# 图片基础卡片插件 - 架构设计

版本：0.1.0  
状态：正式版  

---

## 1. 设计目标

本插件采用“纯插件导出 + 宿主资源桥 + React UI”架构，在不保留任何旧版兼容层的前提下复刻旧版图片基础卡片的正式能力。

核心目标：

- 完整迁移旧版图片配置模型和主要交互能力
- 正式接入卡片根目录资源链路
- 同时适配 Host 托管链路与 EditingEngine 本地运行时
- 保持模块解耦、可测试和可扩展

## 2. 模块划分

### 2.1 `src/schema/card-config.ts`

- 配置校验函数

- 布局派生逻辑

### 2.3 `src/shared/i18n.ts`

- 读取 `zh-CN` / `en-US` 文案
- 创建插件内轻量翻译函数

### 2.4 `src/render/view.tsx`

负责：

- 纯查看态 React 组件
- 根据配置派生有效布局
- 渲染单图 / 网格 / 长图 / 横向滑动四种布局

### 2.5 `src/render/runtime.ts`

负责：

- 渲染态 React 根节点挂载
- 注入 Host / 编辑引擎传入的 `themeCssText`
- 生命周期清理

### 2.6 `src/editor/panel.tsx`

负责：

- 编辑器 React 组件树
- 适配 EditingEngine 右侧窄面板的响应式布局
- 图片导入、URL 添加、排序、删除、清空

- 自定义指针拖拽、浮动缩略图预览、占位插入命中计算

- 无顶部固定工具栏的分区式交互组织
- 图片列表头部仅保留右侧垃圾桶“清空全部”入口，不展示图片总数徽标与撤销 / 重做按钮
- 撤销 / 重做继续保留键盘快捷键能力
- 缩略图资源解析与释放

- 编辑态 React 根节点挂载

- 生命周期清理

## 3. 宿主契约

### 3.1 查看态

- `renderBasecardView({ container, config, themeCssText, resolveResourceUrl, releaseResourceUrl })`
- 对 `source = file` 的图片，优先使用宿主提供的 `resolveResourceUrl(file_path)` 获取运行时 URL
- 当宿主未提供查看态资源解析能力时，回退为配置中的相对 `file_path`
- Host / EditingEngine 负责通过基础卡片 iframe 的 `<base href>` 或运行时 URL 解析卡片根目录资源
- 当卡片从单图状态扩展为多图状态，编辑器默认自动把布局从 `single` 切换到 `grid`，避免继续沿用单图布局造成异常视觉
- 纵向长图布局在查看态始终按内容完整展开，不再引入独立固定高度窗口
- 若临时资源 URL 因切卡等时机失效，查看态在图片加载失败时会主动重新解析一次资源

### 3.2 编辑态

- `renderBasecardEditor({ container, initialConfig, onChange, resolveResourceUrl, releaseResourceUrl, importResource, deleteResource })`
- 图片上传链路：
  1. 选择本地文件
  2. 调用 `importResource({ file, preferredPath })`
  3. 宿主返回 `{ path }`
  4. 插件把 `path` 写入 `images[*].file_path`
- 编辑态缩略图：
  - `source = url` 直接用 `url`
  - `source = file` 必须调用 `resolveResourceUrl(file_path)`
- 对已经持久化到卡片根目录的图片，当前正式宿主优先返回稳定的 `file://` URL；对当前编辑会话里尚未落盘的资源，宿主返回临时 `blob:` URL
- 当编辑器已经提交配置、但 `CardService` 仍在后台写入卡片根目录时，宿主必须继续为这些待落盘资源返回可用的临时 URL，直到文件真正完成持久化，禁止在中间窗口提前切回磁盘路径
- 若资源解析在切卡回切时出现瞬时失败，编辑器会自动进行一次延迟重试；若已拿到的临时 URL 失效，缩略图加载失败时会再次请求资源
- 为了支持“导入后删除 -> 撤销恢复”链路，编辑器内部缓存本次会话导入过的 `File` 对象；当资源被删除后再次恢复引用时，会优先尝试用原文件重新执行 `importResource(...)`
- 释放规则：
- 当图片被移除、重解析或编辑器卸载时，对 `source = file` 已解析资源调用 `releaseResourceUrl(file_path)`
- `releaseResourceUrl(...)` 只负责释放宿主分配的临时运行时资源；对于稳定 `file://` URL，宿主可以安全地视为无操作

### 3.3 编辑引擎注册契约

- 插件入口导出 `basecardDefinition`
- `basecardDefinition.pluginId = "chips.basecard.image"`
- `basecardDefinition.cardType = "base.image"`
- `basecardDefinition.aliases = ["ImageCard"]`
- `basecardDefinition.collectResourcePaths(...)` 负责把 `source = "file"` 的内部资源路径回传给宿主持久化阶段

## 4. 状态模型

### 4.1 持久化配置状态

持久化状态即 `ImageCardConfig`，只包含正式 YAML 字段。

### 4.2 运行时编辑状态

编辑器内部维护：

- 当前草稿配置
- 已解析的图片预览 URL 映射
- 正在上传状态
- 拖拽排序状态
- 浮动拖拽预览状态
- 删除投放区激活状态
- 撤销 / 重做历史栈
- 表单局部错误信息

这些状态都不允许写回正式 YAML。

### 4.3 编辑器布局约束

- 正式编辑器必须能在 `320px` 到 `420px` 的右侧工具面板宽度下完整显示主要控件
- 分区头部动作区、3 列图片网格、列表末尾加号卡片、布局参数区都必须支持响应式换行和窄屏重排
- 长资源路径在窄面板中允许换行显示，不得因单行省略导致关键信息完全不可见
- 图片列表区域需要支持就地继续添加图片，避免用户必须频繁回到顶部工具条
- 图片列表正式采用 3 列无限缩略图插槽编排；布局不依赖 CSS Grid 固定排版，而由编辑器根据容器宽度计算绝对定位插槽
- 列表末尾灰色加号卡片只负责展开列表底部的添加模块，不直接替代添加模块本身
- 拖拽排序采用占位插入式预览：拖拽起始位置先保留空位，目标位置更新后由列表自动让位并形成插入结果
- 图片列表拖拽正式采用自定义指针事件 + 跟手浮动缩略图实现，不依赖原生 HTML5 拖拽拖影
- 跟手浮动缩略图必须提升到 `document.body` 全局浮层渲染，避免被编辑器滚动容器或局部层叠上下文遮挡
- 其他图片的位移必须通过平滑过渡动画完成，交互观感需要接近手机桌面图标整理时的吸附效果
- 删除投放区必须固定在编辑器底部；滚动容器底部要预留额外空间，避免投放区遮挡列表最后一行
- 图片间距正式改为离散模式选择，不再暴露自由数值输入
- 网格溢出计数遮罩必须锚定在最后一个可见网格单元内部，不得以容器级绝对定位覆盖整块网格

## 5. 关键设计决策

- 自动单图模式只在 `images.length <= 1` 时生效；当图片数量大于 `1` 时，多图 `single` 配置在归一化阶段直接改写为 `grid`
- 结构性变更（导入、删除、替换、排序、布局切换）进入撤销 / 重做历史
- 插件本身不设置图片数量上限和文件大小上限，避免把宿主环境约束误固化为插件正式协议
- 资源导入默认优先保留原文件名，冲突处理交由宿主 `importResource(...)`
- 旧版 `resource.write`、`resource.fetch`、Vue 组件、Core 注入全部删除，不再保留任何旧接口痕迹
