# 音乐基础卡片插件架构设计

版本：0.1.0
状态：正式版

## 1. 设计目标

本插件采用“纯插件导出 + 宿主资源桥 + React UI + 嵌入元数据解析”架构，在不引入额外第三方库的前提下，为音乐卡片提供正式的查看、编辑和资源生命周期能力。

核心目标：

- 与图片基础卡片保持一致的卡片根目录资源模型；
- 在查看态通过统一资源打开契约唤起音乐播放器；
- 在编辑态完成音频导入、嵌入元数据自动补全与补充信息编辑；
- 让配置 schema、查看态、编辑态、模板和测试共用同一套字段口径。

## 2. 模块划分

### 2.1 `src/schema/card-config.ts`

负责：

- 音乐卡片正式配置类型定义；
- 默认配置；
- 配置归一函数；
- 配置校验函数。

### 2.2 `src/shared/utils.ts`

负责：

- 卡片根目录相对路径归一；
- 资源路径收集；
- 文件名与扩展名推导；
- 歌手派生与制作团队辅助逻辑。

### 2.3 `src/shared/audio-metadata.ts`

负责：

- 解析 MP3 ID3v2、FLAC 与 M4A/MP4 元数据；
- 提取标题、歌手、专辑、封面、歌词；
- 输出编辑器自动补全所需的中间结果。

### 2.4 `src/shared/default-cover.ts`

负责：

- 暴露插件包内置默认封面 JPG 资源 URL；
- 供查看态与编辑态在 `album_cover` 为空或资源解析失败时统一回退；
- 保持默认占位图与卡片正式资源目录解耦，不写入卡片配置。

### 2.5 `src/render/view.tsx`

负责：

- 渲染长条状音乐卡片；
- 解析音频与封面资源 URL；
- 派生歌曲名、歌手和摘要信息；
- 在宿主提供 `openResource(...)` 时把点击行为转交给外层容器；
- 生成正式 `MusicCardOpenPayload`，把音乐基础卡片配置与已解析资源描述符一并上抛给音乐播放器。

### 2.6 `src/editor/panel.tsx`

负责：

- 音频上传、替换与自动解析；
- 将音频内嵌封面提取为独立图片资源文件；
- 封面与歌词资源管理；
- 制作团队多身份编辑；
- 发行日期、专辑名、语种、流派编辑；
- 表单错误与宿主资源错误展示；
- 在音频未上传前，允许其他可选字段继续参与当前配置草稿编辑与界面回显；
- 采用与视频基础卡片一致的“分组标题 + 列表行 + 资源预览项”布局；
- 在窄窗口下自动折叠为单列设置面板结构。

### 2.7 `src/index.ts`

负责：

- 暴露 `renderBasecardView` / `renderBasecardEditor` 正式导出；
- 暴露 `basecardDefinition`；
- 将资源收集、配置归一与渲染能力统一交给宿主运行时消费。

## 3. 宿主契约

### 3.1 查看态

- `renderBasecardView({ container, config, themeCssText, resolveResourceUrl, releaseResourceUrl, openResource })`
- `audio_file` 与 `album_cover` 优先通过宿主提供的 `resolveResourceUrl(...)` 解析。
- 当宿主提供 `openResource(...)` 时，点击音乐卡片会把音频资源意图上抛给外层统一走 `resource.open`。
- 上抛时除主音频 `resourceId` 外，还会通过 `payload` 正式携带：
  1. 当前音乐基础卡片的正式配置；
  2. 已解析出的音频 / 封面 / 歌词资源描述符；
  3. 当前查看态派生出的展示标题与歌手。

### 3.2 编辑态

- `renderBasecardEditor({ container, initialConfig, onChange, resolveResourceUrl, releaseResourceUrl, importResource, deleteResource, convertTiffToPng })`
- 编辑页采用与视频基础卡片统一的设置面板式结构，不再保留额外大标题和多层表单块。
- 音频上传链路：
  1. 选择本地音频文件；
  2. 调用 `importResource(...)` 写入卡片根目录；
  3. 读取 `File` 二进制并解析嵌入元数据；
  4. 按真实图片字节头识别嵌入封面格式，而不是只信任音频标签声明的 MIME；
  5. 自动补全歌曲名、歌手、专辑名；
  6. 若存在嵌入封面且真实格式不是 TIFF，则再次通过 `importResource(...)` 写入正式独立资源文件；
  7. 若存在嵌入封面且真实格式是 TIFF，则先导入源 TIFF 资源，再调用 `convertTiffToPng(...)` 生成最终 PNG，并删除中间源文件引用；
  8. 若存在嵌入歌词，再次通过 `importResource(...)` 写入正式独立歌词文件；
  9. 通过 `onChange(...)` 输出新的正式配置。
- 替换封面或歌词时，旧资源通过 `deleteResource(...)` 登记删除。
- 若 `album_cover` 为空或资源解析失败，查看态与编辑态统一回退到插件内置默认封面，不向宿主追加任何卡片内部资源写入。
- `convertTiffToPng(...)` 只负责资源格式归一；最终 `album_cover` 字段记录的仍然必须是卡片根目录相对 PNG 路径。

## 4. 关键设计决策

- `audio_file` 是唯一必填正式资源字段，其他字段允许为空。
- 歌手名不单独再建一个独立字段，而是从 `production_team` 中优先派生“歌手”类身份显示。
- 导入新音频时，若音频内嵌标题或专辑信息存在，编辑器会优先采用新音频解析结果更新对应字段；若嵌入信息缺失，则回退保留当前已有值或文件名派生值。
- 资源打开不在基础卡片内部直接启动播放器，而是统一交给宿主资源打开路由。
- 默认占位封面属于插件包静态资源，不进入卡片正式配置，也不参与 `collectResourcePaths(...)`。
- 编辑器中资源存在时显示资源预览项，删除后再回到上传面，减少侧栏里的重复按钮和层级。
- 音频文件尚未上传时，编辑器仍允许用户先编辑标题、制作团队、专辑等可选字段；这些内容先作为当前配置快照参与草稿编辑与界面回显。
- 音乐播放器不应自己猜测卡片目录结构；查看态必须显式把正式播放上下文透传出去，确保“音乐基础卡片 -> 音乐播放器”链路稳定可恢复。
