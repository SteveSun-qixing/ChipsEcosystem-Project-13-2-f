# 富文本基础卡片插件技术方案（Milkdown 版）

版本：2026-03-25  
状态：已实现并对齐当前代码

---

## 1. 现状与目标

### 1.1 当前现状

- 当前 `richtext-BCP` 正式实现已完成：
  - Markdown 内容模型
  - Milkdown 编辑器内核
  - 选区图标悬浮工具框
  - 编辑区右键菜单
  - inline / file 双存储
  - 查看态静态 Markdown 渲染

### 1.2 目标架构

本轮重构只替换富文本插件内部实现，不改变插件在生态中的正式身份与装载方式：

- `manifest.yaml` 继续声明 `base.richtext`
- `basecardDefinition.cardType` 继续为 `base.richtext`
- 内容模型从 **HTML 正文** 切换为 **Markdown 正文**
- 编辑器从 **contenteditable + execCommand** 切换为 **Milkdown 编辑器 + 选区图标悬浮工具框 + 编辑区右键菜单**

### 1.3 兼容策略

- 当前项目未发布第一版，本轮**不做旧 HTML 配置兼容**。
- 代码、测试、README 与内部文档统一以新版 Markdown 契约为准。

## 2. 正式边界

### 2.1 保持不变的正式契约

- 插件仍以 `dist/index.mjs` 作为唯一入口产物。
- 编辑引擎与 Host 继续只消费同一组导出：
  - `renderBasecardView(ctx)`
  - `renderBasecardEditor(ctx)`
  - `basecardDefinition`
- 资源操作继续只走宿主能力：
  - `resolveResourceUrl`
  - `releaseResourceUrl`
  - `importResource`
  - `deleteResource`

### 2.2 本轮不引入的新契约

- 不新增私有 Bridge 写文件接口。
- 不新增富文本专属宿主面板协议。
- 不新增“查看态挂完整编辑器再切只读”的实现路径。


模块职责：

- `schema/card-config.ts`
  - 定义 `RichTextCardConfig`
  - 归一配置、校验字段与来源约束
- `editor/panel.tsx`
  - 封装 Milkdown 实例创建、销毁、内容读写
  - 负责选区图标悬浮工具框、Tooltip、右键菜单与命令分发
- `editor/runtime.ts`
  - 承接 Host 编辑态挂载与卸载
- `render/view.tsx`
  - 负责查看态 Markdown 只读渲染样式
- `render/runtime.ts`
  - 承接 Host 查看态挂载与卸载
- `shared/resource-links.ts`
  - 统一 inline / file 内容读取与相对资源 URL 重写
- `shared/markdown-extensions.ts`
  - 收口 Markdown 解析/序列化扩展、GFM 配置与扩展语法清洗
- `shared/utils.ts`
  - 收口 Markdown 归一、正文字数统计、文件名生成与存储阈值判断

### 3.1 Markdown 扩展层

当前正式 Markdown 能力通过 `shared/markdown-extensions.ts` 统一装配：

- `CommonMark + GFM`
- `==高亮==`
- `++下划线++`
- `^上标^`
- `~下标~`
- `remark-math + KaTeX`

当前正式约束：

- GFM 必须配置 `singleTilde: false`，避免 `~sub~` 被误判为删除线；
- 下划线语法通过插件内自管的 remark 转换层产出 `insert` 语义节点；
- 扩展语法转换后必须统一清理空文本节点，避免 ProseMirror 生成非法空文本节点；
- 所有 `@milkdown/*` 依赖必须固定到同一精确版本，禁止混装不同 minor 版本。

## 4. 编辑态架构

### 4.1 编辑器主流程

1. `renderBasecardEditor(ctx)` 接收当前配置。
2. 运行时根据 `content_source` 读取 Markdown 正文：
   - inline：直接读取 `content_text`
   - file：通过 `resolveResourceUrl(resourcePath)` 获取正式运行时 URL，并按协议选择读取器
     - `file://`：通过 Bridge / SDK `file.read` 读取磁盘正文
     - `blob:` / `http(s):` / 相对 URL：通过 `fetch(resourceUrl)` 读取正文
3. 创建 Milkdown 编辑器实例并载入 Markdown。
4. 监听 Milkdown 文档变化，生成最新 Markdown 字符串。
5. 基于正文可见文本长度执行存储策略判断。
6. 通过 `onChange(nextConfig)` 输出归一后的正式配置快照。
7. 如需写入或删除根目录 Markdown 文件，则通过正式资源能力登记操作。

### 4.2 选区图标悬浮工具框

- 悬浮工具框属于编辑器内部运行时的一部分，不依赖独立宿主窗口。
- 通过监听选区变化、容器滚动、窗口尺寸变化与失焦事件重新定位或关闭。
- 定位锚点来自当前选区的可视矩形。
- 工具框只承载行内文本处理能力。
- 每个按钮当前使用紧凑白底图标按钮承载，优先保证小尺寸编辑区也能完整显示；当前工程实现先以 emoji 作为占位图标，但已保留统一 icon slot。
- Tooltip 遵循 `data-scope="tooltip"` / `data-part="content|arrow"` 契约，按钮通过 `aria-describedby` 关联提示层。
- 点击工具按钮前先恢复选区，命令执行后再同步编辑器内容与浮层状态。

### 4.3 编辑区右键菜单

- 右键菜单与悬浮工具框共享同一个编辑运行时，不引入额外宿主窗口。
- 菜单承载块级结构操作：
  - `setParagraph`
  - `toggleHeading(level)`
  - `toggleBulletList`
  - `toggleOrderedList`
  - `toggleBlockquote`
  - `insertHorizontalRule`
  - `clearMarks`
- 菜单遵循 `data-scope="menu"` / `data-part="content|item"` 契约。
- 菜单提供 `ArrowUp / ArrowDown / Home / End / Escape` 键盘路径。

### 4.4 命令适配层

编辑命令统一映射到 Milkdown 官方命令体系，避免在 UI 组件里散落具体编辑器 API：

- `toggleBold`
- `toggleItalic`
- `toggleInlineCode`
- `toggleLink`
- `setParagraph`
- `toggleHeading(level)`
- `toggleBulletList`
- `toggleOrderedList`
- `toggleBlockquote`
- `insertHorizontalRule`
- `clearMarks`

### 4.5 稳定性要求

- 选区变化只能刷新浮层位置，不能触发编辑器重建。
- 编辑器内容变化只能更新内部状态与外发配置，不能反向覆盖正在输入中的编辑器实例。
- 编辑区根节点与 ProseMirror 正文区必须显式使用文本光标，不能回落成按钮式手型光标。
- 组合输入期间暂停正式提交，待 `compositionend` 后统一发出变更。
- 工具按钮点击与右键菜单命令执行前必须先恢复选区，避免焦点漂移。
- 编辑器滚动、窗口缩放、外部点击与失焦时，Tooltip / 右键菜单必须及时关闭。
- 用户仍在编辑面板内部连续输入时，富文本插件只维护本地编辑器草稿与校验状态，不主动触发正式配置提交。
- 粘贴 Markdown 文本时，编辑器必须拦截原生粘贴并把剪贴板正文按 Markdown 解析后插入文档，不能先以普通字符落入编辑区再等待后续输入触发结构刷新。
- 当前正式链路拆成三层：
  - 编辑器本地草稿：响应逐字输入与 Markdown 输入规则；
  - 复合卡片预览草稿：以短防抖节奏输出 inline Markdown 配置，驱动复合卡片近实时刷新；
  - 正式持久化配置：在编辑器失焦、编辑面板失焦或卸载时再统一收敛为最终存储形态。

## 5. 持久化设计

### 5.1 单一真实内容源

- 插件内部唯一真实内容源为 Markdown 字符串。
- HTML 只允许作为查看态最终渲染产物存在，不能写回配置。

### 5.2 资源写入策略

当正文超过 200 字时，插件直接使用既有资源导入链路：

- 通过 `new File([markdown], fileName, { type: "text/markdown" })` 构造 Markdown 文件对象。
- 调用 `importResource({ file, preferredPath })`，由宿主负责把资源写入卡片根目录。
- 配置中只记录宿主返回的卡片根目录相对路径。
- 长文本文件模式的资源同步必须走“本地草稿优先、失焦 flush 再导入”的节奏，不能在每次按键后立即触发文件导入与预览刷新。
- 当前正式实现中，编辑面板内部的连续输入只更新本地草稿；当用户焦点离开编辑面板或编辑器被卸载时，才统一把最新 Markdown 输出为正式配置并进入宿主持久化链路。
- 为了让复合卡片预览及时响应，编辑期内会以 inline Markdown 形式推送一份运行时预览草稿；这份草稿只用于编辑态显示，不代表最终落盘结构。

当正文缩短回 200 字以内时：

- 配置切回 inline 模式；
- 旧 `.md` 文件通过 `deleteResource(resourcePath)` 登记删除；
- 资源物理删除时机仍由宿主持久化阶段统一控制。

### 5.3 为什么不新增宿主接口

当前正式资源链路已经满足需要：

- 可以写文件
- 可以分配根目录相对路径
- 可以删除旧资源
- 可以在编辑态解析资源 URL

因此本轮重构不需要新增“写文本文件专用 API”。

### 5.4 为什么不直接 `fetch(file://)`

当前编辑引擎对“已落盘的卡片根目录资源”正式返回稳定 `file://` URL，对“待落盘但已进入当前会话”的资源则可能仍持有内存中的临时字节内容。图片卡片的 DOM 展示链路可以继续直接消费 URL，因为图片只需要把地址交给 `<img src>`。

富文本不同：Markdown 文件属于“先读取正文再渲染”的资源，运行时需要获得正文字符串，而不是把 URL 直接交给 DOM。因此正式实现采用以下单一路径规则：

- `resolveResourceUrl(...)` 负责提供统一的运行时定位结果；
- 若定位结果是 `file://`，富文本插件通过正式文件服务读取文本；
- 若定位结果是 `blob:`、`http(s):`、受控渲染协议 URL 或其他可访问 URL，再走 `fetch(...)`；
- Markdown 解析始终由富文本插件自己完成，不再依赖宿主提供类型专属正文读取接口。

这样可以同时覆盖编辑引擎本地运行时的待导入资源、稳定磁盘 URL 与其他通用运行时的可访问 URL，并保持“资源 URL 由宿主提供、内容解析由请求方负责”的统一边界。

## 6. 查看态架构

### 6.1 查看态目标

查看态不再复用旧版“直接渲染 HTML 字符串”的实现，而是改为：

1. 读取 Markdown 正文
2. 转换为只读渲染结构
3. 输出纯展示 DOM

### 6.2 当前实现

- 查看态使用同一份 Markdown 正式内容源。
- 查看态只保留静态渲染结果，不挂载编辑器实例、命令层与浮层逻辑。
- 查看态根节点保持透明，不增加边框、阴影、圆角壳层。
- 相对资源 URL 通过正式资源解析链路重写。
- 查看态与编辑态必须共享同一套 Markdown 扩展装配，避免扩展语法在编辑器中可见而查看态失真。

### 6.3 渲染失败策略

- 文件模式下若 Markdown 文件读取失败，仅隔离当前基础卡片。
- 文件模式下的正文读取同样遵循协议分派：
  - `file://` 走正式文件服务；
  - 其他运行时 URL 走 `fetch(...)`。
- 当前基础卡片显示局部错误态，不影响整张复合卡片。
- 错误信息仍需走 i18n。

## 8. 风险与控制策略

### 8.1 已确认决策

- 采用 Milkdown 官方生态，不再沿用旧版 HTML 编辑方案。
- 删除固定工具栏路径，正式编辑入口为选区图标悬浮工具框与编辑区右键菜单。
- Markdown 是唯一正式持久化格式。
- 不做旧 HTML 配置兼容。

### 8.2 主要风险

- 选区浮层定位与编辑器焦点切换容易引发选区丢失。
- file 模式需要保证读取、写入、切回 inline 三段链路一致。
- 查看态若直接挂完整编辑器，会造成体积与性能浪费。

### 8.3 对应控制策略

- 命令触发统一走“保存选区 -> 恢复选区 -> 执行命令 -> 提交内容”链路。
- 内容来源读取统一收口到 `shared/resource-links.ts`。
- Markdown 语法扩展统一收口到 `shared/markdown-extensions.ts`，禁止在编辑态和查看态各自散落一份配置。
- 存储策略统一收口到 `shared/utils.ts` 与 `schema/card-config.ts`。
