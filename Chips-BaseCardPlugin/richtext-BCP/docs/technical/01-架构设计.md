# 富文本基础卡片插件重构技术方案（Milkdown 版）

版本：2026-03-24  
状态：设计定稿，待开发  

---

## 1. 现状与目标

### 1.1 当前现状

- 当前 `richtext-BCP` 已有实现基于：
  - HTML 正文字符串
  - `contenteditable`
  - 固定工具栏
  - 旧版 `sanitizeRichTextHtml` 清洗链路
- 当前正式导出契约已经对齐生态要求：
  - `renderBasecardView`
  - `renderBasecardEditor`
  - `basecardDefinition`

### 1.2 目标架构

本轮重构只替换富文本插件内部实现，不改变插件在生态中的正式身份与装载方式：

- `manifest.yaml` 继续声明 `base.richtext`
- `basecardDefinition.cardType` 继续为 `base.richtext`
- 内容模型从 **HTML 正文** 切换为 **Markdown 正文**
- 编辑器从 **contenteditable + execCommand** 切换为 **Milkdown 编辑器 + 选区悬浮工具框**

### 1.3 兼容策略

- 当前项目未发布第一版，本轮**不做旧 HTML 配置兼容**。
- 后续实现允许直接切换配置契约、测试基线和模板，不保留“双格式并存”设计。

---

## 2. 正式边界

### 2.1 保持不变的正式契约

- 插件仍以 `dist/index.mjs` 作为唯一入口产物。
- 编辑引擎与 Host 继续只消费同一组导出：
  - `renderBasecardView(ctx)`
  - `renderBasecardEditor(ctx)`
  - `basecardDefinition`
- 资源操作继续只走宿主能力：
  - `resolveResourceUrl`
  - `releaseResourceUrl`
  - `importResource`
  - `deleteResource`

### 2.2 本轮不引入的新契约

- 不新增私有 Bridge 写文件接口。
- 不新增富文本专属宿主面板协议。
- 不新增“查看态挂完整编辑器再切只读”的实现路径。

---

## 3. 目标模块拆分

建议的目标模块结构如下：

```text
src/
  index.ts
  schema/
    card-config.ts
  editor/
    runtime.ts
    milkdown-editor.ts
    selection-toolbar.ts
    command-adapter.ts
  render/
    runtime.ts
    markdown-view.ts
    markdown-source-loader.ts
  shared/
    markdown-storage.ts
    plain-text-length.ts
    markdown-normalize.ts
    i18n.ts
```

模块职责：

- `schema/card-config.ts`
  - 定义新版 `RichTextCardConfig`
  - 归一配置、校验字段与来源约束
- `editor/milkdown-editor.ts`
  - 封装 Milkdown 实例创建、销毁、内容读写
- `editor/selection-toolbar.ts`
  - 负责选区悬浮工具框显示、定位、命令分发
- `editor/command-adapter.ts`
  - 把工具框动作映射到 Milkdown 命令
- `shared/markdown-storage.ts`
  - 统一 inline / file 两种持久化策略
- `shared/plain-text-length.ts`
  - 统一正文“200 字阈值”计数口径
- `render/markdown-source-loader.ts`
  - 统一加载 inline Markdown 或 file Markdown
- `render/markdown-view.ts`
  - 把 Markdown 渲染为只读内容结构

## 4. 编辑态架构

### 4.1 编辑器主流程

1. `renderBasecardEditor(ctx)` 接收当前配置。
2. 运行时根据 `content_source` 读取 Markdown 正文：
   - inline：直接读取 `content_text`
   - file：通过 `resolveResourceUrl -> fetch(resourceUrl)` 读取 `.md`
3. 创建 Milkdown 编辑器实例并载入 Markdown。
4. 监听 Milkdown 文档变化，生成最新 Markdown 字符串。
5. 基于正文可见文本长度执行存储策略判断。
6. 通过 `onChange(nextConfig)` 输出归一后的正式配置快照。
7. 如需写入或删除根目录 Markdown 文件，则通过正式资源能力登记操作。

### 4.2 选区悬浮工具框

悬浮工具框不属于独立宿主窗口，而是编辑器内部运行时的一部分：

- 它由编辑器运行时挂载到当前容器的浮层层级。
- 通过监听编辑器选区变化、容器滚动与窗口尺寸变化重新定位。
- 定位锚点来自当前选区的可视矩形。
- 点击工具按钮后，命令直接作用到当前 Milkdown 选区。

### 4.3 命令适配层

命令适配层负责把 UI 动作映射到 Milkdown 官方命令体系，避免在 UI 组件里散落具体编辑器 API：

- `toggleBold`
- `toggleItalic`
- `toggleStrike`
- `toggleInlineCode`
- `toggleLink`
- `setParagraph`
- `toggleHeading(level)`
- `toggleBulletList`
- `toggleOrderedList`
- `toggleBlockquote`
- `insertHorizontalRule`
- `clearMarks`

### 4.4 稳定性要求

- 选区变化只能刷新浮层位置，不能触发编辑器重建。
- 编辑器内容变化只能更新内部状态与外发配置，不能反向覆盖正在输入中的编辑器实例。
- 组合输入期间暂停正式提交，待 `compositionend` 后统一发出变更。

## 5. 持久化设计

### 5.1 单一真实内容源

- 插件内部唯一真实内容源为 Markdown 字符串。
- HTML 只允许作为查看态最终渲染产物存在，不能写回配置。

### 5.2 资源写入策略

当正文超过 200 字时，插件不需要新增宿主协议，可以直接使用既有资源导入链路：

- 通过 `new File([markdown], fileName, { type: "text/markdown" })` 构造 Markdown 文件对象。
- 调用 `importResource({ file, preferredPath })`，由宿主负责把资源写入卡片根目录。
- 配置中只记录宿主返回的卡片根目录相对路径。

当正文缩短回 200 字以内时：

- 配置切回 inline 模式；
- 旧 `.md` 文件通过 `deleteResource(resourcePath)` 登记删除；
- 资源物理删除时机仍由宿主持久化阶段统一控制。

### 5.3 为什么不新增宿主接口

当前正式资源链路已经满足需要：

- 可以写文件
- 可以分配根目录相对路径
- 可以删除旧资源
- 可以在编辑态解析资源 URL

因此本轮重构不需要新增“写文本文件专用 API”。

## 6. 查看态架构

### 6.1 查看态目标

查看态不再复用旧版“直接渲染 HTML 字符串”的实现，而是改为：

1. 读取 Markdown 正文
2. 转换为只读渲染结构
3. 输出纯展示 DOM

### 6.2 推荐实现

- 继续使用 Milkdown 官方生态中的 Markdown 语义转换能力，确保编辑态与查看态语义一致。
- 查看态只使用静态渲染结果，不保留编辑器实例、命令层与浮层逻辑。
- 查看态根节点保持透明，不增加边框、阴影、圆角壳层。

### 6.3 渲染失败策略

- 文件模式下若 Markdown 文件读取失败，仅隔离当前基础卡片。
- 当前基础卡片显示局部错误态，不影响整张复合卡片。
- 错误信息仍需走 i18n。

## 8. 风险与决策

### 8.1 已确认决策

- 采用 Milkdown 官方生态，不再沿用旧版 HTML 编辑方案。
- 删除固定工具栏路径，正式编辑入口只有选区悬浮工具框。
- Markdown 是唯一正式持久化格式。
- 不做旧 HTML 配置兼容。

### 8.2 主要风险

- 选区浮层定位与编辑器焦点切换容易引发选区丢失。
- file 模式需要保证读取、写入、切回 inline 三段链路一致。
- 查看态若直接挂完整编辑器，会造成体积与性能浪费，因此必须坚持静态渲染实现。

### 8.3 对应控制策略

- 命令触发统一走“保存选区 -> 执行命令 -> 恢复焦点”链路。
- 内容来源读取统一收口到 `markdown-source-loader`。
- 存储策略统一收口到 `markdown-storage`，避免编辑器和 schema 分别判断。
