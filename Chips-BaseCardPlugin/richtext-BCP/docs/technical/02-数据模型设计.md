# 富文本基础卡片插件重构数据模型设计（Milkdown 版）

版本：2026-04-03  
状态：已实现并对齐当前代码  

---

## 1. 目标数据模型

新版富文本基础卡片配置改为以 Markdown 为正式内容源。

```ts
export interface RichTextCardConfig {
  card_type: "RichTextCard";
  theme?: string;
  locale?: string;
  content_format: "markdown";
  content_source: "inline" | "file";
  content_text?: string;
  content_file?: string;
}
```

设计原则：

- `card_type` 保持生态既有口径；
- `content_format` 显式声明为 `markdown`，避免再次混入 HTML 真源；
- `content_source` 描述内容来自配置内联还是根目录 Markdown 文件；
- `content_text` 与 `content_file` 两者只允许存在其一。

---

## 2. 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `card_type` | string | 是 | 固定为 `RichTextCard` |
| `theme` | string | 否 | 主题覆盖标识，空值表示沿用上层主题 |
| `locale` | string | 否 | 当前卡片语言代码，如 `zh-CN` |
| `content_format` | string | 是 | 固定为 `markdown` |
| `content_source` | string | 是 | `inline` 或 `file` |
| `content_text` | string | 条件必填 | `content_source = inline` 时必填，保存 Markdown 文本 |
| `content_file` | string | 条件必填 | `content_source = file` 时必填，保存卡片根目录相对路径 |

---

## 3. 归一规则

### 3.1 基础归一

- 缺失 `card_type` 时，归一为 `RichTextCard`
- 缺失 `content_format` 时，归一为 `markdown`
- 缺失 `content_source` 时，按以下顺序推断：
  1. 有 `content_file` 时推断为 `file`
  2. 否则推断为 `inline`

### 3.2 字段互斥规则

- `content_source = inline` 时：
  - 必须保留 `content_text`
  - 必须清空 `content_file`
- `content_source = file` 时：
  - 必须保留 `content_file`
  - 必须清空 `content_text`

### 3.3 格式约束

- `content_text` 保存原始 Markdown，不保存 HTML。
- `content_file` 必须是卡片根目录相对路径，且扩展名固定为 `.md`。
- 不允许把下列值写回配置：
  - `blob:` URL
  - `file://` 绝对路径
  - 系统磁盘路径
  - 临时缓存 key

## 4. 200 字分流算法

### 4.1 计数字段口径

“200 个字”采用**正文可见文本长度**口径，而不是 Markdown 源字符串长度。

计数步骤：

1. 基于编辑器当前文档提取可见纯文本；
2. 去除零宽字符与仅用于编辑态占位的控制字符；
3. 把连续空白归一为单个空格；
4. 去除已正式支持的 Markdown 语法包裹符号：
   - `~~ ~~`
   - `== ==`
   - `++ ++`
   - `^ ^`
   - `~ ~`
   - 数学公式定界符 `$` / `$$`
5. 使用 Unicode 字符计数（`Array.from(text).length`）。

### 4.2 分流规则

- `plainTextLength <= 200`：
  - 使用 inline 模式；
  - Markdown 写入 `content_text`。
- `plainTextLength > 200`：
  - 使用 file 模式；
  - Markdown 写入卡片根目录 `.md` 文件；
  - 配置中写入 `content_file`。

### 4.3 切换行为

- inline → file：
  - 创建或覆盖当前基础卡片对应 Markdown 文件；
  - 清空 `content_text`；
  - 写入 `content_file`。
- file → inline：
  - 读取当前 Markdown 正文回填 `content_text`；
  - 清空 `content_file`；
  - 登记旧 Markdown 文件删除意图。

## 5. 文件命名

Markdown 文件统一保存到卡片根目录，推荐命名：

```text
richtext-{baseCardId}.md
```

示例：

```text
richtext-a1B2c3D4e5.md
```

设计理由：

- 与 `content/{baseCardId}.yaml` 一一对应；
- 可读性强；
- 不依赖用户手工命名；
- 在单张卡片内天然避免冲突。

## 6. 配置示例

### 6.1 内联模式

```yaml
card_type: "RichTextCard"
theme: ""
locale: "zh-CN"
content_format: "markdown"
content_source: "inline"
content_text: |
  # 今日记录

  这是一个较短的富文本内容。
```

### 6.2 文件模式

```yaml
card_type: "RichTextCard"
theme: ""
locale: "zh-CN"
content_format: "markdown"
content_source: "file"
content_file: "richtext-a1B2c3D4e5.md"
```

对应卡片根目录文件：

```md
# 长文草稿

这里保存超过 200 字的富文本 Markdown 正文……
```

## 7. 读写链路

### 7.1 编辑态读取

- `inline`：直接读取 `content_text`
- `file`：
  - 调用 `resolveResourceUrl(content_file)`
  - 通过返回的受控资源 URL 读取 Markdown 文本

### 7.2 编辑态写入

- inline：
  - 直接通过 `onChange(nextConfig)` 输出配置
- file：
  - 先构造 Markdown `File`
  - 再调用 `importResource(...)`
  - 最终把宿主返回的相对路径写入 `content_file`

### 7.3 查看态读取

- 查看态与编辑态使用同一套 Markdown 加载逻辑；
- 差异仅在于查看态只渲染静态内容，不挂编辑器实例。

---

## 8. 校验规则

正式校验基线如下：

1. `card_type` 必须为 `RichTextCard`
2. `content_format` 必须为 `markdown`
3. `content_source` 必须为 `inline | file`
4. `inline` 模式下 `content_text` 必须存在且不能全空
5. `file` 模式下 `content_file` 必须存在且为根目录相对 `.md` 路径
6. 同一时刻不允许同时持有 `content_text` 与 `content_file`
