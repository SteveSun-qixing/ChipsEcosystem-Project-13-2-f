# 阶段五：文档完善与发布验收

> 阶段目标：在前四个阶段实现与测试基础上，完善脚手架及相关文档，对齐生态共用口径，完成首个可对外宣布版本的发布验收。

---

## 1. 阶段范围

- 审视并完善本仓所有文档（需求/技术/开发计划），消除与实现不一致之处；
- 与生态共用文档维护者协作，补充或更新与应用插件脚手架相关的正式章节；
- 制定并执行发布流程，包括版本号规划、变更日志、发布包构建与验证；
- 对本仓相关的问题工单进行收尾与状态更新。

---

## 2. 关键任务

### 2.1 文档一致性与可读性检查

- 校对以下文档与实际实现：
  - `chips-scaffold-app/需求文档/*`；
  - `chips-scaffold-app/技术文档/*`；
  - `chips-scaffold-app/开发计划/*`；
- 检查并修正：
  - 目录结构、函数签名、文件命名等描述是否与代码与模板一致；
  - 是否存在已过时的暂定口径或 TODO/占位描述；
  - 引用路径是否指向真实存在的文件。

### 2.2 生态共用文档对齐

- 与架构/规范维护者协作，确认以下信息是否需要沉淀到 `生态共用技术文档`：
  - 应用插件工程推荐目录结构；
  - 脚手架/模板体系在生态中的正式定位与使用说明；
  - 与 `chips dev create` 相关的跨仓工作流说明；
- 如需要：
  - 在 `生态共用技术文档/插件开发` 或 `命令行工具` 中新增或更新对应章节；
  - 在本仓文档中将原先作为“本地约定”的内容调整为“引用生态共用文档”。

### 2.3 发布规划与执行

- 版本号规划：
  - 确定脚手架的首个正式版本号（例如 `1.0.0` 或与整体生态版本同步）；
  - 制定后续版本演进策略（仅小版本/补丁版本等）。
- 构建与发布：
  - 通过 CI 或本地脚本构建发布包（例如 NPM 包或内部分发制品）；
  - 在受控环境中验证发布包的可安装性与可用性；
  - 在发布说明中明确：
    - 支持的模板清单；
    - 已知限制与后续规划；
    - 与 `chips dev` 的版本依赖关系。

### 2.4 问题工单与任务收尾

- 梳理与本仓相关的问题工单：
  - 标记已通过实现与测试闭环的工单为“已闭环”；
  - 对仍未解决但不阻断发布的问题，记录为“已知限制”，并规划到后续阶段；
- 在 `项目日志与笔记/任务清单/` 中添加本阶段总结或归档已有任务说明文档。

---

## 3. 依赖与前置条件

- 阶段一至四已通过各自的验收标准；
- 相关生态仓库（`chips dev`、Host/SDK、组件库）在当前版本下已无影响本脚手架发布的阻断缺陷；
- 如仍存在阻断缺陷，必须先通过问题工单与专项任务闭环后再考虑正式发布。

---

## 4. 验收标准

- 文档侧：
  - 本仓全部文档内容与实现保持一致，无明显错误与占位内容；
  - 生态共用文档中已包含必要的脚手架相关正式信息（如被确认需要）；
- 功能与质量侧：
  - 在受支持平台上按发布前检查清单运行全部质量门禁脚本（lint/test/模板一致性/E2E）均通过；
  - 通过 `chips dev create` + 模板组合的端到端工作流稳定可用；
- 管理侧：
  - 本仓相关问题工单状态已更新，且发布说明中包含必要的“已知限制”与后续计划说明；
  - 发布版本号、发布时间与主要变更记录已归档，便于后续追踪与回滚。

---

## 5. 输出物

- 经校对的最新文档集（需求/技术/开发计划）；
- 至少一个正式发布版本的脚手架包及其发布说明；
- 更新后的生态共用文档片段（如有变更）；
- 阶段总结，形成对“第一版应用插件脚手架”的整体评价与后续演进方向建议。
