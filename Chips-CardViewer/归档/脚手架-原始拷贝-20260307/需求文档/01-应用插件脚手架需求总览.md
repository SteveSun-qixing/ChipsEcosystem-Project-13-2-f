# 应用插件脚手架需求总览

> 文档状态：需求基线稿（用于从零开发立项与范围冻结）  
> 对齐基线：生态设计原稿 + 生态共用技术文档（vNext 冻结口径）

## 1. 项目背景与目标

薯片生态采用“一切皆插件”的扩展模型，所有上层应用（编辑器、查看器、资源管理器、设置面板等）均以 **应用插件（`type: app`）** 形式运行在薯片主机内部。为了降低开发者门槛、统一项目结构与质量门禁，需要提供一套官方的 **应用插件脚手架**：

- 通过 `chips dev create` 命令创建新应用插件项目时，默认使用本仓库提供的模板；
- 创建出的项目天然符合生态的架构分层、插件契约、Manifest 规范、多语言与主题规范；
- 创建后的项目在不修改任何代码的前提下即可通过 `chips dev build/test/validate` 等基础门禁。

应用插件脚手架自身是 **开发阶段工具** 而非运行时代码：

- 运行时代码依然由新建的应用插件仓库维护；
- 脚手架只负责生成初始文件结构、配置与示例代码，并不参与运行时装载。

### 1.1 北极星目标

- 提供一套可持续 5–10 年演进的官方应用插件模板体系，覆盖应用插件的主线场景（单窗口应用、多卡片工作台、设置/工具类窗口等）的基础骨架需求。
- 任何通过脚手架创建的应用插件，在遵循模板约定的前提下，都能：
  - 正确声明 `type: app` 的 Manifest 与必要权限；
  - 使用 React + 薯片组件库完成基础 UI 搭建；
  - 通过 Bridge API 或 SDK 正确调用 Host 能力；
  - 内建多语言、主题接入与日志/错误处理基础设施；
  - 拥有可运行的单元测试与端到端测试示例，并通过基础测试门禁。
- 在不修改 Host/SDK/组件库与全局契约的前提下，完成“脚手架 → 新应用插件 → Host 运行时”的端到端链路闭环。

### 1.2 本轮开发目标

本轮任务聚焦于 **从零建设 `Chips-Scaffold/chips-scaffold-app` 仓库的文档与规划基线**，具体包括：

- 完成应用插件脚手架的项目级需求文档：
  - 《01-应用插件脚手架需求总览》
  - 《02-应用插件脚手架功能需求规格》
  - 《03-应用插件脚手架非功能与质量验收需求》
- 完成应用插件脚手架的技术文档骨架：
  - 总体架构、模板组织、参数化与配置、命令行集成、测试与质量门禁等章节；
- 完成分阶段的开发计划：
  - 阶段一：架构与需求冻结（当前阶段，仅文档与规划，不编写实现代码）；
  - 阶段二及后续：模板实现、CLI 集成、测试体系搭建与发布验收（详见 `开发计划/`）。

后续阶段在实现代码前，如发现生态层面的架构缺陷或能力缺口，必须先在《项目日志与笔记/问题工单.md》中登记工单，再继续推进。

### 1.3 成果边界与输出物

本项目最终输出物包括但不限于：

- 一个可通过 `chips dev create` 直接消费的应用插件项目模板集（最小集为“标准应用插件模板”）；
- 与模板配套的 `package.json/tsconfig.json/chips.config.*` 等配置文件规划；
- 完整的应用插件脚手架需求文档、技术文档与开发计划；
- 与生态共用文档对齐的规范引用与补充说明（新出现的跨生态正式口径需沉淀到生态共用文档）。

不直接输出的内容：

- Host 内部实现改动（Kernel/Services/Runtime/渲染引擎等）；
- SDK 与组件库内部实现改动；
- 第三方文档或对外市场宣传材料（仅提供技术基线，不承担市场文案）。

## 2. 范围与边界

### 2.1 In Scope（本项目范围内）

- 应用插件脚手架项目本身的需求、架构设计与实现：
  - 模板目录结构与多模板组织方案；
  - 模板文件内容与占位变量设计；
  - 脚手架对 `chips dev create` 命令的对接方式；
  - 模板内建的多语言、主题接入、日志与错误处理、测试骨架等；
- 与其他仓库的集成接口说明：
  - 如何在模板中消费 `Chips-SDK`（如有）、组件库与 Bridge API；
  - 如何保证生成项目的 Manifest、权限声明与 Host 对外接口基线保持一致；
- 本仓库自身的开发计划、质量门禁与发布流程说明。

### 2.2 Out of Scope（本项目不做）

- 不在本仓直接实现或修改：
  - Host 内核、服务域与插件运行时；
  - 组件库与主题包内部实现；
  - SDK 内部实现与对外 API；
- 不承担以下职责：
  - 设计新的插件类型或改变现有 `app/card/layout/module/theme` 类型系统；
  - 修改生态共用文档中已冻结的协议、文件格式、Bridge API 契约；
  - 引入超出生态既有技术选型的新第三方库（如新增路由框架、状态管理库等）。

### 2.3 信息归属边界

- 插件开发规范、Manifest 字段语义、Bridge API、Host 对外接口等 **跨生态正式口径**，以 `生态共用技术文档` 为准，本项目文档仅引用与补充说明，不重新定义。
- 应用插件脚手架自身的目录结构、模板变量命名约定、脚手架内部实现细节，仅在本仓 `需求文档/技术文档/开发计划` 中维护。
- 若后续确认某些约定（例如“应用插件项目的推荐目录结构”）需要对所有生态项目生效，则应：
  1. 先在 `生态共用技术文档` 中补充正式章节；
  2. 再在本仓文档中引用该章节，不在本仓维护独立口径。

## 3. 不可违反约束（硬性需求）

### 3.1 架构与生态约束

- **不得改变既有架构设计**：
  - 必须遵守 12 层架构与插件架构设计文档中的层级边界与依赖约束；
  - 应用插件模板必须以 `type: app` 插件形态运行，不得引入新的运行时层级。
- **协议与契约优先**：
  - 所有模板示例代码调用 Host 能力时，必须通过 Bridge API 或 SDK，禁止越层访问；
  - Manifest 字段、权限声明、能力枚举等必须符合《Manifest配置规范》《插件契约规范》《Bridge-API规范》；
- **统一技术栈与组件策略**：
  - 官方应用插件模板必须以 React 为主技术栈，优先集成薯片组件库与主题系统；
  - UI 组件应遵循无头组件与组合模式原则（参考 `vercel-composition-patterns` 与组件库规范），避免硬编码样式与 boolean props 泛滥。

### 3.2 工程与质量约束

- 禁止临时版、简化版、占位实现：
  - 模板内不得存在“TODO”、“占位组件”、“空实现函数”等未完成代码；
  - 生成项目必须是可以直接运行、构建与通过基础测试的完整工程。
- 禁止越界修改与未审批依赖：
  - 不得修改任务范围外的仓库目录与文件；
  - 不得引入未在生态技术选型中出现的第三方库，如确有需要必须先提工单并经确认后再实施。
- 测试与门禁：
  - 所有模板必须内建单元测试与基础端到端测试示例；
  - 生成项目在默认配置下必须通过 `npm test` / `chips dev test`、`chips dev validate` 等基础门禁。

### 3.3 问题工单与阻断策略

- 如在脚手架设计或后续实现过程中发现：
  - 生态层面的架构缺陷或能力缺口（例如 Host 未提供必需接口、组件库缺失关键组件等）；
  - 现有规范之间存在冲突且影响脚手架可实现性；
  - `chips dev` 工具链尚未提供脚手架对接所需的扩展点；
- 则必须：
  1. 在《项目日志与笔记/问题工单.md》中登记新工单，说明问题与期望；
  2. 在 `项目日志与笔记/任务清单/` 下创建对应任务文件夹与说明文档；
  3. 暂停当前脚手架实现工作，等待用户处理与决策后再继续推进。

## 4. 干系人与协作边界

| 干系人 | 角色 | 关注点 |
|---|---|---|
| 插件开发者 | 使用脚手架创建应用插件的第一方开发者 | 创建命令简单可靠、项目结构清晰、默认即具备多语言与主题支持、易于扩展 |
| Chips Dev 工具链 | 提供 `chips dev create/build/test/validate` 的命令行工具 | 脚手架导出的模板结构稳定、易于集成、错误信息可诊断 |
| Chips-Host 团队 | 维护 Host 内核、服务与插件运行时 | 脚手架生成的应用插件严格遵守 Host 对外接口基线与权限模型，不引入新的运行时代码路径 |
| Chips-SDK 团队 | 维护 SDK 封装与开发者工具 | 模板示例正确使用 SDK 能力，不绕过或重复实现 SDK 功能 |
| 组件库与主题团队 | 维护无头组件与主题系统 | 模板中正确接入组件库 Provider 与主题包，不硬编码视觉样式，不破坏 token 体系 |
| 生态架构与规范维护者 | 维护生态共用文档与长线架构 | 脚手架文档与实现与全局规范口径一致，新增正式规范时能够同步回生态共用文档 |

## 5. 里程碑与阶段划分

应用插件脚手架项目的开发阶段规划如下（详细见 `开发计划/00-开发计划索引.md`）：

1. **阶段一：架构与需求冻结**  
   - 阅读并对齐生态设计原稿与生态共用技术文档；  
   - 完成本仓需求文档、技术文档骨架与开发计划拆解；  
   - 不编写任何实现代码，确保方向正确且范围清晰。

2. **阶段二：基础模板骨架实现**  
   - 完成标准应用插件模板的目录结构与配置文件；  
   - 内建最小可用的 React 页面、多语言与主题接入、Bridge/SDK 示例调用；  
   - 保证模板在本地可通过构建与基础测试。

3. **阶段三：扩展模板与生态对接**  
   - 设计并实现适用于不同场景的应用插件模板变体（例如查看器型、工具面板型等）；  
   - 与 `chips dev create` 命令完成对接，支持按类型或模板名创建工程；  
   - 补充与 Host/SDK/组件库集成的契约测试。

4. **阶段四：测试体系与质量门禁**  
   - 建立脚手架自身的测试集（模板一致性测试、生成项目 E2E 验证等）；  
   - 定义并实现发布前质量门禁脚本；  
   - 确保后续模板演进不会破坏已有开发者项目的基本假设。

5. **阶段五：文档完善与发布验收**  
   - 完善使用文档与示例说明；  
   - 对齐并必要时补充生态共用技术文档中的脚手架相关章节；  
   - 形成第一版可对外宣布的应用插件脚手架发布版本。

上述阶段仅描述技术实施顺序，不涉及具体人力与时间排期。每个阶段结束前必须完成对应文档与测试的更新，方可进入下一阶段。
