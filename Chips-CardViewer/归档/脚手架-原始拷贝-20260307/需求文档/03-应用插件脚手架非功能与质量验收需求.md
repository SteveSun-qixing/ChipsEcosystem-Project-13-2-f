# 应用插件脚手架非功能与质量验收需求

> 文档状态：质量基线稿  
> 说明：用于定义脚手架自身与生成工程的发布门禁，未达标不得合并至 `develop`。

---

## 1. 性能需求（NFR-SCAFFOLD-PERF）

### NFR-SCAFFOLD-PERF-001 脚手架执行时间

- 在常规开发机（配置基线由生态统一文档定义）上，执行一次 `chips dev create` 创建标准应用插件工程：
  - 模板渲染与文件生成阶段应在 3 秒内完成（不含依赖安装时间）；
  - 如需要额外初始化步骤（例如 Git 仓库初始化），总体耗时应可控并在文档中说明。

### NFR-SCAFFOLD-PERF-002 生成工程启动体验

- 在不改动模板代码的前提下，生成工程在安装依赖后：
  - 启动开发服务器的首屏加载时间应满足生态对前端应用的基本性能要求（具体阈值在组件库/Host 性能规范中定义）；
  - 模板不得引入明显的性能反模式（如同步阻塞大量计算、无必要的瀑布式网络请求等）。

---

## 2. 可靠性与鲁棒性需求（NFR-SCAFFOLD-REL）

### NFR-SCAFFOLD-REL-001 创建过程幂等性

- 在相同参数下重复执行创建命令时：
  - 如目标目录不存在，应生成结构一致的工程；
  - 如目标目录已存在，应给出明确错误提示，并避免覆盖用户已有文件，除非用户在交互式流程中明确选择覆盖策略（覆盖策略需在后续阶段单独设计与论证）。

### NFR-SCAFFOLD-REL-002 失败恢复

- 当创建过程因异常中断（例如磁盘写入失败、权限不足等）时：
  - 脚手架应尽量避免留下半成品目录结构；
  - 无法自动回滚时，应输出清晰的人工清理指引（包括路径列表）。

### NFR-SCAFFOLD-REL-003 生成工程稳定性

- 生成工程在未改动代码的前提下：
  - `npm run dev` / `chips dev server` 不应频繁崩溃或热重载失败；
  - `chips dev build` 不应因为模板自身问题导致构建失败；
  - 如依赖生态其他仓库能力（如 SDK、组件库），需在技术文档中明确版本与兼容性前提。

---

## 3. 安全需求（NFR-SCAFFOLD-SEC）

### NFR-SCAFFOLD-SEC-001 权限最小化原则

- 模板生成的 `manifest.yaml` 必须遵循最小权限原则：
  - 仅声明示例代码实际必需的权限；
  - 不预先打开高危能力（如文件写入、网络访问、凭证管理等）；
  - 如示例中需要使用高危能力，应在 README 与技术文档中显著标注用途与风险。

### NFR-SCAFFOLD-SEC-002 敏感信息处理

- 模板中不得包含：
  - 硬编码的访问令牌、密码、API Key 等敏感信息；
  - 指向生产环境的真实服务地址；
- 示例配置如需展示敏感字段，应使用明显的占位值（例如 `YOUR_TOKEN_HERE`），并在文档中说明由开发者自行配置。

### NFR-SCAFFOLD-SEC-003 依赖安全

- 模板 `package.json` 中的依赖必须：
  - 来自可信源（官方或经过审查的第三方库）；
  - 使用清晰的版本约束（避免无上限的 `*`）；
  - 定期通过 `npm audit` 等方式进行安全检查（检查流程在开发计划与技术文档中说明）。

---

## 4. 可维护性需求（NFR-SCAFFOLD-MNT）

### NFR-SCAFFOLD-MNT-001 文档与实现同步

- 任何对模板目录结构、文件命名、参数化规则的变更：
  - 必须同步更新本仓需求文档与技术文档中对应章节；
  - 如变更影响生态通用约定（例如“应用插件工程推荐目录结构”），必须先更新生态共用文档后再调整本仓实现。

### NFR-SCAFFOLD-MNT-002 代码与配置分层

- 脚手架实现自身（未来的 Node/CLI 代码）应与模板内容隔离：
  - 模板作为“静态资源”或“模板文件”管理，不与脚手架运行时代码混杂；
  - 模板参数与渲染逻辑集中在少量适配层中，方便后续维护与演进。

### NFR-SCAFFOLD-MNT-003 一致性

- 与其他脚手架仓库（basecard/layout/module/theme）在以下方面保持一致：
  - 文档结构与命名；
  - 模板参数命名与默认值约定；
  - 质量门禁与发布流程。

---

## 5. 可观测性与诊断需求（NFR-SCAFFOLD-OBS）

> 脚手架本身多为本地开发工具，可观测性需求主要体现在错误信息与日志可读性上。

### NFR-SCAFFOLD-OBS-001 命令行输出

- 创建过程中的日志输出需满足：
  - 关键信息（创建路径、模板名称、参数摘要）清晰可读；
  - 错误信息包含简要原因与参考解决方案；
  - 不向终端打印大量无意义调试细节（如长栈信息），除非通过显式 `--debug` 开关启用。

### NFR-SCAFFOLD-OBS-002 诊断信息

- 当创建失败时，脚手架应输出：
  - 失败步骤（例如“写入 manifest.yaml 时失败”）；
  - 涉及的路径与文件；
  - 建议用户在必要时将该信息附在问题工单中，便于复现与排查。

---

## 6. 测试与发布门禁需求（NFR-SCAFFOLD-TEST）

### NFR-SCAFFOLD-TEST-001 测试层级

- 本仓在引入实现代码后，必须覆盖至少以下测试层级：
  - 模板渲染单元测试：验证参数化逻辑与生成结果；
  - 集成测试：在临时目录中实际执行一次或多次 `create` 流程并比对输出；
  - 生成工程 E2E 测试：在 CI 中创建工程、安装依赖并运行基础构建与测试命令。

### NFR-SCAFFOLD-TEST-002 覆盖要求

- 针对脚手架实现代码：
  - 核心渲染逻辑与文件系统操作应具备足够的单元测试覆盖；
  - 与生态其他仓库的集成逻辑（如调用 `chips dev` 内部 API）需通过契约测试或模拟替身进行验证。

### NFR-SCAFFOLD-TEST-003 发布阻断条件

以下任一项未通过时，不得发布新的脚手架版本或合并到 `develop`：

- 模板生成工程的 `chips dev validate` 失败；
- 模板生成工程的 `build/test/lint` 任一命令失败；
- 模板生成工程在主流平台（例如 `darwin` 与 `win32`）上的最基本启动流程无法完成；
- 脚手架自身测试集（单元/集成/E2E）存在失败用例。

---

## 7. Git 与交付规范需求（NFR-SCAFFOLD-GIT）

### NFR-SCAFFOLD-GIT-001 分支策略

- 所有脚手架相关开发工作必须在任务分支上进行：
  - 任务分支以 `codex/` 为前缀命名（例如 `codex/app-scaffold-stage1-docs`）；
  - 按用户规范，从 `Develop` 分支或其同步分支上创建任务分支（在不破坏现有工作区前提下执行）。
- 任务完成并通过质量门禁后方可合并至 `Develop`，随后归档任务分支与 worktree。

### NFR-SCAFFOLD-GIT-002 提交流程

- 每次提交应保持语义单一、便于回溯：
  - 文档类提交与代码实现类提交尽量拆分；
  - 提交前需本地通过约定的质量门禁（至少 `lint/test/build` 通过）。

### NFR-SCAFFOLD-GIT-003 变更边界

- 不得修改与当前任务无关的仓库目录与文件；
- 禁止使用破坏性命令（如 `git reset --hard`）回滚用户已有变更；
- 如需移动或废弃文件，必须遵循“移入归档目录而非直接删除”的仓库约定。

以上非功能与质量需求用于约束应用插件脚手架项目的长期稳定性与可维护性，后续开发与版本发布均需以本文件为基线进行验收与回归检查。
