# 阶段二开发计划：认证系统

**阶段编号**：Phase 2  
**前置阶段**：Phase 1（基础架构完成）  
**预计工时**：2-3 天  
**目标**：用户注册、登录、JWT 认证完整闭环，受保护接口鉴权正确。

## 交付标准

- 用户可以通过 `POST /api/v1/auth/register` 注册账号
- 用户可以通过 `POST /api/v1/auth/login` 登录并获取 token
- 受保护接口在携带有效 token 时正常响应，否则返回 401
- Refresh token 可正常刷新 access token
- Logout 后 refresh token 失效
- 前端登录 / 注册页面可正常使用
- 单元测试通过：UserService、AuthService 业务逻辑
- 集成测试通过：所有认证路由

## 任务清单

### 1. 服务端：认证核心逻辑

#### 1.1 依赖安装

- [ ] 安装 `@fastify/jwt`、`bcryptjs`、`@types/bcryptjs`
- [ ] 安装 `@fastify/cookie`（用于 refresh token Cookie）

#### 1.2 UserService（`src/services/user.service.ts`）

- [ ] `createUser(username, password)` → 校验用户名格式、唯一性，bcrypt 哈希密码，写库
- [ ] `findByUsername(username)` → 大小写不敏感查找（查 `username_lower`）
- [ ] `findById(id)` → 通过 UUID 查找用户
- [ ] `validatePassword(user, plainPassword)` → bcrypt.compare
- [ ] `updateUser(id, patch)` → 更新可编辑字段（display_name、bio）
- [ ] `changePassword(id, oldPassword, newPassword)` → 验证旧密码后更新哈希

#### 1.3 AuthService（`src/services/auth.service.ts`）

- [ ] `generateTokenPair(userId)` → 签发 access token（JWT, 15min）+ refresh token（JWT, 30d）
- [ ] `verifyAccessToken(token)` → 验证 JWT，检查 Redis 黑名单（jti）
- [ ] `refreshTokens(refreshToken)` → 验证 refresh token，检查黑名单，签发新 token 对
- [ ] `revokeRefreshToken(jti)` → 将 jti 放入 Redis 黑名单，TTL = refresh token 剩余有效期
- [ ] 登录失败计数器（Redis key: `login_fail:{username}`，TTL 15分钟，超过 5 次封锁）

#### 1.4 JWT Fastify 插件（`src/plugins/auth.plugin.ts`）

- [ ] 注册 `@fastify/jwt`，配置 secret 和算法（HS256）
- [ ] 注册 `@fastify/cookie`
- [ ] 导出 `authenticate` preHandlerHook：验证 Bearer token，将 user payload 挂到 `request.user`
- [ ] 导出 `optionalAuthenticate` preHandlerHook：token 无效时不报错，`request.user` 为 null

#### 1.5 认证路由（`src/routes/auth.ts`）

- [ ] `POST /api/v1/auth/register`
  - 请求体校验（Zod schema）：`{ username, password }`
  - 调用 UserService.createUser
  - 签发 token 对，access token 在响应体返回，refresh token 通过 HttpOnly Cookie 返回
  - 返回 `{ data: { user, accessToken } }`
- [ ] `POST /api/v1/auth/login`
  - 请求体：`{ username, password }`
  - 检查封锁状态
  - 验证密码，失败则计数
  - 成功则重置计数，签发 token 对
  - 返回 `{ data: { user, accessToken } }`
- [ ] `POST /api/v1/auth/refresh`
  - 从 Cookie 读取 refresh token
  - 调用 AuthService.refreshTokens
  - 返回新 access token
- [ ] `POST /api/v1/auth/logout`
  - 从 Cookie 读取 refresh token
  - 吊销 refresh token（加入 Redis 黑名单）
  - 清除 Cookie
  - 返回 204

#### 1.6 Zod 请求体 Schema（`src/schemas/auth.schemas.ts`）

- [ ] `RegisterSchema`：username 正则（`/^[a-zA-Z0-9_-]{3,32}$/`）+ password 最少 8 位
- [ ] `LoginSchema`：username + password 非空

#### 1.7 错误码（`src/errors/codes.ts`）

- [ ] 定义常量：`AUTH_INVALID_CREDENTIALS`、`AUTH_USER_EXISTS`、`AUTH_ACCOUNT_LOCKED`、`AUTH_TOKEN_INVALID`、`AUTH_TOKEN_EXPIRED`

## 关键数据流示例

### 注册流程

```
前端 RegisterPage
  → POST /api/v1/auth/register { username, password }
  → 服务端：校验 → 检查重复 → bcrypt hash → 写 users 表 → 生成 token 对
  → 响应：{ data: { user, accessToken } } + Set-Cookie: refreshToken=...
  → 前端：存 accessToken 到内存，更新 AuthContext，跳转到个人空间
```

### token 自动刷新流程

```
应用加载
  → 检查 Cookie 中是否有 refreshToken（通过调用 /api/v1/auth/refresh）
  → 成功 → 将 accessToken 存入内存，恢复登录态
  → 失败（token 过期/吊销）→ 保持未登录态，不报错
```

## 测试用例要求

### 单元测试（`packages/server/tests/unit/`）

| 用例 | 要求 |
|------|------|
| 注册：合法用户名和密码 | 创建成功，密码哈希存储 |
| 注册：用户名重复 | 抛出 `AUTH_USER_EXISTS` 错误 |
| 注册：用户名格式不合法 | Zod 校验失败，返回 400 |
| 登录：正确凭据 | 返回 token 对 |
| 登录：密码错误 | 失败计数 +1 |
| 登录：封锁后尝试 | 返回 `AUTH_ACCOUNT_LOCKED` |
| 刷新：有效 refresh token | 返回新 access token |
| 刷新：已吊销 refresh token | 返回 401 |

## 验证方式

```bash
# 注册
curl -X POST http://localhost/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"Password123"}'

# 登录
curl -X POST http://localhost/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"Password123"}'

# 访问受保护接口（使用上面返回的 accessToken）
curl http://localhost/api/v1/users/me \
  -H "Authorization: Bearer <accessToken>"
```
