# 阶段三开发计划：房间与空间管理

**阶段编号**：Phase 3  
**前置阶段**：Phase 2（认证系统完成）  
**预计工时**：2-3 天  
**目标**：用户可在 UI 中创建 / 删除房间，个人空间首页可正常访问，公开房间可被任何人浏览。

---

## 交付标准

- 登录用户可创建、查看、删除自己的房间
- 个人空间 `/@{username}` 公开可访问，展示房间列表
- 房间页面 `/@{username}/rooms/{roomSlug}` 可访问（本阶段内容为空，仅展示房间信息）
- 私有房间对非拥有者不可见
- 前端个人空间页面和房间管理页面完整可用
- 集成测试通过：所有房间路由

---

## 任务清单

### 1. 服务端：房间业务逻辑

#### 1.1 RoomService（`src/services/room.service.ts`）

- [ ] `createRoom(userId, data)` → 生成 UUID，自动生成 slug（从 name 转换，同用户下唯一），写库
  - slug 生成规则：将 name 转为小写，空格替换为 `-`，移除非 URL 安全字符，对重复 slug 追加 `-2`、`-3` 等
- [ ] `listUserRooms(username, requesterId?)` → 查询用户所有房间，若 requesterId !== ownerId 则过滤 private
- [ ] `getRoomById(roomId, requesterId?)` → 校验权限（private 仅拥有者可见）
- [ ] `getRoomBySlug(userId, slug, requesterId?)` → 通过 slug 查询
- [ ] `updateRoom(roomId, userId, patch)` → 校验拥有权，更新 name / description / cover_url / visibility
- [ ] `deleteRoom(roomId, userId)` → 校验拥有权，删除（CASCADE 会删除所属 cards/boxes 的 room_id）

#### 1.2 SpaceService（`src/services/space.service.ts`）

- [ ] `getUserSpaceOverview(username, requesterId?)` → 组合查询，返回：
  - 用户基本信息（公开字段）
  - 房间列表（有权限的）
  - 根目录卡片列表（room_id IS NULL，有权限的）
  - 根目录箱子列表（room_id IS NULL，有权限的）

#### 1.3 Slug 唯一性工具（`src/utils/slug.ts`）

- [ ] `generateSlug(name)` → 基础 slug 生成
- [ ] `ensureUniqueSlug(baseSlug, userId, db)` → 检查重复，自动后缀数字

### 2. 服务端：路由

#### 2.1 房间路由（`src/routes/rooms.ts`）

- [ ] `POST /api/v1/rooms`（需认证）
  - 请求体 Zod schema：`{ name, description?, visibility?, cover_url? }`
  - 调用 RoomService.createRoom
  - 返回创建的 room 对象

- [ ] `GET /api/v1/users/:username/rooms`（公开）
  - 调用 RoomService.listUserRooms
  - 支持分页：`?page=1&pageSize=20`

- [ ] `GET /api/v1/rooms/:roomId`（公开）
  - 调用 RoomService.getRoomById
  - 私有房间对非拥有者返回 404（不暴露存在性）

- [ ] `PATCH /api/v1/rooms/:roomId`（需认证）
  - 请求体：可部分更新 `{ name?, description?, visibility?, cover_url? }`
  - 返回更新后的 room 对象

- [ ] `DELETE /api/v1/rooms/:roomId`（需认证）
  - 仅拥有者可删除
  - 返回 204

#### 2.2 空间路由（`src/routes/space.ts`）

- [ ] `GET /api/v1/users/:username/space`（公开）
  - 调用 SpaceService.getUserSpaceOverview
  - 返回用户信息 + 房间列表 + 根目录内容摘要

### 3. 房间封面图上传

- [ ] `POST /api/v1/rooms/:roomId/cover`（需认证）
  - 接收图片文件（JPEG / PNG / WebP，最大 5MB）
  - 校验文件类型（magic bytes）
  - 压缩并转换为 WebP（使用 `sharp` 库，需确认技术选型是否已包含）
  
> **注意**：`sharp` 当前未在技术选型中列出，上传封面图功能如需图片处理请先询问用户。
> 备选方案：接受原始图片，不做服务端压缩，直接上传到 MinIO（文件大小限制 5MB）。
> **当前阶段采用备选方案**，不引入 sharp。

- [ ] 上传到 MinIO（`chips-covers/rooms/{roomId}/cover.{ext}`），设置 public-read
- [ ] 更新 rooms.cover_url

## 测试用例要求

| 用例 | 要求 |
|------|------|
| 创建房间：登录用户 | 成功，返回 201 + room 对象 |
| 创建房间：未登录 | 返回 401 |
| 创建房间：名称超长 | Zod 校验失败，返回 400 |
| 查询房间：公开房间 | 任何人可访问 |
| 查询房间：私有房间（非拥有者） | 返回 404 |
| 删除房间：拥有者 | 成功，返回 204 |
| 删除房间：非拥有者 | 返回 403 |
| 个人空间：包含私有房间 | 自己可见，他人不可见 |
