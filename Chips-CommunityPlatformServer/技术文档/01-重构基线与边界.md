# 重构基线与边界

## 1. 当前代码基线

当前 `packages/web` 仍是多页面站点式结构，核心现状如下：

- `App.tsx` 暴露首页、个人空间页、房间页、工作台、上传中心等多路由；
- `GlobalNav.tsx` 提供全局导航、主题切换、语言切换、后台入口；
- `HomePage.tsx` 承担发现、搜索、最新卡片、最新箱子总站能力；
- `DashboardPage.tsx` 承担资料编辑、房间创建、内容管理；
- `CardUploadPage.tsx` 仍是“卡片 / 箱子二选一”的上传中心；
- `AppPreferencesContext.tsx` 与 `tokens.css` 仍是项目内自建主题 / 多语言运行时。

## 1.1 新边界补充

- 社区服务器与用户设备上的 Host 是**同根同源但完全独立**的两套底层系统；
- 社区服务器不会直接依赖 Host、Bridge、`window.chips` 或 Host 运行时；
- 社区服务器前端需要建设自己的 Web 主题、图标与多语言体系，并按 Web 服务场景独立优化；
- 两端允许共享设计理念、插件生态和部分代码来源，但不能混写成同一运行时链路。

## 2. 当前问题判断

- 页面范围明显超出最新产品目标；
- 主题、图标与多语言仍停留在旧版硬编码实现，没有升级到服务器独立 Web 体系；
- 继续在旧壳层上增量修改，会导致界面收敛失败。

## 3. 已登记前置工单

- `项目日志与笔记/工单075-社区平台前端旧版主题词典与多页面壳层未升级到服务器独立Web体系/任务说明.md`

本手册后续所有方案都默认：

1. 旧版本地 `AppPreferencesContext + tokens.css` 只算待替换基线；
2. 社区服务器需要形成自己的 Web 主题 / 图标 / 多语言模块；
3. 社区服务器前端不得直接依赖 Host 运行时链路；
4. 后续实施前需先按工单075冻结服务器独立 Web 体系的重构方式。

## 4. 本轮技术边界

### 4.1 在范围内

- `packages/web` 的页面、路由、组件拆分与状态模型；
- 对现有 REST API 的消费方式重组；
- 作品聚合、上传队列、进度反馈、页面动效；
- 与主题、多语言有关的消费层接线约束。

### 4.2 不在范围内

- `packages/server` 新增大规模业务能力；
- `packages/admin` 的后台产品设计；
- 房间公开浏览体系继续扩展；
- 通过临时兼容分支同时维护两套前端信息架构。

## 5. 约束原则

- 前端仍使用 React + TypeScript；
- 不新增无必要第三方库；
- 不保留旧版页面结构作为双轨运行；
- 所有文案、图标和视觉最终必须回到社区服务器自身的 Web 主题 / 图标 / 多语言体系。
