# 服务端启动与运行机制

## 1. 入口文件

- 进程入口：`packages/server/src/server.ts`
- 应用装配：`packages/server/src/app.ts`
- 环境配置：`packages/server/src/config/env.ts`

## 2. 启动顺序

`server.ts` 当前启动顺序固定为：

1. 运行数据库迁移 `runMigrations()`
2. 连接 Redis `connectRedis()`
3. 初始化对象存储桶 `initStorageBuckets()`
4. 初始化管理员账号 `initAdminAccount()`
5. 构建 Fastify 应用 `buildApp()`
6. 启动 HTTP 监听 `app.listen(...)`

这意味着当前服务端采用“进程启动即自检并补齐基础设施前置状态”的模式。

## 3. 环境变量加载

### 3.1 加载位置

`config/env.ts` 会优先读取：

- `deploy/.env`

### 3.2 校验方式

使用 `zod` 对关键环境变量做强约束校验。

若校验失败：

1. 打印结构化错误
2. 立即 `process.exit(1)`

### 3.3 关键配置域

当前校验的配置主要包括：

- 服务监听：`PORT`、`HOST`、`BASE_URL`
- 数据库：`DATABASE_URL`
- Redis：`REDIS_URL`
- S3 / MinIO：`S3_*`
- JWT：`JWT_*`
- 上传大小：`MAX_*_SIZE_MB`
- Host 插件安装源：`HOST_*_PLUGIN_PATHS`
- 当前启用主题：`HOST_ACTIVE_THEME_ID`

## 4. Fastify 应用装配

### 4.1 进程级配置

`buildApp()` 当前使用：

- `trustProxy = true`
- 开发模式使用 `pino-pretty`
- 生产模式使用默认 JSON logger

### 4.2 核心插件

按顺序注册：

1. `@fastify/helmet`
   - 当前关闭内置 CSP
   - CSP 交由 Nginx 层治理
2. `@fastify/cors`
3. `@fastify/jwt`
4. `@fastify/cookie`
5. `@fastify/multipart`
6. `@fastify/swagger`
7. `@fastify/swagger-ui`

### 4.3 CORS 规则

- 开发模式：允许本地开发源
- 生产模式：仅允许 `BASE_URL` 同源或无 `origin` 的请求
- `credentials = true`

### 4.4 multipart 规则

上传体大小上限取：

- `max(MAX_CARD_SIZE_MB, MAX_BOX_SIZE_MB)`

## 5. 路由注册顺序

当前顺序为：

1. `health`
2. `auth`
3. `users`
4. `rooms`
5. `space`
6. `upload`
7. `cards`
8. `boxes`
9. `discover`
10. `admin`（带 `/admin` 前缀）

## 6. Host 生命周期

### 6.1 初始化

在 Fastify `onReady` 阶段：

- 动态导入 `host-integration.ts`
- 执行 `hostIntegration.init()`

### 6.2 停止

在 Fastify `onClose` 阶段：

- 执行 `hostIntegration.stop()`

这保证了 Host 工作区与 Fastify 进程生命周期一致。

## 7. 管理员引导账号

`initAdminAccount()` 当前规则：

1. 只有当 `ADMIN_USERNAME` 与 `ADMIN_PASSWORD` 同时配置时才执行
2. 若该用户名已存在，则跳过，不覆盖现有密码
3. 若不存在，则以 `role = admin` 创建管理员账号

## 8. 退出语义

### 8.1 信号退出

当前服务端对以下信号执行优雅关闭：

- `SIGTERM`
- `SIGINT`

这条路径会依次关闭：

1. Fastify 服务器
2. Redis 连接
3. PostgreSQL 连接池

### 8.2 致命异常退出

当前还监听：

- `uncaughtException`
- `unhandledRejection`

但这两类分支目前只会：

1. 记录错误日志
2. 直接 `process.exit(1)`

也就是说，当前代码并不会在这两类致命异常分支里先执行 `shutdown(...)`。

## 9. 失败退出语义

### 9.1 启动前置失败

若以下任一步骤失败，服务端不会继续监听端口：

- 迁移失败
- Redis 连接失败
- bucket 初始化失败
- 环境变量校验失败

### 9.2 监听失败

若 `app.listen(...)` 失败：

1. 记录错误
2. 执行 shutdown
3. 退出进程

## 10. 当前运行形态说明

当前 `packages/server` 是唯一 API 进程，不拆成单独的：

- 鉴权服务
- 内容服务
- 转换服务
- 搜索服务

所有这些能力目前都在一个 Fastify 进程中完成；Host 以嵌入式运行时方式提供正式插件能力。
