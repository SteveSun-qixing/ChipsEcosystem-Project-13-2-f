# 个人社区页技术方案

## 1. 数据来源

个人社区页不再优先依赖旧版 `/users/:username/space` 聚合接口，而是显式拆成三类请求：

1. `GET /api/v1/users/:username`
2. `GET /api/v1/users/:username/cards`
3. `GET /api/v1/users/:username/boxes`

若当前访问者已登录，再额外通过当前登录态判断是否为本人。

## 2. 为什么不再以 `space` 接口为主

旧版 `space` 接口返回的是：

- 用户信息；
- 房间列表；
- 根目录卡片；
- 根目录箱子。

而 V2 目标需要的是：

- 用户信息；
- 全部公开作品；
- 不再以房间为公开浏览主结构。

因此作品主数据应直接来自用户级卡片 / 箱子列表接口，而不是旧版空间聚合口径。

## 3. 视图模型

页面内部统一转换为：

```ts
type CommunityWorkItem = {
  id: string;
  type: 'card' | 'box';
  title: string;
  coverUrl: string | null;
  href: string;
  createdAt: string;
};
```

卡片与箱子都先进入这一统一模型，再按时间倒序混排。

## 4. 作品合并策略

- 卡片列表映射为 `type: 'card'`；
- 箱子列表映射为 `type: 'box'`；
- 统一按 `createdAt desc` 排序；
- 页面只关心展示顺序，不再按内容类别拆段。
- 个人页卡片接口只消费 `public + ready` 的可展示卡片，不把处理中或失败卡片混入公开页面。

## 5. 本人判断

本人状态推荐使用以下规则：

1. 若未登录，则必然不是本人；
2. 若已登录且 `auth.user.username === route.username`，则视为本人；
3. 本人状态影响作品区头部操作入口显隐，不改变作品公开列表定义。

## 6. 本人资料编辑入口

- 头图区改为透明承载层，不再复用 `panel` 底板；
- 仅本人访问自己的页面时，头图区右上角显示资料设置按钮；
- 头图区必须保持单一主视觉区域：头像与资料文案构成 `profile-hero__main`，设置按钮放在资料内容区顶部的右上角工具位，不能再单独占一行或脱离资料区；
- 头图区不再追加额外说明性 eyebrow 文案，优先把可见空间留给名字、用户名、简介和编辑动作本身；
- 设置按钮打开 `ProfileSettingsDialog` 浮层卡片；
- 弹层表单负责编辑 `displayName`、`bio`，并上传 `avatar`；
- 头像上传走 `POST /api/v1/users/me/avatar`；
- 名字与简介更新走 `PATCH /api/v1/users/me`；
- 任一资料更新成功后，前端调用 `refreshUser()` 同步登录态，并把页面头图区本地状态同步刷新；
- 设置表单需要处理图片类型、文件大小、空名字校验以及关闭弹层时的滚动锁定。

## 7. 本人管理态

- 个人页右上角保留工作区 `+` 入口；
- 同时新增“管理卡片”按钮，仅本人可见；
- 作品区头部只保留主标题与操作按钮，不再渲染重复解释性副文案；
- 进入管理态后，公开卡片支持批量勾选；
- 箱子仍保持只读展示，不纳入本轮批量删除；
- 删除动作直接复用 `DELETE /api/v1/cards/:cardId`，前端以 `Promise.allSettled` 编排批量请求；
- 成功删除后前端立即从当前网格移除对应卡片，失败项保留勾选并显示错误提示。

## 8. 头图区与背景氛围

- 页面根节点内增加固定定位的背景氛围层；
- 背景氛围层由三部分组成：彩色球状光晕层、大范围流动色幕层、其上的整页磨砂层；
- 七彩光晕采用配置化节点数组渲染，每个节点独立声明尺寸、位置、颜色、透明度、漂浮位移、时长与延迟，保证氛围覆盖整个视口，而不是只集中在头像附近；
- 单个光晕节点至少包含多段漂浮路径、透明度呼吸与内部高光流动三类动画，避免出现“整体在慢慢挪动，但看起来不流动”的静止感；
- 大范围流动色幕采用独立节点渲染，并与光球层错峰漂移，负责把局部光球串联成完整背景节奏，避免页面只剩零散小球；
- 磨砂层负责把主题色、模糊和细纹统一到个人页背景，不额外引入新的主题系统，同时不能把下层彩光压到看不见；
- 头图区自身只保留头像、文字和设置按钮，不再承担整块大卡片视觉；头像局部光效需要弱化，避免和彩色背景光晕叠加成局部过曝；
- 头图区背景动效必须兼容 `prefers-reduced-motion`，但作品卡片排布本身不再承担漂浮氛围。
- 站点交互主色统一收敛到蓝色 accent token；按钮、聚焦环、进度条、加载态和选中态必须全部消费 `--ccps-sys-accent` / `--ccps-sys-accent-strong`，不允许继续散落硬编码橙色实现。

## 9. 封面渲染策略

- 卡片优先展示独立封面 HTML 入口；
- 前端以 iframe 方式承载卡片封面 HTML，并禁用其在列表中的交互抢占；
- 列表视觉只保留一个封面体块与下方标题，不再附加内层窗框、日期行和类型条；
- 箱子优先展示箱子封面，缺失时回退到统一占位样式；
- 占位样式也必须纳入同一封面视觉体系。
- 作品网格中的封面排布禁止使用自主漂浮、随机位移或按索引扰动的动画；封面只允许保留必要的状态反馈，不再承担“活跃氛围”的职责。
- 作品封面采用三层结构：固定舞台 `cover-stage`、承载阴影与选中描边的 `cover-shell`、负责媒体裁切的 `cover-clip`。阴影不能再和 `overflow: hidden` 绑定在同一层。
- 不同长宽比的封面统一在带安全边距的固定舞台内居中缩放，阴影可见范围必须在舞台内部消化，不能依赖溢出绘制到卡片外部。
- 对带外扩阴影的封面卡片，禁止在条目根节点继续使用 `content-visibility` 或其他会引入 paint containment 的能力，否则会重新引入“看不见的裁切边界”。

## 10. 卡片封面与内容分离

卡片上传正式产物拆分为两条并列链路：

1. 内容链路：重写资源后的 `.card` 经 `card -> html` 转换，生成内容 HTML；
2. 封面链路：读取 `.card/cover.html`，若缺失则生成默认封面 HTML，并将其单独保存为封面入口；
3. 社区服务器直接发布转换模块输出的原生内容 HTML，不再额外改写导出舞台壳；
4. 两个 HTML 入口分别持有对方的关联超链接信息；
5. 数据表中的 `coverUrl` 对卡片场景表示封面 HTML 入口，而不是图片地址。

## 11. 状态流转

### 11.1 首屏

- 并行发起用户、卡片、箱子请求；
- 头图区与网格共用同一页级 loading 生命周期；
- 但视觉上允许骨架分区渲染。

### 11.2 错误

- 用户不存在：直接进入页面级 not found；
- 列表拉取失败：保留头图区错误容器或统一页级错误态，不回退到旧式 toast 堆叠。

### 11.3 空态

- 用户存在但无公开作品：渲染空网格态；
- 本人空态下仍保留“+”入口与资料设置按钮。
