# 数据库模型与持久化设计

## 1. 技术选型

- 数据库：PostgreSQL
- ORM：Drizzle ORM
- 连接驱动：`pg`
- 迁移方式：`drizzle-orm/node-postgres/migrator`

目录：

- schema：`packages/server/src/db/schema/`
- 迁移：`packages/server/src/db/migrations/`
- 连接：`packages/server/src/db/client.ts`

## 2. 连接模型

当前 PostgreSQL 连接池参数：

- `max = 20`
- `idleTimeoutMillis = 30000`
- `connectionTimeoutMillis = 5000`

开发模式下 Drizzle logger 默认开启。

## 4. `users` 表

职责：平台账号主体与个人空间主体。

关键字段：

- `id`
- `username`
- `usernameLower`
- `displayName`
- `passwordHash`
- `bio`
- `avatarUrl`
- `role`
- `isActive`
- `createdAt`
- `updatedAt`

关键约束：

1. `username` 唯一
2. `usernameLower` 唯一
3. 用户名大小写不敏感查找基于 `usernameLower`

## 5. `rooms` 表

职责：内容组织单元。

关键字段：

- `id`
- `userId`
- `name`
- `slug`
- `description`
- `coverUrl`
- `visibility`
- `createdAt`
- `updatedAt`

关系：

- `userId -> users.id`，`onDelete: cascade`

说明：

- 同一用户下 slug 需唯一，但当前唯一性由应用层 `ensureUniqueSlug(...)` 保证
- 数据库层当前没有单独建立 `(userId, slug)` 唯一索引

## 6. `cards` 表

职责：社区卡片主记录。

关键字段：

- `id`
- `cardFileId`
- `userId`
- `roomId`
- `title`
- `coverUrl`
- `htmlUrl`
- `cardMetadata`
- `cardStructure`
- `status`
- `errorMessage`
- `visibility`
- `fileSizeBytes`
- `createdAt`
- `updatedAt`

关系：

- `userId -> users.id`，`onDelete: cascade`
- `roomId -> rooms.id`，`onDelete: set null`

语义：

- `cardFileId` 来源于卡片文件内置 ID
- `roomId = null` 表示卡片位于用户空间根目录
- `status` 记录卡片处理流水线状态
- `cardMetadata` / `cardStructure` 保存结构化快照

## 7. `boxes` 表

职责：社区箱子主记录。

关键字段：

- `id`
- `boxFileId`
- `userId`
- `roomId`
- `title`
- `coverUrl`
- `metadata`
- `structure`
- `layoutPlugin`
- `visibility`
- `fileSizeBytes`
- `createdAt`
- `updatedAt`

关系：

- `userId -> users.id`，`onDelete: cascade`
- `roomId -> rooms.id`，`onDelete: set null`

语义：

- `boxFileId` 来源于 `.box` 文件内置 ID
- `metadata` / `structure` 保存箱子解析结果
- 当前不单独存箱子内卡片引用子表

## 8. 当前关系模型

整体关系为：

- 一个用户有多个房间
- 一个用户有多张卡片
- 一个用户有多个箱子
- 一个房间可挂多张卡片
- 一个房间可挂多个箱子
- 卡片 / 箱子都可以脱离房间挂在根目录

当前没有：

- 团队 / 组织
- 房间成员表
- 权限 ACL 表
- 评论、点赞、关注等关系表

## 9. 删除语义

### 9.1 用户删除

数据库层级联影响：

- 用户删除会级联删除名下房间、卡片、箱子记录

### 9.2 房间删除

房间删除不是单纯依赖数据库 cascade，而是先走业务层：

1. 列出房间内卡片
2. 列出房间内箱子
3. 调用各自 service 完成业务删除
4. 再删除房间记录

这样可以先清理对象存储资源。

### 9.3 卡片删除

卡片删除会：

1. 删除 `chips-card-resources/{userId}/{cardId}/`
2. 删除 `chips-card-html/{userId}/{cardId}/`
3. 删除数据库记录

### 9.4 箱子删除

当前箱子删除只删除数据库记录，不清理额外对象资源。

## 10. 启动时迁移与引导

### 10.1 迁移

`runMigrations()` 会在服务启动最开始执行。

### 10.2 管理员账号初始化

`initAdminAccount()` 会在迁移之后执行，可选地创建首个管理员账号。

## 11. 当前持久化边界

当前数据库只存“社区平台需要直接检索和治理的数据”，不长期保存：

- 原始上传 `.card` 包
- 原始上传 `.box` 包
- 卡片本地资源原件

这些内容的正式公开访问载体已经转移到对象存储与 CDN。
