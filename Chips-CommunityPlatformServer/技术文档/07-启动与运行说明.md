## 1. 运行架构总览

### 1.1 包职责

- `packages/server`
  - 社区服务器 API 主体；
  - 使用 Fastify 提供 REST API；
  - 使用 Drizzle ORM + PostgreSQL 持久化结构化数据；
  - 使用 Redis 处理认证失败计数、token 吊销名单等运行时状态；
  - 使用 MinIO / S3 兼容对象存储保存卡片资源、卡片 HTML、头像与封面；
  - 启动时会初始化 Host 集成层，安装卡片插件、模块插件和主题插件。
- `packages/web`
  - 社区公开前台；
  - 使用 Vite 开发服务器运行；
  - 开发态通过 `/api` 代理到 `packages/server`。
- `packages/admin`
  - 社区管理后台；
  - 也是独立 Vite 前端；
  - 开发态通过 `/api` 与 `/admin/api` 代理到 `packages/server`。
- `deploy`
  - 存放 Docker Compose、Dockerfile 和 Nginx 反向代理配置；
  - 用于生产式或接近生产式的容器化运行。

### 1.2 两种正式运行模式

当前项目存在两条正式启动路径：

1. 本地开发分拆模式
   - Docker 只负责 PostgreSQL、Redis、MinIO；
   - `packages/server` 在本地 Node 进程中运行；
   - `packages/web` / `packages/admin` 在本地 Vite 进程中运行。
2. Docker 全栈模式
   - PostgreSQL、Redis、MinIO、API、Web、Admin、Nginx 全部由 Compose 拉起；
   - 对外统一由 Nginx 暴露 `80` 端口。

本仓库当前最适合日常开发与联调的是第一种模式。

### 1.3 端口与访问入口

| 组件 | 默认端口 | 用途 | 说明 |
| --- | --- | --- | --- |
| API 服务 | `3000` | Fastify REST API | 根路径 `/` 不是前台首页 |
| 社区前台 | `5173` | `packages/web` Vite 开发服务 | 默认推荐访问这里看社区页面 |
| 管理后台 | `5174` | `packages/admin` Vite 开发服务 | 独立后台入口 |
| PostgreSQL | `5432` | 结构化数据存储 | Docker 基础设施 |
| Redis | `6379` | 认证与运行时缓存 | Docker 基础设施 |
| MinIO S3 API | `9000` | 对象存储 API | Docker 基础设施 |
| MinIO Console | `9001` | MinIO 管理台 | Docker 基础设施 |
| Nginx | `80` | 全栈统一入口 | 仅 Docker 全栈模式使用 |

重要说明：

- `http://127.0.0.1:3000/` 不是社区前台页面，看到 `404` 是正常现象；
- API 是否启动成功，应通过 `http://127.0.0.1:3000/api/v1/health` 判断；
- 社区前台页面默认应访问 `http://localhost:5173/` 或显式绑定后访问 `http://127.0.0.1:5173/`。

## 2. 环境变量与基础依赖

### 2.1 基础依赖

- Node.js：`>= 20`
- pnpm：`>= 9`
- Docker / Docker Compose：用于 PostgreSQL、Redis、MinIO 以及可选的全栈模式

### 2.2 环境变量文件位置

服务端环境变量不是从仓库根自动扫描，而是由 `packages/server/src/config/env.ts` 显式读取：

```ts
dotenv.config({ path: path.resolve(__dirname, '../../../../deploy/.env') });
```

因此开发态正式环境文件位置是：

```text
Chips-CommunityPlatformServer/deploy/.env
```

不是：

- 仓库根 `.env`
- `packages/server/.env`

### 2.3 必要环境变量

当前 `deploy/.env` 至少需要保证以下变量有效：

- `PORT`
  - API 监听端口，默认 `3000`
- `HOST`
  - API 监听地址，默认 `0.0.0.0`
- `BASE_URL`
  - 外部访问基地址；
  - Swagger 与部分公开 URL 逻辑会用到
- `DATABASE_URL`
  - PostgreSQL 连接串
- `REDIS_URL`
  - Redis 连接串
- `S3_ENDPOINT`
  - MinIO / S3 兼容对象存储入口
- `S3_ACCESS_KEY`
  - MinIO 访问账号
- `S3_SECRET_KEY`
  - MinIO 访问密码
- `S3_REGION`
  - S3 区域，默认 `us-east-1`
- `S3_FORCE_PATH_STYLE`
  - 本地 MinIO 一般为 `true`
- `JWT_SECRET`
  - 长度必须至少 32 个字符
- `JWT_ACCESS_EXPIRES_IN`
  - Access Token 过期时间
- `JWT_REFRESH_EXPIRES_IN`
  - Refresh Token 过期时间
- `MAX_CARD_SIZE_MB`
  - 卡片上传大小上限
- `MAX_BOX_SIZE_MB`
  - 箱子上传大小上限
- `MAX_AVATAR_SIZE_MB`
  - 头像上传大小上限
- `ENABLE_SWAGGER`
  - 是否启用 `/api/docs`
- `HOST_ACTIVE_THEME_ID`
  - Host 集成层的激活主题

### 2.4 可选环境变量

- `S3_PUBLIC_URL`
  - 若设置，公开资源 URL 将优先使用这个前缀；
  - 若不设置：
    - 开发态优先返回 MinIO 直连地址；
    - 非开发态回退到 `${BASE_URL}/cdn/...`
- `ADMIN_USERNAME`
  - 若设置，服务启动时会尝试初始化管理员账号
- `ADMIN_PASSWORD`
  - 配合 `ADMIN_USERNAME` 使用
- `HOST_CARD_PLUGIN_PATHS`
  - 覆盖默认卡片插件安装路径
- `HOST_THEME_PLUGIN_PATHS`
  - 覆盖默认主题插件安装路径
- `HOST_MODULE_PLUGIN_PATHS`
  - 覆盖默认模块插件安装路径

## 3. 服务端正式启动顺序

`packages/server/src/server.ts` 中的启动顺序是固定的：

1. 执行数据库迁移 `runMigrations()`
2. 连接 Redis `connectRedis()`
3. 初始化对象存储 bucket `initStorageBuckets()`
4. 初始化管理员账号 `initAdminAccount()`
5. 构建并启动 Fastify `buildApp()`

### 3.1 数据库迁移

- 迁移实现位于 `packages/server/src/db/migrate.ts`
- 使用 Drizzle migrator 执行 `packages/server/src/db/migrations`
- 若数据库未准备好，服务不会继续监听端口
- `@ccps/server build` 结束时会通过 `scripts/copy-server-runtime-assets.mjs` 把迁移 SQL 与 `meta/_journal.json` 正式复制到 `packages/server/dist/db/migrations`，保证 `node dist/server.js` 与正式 Docker 镜像都能直接运行迁移

### 3.2 Redis 连接

- 连接实现位于 `packages/server/src/cache/redis.ts`
- 采用 `lazyConnect: true`，由 `connectRedis()` 显式触发
- Redis 主要用于：
  - 登录失败计数
  - 认证锁定
  - Refresh Token 黑名单 / 吊销状态

### 3.3 对象存储初始化

- 初始化逻辑位于 `packages/server/src/storage/init.ts`
- 启动时会确保以下 bucket 存在并设置公开读策略：
  - `chips-card-resources`
  - `chips-card-html`
  - `chips-avatars`
  - `chips-covers`

### 3.4 Host 集成层

Fastify 启动完成后，`buildApp()` 还会在 `onReady` 生命周期中初始化 Host 集成层。

Host 集成层的关键行为：

- 工作区位置固定为：
  - `Chips-CommunityPlatformServer/packages/server/.chips-server-host`
- 每次启动前会先清空并重建这个目录；
- 会启动 `HostApplication`；
- 会从生态仓库中默认安装这些插件：
  - `Chips-BaseCardPlugin/richtext-BCP`
  - `Chips-BaseCardPlugin/image-BCP`
  - `Chips-ModulePlugin/Chips-CardtoHTML-Plugin`
  - `Chips-ModulePlugin/Chips-FileConversion-Plugin`
  - `ThemePack/Chips-default`
- 启动后会执行 `theme.apply`，应用 `HOST_ACTIVE_THEME_ID`

这意味着：

- 社区服务器开发态不是一个“纯后端 API”，它在内容处理链路上依赖本仓库里的 Host / 插件 / 主题资源；
- 如果这些本地插件目录不存在、未构建或路径配置错误，卡片处理相关能力会受影响。

## 4. 推荐本地开发启动方式

### 4.1 启动 Docker 基础设施

```bash

  -f Chips-CommunityPlatformServer/deploy/docker-compose.yml \
  --env-file Chips-CommunityPlatformServer/deploy/.env \
  up -d postgres redis minio
```

```bash
cd Chips-CommunityPlatformServer/deploy

```

```bash

  -f Chips-CommunityPlatformServer/deploy/docker-compose.yml \

  ps
```

```bash

```

这个命令实际会执行：

```bash
pnpm --filter @ccps/server dev
```

```bash

```

期望返回：


```text
http://127.0.0.1:3000/api/docs
```

## 5. Docker 全栈启动方式

如果需要接近生产式的整体验证，可以直接拉起完整 Compose：

```bash
docker compose \
  -f Chips-CommunityPlatformServer/deploy/docker-compose.yml \
  --env-file Chips-CommunityPlatformServer/deploy/.env \
  up -d --build
```

这会启动：

- PostgreSQL
- Redis
- MinIO
- `server`
- `web`
- `admin`
- `nginx`

### 5.1 Docker 全栈模式访问入口

- 社区前台：
  - `http://localhost/`
- 管理后台：
  - `http://localhost/admin/`
- API 健康检查：
  - `http://localhost/api/v1/health`
- API 文档：
  - `http://localhost/api/docs`
- CDN 公开资源：
  - `http://localhost/cdn/...`

### 5.2 全栈模式的反向代理关系

`deploy/nginx/nginx.conf` 当前路由关系为：

- `/api/` -> `server:3000`
- `/admin/api/` -> `server:3000`
- `/cdn/` -> `minio:9000`
- `/admin/` -> `admin:80`
- `/` -> `web:80`

### 5.3 全栈模式的额外注意事项

- `Dockerfile.server` 不是只打包 `packages/server`；
- 它会先在独立的 `ecosystem-artifacts` 阶段正式构建：
  - `Chips-Host`
  - `Chips-BaseCardPlugin/image-BCP`
  - `Chips-BaseCardPlugin/richtext-BCP`
  - `Chips-ModulePlugin/Chips-HtmltoImage-Plugin`
  - `Chips-ModulePlugin/Chips-CardtoHTML-Plugin`
  - `Chips-ModulePlugin/Chips-FileConversion-Plugin`
  - `ThemePack/Chips-default`
- 随后服务镜像再消费这些已构建工件，而不是依赖宿主机预先提交的 `dist` 目录；
- 服务端正式 `dist` 还会额外并入数据库迁移资源，再复制进生产镜像；

因此 Docker 全栈模式要求整个生态仓库相对目录结构完整可用。

## 8. 常见问题与排障

### 8.1 打开 `http://127.0.0.1:3000/` 什么都没有

这是正常现象。

原因：

- `3000` 是 API 服务；
- 根路径 `/` 没有挂社区前台页面；
- 正确的 API 验证地址是 `/api/v1/health`；
- 正确的社区前台地址是 `5173` 或 Docker 全栈模式下的 `80`。

### 8.2 `5173` 只能用 `localhost`，不能用 `127.0.0.1`

原因：

- Vite 默认开发服务器并不总是对 `127.0.0.1` 暴露；
- 若需要明确通过 `127.0.0.1` 访问，请使用：

```bash
pnpm --dir Chips-CommunityPlatformServer/packages/web exec vite --host 0.0.0.0 --port 5173
```

### 8.3 API 没有监听，终端停在迁移或初始化阶段

优先检查：

1. PostgreSQL 是否启动
2. Redis 是否启动
3. MinIO 是否启动
4. `deploy/.env` 是否正确
5. `JWT_SECRET` 是否至少 32 位

### 8.4 启动时 Host 集成层报插件路径问题

优先检查：

- 生态仓库目录是否完整；
- `Chips-Host`、`Chips-BaseCardPlugin`、`Chips-ModulePlugin`、`ThemePack` 是否存在；
- 若使用自定义插件路径，`HOST_*_PLUGIN_PATHS` 是否指向真实文件或目录。

### 8.5 对象资源 URL 不是 `/cdn/...`

这是当前代码定义的结果，不一定是故障。

现有逻辑：

- 若设置了 `S3_PUBLIC_URL`，优先用它；
- 若是开发态且未设置 `S3_PUBLIC_URL`，优先返回 MinIO 直连地址；
- 若是非开发态且未设置 `S3_PUBLIC_URL`，回退到 `${BASE_URL}/cdn/...`

### 8.6 启动后 `.chips-server-host` 内容被重建

这是当前 Host 集成层的正式行为，不是误删。

现有逻辑：

- 每次 `HostIntegrationService.init()` 先删除旧工作区；
- 再重建并重新安装插件、应用主题；
- 目的是保证社区服务器处理链路启动时拥有一致的 Host 运行环境。

## 9. 当前已验证的本地开发口径

截至 2026-04-06，本仓库当前已经人工验证过以下链路：

- Docker 基础设施可独立启动：
  - PostgreSQL
  - Redis
  - MinIO
- 本地 API 可正常启动并通过：
  - `http://127.0.0.1:3000/api/v1/health`
- 本地社区前台可正常启动并通过：
  - `http://localhost:5173/`
  - 显式 `--host 0.0.0.0` 后可通过 `http://127.0.0.1:5173/`
- 本地管理后台可正常监听：
  - `http://localhost:5174/admin/`

## 10. 推荐使用结论

日常开发请优先采用以下口径：

1. 先启动 Docker 基础设施；
2. 再启动本地 API；
3. 再启动本地社区前台；
4. 需要后台时额外启动本地 Admin；
5. 不要把 `3000` 当成社区首页；
6. 需要单端口统一入口时再切换到 Docker 全栈模式。
