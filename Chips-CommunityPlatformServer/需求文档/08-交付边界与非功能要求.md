# 交付边界与非功能要求

## 1. 第一版交付边界

### 已纳入

- 用户注册、登录、登出、刷新令牌
- 个人空间
- 房间创建、编辑、删除
- 卡片上传、异步处理、查看、删除
- 箱子上传、解析、查看、删除
- 首页发现页
- 搜索
- 管理后台基础治理
- Docker Compose 单机部署方案

### 未纳入

- 社交关系
- 评论
- 点赞
- 举报
- 订阅通知
- 协作编辑
- 内容审核流
- 版本历史

## 2. 非功能要求

### 2.1 安全

- 密码使用 bcrypt 哈希。
- JWT 认证。
- 上传类型校验。
- ZIP 解包时防路径穿越。
- 管理接口必须验证管理员角色。

### 2.2 稳定性

- 卡片上传处理失败要可追踪。
- 中间临时目录要及时清理。
- CDN 与数据库状态应尽量保持一致。

### 2.3 可部署性

- 提供 `.env.example`
- 提供 `docker-compose.yml`
- 提供 `server` / `web` / `admin` 三个镜像构建文件
- 提供前后台静态站点容器内 Nginx 配置
- 提供主站反向代理配置
- 支持单机一键启动

### 2.4 可维护性

- 代码模块边界清晰
- 配置集中在环境变量
- 文档与代码同步
- 关键服务链路有最少可执行验证

### 2.5 可扩展性

- Host 插件来源可动态配置
- 主题插件和转换插件可替换
- 对象存储采用 S3 兼容接口

## 3. 文档要求

- 产品文档描述用户可见行为、范围与规则。
- 技术文档描述实现结构、接口、数据、部署和排障。
- 生态公共契约单独沉淀到 `生态共用技术文档/`。
