# 组件库总体架构设计

## 1. 架构定位

薯片组件库位于生态 12 层架构中的 L10（Component Headless），职责是提供“结构 + 状态 + 可访问性 + 契约挂点”的无头组件能力。

- 向下依赖：L11 Theme Runtime 提供主题变量与契约
- 向上服务：L8 Declarative UI、L12 App/Page Domain
- 禁止事项：直接依赖 Host 服务细节；组件内硬编码视觉值；组件内硬编码用户文案

## 2. 技术栈冻结

- React：组件编排、组合 API、Hooks 能力
- Ark UI：交互原语、可访问性与状态机基础
- Style Dictionary：设计 token 构建与校验
- Tamagui (Core)：跨端 token/runtime 语义层，统一 token 消费模型

说明：Ark UI 是组件交互底座，Tamagui Core 是 token 语义与运行时桥接层，二者职责不重叠。

## 3. 分层模型

```text
L12 App/Page
   |
L8 Declarative UI
   |
L10 Chips Component Library
   |-- Ark UI Wrappers
   |-- Component Contracts
   |-- A11y & State Machine
   |-- Token Adapter (Tamagui Core)
   |
L11 Theme Runtime
   |
Style Dictionary Outputs + Theme Packages
```

## 4. 核心设计原则

1. 无头原则：组件只提供行为和结构，不提供视觉样式
2. 契约优先：所有组件必须声明 `data-scope/data-part/data-state`
3. 组合优先：使用复合组件与显式子组件，禁止 boolean props 爆炸
4. 稳定优先：先冻结接口契约，再进入功能扩展
5. 生态一致性：遵循 `namespace.action`、统一错误模型、统一 i18n 与 cpx 规则

## 5. 对外能力边界

组件库对外仅暴露：

- 组件 API（无样式）
- 组件契约定义（scope/part/state）
- token 类型定义与消费工具
- React Hooks（组件库内部状态/契约辅助，不越级访问 Host）

组件库不对外暴露：

- Host IPC 细节
- Theme Service 实现细节
- 业务域状态管理实现

## 6. 组件族分层

- Primitive：布局和语义原语（Box、Inline、Stack、Text）
- Base Interactive：Button/Input/Checkbox/Select/Dialog/Tabs/Menu 等
- Data & Form：DataGrid/VirtualList/FormField/DateTime/CommandPalette
- Workbench：SplitPane/DockPanel/Inspector/CardShell/ToolWindow
- System UX：ErrorBoundary/LoadingBoundary/Toast/Skeleton/EmptyState
- Card Runtime（生态独有）：CardCoverFrame、CompositeCardWindow（均为 iframe 高级组件）

## 7. 稳定性策略

- 状态变化全部投影到标准数据属性，主题和测试可观测
- 组件与主题采用弱耦合（契约键 + token 键）
- 核心组件优先保证行为正确和 a11y 完整，再扩展变体
- 引入新依赖必须先走评审（不允许超出冻结技术选型）

## 8. 主题权重统一口径

- 权重链（低 -> 高）：`global -> app -> box -> composite-card -> base-card -> component`
- 最小组件主题权重最高
- token 查找必须从高权重层开始，缺失后逐层向上回退
