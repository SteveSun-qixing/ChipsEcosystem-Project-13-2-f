# 需求与验收矩阵

## 1. 目标

确保组件库建设按“完整实现、无临时方案、可稳定交付”执行，并可直接作为后续开发验收依据。

## 2. 需求矩阵

| 需求编号 | 需求描述 | 验收标准 |
|---|---|---|
| CL-REQ-001 | 组件库采用 React + Ark UI + Style Dictionary + Tamagui (Core) | 技术依赖清单与架构文档一致，无额外未审批库 |
| CL-REQ-002 | 组件必须无视觉硬编码 | 抽检组件源码，视觉值全部来自 token 或主题变量 |
| CL-REQ-003 | 组件契约统一 | 每个组件存在契约定义，包含 scope/part/state |
| CL-REQ-004 | 主题可覆盖 | 主题切换后组件结构不变、视觉可切换 |
| CL-REQ-004A | 主题权重与查找规则一致 | 权重链为 `global -> app -> box -> composite-card -> base-card -> component`，解析按“高权重优先，缺失逐层向上”执行 |
| CL-REQ-004B | CardCoverFrame 规范落地 | iframe显示封面HTML，窗口下方显示卡片名称，错误与加载态完整 |
| CL-REQ-004C | CompositeCardWindow 规范落地 | 统一通过SDK显示接口传入卡片文件并返回复合iframe窗口 |
| CL-REQ-004D | 基础卡片插件职责闭环 | 每个基础卡片插件包含渲染模块、编辑模块、模板、配置、参数表与说明 |
| CL-REQ-005 | 可访问性达标 | 键盘路径、读屏语义、焦点管理测试通过 |
| CL-REQ-006 | 国际化合规 | 组件内无硬编码用户文案 |
| CL-REQ-007 | 测试门禁完整 | lint/typecheck/test/contract/a11y/perf 全通过 |
| CL-REQ-008 | Git 流程合规 | 使用 `codex/*` 分支，合并目标 `develop` |
| CL-REQ-009 | 缺陷闭环机制可执行 | 架构缺陷可按问题工单流程阻断与追踪 |
| CL-REQ-010 | Base-Interactive 首批组件闭环 | Button/Input/Checkbox 完成 API/契约/token/a11y/测试闭环 |
| CL-REQ-011 | Base-Interactive 次批组件闭环 | Radio/Switch/Select 完成 API/契约/token/a11y/测试闭环 |
| CL-REQ-012 | Base-Interactive 三批组件闭环 | Dialog/Popover/Tabs/Menu/Tooltip 完成 API/契约/token/a11y/测试闭环 |
| CL-REQ-013 | Data-Form 首批组件闭环 | FormField/FormGroup/VirtualList 完成 API/契约/token/a11y/测试闭环 |
| CL-REQ-014 | Data-Form 次批组件闭环 | DataGrid/Tree/DateTime/CommandPalette 完成 API/契约/token/a11y/测试闭环 |
| CL-REQ-015 | Workbench 首批组件闭环 | SplitPane/DockPanel/Inspector/PanelHeader/CardShell/ToolWindow 完成 API/契约/token/a11y/测试闭环 |
| CL-REQ-016 | 阶段七交互文档完整 | 每个阶段七组件提供状态图与键盘清单，文档可追溯 |
| CL-REQ-017 | 阶段七性能基线完整 | 存在可复现基线脚本与报告，结果满足性能门禁 |
| CL-REQ-018 | System-UX 组件闭环 | ErrorBoundary/LoadingBoundary/Notification/Toast/EmptyState/Skeleton 完成 API/契约/token/a11y/测试闭环 |
| CL-REQ-019 | Stage8 跨层联动闭环 | i18n/config 缺失回退与观测字段输出能力完整，回归用例通过 |
| CL-REQ-020 | Stage9 质量门禁脚本闭环 | `lint/typecheck/test/test:contracts/test:a11y/test:perf/build` 可聚合执行并阻断失败 |
| CL-REQ-021 | Stage9 故障注入与恢复闭环 | token/theme/config/i18n/event 异常均有测试覆盖并验证回退 |
| CL-REQ-022 | Stage9 结构化报告闭环 | 质量门禁与性能报告产物可追溯（latest + 历史时间戳） |
| CL-REQ-023 | Stage10 发布包闭环 | 每个发布版本包含 changelog/compatibility/rollback 与 release-bundle 校验记录 |
| CL-REQ-024 | Stage10 首发项目口径一致 | 阶段十不包含历史迁移策略，文档与脚本不依赖旧版映射工具 |
| CL-REQ-025 | Stage10 生态接入验收闭环 | 设置面板/编辑器/查看器场景存在统一验收模板与状态跟踪口径 |
| CL-REQ-026 | Stage11 iframe 安全与隔离闭环 | iframe sandbox 最小权限策略生效，消息通道按来源窗口与 origin 白名单过滤 |
| CL-REQ-027 | Stage11 SDK 适配契约闭环 | `resolveCoverFrame/resolveCompositeWindow` 适配器契约可校验，`mode` 严格限定为 `view | preview` |

## 3. 禁止项验收

- 禁止临时实现、占位实现
- 禁止 TODO 残留
- 禁止绕过测试发布
- 禁止删除文件（仅允许迁移归档）

## 4. 交付物清单

- 组件库内部技术文档（`技术文档/00-31`，不含归档文档）
- 生态共用组件库标准文档（`生态共用技术文档/组件库/00-07`）
- 对应索引与阅读路径
