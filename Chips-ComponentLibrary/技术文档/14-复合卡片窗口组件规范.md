# 复合卡片窗口组件规范

## 1. 组件定位

`CompositeCardWindow` 是薯片生态统一复合卡片展示组件，本质是 iframe 高级窗口组件：

- 输入：卡片文件（路径/ID）与显示参数
- 输出：可直接挂载的复合卡片 iframe 窗口
- 适用：编辑引擎、卡片查看器、未来所有需要显示复合卡片的应用

## 2. 核心原则

- 查看器/编辑器是显示接口的“套壳应用”
- 渲染处理、基础卡片分发、iframe 拼接由 SDK 显示接口完成
- 调用方只负责传入卡片文件和承载窗口

## 3. 接口草案

```ts
export interface CompositeCardWindowProps {
  cardFile: string;
  mode?: 'view' | 'preview';
  loading?: boolean;
  disabled?: boolean;
  allowedOrigins?: string[];
  onReady?: (payload: { cardId: string; nodeCount: number }) => void;
  onNodeError?: (payload: { nodeId: string; error: StandardError }) => void;
  onFatalError?: (error: StandardError) => void;
}
```

## 4. 结构契约

- `data-scope="composite-card-window"`
- `data-part="root|iframe|overlay|status"`
- `data-state="idle|resolving|rendering|ready|degraded|error|disabled"`

## 5. 渲染链路规范

1. SDK 接收 `cardFile`。
2. SDK 解析复合卡片结构并读取基础卡片列表。
3. SDK 根据基础卡片类型分发到对应基础卡片插件。
4. 各基础卡片插件前端渲染模块将内容填入模板，生成完整 HTML。
5. SDK 将基础卡片 iframe 按结构顺序拼接成复合卡片页面。
6. `CompositeCardWindow` 接收最终复合 iframe 并展示。

### iframe 事件协议（组件侧消费）

- 事件通道：`chips.composite`
- 事件类型：
  - `ready`
  - `node-error`
  - `fatal-error`
- 事件来源：仅接收当前 iframe `contentWindow` 的消息
- 可选来源限制：`allowedOrigins` 提供后，仅处理白名单 origin 事件
- 事件解析：无效载荷必须忽略，不得污染组件状态

## 6. 错误与降级

- 单个基础卡片失败：进入 `degraded`，展示局部错误占位，整体仍可显示。
- 关键链路失败：进入 `error`，触发 `onFatalError`。
- 错误对象统一 `StandardError`。

## 7. 性能与稳定性

- 支持增量加载与可见区优先策略。
- 复合窗口不直接依赖业务状态，避免重复渲染。
- 目标：首个可见内容快速可用，主题切换不重建结构。

## 8. 安全规范

- iframe 沙箱隔离必须启用。
- `sandbox` 输入会进行最小权限归一化并剔除危险权限。
- 跨 iframe 通信必须通过受控事件通道。
- 禁止插件窗口直接访问宿主敏感能力。

## 9. 测试验收

- 多基础卡片拼接顺序一致性测试
- 节点级失败降级测试
- 主题切换与状态切换回归测试
- iframe 安全策略测试
- 跨场景接入测试（查看器/编辑器）
