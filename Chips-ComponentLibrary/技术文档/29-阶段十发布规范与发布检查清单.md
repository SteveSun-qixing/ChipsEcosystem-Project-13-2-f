# 阶段十发布规范与发布检查清单

## 1. 目标

建立组件库稳定发布闭环，确保每次发布都具备可追溯变更、兼容说明、回滚说明与门禁证据。

## 2. 发布包结构

发布版本目录：`releases/<version>/`

必须包含：

- `release-bundle.json`
- `CHANGELOG.md`
- `ROLLBACK.md`

当前示例版本：`releases/0.1.0/`。

## 3. release-bundle 字段规范

`release-bundle.json` 必填字段：

- `version`：语义化版本号
- `releaseType`：`major/minor/patch`
- `changeLog`：变更说明文件路径
- `compatibility`：兼容性说明
- `rollback`：回滚说明文件路径
- `artifacts.qualityGateReport`
- `artifacts.perfReport`

## 4. 发布门禁命令

发布前执行：

1. `npm run verify`
2. `npm run quality:gate`
3. `npm run release:check`

`release:check` 会校验：

- 发布包字段完整性
- `package.json` 版本与发布包版本一致
- 质量门禁报告状态为 `passed`
- 性能报告状态为 `passed`

## 5. 发布检查清单

- [ ] 版本号与发布类型已确认
- [ ] 变更说明包含新增、修复、风险说明
- [ ] 兼容说明已覆盖主题作用域链与接入边界
- [ ] 回滚方案可执行且已记录工单路径要求
- [ ] `verify` 与 `quality:gate` 均通过
- [ ] `release:check` 通过并输出结构化报告

## 6. 报告产物

- 报告目录：`reports/release/`
- 模板：`reports/release/release-readiness-template.json`
- 最新报告：`reports/release/release-readiness-latest.json`

## 7. 失败处理

任一步骤失败必须阻断发布，并按以下流程执行：

1. 在当前分支修复问题并补充测试。
2. 如问题涉及生态外部依赖，在 `项目日志与笔记/问题工单.md` 提交工单。
3. 复跑全量门禁通过后再进入发布流程。
