# 应用架构与模块边界设计

## 1. 总体目标

生态设置面板采用“应用壳层 + 功能模块 + 服务适配层 + 共享基础层”四层结构，确保页面持续扩展时依旧保持清晰边界。

## 2. 目录规划

当前实现已将 `src/` 组织为如下结构：

```text
src/
├─ app/                 # 应用壳层、Provider、菜单注册、全局布局
├─ features/
│  ├─ themes/
│  ├─ languages/
│  ├─ app-plugins/
│  └─ component-gallery/
├─ shared/
│  ├─ ui/
│  ├─ hooks/
│  ├─ i18n/
│  └─ runtime/          # Host/SDK 适配器、错误归一、事件源
└─ test-support/
```

## 3. 职责划分

### 3.1 应用壳层

- 负责窗口级布局：左侧菜单、右侧内容区。
- 窗口原生外壳由 `manifest.yaml -> ui.window.chrome` 声明，当前正式方案为保留 Electron 原生标题栏，不在前端重复自绘标题区。
- 负责主题与语言运行时接入。
- 负责菜单注册和页面装配，不承载具体业务逻辑。

### 3.2 功能模块

每个一级菜单是一个独立功能模块。当前实现采用“页面组件 + 专属治理 hook”组合：

- `ThemePage.tsx` / `useThemeGovernance.ts`
- `LanguagePage.tsx` / `useLanguageGovernance.ts`
- `AppPluginsPage.tsx` / `useAppPluginGovernance.ts`
- `ComponentGalleryPage.tsx` / `registry.tsx`

这样做的原因是首版功能规模仍可控，但读取、变更、事件刷新逻辑已经与页面结构分离，后续如果单个模块复杂度继续上升，再向 `model/` 或更细粒度的 `components/` 细分。

### 3.3 服务适配层

服务适配层位于 `src/shared/runtime/settings-runtime-service.ts`，只负责把 Host / SDK 的原始响应映射成页面可消费的模型，例如：

- `ThemeInstallRecord -> ThemeListItemViewModel`
- `PluginRecord -> AppPluginListItemViewModel`
- `locale string -> LanguageListItemViewModel`

页面层不得直接拼接 Host 返回结构，避免契约扩散到全工程。

### 3.4 共享基础层

- 通用空状态、错误态、加载态组件。
- 拖拽安装区域、操作确认对话框、列表骨架等可复用 UI。
- 公共 hooks，例如事件订阅、异步状态管理、Host 错误归一化。
- 反馈队列与通知栈统一由 `useFeedbackQueue` + `NotificationStack` 管理，避免每个页面自行拼装消息状态。

## 4. 技术选型约束

- 状态管理优先使用 React 原生能力（Context、Reducer、自定义 hooks），不额外引入状态管理库。
- 数据请求层优先使用 `chips-sdk` 客户端与项目内适配器，不引入新的网络/查询框架。
- 组件预览页直接使用薯片组件库导出组件，不复制组件实现代码。

## 5. 扩展策略

菜单扩展必须通过注册表完成。新增设置页时只允许：

1. 新增功能模块目录；
2. 在菜单注册表中声明 `id`、`titleKey`、`order`、`component`、`capabilityTags`；
3. 在测试中补充对应页面注册和最小渲染覆盖。

不允许修改多个页面组件中的 if/else 分支来“追加一个新菜单”。

## 6. 源码基线约束

- 正式源码只保留 TypeScript / TSX 文件。
- 脚手架或转译过程中遗留的同名 `.js` 副本必须移出主源码目录，统一归档到 `归档/`。
- 测试、构建、联调全部以 TypeScript 源码为唯一基线。
