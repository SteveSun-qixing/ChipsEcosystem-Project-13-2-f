# 信息架构与交互流程设计

## 1. 页面壳层

采用经典设置面板布局：

- 窗口外壳采用 Host 正式窗口契约，在 `manifest.yaml` 中声明 `ui.window.chrome.frame: true` 与 `titleBarStyle: default`，由 Electron 原生标题栏承接窗口标题与系统控制按钮。
- 左侧：固定宽度菜单栏，包含应用标题、一级菜单、预留扩展入口。
- 右侧：滚动内容区，顶部显示页面标题与上下文操作按钮，下方显示具体内容面板。
- 左侧菜单栏只保留导航职责，不再附加当前主题 / 当前语言运行时信息小窗。
- 首版页面头部统一只保留标题与必要操作按钮，不额外显示说明性副标题或搜索框。

菜单栏需要支持后续继续添加分类，不允许把首版四个菜单写死成不可扩展结构。

## 2. 主题管理流程

1. 页面进入时读取已安装主题记录与当前主题。
2. 用户点击“安装主题”或拖拽主题包，进入文件安装流程。
3. 安装成功后刷新列表并同步当前主题状态。
4. 用户点击“切换”后执行主题应用流程，成功后通过 `theme.changed` 触发页面刷新。
5. 用户点击“详情”时，通过正式弹窗查看主题完整信息。
6. 用户点击“卸载”时弹出确认，成功后刷新列表。

## 3. 多语言流程

1. 页面进入时读取当前语言和可用语言列表。
2. 用户可先点击“详情”查看语言条目完整信息。
3. 用户点击目标语言项后立即提交切换。
4. 收到成功响应和 `language.changed` 事件后刷新页面文案。

## 4. 应用插件流程

1. 页面进入时读取全部应用插件运行时记录。
2. 用户选择文件或拖拽安装插件。
3. 页面根据插件启用状态渲染“启用”或“停用”动作。
4. 页面同步读取每个应用插件的快捷方式状态，并显示“未创建 / 已创建”与目标位置。
5. 若快捷方式尚未创建，用户可点击“创建快捷方式”；若已存在，则可点击“移除快捷方式”或“显示位置”。
6. 若列表项是当前设置面板自身插件，则停用和卸载动作必须被正式阻断，并显示限制说明。
7. 正式阻断不能只停留在按钮禁用层；页面动作入口与 hook 动作层都必须拒绝执行自停用 / 自卸载请求，避免后续 UI 复用时绕过限制。
8. 每个插件项都提供“详情”动作，通过正式弹窗查看完整治理信息。
9. 其他插件卸载前弹出确认，处理完成后刷新列表。

快捷方式治理的正式交互约束：

- 页面只能调用 Host 正式接口，不允许前端直接创建 `.lnk` 或 `.app`；
- Windows 文案统一显示为“桌面快捷方式”，macOS 文案统一显示为“启动台入口”；
- “显示位置”只在 Host 返回了 `launcherPath` 时可用；
- 创建 / 删除成功后必须重新拉取当前插件的运行时记录与快捷方式状态，不允许直接前端改值。

## 5. 组件展示页布局

组件展示页采用 Bento 布局，遵循以下原则：

- 高优先级组件使用更大卡片面积，便于对比主题效果。
- 同类组件集中在同一视觉分组中，避免随机堆砌。
- 每个卡片展示组件名称、用途、关键状态样例。
- 页面应覆盖明确信息层级：基础交互、表单数据、高级数据、工作台系统体验。

当前实现约束：

- 页面按正式组件分组生成多个 `gallery-section` 区块，不再使用 Tab 切换隐藏其余分组。
- 每个组件卡片都包含组件名、scope、parts、用途摘要和真实预览内容。
- 卡片尺寸通过 `hero / wide / standard` 三种强调级别控制，形成稳定的 Bento 信息密度。
- 预览文案全部进入本地 i18n 资源，避免展示页成为未国际化的“演示孤岛”。

## 6. 响应式策略

- 桌面宽屏：左侧菜单固定，右侧内容自适应。
- 中等宽度：保持左右布局，但允许右侧卡片布局自动换列。
- 窄屏场景：菜单折叠为顶部切换器或抽屉；当前首版已采用顶部切换器，桌面侧边栏导航在窄屏下隐藏。
- 主题、多语言、应用插件三类治理页在桌面与窄屏下统一采用正式列表布局，窄屏时单条记录按字段标签折叠显示。
