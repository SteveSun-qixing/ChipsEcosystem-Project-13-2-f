# 运行时数据流与接口映射设计

## 1. 原则

生态设置面板是治理类应用，不直接消费底层文件结构，而是统一通过 Host 正式服务域获取运行时状态。所有接口都必须经过项目内服务适配层归一化后再供页面使用。

## 2. 页面与接口映射

| 页面 | 读取接口 | 写入接口 | 事件 |
| --- | --- | --- | --- |
| 主题管理 | `plugin.query({ type: 'theme' })`、`theme.list()`、`theme.getCurrent()` | `plugin.install()`、`theme.apply()`、`plugin.uninstall()` | `theme.changed`、`plugin.installed`、`plugin.uninstalled` |
| 多语言 | `i18n.getCurrent()`、`i18n.listLocales()` | `i18n.setCurrent()` | `language.changed` |
| 应用插件 | `plugin.query({ type: 'app' })`、`plugin.getShortcut(pluginId)` | `plugin.install()`、`plugin.enable()`、`plugin.disable()`、`plugin.uninstall()`、`plugin.createShortcut()`、`plugin.removeShortcut()` | `plugin.installed`、`plugin.enabled`、`plugin.disabled`、`plugin.uninstalled`、`plugin.shortcut.changed` |
| 组件展示 | 组件库导出清单、`theme.getCurrent()`、必要时 `theme.contract.get()` | 无 | `theme.changed` |

## 3. 主题治理口径

首版主题管理列表必须区分两类数据：

1. **已安装主题记录**：来源于 `plugin.query({ type: 'theme' })`，用于展示所有已安装主题和卸载动作。
2. **可切换主题集合**：来源于 `theme.list()`，只表示当前工作区“已启用”的主题。

页面需要在适配层把两类数据合并，生成统一的主题视图模型，明确：

- 是否已安装
- 是否已启用
- 是否为当前主题
- 是否允许切换
- 是否允许卸载

## 4. 文件安装流程

文件选择与拖拽安装必须遵循正式治理契约：

- 文件对话框调用 `platform.dialogOpenFile`；
- 安装动作统一调用 `plugin.install(manifestPath)`；
- 页面只负责解析用户输入来源，不负责自行解包或扫描插件包；
- 安装结果刷新时以 Host 返回的运行时记录为准，不以本地临时状态伪造成功结果。

当前实现落地方式：

- `SettingsRuntimeService.openPluginFileDialog()` 统一发起 `platform.dialogOpenFile`
- 页面层通过 i18n 传入对话框标题与过滤器名称
- 拖拽安装只负责把 `File` 交给 `platform.getPathForFile()` 解析，再复用相同安装入口

## 5. 事件同步

页面层必须通过统一事件订阅机制同步 Host 运行时变化：

- `theme.changed`：同步当前主题、刷新展示页与主题标签。
- `language.changed`：同步当前语言、触发文案重算。
- `plugin.*`：刷新应用插件页与主题管理页的安装状态。
- `plugin.shortcut.changed`：用于后续增强场景下刷新快捷方式状态；首版实现仍以动作完成后的主动刷新为主。

事件处理必须幂等，不能因为重复事件造成列表重复插入或状态错乱。

当前实现约束：

- 页面级刷新统一走 `useHostRefresh()`
- 主题页监听：`theme.changed`、`plugin.installed`、`plugin.uninstalled`
- 语言页监听：`language.changed`
- 应用插件页监听：`plugin.installed`、`plugin.enabled`、`plugin.disabled`、`plugin.uninstalled`
- `useHostRefresh()` 内部使用稳定事件名快照与最新回调引用，避免组件重渲染时重复拆装同一组 Host 订阅，同时保证事件到达时始终执行最新刷新逻辑
- 当前设置面板插件的“不可停用 / 不可卸载”限制同时存在于页面按钮层和 `useAppPluginGovernance()` 动作层，避免仅靠 UI 禁用造成结构性绕过

## 6. 应用快捷方式治理口径

应用插件页的快捷方式状态来自两层正式数据：

1. `plugin.query({ type: 'app' })`：提供应用插件基础运行时记录；
2. `plugin.getShortcut(pluginId)`：提供该应用插件在当前工作区下的系统快捷方式状态。

当前实现落地方式：

- `SettingsRuntimeService.listAppPlugins()` 先拉取全部应用插件记录，再逐个调用 `client.plugin.getShortcut(record.id)` 合并为页面视图模型；
- `createPluginShortcut(pluginId, replace)` 直接映射 `client.plugin.createShortcut(pluginId, { replace })`；
- `removePluginShortcut(pluginId)` 直接映射 `client.plugin.removeShortcut(pluginId)`；
- “显示位置”动作统一复用 `shell.showItemInFolder`，不自行拼装系统命令。

需要明确的技术细节：

- Host 返回的 `shortcut.exists` 才是正式真值，前端不得自行判断文件是否存在；
- `shortcut.location` 只承担展示语义，不用于推导真实路径；
- `shortcut.args` 是 Host 实际写入快捷方式的参数快照，可用于诊断“错误工作区”或“错误插件”问题；
- 当快捷方式已存在时，页面创建动作当前会以 `replace = true` 触发更新，从而确保图标、名称与目标路径可刷新到最新状态。

## 7. 错误归一化

适配层需要把 Host/SDK 错误统一映射为页面可消费结构，例如：

```ts
interface SettingsPanelError {
  code: string;
  message: string;
  retryable: boolean;
  details?: unknown;
}
```

页面层只依赖归一化错误，不直接读取散落的原始异常对象字段。

应用壳层中的 Host 启动期错误同样必须走统一错误结构，不允许直接把英文异常字符串泄漏到 UI。

当前实现约束：

- SDK Bridge 适配层需要先解码 Host IPC 返回的标准错误封装，再进入设置面板自身的 `normalizeSettingsError()`
- 页面层不得直接展示 Electron `Error invoking remote method ...` 这类宿主异常前缀

## 8. 权限基线

当前 manifest 必须覆盖以下权限：

- `theme.read`
- `theme.write`
- `i18n.read`
- `i18n.write`
- `plugin.read`
- `plugin.manage`
- `platform.read`

如未来新增设置项需要更多平台能力，必须同步更新 manifest 与本技术文档。
