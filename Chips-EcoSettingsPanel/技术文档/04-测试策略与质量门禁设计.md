# 测试策略与质量门禁设计

## 1. 测试分层

### 1.1 单元测试

覆盖纯函数与视图模型映射：

- 菜单注册表排序与过滤
- 主题 / 插件 / 语言 DTO -> ViewModel 映射
- 权限、状态、按钮可用性判定
- 错误归一化与消息映射

### 1.2 组件测试

覆盖页面壳层与功能组件：

- 左侧菜单切换
- 列表空状态、加载态、错误态
- 操作按钮的可用/禁用条件
- 列表详情弹窗入口是否存在
- 组件展示页分组渲染

### 1.3 集成测试

通过 mock Bridge / mock SDK 客户端验证：

- 安装主题 / 插件后页面状态刷新
- 主题切换后事件驱动更新
- 语言切换后的文案刷新
- 插件启用/停用/卸载的完整流程

### 1.4 端到端验证

在真实 Host 开发态至少验证：

1. 应用窗口可正常打开。
2. 主题安装、切换、卸载成功。
3. 语言切换后界面立即刷新。
4. 应用插件安装与停用成功。
5. 组件展示页能随主题变化刷新。

## 2. 质量门禁

- 单元与组件测试覆盖关键状态分支。
- 所有新增一级菜单必须至少补齐一条渲染测试和一条交互测试。
- Host 联调中发现的契约缺口必须先登记工单，再继续开发。
- 合并到 `Develop` 前必须执行 `lint/typecheck/test/build/validate` 全套命令。

## 3. 回归清单

每次功能迭代后至少回归以下场景：

- 当前主题标识是否正确
- 当前语言标识是否正确
- 列表排序和筛选是否稳定
- 当前设置面板插件是否被正确标识为不可停用/卸载
- 三类治理页是否保持正式列表布局且每条记录都有详情入口
- 三类治理页正常态是否不再出现错误边界红色外框
- 主题与插件安装失败时是否有可读反馈
- 组件展示页是否仍覆盖全部正式组件分组
- Host 事件订阅在页面重渲染后是否保持稳定且仍能刷新最新状态

## 4. 执行命令

- `npm run lint`
- `npm run typecheck`
- `npm test`
- `npm run build`
- `npm run validate`
- `npm run verify`
