# 生态设置面板需求总览

## 1. 文档目标

本文定义生态设置面板首版的产品定位、范围边界、关键用户和核心交付结果，为后续技术设计、开发计划和测试验收提供统一依据。

## 2. 产品定位

生态设置面板是运行在薯片 Host 内部的 `type: app` 应用插件，承担薯片生态运行时治理入口的职责。它不是安装器，不负责 Host 本体的系统级安装、卸载、升级、注册表和文件关联写入；它只处理 Host 运行态内可即时生效的配置和治理能力。应用插件快捷方式虽然会落到操作系统桌面/启动台，但必须通过 Host 正式接口完成，因此属于本面板可治理范围。

首版目标是形成一个可持续扩展的设置中枢，而不是一次性页面集合。所有菜单页必须遵循统一的导航注册、服务适配、状态治理和测试基线，确保后续能够继续添加新的设置分类与治理面板。

## 3. 用户与场景

| 角色 | 核心诉求 | 首版关注点 |
| --- | --- | --- |
| 生态终端用户 | 快速切换主题和语言，安装/卸载插件 | 页面清晰、操作即时生效、错误可理解 |
| 生态维护者 | 验证主题包、插件包与组件库在真实 Host 中的联动效果 | 列表信息完整、状态可观测、组件展示可信 |
| 未来功能开发者 | 在现有框架上增加新的设置页 | 菜单扩展机制稳定、模块边界清晰、无需推倒重来 |

## 4. 首版范围

### 4.1 必须交付

1. 左侧菜单 + 右侧内容面板的标准设置界面壳层。
2. 主题管理页。
3. 多语言页。
4. 应用插件页。
5. 组件展示页（Bento 布局）。
6. 完整需求文档、技术文档、开发计划与测试方案。

### 4.2 明确不在首版范围

1. 安装器能力（首次安装向导、注册表写入、Host 本体升级下载与安装）。
2. 系统级文件关联与 URL 协议注册。
3. 云同步、账户登录、远程市场下载。
4. 兼容旧版本数据或迁移旧架构的兼容层。

## 5. 业务目标

- 建立生态统一的运行时设置入口。
- 主题、多语言、应用插件治理全部走 Host 正式接口，不绕过 Bridge。
- 基于薯片组件库构建真实组件展示场景，作为主题验证基准页。
- 页面架构支持后续继续扩展菜单项与配置分类，而不破坏现有实现。

## 6. 成功标准

- 首版四个一级菜单可完整闭环。
- 所有操作具备明确成功/失败反馈，且即时反映到界面状态。
- 主题切换与语言切换能够响应正式事件源刷新界面，而非依赖页面重启。
- 文档、代码结构、测试矩阵、权限声明全部齐备，可直接进入正式开发执行。
