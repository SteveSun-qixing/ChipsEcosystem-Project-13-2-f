# 生态设置面板非功能与验收需求

## 1. 架构约束

- 必须使用 React 技术栈。
- UI 基础能力必须优先使用 `@chips/component-library`。
- Host 能力调用必须通过 `chips-sdk` 或 Bridge API 封装，不允许越层直连 Host 内部实现。
- 不引入未在生态技术选型中列出的第三方库；如确有必要，必须先单独确认。
- 不为规避接口问题而编写临时分支、兼容壳、伪实现或旁路逻辑。

## 2. 主题与多语言约束

- 所有视觉值必须走主题 token 或组件库契约，不允许页面内硬编码颜色、阴影、圆角作为正式结果。
- 所有用户可见文本必须进入 i18n 资源，不允许在业务实现中残留硬编码中文或英文。
- 首屏主题状态必须优先读取 Host preload 注入和 `theme.getCurrent()` 结果，不允许写死默认主题。
- 语言切换必须基于正式 `i18n.*` 接口与 `language.changed` 事件。

## 3. 性能与稳定性

- 菜单切换应在常规开发机上保持可感知顺滑，不允许因为整页重建造成明显卡顿。
- 长列表页必须预留列表虚拟化或分段渲染能力，但是否启用由实际数据规模决定，不能过度设计。
- 异步操作失败时必须能恢复到一致状态，不能出现列表状态与 Host 实际状态长期分叉。

## 4. 可测试性与质量门禁

合并到 `Develop` 前至少必须通过：

1. `npm run lint`
2. `npm run typecheck`
3. `npm test`
4. `npm run build`
5. `npm run validate`
6. 至少一轮 Host 开发态真实联调验证

测试范围必须覆盖：

- 菜单注册与路由切换。
- 主题列表与当前主题状态映射。
- 语言切换与事件刷新。
- 应用插件列表、启用/停用/卸载流程。
- 组件展示注册表与分组渲染。
- 关键错误态与权限不足提示。

## 5. 文档与问题治理

- 任何跨生态正式信息必须同步到生态共用技术文档。
- 若发现组件库缺少正式组件、Host/SDK 契约缺口、主题或 i18n 运行时无法满足实现要求，必须先记录问题工单并停止推进对应功能。
- 当前任务完成后，需在本工程中保留完整文档，作为后续开发与验收基线。
- 正式源码基线统一使用 TypeScript；如存在脚手架或转译遗留的同名 `.js` 副本，必须移出主源码目录并归档。
