# 现行正式链路盘点

## 1. 文档侧正式定义

当前正式文档已经明确：

- Host 负责 L1-L9 运行时与统一渲染；
- SDK 仅负责类型化封装与调用入口；
- 应用层统一通过 SDK 消费 Host 正式渲染链路；
- 不允许应用层自建另一套复合卡片渲染运行时。

关键文档：

- `生态共用技术文档/组件库/07-复合卡片窗口与SDK显示链路标准.md`
- `生态共用技术文档/协议与接口标准/02-Bridge-API规范.md`
- `生态设计原稿/09-卡片渲染机制.md`
- `生态设计原稿/14-编辑引擎设计.md`
- `生态设计原稿/19-SDK与协议.md`

## 2. SDK 当前正式交付

`Chips-SDK/src/api/card.ts` 当前提供：

- `client.card.render(...)`
- `client.card.compositeWindow.render(...)`
- `client.card.editorPanel.render(...)`
- 复合卡片事件：
  - `onReady`
  - `onResize`
  - `onInteraction`
  - `onNodeSelect`
  - `onNodeError`
  - `onFatalError`
- 编辑器事件：
  - `onReady`
  - `onChange`
  - `onError`

但其统一特征是：

- Host 返回 HTML 文档字符串；
- SDK 将 HTML 文档包装成 iframe；
- 应用层消费的是 iframe，而不是基础卡片节点级运行时对象。

## 3. Host 当前正式交付

`Chips-Host/packages/card-service/src/card-service.ts` 当前正式交付的是：

- `createChildFrameDocument()`：基础卡片 iframe 文档；
- `createCompositeDocument()`：复合卡片文档；
- `createEditorFrameDocument()`：编辑器 iframe 文档；
- `render()`：返回复合卡片文档；
- `renderEditor()`：返回编辑器文档。

当前 Host 没有正式交付以下能力：

- 可被编辑引擎直接持有和编排的基础卡片节点运行时；
- 节点树/场景树级别的编辑态输出；
- 应用壳层直接装配基础卡片节点的正式契约。

## 4. 编辑引擎当前接入点

`Chips-EditingEngine` 当前与正式链路的对接是：

- `src/components/CardWindow/CardWindow.tsx`
  - 使用 `client.card.compositeWindow.render(...)`
  - 消费 `onResize/onInteraction/onNodeSelect/onReady/onFatalError/onNodeError`
- `src/components/EditPanel/PluginHost.tsx`
  - 使用 `client.card.editorPanel.render(...)`
  - 消费 `onReady/onChange/onError`

这说明当前编辑引擎已经沿着正式链路完成接入，并不是还停留在 stub 或临时方案。

## 5. 与任务目标的差异

本任务目标希望：

- 去掉复合卡片整体 iframe；
- 让编辑引擎应用层直接持有基础卡片节点的渲染与交互编排权；
- 把大量编辑态交互和数据操作拉回应用壳层。

而当前正式链路交付的是：

- Host 主导的复合文档；
- SDK 封装的 iframe；
- 基础卡片通过子 iframe 失败隔离；
- 编辑器通过独立 iframe 交付。

这两者不是同一层级的“实现细节替换”，而是“架构边界变化”。
