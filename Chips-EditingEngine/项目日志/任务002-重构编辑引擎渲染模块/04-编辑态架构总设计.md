# 编辑态架构总设计

## 1. 总体原则

重构后的编辑引擎采用“双运行时、单数据源”架构：

- 显示链路：按基础卡片粒度调用各自渲染器，生成单卡 iframe，再由编辑引擎完成复合拼装；
- 编辑链路：改为编辑引擎本地编辑器运行时；
- 数据源：仍以 `CardService` 中的解包态卡片数据为唯一真实来源。

## 2. 目标架构分层

### A. Display Layer

职责：

- 负责基础卡片级 iframe 渲染；
- 负责复合卡片节点装配、排序、选中和高度汇总；
- 负责把多个单卡 iframe 组织成编辑引擎中的复合卡片窗口。

核心组件：

- `BasecardRendererRegistry`
- `BasecardFrameHost`
- `CompositeCardAssembler`
- `CardWindow`

### B. Editing Domain Layer

职责：

- 管理当前选中卡片、基础卡片和解包态内容；
- 提供本地编辑器可消费的源数据和提交入口；
- 驱动自动保存和变更事件。

核心对象：

- `CardService`
- `CardContext`
- `globalEventEmitter`

### C. Editor Runtime Layer

职责：

- 根据 `cardType` 选择本地编辑器；
- 管理会话、草稿、校验、提交、销毁；
- 将编辑器与编辑面板容器解耦。

建议模块：

- `EditorRegistry`
- `EditorSessionStore`
- `EditorRuntimeProvider`
- `EditorHost`

### D. Editor UI Layer

职责：

- 每种基础卡片类型的正式编辑器界面；
- 只关心该卡片类型的字段、交互和局部校验；
- 不直接负责持久化和预览编排。

## 3. 数据责任划分

### CardService

负责：

- 打开和维护 `.card/` 解包态数据；
- `updateBasicCard` 后写回内存模型；
- 自动保存到磁盘；
- 发出全局变更事件。

### EditorSession

负责：

- 保存当前基础卡片的本地草稿；
- 处理高频输入、防抖提交、校验状态；
- 将稳定配置提交回 `CardService`。

### CardWindow

负责：

- 消费编辑引擎内部基础卡片装配显示链路；
- 响应变更事件刷新预览；
- 管理窗口态、选中态、预览刷新节流。

## 4. 组件设计原则

为避免后续继续形成臃肿宿主，编辑器运行时采用组合模式：

- 不再给 `EditPanel` 叠加越来越多的布尔参数；
- 由 `EditorHost` 负责统一运行时编排；
- 每个编辑器只实现自己的内容区；
- 工具栏、状态栏、错误区用显式 slot/子组件拆开。

## 5. 性能原则

- 输入流和提交流分离，避免每次敲字都触发整窗重建；
- 单卡 iframe 重建与复合卡片刷新继续通过事件驱动和防抖策略执行；
- 对编辑面板内的短生命周期对象使用局部状态和 refs，减少全局重渲染；
- 不在渲染阶段做文件读写和重计算。
