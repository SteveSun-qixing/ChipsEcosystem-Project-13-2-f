# 数据模型与状态流设计

## 1. 单一真实数据源

本任务保持：

- `CardService` 中的 `CompositeCard.structure.basicCards[*].data` 为唯一真实卡片数据；
- 本地编辑器只编辑草稿；
- 提交后由 `CardService.updateBasicCard(...)` 回写正式数据并触发自动保存。

## 2. 状态流拆分

### 源状态

来源：

- `CardContext`

内容：

- 当前打开卡片；
- 当前选中基础卡片；
- 当前基础卡片正式配置。

### 草稿状态

来源：

- `EditorSessionStore`

内容：

- 当前会话草稿；
- dirty；
- valid；
- 字段级错误；
- 提交中的状态。

### 预览状态

来源：

- `CardWindow + CompositeCardAssembler`

内容：

- 当前基础卡片 iframe 集合；
- 当前复合卡片节点顺序与选中态；
- 预览加载态；
- 预览错误态；
- 复合卡片高度；
- 最近编辑活动时间。

## 3. 标准流转

### 选中基础卡片

1. 用户在复合卡片预览中选中节点；
2. `CardWindow` 通过正式事件回写 `selectedBaseCardId`；
3. `EditPanel` 解析当前基础卡片；
4. `EditorHost` 打开或复用该基础卡片编辑会话。

### 编辑输入

1. 用户在本地编辑器中输入；
2. 编辑器只更新 `draftConfig`；
3. 会话层执行局部校验；
4. 到达提交时机后统一提交。

### 提交与保存

1. `EditorSessionStore` 调用 `updateBasicCard(cardId, baseCardId, nextConfig)`；
2. `CardService` 更新内存模型并标记 dirty；
3. `CardService` 自动持久化到 `.card/content/*.yaml`；
4. `globalEventEmitter` 发出 `card:basic-card-updated`；
5. `CardWindow` 按既有节流策略刷新复合卡片预览。

## 4. 提交策略

建议按编辑器类型区分提交策略：

- 文本连续输入型：防抖提交；
- 开关、选择、点击型：即时提交；
- 复杂结构型：局部防抖 + 显式提交钩子。

但无论哪种类型，最终都统一经过会话层，而不是直接在编辑器组件里各写一套提交逻辑。

## 5. 错误处理

- 字段校验错误停留在会话层，不应立即污染正式数据；
- 提交失败时保留草稿，并向编辑面板显示错误状态；
- 预览刷新失败与编辑器输入失败分开处理，避免相互影响。
