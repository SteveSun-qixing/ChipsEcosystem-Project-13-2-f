# 数据模型与状态流设计

## 1. 单一真实数据源

当前任务保持以下正式约束：

- `CardService` 中的 `CompositeCard.structure.basicCards[*].data` 为唯一真实卡片数据；
- 本地编辑器只编辑草稿；
- 提交后统一由 `CardService.updateBasicCard(...)` 回写正式数据并触发自动保存。

## 2. 状态流拆分

### 源状态

来源：

- `CardService`
- `CardContext`

内容：

- 当前打开卡片；
- 当前选中卡片 ID；
- 当前选中基础卡片 ID；
- 当前基础卡片正式配置；
- 当前基础卡片列表顺序与布局参数。

### 草稿状态

来源：

- `EditorSessionStore`

内容：

- 当前会话草稿；
- dirty；
- valid；
- 字段级错误；
- 提交中的状态；
- 当前会话错误信息。

### 预览状态

来源：

- `CardWindow`
- `CompositeCardAssembler`
- `BasecardFrameHost`

内容：

- 当前基础卡片 iframe 集合；
- 当前复合卡片节点顺序与选中态；
- 单卡 `loading/ready/error` 状态；
- 复合卡片总高度；
- 当前预览错误信息；
- 当前交互代理策略。

## 3. 标准流转

### 选中基础卡片

1. `CardWindow` 把当前 `basicCards` 交给 `CompositeCardAssembler`；
2. `CompositeCardAssembler` 为每个节点创建 `BasecardFrameHost`；
3. 用户在单卡 iframe 内点击基础卡片；
4. `BasecardFrameHost` 通过本地 `onSelect` 回调把选择事件交给上层；
5. `CardWindow` 调用 `setActiveCard/setSelectedBaseCard`；
6. `EditPanel` 解析当前基础卡片；
7. `EditorHost` 打开或复用该基础卡片编辑会话。

### 编辑输入

1. 用户在本地编辑器中输入；
2. 插件通过 `onChange(nextConfig)` 输出新配置；
3. `EditorHost` 调用 `store.updateDraft(...)` 更新草稿；
4. 会话层执行局部校验；
5. 草稿保持在本地会话，不直接污染正式数据。

### 提交与保存

1. `EditorHost` 在草稿合法且变脏后，按 `commitDebounceMs` 触发提交；
2. `EditorSessionStore.commit(...)` 调用外层 `onConfigChange`；
3. `CardContext.updateBasicCard(...)` 转发到 `CardService.updateBasicCard(...)`；
4. `CardService` 归一化数据、执行 `notify()`、进入持久化队列；
5. `CardService` 发出 `card:basic-card-updated`；
6. 持久化完成后继续发出 `card:persisted`；
7. `CardWindow` 因数据变化刷新对应基础卡片 iframe 与整体高度。

## 4. 提交策略

当前正式策略：

- 文本连续输入型：由插件局部节流 + 会话层防抖提交；
- 开关、选择、点击型：仍统一经过会话层，不直接写正式数据；
- 卸载或切换节点时：`EditorHost` 会 flush 当前会话。

正式约束：

- 无论哪种编辑器类型，最终都统一经过会话层；
- 不允许在编辑器组件里各自写一套文件持久化逻辑；
- 不允许绕过描述符归一直接把脏数据写回 `CardService`。

## 5. 错误处理

- 字段校验错误停留在会话层，不应立即污染正式数据；
- 提交失败时保留草稿，并在编辑面板显示错误状态；
- 基础卡片预览失败由对应 `BasecardFrameHost` 局部隔离；
- 预览刷新失败与编辑器输入失败分开处理，避免相互影响。
