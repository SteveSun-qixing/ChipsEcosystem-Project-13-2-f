# 窗口协同与交互设计

## 1. CardWindow 角色不变

`CardWindow` 仍然是：

- 复合卡片拼装容器；
- 选中事件入口；
- 预览刷新调度器；
- 无限画布交互协同层。

本任务不把编辑表单塞回 `CardWindow` 内部。

## 2. EditPanel 新角色

`EditPanel` 改为：

- 纯编辑态容器；
- 当前选中基础卡片的本地编辑入口；
- 显示校验、保存、错误、空态信息；
- 不再承担 iframe 编辑器装载职责。

## 3. 预览与编辑协同

正式行为：

1. `CardWindow` 内部先按顺序渲染各基础卡片 iframe；
2. 用户在 `CardWindow` 中点选基础卡片；
3. 右侧 `EditPanel` 打开对应编辑器；
4. 编辑器修改草稿；
5. 数据提交后，`CardWindow` 通过已有事件和节流策略刷新对应基础卡片 iframe 和复合布局。

这样显示和编辑彻底分层：

- 拼装显示层只负责“渲染、拼装、选中”；
- 编辑面板只负责“改”。

## 4. 窗口状态规则

### 正常态

- `CardWindow` 使用 `preview` 或 `view` 模式展示复合卡片；
- `EditPanel` 展示当前编辑器。

### 收起态

- 卡片窗口继续保持当前已有的 16:9 收起规则；
- 不因本地编辑器运行时重构而改变。

### 封面态

- 继续沿用当前封面逻辑；
- 不把编辑器入口塞进封面窗口。

## 5. 无限画布协同

当前已有的 `onInteraction`、`panByInput`、`zoomByFactorAtPoint` 逻辑继续保留。

本任务只需要保证：

- 编辑面板的本地输入不会误触发画布级交互；
- 编辑器组件内部的高频输入不会导致卡片窗口被频繁重建；
- 节点选中与编辑器切换在窗口层保持一致。

## 6. 交互细节约束

- 不在本地编辑器内部再维护一套“基础卡片列表”作为正式入口；
- 正式入口仍然是点击复合卡片预览节点；
- 编辑器切换时应保留上一节点未 flush 的草稿；
- 当卡片切换或窗口关闭时，必须先完成会话清理。
