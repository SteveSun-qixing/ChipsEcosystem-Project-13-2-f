# 任务004：富文本基础卡片初始内容与内容驱动高度修复

## 1. 任务背景

本次任务聚焦 `Chips-EditingEngine` 里复合卡片装配链路中的两个直接体验问题：

- 新插入富文本基础卡片后，正文下方会被单卡运行时强行保留一段额外空白；
- 新建富文本基础卡片时，初始正文仍为空，不符合当前任务要求中的默认文本约束。

本次修复仍严格遵循编辑引擎正式链路：

- 预览继续走 `BasecardDescriptor -> BasecardFrameHost -> CompositeCardAssembler -> CardWindow`；
- 不新增任何针对富文本类型的主流程特判；
- 新建默认内容统一由描述符层注入，不把默认值散落到应用层或拖拽逻辑中。

## 2. 根因分析

### 2.1 额外空白来自单卡运行时的最小高度保护

排查 `BasecardFrameHost` 后确认，当前单卡宿主与 iframe 在样式层都保留了 `96px` 的最小高度保护。这样会导致：

- 富文本正文内容很短时，即使真实内容高度已经测量完成；
- 单卡 iframe 仍然不能缩到真实内容高度以下；
- `CompositeCardAssembler` 汇总到的仍是被最小高度撑大的结果；
- 最终在复合卡片里表现为正文下方固定多出一片空白。

### 2.2 新建默认内容需要由描述符统一下发

富文本创建链路已经统一收口到 `createInitialBasecardConfig -> BasecardDescriptor.createInitialConfig(...)`。因此：

- 只要在 `base.richtext` 描述符里定义正式初始正文；
- 新建卡片、拖拽插入、首卡创建等入口都会自然获得同一份默认内容；
- 不需要在 `App.tsx`、拖拽处理器或 `CardService` 中再写富文本特判。

## 3. 本次正式实现

### 3.1 恢复 ready 态的内容驱动高度

本次在 `BasecardFrameHost` 中调整了单卡宿主与 iframe 的最小高度策略：

- `loading/error` 态仍保留最小高度保护，避免骨架和错误态塌陷；
- 一旦进入 `ready` 态，立即把宿主与 iframe 的 `min-height` 覆盖为 `0px`；
- 让已经测量到的真实内容高度成为唯一生效高度。

这样短正文富文本不会再被额外撑高，同时不影响加载态和错误态的可感知反馈。

### 3.2 统一新建富文本的初始正文

本次把 `base.richtext` 描述符的 `createInitialConfig(...)` 改为输出：

- `body: "<p>123456789</p>"`
- `locale: "zh-CN"`

这样新建富文本基础卡片时，正式初始正文就是 `123456789`，并且所有创建入口保持一致。
