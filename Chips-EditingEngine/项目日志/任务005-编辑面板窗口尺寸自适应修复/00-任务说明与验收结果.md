# 任务005：编辑面板窗口尺寸自适应修复

## 1. 任务背景

本次任务聚焦 `Chips-EditingEngine` 编辑面板窗口的一个直接体验问题：

- 用户调整编辑面板窗口大小后，内部基础卡片编辑器仍保持旧的固定宽度；
- 导致右侧编辑宿主没有真正跟随工具窗口尺寸变化，不符合编辑态正式链路中“编辑面板只作为容器，编辑器由基础卡片插件直接挂载并适配容器”的要求。

本次修复严格限制在编辑引擎应用层，不改动 Host、SDK 或基础卡片插件协议，也不引入临时兼容分支。

## 2. 根因分析

排查 `EditPanel`、`PluginHost` 与工具窗口承载链路后，确认问题根因不是基础卡片编辑器自身不会伸缩，而是编辑面板根节点在应用层被写死了默认宽度：

- `EditPanel` 根节点在渲染时直接写入 `width: 320px`；
- 外层 `ToolWindow` 虽然允许用户拖拽改变窗口宽度，但内部 `EditPanel` 仍停留在默认 320 像素；
- 基础卡片编辑器宿主容器只能继承这个固定宽度，所以看起来像“编辑器页面不随窗口变化”。

因此这是编辑引擎本地壳层实现问题，不属于新的生态架构缺口，也不需要新建问题工单。

## 3. 本次正式实现

### 3.1 移除编辑面板固定宽度

本次将 `EditPanel` 从“自带固定宽度”改为“完全填充父窗口容器”：

- 不再在 `EditPanel` 根节点写入默认 `320px` 内联宽度；
- 保留编辑面板作为工具窗口内容区的正式角色，不在子组件内部重复声明一套窗口宽度；
- 让窗口尺寸由 `ToolWindow/BaseWindow` 统一管理，编辑面板只消费父容器可用空间。

### 3.2 补齐宽度继承链

为避免 flex 容器下的宽度收缩异常，本次同步补齐了编辑面板和插件宿主的尺寸链：

- `EditPanel` 根节点增加 `flex: 1`、`width: 100%`、`min-width: 0`；
- `edit-panel__content` 增加 `min-width: 0`；
- `PluginHost` 及其内部挂载容器增加 `width: 100%` 与 `min-width: 0`。

这样编辑宿主会稳定跟随工具窗口尺寸变化，基础卡片编辑器可以直接使用完整父容器宽度。

### 3.3 回归测试

本次补充了单测断言，确保编辑面板不再回到“默认 320px 内联宽度”的旧行为。
