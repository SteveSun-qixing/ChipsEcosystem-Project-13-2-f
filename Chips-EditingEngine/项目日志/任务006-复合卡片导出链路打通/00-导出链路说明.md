# 复合卡片导出链路说明

## 适用范围

- 本文只描述 `Chips-EditingEngine` 当前正式生效的“复合卡片设置 -> 导出为 .card”内部实现；
- 对外公共能力边界仍以生态设计原稿与生态共用技术文档中的 `card.pack`、卡片文件格式规范为准。

## 当前实现

编辑引擎的复合卡片设置弹窗中，“导出为 `.card`”现在走的是正式 Host 打包链路，而不是前端假进度。

当前导出顺序如下：

1. 用户在 `CardSettingsDialog` 中进入导出页；
2. 若设置弹窗内存在未提交且合法的封面草稿，先通过 `updateCardCover(...)` 回写到当前卡片会话；
3. `ExportPanel` 调用 SDK 正式接口 `client.platform.saveFile(...)` 让用户选择导出位置；
4. 用户确认位置后，编辑引擎先调用 `saveCard(cardId)`，把当前解包态卡片目录写回：
   - `.card/metadata.yaml`
   - `.card/structure.yaml`
   - `.card/cover.html`
   - `content/*.yaml`
   - 卡片根目录资源
5. 保存完成后，再调用 `client.card.pack(card.path, outputPath)`；
6. Host `card.pack` 通过 `CardPacker` 以 ZIP Store 模式输出正式 `.card` 文件；
7. 编辑引擎再通过正式 `file.stat` 路由校验目标文件确实存在，仅在真实落盘后才显示“导出完成：{path}”。

## 默认导出路径

- 默认导出文件名优先使用当前卡片名称，并清洗非法文件名字符；
- 若默认文件名与当前解包态卡片目录重名，则自动改为 `*-export.card`，避免把导出目标落回源目录路径。
- `client.platform.saveFile(...)` 中传入的 `defaultPath` 只用于给系统保存对话框提供初始建议位置，不能绕过用户确认。

## 当前边界

- 当前正式只接通 `.card` 导出；
- HTML / PDF / 图片导出尚未接到正式转换链路，因此在导出面板中保持不可用状态；
- 本实现不自行拼装 ZIP，也不绕过 Host 直接写归档文件。
