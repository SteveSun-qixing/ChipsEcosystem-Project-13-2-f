# 复合卡片导出链路说明

## 适用范围

- 本文描述 `Chips-EditingEngine` 当前正式生效的“复合卡片设置 -> 导出”内部实现；
- 对外公共能力边界仍以生态设计原稿与生态共用技术文档中的 `card.pack`、`converter.file.convert` 与文件转换能力契约为准。

## 当前实现

编辑引擎的复合卡片设置弹窗中，`.card`、HTML、PDF、图片四种导出现在都走正式链路，而不是前端假进度或占位按钮。

当前导出顺序如下：

1. 用户在 `CardSettingsDialog` 中进入导出页；
2. 若设置弹窗内存在未提交且合法的封面草稿，先通过 `updateCardCover(...)` 回写到当前卡片会话；
3. `ExportPanel` 调用 SDK 正式接口 `client.platform.saveFile(...)` 让用户选择导出位置；
4. 用户确认位置后，编辑引擎先调用 `saveCard(cardId)`，把当前解包态卡片目录写回：
   - `.card/metadata.yaml`
   - `.card/structure.yaml`
   - `.card/cover.html`
   - `content/*.yaml`
   - 卡片根目录资源
5. 之后按目标格式分流：
   - `.card`：调用 `client.card.pack(card.path, outputPath)`；
   - `html/pdf/image`：调用 `client.module.invoke({ capability: "converter.file.convert", method: "convert", ... })`；
6. 若文件转换模块以 job 模式返回，编辑引擎继续通过 `client.module.job.get(jobId)` 轮询正式任务状态，并把模块进度投影到导出面板；
7. `.card` 导出由 Host `card.pack` 通过 `CardPacker` 以 ZIP Store 模式输出正式 `.card` 文件；
8. HTML / PDF / 图片导出由 `Chips-FileConversion-Plugin` 编排模块继续分发到：
   - `Chips-CardtoHTML-Plugin`
   - `Chips-HtmltoPDF-Plugin`
   - `Chips-HtmltoImage-Plugin`
9. 编辑引擎最终统一通过正式 `file.stat` 路由校验目标文件确实存在，仅在真实落盘后才显示“导出完成：{path}”。

## 格式映射

- `.card`：输出正式卡片归档文件，默认扩展名 `.card`
- `HTML`：输出 HTML 导出压缩包，默认扩展名 `.zip`
- `PDF`：输出 PDF 文件，默认扩展名 `.pdf`
- `图片`：当前正式输出 PNG 文件，默认扩展名 `.png`

编辑引擎会在保存对话框返回路径后补齐缺失扩展名，并对文件转换链路统一传入 `overwrite: true`，与系统保存对话框的覆盖确认保持一致。

## 默认导出路径

- 默认导出文件名优先使用当前卡片名称，并清洗非法文件名字符；
- 若默认文件名与当前解包态卡片目录重名，则自动改为 `*-export.card`，避免把导出目标落回源目录路径。
- 其他格式按 `.zip / .pdf / .png` 生成默认导出建议路径。
- `client.platform.saveFile(...)` 中传入的 `defaultPath` 只用于给系统保存对话框提供初始建议位置，不能绕过用户确认。

## 当前边界

- 编辑引擎自身不实现格式转换，只负责保存当前卡片状态、发起正式能力调用和汇总任务状态；
- HTML / PDF / 图片导出效果以文件转换模块和 Host 正式渲染导出能力为准；
- 本实现不自行拼装 ZIP，不绕过 Host 直接写归档文件，也不直接访问 Electron 导出接口。
