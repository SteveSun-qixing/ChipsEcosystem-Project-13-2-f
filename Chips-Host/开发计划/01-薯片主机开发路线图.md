# 薯片主机开发路线图

> 文档状态：执行基线稿
> 目标：以“先底座、再链路、后治理、最后发布”的顺序完成开发。

---

## 1. 总体节奏

| 阶段 | 目标 | 关键产物 |
|---|---|---|
| Phase 0 | 基线冻结与风险清理 | 需求/技术/计划文档冻结，风险清单 |
| Phase 1 | L1-L2 底座开发 | PAL + Kernel 可运行基线 |
| Phase 2 | L3 服务域开发 | 16 服务域动作面与契约测试 |
| Phase 3 | L4-L6 通信链路开发 | Plugin Runtime + Bridge + Runtime Client |
| Phase 4 | 质量治理与发布 | 质量门禁全绿、灰度验证、发布回滚方案 |
| Phase 5 | L8 声明式 UI 分阶段交付 | 语义原语、组合模式、事件/副作用模型、门禁验收 |
| Phase 6 | L9 统一渲染层分阶段交付 | 渲染流水线、目标适配器、一致性约束、失败隔离、增量调度 |
| Phase 7 | L11 主题运行时分阶段交付 | 五层 token 解析、主题作用域链、覆盖规则、契约门禁与管理工具 |

---

## 2. 阶段详情

## 2.1 Phase 0：基线冻结

- 冻结需求与边界，不允许边做边改架构主线。
- 对齐生态共用文档中的协议与接口口径。
- 建立风险工单流与阻断规则。

## 2.2 Phase 1：PAL + Kernel

- PAL 三平台能力接口统一。
- Kernel 路由/事件/生命周期能力可用。
- 路由契约描述符与 `route-manifest` 生成链路可用。

## 2.3 Phase 2：Host Services

- 落地 16 服务域动作面。
- 完成 schema、权限、超时、错误码定义。
- 完成动作契约测试与核心集成测试。

## 2.4 Phase 3：Runtime + Bridge

- 插件运行时安全流水线落地。
- Bridge IPC 主/子通道落地。
- Runtime Client 完成错误归一与重试策略。

## 2.5 Phase 4：质量与发布

- 性能、稳定性、安全门禁全部通过。
- 完成灰度验证、回滚预案演练。
- 合并至 `develop` 并输出发布说明。

## 2.6 Phase 5：L8 声明式 UI 分阶段交付

- 以“先契约、再原语、再组合、再事件、最后验收”顺序推进。
- 输出 Host 内置 Declarative UI 标准库，覆盖 `View/Stack/Grid/Form/List`。
- 固化 Vercel 组合模式与副作用分级约束，保证无 boolean 模式扩张、无渲染阶段 `runtime-effect`。
- 通过分层测试与文档收口后进入主干合并流程。

## 2.7 Phase 6：L9 统一渲染层分阶段交付

- 以“先契约、再流水线、再适配器、再隔离调度、最后集成验收”顺序推进。
- 实现 6 阶段渲染流水线：Normalize/Validate/Resolve/Compute/Commit/Dispatch。
- 完成 4 类目标适配器：`App Root`、`Card Iframe`、`Module Slot`、`Offscreen Render`。
- 落地一致性约束与失败隔离，保证单节点失败不影响整页渲染。
- 实现 `render queue` 与可见区域增量提交，达成性能门禁后再进入发布流程。

## 2.8 Phase 7：L11 主题运行时分阶段交付

- 以“先模型与契约、再 token 解析、再作用域链、再解析算法、最后契约门禁与管理工具”顺序推进。
- 按生态规范实现五层 token 体系：`ref`（原始值）、`sys`（语义值）、`comp`（组件值）、`motion`（动效值）、`layout`（布局密度值）。
- 落地主题作用域链与覆盖规则：
  - 对外语义：`component → card → workspace → app → global`
  - 内部实现：`component → base-card → composite-card → box → app → global`
- 将 `theme.resolve` 从简单 token 合并演进为完整解析算法：加载作用域链快照、按层级合并、冲突裁决、生成变量表与组件映射表、注入作用域样式并触发 `theme.changed`。
- 实现主题契约门禁：token 完整性校验、组件接口点校验、动效参数安全校验、颜色对比度校验，并提供主题管理与诊断工具。

---

## 3. 关键依赖关系

- Phase 2 依赖 Phase 1 完成（服务依赖 Kernel）。
- Phase 3 依赖 Phase 2 完成（Bridge 需服务动作稳定）。
- Phase 4 依赖 Phase 1-3 完成并通过联调。
- Phase 5 依赖 Phase 3 完成（L8 依赖 Runtime Client + UI Hooks 稳定接口）。
- Phase 6 依赖 Phase 5 完成（L9 依赖 L8 声明树与事件语义稳定）。
- Phase 7 依赖：
  - Phase 6 完成（L11 解析结果需与 L9 统一渲染链路协同）。
  - 组件库阶段四“主题运行时与 Tamagui-Core 适配”完成（组件消费侧契约稳定）。
  - 生态共用层主题系统与工具链工单闭环（如 Style Dictionary 统一 preset 等）。

---

## 4. 执行原则

- 每阶段必须有可演示、可测试、可回滚的中间产物。
- 阶段验收未通过，下一阶段不得启动。
- 不允许引入临时实现来“先跑通再补”。
