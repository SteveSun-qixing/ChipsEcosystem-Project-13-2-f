# L11 主题运行时开发计划

> 文档状态：新增阶段执行基线稿  
> 适用范围：`Chips-Host` L11 Theme Runtime（主题运行时）  
> 目标：交付完整的主题运行时能力（五层 token 解析、作用域链与覆盖规则、解析算法、契约门禁与管理工具），并与 L8/L9/组件库对齐。

---

## 1. 阶段目标与边界

- L11 Theme Runtime 归属 Host 运行时主实现，本计划在 `Chips-Host` 仓内完成，不在组件库侧重复实现。
- 严格对齐生态共用文档：
  - `生态共用技术文档/架构设计/15-主题系统设计.md`
  - `生态共用技术文档/主题系统/*`
  - `生态共用技术文档/组件库/03-Token与主题对接标准.md`
- 明确职责边界：
  - **本阶段负责**：主题 token 五层解析（`ref/sys/comp/motion/layout`）、作用域链解析与覆盖规则、`theme.resolve` 运行时算法、`theme.contract.get` 契约门禁、主题管理工具与诊断。
  - **不负责**：具体主题包视觉设计、组件库内部样式实现（由组件库与主题包负责）。
- 与其他层协作关系：
  - L8 Declarative UI 通过 `themeScope` 提供作用域线索，不参与 token 合并。
  - L9 Unified Rendering 只依赖 Theme Runtime 暴露的解析结果，不自行管理 token。
  - L10 Component Headless 只消费 Theme Runtime 提供的变量与契约。
- 开发约束：
  - 不引入技术选型外第三方依赖（如需新增，必须先评审并更新生态共用文档）。
  - 禁止临时实现、占位实现和 TODO；所有入口必须达到可测试、可验收状态。
  - 禁止改动任务范围外模块（PAL/Kernel/其他服务域等）。

### 3.1 R0 架构基线与契约冻结

**目标**

- 将 L11 Theme Runtime 的职责和边界与 L8/L9/L10 明确切分。
- 冻结核心数据模型与错误码，为后续实现提供稳定基线。

**任务清单**

1. 契约梳理与口径统一：
   - 汇总并对齐以下文档中的主题相关口径：
     - `生态设计原稿/06-前端多样化和主题系统.md`
     - `生态共用技术文档/架构设计/15-主题系统设计.md`

     - `生态共用技术文档/主题系统/02-主题接口规范.md`
     - `生态共用技术文档/组件库/03-Token与主题对接标准.md`
   - 确认 Host 侧 `theme.list/apply/getCurrent/getAllCss/resolve/contract.get` 动作的最终参数与返回结构。
2. 数据模型设计：
   - 定义内部核心模型（以 TypeScript 类型或设计说明形式落地）：
     - `ThemePackageManifest`
     - `ThemeTokenLayers`：`ref/sys/comp/motion/layout`
     - `ThemeScopeId` 与 `ThemeScopeChain`
     - `ThemeSnapshot`（包含五层 token 与作用域局部覆盖）
     - `ResolvedThemeVariables` 与 `ComponentTokenMap`
   - 对齐组件库侧 `createScopedTokenResolver` / `ThemeSnapshot` 预期结构，确保兼容。
3. 作用域链语义映射：
   - 将全局开发大纲中的 `component → card → workspace → app → global` 与生态共用规范中的 `global -> app -> box -> composite-card -> base-card -> component` 做一一映射：
     - 对外语义：`component/card/workspace/app/global`
     - 内部实现：`component/base-card/composite-card/box/app/global`
   - 明确 Host theme 服务在 workspace 级别与 card/box 级别的配置来源与存储方式。
4. 错误码与诊断字段冻结：
   - 整理并扩展主题相关错误码：
     - `THEME_NOT_FOUND`
     - `THEME_CHAIN_TOO_DEEP`
     - `THEME_CONTRACT_INVALID`
     - `THEME_DEPENDENCY_ERROR`
     - `THEME_LOAD_FAILED`
     - `THEME_CSS_PARSE_ERROR`
     - `THEME_TOKEN_MISSING` / `THEME_TOKEN_FALLBACK_APPLIED`
   - 统一诊断结构：`{ code, message, details, scope, tokenKey?, component?, retryable? }`。
5. 文档与任务拆解输出：
   - 在 `Chips-Host/技术文档` 中新增/补充 L11 相关小节（可挂在总体架构或主题系统章节下）。
   - 明确各阶段 R1-R4 的子任务与验收标准，更新本开发计划与迭代任务拆解文档。

**交付物**

- L11 Theme Runtime 架构说明与数据模型定义。
- 主题服务错误码与诊断规范。
- 更新后的任务拆解与阶段计划（本文件 + `02-薯片主机迭代任务拆解.md` 衔接）。

**验收标准**

- 架构与数据模型通过审查，且与生态共用文档口径完全一致。
- 关键接口签名稳定，并在 Host 技术文档中有明确记录。
- 未发现跨层依赖违规（L11 只依赖主题包与 Host 内部基础设施）。

### 3.2 R1 token 五层解析与主题快照

**目标**

- 从主题包产物中解析并构建五层 token（`ref/sys/comp/motion/layout`）。
- 生成可被 L9/L10 消费的标准化 `ThemeSnapshot`，并支持缓存与诊断。

**任务清单**

1. 主题包载入与 manifest 解析：
   - 复用/补充现有主题安装与登记流程，将主题包 manifest（`manifest.yaml`）与 tokens 路径解析为 Host 内部记录。
   - 设计 `ThemeRecord` 扩展结构，从当前的 `{ id,name,publisher,css,tokens }` 演进为同时持有：
     - `layers: { ref/sys/comp/motion/layout }`
     - `rawContracts`（来自 `contracts/theme-interface.contract.json`）
     - 必要的依赖信息与版本。
2. 五层 token 解析流水线：
   - 从主题包 `tokens/ref.json/sys.json/comp.json/motion.json/layout.json` 构建层级化 token 对象。
   - 明确内部引用解析规则（如 `{color.blue}`）与循环依赖检测策略。
   - 输出统一的 `ThemeTokenLayers` 结构，并支持基础 schema 校验（键名、值类型）。
3. ThemeSnapshot 构建：
   - 设计 `ThemeSnapshot` 结构，至少包含：
     - 基础 token 层（五层）
     - 可选 scopes 覆盖（按组件/卡片/盒子等 scope 组织）
   - 实现 `buildThemeSnapshot(themeId: string): ThemeSnapshot`：
     - 从已安装主题包中加载并缓存 snapshot。
     - 提供版本化 cache key（`themeId + version`）。
4. 诊断与调试支持：
   - 在 snapshot 构建过程中记录：
     - 缺失 token 列表（候选默认值）。
     - 冗余 token 列表。
     - 解析失败位置与原因。
   - 输出到统一日志或诊断通道，便于后续契约校验与开发调试。
5. 单元测试与样例主题：
   - 准备最小可用主题包 fixture，用于验证五层解析 correctness。
   - 覆盖：
     - 合法主题包解析路径。
     - token 引用展开路径。
     - 错误场景（文件缺失、格式错误）与对应错误码。

**交付物**

- 主题包五层 token 解析实现。
- `ThemeSnapshot` 构建工具与缓存策略。
- 对应单元测试与解析日志。

**验收标准**

- 五层 token 解析结果与 `15-主题系统设计.md` 示例保持一致。
- 对于非法主题包，错误码与诊断信息清晰可追踪。
- 解析性能满足基础门禁（在预期主题包规模下耗时可接受）。

### 3.3 R2 作用域链与覆盖规则引擎

**目标**

- 实现主题作用域链与覆盖规则，使 token 解析遵循近者优先、选择器特异性与声明顺序三层规则。

**任务清单**

1. 作用域模型定义：
   - 对外统一语义：`component → card → workspace → app → global`。
   - 内部实现映射：`component → base-card → composite-card → box → app → global`。
   - 定义 `ThemeScopeId`（字符串或结构体）与 `ThemeScopeChain` 类型，并约束最大长度（<= 6）。
2. 作用域链构建：
   - 根据当前工作区、应用、箱子、卡片、组件配置构建作用域链：
     - workspace/app: 来自配置服务与当前 workspace 设置。
     - box/card: 来自卡片/箱子元数据与结构文件。
     - component: 来自声明式 UI 节点或组件库接口提供的 `themeScope`。
   - 实现 `buildScopeChain(context): ThemeScopeChain`，作为 `theme.resolve` 输入。
3. 覆盖规则引擎：
   - 实现逐 key 解析规则：
     - 先按作用域优先级：component > base-card > composite-card > box > app > global。
     - 同一作用域内按 CSS 选择器特异性排序。
     - 特异性相同按声明顺序（后声明覆盖前声明）。
   - 明确与 CSS 变量注入策略的对应关系：
     - `:root` / `[data-chips-app]` / `[data-chips-box]` / `[data-chips-composite-card]` / `[data-chips-base-card]` / `[data-chips-component]`。
4. 回退与容错：
   - 当某一作用域未提供特定 token 时，逐层回退至更低权重作用域。
   - 当所有作用域均缺失时：
     - 默认回退至系统默认主题。
     - 记录 `THEME_TOKEN_FALLBACK_APPLIED` 诊断事件。
5. 测试场景设计：
   - 构造多层覆盖场景：
     - 只在 global 定义。
     - global + app + component 多级覆盖。
     - 同一作用域内多条规则的特异性与声明顺序冲突。
   - 对应单测验证解析结果与预期一致。

**交付物**

- 作用域链构建与覆盖规则引擎实现。
- 作用域链诊断工具（可输出当前节点的作用域解析路径）。
- 覆盖规则单元测试。

**验收标准**

- 任意给定 scope 与 token key，解析结果与文档规范一致。
- 回退行为可预测且具备日志/诊断记录。
- 不引入跨层硬耦合（作用域信息通过标准接口传递）。

### 3.4 R3 主题解析算法与运行时集成

**目标**

- 严格按照生态规范实现 `theme.resolve` 解析算法，并将结果接入 L9 统一渲染与组件库消费链路。

**任务清单**

1. 解析算法实现：
   - 按 `15-主题系统设计.md` 的参考伪代码实现：
     1. 加载作用域链主题快照（R1 + R2 输出）。
     2. 按 token 层级合并（`ref -> sys -> comp/motion/layout`）。
     3. 执行冲突裁决（依据作用域优先级、特异性、声明顺序）。
     4. 生成变量表（`--chips-*`）与组件映射表（`component -> tokens`）。
     5. 注入作用域样式（按 CSS 变量策略）并触发 `theme.changed`。
   - 设计 `resolveTheme(chain: ThemeScopeChain): ResolvedTheme` 内部接口，并通过 `theme.resolve` 暴露。
2. Host 服务域集成：
   - 将 `src/main/services/register-host-services.ts` 中现有的 `theme.resolve` 从简单 token 合并演进为调用上述解析算法：
     - 支持空链回退到当前主题与默认作用域。
     - 保持 `THEME_CHAIN_TOO_DEEP` 防御逻辑。
   - 确保 `theme.list/apply/getCurrent/getAllCss` 语义与解析算法保持一致（apply 后 resolve chain 默认包含新主题等）。
3. L9 统一渲染层集成：
   - 对 `packages/unified-rendering/src/theme-resolver.ts` 做架构对齐：
     - 调整 `ThemeSnapshot` 结构，与 R1/R2 输出一致。
     - 将当前直接在 L9 层解析 `token.*` 的逻辑收敛为消费 Theme Runtime 提供的 `ResolvedThemeVariables`（避免 L9 内部自建解析器）。
   - 确保：
     - L9 仍然只负责将 `token.*` 替换为具体值，不管理作用域链与覆盖规则。
     - 主题解析失败时抛出统一错误码（如 `RENDER_THEME_TOKEN_NOT_FOUND`），并可从诊断信息中追溯到 L11。
4. 与组件库/主题 Provider 配合：
   - 明确组件库侧 `ChipsThemeProvider/useToken/useComponentTokens` 与 Host Theme Runtime 的接口契合点：
     - 通过 Bridge `theme.*` 动作获取 tokens 和契约。
     - 订阅 `theme.changed` 事件，在变量表更新后进行变量级刷新。
   - 在本计划中只定义 Host 侧行为，不主动修改组件库仓代码，但要保证接口契约稳定。
5. 测试与性能基线：
   - 更新 Host 集成测试：
     - `Chips-Host/tests/integration/host-services.test.ts` 中主题相关用例改为验证五层解析+作用域链行为。
     - 增加多作用域、多层 token 的端到端用例。
   - 建立基础性能采样：
     - 单次 `theme.resolve` 调用耗时。
     - 主题切换触发的 L9 渲染耗时（不做过度优化，但要有数据基线）。

**交付物**

- 完整 `theme.resolve` 解析实现。
- L9 统一渲染层对 Theme Runtime 的集成改造。
- 端到端主题解析与渲染测试。

**验收标准**

- `theme.resolve` 的行为与生态规范完全一致（作用域链、覆盖规则、冲突裁决）。
- L9 渲染在主题变更前后保持结构不变，只更新变量。
- 所有相关单元/集成测试通过，且无回归 L8/L9 既有用例。

### 3.5 R4 主题契约门禁与管理工具

**目标**

- 实现主题契约校验闭环（token 完整性、组件接口点、动效安全、颜色对比度），并提供开发与运营可用的主题管理工具。

**任务清单**

1. token 完整性校验：
   - 基于组件库公开的 token 清单（`chips.comp.*`）与主题包声明：
     - 检查必需 token 是否全部提供。
     - 检查是否存在不在契约内的冗余 token（发出警告而非硬失败）。
   - 在主题安装/载入与 `theme.apply` 时进行校验：
     - 严重缺失时阻断应用，返回 `THEME_CONTRACT_INVALID`。
2. 组件接口点校验：
   - 基于主题包 `contracts/theme-interface.contract.json` 与组件库契约：
     - 校验组件 `scope/parts/states/tokens` 是否完整。
     - 校验主题包中的 CSS 是否仅使用契约定义的接口点（可通过静态扫描或构建时工具完成）。
   - 在 Host 侧维护契约快照，方便 `theme.contract.get` 返回结构化结果。
3. 动效参数安全校验：
   - 定义动效白名单规则：
     - 最大时长（例如 `<= 3000ms`）。
     - 支持的 easing 曲线集合。
     - 禁止闪烁/癫痫风险动画模式。
   - 对 `motion` 层 token 与 animations CSS 做静态校验：
     - 不满足安全约束直接阻断主题启用并返回错误。
4. 颜色对比度校验：
   - 实现基础 WCAG 对比度计算工具（优先使用现有内部工具，如无则在不引入新依赖前提下实现简易版本）。
   - 对常见文本与背景组合（primary/secondary/error 等）计算对比度：
     - 不达标时给出告警并标记为“需设计评审”，是否阻断作为可配置策略。
5. `theme.contract.get` 实现与诊断接口：
   - 将现有临时实现替换为基于真实契约的实现：
     - 支持按 `component` 过滤返回。
     - 提供 `scope/parts/states/tokens` 等字段。
   - 扩展返回结构以包含校验结果与诊断信息（例如缺失 token 列表）。
6. 主题管理工具：
   - 提供至少一类管理入口（具体形态在实现阶段细化）：
     - CLI：如 `npm run theme:validate` / `npm run theme:list`。
     - 或 Host 内部调试动作：如 `theme.inspect`（内部使用，不对外公开）。
   - 支持输出：
     - 已安装主题列表与基本信息。
     - 单个主题的契约校验报告。
     - 颜色对比度报告与动效安全报告。
7. 门禁集成：
   - 将主题校验纳入 Host 质量门禁链路：
     - `npm run verify` 或专门 `npm run quality:theme`。
   - 确保 CI 中主题相关门禁不过则禁止合并到 `develop`。

**交付物**

- 主题契约校验实现与错误码。
- `theme.contract.get` 与主题管理工具。
- 质量门禁脚本与主题校验报告模板。

**验收标准**

- 所有官方主题包通过契约门禁与安全校验。
- 对于不合规主题包，Host 能给出明确错误信息与可操作建议。
- 主题相关门禁已纳入常规发布前检查流程。

## 5. 阻断与工单规则

出现以下任一情况，必须立即在 `项目日志与笔记/问题工单.md` 中新增工单，并暂停本阶段进一步实现，等待上游闭环后再继续：

- 生态共用文档中主题相关规范存在冲突或缺失，导致无法确定最终口径。
- 主题包工具链（Style Dictionary preset、统一 schema 校验工具等）未交付，导致无法实现统一的五层 token 解析。
- 组件库侧主题契约或 token 命名规范不完整，无法完成 `theme.contract.get` 与 token 完整性校验。
- 颜色对比度或动效安全校验在“禁止新增第三方依赖”的前提下无法安全实现。
- Host/Kernel/公共基础层未提供主题包发现与加载所需的基础能力。

阻断时处理步骤：

1. 在 `项目日志与笔记/问题工单.md` 中创建新工单，描述问题背景、阻断影响与期望行为。
2. 在 `项目日志与笔记/任务清单/<工单编号>/任务说明.md` 中创建专项任务文档。
3. 立即停止本阶段功能实现，等待工单状态更新为“已闭环”后再继续执行本计划。
