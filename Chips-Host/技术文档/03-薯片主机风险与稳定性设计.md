# 薯片主机风险与稳定性设计

> 文档状态：稳定性设计基线稿
> 目的：在大规模底层开发中控制架构风险、集成风险与交付风险。

---

## 1. 风险分类

| 风险类型 | 典型场景 | 影响面 |
|---|---|---|
| 架构风险 | 跨层依赖、绕过内核路由 | 全链路耦合、后续难维护 |
| 接口风险 | 动作名漂移、schema 漂移 | SDK/插件兼容性中断 |
| 运行风险 | 插件越权、资源超配额 | 系统不稳定或安全问题 |
| 质量风险 | 测试不足、性能回退 | 线上故障率上升 |
| 发布风险 | 灰度/回滚策略不完善 | 版本不可控 |

---

## 2. 风险控制策略

## 2.1 架构风险控制

- 采用分层依赖扫描，阻断 `Ln -> Ln-2` 等违规依赖。
- Kernel 层统一治理服务调用，禁止服务间直接 import。
- 所有开发变更先更新契约再实现，避免隐式行为变更。

## 2.2 接口风险控制

- 路由注册自动生成 `route-manifest`，用于 SDK 与测试比对。
- 兼容动作映射集中在 Runtime Client，禁止在业务层散落别名逻辑。
- 关键动作建立快照测试，防止无感破坏。

## 2.3 运行风险控制

- 高敏动作启用强制审计日志。
- 插件会话必须执行权限快照校验。
- 配额触发降级后必须有明确告警与恢复路径。

## 2.4 质量风险控制

- 所有门禁前置到 CI：lint、typecheck、unit、contract、integration、perf。
- 性能基线以 p50/p95 统计值判定，禁止“人工体感通过”。
- 引入故障注入回归：超时、断连、重试耗尽、熔断恢复。

---

## 3. 回退与恢复设计

## 3.1 代码回退

- 按里程碑切分可独立回滚的提交批次。
- 每个批次配套变更说明、影响域和验证脚本。

## 3.2 运行回退

- 插件更新失败回滚至上一稳定版本。
- 主题/配置异常回退默认安全配置。
- Bridge 通道异常时启用降级提示，避免界面冻结。

## 3.3 数据恢复

- 写操作采用幂等 requestId 去重。
- 关键事务支持崩溃恢复后的补偿重放（白名单动作）。

---

## 4. 问题发现与工单流程

当开发过程中发现以下问题，必须先进入工单流程，再推进开发：

- 生态底层未打通（主机/内核/公共基础层联动断裂）。
- 组件库缺失关键组件导致功能无法按规范落地。
- SDK 工具或标准接口不可用，无法满足既定需求。
- 出现临时开发代码或无法满足架构原则的历史残留。

### 流程

1. 在《项目日志与笔记》的《问题工单.md》登记问题。
2. 新建对应任务文件夹，补充复现路径、影响范围、预期修复方案。
3. 暂停当前实现型开发，等待问题处理闭环后再继续。

---

## 5. 稳定性验收标准

- 无跨层依赖违规。
- 无未契约化动作。
- 无未审计高风险动作。
- 无关键链路性能超阈值。
- 无阻断级缺陷（崩溃、数据损坏、权限失控）。
