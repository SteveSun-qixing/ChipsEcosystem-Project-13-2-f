## 1. L11 Theme Runtime 层级定位

### 1.1 在十二层架构中的位置

结合生态设计原稿与架构设计手册，L11 Theme Runtime 的定位如下：

- L9 Unified Rendering：负责 Normalize / Validate / Resolve / Compute / Commit / Dispatch 渲染流水线，不直接管理主题 token 体系。
- L10 Component Headless：只负责组件状态机、结构与 a11y，通过 Theme Runtime 提供的变量与契约消费主题。
- **L11 Theme Runtime**：负责主题 token 五层解析、作用域链与覆盖规则、变量注入与契约门禁。

> 运行时依赖关系：  
> L8 Declarative UI → L9 Unified Rendering → **L11 Theme Runtime** → Theme Package。

### 1.2 与 Host Services 的关系

L11 Theme Runtime 与 L3 `theme` 服务域的职责分工：

- L3 `theme` 服务：
  - 对外暴露固定 6 个动作：`theme.list/apply/getCurrent/getAllCss/resolve/contract.get`；
  - 管理主题安装、当前主题状态、权限与错误码；
  - 不直接实现五层 token 解析与作用域算法，而是委托 L11 Theme Runtime。
- L11 Theme Runtime：
  - 实现主题包解析、五层 token 结构化、作用域链构建、覆盖规则与冲突裁决；
  - 输出标准化的 ThemeSnapshot / ResolvedTheme 结果供 L3 服务与 L9/L10 消费；
  - 承担主题契约门禁（token 完整性、组件接口点、动效安全、颜色对比度）。

## 2. 数据模型与核心类型

> 本节定义 Host 内部使用的数据模型，类型实现统一落地在 `src/shared` 或 `src/main/theme-runtime` 目录中，确保 L3/L8/L9/L10 可共享。

### 2.1 ThemeTokenLayers（五层 token 结构）

五层 token 结构是主题解析的基础输入模型：

- `ref`：原始值层（色板、基础间距、字体参数等）；
- `sys`：语义值层（primary/secondary/surface 等语义 token）；
- `comp`：组件值层（button/input/card-shell 等组件语义）；
- `motion`：动效层（过渡时长、曲线、关键帧命名等）；
- `layout`：布局密度层（间距密度、列表高度、区域间距等）。

内部类型在 `Chips-Host/src/main/theme-runtime/types.ts` 中实现：

```ts
export interface ThemeTokenLayers {
  ref: Record<string, unknown>;
  sys: Record<string, unknown>;
  comp: Record<string, unknown>;
  motion: Record<string, unknown>;
  layout: Record<string, unknown>;
}
```

### 2.2 ThemeScopeId 与 ThemeScopeChain

主题作用域链采用“近者优先”的六级模型，与生态共用文档约定一致：

- 内部实现链路（低 → 高）：`global -> app -> box -> composite-card -> base-card -> component`
- 对外语义映射：`component → card → workspace → app → global`

内部使用类型：

- `ThemeScopeId`：单一作用域标识，采用字符串表示（如 `global`、`app:main`、`box:overview`、`card:todo-list`、`component:button`）；
- `ThemeScopeChain`：有序数组，自低权重向高权重排列，最大长度 6。

### 2.3 ThemeSnapshot 与 ResolvedTheme

L11 Theme Runtime 内部使用的主题快照模型：

- `ThemeSnapshot`：
  - `id`: 当前主题或组合主题链的标识；
  - `layers`: `ThemeTokenLayers`，五层 token 结构；
  - `scopes?`: 作用域局部覆盖（按 `ThemeScopeId` 组织的 token 片段）；
  - `version?`: 主题版本信息，用于 cache key；
  - `diagnostics?`: 解析阶段产生的告警与信息。

- `ResolvedTheme`：
  - `variables`: 扁平化 CSS 变量表（`--chips-*`）；
  - `componentTokens`: 组件级 token 映射（`componentId -> tokenKey -> value`）；
  - `diagnostics`: 合并与冲突裁决过程中的诊断信息。

> 注意：L9 统一渲染层继续使用其本地 `ThemeSnapshot` 结构作为上下文输入，但实际 token 内容由 L11 Theme Runtime 负责构建与维护，确保 L9 不再实现独立解析器。

### 2.4 ThemeRecord 扩展模型

在 Host `RuntimeState` 中，主题元信息使用扩展后的 `ThemeRecord` 结构：

- 基本字段：
  - `id: string`：主题唯一标识；
  - `name: string`：显示名称；
  - `publisher: string`：发行者；
  - `css: string`：主题打包后的 CSS 代码（供 `theme.getAllCss` 使用）。
- 运行时字段：
  - `layers: ThemeTokenLayers`：五层 token 数据；
  - `resolvedSnapshot?`: `ThemeSnapshot`：按需构建并缓存的主题快照；
  - `contracts?`: 主题契约原始数据（来自 `contracts/theme-interface.contract.json`）；
  - `diagnostics?`: 解析与校验阶段的诊断信息。

现有实现中 `RuntimeState.themes` 已持有基础 `ThemeRecord` 列表，后续阶段会按上述模型完整落地，并保证向下兼容。

## 3. 解析与合并流程总览

本节对应生态共用《主题系统设计》中的“主题解析算法”部分，在 Host 内部拆解为三个子阶段：

1. **主题包解析（Theme Package Parsing）**：  
   - 从主题包目录或内置配置加载 manifest 与 token 文件；
   - 生成 `ThemeTokenLayers` 与基础 `ThemeSnapshot`。
2. **作用域链构建与覆盖（Scope Chain & Override）**：  
   - 根据 workspace/app/box/card/component 环境构建 `ThemeScopeChain`；
   - 将不同层级的主题快照合并为统一视图。
3. **变量表与组件映射生成（Variable & Component Mapping）**：  
   - 扁平化五层 token，并按组件契约生成组件级 token 映射；
   - 注入 CSS 变量并触发 `theme.changed` 事件。

> 本文档仅定义 Host 内部职责与模块分工，具体阶段实现进度以 `Chips-Host/开发计划/07-L11主题运行时开发计划.md` 中的 R0-R4 阶段说明为准。

## 4. 模块划分与代码结构

> 目标：保证 Theme Runtime 实现与 Host 其他模块解耦，进入仓库后能独立测试与演进。

当前已落地的模块结构：

- `src/main/theme-runtime/`
  - `types.ts`：`ThemeTokenLayers/ThemeScopeId/ThemeSnapshot/ResolvedTheme` 等核心类型定义；
  - `token-layers.ts`：`asThemeTokenLayers` 实现，对 `ThemeRecord.tokens` 进行五层结构化与基础校验；
  - `resolve-algorithm.ts`：主题解析算法内核，包含：
    - `mergeThemeLayers`：按主题链合并五层 token；
    - `resolveThemeFromLayers`：按 `ref -> sys -> comp/motion/layout` 顺序解析引用并生成变量表与 `componentTokens` 组件映射表；
    - `buildThemeSnapshot`：从 `ThemeRecord.tokens` 构建 `ThemeSnapshot`。
  - `scope-chain.ts`：`ThemeScopeId/ThemeScopeChain` 定义与作用域链构建逻辑（`buildThemeScopeChain`），并配套单元测试验证顺序与边界；
  - `render-bridge.ts`：将 L11 `ResolvedTheme` 转换为 L9 `ThemeSnapshot` 的桥接模块（`toRenderThemeSnapshot`），供统一渲染层按 `token.*` 语义消费。

后续待落地的模块（在后续阶段实现）：

- `diagnostics.ts`：诊断事件结构与记录工具。

- 在 L3 `theme` 服务域中（`src/main/services/register-host-services.ts`）：

- 通过 Theme Runtime 内部接口实现：
  - `theme.resolve`：调用 `mergeThemeLayers` 与 `resolveThemeFromLayers` 组合主题链路并返回 `{ resolved, tokens }`；
  - `theme.apply`：在切换主题前使用 `mergeThemeLayers` 和 `resolveThemeFromLayers` 解析当前主题 tokens，并通过 `validateThemeContractWithTokens` 执行契约门禁，失败时抛出 `THEME_CONTRACT_INVALID` 阻断主题应用；
  - `theme.contract.get`：通过 `buildThemeContractsView` 返回当前主题的组件契约视图。
- 继续管理 `theme.list/apply/getCurrent/getAllCss` 的服务级职责（状态管理与 CSS 分发），并在 `buildState` 中为默认主题提供基础五层 token 结构：
  - 默认主题 `tokens` 使用 `{ ref: {...}, sys: {}, comp: {}, motion: {}, layout: {} }` 形式，满足五层结构要求。

在 L9 统一渲染层中：

- `packages/unified-rendering/src/theme-resolver.ts` 将演进为：
  - 仅负责根据 L11 提供的 tokens 将 `token.*` 形式的属性替换为具体值；
  - 不再自行定义独立的 token 查找规则或作用域逻辑。

## 5. 错误码与诊断模型

Theme Runtime 遵循统一错误对象结构（`StandardError`），并在主题相关场景中使用以下错误码（与生态共用文档对齐）：

- `THEME_NOT_FOUND`：指定主题不存在或不可用；
- `THEME_CHAIN_TOO_DEEP`：`theme.resolve` 输入的作用域链长度超出限制（> 6）；
- `THEME_CONTRACT_INVALID`：主题契约校验失败（缺失必要 token 或接口点）；
- `THEME_DEPENDENCY_ERROR`：主题依赖主题或资源加载失败；
- `THEME_LOAD_FAILED`：主题包读取或解析失败；
- `THEME_CSS_PARSE_ERROR`：主题 CSS 解析失败；
- `THEME_TOKEN_MISSING`：解析过程中找不到必需的 token；
- `THEME_TOKEN_FALLBACK_APPLIED`：已应用回退 token（通过诊断记录，不一定作为硬错误）。

诊断信息统一通过以下渠道暴露：

- Host 内部日志（结构化字段包含 `scope/themeId/tokenKey/stage` 等信息）；
- 主题管理工具输出的报告（后续阶段在 CLI 或调试接口中提供）；
- L9 渲染诊断结构中的 `RenderNodeDiagnostic`，在主题解析失败时带出关联节点信息。

## 6. 与组件库与主题包的协作

### 6.1 与组件库（L10）的协作

- 组件库只通过以下接口消费 Theme Runtime 成果：
  - `theme.resolve` 返回的 token 映射；
  - `theme.contract.get` 返回的组件契约定义；
  - `theme.changed` 事件通知（通过 UI Hooks 或 Provider 订阅）。
- 组件库侧 `ChipsThemeProvider/useToken/useComponentTokens` 负责：
  - 通过 Bridge 调用 Host 主题服务获取 tokens 与契约；
  - 在 `theme.changed` 时执行变量级刷新，避免重建组件树。
- Theme Runtime 不直接依赖组件实现，只依赖组件契约与 token 命名规范（`chips.comp.*`）。

### 6.2 与主题包（Theme Package）的协作

- 主题包必须满足：
  - 目录结构与 manifest 要求（详见《主题包开发指南》）；
  - 提供完整的五层 token 定义文件与契约文件；
  - 不在包内混合多种模式（例如 light/dark），不同外观使用不同 `themeId`。
- Theme Runtime 在 Host 侧负责：
  - 加载并解析主题包的 manifest 与 token 文件；
  - 执行主题契约门禁（当前版本主要聚焦组件 token 完整性，动效参数安全与颜色对比度校验仍由组件库侧脚本与后续版本补齐）；
  - 仅在通过门禁后允许 `theme.apply` 与后续解析。

### 6.3 2026-03-10 主题链路闭环补充

本轮主题链路修正后，Host 侧补齐了以下正式行为：

- 主题来源收敛到当前工作区中已启用的主题插件，`loadInstalledThemes` 在装载时显式过滤 `plugin.enabled === true`；
- 主题插件发生 `install/enable/disable/uninstall` 时，通过 `syncInstalledThemes` 立即刷新 `state.themes` 与当前主题状态；
- `resolveThemedWindowChrome` 在应用未显式指定背景色时，从 `chips.sys.color.canvas` / `chips.sys.color.surface` 推导原生窗口背景；
- preload 通过 `theme.getCurrent/getAllCss/resolve` 把主题 CSS、变量和 `data-chips-theme-id` / `data-chips-theme-version` 注入应用文档；
- 复合卡片渲染链只负责注入主题包 CSS 与变量，不再附加固定的浅色模式声明；
- 应用层可通过 `theme.changed` 和主题运行时缓存键，驱动复合卡片窗口重建 iframe 内容。

### 6.4 2026-03-26 主题 CSS 资源路径重写补充

- Host 在读取主题包 `dist/theme.css` 时，会以该 CSS 文件所在目录为基准，把其中相对 `url(...)` 资源路径重写为绝对 `file://` URL；
- 该重写发生在 `loadThemeRecordFromPlugin -> rewriteThemeCssAssetUrls(...)` 阶段，随后再把 CSS 文本注入 preload 和卡片渲染链；
- 这样即使主题 CSS 通过 `<style>` 内联注入文档，图标字体等静态资源仍能正确解析，不会因为失去原始 CSS 文件相对路径上下文而 404；
- `data:`、`blob:`、`file:`、协议 URL、根路径 URL 与片段引用不会被重写。
