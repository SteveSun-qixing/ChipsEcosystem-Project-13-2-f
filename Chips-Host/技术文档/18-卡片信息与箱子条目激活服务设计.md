# 卡片信息与箱子条目激活服务设计

## 1. 文档定位

本文档只描述 `Chips-Host` 内部实现细节。  
对外正式契约以以下共享文档为准：

- `生态共用技术文档/协议与契约/09-卡片信息与打开契约.md`
- `生态共用技术文档/协议与契约/08-箱子布局运行时契约.md`

## 2. 新增内部服务包

### 2.1 `packages/card-info-service`

职责：

1. 统一读取卡片 `status / metadata / cover`；
2. 输出标准化卡片信息视图；
3. 作为箱子、资源管理器、选择器等上层能力的唯一卡片信息入口。

依赖：

1. `CardService.validate`
2. `CardService.readMetadata`
3. `CardService.renderCover`

### 2.2 `packages/card-open-service`

职责：

1. 统一执行本地卡片打开；
2. 优先复用正式 `file-handler:.card` 应用插件；
3. 没有专门处理器时退回宿主窗口承载。

依赖：

1. `PluginRuntime.query`
2. `openPluginWindow(...)`
3. `PAL.window.create(...)`
4. `card-info-service` 的 `status` 检查结果

## 3. 路由接线

### 3.1 `card.readInfo`

路由位于 `card` 服务域，由 `card-info-service` 提供实现。  
Host 对外返回：

1. `status`
2. `metadata`
3. `cover`

### 3.2 `card.open`

路由位于 `card` 服务域，由 `card-open-service` 提供实现。  
Host 保证：

1. 调用前先确认卡片可用；
2. 优先交给正式应用插件处理；
3. 返回结构化窗口结果，而不是把具体窗口实现泄露给调用方。

### 3.3 `box.openEntry`

`box.openEntry` 是箱子查看会话上的正式条目激活动作。  
实现策略：

1. 先根据 `sessionId + entryId` 解析条目 URL；
2. 若为本地 `file://` 卡片，则复用 `card-open-service`；
3. 若为外部 URL，则走 `PAL.shell.openExternal(...)`；
4. 箱子层不自行生成卡片窗口逻辑。

## 4. `BoxService` 内部调整

### 4.1 `readEntryDetail`

旧的 `cardMetadata` 字段被收口为 `cardInfo`。  
箱子层不再自己拼装第二套卡片详情结构，而是直接消费 `card-info-service` 的结果。

### 4.2 `resolveEntryResource`

当条目封面不是箱子内置 `asset` 时，`BoxService` 通过 `card-info-service` 读取正式封面资源，而不是直接散落调用 `CardService.renderCover`。

### 4.3 会话能力位

`BoxSessionInfo.capabilities` 新增：

- `openEntry: true`

用于明确布局插件可以通过正式会话动作触发条目激活。

## 5. 文件关联链路收口

`src/main/core/file-association.ts` 中 `.card` 打开逻辑已切到 `card.open`，避免 CLI、文件关联入口和布局点击各走一套卡片打开实现。

## 6. 设计结论

当前 Host 内部正式分工如下：

1. `CardService` 负责卡片解析、渲染和封面生成；
2. `CardInfoService` 负责标准化卡片信息读取；
3. `CardOpenService` 负责正式卡片打开；
4. `BoxService` 负责箱子会话、箱子资源和条目激活委托；
5. `register-host-services.ts` 负责统一路由编排，不在路由层重复实现卡片信息或卡片打开逻辑。
