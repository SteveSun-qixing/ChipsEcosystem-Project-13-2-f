# 资源打开路由服务设计

## 1. 文档定位

本文档只描述 `Chips-Host` 内部的资源打开路由实现。
对外正式契约以以下共享文档为准：

- `生态共用技术文档/协议与契约/10-资源打开契约.md`
- `生态共用技术文档/协议与接口标准/06-Chips-Host对外接口基线.md`

## 2. 新增内部服务包

### 2.1 `packages/resource-open-service`

职责：

1. 统一承接 `resource.open`；
2. 负责资源处理器应用发现；
3. 负责 `resource-handler` / `file-handler` 能力匹配；
4. 负责目标应用启动参数归一；
5. 在没有应用处理器时执行系统级回退。

依赖：

1. `PluginRuntime.query`
2. `plugin.launch(...)`
3. Host 内部提供的受控资源落点解析函数
4. `PAL.shell.openPath(...)`
5. `PAL.shell.openExternal(...)`

## 3. 路由接线

### 3.1 `resource.open`

路由位于 `resource` 服务域，由 `resource-open-service` 提供实现。

Host 内部保证：

1. 先归一 `intent/resourceId/mimeType/extension/fileName`；
2. 若调用方提供 `resource.payload`，把它视为受控透传字段；
3. 再解析可用应用处理器；
4. 最后才进入系统级回退。

### 3.2 路由优先级

内部匹配顺序固定为：

1. `resource-handler:<intent>:<mime>`
2. `resource-handler:<intent>:<primary>/*`
3. `file-handler:<ext>`

当前只把 `type: app` 且 `enabled = true` 的插件视为正式处理器候选。

### 3.3 启动参数注入

命中应用插件后，Host 统一注入：

1. `trigger: 'resource-open-service'`
2. `resourceOpen.intent`
3. `resourceOpen.resourceId`
4. `resourceOpen.filePath?`
5. `resourceOpen.mimeType?`
6. `resourceOpen.extension?`
7. `resourceOpen.fileName?`
8. `resourceOpen.title?`
9. `resourceOpen.matchedCapability?`
10. `resourceOpen.payload?`

当 Host 已解析出真实本地文件路径时，还会继续注入：

1. `targetPath`

这样可以让既有文件查看器应用继续复用旧的启动恢复逻辑，同时逐步收口到新的 `resourceOpen` 正式结构。

## 4. 卡片渲染链路集成

### 4.1 基础卡片运行时注入

`packages/card-service/src/card-service.ts` 当前会在基础卡片查看态渲染上下文中注入：

```ts
openResource(input)
```

基础卡片插件只负责表达资源打开意图，不感知具体目标应用插件。

### 4.2 iframe 事件转发

Host 在复合卡片壳层内部维护两级转发：

1. `chips.basecard:resource-open`
2. `chips.composite:resource-open`

当前两级事件都会继续保留可选 `payload` 字段，用于把结构化业务上下文从基础卡片 iframe 透传到外层应用，再由外层统一交给 `resource.open`。

这样做的目的：

1. 基础卡片 iframe 与外层应用保持正式边界；
2. 卡片查看器、箱子查看器、未来富媒体容器可以复用同一事件模型；
3. 避免把 `window.chips` 或主进程能力直接暴露给基础卡片 iframe。

## 5. 系统级回退

当没有命中应用处理器时：

1. 如果资源已归一为本地文件路径，则走 `openPath(...)`；
2. 如果资源是 `http/https` URL，则走 `openExternal(...)`；
3. 其它情况返回 `RESOURCE_HANDLER_MISSING`。

## 6. 设计结论

当前 Host 内部正式分工如下：

1. `CardOpenService` 负责“打开整张卡片”；
2. `ResourceOpenService` 负责“打开任意资源”；
3. 卡片内部资源点击不再直连目标应用，而是统一转交 `resource.open`；
4. 处理器解析、启动参数兼容、系统级回退全部收口在 Host 内部，不进入应用插件业务层。
