> 文档状态：需求基线稿（用于从零开发立项与范围冻结）
> 适用阶段：阶段18及后续开发实施阶段
> 对齐基线：生态设计原稿 + 生态共用技术文档（vNext 冻结口径）

---

## 1. 项目背景与目标

薯片主机（Chips Host）是薯片生态唯一宿主程序，承载 PAL、Host Kernel、Host Services、Plugin Runtime 与 Bridge Transport。当前项目进入从零开发阶段，目标是一次性完成底层能力的标准化、契约化和可持续演进。

### 1.1 北极星目标

- 形成可持续 5-10 年演进的主机底座。
- 用一套统一标准支撑 `app/card/layout/module/theme` 五类插件。
- 保证用户侧体验稳定：行为一致、错误可恢复、升级可回滚。
- 降低开发侧复杂度：接口统一、契约统一、测试门禁统一。

### 1.2 本轮开发目标

- 完成 L1-L6 关键链路开发：PAL、Kernel、Services、Runtime、Bridge、Runtime Client。
- 完成 16 个 Host 服务域的动作面落地与契约测试覆盖。
- 完成主机运行治理能力：健康检查、指标、诊断、审计日志。
- 完成“零临时实现”交付：禁止占位实现、禁止 TODO 留存、禁止绕过契约。

## 2. 范围与边界

### 2.1 In Scope（本次范围）

- 主进程底层：PAL、Kernel、Host Services、Plugin Runtime。
- 跨进程通信：Bridge Transport、Runtime Client。
- 主机命令行基础能力：启动/停止/状态/配置/日志/诊断/插件管理。
- 测试与质量体系：契约测试、单测、集成测试、性能门禁、稳定性门禁。

### 2.2 Out of Scope（本次不做）

- L8-L12 页面业务实现（具体应用插件 UI）。
- 社区服务器能力实现（仅保留对接边界）。
- 未在现有技术选型中批准的第三方库引入。

### 2.3 全局信息存放边界

- 面向生态全局、对主机外公开的接口与契约，统一维护在生态共用技术文档。
- 仅主机内部流转、不对外承诺的实现细节，维护在 `Chips-Host/技术文档`。

## 3. 不可违反约束（硬性需求）

### 3.1 架构约束

- 中心路由原则：服务间调用必须经内核路由，禁止跨服务直接调用。
- Bridge 隔离原则：插件仅可通过 `window.chips.*` 访问系统能力。
- 平台无关性原则：平台差异统一收敛到 PAL。
- 契约优先原则：动作必须声明输入/输出 schema、权限、超时、错误码。

### 3.2 工程约束

- 禁止临时版、简化版、占位版交付。
- 禁止删除文件；确需移除时必须迁移到归档目录。
- 禁止修改任务范围外文件。
- 禁止在当前任务中留下 TODO。

### 3.3 质量约束

- 质量门禁不通过不得合并。
- 所有关键路径必须有自动化测试覆盖。
- 所有错误必须使用标准错误对象 `StandardError`。

## 4. 干系人与职责

| 角色 | 关注点 | 交付责任 |
|---|---|---|
| 架构负责人 | 分层边界、契约一致性、长期演进 | 架构决策冻结与变更审批 |
| Host 开发负责人 | 主机底层稳定性与性能 | L1-L6 能力落地 |
| SDK/插件开发者 | 接口稳定性与兼容性 | 接口消费与回归验证 |
| QA | 回归质量、故障恢复、性能门禁 | 测试策略与验收报告 |
| 运维/发布 | 可观测、诊断、发布可回滚 | 发布与运行治理 |

---

## 5. 里程碑级验收目标

- M1：PAL + Kernel 基线可运行，路由/事件/生命周期可观测。
- M2：16 服务域动作面完成，契约测试全绿。
- M3：Bridge + Runtime Client + 权限审计完成，插件链路可闭环。
- M4：质量门禁（测试、性能、鲁棒）达标，版本可发布。
