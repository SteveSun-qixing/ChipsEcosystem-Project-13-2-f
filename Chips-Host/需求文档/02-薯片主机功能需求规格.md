# 薯片主机功能需求规格

> 文档状态：需求规格冻结候选稿
> 说明：本文件定义主机内部功能需求；跨生态公开接口以生态共用文档为准。

---

## 1. 功能域总览

| 功能域 | 对应层级 | 核心目标 |
|---|---|---|
| PAL | L1 | 屏蔽平台差异并提供统一系统能力 |
| Host Kernel | L2 | 提供中心路由、注册表、事件总线、生命周期 |
| Host Services | L3 | 提供 16 服务域统一能力 |
| Plugin Runtime | L4 | 插件加载、会话、权限、隔离、配额治理 |
| Bridge Transport | L5 | 提供 `window.chips.*` 统一入口与 IPC 通道 |
| Runtime Client | L6 | 协议编解码、错误归一、超时与重试 |

---

## 2. PAL 需求（FR-PAL）

### FR-PAL-001 跨平台适配
- 系统应支持 `win32/darwin/linux` 三平台统一行为。
- 内核与服务层禁止直接调用平台特定 API。

### FR-PAL-002 窗口能力
- 提供窗口创建、聚焦、尺寸调整、状态切换、关闭。
- 支持窗口句柄追踪与异常回收。

### FR-PAL-003 文件系统能力
- 提供路径标准化、读写、列举、监听、状态查询。
- 支持跨平台路径分隔符与大小写差异处理。

### FR-PAL-004 系统能力
- 提供对话框、剪贴板、托盘、电源、快捷键、IPC 本地能力。
- 所有能力失败时返回标准错误对象，不抛裸异常。

---

## 3. Kernel 需求（FR-KERNEL）

### FR-KERNEL-001 路由契约
- 每个路由动作必须以 `namespace.action` 命名。
- 每个路由必须注册 `RouteDescriptor`。

### FR-KERNEL-002 路由执行保障
- 必须执行输入/输出 schema 校验。
- 必须执行权限校验与超时控制。
- 必须支持幂等动作的安全重试。

### FR-KERNEL-003 路由鲁棒性
- 高频失败路由需触发软熔断。
- 熔断必须支持半开探测与自动恢复。

### FR-KERNEL-004 生命周期与事件
- 服务生命周期状态必须可观测。
- 事件命名必须采用点语义（如 `theme.changed`）。

---

## 4. Host Services 需求（FR-SVC）

### FR-SVC-001 服务域完整性
- 必须实现 `file/resource/config/theme/i18n/window/plugin/module/platform/log/credential/card/box/zip/serializer/control-plane` 共 16 服务域。

### FR-SVC-002 动作面契约
- 每动作必须声明：`schemaIn/schemaOut/permission/timeoutMs/idempotent/retries`。
- 每动作必须具备同名契约测试 `<action>.contract.test.ts`。

### FR-SVC-003 关键能力
- `card` 服务应提供 parse/render/validate 能力。
- `box` 服务应提供 pack/unpack/inspect 能力。
- `control-plane` 服务应提供 health/check/metrics/diagnose 能力。

### FR-SVC-004 安全与审计
- `credential/shell/window` 高风险动作必须输出独立审计日志。
- 非幂等写操作必须具备 requestId 防重放策略。

---

## 5. Plugin Runtime 需求（FR-RUNTIME）

### FR-RUNTIME-001 插件类型支持
- 必须支持 `app/card/layout/module/theme` 五类插件加载。

### FR-RUNTIME-002 启动握手
- 插件进入 running 前必须完成 `plugin-init` 握手。
- 握手失败不得进入运行态。

### FR-RUNTIME-003 权限与隔离
- 插件只能访问授权能力；越权调用返回 `PERMISSION_DENIED`。
- 插件默认沙箱运行，仅暴露 `window.chips.*`。

### FR-RUNTIME-004 配额治理
- 必须具备 CPU/内存/消息频率软配额能力。
- 超配额需执行限频、降级、告警。

---

## 6. Bridge + Runtime Client 需求（FR-BRIDGE）

### FR-BRIDGE-001 Bridge 入口
- 必须暴露 `invoke/on/once/emit` 与 `window/dialog/plugin/clipboard/shell` 子域。

### FR-BRIDGE-002 IPC 通道
- 主通道固定为 `chips:invoke`。
- 子通道按 `chips:<domain>:*` 命名并可审计。

### FR-BRIDGE-003 Runtime Client 语义
- 统一参数规范化、动作名兼容映射、错误归一。
- 超时默认值、重试次数、退避策略可配置。

### FR-BRIDGE-004 错误模型
- 错误必须归一到 `BRIDGE_* / SERVICE_* / RUNTIME_*`。
- UI 侧仅消费标准错误对象，不暴露底层堆栈细节。

---

## 7. CLI 与运行治理需求（FR-OPS）

### FR-OPS-001 主机命令行能力
- 支持 `start/stop/status/config/logs/plugin/update/doctor` 基础命令。
- 命令应支持结构化输出（JSON）供自动化消费。

### FR-OPS-002 运行诊断
- 提供健康检查、错误码 TOP、延迟分布、权限拒绝统计。
- 诊断输出支持导出用于问题定位。

### FR-OPS-003 可追踪性
- 调用链路必须贯穿 `requestId/traceId`。
- 关键动作必须输出结构化日志。

---

## 8. 需求验收映射

| 需求组 | 验收方式 | 通过标准 |
|---|---|---|
| FR-PAL | 跨平台回归 + 单元测试 | 三平台行为一致，关键能力全绿 |
| FR-KERNEL | 契约测试 + 压测 + 故障注入 | 路由稳定、熔断恢复生效 |
| FR-SVC | 动作契约测试 + 集成测试 | 16 服务域动作面完整 |
| FR-RUNTIME | 权限测试 + 隔离测试 | 无越权，握手失败可阻断 |
| FR-BRIDGE | IPC 回归 + 错误归一测试 | 接口可用，错误语义一致 |
| FR-OPS | 命令行 E2E + 诊断演练 | 管理能力完整可用 |
