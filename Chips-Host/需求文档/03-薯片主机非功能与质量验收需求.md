# 薯片主机非功能与质量验收需求

> 文档状态：质量基线稿
> 说明：用于定义发布门禁，未达标不得合并至 `develop`。

---

## 1. 性能需求（NFR-PERF）

| 指标 | 目标阈值 | 适用范围 |
|---|---|---|
| 路由延迟 p50 | <= 10ms | Kernel 路由 |
| 路由延迟 p95 | <= 50ms | Kernel 路由 |
| 服务启动时间 | <= 500ms | Host Services |
| 服务调用超时率 | <= 0.1% | Host Services |
| Bridge 调用延迟 p95 | <= 100ms | Bridge + Runtime Client |
| 事件投递延迟 | <= 20ms | Bridge Transport |
| 重试成功率 | >= 95% | Runtime Client |

### NFR-PERF-001
所有关键性能指标必须纳入自动化采样与趋势对比，禁止仅凭人工感知判定性能通过。

---

## 2. 可靠性与鲁棒性需求（NFR-REL）

### NFR-REL-001 失败隔离
- 单服务失败不得导致主进程崩溃。
- 插件失败不得扩散至其他插件会话。

### NFR-REL-002 回退策略
- 主题解析失败自动回退默认主题。
- Bridge 超时返回标准可重试错误。
- 插件升级失败支持回滚上个稳定版本。

### NFR-REL-003 熔断恢复
- 路由连续失败触发熔断。
- 半开探测成功后自动恢复闭合状态。

---

## 3. 安全需求（NFR-SEC）

### NFR-SEC-001 权限最小化
- 插件权限必须 manifest 显式声明并在入站执行校验。
- 未声明权限调用一律拒绝。

### NFR-SEC-002 敏感能力隔离
- `credential`、`shell.openExternal`、`window` 高风险动作必须审计。
- 审计日志至少记录：时间、插件ID、动作、结果、错误码。

### NFR-SEC-003 数据安全
- 凭证不得以明文持久化到业务配置。
- 写操作必须支持 requestId 去重窗口，防重放。

---

## 4. 可维护性需求（NFR-MNT）

### NFR-MNT-001 契约优先
- 所有服务动作必须先定义契约后实现。
- 禁止“仅实现 handler 不注册契约”的代码形态。

### NFR-MNT-002 分层治理
- 仅允许 `Ln -> Ln-1` 依赖。
- 禁止 L12 直连 L1-L5 等跨层跳跃。

### NFR-MNT-003 一致性
- 错误对象、日志字段、动作命名、事件命名必须全栈统一。

---

## 5. 可观测性需求（NFR-OBS）

### NFR-OBS-001 日志标准字段
- 统一字段：`traceId/requestId/pluginId/namespace/action/durationMs/result/errorCode`。

### NFR-OBS-002 指标与诊断
- 必须提供路由性能、权限拒绝、插件状态、主题解析失败率指标。
- 必须提供可导出的诊断报告（用于问题工单与回归复盘）。

### NFR-OBS-003 链路追踪
- requestId 必须贯穿 Transport -> Runtime Client -> Kernel -> Service。

---

## 6. 测试与发布门禁（NFR-TEST）

### NFR-TEST-001 测试层级
- 必须覆盖契约测试、单元测试、集成测试、E2E、性能测试、鲁棒性测试。

### NFR-TEST-002 最低覆盖
- 服务模块每模块 >= 8 个单元测试用例。
- 契约测试通过率必须为 100%。

### NFR-TEST-003 发布阻断条件
以下任一项失败即阻断发布：
- 契约漂移。
- 关键链路测试失败。
- 性能预算超阈值。
- 高危安全测试失败。

---

## 7. Git 与交付规范（NFR-GIT）

### NFR-GIT-001 分支策略
- 所有开发工作在任务分支进行；完成后通过审查合并至 `develop`。
- 本仓当前任务分支命名遵循 `codex/*` 前缀。

### NFR-GIT-002 提交流程
- 先过本地质量门禁，再提交。
- 单次提交应保持语义单一、可回溯。

### NFR-GIT-003 变更边界
- 不得修改任务范围外文件。
- 不得使用破坏性命令回滚非本任务改动。
