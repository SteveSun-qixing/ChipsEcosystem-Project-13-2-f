# 颜色选取模块技术方案

## 1. 文档目的

本文档用于说明 `Chips-ColorPicker` 模块的当前正式技术实现，覆盖以下内容：

- 模块在生态中的定位与职责边界；
- `manifest.yaml`、能力契约与构建产物的真实口径；
- 核心源码文件职责划分；
- 图片采样、颜色分析、结果调优与缓存策略；
- 性能表现、测试验证、当前边界与接入建议。

本文档只记录当前仓库内部实现细节。对生态外公开的正式能力口径，以共享契约文档 `生态共用技术文档/协议与契约/12-图像颜色提取模块能力契约.md` 为准。

## 2. 模块概览

### 2.1 模块定位

`Chips-ColorPicker` 是一个 `type: module` 的无界面模块插件，用于接收单张本地图片并返回两个代表色：

- `backgroundColor`
  - 适合大面积铺底、纯色背景、卡片底色；
  - 颜色浓郁、稳重，但不会被压成过暗脏色。
- `accentColor`
  - 适合按钮、强调点、个性化视觉元素；
  - 颜色更亮、更高饱和、更显眼。

模块的目标不是返回平均色，而是尽量逼近成品设计里真正会被采用的“背景主色 + 强调色”组合。

### 2.2 当前正式元信息

基于当前 `manifest.yaml` 与实际构建链路，模块的正式元信息如下：

| 项目 | 当前值 |
| --- | --- |
| 插件 ID | `chips.module.chips.colorpicker` |
| 名称 | `Chips ColorPicker` |
| 类型 | `module` |
| 入口 | `dist/index.mjs` |
| 运行时 | `worker` |
| 激活方式 | `onDemand` |
| 权限 | `file.read` |
| capability | `image.color.pick` |
| method | `pick` |
| mode | `sync` |
| 输入契约 | `contracts/pick.input.schema.json` |
| 输出契约 | `contracts/pick.output.schema.json` |
| 运行目标 | `desktop: true`、`headless: true`、`web/mobile: false` |

### 2.3 构建策略

该模块当前不使用默认的浏览器向 Vite lib 构建作为正式产物，而是使用自定义 `esbuild` Node 构建脚本：

- 构建脚本：`scripts/build.mjs`
- 输出：`dist/index.mjs`
- `platform: "node"`
- `format: "esm"`
- `target: "node22"`
- `packages: "external"`

这样做的原因是模块依赖 `sharp`，其运行时语义属于 Node 侧图像解码库，不能按浏览器 bundle 的方式收口。

## 3. 源码结构与职责划分

当前模块的核心文件如下：

| 文件 | 职责 |
| --- | --- |
| `src/index.ts` | 模块入口，导出 `image.color.pick` provider 与 `pick` method |
| `src/picker.ts` | 调用编排层，负责采样、分析、进度上报、结果缓存与日志 |
| `src/image-sampler.ts` | 输入路径规范化、`file.stat` / `file.read` 调用、`sharp` 缩采样、采样缓存 |
| `src/color-analysis.ts` | OKLab / OKLCH 颜色空间转换、像素聚合、聚类、评分、最终颜色调优 |
| `src/binary.ts` | Host `file.read` 二进制返回值归一化 |
| `src/errors.ts` | 模块内统一错误码与错误对象封装 |
| `src/types.ts` | 输入输出类型、Host 上下文类型、文件元信息类型 |
| `contracts/*.json` | method 输入输出 JSON Schema |
| `tests/unit/*.test.ts` | 单元测试，覆盖结果形态、亮度区间、缓存行为、错误路径 |

## 4. 正式调用链路

### 4.1 运行链路

```text
调用方
  -> module.invoke(image.color.pick / pick)
    -> src/index.ts
      -> src/picker.ts
        -> src/image-sampler.ts
          -> ctx.host.invoke("file.stat")
          -> ctx.host.invoke("file.read")
          -> sharp 解码 + 缩采样
        -> src/color-analysis.ts
          -> 像素聚合
          -> 聚类
          -> 背景色评分
          -> 个性色评分
          -> OKLCH 后处理
        -> 返回 { backgroundColor, accentColor }
```

### 4.2 Host 交互边界

模块当前只通过 Host 正式上下文调用以下动作：

- `file.stat`
  - 校验输入文件是否存在、是否为文件；
  - 同时获取 `size` 与 `mtimeMs`，用于缓存身份。
- `file.read`
  - 读取原图二进制内容；
  - 读取选项使用 `encoding: "binary"`。

模块没有自行引入 Host 内部实现，也没有额外创建第二套服务包装。

### 4.3 调用模式

当前 method `pick` 为同步返回模式：

- 不创建 job；
- 不进行异步轮询；
- 直接返回：

```ts
{
  backgroundColor: "#rrggbb",
  accentColor: "#rrggbb"
}
```

## 5. 输入输出与约束

### 5.1 输入

当前输入类型：

```ts
interface ColorPickRequest {
  imagePath: string;
  options?: {
    sampleSize?: number;
  };
}
```

约束：

- `imagePath` 只接受本地绝对路径或 `file://` 本地 URL；
- 远程 URL 会直接报错；
- `sampleSize` 会被夹紧到 `48-160`；
- 默认 `sampleSize = 96`。

### 5.2 输出

当前输出类型：

```ts
interface ColorPickResult {
  backgroundColor: string;
  accentColor: string;
}
```

约束：

- 两个字段都必须是 `#rrggbb`；
- 当前只返回两个颜色，不返回完整调色板；
- 不附带文字前景色、对比度评分或语义命名。

## 6. 采样与缓存设计

### 6.1 路径与输入规范化

`src/image-sampler.ts` 首先对 `imagePath` 做规范化：

- 去除首尾空白；
- 若为 `file://`，转换为本地路径；
- 若为其他协议 URL，直接报 `COLOR_PICKER_INPUT_INVALID`；
- 普通路径按本地绝对路径解析。

### 6.2 文件存在性校验

采样前先调用 `file.stat`：

- 若 Host 调用失败或不存在文件，报 `COLOR_PICKER_INPUT_NOT_FOUND`；
- 若 `meta.isFile !== true`，同样视为无效输入；
- 返回的 `size` 与 `mtimeMs` 会参与采样缓存键构建。

### 6.3 采样缓存

模块内维护一个进程级采样缓存：

- 容量上限：`24`
- 缓存键：

```text
${imagePath}::${sampleSize}::${size}::${mtimeMs}
```

语义：

- 同一路径、同一缩采样尺寸、且文件大小和修改时间都未变化时，直接复用采样结果；
- 这是一个轻量 LRU 风格缓存，超过上限后淘汰最早写入项；
- 缓存只在模块当前进程内有效，跨进程不共享。

### 6.4 图像缩采样

缩采样使用 `sharp`，当前参数如下：

- `animated: false`
- `failOn: "error"`
- `ensureAlpha()`
- `resize(sampleSize, sampleSize, { fit: "contain" })`
- 背景为透明像素
- `kernel: sharp.kernel.bilinear`
- `fastShrinkOnLoad: true`
- `withoutEnlargement: true`
- 输出为 `raw()` RGBA 像素

设计意图：

- `contain + 透明背景` 可以保留原图完整构图；
- `ensureAlpha` 保证后续像素处理统一按 RGBA 进行；
- `bilinear + fastShrinkOnLoad` 比更重的高质量核更快，适合本模块“以代表色分析为主、不是做最终显示”的目标；
- 对于颜色选取任务，缩采样保留整体色彩关系比追求锐利边缘更重要。

## 7. 颜色分析算法

### 7.1 总体原则

当前算法由以下阶段组成：

1. 过滤不可见像素；
2. 对剩余像素做量化聚合；
3. 在 OKLab 空间进行代表色聚类；
4. 分别为背景色和个性色打分；
5. 在 OKLCH 空间做最终亮度 / 色度调优；
6. 若背景色和个性色分离度不够，再做一次强调色拉开处理。

算法核心不是“找最大面积颜色”或“找最亮颜色”，而是综合代表性、面积、空间位置、亮度、色度和相互差异做平衡。

### 7.2 像素过滤与量化

在 `collectQuantizedPoints()` 中：

- 忽略 `alpha < 0.08` 的像素；
- 对 RGB 每通道右移 4 位，形成 4bit 量化桶；
- 每个桶累计：
  - RGB 加权和；
  - 像素权重；
  - 中心权重；
  - 边缘权重。

额外位置权重：

- 中心权重：
  - 距离图像中心越近，值越高；
  - 用于强调画面主体附近颜色。
- 边缘权重：
  - 靠近四周边缘的像素记为边缘；
  - 用于识别更适合铺底的大面积边缘色。

### 7.3 颜色空间

模块使用 OKLab / OKLCH 作为核心分析空间，原因如下：

- 比直接使用 RGB 更符合感知均匀性；
- 更适合比较颜色差异；
- 更适合对亮度与色度做独立控制；
- 更适合“保持图片气质，同时把颜色推到目标区间”这一后处理策略。

当前实现提供：

- `rgbToOklab`
- `oklabToRgb`
- `oklabToOklch`
- `fitOklchToRgb`

其中 `fitOklchToRgb()` 会在超出 RGB gamut 时回退到可显示范围内的最近结果。

### 7.4 聚类策略

聚类并非固定 K 值，而是根据点数量动态选择：

- 点数 `<= 1`，取 1 类；
- 点数 `<= 4`，取 `2 ~ 4` 类；
- 否则按 `sqrt(points.length / 3)` 估算；
- 最终限制在 `3 ~ 8` 类之间。

初始质心使用带色度偏好的最远点初始化策略：

- 第一类优先选择 `weight * (1 + chroma)` 最大的点；
- 后续类选择与现有质心距离最远、且权重足够高的点；
- 再进行最多 `12` 轮重新分配和质心更新。

这比随机初始化更稳定，更适合模块化能力场景下对结果一致性的要求。

## 8. 背景色与个性色选择逻辑

### 8.1 背景色评分方向

背景色的候选选择更重视：

- 面积占比；
- 整体代表性；
- 一定的色彩浓度；
- 更适合铺底的亮度区间；
- 来自边缘的大面积背景信号。

当前评分倾向：

- `share * 4.2`
- `normalizeChroma(cluster.chroma) * 2`
- `closeness(cluster.lightness, 0.54, 0.2) * 2.7`
- `edgeRatio * 0.45`

当前惩罚项：

- 面积太小；
- 亮度过低；
- 亮度过高。

这意味着背景色虽然仍然偏稳重，但不会像早期版本那样被压到近黑区域。

### 8.2 个性色评分方向

个性色候选更重视：

- 色度；
- 亮度；
- 与背景色的差异；
- 适度面积；
- 主体附近颜色。

当前评分倾向：

- `normalizeChroma(cluster.chroma) * 3.4`
- `closeness(cluster.lightness, 0.58, 0.2) * 2.1`
- `Math.min(share, 0.2) * 1.5`
- `centerRatio * 0.35`
- `Math.min(separation / 0.16, 1) * 1.1`

当前惩罚项：

- 面积极小；
- 亮度过低；
- 亮度过高；
- 与背景色差异不足。

### 8.3 最终后处理区间

选出候选 cluster 后，不直接返回原始 cluster 色，而是继续把颜色推入更稳定的使用区间。

#### 背景色后处理

非中性色时：

- 亮度 `L` 收敛到 `0.48 - 0.62`
- 色度 `C` 收敛到 `0.11 - 0.20`

中性色时：

- 亮度 `L` 收敛到 `0.48 - 0.62`
- 色度 `C` 限制在更低区间，避免发脏发彩

#### 个性色后处理

非中性色时：

- 亮度 `L` 收敛到 `0.72 - 0.84`
- 色度 `C` 收敛到 `0.18 - 0.30`

中性色时：

- 亮度继续拉高到强调区间；
- 色度仅做轻微提升，避免假彩色失真。

### 8.4 分离度保障

若最终背景色与个性色的 OKLab 距离低于 `0.12`，模块会再次提升个性色：

- 非中性色时继续提亮、增艳；
- 中性色时优先拉亮，少量拉色度；
- 最终仍通过 `fitOklchToRgb()` 保证结果落在可显示 RGB 范围内。

这样可以降低两色过于接近导致“按钮埋进背景里”的情况。

## 9. 结果缓存与性能策略

### 9.1 结果缓存

模块内还维护一个结果缓存：

- 容量上限：`48`
- 缓存键与采样缓存一致；
- 命中后直接返回 `ColorPickResult`；
- 同样采用轻量 LRU 风格淘汰。

这意味着对于同一张未变化图片：

- 第一次调用会经历 `stat -> read -> decode -> sample -> analyze`；
- 后续调用只需 `stat` 校验文件身份后即可直接复用结果。

### 9.2 当前实测表现

基于当前本地开发环境与测试图片的实际测量：

- 模块本体首次调用约 `20ms ~ 30ms`
- 同图重复调用命中缓存后约 `< 1ms`

当前一次内部实测值为：

- 首次模块侧处理约 `23ms`
- 第二次同图调用约 `0.07ms`

### 9.3 关于 `chipsdev module invoke` 的耗时说明

开发联调命令 `chipsdev module invoke` 的总耗时明显高于模块本体耗时，这是因为该命令包含：

- 工程构建；
- 拉起真实 Electron Host；
- 开发工作区插件安装启用；
- Host 内 capability 调用；
- 进程退出。

因此：

- `chipsdev module invoke` 的秒级耗时，不代表模块算法本身慢；
- 真正集成到长驻 Host 进程后，模块本体的实际调用耗时远低于开发命令总耗时。

## 10. 错误语义

当前模块内部使用统一错误封装 `createColorPickerError()`，主要错误码如下：

| 错误码 | 触发场景 |
| --- | --- |
| `COLOR_PICKER_INPUT_INVALID` | `imagePath` 为空或使用了不支持的 URL 协议 |
| `COLOR_PICKER_INPUT_NOT_FOUND` | 文件不存在、不是文件或 `file.stat` 失败 |
| `COLOR_PICKER_IMAGE_EMPTY` | 采样后没有可见像素，或无有效颜色信息 |
| `COLOR_PICKER_IMAGE_SAMPLE_FAILED` | `file.read` / `sharp` 解码采样失败 |
| `COLOR_PICKER_IMAGE_SAMPLE_INVALID` | `sharp` 返回的采样尺寸或通道异常 |
| `COLOR_PICKER_IMAGE_SAMPLE_UNSUPPORTED` | 预留错误码，供后续图片格式边界使用 |
| `COLOR_PICKER_PNG_INVALID` | 历史 PNG 草稿链路相关错误码，当前实现未作为主路径使用 |
| `COLOR_PICKER_ANALYSIS_FAILED` | 聚类或颜色分析阶段失败 |

## 12. 当前边界

当前版本明确不做以下能力：

- 网络图片 URL 读取；
- 多颜色调色板输出；
- 自动前景文字颜色推荐；
- 对比度评级；
- 语义色名生成；
- 跨进程持久缓存；
- 调用方样式 token 自动映射。

这些都应作为后续新增能力扩展，而不是直接改变当前 `pick` 输出结构。

## 13. 接入建议

调用方当前建议如下：

- `backgroundColor`
  - 直接用于卡片背景、封面背景、标题区大面积底色；
  - 不建议再进行大幅度整体压暗，否则容易回到早期“过黑”问题。
- `accentColor`
  - 适合按钮、图标高亮、进度条高亮、个性化强调点；
  - 如需 hover / active 态，建议围绕当前色做有限度亮暗变化，而不是重新选色。

若调用方还需要文字颜色，应独立做对比度判断，不要把该模块的两个返回色直接当成前景文字色使用。

## 14. 后续可扩展方向

后续如需继续演进，优先建议新增而不是改坏现有口径：

- 增加 `palette` 输出，返回更多候选色；
- 增加 `recommendedForeground`；
- 增加对比度评级；
- 增加“背景色偏冷 / 个性色偏暖”等风格偏好参数；
- 增加跨进程缓存或持久化缓存；
- 增加批量图片取色能力。

在扩展前，需要继续保持：

- 当前 `image.color.pick/pick` 契约不破坏；
- 当前 `backgroundColor` 与 `accentColor` 语义不漂移；
- 当前长驻 Host 场景下的快速返回能力不退化。
