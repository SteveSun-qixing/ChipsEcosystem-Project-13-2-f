# 文件转换编排模块技术方案

## 1. 当前仓库基线

本仓已由 `chips-scaffold-module` 初始化，具备：

- `manifest.yaml`
- `contracts/`
- `src/`
- `tests/`

当前代码仍是脚手架示例能力，后续要替换为正式文件转换编排能力。

## 2. 目标结构

本仓内部实现建议分为：

- `request-normalizer`
- `pipeline-planner`
- `step-executor`
- `job-progress-aggregator`
- `result-normalizer`

## 3. 流水线规划规则

- `card -> html`：单步
- `card -> pdf`：两步
- `card -> image`：两步
- `html -> pdf/image`：单步

流水线规划本身不依赖具体 provider 仓库，只依赖正式 capability 名称。

## 4. 原子模块调用方式

编排模块调用原子模块时，必须使用 Host 注入的 `ctx.module.invoke(...)`：

- 不直接 import 原子模块代码
- 不复制原子模块 schema
- 不复写原子模块内部输出逻辑

## 5. 中间产物治理

`pdf/image` 链路中的 HTML 中间产物必须被视为临时目录：

- 由编排模块统一创建
- 由编排模块统一传递给下游步骤
- 成功或失败后统一清理

## 6. 测试重点

- 流水线矩阵测试
- 原子模块失败传播测试
- job 取消测试
- 中间目录清理测试
- 结果归一测试
