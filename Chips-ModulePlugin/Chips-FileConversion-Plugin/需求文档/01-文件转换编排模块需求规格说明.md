# 文件转换编排模块需求规格说明

## 1. 模块定位

`Chips-FileConversion-Plugin` 是整个转换系统的对外统一入口。调用方只与它交互，不直接依赖 `card -> html`、`html -> pdf`、`html -> image` 原子模块。

## 2. 功能目标

本模块必须完成以下职责：

- 对应用提供统一转换入口
- 根据源类型与目标类型规划流水线
- 调用原子模块完成实际转换
- 汇总 job 进度、错误和结果
- 清理中间产物

## 3. 功能需求

- `FR-01` 支持 `card -> html`、`card -> pdf`、`card -> image`
- `FR-02` 支持 `html -> pdf`、`html -> image`
- `FR-03` `pdf/image` 导出必须复用 `card -> html(directory)` 中间产物
- `FR-04` 统一输出 artifact、pipeline、warning、outputPath
- `FR-05` 统一支持 job 查询与取消
- `FR-06` 流水线失败时必须返回结构化错误并清理中间目录

## 4. 非功能需求

- `NFR-01` 不新增新的 Host 服务域
- `NFR-02` 不在调用方暴露原子模块内部细节
- `NFR-03` 进度阶段必须可追踪
- `NFR-04` 流水线规划必须可测试
- `NFR-05` 模块替换或多 provider 并存时，调用方入口不变

## 5. 当前仓库基线要求

本仓当前已完成模块脚手架初始化，但 `src/`、`contracts/`、`tests/` 仍为脚手架示例基线。后续实现时必须用本需求文档定义的正式能力替换示例能力，不得把脚手架示例直接作为正式交付。

## 6. 验收口径

本仓验收完成时必须满足：

- 调用方能够用一个模块入口完成 3 类卡片导出
- 统一 job 进度和取消生效
- 原子模块错误能被编排模块归一化返回
- 与公共契约文档完全一致
