# 启动恢复与音频资源解析设计

## 1. 正式输入来源

音乐播放器当前接受五类正式输入：

1. `launchParams.resourceOpen.filePath`
2. `launchParams.targetPath`
3. `launchParams.resourceOpen.resourceId`
4. `launchParams.resourceOpen.payload` 中的音乐基础卡片播放上下文
5. 文件选择器 / 拖放得到的一组本地文件路径

其中第五类允许同时出现：

1. 一首音频；
2. 一张封面；
3. 一个歌词文件。

## 2. 启动目标归一

实现位置：

- `src/utils/launch-resource.ts`
- `src/App.tsx`

归一顺序：

1. `resourceOpen.filePath`
2. `targetPath`
3. `resourceOpen.resourceId`

输出结构：

```ts
interface LaunchAudioTarget {
  sourceId: string;
  filePath?: string;
  fileName?: string;
  mimeType?: string;
  title?: string;
  musicCard?: MusicCardOpenPayload;
}
```

补充：

1. 当 `resourceOpen.payload` 符合正式 `MusicCardOpenPayload` 结构时，播放器会把它挂到 `LaunchAudioTarget.musicCard`；
2. 该 payload 保存音乐基础卡片配置，以及已解析好的音频 / 封面 / 歌词资源描述符；
3. 这样音乐播放器在“从卡片查看器打开音乐基础卡片”场景下，不再只拿到一条音频路径，而是能恢复完整音乐卡片上下文。

## 3. 播放 URI 解析

实现位置：

- `src/App.tsx`

解析策略：

1. 若目标本身已是 `file://`、`http://`、`https://`、`blob:` 或 `data:`，直接作为可播放 URI；
2. 否则调用正式 `client.resource.resolve(...)`；
3. 解析完成后才进入音乐播放会话。

这样做的原因：

1. 本地文件路径必须通过正式资源解析转为可消费 URI；
2. 远端 URL 不应被错误当作本地路径二次包装；
3. 上层应用无需关心 Host 路由内部资源格式，只需要交给音乐播放器恢复会话。

## 4. 嵌入元数据读取目标

实现位置：

- `src/App.tsx`
- `src/utils/audio-metadata.ts`

读取策略：

1. 若 launch context 能解析出稳定本地路径，优先读取该本地路径；
2. 若没有本地路径，但 `resourceOpen.resourceId` 不是可直接播放的 `http/https/blob/data` URI，则继续通过正式 `client.resource.readBinary(...)` 读取该资源；
3. `client.resource.readBinary(...)` 对播放器侧始终按“裸字节”消费；SDK 已在内部统一解包 Host / Bridge 可能返回的 `Buffer` 包装对象与桥接序列化后的 Buffer JSON；
4. 只有在来源是纯外部直链且没有 Host 可读资源标识时，才跳过嵌入元数据解析。

这样做的原因：

1. Host 通过 `resource.open` 唤起播放器时，不一定总能同时提供本地文件路径；
2. 资源 ID 与本地路径都必须走正式资源读取能力，而不是由应用自己直连文件系统；
3. 嵌入封面、歌词、歌曲信息应在“本地打开”和“Host 路由唤起”两条链路下保持一致。
4. 业务层不需要也不允许自行兼容 `readBinary().data` 之类的错误共享形状。

## 5. 本地伴生资源发现

实现位置：

- `src/App.tsx`
- `src/utils/music-player.ts`
- `src/utils/binary.ts`

当音频存在稳定本地路径且当前不是从音乐基础卡片 payload 恢复时，播放器会先查看用户显式选择的补充文件；若没有，再到同目录发现伴生资源。

封面候选名：

1. 与音频同名的 `.jpg/.jpeg/.png/.webp/.avif`
2. `cover.*`
3. `folder.*`
4. `front.*`

歌词候选名：

1. 与音频同名的 `.lrc`
2. `lyrics.lrc`
3. 与音频同名的 `.txt`

伴生歌词读取补充：

1. 伴生歌词统一通过 `client.file.read(path, { encoding: "binary" })` 读取；
2. Host 当前 `file.read` 返回形态为 `{ content }`，应用侧需要先解包 `content`；
3. 若解包后的结果是 `Uint8Array`，则直接进入 `decodeTextWithFallback(...)`；
4. 若解包后的结果是字符串，则先按生态现有链路还原为 `base64` 二进制内容，再做编码探测；
5. 这样可以避免 GBK / Big5 / Shift_JIS 等歌词文件被错误按 UTF-8 二次编码，导致乱码或解析失败。

## 6. 嵌入元数据解析

实现位置：

- `src/utils/audio-metadata.ts`

当前正式支持：

1. MP3：
   - `ID3v2.2 / v2.3 / v2.4`
   - 标题：`TIT2 / TT2`
   - 歌手：`TPE1 / TP1`，缺失时回退 `TPE2 / TP2`
   - 专辑：`TALB / TAL`
   - 歌词：`USLT / ULT`、`SYLT / SLT`
   - 歌词扩展：`COMM / COM`、`TXXX / TXX` 中描述为 `LYRICS / UNSYNCEDLYRICS / UNSYNCED LYRICS / LYRIC` 的内容
   - 封面：`APIC / PIC`
   - 当 `ID3v2` 缺失时，继续回退尾部 `ID3v1`
2. FLAC：
   - Vorbis Comment 中的 `TITLE / ARTIST / ALBUM / LYRICS / UNSYNCEDLYRICS / SYNCEDLYRICS`
   - `PICTURE` block
3. MP4 / M4A / AAC 容器：
   - `moov.udta.meta.ilst`
   - 标题：`©nam`
   - 歌手：`©ART / aART`
   - 专辑：`©alb`
   - 歌词：`©lyr`
   - 封面：`covr`
   - Freeform 条目 `----` 中 `name = LYRICS / UNSYNCEDLYRICS / UNSYNCED LYRICS / LYRIC`
4. Ogg Vorbis / Opus：
   - `vorbis comment packet`
   - `OpusTags`
   - 标题 / 歌手 / 专辑：`TITLE / ARTIST / ALBUM`
   - 歌词：`LYRICS / UNSYNCEDLYRICS / SYNCEDLYRICS / TIMEDLYRICS`
   - 封面：`METADATA_BLOCK_PICTURE`、`COVERART + COVERARTMIME`

补充策略：

1. 当歌词内容本身已是 LRC 时间轴格式时，即使标签名是普通 `LYRICS`，也会自动提升为时间轴歌词；
2. 封面图片 MIME 不盲信标签显式声明，而是优先根据图片字节魔数识别真实格式；当标签声明与真实字节不一致时，以真实字节格式为准；
3. 当前运行时若遇到浏览器不稳定直显的嵌入封面格式，会优先转成 `PNG` 再交给界面层显示：
   - `TIFF` 优先走应用内基线解码链路，当前正式支持 `baseline / uncompressed / RGB / chunky / 8-bit`；
   - 其他格式如 `HEIC / HEIF` 继续优先尝试 `ImageDecoder` 或 `createImageBitmap`；
4. 统一输出结构保持为 `EmbeddedAudioMetadata`，供应用层按优先级合并。

嵌入封面落盘策略：

1. 当存在 `launchParams.workspacePath` 且解析到嵌入封面时，播放器优先把封面归一为 Host 工作区临时 PNG；
2. 缓存目录固定为 `<workspacePath>/.music-player-artwork-cache`；
3. 若嵌入封面本身就是 PNG，则直接写入缓存目录；
4. 若嵌入封面是 TIFF，则先把原始字节写入临时 `.tiff` 文件，再通过正式 `client.resource.convertTiffToPng(...)` 产出 PNG，最后删除临时 TIFF；
5. 若是其他浏览器直显不稳定格式，则继续走运行时转码为 PNG 字节后落盘；
6. 创建缓存目录时必须把“目录已存在”的并发场景视为成功，不得因为 `EEXIST` 中断整条打开链路；
7. 只有当 Host 工作区临时 PNG 落盘失败或当前上下文没有工作区路径时，才回退为运行时 `blob:` URL。

优先级策略：

1. 标题：`launch title > musicCard.display.title > embedded title > 歌词元数据 title > 文件名`
2. 歌词：`musicCard.resources.lyrics > 显式歌词文件 > 自动发现歌词文件 > 嵌入歌词`
3. 封面：`musicCard.resources.cover / 显式或自动发现封面 > 嵌入封面 > 默认封面`

## 7. 支持格式判断

实现位置：

- `src/utils/music-player.ts`

判断规则：

1. `mimeType` 以 `audio/` 开头时优先判为支持；
2. 否则按本地路径或资源 ID 后缀匹配扩展名白名单；
3. 未命中白名单则拒绝打开并给出反馈。

## 8. 保存链路

实现位置：

- `src/App.tsx`

保存策略：

1. 本地文件路径存在时，使用 `platform.saveFile + file.copy`；
2. 远端或无稳定本地路径时，回退为下载链接；
3. 覆盖同路径时显式拒绝。
