# 音乐播放器非功能与验收需求

## 1. 架构与工程要求

1. 应用必须基于官方应用脚手架初始化；
2. 外层仍然是标准 `type: app` 应用插件；
3. 系统能力调用只允许通过 `chips-sdk` / `window.chips.*`；
4. 所有界面文案必须接入多语言词典；
5. 所有视觉样式必须走主题系统与 CSS token。

## 2. 质量要求

1. 不得保留模板身份泄露、示例占位、TODO 或临时兼容分支；
2. 运行时必须对启动参数、资源解析失败、歌词读取失败、媒体解码失败给出稳定反馈；
3. 封面区、歌词区和控制区在缩放窗口时应保持布局稳定；
4. 移动端纵向比例下的双页面滑动切换必须平稳，且不能破坏桌面端左右布局；
5. 小屏场景不得退回封面区和歌词区的上下堆叠布局；
6. 文档必须与真实 `manifest.yaml`、测试和正式脚本一致；
7. 资源来源说明必须写清复用的开源素材与许可边界。

## 3. 文档要求

本项目必须至少维护以下文档体系：

1. `需求文档/`
2. `技术文档/`
3. `开发计划/`
4. `README.md`

## 4. 验收门禁

首版验收至少包括：

1. `resource.open` 能正确拉起音乐播放器；
2. 系统文件关联能把受支持的音频扩展名交给音乐播放器；
3. 应用内手动打开、拖放打开、本地副本保存可正常工作；
4. 播放、暂停、时间轴、音量、静音、循环行为符合预期；
5. 左封面右歌词界面在有歌词与无歌词场景下都保持稳定；
6. 移动端纵向比例下可以左右滑动切换“播放器页 / 歌词页”，顶部小圆点能同步反映当前页面；
7. 移动端分页在触摸屏、触摸板和鼠标拖拽场景下都能完成页面切换；
8. 自动发现伴生封面 / 歌词、解析 MP3 / FLAC 嵌入元数据以及移动端布局判定 / 翻页阈值逻辑具备单元测试覆盖；
9. `npm run lint`
10. `npm test`
11. `npm run build`
12. `npm run validate`

## 5. 风险与边界

当前可接受但必须显式标注的边界：

1. 实际可播放的编解码能力受当前 Host / 浏览器运行时限制；
2. 远端资源的“保存”是下载语义，不保证与本地复制完全等价；
3. MP3 / FLAC 之外的嵌入元数据当前不做正式保证；
4. 参考仓库只作为设计与体验参考，不改变本项目对外契约。
