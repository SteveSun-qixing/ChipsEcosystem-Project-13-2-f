# 图片查看器技术文档

## 1. 文档边界

本文只描述 `Chips-PhotoViewer` 本仓内部实现与当前已落地的 Host / SDK 接线事实。涉及未来生态通用资源票据、默认处理器治理等公共契约，不在本文件中重新定义。

## 2. 当前正式外部链路

### 2.1 Host 资源打开入口

当前 Host 已新增正式 `resource.open` 路由，图片查看器可以通过两类能力参与该路由：

1. `resource-handler:view:image/*`
2. `file-handler:.png/.jpg/...`

Host 当前处理顺序如下：

1. 优先匹配 `resource-handler:view:image/*` 这一类 MIME 处理器；
2. 若未命中，再回退到 `file-handler:<ext>`；
3. 命中后统一通过 `plugin.launch(...)` 启动图片查看器；
4. 启动上下文中注入 `launchParams.resourceOpen`，并在可解析真实文件路径时继续补充 `targetPath`。

这意味着：

1. 卡片查看器、箱子查看器、未来的媒体容器都不需要硬编码图片查看器插件 ID；
2. 只要调用正式 `resource.open`，Host 就会自动寻找图片处理器。

### 2.2 Host 文件关联入口

当前 Host 已经完成两层接线：

1. `Chips-Host/src/main/electron/app-entry.ts`
   - 进程参数、macOS `open-file`、`second-instance` 现在会接收任意带扩展名的文件路径；
2. `Chips-Host/src/main/core/file-association.ts`
   - `.card` 继续走卡片链路；
   - `.box` 继续走箱子链路；
   - 其它扩展名会查询 `plugin.query({ type: "app", capability: "file-handler:<ext>" })`；
   - 命中后调用 `plugin.launch(...)`，把 `targetPath` 交给目标应用插件。

因此，图片查看器只要声明 `file-handler:.png/.jpg/...`，就能通过正式 Host 文件关联链路接收图片文件。

### 2.3 安装器条件注册

`Chips-Host/src/main/installer/macos-installer.ts` 现在会：

1. 扫描内置 app 插件 manifest 中的 `file-handler:<ext>`；
2. 推导要写入 `Info.plist` 的文档类型；
3. 只为真正随包携带的内置插件注册对应文件类型。

这意味着：

1. 标准 Host 发行包如果携带 `Chips-PhotoViewer`，会注册图片类型；
2. 如果发行包不携带图片查看器，安装器不会注册这些图片类型；
3. Host 运行时也会把缺失的图片查看器视为可选内置插件，不会因为缺 bundle 而启动失败。

### 2.4 SDK / Bridge 能力

图片查看器当前使用的正式能力：

1. `client.platform.getLaunchContext()`：读取 `trigger`、`targetPath` 与 `launchParams.resourceOpen`
2. `launchParams.resourceOpen`：读取由 Host 资源路由注入的资源上下文
3. `client.platform.openFile(...)`：手动选择图片
4. `client.platform.saveFile(...)`：选择保存位置
5. `client.file.copy(...)`：保存图片副本
6. `client.resource.resolve(...)`：将本地路径、远端 URL 或 Web 本地文件 token 转换为可直接显示的资源 URI
7. `client.theme.getCurrent()` / `theme.changed`
8. `client.i18n.getCurrent()` / `language.changed`

## 3. 应用内部架构

### 3.1 Launch Intake

打开入口当前统一归一为“可解析资源标识”：

1. 启动上下文里的 `launchParams.resourceOpen.filePath`
2. 启动上下文里的 `launchParams.resourceOpen.resourceId`
3. 启动上下文里的 `launchParams.targetPath`
4. 文件对话框返回的本地路径

入口层只做三件事：

1. 归一启动参数中的本地路径或远端资源 URL；
2. 校验扩展名是否在支持列表内；
3. 把资源标识交给后续图片会话层。

### 3.2 Image Session

应用内部会话状态由以下信息组成：

```ts
interface ImageSource {
  filePath: string;
  fileName: string;
  resourceUri: string;
  revision: number;
}
```

以及以下派生状态：

1. `isResolving`
2. `isImageLoaded`
3. `feedback`
4. `imageDimensions`
5. `camera`

其中：

1. `filePath` 当前保存的是应用内部的“打开标识”，既可能是本地路径，也可能是远端 URL；
2. `resourceUri` 来自 `resource.resolve(...)`
3. `revision` 用于在重复打开同一路径时强制刷新图片元素

其中 `camera` 统一收口图片查看态的瞬时交互状态：

```ts
interface ViewerCameraState {
  zoomMode: "fit" | "manual";
  manualScale: number;
  viewportSize: ViewportSize | null;
  panOffset: { x: number; y: number };
}
```

当前正式实现已经把这部分逻辑从应用入口拆到独立 `usePhotoViewerCamera(...)` Hook 中，避免文件接入、主题/多语言、图片手势和布局骨架继续混在一个大组件里。

### 3.3 缩放模型

缩放逻辑抽到 `src/utils/image-viewer.ts`，核心规则如下：

1. `fit` 模式根据图片原始尺寸和“稳定视口窗口”尺寸实时计算 `fitScale`
2. `manual` 模式维护独立的 `manualScale`
3. 放大 / 缩小以固定倍率 `1.25` 递进
4. 缩放范围统一钳制在 `0.1 ~ 8`
5. `fitScale` 采用正式“长边 80%”规则：
   横图使用 `viewport.width * 0.8 / natural.width`，竖图使用 `viewport.height * 0.8 / natural.height`，方图取宽高两者中的更小值

### 3.4 跨输入方式交互

视口交互当前统一由 `Wheel + Pointer Events` 驱动：

1. 鼠标滚轮
   - 直接作为缩放输入；
2. 触摸板
   - 普通双指滚动继续交给视口原生滚动；
   - `ctrl/meta + wheel` 等价于捏合缩放；
3. 鼠标 / 触控笔 / 触摸屏
   - 单指或主按键拖动画面平移；
4. 触摸屏双指
   - 通过 Pointer Events 计算两点距离与中点，执行锚点缩放。

补充规则：

1. 未带修饰键的 `wheel` 事件会根据增量形态区分“鼠标滚轮缩放”与“触控板滚动平移”；
2. 图片平移不再依赖滚动容器，而是维护独立 `panOffset`，因此任何缩放倍率下都允许直接拖动；
3. 缩放时会基于当前指针锚点重新投影 `panOffset`，尽量保持用户正在观察的区域不跳变；
4. 新图片打开、切回 `fit` 模式或切到实际大小时，会把平移偏移归零，保证默认打开位置稳定。

### 3.5 查看层布局

当前查看层已经拆成三层稳定结构：

1. `PhotoViewerStage`
   - 作为唯一的全窗口舞台根节点；
2. `photo-viewer-frame`
   - 作为稳定的图片显示窗口，始终铺满整个应用窗口；
3. `photo-viewer-chrome`
   - 单独承载底部 dock 与反馈浮层，不参与图片布局，也不受图片缩放影响。

其中：

1. 图片本体始终渲染在 `photo-viewer-image-frame` 中，并围绕视口中心做缩放与平移；
2. 底部居中的浮动 dock 固定挂在 `photo-viewer-chrome` 上；
3. 状态反馈仅在保存成功、打开失败等需要提示时短时出现，不默认展示图片信息面板。

### 3.6 保存链路

保存不读取图片二进制后重写，而是直接走正式文件复制链路：

1. `platform.saveFile(...)` 获取目标路径；
2. 若用户取消，直接结束；
3. 若目标路径等于源路径，返回提示；
4. 其它情况调用 `file.copy(sourcePath, destPath)`。

这样可以保证首版“保存图片”语义稳定为“保留原始内容的副本输出”。

### 3.7 主题与多语言

应用壳层通过以下方式接入生态：

1. `ChipsThemeProvider` 负责主题运行时同步；
2. 首屏先读取文档注入的主题快照，减少闪烁；
3. 文案统一来自 `i18n/zh-CN.json` 与 `i18n/en-US.json`；
4. 通过 `language.changed` 跟随 Host 当前语言变化。

## 4. Manifest 与能力声明

当前 `manifest.yaml` 关键声明如下：

### 4.1 `capabilities`

```yaml
capabilities:
  - resource-handler:view:image/*
  - file-handler:.png
  - file-handler:.jpg
  - file-handler:.jpeg
  - file-handler:.webp
  - file-handler:.gif
  - file-handler:.bmp
  - file-handler:.svg
  - file-handler:.avif
```

### 4.2 `permissions`

```yaml
permissions:
  - resource.read
  - file.write
  - platform.read
  - theme.read
  - i18n.read
```

这些权限分别对应：

1. 解析图片资源 URL；
2. 保存图片副本；
3. 打开/保存系统对话框；
4. 主题读取；
5. 当前语言读取。

## 5. “没安装插件时 Host 不接图片”的实现方式

当前正式行为如下：

1. `Chips-PhotoViewer` 被设计为 Host 的可选内置 app 插件；
2. 只有当发行包真正携带该插件时，安装器才会根据它的 `file-handler:<ext>` 能力写入图片文件关联；
3. 若发行包不带该插件：
   - 安装器不会注册图片类型；
   - Host 启动时也不会因为缺少图片查看器 bundle 而报错；
   - 结果就是 Host 不会接住这些图片文件。

当前尚未处理的边界是：

1. 如果用户已经安装过带图片查看器的 Host，再通过其它方式移除插件，操作系统已有的文件关联回收仍需要依赖安装器 / 卸载链路治理；
2. 这不影响“未安装图片查看器插件时，Host 不注册也不接图片”的首版结论。

## 6. 卡片查看器点击图片的正式闭环

当前已经正式闭环的链路如下：

1. 图片基础卡片点击图片后，只表达 `openResource(...)` 意图；
2. Host 复合卡片运行时把该意图转发给卡片查看器；
3. 卡片查看器调用 `client.resource.open(...)`；
4. Host 自动匹配图片查看器的 `resource-handler:view:image/*`；
5. 图片查看器通过 `launchParams.resourceOpen.filePath` 或 `targetPath` 恢复查看状态。

## 7. 当前剩余边界

本轮仍未扩展：

1. 纯内存二进制资源、不落地文件路径时的专用查看器载荷协议；
2. 多个图片查看器并存时的默认处理器优先级治理。

这两项属于后续生态扩展，不是当前图片查看器首版阻塞。
