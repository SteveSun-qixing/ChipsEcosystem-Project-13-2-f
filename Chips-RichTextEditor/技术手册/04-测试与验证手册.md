# 富文本编辑器测试与验证手册

## 1. 验证目标

本项目验证分为三层：

1. 文件模型与辅助模块单元测试；
2. 应用壳层基础可用性测试；
3. 正式脚本构建与校验。

## 2. 单元测试重点

### 2.1 文件模型

需要覆盖：

1. 单富文本基础卡片复合卡片的构建；
2. 单富文本基础卡片复合卡片的解析；
3. 资源路径收集与 `manifest.resources` 重建逻辑；
4. 标题推导、时间戳、封面 HTML 输出。

### 2.2 路径与会话辅助

需要覆盖：

1. 运行时目录路径生成；
2. `.card` 扩展名收敛；
3. 相对资源路径清洗。

### 2.3 应用壳

至少覆盖：

1. `App` 正常导出；
2. 关键纯函数的异常分支与口径稳定性。

## 3. 手动联调重点

至少执行以下手动场景：

1. 启动后新建草稿并保存；
2. 再次打开刚保存的 `.card` 文件；
3. 输入短正文，确认走内联模式；
4. 输入长正文，确认走 Markdown 资源文件模式；
5. 另存为到新路径；
6. 打开右上角菜单，确认文件组与信息组均可用；
7. 打开信息弹窗，确认卡片 ID、基础卡片 ID、文件路径、存储模式与运行时状态准确；
8. 在有未保存改动时执行新建 / 打开，确认弹窗生效；
9. 切换主题与语言，确认壳层和正文区均稳定刷新。

## 4. 正式脚本验证

开发收口前必须执行：

1. `npm run lint`
2. `npm test`
3. `npm run build`
4. `npm run validate`

验证口径：

- 至少与本轮改动范围匹配；
- 若任一脚本失败，不允许把当前实现视为完成。

## 5. 问题治理

若在实现或验证过程中发现以下问题，必须先登记工单再继续：

1. Host `card.*` 链路行为与文档不一致；
2. `editorPanel` 资源桥协议存在缺口；
3. 富文本基础卡片首版行为与当前协议无法收口；
4. 主题、多语言或正式文件服务在本应用场景下不可用。
