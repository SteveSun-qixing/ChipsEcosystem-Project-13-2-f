# 富文本编辑器产品定位与范围

## 1. 文档目标

本文档用于冻结 `Chips-RichTextEditor` 首版的产品定位、生态边界与交付范围，作为后续技术手册、开发计划和实现验收的统一依据。

## 2. 产品定位

`Chips-RichTextEditor` 是运行在薯片 Host 内部的 `type: app` 应用插件，目标是提供一个“像 Typora 一样专注单文档写作”的 Markdown 所见即所得编辑器。

它不是独立重写富文本编辑内核，而是：

1. 调用正式 `base.richtext` 富文本基础卡片编辑器；
2. 在应用层提供单文档壳、右上角菜单、信息弹窗与保存链路；
3. 把最终结果保存为“只包含一个富文本基础卡片”的复合 `.card` 文件。

## 3. 首版核心结论

首版必须满足以下冻结结论：

1. 一个编辑窗口只编辑一份文稿。
2. 一份文稿在文件层面就是一个复合卡片文件。
3. 该复合卡片内部正式结构只允许包含一个 `base.richtext` 基础卡片节点。
4. 文稿正文由富文本基础卡片承担；应用壳负责文稿元数据、文件路径、打开/保存/另存为、自动保存、未保存提示与信息查看等外围能力。

## 4. 正式支持的使用场景

### 4.1 应用内新建文稿

- 用户启动应用后，直接进入一份新的单文档草稿。
- 草稿初始状态尚未绑定正式 `.card` 文件路径。

### 4.2 应用内手动打开

- 用户通过正式文件对话框选择一个 `.card` 文件；
- 应用仅接受“只包含一个富文本基础卡片”的复合卡片；
- 若文件结构不符合约束，应用必须明确拒绝并提示原因。

### 4.3 Host 启动参数恢复

- Host 或其他生态能力可通过 `launchParams.targetPath` 把 `.card` 文件路径交给本应用；
- 应用启动后需要恢复到该目标卡片的编辑状态。

### 4.4 应用内保存与另存为

- 用户可以把当前草稿保存为正式 `.card` 文件；
- 已绑定路径的文稿可继续原位保存；
- 用户可在任意时刻执行“另存为”，把当前文稿导出到新的 `.card` 文件路径。

## 5. 首版必须交付的能力

### 5.1 编辑体验

1. 单文档极简编辑界面；
2. 右上角单入口菜单；
3. 主体正文由正式富文本基础卡片编辑器承载；
4. 信息弹窗可查看当前卡片、富文本与运行时状态；
5. 明确的空态、加载态、错误态与保存状态反馈；
6. 整体体验尽量接近 Typora 的“写作优先、弱工具栏、弱管理感”。

### 5.2 文件能力

1. 新建；
2. 打开；
3. 保存；
4. 另存为；
5. 已绑定文稿的自动保存；
6. 未保存改动放弃确认。

### 5.3 生态接入

1. 正式 `chips-sdk` / `window.chips.*` 链路；
2. 主题系统；
3. 多语言系统；
4. Host 启动上下文；
5. `card.pack / card.unpack` 正式文件链路；
6. `client.card.editorPanel.render(...)` 正式富文本编辑器装配链路。

## 6. 明确不在首版范围

以下能力不进入本轮交付：

1. 多标签页文稿管理；
2. 多基础卡片拼装编辑；
3. 左侧目录树、文库管理器、搜索中心；
4. 自建第二套 Markdown 编辑器实现；
5. 绕过 SDK / Host 直接 import 富文本基础卡片源码运行；
6. 通用 `.card` 文件处理器注册。

关于第 6 点，当前 Host 对同一扩展名的 app 处理器优先级治理尚未完善；若本应用直接声明 `.card` 文件处理能力，会与现有 `Chips-CardViewer` 形成路由歧义。因此首版不在 manifest 中声明通用 `.card` 处理能力，只支持应用内打开与显式启动参数恢复。
