# 富文本编辑器核心流程与交互需求

## 1. 新建流程

### FR-NEW-001 新建草稿

- 应用启动后默认提供一份新的草稿文档。
- 草稿需自动生成：
  - 复合卡片 `card_id`
  - 单一富文本基础卡片 `baseCardId`
  - 初始封面 HTML
  - 初始标题
  - 初始 Markdown 正文

### FR-NEW-002 草稿状态

- 新建草稿在首次保存前没有正式文件路径。
- 应用至少在信息弹窗或轻提示中可确认“尚未保存”状态。

## 2. 打开流程

### FR-OPEN-001 打开入口

- 应用必须支持：
  - 点击“打开”按钮后使用正式文件对话框选择文件；
  - 通过 `launchParams.targetPath` 恢复目标卡片。

### FR-OPEN-002 打开校验

- 仅接受扩展名为 `.card` 的文件路径。
- 打开后必须校验复合卡片结构是否满足：
  1. `structure` 中只有一个节点；
  2. 节点类型为 `base.richtext` 或其正式别名；
  3. 对应 `content/<baseCardId>.yaml` 可读取并解析为富文本基础卡片配置。

### FR-OPEN-003 失败反馈

- 当文件不存在、读取失败、格式不合法、不是单富文本基础卡片复合卡片时，应用必须给出明确错误提示。

## 3. 编辑流程

### FR-EDIT-001 正文编辑

- 正文编辑区域必须由正式富文本基础卡片编辑器提供。
- 应用壳不得用临时 `<textarea>` 或自定义简化编辑器替代正式编辑器。

### FR-EDIT-002 标题维护

- 首版不提供常驻标题输入栏；
- 文稿标题仍需作为复合卡片元数据正式保存；
- 新建草稿在首次保存前若仍保留默认草稿标题，则保存时必须回写为用户选择的文件名推导值；
- 信息弹窗必须能够显示当前标题。

### FR-EDIT-003 状态反馈

- 应用必须至少通过轻提示或信息弹窗反馈：
  - 编辑器加载中；
  - 编辑器已就绪；
  - 编辑器装载失败；
  - 当前文稿是否已保存。

### FR-EDIT-004 界面结构

- 常驻界面只保留：
  - 中央正文编辑区；
  - 右上角菜单按钮。
- 应用窗口本身必须使用 Host / Electron 原生标题栏，而不是页面内自绘假标题栏。
- 文件操作入口统一收纳在菜单中。
- “信息”入口必须打开弹窗，显示当前复合卡片、富文本正文与运行时状态。

## 4. 保存流程

### FR-SAVE-001 保存

- 已绑定文件路径的文稿，点击“保存”必须直接更新该 `.card` 文件。
- 首次保存时必须调用正式保存对话框选择目标文件路径。

### FR-SAVE-002 另存为

- 点击“另存为”必须始终让用户选择新的输出文件路径。
- 另存为完成后，当前会话应切换为新的绑定路径继续编辑。

### FR-SAVE-003 自动保存

- 文稿已经绑定正式路径后，应用必须支持自动保存。
- 自动保存只作为“减轻用户显式保存负担”的增强，不替代手动保存入口。

### FR-SAVE-004 放弃确认

- 当当前文稿存在未保存改动时，执行“新建”“打开其他文件”等会丢弃当前状态的动作前，应用必须给出确认提示。

## 5. Typora 风格体验要求

### UX-001 布局风格

- 整体界面应以单页文稿为中心，不做多窗格工程化布局。
- 页面内常驻 chrome 只保留右上角菜单按钮，正文区域尽量连续、稳定、无多余边界干扰。
- 窗口外壳层使用 Electron 原生标题栏，页面不再重复绘制标题栏。

### UX-002 写作优先

- 编辑器不应因菜单开合、状态提示刷新、主题刷新而频繁重挂载正文 iframe。
- 编辑过程必须尽量避免失焦、闪烁和选区丢失。

### UX-003 弱管理感

- 首版不提供卡片节点列表、复杂配置面板或卡片结构编辑器。
- 用户心智必须始终是“我在编辑一篇文稿”，而不是“我在拼一张卡片”。
