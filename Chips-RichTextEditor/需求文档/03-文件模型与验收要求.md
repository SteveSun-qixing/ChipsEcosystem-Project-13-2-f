# 富文本编辑器文件模型与验收要求

## 1. 正式保存模型

本应用创建和保存的正式文件始终是复合 `.card` 文件，目录态结构必须满足：

```text
<name>.card
├─ .card/
│  ├─ metadata.yaml
│  ├─ structure.yaml
│  └─ cover.html
├─ content/
│  └─ <baseCardId>.yaml
└─ <optional-root-resources>
```

其中：

1. `structure.yaml` 只允许声明一个基础卡片节点；
2. 该节点正式类型为 `base.richtext`；
3. `content/<baseCardId>.yaml` 保存富文本基础卡片配置；
4. 若正文超过富文本基础卡片的内联阈值，可转为卡片根目录下的 Markdown 资源文件，并在基础卡片配置中通过相对路径引用；
5. `structure.yaml manifest.resources` 必须根据卡片根目录真实资源文件重建，不允许写死占位清单。

## 2. 复合卡片与富文本基础卡片关系

首版固定采用“一层复合卡片壳 + 一个富文本基础卡片正文”的模型：

- 复合卡片负责：
  - 卡片 ID
  - 名称
  - 封面
  - 时间戳
  - 资源清单
- 富文本基础卡片负责：
  - Markdown 正文
  - 内联 / 文件型正文存储模式

## 3. 非功能要求

### NFR-001 架构边界

- 必须使用 React 技术栈；
- 必须通过 `chips-sdk` 或正式 Bridge 链路访问 Host；
- 不得引入未在现有技术选型中确认的新第三方库；
- 不得留临时实现、占位实现或 TODO。

### NFR-002 主题与多语言

- 所有用户可见文案必须进入 i18n 资源；
- 应用壳必须接入 `ChipsThemeProvider` 和正式主题事件；
- 不允许把主题切换做成页面重建。

### NFR-003 稳定性

- 编辑器 iframe 的重建条件只能来自新建、打开其他文稿或明确重置会话；
- 正文输入过程不得因为普通状态变更反复重挂载编辑器。

## 4. 验收基线

首版验收至少包括：

1. 新建后可直接进入正文编辑；
2. 可手动打开合法的单富文本基础卡片复合 `.card` 文件；
3. 打开非法 `.card` 文件时会明确报错；
4. 可保存为 `.card` 文件，并能再次打开继续编辑；
5. 长正文能够通过资源文件模式稳定保存和再次读取；
6. 已绑定路径的文稿可自动保存；
7. 主题和语言切换不会破坏应用壳与正文编辑区；
8. 项目级 `npm run lint`、`npm test`、`npm run build`、`npm run validate` 通过。
