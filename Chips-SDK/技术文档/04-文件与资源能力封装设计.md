# 文件与资源能力封装设计

> 文档状态：设计基线稿  
> 关联需求：FR-SDK-FILE-* 系列

---

## 1. 设计目标

- 提供类型安全、易用的文件、卡片、箱子与资源访问封装。
- 将 Host `file/resource/card/box` 服务能力暴露为高层 API。
- 坚持“无业务逻辑”的封装原则，仅做参数整理与错误归一。

---

## 2. 文件服务封装

### 2.1 API 设计

```ts
// src/api-wrapper/file/index.ts

export interface FileApi {
  read(path: string, options?: FileReadOptions): Promise<FileContent>;
  write(path: string, content: FileContent, options?: FileWriteOptions): Promise<void>;
  list(dir: string, options?: FileListOptions): Promise<FileEntry[]>;
  stat(path: string): Promise<FileStat>;
}
```

实现要点：

- 映射关系：
  - `file.read` → Host `file.read`
  - `file.write` → Host `file.write`
  - `file.list` → Host `file.list`
  - `file.stat` → Host `file.stat`
- 参数校验：
  - `path` 不可为空字符串。
  - 根据 `options.encoding` 决定返回内容类型。

### 2.2 二进制读取归一语义

- `createFileApi().read(path, { encoding: "binary" })` 对调用方的正式语义始终是“返回 `Uint8Array` 裸字节”；
- SDK 必须在封装层统一兼容 Host / Bridge 可能出现的：
  - `latin1` 字节串
  - Base64 字符串
  - `{ type: "Buffer", data: number[] }`
  - `ArrayBuffer`
  - TypedArray / DataView
- 业务层不得再感知或处理上述桥接中间形状。

## 3. 卡片服务封装

### 3.1 解析与验证

```ts
export interface CardApi {
  parse(cardFile: string): Promise<CardDocument>;
  validate(card: CardDocument): Promise<ValidationResult>;
  render(cardFile: string, options?: RenderOptions): Promise<CardRenderView>;
}
```

- 映射关系：
  - `parse` → Host `card.parse`
  - `validate` → Host `card.validate`
  - `render` → Host `card.render`
- 参数约束：
  - `cardFile` 必须为有效路径或 URI。
  - `options.target`、`options.viewport`、`options.verifyConsistency` 必须符合 `card.render` schema。
- 返回值：
  - `CardRenderView` 必须包含 `semanticHash` 与诊断信息，类型与 Host 对外接口基线一致。

### 3.2 与统一显示链路的衔接

- `card.render` 封装返回结果供统一显示接口复用：
  - 例如对 `CardRenderView` 中的内容文件列表进行二次处理。
- 统一显示链路具体设计见《05-统一卡片显示链路技术方案》。

## 4. 箱子服务封装

### 4.1 API 设计

```ts
export interface BoxApi {
  pack(boxDir: string, options?: { outputPath?: string }): Promise<string>;
  unpack(boxFile: string, outputDir: string): Promise<string>;
  inspect(boxFile: string): Promise<BoxInspectionResult>;
  validate(boxFile: string): Promise<BoxValidationResult>;
  readMetadata(boxFile: string): Promise<BoxMetadata>;
  openView(
    boxFile: string,
    options?: { layoutType?: string; initialQuery?: BoxEntryQuery }
  ): Promise<BoxOpenViewResult>;
  listEntries(sessionId: string, query?: BoxEntryQuery): Promise<BoxEntryPage>;
  readEntryDetail(
    sessionId: string,
    entryIds: string[],
    fields: BoxEntryDetailField[]
  ): Promise<BoxEntryDetailItem[]>;
  resolveEntryResource(
    sessionId: string,
    entryId: string,
    resource: {
      kind: BoxEntryResourceKind;
      key?: string;
      sizeHint?: { width?: number; height?: number };
    }
  ): Promise<ResolvedRuntimeResource>;
  readBoxAsset(sessionId: string, assetPath: string): Promise<ResolvedRuntimeResource>;
  prefetchEntries(sessionId: string, entryIds: string[], targets: BoxPrefetchTarget[]): Promise<void>;
  closeView(sessionId: string): Promise<void>;
}
```

- 映射关系：
  - `pack` → Host `box.pack`
  - `unpack` → Host `box.unpack`
  - `inspect` → Host `box.inspect`
  - `validate` → Host `box.validate`
  - `readMetadata` → Host `box.readMetadata`
  - `openView` → Host `box.openView`
  - `listEntries` → Host `box.listEntries`
  - `readEntryDetail` → Host `box.readEntryDetail`
  - `resolveEntryResource` → Host `box.resolveEntryResource`
  - `readBoxAsset` → Host `box.readBoxAsset`
  - `prefetchEntries` → Host `box.prefetchEntries`
  - `closeView` → Host `box.closeView`

### 4.2 典型使用场景

- 编辑器或工具链通过 SDK 将工作目录打包为 `.box` 文件。
- 查看器先通过 `inspect` 读取 `layout_configs`，再调用 `openView` 建立正式查看会话。
- 布局插件宿主通过 `listEntries / readEntryDetail / resolveEntryResource / readBoxAsset` 按需消费箱子数据与资源。

## 5. 资源服务封装

### 5.1 API 设计

```ts
export interface ResourceApi {
  resolve(resourceId: string): Promise<ResourceUri>;
  open(request: ResourceOpenRequest): Promise<ResourceOpenResult>;
  readMetadata(resourceId: string): Promise<ResourceMeta>;
  readBinary(resourceId: string): Promise<ArrayBuffer>;
  convertTiffToPng(request: ResourceConvertTiffToPngRequest): Promise<ResourceConvertTiffToPngResult>;
}
```

- 映射关系：
  - `resolve` → Host `resource.resolve`
  - `open` → Host `resource.open`
  - `readMetadata` → Host `resource.readMetadata`
  - `readBinary` → Host `resource.readBinary`
  - `readBinary` 对调用方继续维持“返回裸二进制”的稳定语义；若 Host / Bridge 内部返回 `{ data: Buffer }` 或桥接后的 Buffer JSON，SDK 必须在 `createResourceApi()` 内部统一解包，不能把包装形状泄露给业务层
  - `convertTiffToPng` → Host `resource.convertTiffToPng`

### 5.2 TIFF 转 PNG 封装补充

- `createResourceApi().convertTiffToPng(...)` 负责做基础参数校验，并把请求直接透传到正式 Host 路由；
- SDK 不承载 TIFF 解析、图片格式识别或平台命令执行逻辑；
- 编辑器链路若通过 `CardEditorResourceBridge.convertTiffToPng` 暴露该能力，仍应复用同一条正式 Host 路由，不得在调用方仓库复制转换实现。

### 5.3 缓存策略

- SDK 本身不维护持久缓存，仅提供可选的内存缓存辅助（具体策略在实现阶段评估）。
- 所有缓存行为必须可关闭，默认以“正确性优先”。

---

## 6. 错误处理与验收

- 所有文件与资源相关 API 必须统一处理典型错误：
  - `FILE_NOT_FOUND`
  - `FILE_PERMISSION_DENIED`
  - `BOX_FORMAT_INVALID`
  - `RESOURCE_NOT_FOUND`
  - `CARD_SCHEMA_INVALID`
- 验收标准：
  - 在 Host 契约测试通过的前提下，SDK 封装对错误码不做自定义映射，仅做标准错误包装。
  - 单元测试覆盖正常路径、错误路径与边界路径。

---

## 7. 小结

本设计为 `file/card/box/resource` 四类能力提供封装蓝图。在实现时必须严格遵守 Host 服务域设计与生态共用契约，避免在 SDK 层添加与上游不一致的额外语义。
