# 文件与资源能力封装设计

> 文档状态：设计基线稿  
> 关联需求：FR-SDK-FILE-* 系列

---

## 1. 设计目标

- 提供类型安全、易用的文件、卡片、箱子与资源访问封装。
- 将 Host `file/resource/card/box` 服务能力暴露为高层 API。
- 坚持“无业务逻辑”的封装原则，仅做参数整理与错误归一。

---

## 2. 文件服务封装

### 2.1 API 设计

```ts
// src/api-wrapper/file/index.ts

export interface FileApi {
  read(path: string, options?: FileReadOptions): Promise<FileContent>;
  write(path: string, content: FileContent, options?: FileWriteOptions): Promise<void>;
  list(dir: string, options?: FileListOptions): Promise<FileEntry[]>;
  stat(path: string): Promise<FileStat>;
}
```

实现要点：

- 映射关系：
  - `file.read` → Host `file.read`
  - `file.write` → Host `file.write`
  - `file.list` → Host `file.list`
  - `file.stat` → Host `file.stat`
- 参数校验：
  - `path` 不可为空字符串。
  - 根据 `options.encoding` 决定返回内容类型。

## 3. 卡片服务封装

### 3.1 解析与验证

```ts
export interface CardApi {
  parse(cardFile: string): Promise<CardDocument>;
  validate(card: CardDocument): Promise<ValidationResult>;
  render(cardFile: string, options?: RenderOptions): Promise<CardRenderView>;
}
```

- 映射关系：
  - `parse` → Host `card.parse`
  - `validate` → Host `card.validate`
  - `render` → Host `card.render`
- 参数约束：
  - `cardFile` 必须为有效路径或 URI。
  - `options.target`、`options.viewport`、`options.verifyConsistency` 必须符合 `card.render` schema。
- 返回值：
  - `CardRenderView` 必须包含 `semanticHash` 与诊断信息，类型与 Host 对外接口基线一致。

### 3.2 与统一显示链路的衔接

- `card.render` 封装返回结果供统一显示接口复用：
  - 例如对 `CardRenderView` 中的内容文件列表进行二次处理。
- 统一显示链路具体设计见《05-统一卡片显示链路技术方案》。

## 4. 箱子服务封装

### 4.1 API 设计

```ts
export interface BoxApi {
  pack(inputDir: string, options?: PackOptions): Promise<string>;
  unpack(boxFile: string, outputDir: string): Promise<void>;
  inspect(boxFile: string): Promise<BoxInspectResult>;
}
```

- 映射关系：
  - `pack` → Host `box.pack`
  - `unpack` → Host `box.unpack`
  - `inspect` → Host `box.inspect`

### 4.2 典型使用场景

- 插件或外部应用通过 SDK 将数据目录打包为 `.box` 文件。
- 验证 `.box` 文件结构或提取箱子元信息，用于 UI 显示。

## 5. 资源服务封装

### 5.1 API 设计

```ts
export interface ResourceApi {
  resolve(resourceId: string): Promise<ResourceUri>;
  readMetadata(resourceId: string): Promise<ResourceMeta>;
  readBinary(resourceId: string): Promise<ArrayBuffer>;
}
```

- 映射关系：
  - `resolve` → Host `resource.resolve`
  - `readMetadata` → Host `resource.readMetadata`
  - `readBinary` → Host `resource.readBinary`

### 5.2 缓存策略

- SDK 本身不维护持久缓存，仅提供可选的内存缓存辅助（具体策略在实现阶段评估）。
- 所有缓存行为必须可关闭，默认以“正确性优先”。

---

## 6. 错误处理与验收

- 所有文件与资源相关 API 必须统一处理典型错误：
  - `FILE_NOT_FOUND`
  - `FILE_PERMISSION_DENIED`
  - `BOX_FORMAT_INVALID`
  - `RESOURCE_NOT_FOUND`
  - `CARD_SCHEMA_INVALID`
- 验收标准：
  - 在 Host 契约测试通过的前提下，SDK 封装对错误码不做自定义映射，仅做标准错误包装。
  - 单元测试覆盖正常路径、错误路径与边界路径。

---

## 7. 小结

本设计为 `file/card/box/resource` 四类能力提供封装蓝图。在实现时必须严格遵守 Host 服务域设计与生态共用契约，避免在 SDK 层添加与上游不一致的额外语义。
