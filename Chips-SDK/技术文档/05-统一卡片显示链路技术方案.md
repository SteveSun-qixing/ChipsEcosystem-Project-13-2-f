## 1. 目标

- 为所有生态内部应用（查看器、编辑引擎等）提供统一的卡片显示接口。
- 确保“同一卡片文件通过 SDK 在任意生态内部应用中显示结果一致”。
- 与组件库的 `CardCoverFrame` 与 `CompositeCardWindow` 组件契约完全对齐。

---

## 2. 设计原则

1. **Host 负责渲染主实现**：基础卡片分发、模板编译、iframe 拼接由 Host 内置渲染运行时完成。
2. **SDK 仅封装入口**：SDK 不负责 HTML 模板渲染与 iframe 拼接，只负责参数封装与事件协作。
3. **应用负责套壳**：查看器、编辑引擎等生态内部应用仅承担窗口容器与业务外壳。
4. **事件协议统一**：复合卡片窗口通过统一事件协议与外部通信。

## 3. 接口设计

### 3.1 封面窗口接口

```ts
interface CoverFrameRenderOptions {
  cardFile: string;
  cardName?: string;
}

interface FrameRenderResult {
  frame: HTMLIFrameElement;
  origin: string;
}

interface CardCoverFrameApi {
  render(options: CoverFrameRenderOptions): Promise<FrameRenderResult>;
}
```

### 3.2 复合窗口接口

```ts
type CompositeMode = 'view' | 'preview';

interface CompositeWindowRenderOptions {
  cardFile: string;
  mode?: CompositeMode;
}

interface CompositeCardWindowApi {
  render(options: CompositeWindowRenderOptions): Promise<FrameRenderResult>;

  onReady(frame: HTMLIFrameElement, handler: () => void): () => void;
  onResize(
    frame: HTMLIFrameElement,
    handler: (payload: { height: number; nodeCount: number; reason: string }) => void,
  ): () => void;
  onNodeError(frame: HTMLIFrameElement, handler: (payload: CompositeNodeError) => void): () => void;
  onFatalError(frame: HTMLIFrameElement, handler: (error: StandardError) => void): () => void;
}
```

### 3.3 基础卡片编辑器面板接口

```ts
interface CardEditorRenderOptions {
  cardType: string;
  initialConfig?: Record<string, unknown>;
  baseCardId?: string;
}

interface CardEditorChangePayload {
  baseCardId?: string;
  cardType: string;
  pluginId: string;
  config: Record<string, unknown>;

interface CardEditorErrorPayload {
  baseCardId?: string;

  onError(frame: HTMLIFrameElement, handler: (payload: CardEditorErrorPayload) => void): () => void;
}
```

## 4. Host 渲染链路调用

### 4.1 card.render 封装

- `CompositeCardWindowApi.render` 内部逻辑：
  1. 调用 `client.card.render(cardFile, options)`，映射到 Host `card.render`。
  2. 从返回结果中读取渲染目标 URL 或渲染上下文。
  3. 构造 iframe：
     - 设置 `src` 或 `srcdoc`。
     - 配置受控 `sandbox` 与 `allow` 属性。
  4. 计算 `origin`（从 `src` 或 Host 提供的元信息）并写入 `FrameRenderResult`。

### 4.2 coverFrame 封装

- `CardCoverFrameApi.render` 可复用相同链路，但仅展示封面视图：
  - Host 渲染链路可提供专用封面模板；
  - 或通过统一渲染结果中标记“封面区域”。

## 5. 事件协议与 postMessage

### 5.1 事件类型

- `chips.composite:ready`：复合窗口渲染完成。
- `chips.composite:resize`：复合窗口当前总高度变化。
- `chips.composite:node-error`：某个子节点渲染或主题/契约校验失败。
- `chips.composite:fatal-error`：全局性错误，整窗无法正常运行。

### 5.2 事件派发链路

1. Host 内置渲染运行时在复合窗口内部通过 `postMessage` 向父窗口发送事件：
   - `window.parent.postMessage({ type, payload }, '*')`。
2. SDK 渲染接口在创建 iframe 时注册 `message` 事件监听：
   - 监听窗口为 `window` 或调用方提供的自定义对象。
3. 收到消息后：
   - 根据 `origin` 校验消息来源是否处于白名单。
   - 根据 `type` 分发到对应的事件处理器。

### 5.3 安全策略

- SDK 必须在内部维护 `allowedOrigins` 列表：
  - 默认包含 Host 所在 origin。
  - 允许通过配置追加额外测试环境 origin。
- 对 `origin` 不在白名单内的消息直接忽略。

当前 SDK 实现中：

- 通过 `frame.contentWindow` 与 `window.location.origin` 组合，校验 `event.source` 与 `event.origin`；
- 默认白名单仅包含当前页面 origin，适用于通过 `srcdoc` 渲染的场景；
- 如需扩展为跨域 Host iframe，可在后续版本在配置对象中增加 `allowedOrigins` 参数，并同步更新生态共用文档。

## 6. 组件库适配与工单依赖

### 6.1 组件库适配现状

- 组件库当前通过适配器契约：
  - `resolveCoverFrame({ cardId, cardFile, cardName, signal })`
  - `resolveCompositeWindow({ cardFile, mode, signal })`
- 在真实 SDK 未落地前，组件库适配器通过本地实现保持可运行状态。

### 6.2 SDK 落地后的改造方向

- 组件库适配器将切换为调用 SDK：
  - `resolveCoverFrame` 内部调用 `client.card.coverFrame.render`。
  - `resolveCompositeWindow` 内部调用 `client.card.compositeWindow.render`。
- SDK 必须提供与现有适配器等价或更友好的 API，以避免组件库大范围改动。

### 6.3 工单依赖

- 统一显示接口实现受 `工单001-SDK-UNIFIED-CARD-DISPLAY-API` 约束：
  - 在工单未闭环前，仅允许落地设计与文档，不实施临时代码。
  - 实施阶段需更新工单状态与本技术文档，记录最终接口形态与验收结果。

---

## 7. 验收标准

- 使用组件库 demo，接入 SDK 接口后：
  - 同一卡片文件在查看器与编辑引擎中的显示完全一致。
  - 子节点错误触发降级而非整窗崩溃。
  - fatal 错误触发统一错误视图与日志记录。
- E2E 测试覆盖：
  - `ready`、`resize`、`node-error` 与 `fatal-error` 四类事件。
  - 合法/非法 `origin` 场景。
  - `mode` 非法值阻断行为。

---

## 8. 小结

本方案定义了统一卡片显示链路在 SDK 层的职责、接口与事件模型。实现时必须严格遵守生态共用与组件库文档中的约束，禁止在 SDK 层发明临时事件或未文档化行为；如需调整现有口径，必须先通过工单流程更新生态共用文档并同步本技术文档。

补充说明：编辑引擎正式落地后，统一卡片显示链路已经与基础卡片编辑器面板链路配套成型，查看器、编辑引擎和未来其他应用必须统一复用这两条正式 Host + SDK 链路。
