## 1. 主题能力封装

### 1.1 API 设计

```ts
export interface ThemeApi {
  list(publisher?: string): Promise<ThemeMeta[]>;
  apply(themeId: string): Promise<void>;
  getCurrent(options?: { appId?: string; pluginId?: string }): Promise<ThemeState>;
  getAllCss(): Promise<{ css: string; themeId: string }>;
  resolve(chain: string[]): Promise<{
    resolved: Array<{ id: string; displayName: string; order: number }>;
    tokens: Record<string, unknown>;
  }>;
  contract: {
    get(component?: string): Promise<ThemeContract>;
  };
}
```

### 1.2 契约对齐

- 所有类型与字段必须对齐：
  - 《主题接口规范》：`生态共用技术文档/主题系统/02-主题接口规范.md`
  - 《Host 服务域设计》：`生态共用技术文档/架构设计/14-Host服务域设计.md`
- 若发现 Host 实现与文档不一致（如 `theme.resolve` 响应结构），必须先提工单处理，禁止在 SDK 层自行发明折衷结构。

### 1.3 常见用法示例（设计级）

- 获取所有主题并应用：

  ```ts
  const themes = await client.theme.list();
  await client.theme.apply(themes[0].id);
  ```

- 获取当前主题 CSS 并注入到页面（容器应用）：

  ```ts
  const { css, themeId } = await client.theme.getAllCss();
  // 容器应用负责将 CSS 注入到 <style> 或 Shadow DOM，并记录当前 themeId
  ```

---

### 1.4 当前实现补充（2026-03-10）

- `theme.list()` 的结果来源于当前工作区中已启用主题插件，不包含未启用主题；
- `theme.getAllCss()` 的返回结构为 `{ css, themeId }`，供应用建立显式主题快照；
- 应用启动时必须优先使用 Host 当前主题或 preload 注入的文档主题状态，不能在 SDK 消费端写死默认主题；
- 当主题切换后，依赖主题运行时缓存键的复合窗口应主动重建内容，而不是只刷新外层壳层。

## 3. 插件能力封装

### 3.1 插件信息查询

```ts
export interface PluginApi {
  getSelf(): Promise<PluginInfo>;
  list(options?: { type?: PluginType; capability?: string }): Promise<PluginInfo[]>;
  get(pluginId: string): Promise<PluginInfo | undefined>;
  getCardPlugin(cardType: string): Promise<PluginInfo | undefined>;
  getLayoutPlugin(layoutType: string): Promise<PluginInfo | undefined>;
  install(manifestPath: string): Promise<{ pluginId: string }>;
  enable(pluginId: string): Promise<void>;
  disable(pluginId: string): Promise<void>;
  uninstall(pluginId: string): Promise<void>;
  query(options?: { type?: PluginType; capability?: string }): Promise<PluginRecord[]>;
}
```

类型定义：

- `PluginInfo` 与 Bridge API 规范保持一致。
- `PluginType`：`'app' | 'card' | 'layout' | 'theme' | 'module'`。
- `PluginRecord` 对应 Host `plugin.query` 返回的运行时记录结构，包含治理列表所需的标准元数据与类型专属字段。

### 3.2 管理操作（谨慎封装）

- SDK 已提供 `list/get/getSelf/getCardPlugin/getLayoutPlugin/install/enable/disable/uninstall/query` 插件能力，严格映射到 Host `plugin.*` 服务动作：
  - `list/get/getSelf/getCardPlugin/getLayoutPlugin/query` 属于读取动作，统一使用 `plugin.read` 权限；
  - `install/enable/disable/uninstall` 属于治理变更动作，统一使用 `plugin.manage` 权限；
  - SDK 不做额外的缓存或“智能重试”，仅包装为类型安全的调用入口；
  - 插件管理行为（安装/卸载/启停）应主要用于 Host 自身、开发工具与运维脚本场景，在普通业务插件中不推荐直接调用。

---

### 3.3 开发工具集成

- `chipsdev` 已正式复用 Host 命令核心，在开发工作区内提供 `start/stop/status/config/logs/plugin/theme/update/doctor/open`；
- SDK 层不额外发明另一套开发态插件或主题管理协议，而是复用 Host `plugin.*` 与 `theme.*` 服务；
- 开发者若需要联调主题问题，应优先使用 `chipsdev plugin.*` 与 `chipsdev theme.*`，避免直接编辑 `.chips-host-dev`。

## 4. 窗口与模块能力封装

### 4.1 window 封装

```ts
export interface WindowApi {
  open(config: WindowConfig): Promise<WindowState>;
  focus(id: string): Promise<void>;
  resize(id: string, size: { width: number; height: number }): Promise<void>;
  setState(id: string, state: Partial<WindowState>): Promise<void>;
}
```

注意：

- `WindowConfig` 与 `WindowState` 必须与 Host 对外接口定义一致。
- 高风险能力（如关闭窗口、全屏等）应在文档中明确使用场景与权限要求。

### 4.2 module 封装

```ts
export interface ModuleApi {
  mount(
    slot: string,
    moduleId: string,
    options?: { requiredCapabilities?: string[] }
  ): Promise<ModuleState>;
  unmount(slot: string): Promise<void>;
  query(slot: string): Promise<ModuleState | undefined>;
  list(): Promise<ModuleState[]>;
}
```

约束：

- SDK 对外只暴露 `mount/unmount/query/list`，不再保留 `load/unload` 历史动作；
- `mount` 入参固定为 `slot + moduleId`，并支持可选 `requiredCapabilities`；
- `slot` 必须使用 namespaced dot 格式；
- Host 返回的 `{ module }` / `{ modules }` 由 SDK 在封装层解包；
- `ModuleState` 需与 Host 对齐，至少包含 `slot/moduleId/entry?/capabilities?/requiredCapabilities?/active/mountedAt`；
- `mount` 返回的 `ModuleState` 在成功挂载时可附带一次性的 `bridgeScopeToken`，用于调用 `createClient({ bridgeScope: { token } })` 建立模块独立作用域。
