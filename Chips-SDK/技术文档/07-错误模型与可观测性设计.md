# 错误模型与可观测性设计

> 文档状态：设计基线稿  
> 目标：统一 SDK 内部错误与日志模型，支撑调试与问题工单

---

## 1. 错误模型

### 1.1 标准错误对象

```ts
export interface StandardError {
  code: string;
  message: string;
  details?: unknown;
  retryable?: boolean;
  requestId?: string;
  traceId?: string;
}
```

来源：

- Bridge 层：`BRIDGE_*`
- Service 层：`SERVICE_*`
- Runtime 层：`RUNTIME_*`
- 权限层：`PERMISSION_*`

SDK 不新增自创前缀，仅在必要时使用 `INTERNAL_ERROR` 等约定错误码包装未知异常。

### 1.2 错误归一策略

- 若 Host 返回标准错误对象，SDK 直接透传，附加环境信息（如环境类型）。
- 若 Bridge 抛出 `Error` 实例或字符串：
  - 包装为 `StandardError`，`code` 设为 `INTERNAL_ERROR`，`details` 包含原始错误。
- 若调用方传入非法参数：
  - 在 SDK 层直接抛出 `StandardError`，`code` 为 `INVALID_ARGUMENT`。

## 2. 日志模型

### 2.1 日志记录结构

```ts
export interface SdkLogRecord {
  level: 'debug' | 'info' | 'warn' | 'error';
  time: string; // ISO 8601
  action?: string; // namespace.action
  requestId?: string;
  pluginId?: string;
  message?: string;
  details?: unknown;
}
```

### 2.2 日志入口

- 通过 `ClientConfig.logger` 注入日志实现。
- SDK 不强制输出日志，仅在提供 logger 时调用。

## 3. 调试模式

### 3.1 开关设计

- 通过客户端配置启用调试模式：

  ```ts
  const client = createClient({
    debug: true,
    logger: consoleLogger,
  });
  ```

- 或通过环境变量（仅 Node 环境）：
  - `CHIPS_SDK_DEBUG=1`。

### 3.2 调试输出内容

- 环境探测结果。
- 每次 `invoke` 的动作名与耗时（参数与返回值需适当脱敏）。
- 错误归一结果与原始错误信息摘要。

---

## 4. 性能观测

### 4.1 采样工具

- 在 `tooling/perf-benchmark.ts` 中提供测量工具：
  - 记录封装层额外开销（不含 Host 内部执行时间）。
  - 输出统计数据（p50/p95 等）。

### 4.2 门禁集成

- 在 CI 中运行性能基线脚本：
  - 若封装层额外开销超过阈值，CI 失败。

---

## 5. 问题工单协作

- SDK 出现无法由单仓解决的错误（如 Host 返回非文档化错误码、契约漂移）时：
  - 必须在 `项目日志与笔记/问题工单.md` 中登记工单；
  - 在工单任务说明中记录：调用路径、错误码、requestId/traceId 与上下文信息；
  - 禁止在 SDK 层通过“硬编码兼容逻辑”掩盖问题。

---

## 6. 小结

统一的错误与可观测性设计保证了 SDK 在复杂生态环境下的可调试性与可维护性。实现时必须严格遵守本设计，并确保所有公共 API 使用标准错误对象与日志模型。
