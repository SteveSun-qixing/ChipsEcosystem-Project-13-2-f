# 类型系统与契约集成设计

> 文档状态：设计基线稿  
> 目标：确保 SDK 类型定义与生态共用契约保持一致，并为后续自动化校验提供基础

---

## 1. 类型定义策略

### 1.1 手工维护 vs 代码生成

- 初期阶段：以手工维护类型为主，优先保证正确性与可读性。
- 中长期：根据 Host 导出的 `route-manifest` 与 schema，部分类型可通过脚本生成。

### 1.2 类型文件组织

- `src/types/errors.ts`：错误模型与错误码枚举。
- `src/types/card.ts`：卡片文档、渲染结果与诊断类型。
- `src/types/box.ts`：箱子检查结果与结构描述。
- `src/types/theme.ts`：主题元信息、状态与契约类型。
- `src/types/plugin.ts`：插件信息结构与插件类型。
- `src/types/client.ts`：客户端配置与实例接口。

---

## 2. 契约集成

### 2.1 route-manifest

- Host 在构建或发布阶段导出 `route-manifest.json`，内容包含：
  - 动作名：`namespace.action`。
  - 输入/输出 schema ID。
  - 幂等性与重试策略。
  - 权限需求。

- SDK 在 `src/contracts/route-manifest.json` 中保留该清单的快照（只读）。

### 2.2 契约校验工具

- 在 `src/tooling/route-manifest.ts` 中实现：
  - `assertKnownAction(action: string)`：检查动作是否存在于清单中。
  - `listActionsByNamespace(namespace: string)`：列出某命名空间的动作。

- 在测试中使用：
  - 确认所有封装调用的动作都存在于清单中。
  - 监控 Host 侧新增或删除动作引起的漂移。

---

## 3. 类型与契约的一致性

### 3.1 手工对照流程

1. 修改或新增类型定义前，先查阅生态共用文档与 Host 服务域设计。
2. 对照 `route-manifest` 与对应 schema 文档。
3. 若发现差异：
   - 判定是文档、Host 实现还是现有类型定义的问题；
   - 必要时提工单并等待裁决。

### 3.2 漂移检测

- 在 `tests/tooling/contract-drift.test.ts` 中：
  - 检查已封装动作集合与 `route-manifest` 中公开动作集合之间的差异；
  - 将未封装但公开的动作标记为“待规划”而非立即实现，避免临时实现；
  - 将已封装但不在清单中的动作视为错误。

---

## 4. 对外类型导出原则

- 对外仅导出稳定类型：
  - 与生态共用文档中已经冻结的接口与数据结构一致。
- 对内部实现细节类型（如内部错误封装、适配器私有结构）不对外导出。
- 对尚在演进中的结构（如实验性能力），需在类型命名与文档中清晰标记实验性状态，并避免作为跨仓正式依赖。

---

## 5. 小结

通过明确类型文件组织、契约集成与漂移检测机制，`Chips-SDK` 能够在长期演进过程中保持与 Host 与生态共用文档的一致性，避免“文档一套、实现一套”的情况。实现过程中如需引入代码生成或更复杂的契约集成机制，必须先更新本设计并确保与上游仓库协同。
