## 1. 项目背景与目标

薯片 SDK 是薯片生态内部使用的开发者工具包，主要面向薯片主机内部的应用插件和生态维护者，为这些内部应用提供统一的类型化封装与调用入口。

在 12 层架构中，运行时主实现（L1-L9）均由 `Chips-Host` 内置实现；`Chips-ComponentLibrary` 负责 L10 无头组件，SDK 仅负责：

- 对 Bridge API 与 Host 对外接口的类型化封装；
- 提供一致的客户端实例与能力域调用接口；
- 提供脚手架、工具函数与示例工程，降低集成成本；
- 提供测试与契约校验辅助能力。

终端用户设备仅需安装 Host 即可运行完整功能；SDK 不作为用户侧运行依赖，只在开发与生态内部集成阶段按需使用。

### 1.1 北极星目标

- 形成可持续 5–10 年演进的统一开发者工具包，覆盖插件开发与生态内部应用开发两大场景。
- 用一套 SDK 标准支撑 `app/card/layout/module/theme` 五类插件及 Host 内部应用对 Host 能力的访问。
- 确保“任意应用调用同一显示接口即可得到一致卡片窗口”，统一卡片显示链路。
- 降低开发者复杂度：接口命名统一、错误模型统一、类型系统统一、测试门禁统一。
- 保持 SDK 自身无运行时硬依赖：不重新实现 Runtime Client 或渲染引擎，只做封装与工具。

### 1.2 本轮开发目标

- 交付 JS/TS 版本 `chips-sdk` 核心包，优先支持运行在薯片主机内部的 Web 环境（插件窗口、设置面板等），以及生态内部工具所需的 Node.js 场景。
- 完成文件／资源／配置／主题／插件等核心能力域的封装接口。
- 完成卡片统一显示链路封装：
  - `client.card.render` 统一渲染入口封装；
  - `client.card.coverFrame.render` 与 `client.card.compositeWindow.render` 两个统一显示接口；
  - `client.box.renderCover` 箱子本体封面接口；
  - 与组件库 `EmbeddedDocumentFrame` / `CardCoverFrame` / `CompositeCardWindow` 的事件协议与 origin 安全联动。
- 建立完整的错误模型、日志与可观测性约束，并提供调试辅助工具。
- 建立测试与质量门禁（契约测试、单元测试、集成测试），达到“零临时实现、零占位代码”标准。

## 2. 范围与边界

### 2.1 In Scope（本轮范围内）

- SDK 核心客户端：
  - 客户端实例创建与配置（Host 连接信息、认证、环境探测）。
  - Bridge API 封装：通过 `window.chips.*` 或等价通道调用 Host 对外接口。
  - 标准错误对象与重试策略的轻量封装（不重写 Runtime Client 内部实现）。
- 能力域封装：
  - `file/card/box/resource`：文件与卡片操作封装。
  - `theme/i18n/config`：主题、多语言与配置能力封装。
  - `plugin/module/window/platform`：插件与窗口、平台接口的常用调用封装。
- 卡片显示链路：
  - `client.card.render` 对 `card.render` 统一渲染入口的封装。
  - 对 `card.coverFrame` / `card.compositeWindow` 显示链路的统一封装与事件模型定义。
- 开发者工具：
  - TypeScript 类型定义与公共类型导出。
  - 开发环境与示例工程（仅文档与目录规划，不在本轮实现代码骨架）。
  - 与未来命令行工具（`chips dev` 等）的协作边界说明。

### 2.2 Out of Scope（本轮不做）

- Host 内部实现改动（Kernel、Services、Runtime Client、渲染引擎等）。
- 组件库内部实现改动（仅消费其对外契约，不在 SDK 仓维护 UI 组件）。
- 自行实现 Bridge Transport / Runtime Client 替代品。
- 非技术选型中第三方库（如额外状态管理、网络库等）的引入。
- 任何“临时版、占位版、简化版”的 SDK 实现。
- 将 SDK 作为博客系统、知识库、课程平台等完全独立第三方业务系统的通用集成工具使用。

### 2.3 信息归属边界

- 跨生态公开的能力口径、接口签名与协议描述，统一归档在：
  - `生态设计原稿`：概念和宏观设计；
  - `生态共用技术文档`：正式对外接口与开发者指南。
- `Chips-SDK/需求文档` 仅描述本项目的实现范围、阶段目标与内部约束，不直接承担对外正式口径。

## 3. 不可违反约束（硬性需求）

### 3.1 架构约束

- SDK 不得重新实现 Host Runtime Client、渲染引擎或 Bridge Transport，必须通过既有接口封装实现。
- SDK 只能调用 Host 已冻结的对外接口命名与语义，不得在 SDK 层发明新的“半官方”动作名。
- 所有对外公开的 SDK API 必须能映射到 Host/Bridge/协议层明确的动作或能力。
- SDK 必须遵守 12 层分层约束：自身定位于“开发者工具包”，不越界承担运行时职责。

### 3.2 工程约束

- 禁止临时版、简化版、占位实现；所有对外 API 必须有完整实现与测试。
- 禁止删除文件；如需废弃实现，迁移到 `Chips-SDK/归档/`（未来创建）并在文档中标注。
- 禁止修改任务范围外文件，尤其是其他项目（Host、组件库）代码与文档。
- 禁止在当前任务范围内留下 `TODO`；需求内事项必须在当前阶段闭环。
- 禁止引入未在技术选型中批准的第三方库（需要时先提案再获批）。

### 3.3 质量约束

- 所有 SDK API 必须返回或抛出标准错误对象，错误模型与 Host/Bridge 保持一致。
- 所有关键路径必须有自动化测试覆盖，质量门禁不过关不得合并到 `develop`。
- 对外暴露的类型定义不得依赖 `any`；如确需宽松类型，需在文档中明确说明原因与边界。

### 3.4 阻塞与工单流程

- 若发现 Host/Bridge/组件库缺失必要能力或契约不一致，导致无法按规范实现 SDK：
  - 必须先在 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/项目日志与笔记/问题工单.md` 中登记问题；
  - 在 `项目日志与笔记/任务清单` 下创建对应任务文件夹与任务说明；
  - 暂停当前任务推进，等待工单裁决后再继续；禁止以临时实现绕过缺口。

## 4. 干系人与职责

| 角色 | 关注点 | 交付责任 |
|---|---|---|
| 架构负责人 | SDK 在 12 层架构中的定位、分层边界、长期演进 | 冻结 SDK 架构与接口原则，裁决跨仓争议 |
| SDK 开发负责人 | SDK 功能完整性、稳定性与测试覆盖 | 组织实现能力域封装与质量门禁 |
| Host 开发负责人 | Host 对外接口稳定性与契约一致性 | 提供稳定 Host/Bridge 接口与路由清单 |
| 组件库负责人 | 统一显示链路与组件契约一致性 | 提供卡片显示组件契约与联调支持 |
| 插件与生态内部应用开发者 | SDK 可用性、易用性、文档质量 | 反馈集成问题与改进需求 |
| QA/测试 | 回归质量、契约漂移、性能与鲁棒性 | 建立测试策略、执行门禁与回归 |
