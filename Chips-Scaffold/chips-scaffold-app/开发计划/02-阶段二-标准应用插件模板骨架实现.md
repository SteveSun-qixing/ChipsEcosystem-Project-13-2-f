# 阶段二：标准应用插件模板骨架实现

> 阶段目标：在不引入多模板与复杂集成的前提下，完成 `app-standard` 标准应用插件模板及基础渲染能力，使其在手工调用时即可生成可运行、可构建、可测试的工程。

---

## 1. 阶段范围

- 在本仓内创建标准应用插件模板目录与文件：`templates/app-standard/`；
- 为脚手架核心层实现最小可用的模板渲染能力：
  - 支持 `CreateAppProjectOptions` → 工程目录的生成流程；
  - 不与 `chips dev` 直接集成（集成工作在阶段三进行）；
- 为 `app-standard` 模板补齐：
  - Manifest、package、TS 配置与 chips 配置模板；
  - React + 组件库 + 主题 + 多语言 + Bridge/API/SDK 调用示例；
  - 最小可用的单元测试与 E2E 测试骨架。

---

## 2. 关键任务

### 2.1 仓库初始化与基础配置

- 在 `chips-scaffold-app` 根目录创建：
  - `package.json`：声明包名、基本脚本（lint/test/build 占位即可，但需可运行）；
  - `tsconfig.json`：脚手架实现代码的 TS 配置；
  - `.eslintrc` / `.prettierrc`（如生态已有统一模板，可复用）。
- 初始化 `src/core/`、`tests/` 等目录结构，但不写入临时代码。

### 2.2 app-standard 模板资产落地

- 按《技术文档/02-模板目录与文件组织设计》创建 `templates/app-standard/` 目录结构：
  - `manifest.yaml.tpl`、`package.json.tpl`、`tsconfig.json.tpl`、`chips.config.mjs.tpl`、`README.md.tpl` 等；
  - `src/`、`config/`、`i18n/`、`tests/`、`assets/` 子目录与对应模板文件；
- 确保模板内容符合：
  - 应用插件开发规范；
  - Manifest 配置规范；
  - 组件库与主题接入规范；
  - 多语言与前后端分离规范。

### 2.3 渲染核心最小实现

- 在 `src/core/` 中实现：
  - 模板元数据加载与验证函数；
  - 占位符替换与文件复制函数；
  - `createAppProject(options)` 的最小实现（接受参数、渲染 `app-standard` 模板到给定目录）。
- 暂不暴露对 `chips dev` 的公共 API，仅提供内部可调用函数，便于阶段三集成。

### 2.4 基础测试与自测脚本

- 单元测试：
  - 针对占位符替换与模板元数据解析实现测试；
- 模板一致性测试：
  - 检查 `app-standard` 模板目录完整性与关键文件存在；
- 生成工程自测脚本：
  - 在临时目录中调用内部 `createAppProject` 生成工程；
  - 在生成工程中执行 `npm install` + `npm test` + `npm run lint`（可配置跳过较重命令，视 CI 能力而定）。

---

## 3. 依赖与前置条件

- 阶段一文档与规划已完成，且未发现阻断脚手架实现的架构级缺陷；
- 生态对 React、组件库、主题、测试框架（Vitest）等技术选型已冻结或有明确临时口径；
- 如在实现过程中发现模板所需能力在 Host/SDK/组件库中尚未落地，需立即：
  - 在问题工单中登记；
  - 评估是否可以通过“示例占位”暂缓相关调用（仍需保证生成工程可运行）；
  - 或将相关模板功能下沉至后续阶段。

---

## 4. 验收标准

- `templates/app-standard/` 目录结构与内容完整，与技术文档描述一致；
- `src/core/` 中已实现最小可用的 `createAppProject`，可在本地通过脚本调用生成工程；
- 从模板生成的工程在本机（至少一个支持平台）满足：
  - `npm install` 可成功；
  - `npm test` 能运行并通过模板自带用例；
  - `npm run lint` 无错误；
- 本阶段新增的代码与模板已通过本仓的基础质量门禁（lint + 单元测试）。

---

## 5. 输出物

- `templates/app-standard/` 标准应用插件模板；
- `src/core/` 中的模板渲染最小实现；
- 覆盖模板渲染与模板一致性的单元/集成测试；
- 阶段总结，记录：
  - 模板设计中做出的关键取舍（例如哪些能力暂不内建）；
  - 与后续阶段（扩展模板与 CLI 集成）的依赖关系与注意事项。
