# 阶段四：测试体系与质量门禁

> 阶段目标：在前几阶段代码与模板基础上，建立完整的测试体系与发布前质量门禁，确保脚手架与生成工程在演进过程中保持稳定与可回归。

---

## 1. 阶段范围

- 完成本仓单元测试、模板一致性测试、集成测试的用例补齐与覆盖率提升；
- 建立“生成工程 E2E 测试”脚本与 CI 流程；
- 定义并在 CI 中落地发布前质量门禁脚本与策略；
- 与相关仓库（特别是 `chips dev` 与 Host/SDK）协作，确保接口变更能通过测试及时发现。

---

## 2. 关键任务

### 2.1 单元测试与模板一致性测试完善

- 单元测试：
  - 覆盖模板元数据加载、占位符替换、错误码映射等核心逻辑；
  - 覆盖多模板场景下的边界情况（未知模板 ID、重复 ID 等）。
- 模板一致性测试：
  - 针对所有模板目录（包括新增变体）执行完整性检查；
  - 检测未使用或拼写错误的占位符；
  - 校验模板结构与技术文档的一致性。

### 2.2 集成测试与 E2E 测试脚本

- 集成测试：
  - 在临时目录对 `createAppProject` 执行多组参数测试；
  - 验证生成工程关键文件存在且可解析。
- E2E 脚本：
  - 实现 `scripts/run-generated-e2e.mjs` 或等价工具；
  - 对每个模板执行“创建 → 安装依赖 → lint/test/build/validate”的全链路验证；
  - 支持通过环境变量控制是否运行耗时较长的步骤（例如在快 CI 与完整回归模式之间切换）。

### 2.3 CI 集成与质量门禁

- 在 CI 配置中接入：
  - `npm run lint`；
  - `npm test`（包含单元与模板一致性测试）；
  - （在适当分支/触发条件下）`npm run test:templates` 与 `npm run test:e2e`；
- 定义并文档化发布前必须执行的流水线：
  - 至少在发布分支或打标签时执行完整的 E2E 测试；
  - 清晰声明如何在本地复现 CI 流程。

### 2.4 与生态其他仓库的联动测试

- 与 `chips dev` 仓库协作：
  - 在其 CI 中增加“通过 `chips dev create` 使用本脚手架创建工程并执行基本命令”的测试；
  - 确保命令行交互与 API 变更能够被及时发现。
- 与 Host/SDK/组件库团队建立变更通知机制：
  - 当对外契约（例如 Bridge API、Manifest 字段）发生调整时，触发脚手架测试回归；
  - 必要时在问题工单中同步记录影响范围。

---

## 3. 依赖与前置条件

- 阶段二、三相关实现已基本完成并通过基础测试；
- CI 基础设施可用，支持运行 Node/Vitest/必要的 E2E 工具；
- 各相关仓库（`chips dev`、Host/SDK 等）同意在自己的 CI 流程中配合添加联动测试（如有需要）。

---

## 4. 验收标准

- 本仓在目标分支上：
  - `npm run lint`、`npm test`、`npm run test:templates` 全部通过；
  - 在至少一种受支持平台上，`npm run test:e2e` 能够完整运行并通过；
- 对每个模板至少有一条 E2E 测试用例覆盖从创建到 validate 的完整链路；
- CI 配置中定义了清晰的流水线阶段（lint/test/e2e）与触发规则；
- 文档（特别是《技术文档/05-测试策略与质量门禁设计》）已更新为与实际实现一致。

---

## 5. 输出物

- 本仓完整的测试用例集与质量门禁脚本；
- 更新后的 CI 配置文件（路径与内容视实际工具链而定）；
- 阶段总结，说明：
  - 测试覆盖范围与已知限制；
  - 对其他仓库（`chips dev`、Host/SDK 等）联动测试的现状；
  - 后续可考虑的测试增强方向（如性能回归测试等）。
