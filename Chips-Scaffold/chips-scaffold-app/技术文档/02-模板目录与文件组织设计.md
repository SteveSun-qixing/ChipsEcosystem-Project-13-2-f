# 模板目录与文件组织设计

> 文档状态：设计基线稿  
> 说明：本文件定义 `chips-scaffold-app` 仓库内部模板组织方式以及生成工程的目录结构约定，是实现与演进的参考基线。

---

## 1. 仓库内部目录结构规划

### 1.1 顶层目录规划

`chips-scaffold-app` 仓库在完成实现后的预期顶层目录结构如下（当前阶段仅部分目录存在）：

```text
chips-scaffold-app/
├─ templates/                # 模板资产层（按模板 ID 划分）
│  ├─ app-standard/          # 标准应用插件模板
│  └─ ...                    # 未来扩展模板
├─ src/                      # 脚手架实现代码
│  ├─ core/                  # 模板元数据与渲染管线
│  └─ cli/                   # 与 chips dev 的集成适配层
├─ tests/                    # 脚手架实现的单元与集成测试
├─ 需求文档/                 # 本仓需求文档
├─ 技术文档/                 # 本仓技术文档
├─ 开发计划/                 # 本仓开发计划
├─ package.json              # 脚手架自身 NPM 包定义（后续阶段添加）
├─ tsconfig.json             # 脚手架实现代码的 TS 配置
└─ README.md                 # 仓库级说明（面向生态内部）
```

约束：

- 模板资产与脚手架实现必须物理分离：`templates/` 下不得包含脚手架实现代码；
- 文档与实现同仓管理，但发布到 NPM 时仅打包必要的实现代码与模板资产；
- 如需新增辅助脚本（例如 CI 脚本、示例生成脚本），放入 `scripts/` 目录，并在技术文档中说明用途。

### 1.2 模板目录命名规范

- 所有模板目录放置在 `templates/` 下，命名规则：
  - 使用小写短横线风格，例如：`app-standard`、`app-viewer`；
  - 前缀 `app-` 表示应用插件模板类型，后缀描述场景；
- 每个模板目录内部必须包含一个元数据文件（具体格式在《03-模板参数化与渲染引擎设计》中定义），用于声明：
  - 模板 ID、名称、描述；
  - 支持的特性（是否集成 SDK、组件库、多窗口等）；
  - 模板版本号（便于后续迁移与兼容性判断）。

## 2. 模板目录结构设计（以 app-standard 为例）

以标准应用插件模板 `templates/app-standard/` 为例，其内部目录结构规划如下：

```text
templates/app-standard/
├─ .eslintrc.cjs.tpl         # 工程级 ESLint 配置模板
├─ manifest.yaml.tpl          # 插件 Manifest 模板（含占位符）
├─ package.json.tpl           # NPM 配置模板
├─ tsconfig.json.tpl          # TS 配置模板
├─ chips.config.mjs.tpl       # 与 chips dev 集成的配置模板
├─ README.md.tpl              # 生成工程 README 模板
├─ src/                       # 应用源码模板
│  ├─ main.tsx.tpl            # 入口文件
│  ├─ App.tsx.tpl             # 根组件
│  ├─ routes/                 # 路由与页面骨架（如采用）
│  ├─ components/             # 示例组件
│  ├─ hooks/                  # 示例 hooks（如 useChipsClient 等）
│  └─ layouts/                # 布局与外壳组件
├─ config/                    # 配置模板
│  ├─ app-config.ts.tpl       # 应用级配置（Feature Flag 等）
│  └─ logging.ts.tpl          # 日志封装
├─ i18n/                      # 多语言资源模板
│  ├─ zh-CN.json.tpl
│  └─ en-US.json.tpl
├─ assets/                    # 静态资源模板
│  └─ icons/
│     └─ app-icon.ico         # 应用快捷方式图标
├─ tests/                     # 测试模板
│  ├─ unit/                   # 单元测试
│  │  └─ app.test.tsx.tpl
│  └─ e2e/                    # 端到端测试
│     └─ basic-flow.test.ts.tpl
└─ .gitignore.tpl             # Git 忽略规则模板
```

说明：

- 所有模板文件使用 `.tpl` 后缀区分于最终生成文件，渲染后去掉 `.tpl`；
- 模板内部使用统一的占位符语法（在《03-模板参数化与渲染引擎设计》中定义），避免各文件采用不同占位风格；
- 目录结构与 `需求文档/02-应用插件脚手架功能需求规格.md` 中对生成工程结构的描述保持一致。

## 3. 生成工程目录结构约定

模板渲染完成后，开发者最终看到的应用插件工程目录结构预计如下（略去部分细节文件）：

```text
my-app/
├─ manifest.yaml
├─ package.json
├─ tsconfig.json
├─ chips.config.mjs
├─ README.md
├─ src/
│  ├─ main.tsx
│  ├─ App.tsx
│  ├─ i18n.ts
│  ├─ routes/
│  │  └─ index.tsx
│  ├─ components/
│  │  └─ ExampleCard.tsx
│  ├─ hooks/
│  │  └─ useChipsClient.ts
│  └─ layouts/
│     └─ ShellLayout.tsx
├─ config/
│  ├─ app-config.ts
│  └─ logging.ts
├─ i18n/
│  ├─ zh-CN.json
│  └─ en-US.json
├─ tests/
│  ├─ unit/
│  │  └─ app.test.tsx
│  └─ e2e/
│     └─ basic-flow.test.ts
└─ assets/
   └─ icons/
      └─ app-icon.ico
```

约束：

- `manifest.yaml` 必须位于工程根目录，符合《Manifest配置规范》，并在标准应用模板中内置 `ui.launcher` 与 `ui.window.chrome` 基线；
- `src/` 为主入口源码目录，默认入口文件为 `src/main.tsx`（可在技术文档与 `chips.config` 中统一约定）；
- `config/`、`i18n/`、`tests/`、`assets/` 目录结构需保持稳定，以便未来工具链可以基于这些约定做自动化分析与操作（如多语言抽取、测试扫描等）。

## 4. 模板文件类型与处理策略

根据文件在生成工程中的角色，模板文件分为以下几类：

1. **配置文件模板**  
   - 如 `.eslintrc.cjs.tpl`、`manifest.yaml.tpl`、`package.json.tpl`、`tsconfig.json.tpl`、`chips.config.mjs.tpl`；
   - 含有较多占位符（插件 ID、名称、版本、脚本命令等）；
   - 渲染时需做严格的 Schema 校验，避免生成不合法配置。

2. **源码文件模板**  
   - 如 `src/*.tsx.tpl`、`config/*.ts.tpl`；
   - 主要包含：
     - React 组件骨架；
     - hooks 与工具函数；
     - Bridge/API/SDK 调用示例；
     - 标题栏拖拽区与主题事件订阅等窗口宿主化基线；
     - `platform.getLaunchContext()` 启动上下文示例；
   - 必须为有效的 TypeScript/TSX 代码，渲染后可直接通过编译与 lint 检查。

3. **资源与文案模板**  
   - 如 `i18n/*.json.tpl`、`assets/*`；
   - 资源文件通常无需占位符，直接复制即可；
   - 快捷方式图标属于必须交付的正式资源，不能在生成工程中缺省；
   - 文案文件应尽量使用占位符或示例文案，避免与具体业务耦合。

4. **测试文件模板**  
   - 如 `tests/unit/*.test.tsx.tpl`、`tests/e2e/*.test.ts.tpl`；
   - 提供最小可运行示例，用于验证模板生成工程的基础质量门禁；
   - 不应引用尚未在模板中定义的模块或路径。

每类文件在渲染与验证过程中的处理规则将在《03-模板参数化与渲染引擎设计》与《05-测试策略与质量门禁设计》中补充细节。

## 5. 与其他脚手架仓库的对齐要求

为避免脚手架体系碎片化，`chips-scaffold-app` 在目录与文件组织上需要与其他脚手架仓库对齐：

- 顶层结构（`templates/src/tests/需求文档/技术文档/开发计划`）保持一致；
- 模板文件 `.tpl` 后缀与占位符语法在所有脚手架中统一；
- 生成工程的基本目录结构在不同插件类型间尽可能一致（例如都包含 `src/`、`config/`、`tests/`、`i18n/`），仅在业务特性上做差异化；
- 如需在多个脚手架之间共享模板片段（例如通用 README 或配置片段），可在 `Chips-Scaffold` 顶层设计共享目录或工具，但需在单独文档中规划。

本文件定义了模板与生成工程的目录与文件组织基线，后续实现与调整必须保持与本设计一致，如需变更需先更新文档并评估对现有工程与工具链的影响。
