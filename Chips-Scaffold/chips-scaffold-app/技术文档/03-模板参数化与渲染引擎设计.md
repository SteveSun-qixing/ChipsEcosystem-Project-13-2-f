# 模板参数化与渲染引擎设计

> 文档状态：设计基线稿  
> 说明：本文件定义模板参数 Schema、占位符语法以及渲染管线，为后续实现脚手架核心层提供约束与参考。

---

## 1. 参数模型与校验

### 1.1 参数 Schema 概览

脚手架核心层在创建应用插件工程时接收如下参数（与《01-应用插件脚手架总体架构设计》中的概念模型对齐）：

```ts
interface CreateAppProjectOptions {
  projectName: string;          // 工程目录名，如 'my-app'
  targetDir: string;            // 绝对路径，如 '/path/to/my-app'
  templateId: string;           // 模板 ID，如 'app-standard'
  pluginId: string;             // 插件 ID，反向域名格式
  displayName: string;          // 插件显示名称（默认语言）
  version: string;              // 初始版本号，语义化版本
  authorName?: string;          // 作者名
  authorEmail?: string;         // 作者邮箱
}
```

约束：

- `pluginId`：
  - 必须满足 Manifest 规范中的 ID 正则（反向域名；小写字母、数字与短横线）；
  - 不在本仓重复定义正则，直接引用生态共用规范中的约束。
- `version`：
  - 必须满足语义化版本规范 `MAJOR.MINOR.PATCH`；
  - 初始版本号默认为 `0.1.0`（可在 CLI 层提供覆盖能力）。
- `targetDir`：
  - 必须是可写路径；
  - 如路径不存在，可由脚手架创建；如路径存在且非空，应在 CLI 层进行确认或中止。

### 1.2 参数来源

参数可来自：

- 命令行参数（如 `chips dev create my-app --type app --template app-standard`）；
- 交互式问答（如在终端中逐项询问插件 ID、显示名称、作者等）；
- 配置文件或环境变量（未来可选扩展，例如为 CI 批量创建工程）。

脚手架核心层仅接收已整理好的 `CreateAppProjectOptions` 对象，不直接关心参数如何采集，保持与 `chips dev` 的解耦。

## 2. 占位符语法设计

### 2.1 基本语法

为保持简洁与可读性，模板文件采用如下占位符语法：

- 占位符格式：`{{ VARIABLE_NAME }}`（花括号包裹，允许空格）；
- 变量名使用大写蛇形命名，例如：
  - `{{ PROJECT_NAME }}`
  - `{{ PLUGIN_ID }}`
  - `{{ DISPLAY_NAME }}`
  - `{{ VERSION }}`
  - `{{ AUTHOR_NAME }}`

渲染前，脚手架核心层将构造一个键值对象：

```ts
interface TemplateContext {
  PROJECT_NAME: string;
  TARGET_DIR: string;
  TEMPLATE_ID: string;
  PLUGIN_ID: string;
  DISPLAY_NAME: string;
  VERSION: string;
  AUTHOR_NAME?: string;
  AUTHOR_EMAIL?: string;
}
```

并对模板文件执行文本替换。

### 2.2 特殊占位符与扩展

后续如需支持更复杂的逻辑（例如条件渲染、列表渲染），原则如下：

- 优先通过 CLI 选择不同模板变体（例如为“是否集成 SDK”提供不同模板），避免在一个模板文件中引入复杂逻辑；
- 如确需在模板中使用简单条件，可限定为有限的注释约定，避免引入重量级模板引擎；
- 引入额外模板引擎（如 Handlebars/EJS 等）前必须评估其是否在生态技术选型范围内，必要时提工单讨论。

当前设计阶段仅采用 **纯占位符替换**，不引入控制流语法。

## 3. 渲染管线设计

### 3.1 渲染步骤总览

渲染管线由脚手架核心层实现，典型流程如下：

1. **加载模板元数据**  
   - 根据 `templateId` 在 `templates/` 目录中找到对应模板；
   - 读取模板元数据文件，校验模板版本与兼容性信息；
   - 模板元数据文件仅用于脚手架内部，不得写入生成工程。

2. **构造模板上下文**  
   - 根据 `CreateAppProjectOptions` 构建 `TemplateContext`；
   - 根据需要派生额外字段（例如 NPM 包名、仓库 URL 等）。

3. **遍历模板文件树**  
   - 从模板根目录递归遍历所有文件；
   - 对每个文件计算目标路径：
     - 去掉 `.tpl` 后缀；
     - 允许在路径中使用占位符（例如 `src/{{ PROJECT_NAME }}/index.tsx.tpl`）。

4. **渲染与写入**  
   - 对文本文件执行占位符替换，将结果写入目标路径；
   - 对二进制文件（图片、字体等）直接复制，不做替换；
   - 写入前检查目标路径是否已存在，避免误覆盖。

5. **校验与总结**  
   - 在渲染完成后执行一轮快速自检：
     - `manifest.yaml` 是否存在且可被解析；
     - `package.json` 是否为有效 JSON；
     - `manifest.yaml` 中的 `ui.launcher.icon` 是否指向已生成的资源；
     - 必要脚本是否存在（`dev/build/test/validate` 等）；
   - 输出创建结果摘要（成功/失败、生成文件数、关键文件路径等）。

### 3.2 文本文件识别与编码

- 通过扩展名与简单的文件头判断文本/二进制类型：
  - `*.ts`/`*.tsx`/`*.json`/`*.yaml`/`*.md` 等始终按 UTF-8 文本处理；
  - `*.png`/`*.jpg`/`*.svg`/`*.ico`/`*.ttf` 等按二进制复制；
- 模板文件应统一使用 UTF-8 编码，避免在渲染过程中出现编码问题。

对快捷方式相关资源的补充要求：

- `assets/icons/app-icon.ico` 必须作为二进制文件直接复制，不允许被错误地当作文本文档做占位替换；
- 若模板未来引入 `icns/png` 等额外图标资源，也必须保持“原样复制”策略；
- 生成后校验阶段应确认默认图标文件存在，避免创建出来的工程在 Host 快捷方式治理链路中缺图。

### 3.3 错误处理策略

- 参数校验失败（例如 ID 不合法）应在渲染前即抛出错误，避免产生任何文件；
- 渲染过程中如遇文件系统错误（无法写入、权限不足等）：
  - 尽量在失败时记录已写入文件列表；
  - 由 CLI 层决定是否执行清理或提示用户手动清理；
- 对于模板本身不合法（例如缺失关键文件）等错误，应在 CI 与自测阶段尽早发现，避免在用户环境暴露。

## 4. 模板元数据与兼容性

### 4.1 元数据文件

每个模板目录下应包含一个元数据文件（形式待定，例如 `template.json`）：

```json
{
  "id": "app-standard",
  "name": "标准应用插件模板",
  "description": "包含 React + 组件库 + 多语言 + 测试骨架的标准应用插件模板",
  "version": "1.0.0",
  "supports": {
    "sdk": true,
    "componentLibrary": true,
    "multiWindow": false
  }
}
```

脚手架核心层使用该文件：

- 在 CLI 中向用户展示可用模板列表与描述；
- 在测试与 CI 中校验模板清单是否与文档描述一致；
- 在未来引入模板版本迁移工具时，作为版本兼容判断的基础。

### 4.2 模板版本与迁移

- 模板自身版本号与仓库版本号相互独立：
  - 可在一次仓库发布中包含多个模板版本变更；
  - 也可在不改变模板内容的前提下发布包含其他修改的仓库版本。
- 未来如需为已有工程提供“模板迁移向导”，需新增专门文档与工具，本阶段仅为模板版本预留字段，不实现迁移逻辑。

## 5. 与 vercel-* 组合与性能规范的对齐

虽然脚手架核心层本身不直接执行 React 代码，但模板中的 React 组件与页面结构必须提前考虑性能与组合模式：

- 在设计模板组件 API 时：
  - 避免滥用 boolean props 控制行为，优先采用显式变体组件与组合模式（参考 `vercel-composition-patterns`）；
  - 将状态管理集中在 Provider 或上层组件，通过 context/props 下发，避免难以维护的深层状态共享；
- 在编写示例代码时：
  - 避免在组件中引入不必要的同步阻塞与瀑布式异步请求（参考 `vercel-react-best-practices` 中的 waterfall 相关规则）；
  - 提前为后续添加 Suspense、并发特性留出结构空间（例如通过清晰的数据加载层与展示层划分）。

这些约束不会在脚手架渲染引擎中直接编码，但会体现在模板源码文件的具体写法上，并通过测试与代码审查确保落地。

本文件为模板参数化与渲染引擎的设计基线，后续实现必须按照本设计完成具体 TypeScript 代码，并在测试与文档中验证其行为与约束一致。
