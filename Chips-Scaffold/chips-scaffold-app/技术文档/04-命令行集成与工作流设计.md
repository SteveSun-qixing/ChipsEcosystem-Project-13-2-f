# 命令行集成与工作流设计

> 文档状态：设计基线稿  
> 说明：本文件描述 `chips-scaffold-app` 与 `chips dev` 开发者命令行工具的集成方式，以及典型开发工作流。

---

## 1. 与 chips dev 的职责划分

根据《Chips Dev 开发者命令行手册》，`chips dev` 负责提供如下能力：

- `chips dev create`：创建新插件工程；
- `chips dev server`：启动开发服务器；
- `chips dev build`：构建 `cpk` 插件包；
- `chips dev test`：运行测试；
- `chips dev validate`：验证插件是否符合规范。

在该工作流中：

- `chips-scaffold-app` 负责：
  - 定义应用插件模板；
  - 提供创建工程所需的渲染能力与参数约束；
- `chips dev` 负责：
  - 解析命令行参数与交互式输入；
  - 调用 `chips-scaffold-app` 提供的接口生成工程；
  - 在生成工程目录中执行 `server/build/test/validate` 等命令。

两者通过 TypeScript API 而非进程间通信集成，避免不必要的复杂性。

## 2. 对外导出接口设计

### 2.1 模块入口

`chips-scaffold-app` 作为生态根工作区内的一方脚手架包，对外提供一个主入口模块，例如：

```ts
// index.ts
export * from './src/cli/app-scaffold-api';
```

在 `src/cli/app-scaffold-api.ts` 中导出与 `chips dev` 集成所需的函数：

```ts
export interface AppScaffoldTemplateSummary {
  id: string;
  name: string;
  description: string;
}

export interface CreateAppProjectOptions {
  projectName: string;
  targetDir: string;
  templateId: string;
  pluginId: string;
  displayName: string;
  version: string;
  authorName?: string;
  authorEmail?: string;
}

export interface CreateAppProjectResult {
  projectDir: string;
  templateId: string;
  filesCreated: number;
}

export function listAppTemplates(): Promise<AppScaffoldTemplateSummary[]>;

export function createAppProject(options: CreateAppProjectOptions): Promise<CreateAppProjectResult>;
```

说明：

- `listAppTemplates` 供 `chips dev create` 在交互式模式下展示可选模板列表；
- `createAppProject` 执行模板渲染与文件写入，不负责参数采集与命令行交互。

### 2.2 错误模型

- `createAppProject` 抛出的错误必须使用标准错误对象：

  ```ts
  interface StandardError {
    code: string;
    message: string;
    details?: unknown;
  }
  ```

- 典型错误码包括：
  - `INVALID_ARGUMENT`：参数不合法（例如 pluginId 格式错误）；
  - `TARGET_DIR_EXISTS`：目标目录已存在且非空；
  - `TEMPLATE_NOT_FOUND`：指定模板不存在；
  - `TEMPLATE_INVALID`：模板缺少关键文件或元数据不合法；
  - `FS_ERROR`：文件系统相关错误（写入失败等）。

`chips dev` 在捕获错误后负责将错误信息转换为用户可读的命令行提示。

## 3. create 命令工作流

### 3.1 标准工作流（交互式）

1. 用户执行：

   ```bash
   chips dev create my-app --type app
   ```

2. `chips dev`：
   - 校验 `--type app`；
   - 通过 `listAppTemplates()` 获取可用模板列表并展示（如仅一个模板则可直接选中）；
   - 通过交互式命令行询问：
     - 插件 ID（建议默认 `com.example.my-app` 格式）；
     - 插件显示名称；
     - 作者姓名与邮箱；
   - 构造 `CreateAppProjectOptions` 对象。

3. `chips dev` 调用 `createAppProject(options)`：
   - 由 `chips-scaffold-app` 完成模板渲染与文件生成；
   - 返回 `projectDir` 与 `filesCreated` 等信息。

4. `chips dev` 输出：
   - 创建成功提示与项目路径；
   - 下一步建议命令：

   ```text
   ✔ 已创建应用插件工程 my-app

   下一步：
     cd <生态根工作区>
     npm install
     cd my-app
     npm run dev   # 或 chips dev server
   ```

### 3.2 非交互式工作流（可选）

未来可支持通过命令行一次性传入所有参数，例如：

```bash
chips dev create my-app \
  --type app \
  --template app-standard \
  --plugin-id com.example.my-app \
  --display-name "My App" \
  --author "Alice <alice@example.com>"
```

在该模式下：

- `chips dev` 需要完成参数校验与默认值填充；
- 如参数不完整或不合法，应直接在 CLI 层提示错误，不调用 `createAppProject`。

本阶段仅在设计上预留该能力，实际 CLI 行为由 `chips dev` 仓库具体实现。

## 4. 与 build/test/validate 工作流的关系

模板生成工程后，开发者将通过 `chips dev` 执行如下命令：

- `chips dev server`：启动开发服务器；
- `chips dev build`：构建 `cpk` 包；
- `chips dev test`：运行测试；
- `chips dev validate`：校验清单、权限与依赖。

`chips-scaffold-app` 在这些命令中的职责体现在：

- 通过模板确保生成工程包含：
  - 合法且完整的 `manifest.yaml`；
  - 正确配置的 `chips.config.*`；
  - 合理的 NPM 脚本映射；
  - 最小可用的测试与 lint 配置；
- 生成工程默认只输出 README 与运行时必要目录，不预创建 `需求文档/`、`技术文档/`、`技术手册/`、`开发计划/`；
- 不直接实现 `build/test/validate` 逻辑，只负责让生成工程“默认即通过”。

任何对模板结构或配置的变更，都必须验证不会破坏上述命令的基本工作流，并在《05-测试策略与质量门禁设计》中补充对应测试用例。

## 5. 本地与 CI 集成

### 5.1 本地开发者使用

典型本地开发工作流：

1. 创建工程：`chips dev create my-app --type app`；
2. 在生态根工作区执行依赖安装：`cd <ecosystem-root> && npm install`；
3. 进入工程目录并启动开发服务器：`cd my-app && npm run dev`；
4. 编写业务代码与测试；
5. 运行测试与校验：`npm test`、`npm run lint`、`chips dev validate`；
6. 构建发布包：`chips dev build`。

脚手架的设计目标是让上述步骤在“空工程”状态下即可顺利执行，不需要开发者先处理脚手架遗留问题。

### 5.2 CI 中的验证

在 CI 或本仓自测中，建议增加如下自动化步骤：

1. 在临时生态工作区中通过 `chipsdev create app` 生成工程，并完成工作区注册；
2. 在工作区根目录执行：
   - `npm install`（可使用缓存加速）；
3. 再在生成工程中执行：
   - `npm test`；
   - `npm run lint`；
   - `chips dev build`；
   - `chips dev validate`；
4. 若任一步骤失败，则阻断本仓的发布或合并。

上述 CI 步骤的具体实现将在《05-测试策略与质量门禁设计》中详细说明。

## 6. 兼容性与演进

- 本设计不涉及对外接口稳定性的承诺（即 `chips-scaffold-app` 本身为内部依赖），但应尽可能保持 `listAppTemplates` 与 `createAppProject` 两个接口的稳定，以降低 `chips dev` 与脚手架之间的耦合成本；
- 如需对接口做不兼容调整，应：
  - 先更新本仓技术文档与开发计划，说明变更原因与影响范围；
  - 同步更新 `chips dev` 仓库的调用逻辑与文档；
  - 通过 CI 验证现有工作流未被破坏；
- 若未来需要将脚手架能力开放给第三方工具使用（例如 IDE 插件），需单独设计对外 API 或命令行接口，并在生态共用文档中沉淀正式口径。

本文件定义了 `chips-scaffold-app` 与 `chips dev` 的集成边界与工作流，为后续实现提供结构化指引。
