# 应用插件脚手架功能需求规格

> 文档状态：需求规格候选稿  
> 说明：本文件定义应用插件脚手架的功能需求；跨生态公开接口与协议口径以生态共用文档为准。

---

## 1. 能力域总览

| 能力域 | 核心目标 |
|---|---|
| 脚手架核心能力 | 通过 `chips dev create` 等命令创建应用插件工程，并完成基本参数收集与模板渲染 |
| 模板结构与内容 | 提供符合生态规范的项目目录结构、配置文件与示例代码 |
| 应用基础设施 | 在模板中内建多语言、主题、日志与错误处理、窗口管理与 Host 能力接入骨架 |
| 质量与门禁支撑 | 在模板中内建测试、校验与构建脚本，确保生成工程可立即通过基础质量门禁 |
| 扩展与文档支持 | 支持未来多模板扩展，并提供清晰的使用说明与自定义扩展入口 |

以下按能力域描述功能需求，采用 `FR-SCAFFOLD-*` 编号。

---

## 2. 脚手架核心能力需求（FR-SCAFFOLD-CORE）

### FR-SCAFFOLD-CORE-001 项目创建入口

- 脚手架必须支持通过 `chips dev create` 命令创建应用插件工程：
  - 用户至少可以指定项目名称与插件类型（`app`）。
  - 命令行工具负责解析参数并调用本仓库导出的模板生成能力。
- 首版可只支持交互式问答或最小参数集，但需要在技术文档中固定接口约定，避免后续破坏兼容性。

### FR-SCAFFOLD-CORE-002 模板选择与变体

- 首版至少提供一个 **标准应用插件模板**（例如 `app-standard`），满足通用业务应用场景；
- 后续阶段可新增模板变体（例如 `app-viewer`、`app-tool-panel` 等），脚手架需要：
  - 为模板分配稳定的标识符与描述；
  - 支持在 `chips dev create` 中通过参数或交互选择模板；
- 当模板变体尚未实现时，不得在文档或命令行帮助中对外暴露对应选项。

### FR-SCAFFOLD-CORE-003 参数化与占位符

- 模板必须支持对至少以下信息进行参数化渲染：
  - 插件 ID（反向域名格式，例如 `com.example.my-app`）；
  - 插件显示名称（多语言可本地化的名称）；
  - 初始版本号（默认 `0.1.0` 或统一约定的开发期版本）；
  - 作者信息（名称、邮箱等）；
  - 项目根路径与包名（供 `package.json` 使用）。
- 参数化渲染必须保持幂等：同一参数集生成的项目结构与内容必须一致。

### FR-SCAFFOLD-CORE-004 执行结果与错误提示

- 创建命令执行完成后必须输出：
  - 新项目的绝对路径；
  - 下一步建议命令（例如 `cd <生态根工作区> && npm install && cd <project> && chips dev server`）；
- 创建失败时：
  - 给出明确的错误信息（如路径已存在、参数不合法等）；
  - 不得生成残缺的工程结构；若已生成部分文件，应在安全前提下回滚或提示用户手工清理方案。

## 3. 模板结构与内容需求（FR-SCAFFOLD-TEMPLATE-STRUCT）

### FR-SCAFFOLD-TEMPLATE-STRUCT-001 项目基础目录结构

- 模板生成的工程至少应包含以下目录与文件（可根据后续技术文档进一步细化）：

  - 根目录：
    - `.eslintrc.cjs`：工程级代码规范配置，供 `chips dev lint` 直接复用；
    - `package.json`：NPM 包描述与脚本定义；
    - `tsconfig.json`：TypeScript 配置；
    - `chips.config.*`：与 `chips dev` 集成的构建/运行配置；
    - `manifest.yaml`：插件 Manifest（`type: app`）；
    - `README.md`：项目使用说明；
  - 源码目录：`src/`
    - `main.tsx` 或等价入口文件；
    - `App.tsx` 及页面组件结构；
    - `features/`：按领域划分的业务模块目录（可为空但结构需预留）；
    - `hooks/` 与 `components/`：示例 Hook 与组件目录；
  - 配置与资源目录：
    - `config/`：应用级配置（例如多语言、主题、Feature Flag 等）；
    - `i18n/`：多语言资源文件；
    - `assets/`：静态资源；
  - 测试目录：`tests/`
    - `unit/`：单元测试示例；
    - `e2e/`：端到端测试示例。

- 所有目录名称与文件命名必须在技术文档中固定下来，作为后续演进与生态对接的基础约定。

### FR-SCAFFOLD-TEMPLATE-STRUCT-002 Manifest 与插件元信息

- 模板必须生成符合《Manifest配置规范》的 `manifest.yaml` 文件，至少包含：
  - `id`、`name`、`version`、`type: app`、`entry` 等必填字段；
  - `ui.layout` 固定布局声明；
  - `ui.launcher.displayName` 与 `ui.launcher.icon`，用于 Host 创建应用快捷方式；
  - 标准应用模板默认携带 `ui.window.chrome` 基线，用于声明白色窗口外壳与原生窗口标题栏；
  - 合理的默认 `description` 与 `author` 信息占位；
  - 必要且最小化的 `permissions` 列表；
- 模板不得默认启用高风险权限（如文件写入、网络访问等）；如示例代码确需某权限，必须在文档中说明用途与风险。
- 模板必须同时生成 `assets/icons/app-icon.ico`，并保证 `manifest.yaml` 中的 `ui.launcher.icon` 与该资源路径一致。

### FR-SCAFFOLD-TEMPLATE-STRUCT-003 NPM 脚本与 chips 命令集成

- `package.json` 中必须预置以下脚本（名称可根据生态统一命名调整）：
  - `dev`：启动开发服务器（例如 `chips dev server` 或等价命令）；
  - `build`：构建插件包（例如 `chips dev build`）；
  - `test`：运行单元测试与基础集成测试；
  - `lint`：运行代码规范检查；
  - `validate`：执行插件规范校验（`chips dev validate` 或等价命令）。
- 所有脚本必须在模板生成后即可无错误运行（安装依赖后）。

### FR-SCAFFOLD-TEMPLATE-STRUCT-004 TypeScript 与编码规范

- 模板必须默认启用 TypeScript，并提供：
  - 严格模式配置（`strict: true` 或与生态统一的配置）；
  - 路径别名与编译输出目录约定；
- 模板代码需符合生态统一的编码规范（ESLint/Prettier 配置），并在技术文档中明确规范来源与规则集合。

## 4. 应用基础设施需求（FR-SCAFFOLD-APP-INFRA）

### FR-SCAFFOLD-APP-INFRA-001 多语言与国际化

- 模板必须内建多语言支持：
  - 所有用户可见文案通过 i18n 资源文件管理，不得在组件中硬编码文本；
  - 至少提供示例的 `zh-CN` 与 `en-US` 语言包；
  - 提供简单的语言切换示例（菜单或设置入口）。

### FR-SCAFFOLD-APP-INFRA-002 主题与组件库接入

- 模板必须：
  - 在根组件中接入薯片组件库 Provider 与主题运行时；
  - 通过 `theme.changed` 事件驱动主题刷新；
  - 使用组件库基础组件（如 Button/Input/Dialog 等）构建示例 UI；
  - 不在业务代码中硬编码颜色、阴影、圆角等视觉常量，而是通过主题 token 获取。

### FR-SCAFFOLD-APP-INFRA-003 Bridge/API/SDK 接入示例

- 模板必须包含至少一个完整的 Host 能力调用示例，例如：
  - 通过 Bridge API 或 SDK 读取当前工作区的卡片/箱子列表；
  - 在 UI 中展示结果，并对错误进行标准化提示；
- 示例代码需要：
  - 走统一的 `window.chips.*` 或 SDK 客户端调用链；
  - 捕获并展示标准错误对象（`code/message`），不向用户泄露底层堆栈；
  - 在技术文档中标注依赖的 Host 路由与权限。
- 标准应用模板还必须包含 `client.platform.getLaunchContext()` 或等价 Bridge 读取示例，帮助开发者理解快捷方式启动、文件关联启动与开发入口启动的上下文恢复方式。

### FR-SCAFFOLD-APP-INFRA-004 窗口与路由骨架

- 模板必须提供一个基础的应用外壳：
  - 顶部或侧边导航区域；
  - 主内容区；
  - 当模板切换为 `ui.window.chrome.titleBarStyle: hidden` 等自绘标题栏模式时，顶部区域必须预留可拖拽标题栏空间。
- 内部可使用 React Router 等路由方案，但必须符合生态技术选型；如路由库尚未在技术选型中确认，则应优先采用最简页面结构，并在问题工单中记录路由技术选型议题。

### FR-SCAFFOLD-APP-INFRA-005 错误边界与日志

- 模板必须内建：
  - 全局 Error Boundary 组件（可使用组件库提供的 `ErrorBoundary`）；
  - 统一的日志输出工具（封装 `console.*`，为未来接入 Host 日志服务预留接口）；
- 示例代码中应展示：
  - 捕获组件内部异常并展示友好错误提示；
  - 在开发模式下输出更详细的调试信息。

## 5. 质量与门禁支撑需求（FR-SCAFFOLD-QUALITY）

### FR-SCAFFOLD-QUALITY-001 单元测试与组件测试

- 模板必须内建单元测试基础设施：
  - 测试框架使用 Vitest，与生态开发环境配置指南保持一致；
  - 至少提供 1–2 个示例测试文件，覆盖：
    - 基础组件渲染与交互；
    - Bridge/API/SDK 调用的错误处理逻辑（可通过 mock 实现）。

### FR-SCAFFOLD-QUALITY-002 端到端测试

- 模板必须预留端到端测试目录与示例：
  - 使用生态统一的 E2E 工具（若已选型，例如 Playwright），否则在文档中标记待接入，并通过问题工单推动选型；
- 示例用例至少覆盖“启动应用 → 打开主页面 → 触发一次 Host 能力调用并看到结果”。
- 模板级结构检查至少覆盖：
  - 生成工程包含 `assets/icons/app-icon.ico`；
  - `manifest.yaml` 包含 `ui.launcher.displayName` 与 `ui.launcher.icon`；
  - 示例入口可读取启动上下文而不报错。

### FR-SCAFFOLD-QUALITY-003 校验与构建门禁

- 模板必须保证在依赖安装完成后：
  - `npm run lint` / `chips dev lint` 无错误；
  - `npm test` / `chips dev test` 用例全部通过；
  - `chips dev validate` 对 Manifest、权限、依赖等校验全部通过；
  - `chips dev build` 可生成合法的 `cpk` 包。
- 如生态尚未提供某个门禁命令，则在技术文档与开发计划中明确依赖关系，并通过问题工单推进。

## 6. 扩展与文档支持需求（FR-SCAFFOLD-EXTENSIBILITY）

### FR-SCAFFOLD-EXTENSIBILITY-001 多模板扩展能力

- 脚手架内部设计必须支持未来新增模板而不破坏现有接口：
  - 模板元数据（ID、名称、描述、适用场景）应以结构化方式存储；
  - `chips dev create` 接口与内部实现需预留模板列表查询与筛选能力；
- 新增模板时必须更新：
  - 本仓技术文档中与模板相关章节；
  - 生态共用文档中如有涉及到模板清单或示例的部分。

### FR-SCAFFOLD-EXTENSIBILITY-002 使用说明与示例

- 模板生成的工程必须包含 `README.md`，至少说明：
  - 如何安装依赖、运行开发服务器、执行测试与构建；
  - 模板中各主要目录与文件的作用；
  - 如何扩展页面、接入新的 Host 能力、增加多语言条目等；
- 本仓技术文档中需要提供：
  - 面向生态内部的脚手架使用说明；
  - 面向未来公开的高层次说明可通过 `生态共用技术文档` 进行对外整理。

### FR-SCAFFOLD-EXTENSIBILITY-003 与其他脚手架仓库的一致性

- `chips-scaffold-app` 的能力规划与文档结构应与以下仓库保持一致：
  - `chips-scaffold-basecard`
  - `chips-scaffold-boxlayout`
  - `chips-scaffold-module`
  - `chips-scaffold-theme`
- 一致性体现在：
  - 需求/技术/开发计划三类文档均采用相同结构与命名约定；
  - 模板参数（例如插件 ID、名称、版本等）的命名与渲染机制尽量统一；
  - 如需在多个脚手架间共享规范，应优先沉淀到 `生态共用技术文档` 再在各仓引用。

以上功能需求构成应用插件脚手架项目的基础能力清单，后续技术方案与实现必须以本文件为依据进行细化与验证。如发现本文件与生态共用文档存在冲突，以生态共用文档与用户最新决策为准，并通过问题工单同步调整。
