# 阶段一：架构与需求冻结

> 阶段目标：在不编写实现代码的前提下，完成基础卡片插件脚手架的需求与技术方案基线，为后续模板与实现落地打下稳定基础。

---

## 1. 阶段范围

- 阅读与对齐以下文档体系：
  - `生态设计原稿/` 中与插件、卡片、SDK、渲染架构相关的所有章节；
  - `生态共用技术文档/插件开发`、`协议与接口标准`、`组件库`、`文件格式规范`、`命令行工具` 等；
  - `Chips-Host`、`Chips-SDK`、`Chips-ComponentLibrary` 的需求与技术文档中与基础卡片相关的部分；
  - 现有《项目日志与笔记/问题工单.md》中与脚手架、SDK、卡片开发体验相关的工单（如有）。
- 在本仓完成完整的文档与规划基线：
  - 需求文档 `00-03` 系列；
  - 技术文档 `00-05` 系列；
  - 开发计划 `00-05` 系列。
- 不编写任何脚手架实现代码或模板内容，仅在必要时创建空目录结构以支撑文档描述。

---

## 2. 关键任务

### 2.1 文档对齐与角色澄清

- 明确 `chips-scaffold-basecard` 在生态中的定位：
  - 仅作为开发期工具，不参与运行时；
  - 负责为 `type: card` 插件提供标准工程模板；
  - 与 `chips dev`、Host、SDK、组件库的边界与协作方式清晰可见。
- 对齐并记录：
  - 插件类型与生命周期（参考《插件架构设计》《插件开发规范》）；
  - 卡片插件开发规范与 Manifest/权限要求；
  - 卡片文件结构、卡片服务动作面与渲染链路；
  - CLI 能力与工作流（`chips dev create/build/test/validate`）。

### 2.2 本仓需求与技术文档落地

- 需求文档：
  - 《01-基础卡片插件脚手架需求总览》：背景、目标、范围、约束、干系人与里程碑；
  - 《02-基础卡片插件脚手架功能需求规格》：按能力域拆分脚手架与模板需求；
  - 《03-基础卡片插件脚手架非功能与质量验收需求》：性能、安全、可维护性、测试与 Git 规范。
- 技术文档：
  - 《01-基础卡片插件脚手架总体架构设计》：仓库角色与内部分层；
  - 《02-模板目录与文件组织设计》：模板与生成工程目录结构规划；
  - 《03-模板参数化与渲染引擎设计》：参数模型、占位符语法与渲染管线；
  - 《04-命令行集成与工作流设计》：与 `chips dev` 的集成接口与工作流；
  - 《05-测试策略与质量门禁设计》：测试层级与 CI/发布门禁。
- 开发计划：
  - 《00-开发计划索引》与阶段二至阶段五拆解文档。

### 2.3 阻塞项识别与问题工单

- 在阅读与设计过程中，如发现：
  - 生态文档中对基础卡片插件工程目录结构、模板体系缺乏正式口径；
  - `chips dev` 尚未提供卡片插件脚手架所需的扩展点或命令行参数规范；
  - 卡片服务、组件库或 SDK 缺少标准基础卡片模板所必需的能力；
- 则必须：
  1. 在《项目日志与笔记/问题工单.md》中登记新工单，说明问题与期望；
  2. 在 `项目日志与笔记/任务清单/` 下创建对应任务文件夹与任务说明文档；
  3. 在本阶段总结中记录阻塞项列表及其工单编号。

## 3. 验收标准

- 文档侧：
  - `chips-scaffold-basecard/需求文档/00-03` 均已创建且内容完整，无明显占位或 TODO；
  - `chips-scaffold-basecard/技术文档/00-05` 均已创建且能指导后续实现；
  - `chips-scaffold-basecard/开发计划/00-05` 明确后续阶段任务拆解与依赖关系。
- 一致性与边界：
  - 本仓文档未与生态共用文档产生明显冲突；
  - 对于需要跨仓库引用的信息，均以绝对路径或明确引用方式标注来源。
- 阻塞项：
  - 与基础卡片插件脚手架相关的潜在阻塞项已在问题工单中登记；
  - 当前阶段不会在存在重大不确定性的前提下继续推进实现代码。
