# 阶段二：标准基础卡片插件模板骨架实现

> 阶段目标：在不破坏全局规范的前提下，实现标准基础卡片插件模板的完整工程骨架，使其能够在本地完成一次端到端渲染/编辑链路。

---

## 1. 阶段范围

- 在 `chips-scaffold-basecard` 仓库中：
  - 创建 `templates/card-standard/` 目录与元数据文件；
  - 实现标准基础卡片插件模板的目录结构与模板文件；
  - 实现脚手架核心层的最小渲染能力（仅支持 `card-standard` 模板）；
  - 为模板生成工程补齐 README 与最小测试骨架。
- 不在本阶段：
  - 集成更多模板变体；
  - 与 `chips dev` CLI 完整对接（CLI 集成可在阶段三推进）。

---

## 2. 关键任务

### 2.1 模板目录与元数据落地

- 按《技术文档/02-模板目录与文件组织设计》创建 `templates/card-standard/` 目录；
- 编写 `template.json`（或等价元数据文件）：
  - 填写模板 ID、名称、描述、版本、支持特性等；
  - 确保字段与技术文档中的定义一致；
- 在本仓测试中增加模板元数据加载与校验用例。

### 2.2 模板文件与占位符实现

- 在模板目录中实现：
  - `manifest.yaml.tpl`、`package.json.tpl`、`tsconfig.json.tpl`、`chips.config.mjs.tpl`、`README.md.tpl`；
  - `src/index.ts.tpl`、`src/render/*`、`src/editor/*`、`src/schema/card-config.ts.tpl` 等源码模板；
  - `i18n/`、`templates/`、`tests/` 目录中的示例模板文件。
- 所有模板文件使用统一的占位符语法（`{{ VARIABLE_NAME }}`），并在技术文档中列出支持的变量清单。

### 2.3 脚手架核心层最小实现

- 在 `src/core/` 中实现：
  - 模板元数据加载函数；
  - 模板文件树遍历与占位符替换函数；
  - 对外暴露的 `createBasecardProject` 核心实现（目前仅支持 `card-standard` 模板）。
- 为核心函数编写单元测试：
  - 参数校验与错误码覆盖；
  - 占位符替换正确性；
  - 模板缺失或不合法时的错误行为。

### 2.4 标准基础卡片插件示例行为

- 在模板中提供一个最小可运行示例：
  - 渲染模块：渲染一个标题 + 主体内容的简单基础卡片视图；
  - 编辑模块：提供标题与主体内容的表单编辑界面，演示配置更新与本地校验；
  - 配置模板：默认配置文件与参数表说明。
- 通过 `npm run dev` 本地验证：
  - 能够看到基础卡片的渲染效果；
  - 能够在编辑界面修改配置并看到更新结果（即使当前不与 Host 真正集成，也需模拟回调链路）。

---

## 3. 验收标准

- 模板侧：
  - `templates/card-standard/` 目录存在且结构与技术文档描述一致；
  - 元数据文件可被解析并通过模板一致性测试；
  - 模板文件渲染后能生成完整工程，无 `.tpl` 遗留。
- 脚手架核心层：
  - 存在 `createBasecardProject` 实现；
  - 单元测试覆盖参数校验与基本渲染行为；
  - 集成测试能在临时目录中成功生成工程。
- 生成工程：
  - `npm install` 成功；
  - `npm test` 至少运行通过 1–2 个示例测试；
  - `chips dev build` 与 `chips dev validate` 在可用环境中通过（如暂不可用需在问题工单中记录）。

---

## 4. 输出物

- 标准基础卡片插件模板 `card-standard` 的完整目录与模板文件；
- 脚手架核心层最小实现与配套单元/集成测试；
- 一份阶段总结（建议记录在项目日志中），说明：
  - 模板工程结构与行为概述；
  - 已识别的风险与待办事项（例如与 Host/CLI 集成的依赖）。

本阶段完成后，可进入阶段三，扩展模板与完成 CLI 集成。
