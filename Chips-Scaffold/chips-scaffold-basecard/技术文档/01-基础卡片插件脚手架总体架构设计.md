# 基础卡片插件脚手架总体架构设计

> 文档状态：内部技术基线稿  
> 边界说明：仅描述 `Chips-Scaffold/chips-scaffold-basecard` 仓库内部角色与分层，不重复外部契约定义。

---

## 1. 仓库角色与生态定位

### 1.1 在 12 层架构中的位置

在薯片生态的 12 层架构中：

- L1–L9：由 `Chips-Host` 提供的运行时能力（PAL、Kernel、Host Services、Plugin Runtime、Bridge、Runtime Client、Declarative UI、Unified Rendering 等）；
- L10：由 `Chips-ComponentLibrary` 提供的无头组件与可访问性能力；
- L11：由主题包提供的视觉层；
- L12：由各类插件与页面业务代码组成，包括应用插件、基础卡片插件、布局插件等。

`Chips-Scaffold/chips-scaffold-basecard` **不直接参与任何运行时层级**，而是作为 **开发期工具仓库**，为 L12 的基础卡片插件（`type: card`）提供标准工程模板与初始化流程：

- 仓库产物（基础卡片插件模板工程）在开发期通过 `chips dev create --type card` 命令被复制到开发者本地；
- 生成出来的基础卡片插件工程，在运行时作为独立插件由 Host 加载，与 `Chips-Scaffold` 仓库本身无运行时依赖关系；
- 仓库自身仅在开发环境与 CI 中使用，不会被打包进最终用户环境。

### 1.2 与其他仓库的协作关系

| 仓库 | 角色 | 与 `chips-scaffold-basecard` 的关系 |
|---|---|---|
| `Chips-Host` | 主机运行时与对外接口（含卡片服务、渲染器等） | 生成的基础卡片插件在运行时依赖 Host，模板需遵守 Host 对外接口基线与卡片渲染/编辑架构设计 |
| `Chips-SDK` | 开发者 SDK 与封装 | 模板可选集成 SDK，用于通过 typed 客户端调用卡片服务、文件服务等能力 |
| `Chips-ComponentLibrary` | 无头组件库与 UI 能力 | 模板在渲染/编辑 UI 层默认集成组件库与主题系统 |
| `Chips-Scaffold/*` | 其他插件类型脚手架 | 与本仓共享文档结构、参数化规则与质量门禁，保持整体一致性 |
| `chips dev` 工具 | 开发者命令行入口 | 通过内部 API 或约定的目录结构消费本仓模板能力，为 `chips dev create --type card` 提供实现 |

## 2. 脚手架内部分层

从实现视角，将 `chips-scaffold-basecard` 内部分为三层：

1. **Template Assets Layer（模板资产层）**  
   - 目录：`templates/`（后续阶段创建）  
   - 内容：按模板 ID 划分的静态文件树，每个模板完整描述一个基础卡片插件工程的初始形态（含 `manifest.yaml`、`package.json`、`src/render`、`src/editor`、`schema/`、`tests/` 等）。  
   - 约束：
     - 不包含任何脚手架实现代码或运行时代码逻辑（仅工程模板）；
     - 不直接依赖 Node API 或 CLI 实现；
     - 通过占位符与元数据描述可参数化的部分。

2. **Scaffold Core Layer（脚手架核心层）**  
   - 目录：`src/core/`（后续阶段创建）  
   - 职责：
     - 描述模板元数据（ID、名称、适用场景、是否集成 SDK/组件库等）；
     - 定义参数化 Schema 与默认值（例如插件 ID、卡片类型标识、版本号等）；
     - 实现模板渲染管线（读取模板 → 替换占位符 → 写入目标目录 → 基本自检）。  
   - 依赖：
     - 仅依赖 Node.js 标准库与生态允许的少量工具库（如路径处理）；
     - 不直接依赖 Host/SDK/组件库运行时代码。

3. **CLI Integration Layer（命令行集成层）**  
   - 目录：`src/cli/`（后续阶段创建）  
   - 职责：
     - 为 `chips dev create --type card` 暴露可调用的 API（例如 `createBasecardProject(options)`、`listBasecardTemplates()`）；
     - 实现与命令行参数解析、交互式问答的对接（具体交互逻辑在 `chips dev` 仓库中实现，本仓以纯函数接口为主）。  
   - 依赖：
     - 依赖 Scaffold Core Layer；
     - 不直接依赖 CLI 的实现细节，避免双向耦合。

依赖方向：

- Template Assets Layer ← Scaffold Core Layer ← CLI Integration Layer  
- 不允许出现“模板反向依赖脚手架实现”的情况（模板中不得 import 脚手架代码）。

## 3. 典型调用链与数据流

### 3.1 创建基础卡片插件工程的调用链

1. 开发者在终端执行：

   ```bash
   chips dev create my-basecard --type card
   ```

2. `chips dev` 工具解析参数后，根据 `type: card`：
   - 通过 `listBasecardTemplates()` 获取可用基础卡片模板列表；
   - 引导用户选择模板（如 `card-standard`），并收集：
     - 插件 ID、显示名称、版本号、作者信息；
     - 基础卡片类型标识（例如 `base.text` 或 `vendor.example-card`）。

3. `chips dev` 构造 `CreateBasecardProjectOptions` 并调用 `createBasecardProject(options)`：
   - 根据模板 ID 加载模板目录元数据；
   - 校验参数（ID/卡片类型格式、路径合法性、目录是否为空等）；
   - 将模板文件树渲染到目标路径中，完成占位符替换；
   - 输出结果摘要（生成路径、模板名、后续命令建议）。

4. 开发者进入生成工程目录，安装依赖并运行：

   ```bash
   npm install
   npm run dev   # 或 chips dev server
   ```

5. 生成工程在运行时作为 `type: card` 插件由 Host 加载：
   - 卡片服务在渲染复合卡片时，根据基础卡片类型查找到该插件；
   - 通过插件暴露的渲染/编辑接口将配置数据填入模板并在 iframe 中渲染；
   - 编辑引擎与查看器通过 SDK/Bridge 与卡片服务交互，间接使用插件能力。

### 3.2 数据模型概览

核心数据模型包括（概念层）：

```ts
export interface BasecardScaffoldTemplateMeta {
  id: string;                // 模板 ID，例如 'card-standard'
  name: string;              // 模板展示名称
  description: string;       // 模板用途说明
  tags?: string[];           // 适用场景标签
  supports: {
    sdk?: boolean;           // 是否默认接入 SDK
    componentLibrary?: boolean; // 是否默认接入组件库
  };
}

export interface CreateBasecardProjectOptions {
  projectName: string;
  targetDir: string;
  templateId: string;        // 例如 'card-standard'
  pluginId: string;          // 反向域名格式
  cardType: string;          // 基础卡片类型标识，例如 'base.text'
  displayName: string;       // 插件显示名称（默认语言）
  version: string;           // 初始版本号
  authorName?: string;
  authorEmail?: string;
}
```

具体字段与约束将在《03-模板参数化与渲染引擎设计》中细化。

## 4. 与生态共用文档的关系

- 插件类型、生命周期、权限模型、Manifest 字段、卡片文件格式、卡片服务接口与 Bridge API 等 **对外正式信息**，全部以 `生态共用技术文档` 为准，本仓技术文档仅引用：
  - 插件开发规范；
  - 卡片插件开发指南；
  - Manifest 配置规范；
  - 卡片文件结构规范与卡片业务规范；
  - Bridge API 规范与 Host 对外接口基线；
  - 组件库与主题系统对外使用总览。
- 模板内部如何组织文件、如何命名目录与占位符属于本仓实现细节，但当这些约定需要对所有基础卡片插件生效时，应：
  1. 优先在生态共用文档中补充“基础卡片插件工程目录结构”“卡片配置 Schema 约定”等正式章节；
  2. 本仓再引用该章节作为实现依据，避免出现“双口径”。

## 5. 架构演进原则

为保证脚手架长期可维护与可扩展，约定以下演进原则：

1. **模板优先、逻辑最小化**  
   - 尽可能通过模板文件表达工程结构，脚手架逻辑只做参数填充、简单校验与少量自检；  
   - 避免在脚手架中实现复杂业务逻辑或运行时代码，基础卡片插件的业务行为应由插件工程自身负责。

2. **向后兼容优先**  
   - 在不破坏开发者已有基础卡片插件项目前提下演进模板；  
   - 若必须做不兼容变更，应在文档与 CLI 提示中明确版本与迁移方案。

3. **契约先行**  
   - 在调整模板目录、参数命名、CLI 集成接口前，先更新需求与技术文档中的契约描述；  
   - 通过测试与质量门禁验证契约未被破坏。

4. **与其他脚手架仓库保持结构一致**  
   - 技术文档与开发计划结构尽量与 `chips-scaffold-app`、`chips-scaffold-boxlayout` 等仓库统一；  
   - 模板参数与渲染逻辑复用公共约定，避免每个脚手架独立发明规则。

5. **问题工单优先级高于临时方案**  
   - 当生态缺失必要能力（例如 CLI 不支持 `type: card` 扩展点、组件库缺少关键组件、Host 未暴露必要的卡片服务动作）时，不得通过“临时代码”绕过，而应先在《问题工单.md》中登记并暂停相关实现。

本文件为基础卡片插件脚手架的总体架构约束，后续技术方案与实现必须以此为基线进行细化与评审。
