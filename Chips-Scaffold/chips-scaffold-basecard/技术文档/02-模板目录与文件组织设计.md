# 模板目录与文件组织设计

> 文档状态：设计基线稿  
> 说明：本文件定义 `chips-scaffold-basecard` 仓库内部模板组织方式以及生成基础卡片插件工程的目录结构约定，是实现与演进的参考基线。

---

## 1. 仓库内部目录结构规划

### 1.1 顶层目录规划

`chips-scaffold-basecard` 仓库在完成实现后的预期顶层目录结构如下（当前阶段仅部分目录存在）：

```text
chips-scaffold-basecard/
├─ templates/                # 模板资产层（按模板 ID 划分）
│  ├─ card-standard/         # 标准基础卡片插件模板
│  └─ ...                    # 未来扩展模板
├─ src/                      # 脚手架实现代码
│  ├─ core/                  # 模板元数据与渲染管线
│  └─ cli/                   # 与 chips dev 的集成适配层
├─ tests/                    # 脚手架实现的单元与集成测试
├─ 需求文档/                 # 本仓需求文档
├─ 技术文档/                 # 本仓技术文档
├─ 开发计划/                 # 本仓开发计划
├─ package.json              # 脚手架自身 NPM 包定义（后续阶段添加）
├─ tsconfig.json             # 脚手架实现代码的 TS 配置
└─ README.md                 # 仓库级说明（面向生态内部）
```

约束：

- 模板资产与脚手架实现必须物理分离：`templates/` 下不得包含脚手架实现代码；
- 文档与实现同仓管理，但发布到 NPM 时仅打包必要的实现代码与模板资产；
- 如需新增辅助脚本（例如 CI 脚本、示例生成脚本），放入 `scripts/` 目录，并在技术文档中说明用途。

### 1.2 模板目录命名规范

- 所有基础卡片模板目录放置在 `templates/` 下，命名规则：
  - 使用小写短横线风格，例如：`card-standard`、`card-media-standard`；
  - 前缀 `card-` 表示基础卡片插件模板类型，后缀描述场景；
- 每个模板目录内部必须包含一个元数据文件（形式与字段在《03-模板参数化与渲染引擎设计》中定义），用于声明：
  - 模板 ID、名称、描述；
  - 支持的特性（是否集成 SDK、组件库、多语言等）；
  - 模板版本号（便于后续迁移与兼容性判断）。

## 2. 模板目录结构设计（以 card-standard 为例）

以标准基础卡片插件模板 `templates/card-standard/` 为例，其内部目录结构规划如下：

```text
templates/card-standard/
├─ .eslintrc.cjs.tpl          # 工程级 ESLint 配置模板
├─ manifest.yaml.tpl            # 插件 Manifest 模板（含占位符）
├─ package.json.tpl             # NPM 配置模板
├─ tsconfig.json.tpl            # TS 配置模板
├─ chips.config.mjs.tpl         # 与 chips dev 集成的配置模板
├─ README.md.tpl                # 生成工程 README 模板
├─ src/                         # 插件源码模板
│  ├─ index.ts.tpl              # 插件入口（注册渲染/编辑模块）
│  ├─ render/                   # 渲染模块
│  │  ├─ view.tsx.tpl           # 基础卡片视图组件
│  │  └─ runtime.ts.tpl         # 渲染运行时挂载逻辑
│  ├─ editor/                   # 编辑模块
│  │  ├─ panel.tsx.tpl          # 编辑面板组件
│  │  └─ runtime.ts.tpl         # 编辑运行时挂载逻辑
│  ├─ schema/                   # 配置 Schema 与类型定义
│  │  └─ card-config.ts.tpl
│  └─ shared/                   # 渲染/编辑共享工具
│     ├─ utils.ts.tpl
│     └─ i18n.ts.tpl
├─ config/                      # 配置模板
│  └─ logging.ts.tpl            # 日志封装
├─ i18n/                        # 多语言资源模板
│  ├─ zh-CN.json.tpl
│  └─ en-US.json.tpl
├─ templates/                   # 内容模板与参数表
│  ├─ default-card-config.yaml.tpl
│  └─ parameters.md.tpl
├─ tests/                       # 测试模板
│  ├─ unit/                     # 单元测试
│  │  ├─ render-view.test.tsx.tpl
│  │  ├─ editor-panel.test.tsx.tpl
│  │  └─ schema.test.ts.tpl
│  └─ integration/              # 链路测试
│     └─ card-flow.test.ts.tpl
└─ .gitignore.tpl               # Git 忽略规则模板
```

说明：

- 所有模板文件使用 `.tpl` 后缀区分于最终生成文件，渲染后去掉 `.tpl`；
- 模板内部使用统一的占位符语法（在《03-模板参数化与渲染引擎设计》中定义），避免各文件采用不同占位风格；
- 目录结构与《需求文档/02-基础卡片插件脚手架功能需求规格.md》中对生成工程结构的描述保持一致。

## 3. 生成工程目录结构约定

模板渲染完成后，开发者最终看到的基础卡片插件工程目录结构预计如下（略去部分细节文件）：

```text
my-basecard/
├─ manifest.yaml
├─ package.json
├─ tsconfig.json
├─ chips.config.mjs
├─ README.md
├─ src/
│  ├─ index.ts
│  ├─ render/
│  │  ├─ view.tsx
│  │  └─ runtime.ts
│  ├─ editor/
│  │  ├─ panel.tsx
│  │  └─ runtime.ts
│  ├─ schema/
│  │  └─ card-config.ts
│  └─ shared/
│     ├─ utils.ts
│     └─ i18n.ts
├─ config/
│  └─ logging.ts
├─ i18n/
│  ├─ zh-CN.json
│  └─ en-US.json
├─ templates/
│  ├─ default-card-config.yaml
│  └─ parameters.md
├─ tests/
│  ├─ unit/
│  │  ├─ render-view.test.tsx
│  │  ├─ editor-panel.test.tsx
│  │  └─ schema.test.ts
│  └─ integration/
│     └─ card-flow.test.ts
└─ assets/
   └─ icons/
      └─ basecard-icon.svg
```

约束：

- `manifest.yaml` 必须位于工程根目录，符合《Manifest配置规范》；
- `src/` 为主入口源码目录，默认入口文件为 `src/index.ts`（具体导出接口由 Host/卡片服务规范决定，在本仓仅固定工程结构）；
- `render/` 与 `editor/` 为基础卡片渲染与编辑模块的物理分层，避免混杂；
- `schema/`、`templates/` 与 `tests/` 目录结构需保持稳定，以便未来工具链可以基于这些约定做自动化分析与操作（如配置 Schema 生成、参数表自动校验、测试扫描等）。

## 4. 模板文件类型与处理策略

根据文件在生成工程中的角色，模板文件分为以下几类：

1. **配置文件模板**  
   - 如 `.eslintrc.cjs.tpl`、`manifest.yaml.tpl`、`package.json.tpl`、`tsconfig.json.tpl`、`chips.config.mjs.tpl`；  
   - 含有较多占位符（插件 ID、名称、版本、脚本命令、卡片类型标识等）；  
   - 渲染时需做严格的 Schema 校验，避免生成不合法配置。

2. **源码文件模板**  
   - 如 `src/*.ts.tpl`、`src/**/*.tsx.tpl`、`config/*.ts.tpl`；  
   - 主要包含：
     - 渲染与编辑模块骨架；
     - 配置类型定义；
     - Bridge/API/SDK 调用示例；
   - 必须为有效的 TypeScript/TSX 代码，渲染后可直接通过编译与 lint 检查；
   - 若文件扩展名为 `.ts`，则不得直接写 JSX 语法，必须使用 `React.createElement(...)` 或改为 `.tsx` 模板文件。

3. **资源与文案模板**  
   - 如 `i18n/*.json.tpl`、`assets/*`；  
   - 资源文件通常无需占位符，直接复制即可；  
   - 文案文件应尽量使用占位符或示例文案，避免与具体业务耦合。

4. **内容模板与参数表模板**  
   - 如 `templates/default-card-config.yaml.tpl`、`templates/parameters.md.tpl`；  
   - 用于指导开发者理解配置结构与字段含义；  
   - 应与 `schema/` 中的类型定义保持同步。

5. **测试文件模板**  
   - 如 `tests/unit/*.test.tsx.tpl`、`tests/integration/*.test.ts.tpl`；  
   - 提供最小可运行示例，用于验证模板生成工程的基础质量门禁；  
   - 不应引用尚未在模板中定义的模块或路径；
   - `tests/unit/schema.test.ts.tpl` 为正式基线文件，用于约束模板生成工程至少具备配置归一与校验的最小回归测试。

每类文件在渲染与验证过程中的处理规则将在《03-模板参数化与渲染引擎设计》与《05-测试策略与质量门禁设计》中补充细节。

## 5. 与其他脚手架仓库的对齐要求

为避免脚手架体系碎片化，`chips-scaffold-basecard` 在目录与文件组织上需要与其他脚手架仓库对齐：

- 顶层结构（`templates/src/tests/需求文档/技术文档/开发计划`）保持一致；
- 模板文件 `.tpl` 后缀与占位符语法在所有脚手架中统一；
- 生成工程的基本目录结构在不同插件类型间尽可能一致（例如都包含 `src/`、`config/`、`tests/`、`i18n/`、`templates/`），仅在业务特性上做差异化；
- 如需在多个脚手架之间共享模板片段（例如通用 README 或配置片段），可在 `Chips-Scaffold` 顶层设计共享目录或工具，但需在单独文档中规划。

本文件定义了模板与生成工程的目录与文件组织基线，后续实现与调整必须保持与本设计一致，如需变更需先更新文档并评估对现有工程与工具链的影响。
