# 测试策略与质量门禁设计

> 文档状态：设计基线稿  
> 说明：本文件定义 `chips-scaffold-basecard` 仓库及其生成工程的测试策略与发布门禁，对应《需求文档/03-基础卡片插件脚手架非功能与质量验收需求》中的 NFR-SCAFFOLD-BASECARD-TEST。

---

## 1. 测试目标与范围

### 1.1 测试目标

- 确保脚手架核心逻辑在各类输入下行为正确、错误可预期；
- 确保模板在渲染后得到的基础卡片插件工程是 **可立即运行、可构建、可测试** 的完整项目；
- 防止模板或渲染逻辑的变更悄然破坏 `chips dev` 标准工作流或卡片渲染链路。

### 1.2 测试范围

- 本仓 TypeScript 实现代码（`src/core` 与 `src/cli`）；
- 模板文件（`templates/`）及其元数据；
- 渲染后的示例基础卡片插件工程（通过自动化测试临时生成）。

---

## 2. 测试层级设计

### 2.1 单元测试（Unit Tests）

- **对象**：
  - 模板元数据加载与校验逻辑；
  - 占位符替换与路径映射函数；
  - 错误对象构造与错误码映射。
- **示例**（伪代码）：

  ```ts
  describe('renderTemplateFile', () => {
    it('replaces placeholders with context values', () => {
      const input = 'id: {{ PLUGIN_ID }}';
      const ctx = { PLUGIN_ID: 'com.example.basecard' } as TemplateContext;
      expect(renderTemplateFile(input, ctx)).toBe('id: com.example.basecard');
    });

    it('throws INVALID_ARGUMENT when required context missing', () => {
      const input = 'id: {{ PLUGIN_ID }}';
      const ctx = {} as TemplateContext;
      expect(() => renderTemplateFile(input, ctx)).toThrowErrorMatchingObject({
        code: 'INVALID_ARGUMENT',
      });
    });
  });
  ```

- 目标：对核心纯函数实现做到较高覆盖率，便于快速回归。

### 2.2 模板一致性测试（Template Consistency Tests）

- **对象**：`templates/` 目录下所有模板文件与元数据。  
- **检查内容**：
  - 每个模板目录都存在元数据文件，并且能够被解析；
  - 关键模板文件存在（例如 `.eslintrc.cjs.tpl`、`manifest.yaml.tpl`、`package.json.tpl`、`src/index.ts.tpl`、`src/shared/i18n.ts.tpl`、`tests/unit/schema.test.ts.tpl`、`src/render/view.tsx.tpl` 等）；
  - 模板文件中不存在未被允许的占位符（如拼写错误的变量名）；
  - 模板结构与《技术文档/02-模板目录与文件组织设计》中约定一致。

### 2.3 集成测试（Integration Tests）

- **对象**：`createBasecardProject` 接口与文件系统操作。  
- **策略**：
  - 在临时目录中为多组参数执行 `createBasecardProject`；
  - 渲染后检查：
    - 关键文件是否存在且内容符合预期；
    - 未出现 `.tpl` 后缀遗留；
    - `manifest.yaml` 与 `package.json` 可被正确解析；
    - `schema/card-config.ts` 存在且导出预期类型；
    - `src/shared/i18n.ts` 与 `tests/unit/schema.test.ts` 已被正式生成。

### 2.4 生成工程 E2E 测试（Generated Project E2E Tests）

- **对象**：从模板生成的实际基础卡片插件工程。  
- **场景**：
  1. 创建工程 → 校验关键模板文件已经生成；
  2. 创建工程 → 运行 `tsc -p tsconfig.json --noEmit`；
  3. 创建工程 → 运行 `vitest run`；
  4. 创建工程 → 安装依赖后运行 `chips dev build`；
  5. 创建工程 → 运行 `chips dev validate`；
  6. （可选）在受控环境中加载示例卡片配置并调用 Host 卡片渲染链路进行一次集成验证。
- **结构断言**：
  - 关键运行文件必须存在；
  - `需求文档/`、`技术文档/`、`技术手册/`、`开发计划/` 目录默认不得出现在生成工程中。

- **约束**：
  - 该层测试在本仓 CI 中运行频率可适当降低（例如只在主干与发布分支中运行），以平衡用时与覆盖。

---

## 3. 测试工具与环境

### 3.1 测试框架

- 单元测试与大部分集成测试使用 **Vitest**，与生态统一的前端测试工具保持一致；
- 生成工程 E2E 测试可复用生态既有的 E2E 方案（如 Playwright），具体取决于 `chips dev` 与 Host 的集成能力；  
- 如 E2E 工具尚未在技术选型中冻结，应在本仓开发计划与问题工单中标注为待决项，暂以最小可行方案实现。

### 3.2 运行环境

- CI 环境需至少覆盖：
  - `darwin`；
  - `win32`；
  - （如有必要）`linux`；
- 在无法覆盖多平台时，应优先保证至少一套环境稳定运行，并在问题工单中记录缺失的平台覆盖情况。

## 4. 质量门禁脚本

### 4.1 本仓质量门禁

在 `package.json` 中预期定义如下脚本（名称可在实现阶段根据实际情况微调）：

```json
{
  "scripts": {
    "lint": "eslint src --ext .ts,.tsx",
    "test": "vitest run",
    "test:watch": "vitest",
    "test:templates": "node scripts/check-templates.mjs",
    "test:e2e": "node scripts/run-generated-e2e.mjs"
  }
}
```

质量门禁要求：

- 合并到 `develop` 分支前至少必须通过：
  - `npm run lint`；
  - `npm test`（包含单元与模板一致性测试）；  
- 在发布前（例如打标签或合并到主干时）必须额外通过：
  - `npm run test:templates`；
  - `npm run test:e2e`（可在受控环境下执行）。

### 4.2 生成工程门禁

自动化脚本 `scripts/run-generated-e2e.mjs` 的理想行为：

1. 在临时目录中循环执行：

   ```bash
   cd tmp-basecard
   test -f src/shared/i18n.ts
   test -f tests/unit/schema.test.ts
   npx tsc -p tsconfig.json --noEmit
   npx vitest run
   npm install
   chips dev build
   chips dev validate
   ```

2. 收集每一步的退出码与关键日志；  
3. 任何一步失败即将 CI 标记为失败。

实际实现中可根据 CI 性能与时间预算进行裁剪（例如缓存依赖、减少重复安装）。

## 5. 回归策略与变更流程

### 5.1 模板变更

- 任何对模板文件的修改（包括文案与配置）均视为潜在破坏性变更：
  - 必须更新相关测试用例（尤其是模板一致性测试）；  
  - 必须在提交说明中明确“影响范围”（例如：仅更新 README；或变更了 manifest 字段）。

### 5.2 渲染逻辑变更

- 对占位符解析、路径映射、文件复制逻辑的调整，必须：
  - 增加或更新单元测试覆盖新行为；  
  - 在集成测试与生成工程 E2E 测试中至少完成一次全流程验证。

### 5.3 与 chips dev/Host/SDK 的联动变更

- 当 `chips dev` 或 Host/SDK 的对外契约发生变更（例如命令名称或参数调整）时：
  - 应先更新生态共用文档与对应仓库文档；
  - 在本仓开一个明确的任务（必要时配套问题工单），更新模板与测试；
  - 在 CI 中验证新旧工作流是否需要兼容策略。

---

## 6. 发布前检查清单

在发布新的 `chips-scaffold-basecard` 版本或完成阶段性任务前，应完成如下检查：

1. 本仓质量门禁全部通过：`lint`、`test`、`test:templates`、`test:e2e`；  
2. 模板生成工程在至少一种受支持平台上通过了完整工作流测试；  
3. 文档已同步更新：
   - 需求文档中无过时描述；
   - 技术文档中的目录结构、参数模型、工作流描述与实现一致；  
4. 如有与生态共用文档相关的正式口径变更，已在对应文档中落地；  
5. 问题工单中与本仓相关的开放项已梳理清楚，必要时在发布说明中说明当前版本的已知限制。

本文件为脚手架测试与质量门禁的设计基线，后续实现与 CI/发布流程应以此为依据进行完善与调整。
