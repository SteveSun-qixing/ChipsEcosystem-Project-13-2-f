> 对齐基线：生态设计原稿 + 生态共用技术文档（vNext 冻结口径）

## 1. 项目背景与目标

薯片生态采用“一切皆插件”的扩展模型。内容侧的最小单位是 **基础卡片**，每一种基础卡片类型都由一个独立的 **卡片插件（`type: card`）** 提供渲染与编辑能力：

- 卡片文件（`.card`）只保存内容数据与结构（`metadata.yaml/structure.yaml/content/*.yaml`）；
- 基础卡片插件保存前端代码与交互逻辑（渲染组件 + 编辑组件）；
- Host 内置渲染运行时负责将卡片文件解析为 AST，按类型路由到对应的基础卡片插件，在 iframe 中完成渲染与编辑；
- 编辑引擎、查看器等应用统一通过 SDK/Bridge 调用卡片渲染链路，本身不内嵌各类基础卡片实现。

随着基础卡片类型不断扩展，单个插件仓库临时起项目结构、随意选择技术栈会导致：

- 不同卡片插件目录结构与配置风格割裂，难以维护；
- Manifest、权限、Bridge 调用等细节容易偏离生态共用规范；
- 新开发者上手成本高，每次都要从零搭框架与测试；
- 未来无法在 CI 中批量验证“所有基础卡片插件是否满足最低质量门禁”。

为此，需要在生态层面提供一套官方的 **基础卡片插件脚手架**：

- 通过 `chips dev create` 命令创建 `type: card` 插件时，默认使用本仓模板；
- 创建出的基础卡片插件天然符合卡片文件格式、卡片服务契约、插件开发规范、Manifest 与权限规范、组件库与主题系统规范；
- 在不改动任何代码的前提下，生成工程即可通过 `chips dev build/test/validate` 等基础门禁，并能在 Host 中完成一次端到端渲染与编辑闭环。

基础卡片插件脚手架自身是 **开发阶段工具** 而非运行时代码：

- 运行时代码位于新建的卡片插件工程中，由各基础卡片仓库独立维护；
- 脚手架只负责生成初始目录结构、配置文件、渲染/编辑模块骨架、参数表与示例测试，不参与运行时装载。

### 1.1 北极星目标

- 提供一套可持续 5–10 年演进的官方基础卡片插件模板体系，覆盖常见基础卡片类型（文本、图片、视频、音频、代码、列表等）的通用工程骨架需求。
- 任何通过脚手架创建的基础卡片插件，在遵循模板约定的前提下，都能：
  - 正确声明 `type: card` 与卡片类型能力（例如 `capabilities.cardTypes`）；
  - 完整包含 **渲染模块 + 编辑模块 + 配置 Schema + 内容模板 + 参数表与填写说明**；
  - 通过 Bridge/API/SDK 正确接入卡片服务与文件系统能力；
  - 内建多语言、主题接入与日志/错误处理基础设施；
  - 拥有可运行的单元测试与链路验证测试，并通过基础质量门禁。
- 在不修改 Host/SDK/组件库与全局契约的前提下，完成“脚手架 → 新基础卡片插件 → Host 渲染运行时 → 编辑引擎/查看器”的端到端链路闭环。

### 1.2 本轮开发目标

当前阶段聚焦于 **从零建设 `Chips-Scaffold/chips-scaffold-basecard` 仓库的文档与规划基线**，具体包括：

- 完成基础卡片插件脚手架的项目级需求文档：
  - 《01-基础卡片插件脚手架需求总览》
  - 《02-基础卡片插件脚手架功能需求规格》
  - 《03-基础卡片插件脚手架非功能与质量验收需求》
- 完成基础卡片插件脚手架的技术文档骨架：
  - 总体架构、模板组织、参数化与渲染管线、命令行集成、测试与质量门禁等章节；
- 完成分阶段的开发计划：
  - 阶段一：架构与需求冻结（当前阶段，仅文档与规划，不编写实现代码）；
  - 阶段二及后续：模板实现、CLI 集成、测试体系搭建与发布验收（详见 `开发计划/`）。

后续阶段在开始实现代码前，如发现生态层面的架构缺陷或能力缺口（例如：卡片服务缺少必要接口、组件库缺少基础卡片模板所需组件、`chips dev` 尚未支持 `type: card` 的模板入口等），必须先在《项目日志与笔记/问题工单.md》中登记工单，再继续推进。

### 1.3 成果边界与输出物

本项目最终目标输出物包括但不限于：

- 一套可通过 `chips dev create` 直接消费的基础卡片插件模板集（最小集为“标准基础卡片插件模板”），支持为单一基础卡片类型生成完整插件工程；
- 与模板配套的 `package.json/tsconfig.json/chips.config.*/manifest.yaml` 等配置文件规划，以及渲染/编辑模块的工程骨架结构；
- 完整的基础卡片插件脚手架需求文档、技术文档与开发计划；
- 与生态共用文档对齐的规范引用与补充说明（新出现的跨生态正式口径需沉淀到生态共用文档）。

不直接输出的内容：

- Host 内核、卡片服务、渲染引擎、编辑引擎与查看器内部实现改动；
- SDK 与组件库内部实现改动；
- 具体某一种业务基础卡片（例如“课程章节卡片”“收支明细卡片”）的业务需求说明；
- 对外市场宣传材料（本项目仅提供技术与工程基线）。

## 2. 范围与边界

### 2.1 In Scope（本项目范围内）

- `Chips-Scaffold/chips-scaffold-basecard` 仓库自身的需求、架构设计与实现：
  - 模板目录结构与多模板组织方案（如 `card-standard` 等）；
  - 模板文件内容与占位变量设计（渲染组件、编辑组件、配置 Schema、参数表、内容模板等）；
  - 脚手架对 `chips dev create --type card` 命令的对接方式；
  - 模板内建的多语言、主题接入、日志与错误处理、测试骨架等；
- 与其他仓库的集成接口说明：
  - 如何在模板中消费 `Chips-SDK`（如有）、组件库与 Bridge API；
  - 如何保证生成插件的 Manifest、权限声明与卡片服务接口（`card.parse/validate/render` 等）保持一致；
  - 如何与编辑引擎、查看器通过 SDK 统一显示链路对齐（统一走复合卡片渲染链路，不在卡片插件中重复实现渲染器）；
- 本仓库自身的开发计划、质量门禁与发布流程说明。

### 2.2 Out of Scope（本项目不做）

- 不在本仓直接实现或修改：
  - Host 内核、服务域与插件运行时；
  - 卡片服务的打包/解析/渲染主逻辑；
  - 组件库与主题包内部实现；
  - SDK 内部实现与对外 API；
- 不承担以下职责：
  - 设计新的插件类型或改变现有 `app/card/layout/module/theme` 类型系统；
  - 变更生态共用文档中已冻结的协议、文件格式、Bridge API 契约；
  - 引入超出生态既有技术选型的新第三方库（例如新的路由或状态管理库）；
  - 定义具体业务基础卡片的业务字段与交互（这些属于各卡片插件工程自身职责）。

### 2.3 信息归属边界

- 插件开发规范、卡片文件格式、卡片服务动作、Bridge API、组件库与主题系统契约等 **跨生态正式信息**，统一以 `生态共用技术文档` 为准，本仓文档仅引用与补充实现细节：
  - 卡片插件开发指南；
  - Manifest 配置规范与权限模型；
  - 卡片文件结构规范与 ID 生成规范；
  - Bridge API 规范与 Host 对外接口基线；
  - 组件库与主题系统对外使用规范。
- 基础卡片插件脚手架自身的：
  - 模板目录与命名约定；
  - 模板参数与占位符命名；
  - 与 `chips dev` 的集成接口；
  - 测试策略与 CI 流程；
 仅在 `chips-scaffold-basecard` 目录中记录，不作为生态正式对外规范。
- 如需将某些约定推广为所有卡片插件必须遵守的统一规范（例如统一的 `src/render` / `src/editor` 目录结构），需要：
  1. 先在 `生态共用技术文档` 中增加相应章节；
  2. 再在本仓文档中引用该章节作为实现依据。

## 3. 约束与设计原则

### 3.1 架构与契约约束

- **中心路由原则**：基础卡片插件在运行时只能通过 Host 提供的 Bridge API 或 SDK 调用系统能力，禁止越层访问 Node/Electron API 或跨插件直接通信。
- **运行时归属**：L1–L9（含 Runtime Client、声明式 UI、统一渲染链路）由 `Chips-Host` 提供，基础卡片插件仅在 L12 页面域中实现自身 UI 与交互；渲染器与卡片服务仍由 Host 负责。
- **文件与格式约束**：
  - 生成工程必须遵守卡片文件结构规范与 ID 生成规范；
  - 插件不得自行扩展卡片文件结构，所有结构扩展需通过生态共用文档评审后统一定义。
- **插件契约约束**：
  - Manifest 字段、插件 ID、权限声明、能力标识等必须符合插件开发规范与 Manifest 配置规范；
  - 基础卡片类型标识（例如 `base.text`, `base.image` 或 `vendor.custom-card`）需遵循统一命名约定（在生态共用文档中定义），脚手架模板只负责参数化填充。

### 3.2 工程与质量约束

- **禁止临时版、简化版、占位实现**：
  - 模板中不得存在“TODO”、“占位组件”、“空实现函数”等未完成代码；
  - 生成的基础卡片插件工程必须可以直接运行、构建并通过基础测试，且能够完成至少一次示例渲染与编辑流程。
- **禁止越界修改与未审批依赖**：
  - 不得修改当前任务范围外的仓库目录与文件；
  - 不得引入未在生态技术选型中列出的第三方库（包括打包工具、路由库、状态管理库等）；如确有需要，必须先在问题工单中提出并获批。
- **测试与门禁**：
  - 所有模板必须内建单元测试（至少覆盖渲染/编辑模块骨架）与基础链路测试（至少验证一次配置读写与渲染调用）；
  - 生成工程在默认配置下必须通过 `npm test` / `chips dev test`、`chips dev validate`、`chips dev build` 等基础门禁。

### 3.3 问题工单与阻断策略

在基础卡片插件脚手架设计或后续实现过程中，如发现：

- 生态层面的架构缺陷或能力缺口（例如：Host 尚未暴露卡片插件初始化所需上下文、卡片服务未提供必要的渲染模式、组件库缺少基础卡片常用布局组件等）；
- 现有规范之间存在冲突且影响脚手架可实现性（例如卡片文件格式与插件开发规范对字段口径不一致）；
- `chips dev` 工具链尚未提供卡片插件脚手架所需的扩展点或命令行参数规范；

必须：

1. 在《项目日志与笔记/问题工单.md》中登记新工单，说明问题、影响范围与期望解决方向；
2. 在 `项目日志与笔记/任务清单/` 下创建对应任务文件夹与说明文档，记录讨论与决策；
3. 暂停当前脚手架相关实现工作，等待用户处理与决策后再继续推进，不得采用临时代码绕过问题。

## 4. 干系人与协作边界

| 干系人 | 角色 | 关注点 |
|---|---|---|
| 基础卡片插件开发者 | 使用脚手架创建新基础卡片插件的一方或第三方开发者 | 创建命令简单可靠、项目结构清晰、渲染/编辑链路示例完整、易于扩展 |
| Chips Dev 工具链维护者 | 提供 `chips dev create/build/test/validate` 的命令行工具 | 脚手架导出的模板结构稳定、接口简单、错误信息可诊断，易于在 CLI 中集成 |
| Chips-Host / 卡片服务团队 | 维护 Host 内核、服务域与卡片渲染运行时 | 脚手架生成的卡片插件严格遵守 Host 对外接口与权限模型，不引入新的运行时代码路径 |
| Chips-SDK 团队 | 维护 SDK 封装与开发者工具 | 模板示例正确使用 SDK 能力与 Card/Box 等 typed 客户端，不绕过或重复实现 SDK 功能 |
| 组件库与主题团队 | 维护无头组件库与主题系统 | 模板中正确接入组件库 Provider 与主题运行时，不硬编码视觉样式，不破坏 token 体系 |
| 生态架构与规范维护者 | 维护生态共用文档与长线架构 | 脚手架文档与实现与全局规范口径一致，新增正式规范时能够同步回生态共用文档 |

## 5. 里程碑与阶段划分

基础卡片插件脚手架项目开发阶段规划如下（详细见 `开发计划/00-开发计划索引.md`）：

1. **阶段一：架构与需求冻结**  
   - 阅读并对齐生态设计原稿与生态共用技术文档中与卡片/插件相关的全部章节；  
   - 完成本仓需求文档、技术文档骨架与开发计划拆解；  
   - 不编写任何实现代码，仅在必要时创建空目录结构支撑文档。
2. **阶段二：标准基础卡片插件模板骨架实现**  
   - 设计并实现标准基础卡片插件模板的目录结构与配置文件；  
   - 内建最小可用的渲染模块与编辑模块骨架、多语言与主题接入、配置 Schema 与参数表；  
   - 保证模板在本地可通过构建与基础测试。
3. **阶段三：模板扩展与生态对接**  
   - 设计并实现适用于不同场景的基础卡片插件模板变体（如媒体类、文本类等）；  
   - 与 `chips dev create --type card` 完成对接，支持按模板 ID 创建工程；  
   - 补充与 Host/SDK/组件库集成的契约测试与示例。
4. **阶段四：测试体系与质量门禁**  
   - 建立脚手架自身的测试集（模板一致性测试、生成工程 E2E 验证等）；  
   - 定义并实现发布前质量门禁脚本；  
   - 确保后续模板演进不会破坏已有开发者项目的基本假设。
5. **阶段五：文档完善与发布验收**  
   - 完善使用文档与示例说明；  
   - 对齐并必要时补充生态共用技术文档中的基础卡片插件/脚手架相关章节；  
   - 形成第一版可对外宣布的基础卡片插件脚手架发布版本。

上述阶段仅描述技术实施顺序，不涉及具体人力与时间排期。每个阶段结束前必须完成对应文档与测试的更新，方可进入下一阶段。
