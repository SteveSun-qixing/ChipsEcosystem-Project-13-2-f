# 基础卡片插件脚手架功能需求规格

> 文档状态：需求规格候选稿  
> 说明：本文件定义基础卡片插件脚手架的功能需求；跨生态公开接口与协议口径以生态共用文档为准。

---

## 1. 能力域总览

| 能力域 | 核心目标 |
|---|---|
| 脚手架核心能力 | 通过 `chips dev create` 等命令创建基础卡片插件工程，并完成基本参数收集与模板渲染 |
| 模板结构与内容 | 提供符合生态规范的卡片插件项目目录结构、配置文件与渲染/编辑模块骨架 |
| 卡片配置与契约 | 在模板中内建卡片配置 Schema、内容模板与参数表，确保与卡片文件格式和卡片服务契约对齐 |
| 渲染与编辑基础设施 | 在模板中内建渲染/编辑链路示例、多语言与主题接入、日志与错误处理骨架 |
| 质量与门禁支撑 | 在模板中内建测试、校验与构建脚本，确保生成工程可立即通过基础质量门禁 |
| 扩展与文档支持 | 支持未来多模板扩展，并提供清晰的使用说明与自定义扩展入口 |

以下按能力域描述功能需求，采用 `FR-SCAFFOLD-BASECARD-*` 编号。

---

## 2. 脚手架核心能力需求（FR-SCAFFOLD-BASECARD-CORE）

### FR-SCAFFOLD-BASECARD-CORE-001 项目创建入口

- 脚手架必须支持通过 `chips dev create` 命令创建基础卡片插件工程：
  - 用户至少可以指定项目名称与插件类型（`card`）；
  - 命令行工具负责解析参数并调用本仓导出的模板生成能力。
- 首版可只支持交互式问答或最小参数集，但需要在技术文档中固定接口约定（例如 `createBasecardProject(options)`），避免后续破坏兼容性。

### FR-SCAFFOLD-BASECARD-CORE-002 模板选择与变体

- 首版至少提供一个 **标准基础卡片插件模板**（例如 `card-standard`），满足通用基础卡片类型的开发场景：
  - 单卡片类型、单渲染模块、单编辑模块；
  - 支持配置 Schema 与参数表定义；
  - 支持多语言与主题接入示例。
- 后续阶段可新增模板变体（例如 `card-media-standard`、`card-text-standard` 等），脚手架需要：
  - 为模板分配稳定的标识符与描述；
  - 支持在 `chips dev create` 中通过参数或交互选择模板；
- 当模板变体尚未实现时，不得在文档或命令行帮助中对外暴露对应选项。

### FR-SCAFFOLD-BASECARD-CORE-003 参数化与占位符

- 模板必须支持对至少以下信息进行参数化渲染：
  - 插件 ID（反向域名格式，例如 `com.example.basecard-text`）；
  - 基础卡片类型标识（例如 `base.text`, `base.image` 或统一约定的命名）；
  - 插件显示名称（默认语言的名称）；
  - 初始版本号（默认 `0.1.0` 或统一约定的开发期版本）；
  - 作者信息（名称、邮箱等）；
  - 项目根路径与 NPM 包名（供 `package.json` 使用）。
- 参数化渲染必须保持幂等：同一参数集生成的项目结构与内容必须一致。
- 占位符语法与参数模型需在技术文档《模板参数化与渲染引擎设计》中固定，并通过单元测试验证。

### FR-SCAFFOLD-BASECARD-CORE-004 执行结果与错误提示

- 创建命令执行完成后必须输出：
  - 新项目的绝对路径；
  - 选择的模板 ID；
  - 下一步建议命令（例如 `cd <project> && npm install && npm run dev`）；
- 创建失败时：
  - 给出明确的错误信息（如路径已存在、参数不合法、模板缺失等）；
  - 不得生成残缺的工程结构；若已生成部分文件，应在安全前提下回滚或提示用户手工清理方案。

## 3. 模板结构与内容需求（FR-SCAFFOLD-BASECARD-TEMPLATE-STRUCT）

### FR-SCAFFOLD-BASECARD-TEMPLATE-STRUCT-001 项目基础目录结构

- 模板生成的基础卡片插件工程至少应包含以下目录与文件（可在技术文档中进一步细化）：

  - 根目录：
    - `package.json`：NPM 包描述与脚本定义；
    - `tsconfig.json`：TypeScript 配置；
    - `chips.config.*`：与 `chips dev` 集成的构建/运行配置；
    - `manifest.yaml`：插件 Manifest（`type: card`）；
    - `README.md`：项目使用说明；
  - 源码目录：`src/`
    - `index.ts`：卡片插件入口（导出渲染/编辑模块注册接口等）；
    - `render/`：渲染模块源码（例如 `render/view.tsx`、`render/runtime.ts`）；
    - `editor/`：编辑模块源码（例如 `editor/panel.tsx`、`editor/runtime.ts`）；
    - `schema/`：卡片配置类型定义与默认值（例如 `schema/card-config.ts`）；
    - `shared/`：渲染与编辑共享的工具与类型；
  - 配置与资源目录：
    - `config/`：插件级配置（例如 Feature Flag、日志配置等）；
    - `i18n/`：多语言资源文件（至少 `zh-CN` 与 `en-US` 示例）；
    - `templates/`：内容模板与参数表（例如默认配置示例、参数填写说明）；
    - `assets/`：静态资源（图标等）；
  - 测试目录：`tests/`
    - `unit/`：单元测试示例（渲染/编辑模块的最小测试）；
    - `integration/` 或 `e2e/`：链路测试示例（验证一次配置读写与渲染调用）。

- 所有目录名称与文件命名必须在《技术文档/02-模板目录与文件组织设计》中固定下来，作为后续演进与生态工具对接的基础约定。

### FR-SCAFFOLD-BASECARD-TEMPLATE-STRUCT-002 Manifest 与插件元信息

- 模板必须生成符合《Manifest配置规范》的 `manifest.yaml` 文件，至少包含：
  - `id`、`name`、`version`、`type: card`、`entry` 等必填字段；
  - `capabilities` 字段中声明支持的卡片类型标识（例如 `cardTypes: ['base.text']`）；
  - 合理的默认 `description` 与 `author` 信息占位；
  - 必要且最小化的 `permissions` 列表（例如仅 `card.read` / `card.write` / `resource.read` 等）。
- 模板不得默认启用高风险权限（如文件任意写入、网络访问、凭证管理等）；如示例代码确需某权限，必须在 README 与本仓技术文档中说明用途与风险。

### FR-SCAFFOLD-BASECARD-TEMPLATE-STRUCT-003 NPM 脚本与 chips 命令集成

- `package.json` 中必须预置以下脚本（名称可根据生态统一命名调整）：
  - `dev`：启动开发服务器（例如 `chips dev server` 或等价命令）；
  - `build`：构建插件包（例如 `chips dev build`）；
  - `test`：运行单元测试与基础链路测试（例如 `chips dev test` 或 `vitest`）；
  - `lint`：运行代码规范检查；
  - `validate`（可选）：封装 `chips dev validate`，便于直接在项目内执行。
- 模板必须保证在依赖安装完成后：
  - `npm run lint` / `chips dev lint` 无错误；
  - `npm test` / `chips dev test` 用例全部通过；
  - `chips dev validate` 对 Manifest、权限、依赖等校验全部通过；
  - `chips dev build` 可生成合法的 `cpk` 包。

### FR-SCAFFOLD-BASECARD-TEMPLATE-STRUCT-004 内容模板与参数表

- 模板必须为基础卡片配置提供：
  - 至少一个默认配置模板文件（例如 `templates/default-card-config.yaml`），符合卡片文件结构规范；
  - 一份参数表/填写说明文档（可以是 `templates/parameters.md`），包括每个配置字段的含义、类型、取值范围与示例；
- 上述内容模板与参数表应与渲染/编辑模块中实际使用的配置结构保持一致，避免出现“文档字段与实现字段不一致”的情况。

## 4. 卡片配置与契约需求（FR-SCAFFOLD-BASECARD-CONTRACT）

### FR-SCAFFOLD-BASECARD-CONTRACT-001 配置 Schema 与类型定义

- 模板必须为基础卡片配置提供 TypeScript 类型定义，例如：

  ```ts
  export interface BasecardConfig {
    id: string;              // 基础卡片实例 ID（10 位 62 进制，可由上层生成）
    title?: string;          // 标题（可选）
    content?: string;        // 主体内容（具体字段根据卡片类型扩展）
    // ...
  }
  ```

- 类型定义必须：
  - 能与卡片文件中 `content/*.yaml` 的结构一一对应；
  - 便于在渲染/编辑模块中获得完整的类型检查；
  - 为后续 SDK/工具链进行 Schema 校验预留扩展点（例如通过 JSON Schema 或 Zod 等形式，但具体实现需在生态技术选型中确认）。

### FR-SCAFFOLD-BASECARD-CONTRACT-002 渲染与编辑接口形态

- 模板必须约定并示例基础卡片插件对外暴露的核心接口形态，例如：

  ```ts
  export interface BasecardRenderContext {
    container: HTMLElement;
    config: BasecardConfig;
    themeCssText?: string;
  }

  export function renderBasecardView(ctx: BasecardRenderContext): CleanupFn;

  export interface BasecardEditorContext {
    container: HTMLElement;
    initialConfig: BasecardConfig;
    onChange: (next: BasecardConfig) => void;
  }

  export function mountBasecardEditor(ctx: BasecardEditorContext): CleanupFn;
  ```

- 上述接口仅作为脚手架内部约定，实际与 Host 的对接方式需在 Host/卡片服务中统一定义；模板需要：
  - 在注释中明确哪些字段来自 Host（例如 `container`、`themeCssText`）；
  - 在 README 与技术文档中说明这些接口如何映射到 Host 的卡片插件加载流程。

### FR-SCAFFOLD-BASECARD-CONTRACT-003 卡片服务与 Bridge 调用示例

- 模板必须包含至少一个完整的卡片服务调用示例，例如在编辑模块中：
  - 通过 Bridge API 调用 `card.update` 或相应动作，将配置写回卡片文件；
  - 通过 Bridge API 或 SDK 调用 `card.validate` 或 `card.parse` 进行本地校验；
- 示例代码需要：
  - 使用 `window.chips.*` 或 SDK 客户端，禁止直接访问 Node/Electron API；
  - 捕获并展示标准错误对象（`code/message`），不向用户泄露底层堆栈；
  - 在技术文档中标注依赖的 Host 路由与权限（例如需要 `card.write` 权限）。

## 5. 渲染与编辑基础设施需求（FR-SCAFFOLD-BASECARD-INFRA）

### FR-SCAFFOLD-BASECARD-INFRA-001 多语言与文案结构

- 模板必须内建多语言支持：
  - 至少提供 `zh-CN` 与 `en-US` 两个语言包文件；
  - 为渲染与编辑界面中所有文本提供多语言键值，而非直接硬编码；
  - 展示一个简单的语言切换示例，说明如何根据 Host 当前语言或用户选择切换语言。

### FR-SCAFFOLD-BASECARD-INFRA-002 主题与组件库接入

- 模板必须：
  - 在渲染与编辑模块根组件中接入组件库 Provider 与主题运行时；
  - 使用组件库基础组件（如 `Button/Input/Dialog` 等）构建示例 UI；
  - 不在业务代码中硬编码颜色、阴影、圆角等视觉常量，而是通过主题 token 或组件库契约获取。

### FR-SCAFFOLD-BASECARD-INFRA-003 渲染示例与复合卡片链路对齐

- 模板必须提供一个最小可运行的渲染示例：
  - 在独立 iframe 环境中渲染基础卡片内容；
  - 与《卡片渲染机制》中“基础卡片在复合卡片中的 iframe 渲染模型”保持一致；
  - 不在插件内部实现复合卡片拼接逻辑。
- 示例需要说明：
  - 基础卡片 iframe 如何从 Host 获取配置与主题信息；
  - 如何与复合卡片窗口组件（`CompositeCardWindow`）的整体渲染链路协同。

### FR-SCAFFOLD-BASECARD-INFRA-004 编辑面板嵌入示例

- 模板必须提供基础卡片编辑面板在编辑引擎中的嵌入示例：
  - 假设编辑引擎提供一个 iframe 容器用于挂载基础卡片编辑模块；
  - 示例展示如何在面板中渲染配置表单、校验用户输入，并通过回调或 Bridge API 将配置写回；
  - 示例不直接依赖具体编辑引擎实现，仅在文档中说明与编辑引擎的集成点（例如 `onSelectBasecard` 时加载对应编辑器）。

### FR-SCAFFOLD-BASECARD-INFRA-005 错误边界与日志

- 模板必须内建：
  - 渲染模块中的错误边界组件（可使用组件库提供的 `ErrorBoundary` 或等效封装）；
  - 统一的日志输出工具（封装 `console.*`，为未来接入 Host 日志服务预留接口）；
- 示例代码中应展示：
  - 捕获渲染/编辑过程中的异常并展示友好错误提示；
  - 在开发模式下输出更详细的调试信息。

## 6. 质量与门禁支撑需求（FR-SCAFFOLD-BASECARD-QUALITY）

### FR-SCAFFOLD-BASECARD-QUALITY-001 单元测试与组件测试

- 模板必须内建单元测试基础设施：
  - 测试框架使用 Vitest，与生态开发环境配置指南保持一致；
  - 至少提供 1–2 个示例测试文件，覆盖：
    - 渲染模块的基础渲染与交互；
    - 编辑模块的表单验证与配置更新逻辑（可通过 mock Bridge/API 实现）。

### FR-SCAFFOLD-BASECARD-QUALITY-002 链路与 E2E 测试

- 模板必须预留链路测试目录与示例：
  - 使用生态统一的 E2E 工具（若已选型，例如 Playwright），否则在文档中标记待接入，并通过问题工单推动选型；
  - 示例用例至少覆盖“加载基础卡片配置 → 打开渲染 iframe → 执行一次配置更新 → 重新渲染并验证结果”。

### FR-SCAFFOLD-BASECARD-QUALITY-003 校验与构建门禁

- 模板必须保证在依赖安装完成后：
  - `npm run lint` / `chips dev lint` 无错误；
  - `npm test` / `chips dev test` 用例全部通过；
  - `chips dev validate` 对 Manifest、权限、依赖等校验全部通过；
  - `chips dev build` 可生成合法的 `cpk` 包；
- 如生态尚未提供某个门禁命令，则在技术文档与开发计划中明确依赖关系，并通过问题工单推进。

## 7. 扩展与文档支持需求（FR-SCAFFOLD-BASECARD-EXTENSIBILITY）

### FR-SCAFFOLD-BASECARD-EXTENSIBILITY-001 多模板扩展能力

- 脚手架内部设计必须支持未来新增基础卡片模板而不破坏现有接口：
  - 模板元数据（ID、名称、描述、适用场景、是否集成 SDK/组件库等）应以结构化方式存储；
  - `chips dev create` 接口与内部实现需预留模板列表查询与筛选能力；
- 新增模板时必须更新：
  - 本仓技术文档中与模板相关章节；
  - 生态共用文档中如有涉及到模板清单或示例的部分。

### FR-SCAFFOLD-BASECARD-EXTENSIBILITY-002 使用说明与示例

- 模板生成的工程必须包含 `README.md`，至少说明：
  - 如何安装依赖、运行开发服务器、执行测试与构建；
  - 模板中各主要目录与文件的作用；
  - 如何扩展基础卡片类型的配置字段、接入新的 Host 能力、增加多语言条目等；
- 本仓技术文档中需要提供：
  - 面向生态内部的基础卡片插件脚手架使用说明；
  - 面向未来公开的高层次说明可通过 `生态共用技术文档` 进行对外整理。

### FR-SCAFFOLD-BASECARD-EXTENSIBILITY-003 与其他脚手架仓库的一致性

- `chips-scaffold-basecard` 的能力规划与文档结构应与以下仓库保持一致：
  - `chips-scaffold-app`
  - `chips-scaffold-boxlayout`
  - `chips-scaffold-module`
  - `chips-scaffold-theme`
- 一致性体现在：
  - 需求/技术/开发计划三类文档均采用相同结构与命名约定；
  - 模板参数（例如插件 ID、名称、版本、卡片类型等）的命名与渲染机制尽量统一；
  - 如需在多个脚手架间共享规范，应优先沉淀到 `生态共用技术文档` 再在各仓引用。

以上功能需求构成基础卡片插件脚手架项目的基础能力清单，后续技术方案与实现必须以本文件为依据进行细化与验证。如发现本文件与生态共用文档存在冲突，以生态共用文档与用户最新决策为准，并通过问题工单同步调整。
