# 布局插件脚手架需求总览

> 文档状态：需求基线稿  
> 对齐基线：生态设计原稿 + 生态共用技术文档（2026-03-24 当前冻结口径）

---

## 1. 项目背景与目标

薯片生态中的布局插件承担“箱子页面前端”职责。  
它既不是 Host 运行时的一部分，也不是单纯的样式模板，而是与箱子运行时契约、查看器、编辑引擎、主题系统、多语言系统、组件库和 SDK 一起协作的正式插件类型。

当前生态已经冻结了布局插件的正式公共契约，但工作区仍缺少一套官方布局插件脚手架。这会带来以下问题：

1. 新布局插件工程缺少统一目录结构；
2. `manifest.yaml`、`layout.layoutType`、`layoutDefinition` 的基线无法统一；
3. 查看态、编辑态、schema、测试和 i18n 接线容易出现仓库间漂移；
4. `chipsdev create` 无法给出布局插件的正式创建入口。

因此必须建立 `Chips-Scaffold/chips-scaffold-boxlayout`，作为布局插件工程的唯一正式初始化来源。

### 1.1 北极星目标

- 提供一套可持续 5–10 年演进的官方布局插件模板体系；
- 任何通过脚手架创建的布局插件项目，在不手工补基线的前提下，即可：
  - 声明正式 `type: layout` Manifest；
  - 正确导出 `layoutDefinition`；
  - 正确接入箱子运行时会话、主题、多语言和组件库；
  - 内建查看态、编辑态、schema、测试和 README；
  - 通过 `build / test / validate` 等正式门禁。

### 1.2 本轮开发目标

本轮先完成布局插件脚手架项目级文档与实施规划冻结，包括：

- 完成需求文档 `00-03`；
- 完成技术文档骨架；
- 完成分阶段开发计划；
- 明确与大任务003（箱子全链路）之间的前后依赖；
- 为后续脚手架实现和首个网格布局插件工程提供唯一正式基线。

### 1.3 成果边界与输出物

本项目最终输出物包括但不限于：

- 可被 `chipsdev create` 消费的布局插件模板；
- 与模板配套的工程结构、参数模型、测试基线和验证脚本；
- 完整的需求、技术和开发计划文档；
- 需要进入共享文档的跨生态正式规范补充说明。

本项目不直接承担以下内容：

- Host 箱子运行时实现；
- SDK 箱子 API 实现；
- 查看器与编辑引擎的箱子接入；
- 具体业务布局插件的功能实现（例如网格布局插件本身），这些属于下游仓库或大任务阶段。

## 2. 范围与边界

### 2.1 In Scope

- 布局插件脚手架仓库自身的需求、架构、模板设计与质量门禁；
- `layout` 类型插件的正式工程骨架；
- 模板参数化、CLI 接入、生成工程 README 与示例代码；
- 查看态、编辑态、schema、i18n、测试、资源目录的模板基线；
- 与 `chipsdev create`、SDK、组件库、主题系统和箱子运行时契约的集成说明。

### 2.2 Out of Scope

- Host 运行时、Bridge、SDK 或组件库内部实现改动；
- 布局插件业务逻辑本身的深度算法实现；
- 修改生态中已冻结的插件类型系统；
- 引入未在生态技术选型中的第三方框架。

### 2.3 信息归属边界

- `layout.layoutType`、`layoutDefinition`、箱子运行时动作等跨生态信息，必须以共享文档为主发布位置；
- 脚手架项目内部只定义模板组织、参数模型、CLI 对接与测试策略；
- 生成工程默认不创建 `需求文档/`、`技术文档/`、`开发计划/` 目录，这些仅存在于脚手架仓本身。

---

## 3. 不可违反约束

### 3.1 契约优先

- 模板只能消费已经冻结的生态公共契约；
- 若 Host、SDK、共享文档三者对布局插件口径不一致，必须先登记工单，不得让模板先行定义事实标准。

### 3.2 正式工程优先

- 生成的布局插件工程必须是正式可开发基线；
- 不允许输出占位工程、简化工程、无测试工程；
- 不允许把关键接线留给业务仓手工补齐。

### 3.3 统一技术栈

- 官方布局插件模板必须以 React 为主技术栈；
- 默认接入组件库、主题系统和多语言系统；
- 布局查看态与编辑态都要遵守正式布局插件开发文档。

### 3.4 问题工单优先

如在脚手架设计过程中发现：

- `chipsdev create` 缺少布局插件接入点；
- 共享文档仍缺布局插件工程公共口径；
- 组件库缺布局插件正式模板必需组件；
- SDK 缺少模板示例所需正式类型；

必须先登记工单，再决定是否继续推进。

## 4. 干系人与协作边界

| 干系人 | 角色 | 关注点 |
|---|---|---|
| 布局插件开发者 | 使用脚手架创建布局插件项目 | 创建流程简单、模板完整、无需手工补主干基线 |
| `chipsdev` 工具链维护方 | 提供 `create/build/test/validate` 入口 | 模板接入清晰、参数可诊断、E2E 可验证 |
| Host 维护方 | 提供箱子运行时 | 模板不反向定义 Host 契约，不发明第二套会话模型 |
| SDK 维护方 | 提供开发者 API 封装 | 模板示例正确使用 SDK 或 Bridge，不私建旁路 |
| 组件库与主题团队 | 提供 UI 和视觉基础 | 模板默认正确接入主题和组件规范 |
| 箱子全链路任务负责人 | 下游消费脚手架生成工程 | 可以直接用脚手架初始化 `Chips-BoxLayoutPlugin` 并进入正式开发 |

## 5. 里程碑与阶段划分

布局插件脚手架项目开发阶段规划如下（详见 `开发计划/00-开发计划索引.md`）：

1. **阶段一：架构与需求冻结**  
   - 完成本仓需求、技术、开发计划文档；  
   - 冻结布局插件模板的正式边界、输入输出和验证方式。

2. **阶段二：标准布局插件模板骨架实现**  
   - 完成标准布局插件模板目录、Manifest、入口、查看态/编辑态骨架；  
   - 确保生成工程默认可构建、可测试。

3. **阶段三：CLI 集成与模板参数化**  
   - 完成与 `chipsdev create` 的正式对接；  
   - 建立模板参数模型和占位符渲染管线。

4. **阶段四：模板验证与生成工程 E2E**  
   - 完成脚手架测试体系、模板校验与生成工程联调；
   - 确保生成工程能作为 `Chips-BoxLayoutPlugin` 的正式起点。

5. **阶段五：文档完善与发布验收**  
   - 收口 README、技术文档、共享文档引用与验收门禁。

以上阶段仅描述技术实施顺序，不涉及时间和人力排期。
