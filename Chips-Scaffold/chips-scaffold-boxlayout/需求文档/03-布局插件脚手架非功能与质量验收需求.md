# 布局插件脚手架非功能与质量验收需求

## 1. 稳定性要求

- 脚手架输出必须稳定可复现；
- 相同输入参数生成的工程结构必须一致；
- 模板渲染失败时必须输出可诊断错误。

## 2. 可维护性要求

- 模板目录、参数模型、CLI 接口、测试与 README 必须同步演进；
- 不允许模板与共享文档长期漂移；
- 不允许生成工程继续依赖脚手架仓内部私有文件。

## 3. 可扩展性要求

- 当前至少支持标准布局插件模板；
- 后续新增瀑布流、看板等模板变体时，不应破坏现有模板参数模型；
- 模板组织应支持一仓多模板扩展。

## 4. 质量门禁要求

脚手架仓必须具备并持续通过：

- `build`
- `test`
- `test:templates`
- `test:e2e`

生成工程必须默认具备并通过：

- `build`
- `test`
- `lint`
- `validate`

## 5. 文档完整性要求

- 本仓必须具备完整的需求文档、技术文档和开发计划；
- 任何影响模板正式结构的改动，都必须同步更新相应文档；
- 不允许把关键规则只留在 README 或提交记录中。

## 6. 安全与规范要求

- 不引入未在生态技术选型中的第三方依赖；
- 不在模板中写入临时调试代码、开发者私有路径、隐式全局依赖；
- 不输出 TODO、空实现或示例残留。

## 7. 验收判定

只有同时满足以下条件，才算完成首版布局插件脚手架：

1. 项目级文档已冻结；
2. 脚手架仓自身门禁通过；
3. 生成工程门禁通过；
4. 生成工程可以直接作为正式布局插件开发起点；
5. 与大任务003阶段五衔接顺畅，无需另造工程标准。
