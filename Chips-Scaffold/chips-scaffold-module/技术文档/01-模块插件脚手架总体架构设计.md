# 模块插件脚手架总体架构设计

> 文档状态：内部技术基线稿

---

## 1. 仓库定位

`Chips-Scaffold/chips-scaffold-module` 是开发期工具仓库，不参与最终用户环境的运行时分层。它负责为 `type: module` 插件生成标准工程模板，并向 `chipsdev` 暴露可复用的创建 API。

运行时归属关系如下：

- Host / Bridge / SDK：提供正式运行时协议与系统能力；
- 模块脚手架仓库：生成模块工程，不承载运行时主实现；
- 生成出来的模块工程：在运行时作为独立 `type: module` 插件存在。

## 2. 内部分层

仓库内部划分为四层：

1. Template Assets Layer  
   - 目录：`templates/`  
   - 责任：保存模板文件树与 `template.json` 元数据。

2. Scaffold Core Layer  
   - 目录：`src/core/`  
   - 责任：模板发现、参数校验、占位符渲染、目标目录保护、输出统计。

3. CLI Integration Layer  
   - 目录：`src/cli/`  
   - 责任：向 `chipsdev` 暴露 `listModuleTemplates/createModuleProject` 等接口。

4. Quality Gate Layer  
   - 目录：`tests/`、`scripts/`、文档目录  
   - 责任：模板检查、单测、生成工程自检、技术文档与开发计划沉淀。

依赖方向固定为：

`templates -> core -> cli -> SDK CLI`

模板文件不得反向依赖脚手架实现代码。

## 3. 典型调用链

1. 开发者执行 `chipsdev create module <targetDir>`；
2. `chipsdev` 解析类型为 `module`；
3. CLI 侧确保 `chips-scaffold-module` 已构建；
4. 通过 `createModuleProject(options)` 渲染 `module-standard`；
5. CLI 自动注册工作区并写入 `volta.extends`；
6. 开发者在生成工程中运行 `lint/build/test/validate/package`。

## 4. 关键设计决策

- 模板仓库只提供最少必要的渲染引擎，不内嵌复杂业务逻辑；
- 标准模板直接对齐当前正式契约：`module.mount/unmount/query/list + slot/moduleId`；
- 标准模板同时对齐新的运行时治理口径：`module.mount` 可附带 `requiredCapabilities`，成功后返回 `bridgeScopeToken` 供模块建立独立 Host 身份；
- 模块能力使用 `capabilities: string[]`，不使用未正式实现的 `moduleType` 字段；
- 主题与多语言通过 `chips-sdk + 本地词典` 组合接入；
- React 运行时通过 provider 分层，避免把 Bridge、主题、语言与 UI 堆叠在单组件中；
- 模块模板不再默认把宿主页面 `window.chips` 身份当作模块正式身份，而是优先消费 Host 发放的 scoped bridge token。

## 5. 跨仓依赖边界

| 仓库 | 依赖方向 | 作用 |
|---|---|---|
| `Chips-SDK` | 模板工程依赖 | 提供主题、语言、模块治理与事件订阅能力 |
| `Chips-Host` | 契约对齐 | 提供模块正式动作与插件治理行为 |
| `生态共用技术文档` | 规范源 | 约束 Manifest、Bridge、CLI、插件开发口径 |
| `项目日志与笔记` | 记录源 | 沉淀发现的问题工单与修复结论 |

## 6. 演进原则

- 契约先于模板；
- 模板先于 CLI；
- 测试与文档必须跟随实现同步收口；
- 发现生态缺口时优先走问题工单流程，而不是在模板中做兼容性绕路。
