# 模块插件脚手架总体架构设计

> 文档状态：已按模块插件重构实现更新

## 1. 仓库定位

`chips-scaffold-module` 是开发期工具仓库，只负责生成标准模块工程，不承载 Host 运行时主实现。

生成后的模块工程在生态中的正式定位是：

- 安装到 Host 的无界面能力模块
- 通过 `manifest.module` 声明 provider
- 通过 `capability + method` 对外提供能力

## 2. 内部分层

本仓分为四层：

1. `templates/`
   保存模板文件和模板元数据。
2. `src/core/`
   负责模板发现、参数校验、文件渲染和目录保护。
3. `src/cli/`
   负责向 `chipsdev` 暴露创建 API。
4. `tests/` 与 `scripts/`
   负责模板检查、单测和生成工程自检。

依赖方向固定为：

`templates -> core -> cli -> chipsdev`

## 3. 关键设计决策

- 标准模板直接对齐 `module.listProviders/resolve/invoke/job.*` 新契约。
- 模板 Manifest 以 `module.provides/module.consumes` 为正式契约来源。
- 模板默认生成同步方法与异步 job 方法示例。
- 模板默认生成输入输出 schema 文件。
- 模板不生成 React、主题、多语言和页面挂载运行时。
- 历史错误模板统一移入 `归档/`，渲染器会主动跳过。

## 4. 跨仓边界

- 公共协议、Manifest、Host 动作口径以 `生态共用技术文档/` 为准。
- SDK 类型化封装以 `Chips-SDK` 为准。
- Host 模块运行时治理以 `Chips-Host` 为准。
- 本仓只维护模板结构、渲染逻辑和生成工程基线。
