# 模板目录与文件组织设计

## 1. 模板目录结构

当前第一版正式模板为 `templates/module-standard/`，目录设计如下：

```text
templates/module-standard/
├─ template.json
├─ .eslintrc.cjs.tpl
├─ .gitignore.tpl
├─ README.md.tpl
├─ chips.config.mjs.tpl
├─ manifest.yaml.tpl
├─ package.json.tpl
├─ tsconfig.json.tpl
├─ vitest.config.mts.tpl
├─ config/
│  └─ logging.ts.tpl
├─ i18n/
│  ├─ zh-CN.json.tpl
│  └─ en-US.json.tpl
├─ src/
│  ├─ index.ts.tpl
│  └─ module/
│     ├─ runtime.tsx.tpl
│     ├─ types.ts.tpl
│     └─ i18n.ts.tpl
└─ tests/
   └─ unit/
      └─ module-runtime.test.tsx.tpl
```

## 2. 生成工程结构

生成后的模块工程遵循以下组织：

- 根目录保存清单、构建配置、包配置与 README；
- `src/index.ts` 作为唯一公共导出入口；
- `src/module/` 收口运行时实现与模块域类型；
- `config/` 放置与运行时无关的通用配置工具；
- `i18n/` 放置本地业务文案；
- `tests/unit/` 放置生成工程最小可运行测试。

## 3. 文件组织原则

- Bridge/SDK 调用逻辑不直接散落到多个 UI 组件，而是集中在模块运行时层；
- 词典读取与格式化逻辑单独放在 `src/module/i18n.ts`；
- 运行时公共类型单独放在 `src/module/types.ts`；
- `README.md` 对生成工程使用者可见，必须和模板内容同步。

## 4. 为什么不额外拆分更多模板

第一版先冻结 `module-standard`，原因如下：

- 当前生态首先需要一条正式、稳定、可测试的“模块工程创建入口”；
- 在契约刚刚收敛完成的阶段，先以单模板闭环比一次性铺多个变体更稳妥；
- 后续可以在此基础上新增例如播放器类、渲染器类、编辑器类等模块模板变体，但不能影响第一版正式链路。
