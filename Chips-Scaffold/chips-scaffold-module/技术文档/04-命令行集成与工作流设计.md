# 命令行集成与工作流设计

## 1. CLI 入口

`chipsdev` 当前正式支持：

```bash
chipsdev create module <targetDir>
```

与模块脚手架的集成位于 `Chips-SDK/cli/index.js`，流程如下：

1. 解析 `type === "module"`；
2. 定位 `Chips-Scaffold/chips-scaffold-module`；
3. 确保脚手架包已构建；
4. 构造 `CreateModuleProjectOptions`；
5. 调用 `createModuleProject(options)`；
6. 对生成工程执行工作区注册与 `volta.extends` 适配。

## 2. 默认参数推导

- `projectName`：取目标目录名；
- `slug`：由目标目录名规整化而来；
- `pluginId`：`chips.module.${slug.replace(/-/g, ".")}`；
- `moduleCapability`：`module.${slug.replace(/-/g, ".")}`；
- `displayName`：由目录名自动转为标题格式；
- `version`：默认 `0.1.0`。

## 3. 工作区适配

当目标目录位于生态根工作区内时：

- 若当前工作区配置未覆盖该路径，自动把工程加入根 `workspaces`；
- 自动写入相对根目录的 `volta.extends`；
- 不注入 `file:` 或其他临时依赖。

## 4. 错误处理

- 脚手架包缺失：直接报错并停止；
- 脚手架未构建：先触发构建，再继续；
- 模板渲染失败：直接透传明确错误；
- 不支持的 create 类型：列出当前正式支持类型。

## 5. 后续扩展点

未来若需要增加 `--template`、`--plugin-id`、`--capability` 等显式参数，应遵循：

- 不破坏当前默认行为；
- 不引入第二套模块能力推导口径；
- 在 CLI 文档与模板 README 中同步更新。
