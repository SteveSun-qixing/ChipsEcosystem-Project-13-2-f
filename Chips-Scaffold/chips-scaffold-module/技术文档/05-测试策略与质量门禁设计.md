# 模块插件脚手架测试策略与质量门禁设计

## 1. 测试分层

### 1.1 脚手架仓库单测

- `tests/core/template-engine.test.ts`
- 覆盖模板枚举、模板渲染、关键文件存在与核心占位符落地。

### 1.2 模板完整性检查

- `scripts/check-templates.cjs`
- 校验 `template.json` 与模板最小文件集是否齐全。

### 1.3 生成工程最小自检

- `scripts/run-generated-e2e.cjs`
- 验证模板可生成基本工程骨架，且关键文件存在。

### 1.4 CLI 工作区集成测试

- `Chips-SDK/tests/run-cli-create-module-workspace-tests.cjs`
- 在临时生态工作区中执行：
  - `chipsdev create module`
  - 根工作区 `npm install`
  - 生成工程 `lint/build/test/validate/package`

## 2. 质量门禁

第一版正式门禁如下：

1. `chips-scaffold-module` 可构建；
2. 模板渲染单测通过；
3. 模板目录检查通过；
4. `chipsdev create module` 工作区集成测试通过；
5. 相关文档与问题工单同步更新。

## 3. 为什么保留双层测试

- 脚手架单测用于快速定位模板引擎问题；
- CLI 集成测试用于验证真实开发者路径，不让“脚手架能渲染但 CLI 不可用”再次出现；
- 两层测试共同约束“模板、CLI、工作区、打包”四条链路。

## 4. 后续增强方向

- 增加多模板矩阵测试；
- 增加生成工程的产物结构断言；
- 增加模板 README 与文档同步漂移检测；
- 必要时引入跨仓契约测试，验证模板与 Host/SDK 的模块字段长期一致。
