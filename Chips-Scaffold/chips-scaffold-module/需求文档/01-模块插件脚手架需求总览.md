# 模块插件脚手架需求总览

> 文档状态：已按模块插件重构实现更新
> 对齐基线：生态设计原稿 + 生态共用技术文档 + Host / SDK / CLI 当前实现

## 1. 项目目标

`chips-scaffold-module` 的职责，是为生态提供“官方标准模块工程起点”。

当前正式目标已经收敛为：

- 模块插件是安装到 Host 中的无界面能力模块；
- 调用模型是 `capability + method`，而不是 `slot + mount`；
- 工程模板必须默认生成 `manifest.module`、schema 契约、方法实现示例和测试；
- `chipsdev create module` 生成的工程必须能直接通过 `lint/build/test/validate/package`；
- 生成的包必须能被 Host 安装、启用并通过 `module.invoke` 正常调用。

## 2. 当前范围

本仓负责：

- 模板目录与模板元数据；
- 模板渲染引擎；
- `createModuleProject` 对外 API；
- `chipsdev create module` 所依赖的模板基线；
- 模板自身测试、模板检查和生成工程自检；
- 本仓内部需求、技术和计划文档。

本仓不负责：

- Host 模块运行时主实现；
- SDK 模块 API 主实现；
- 生态公共契约的发布口径。

## 3. 生成工程的正式定位

脚手架生成的模块工程必须满足以下定位：

- `type: module`
- `entry: dist/index.mjs`
- 使用 `manifest.module.provides/module.consumes`
- 默认提供一个 capability
- 默认提供一个同步方法和一个异步 job 方法
- 默认提供与方法一一对应的 schema 文件
- 默认提供最小可运行单测

模板不得再生成：

- React 运行时
- `runtime.tsx`
- 页面挂载入口
- 主题注入逻辑
- i18n UI 词典
- `module.mount(...)` / `bridgeScopeToken` 旧口径示例

## 4. 关键质量目标

- 文档、模板、CLI、Host、SDK 口径一致；
- 归档目录中的历史模板文件不会再被复制到新工程；
- 生成工程在默认状态下具备完整可验证链路；
- 不留下 TODO、占位实现或临时兼容逻辑。

## 5. 里程碑

1. 冻结模块插件新契约与脚手架边界。
2. 重做 `module-standard` 模板为无界面 capability 模块模板。
3. 打通 `chipsdev create module`、`validate`、`package`。
4. 通过生成工程到 Host 安装调用的全链路验证。
5. 同步收口本仓与相关仓的文档。
