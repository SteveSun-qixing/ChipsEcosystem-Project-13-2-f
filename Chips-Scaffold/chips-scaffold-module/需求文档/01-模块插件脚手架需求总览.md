# 模块插件脚手架需求总览

> 文档状态：需求基线稿  
> 对齐基线：生态设计原稿 + 生态共用技术文档 + Host/SDK/CLI 当前实现

---

## 1. 项目背景与目标

薯片生态把模块插件定义为五类正式插件之一。模块插件承担“共享功能模块”的职责，用于为 Host 与上层插件补充可热插拔的通用能力，例如渲染器、编辑器、播放器、预览器与其他可复用界面模块。

在本项目开始前，生态存在三个明显问题：

- 模块正式契约在 Bridge 文档、Host 实现、SDK 封装之间存在漂移；
- `chipsdev create` 没有 `module` 正式创建入口；
- `Chips-Scaffold/` 中没有模块插件脚手架仓库，无法以官方标准方式创建模块工程。

模块插件脚手架项目的目标，是为生态提供一条从文档、模板、CLI 到测试门禁都完整闭环的正式开发入口：

- `chipsdev create module <targetDir>` 可直接生成标准模块插件工程；
- 生成工程默认符合 `type: module`、`entry: dist/index.mjs`、`capabilities`、权限、主题、多语言、React 与 SDK 使用规范；
- 生成工程无需修改模板代码即可通过 `npm run lint/build/test/validate` 与 `chipsdev package`；
- 脚手架自身具备模板检查、单测与工作区集成测试。

### 1.1 北极星目标

- 形成官方标准模块插件模板体系，作为未来所有模块插件工程的统一起点。
- 让模块插件的工程结构、Manifest、Bridge/SDK 用法、主题接入、多语言接入与测试门禁保持一致。
- 将“模块契约修复 + 模块脚手架 + CLI 工作流 + 文档体系”一次性交付为正式能力，而不是临时拼装。

### 1.2 本轮交付目标

本轮交付聚焦于一套完整的第一版正式闭环：

- 修正模块正式契约相关文档；
- 实现 `Chips-Scaffold/chips-scaffold-module`；
- 实现 `chipsdev create module`；
- 补齐模板自身测试与工作区集成测试；
- 补齐项目级需求文档、技术文档、开发计划；
- 更新问题工单，记录本轮修复结果与落地范围。

## 2. 范围与边界

### 2.1 In Scope

- `chips-scaffold-module` 仓库自身的模板、模板渲染引擎、对外 API、测试与文档；
- 模块插件的标准模板 `module-standard`；
- `chipsdev create module` 的 CLI 接入；
- 与 Host/SDK/Bridge 正式契约对齐所需的文档更新；
- 模块脚手架项目自身的开发计划、质量门禁与工单收口。

### 2.2 Out of Scope

- 不在本仓中实现新的 Host 模块运行时协议；
- 不新增未经过批准的第三方库；
- 不修改与当前任务无关的插件或应用工程；
- 不引入向后兼容层、双协议兼容层或临时字段。

### 2.3 归属边界

- Host `module.*` 动作、Manifest 正式字段、Bridge/API 正式口径归生态共用文档与 Host/SDK 实现；
- 模块脚手架内部的模板组织、占位变量、目录结构、测试脚本与 CLI 调用适配归本仓内部设计；
- 若后续需要把模板目录结构升级为“所有模块插件必须遵守”的生态统一规范，应先更新生态共用技术文档，再由本仓引用。

---

## 3. 核心约束

- 只能基于当前已冻结的正式契约开发，不允许在模板中固化未正式实现的临时协议。
- 生成工程必须完整可运行，不得包含 TODO、占位函数、简化版运行时或临时开发代码。
- 前端技术栈为 React，且必须遵循生态主题系统、多语言系统与 SDK 使用规范。
- 不做大版本兼容设计；本轮直接以第一版正式架构为准。
- 每完成一块实现，都要同步更新对应文档与记录。

## 4. 干系人与协作关系

| 干系人 | 角色 | 关注点 |
|---|---|---|
| 模块插件开发者 | 脚手架直接使用者 | 工程创建体验、目录结构、主题/i18n/SDK 用法、测试可运行性 |
| `Chips-SDK` 维护者 | CLI 与 SDK 提供方 | `chipsdev create module` 对接稳定，模板示例正确使用 SDK |
| `Chips-Host` 维护者 | Host 服务与插件治理提供方 | 模块工程与 `module.*`、Manifest、权限与安装链路保持一致 |
| 生态架构维护者 | 正式规范维护方 | 文档、实现与问题工单一致，不出现新漂移 |
| 未来模块插件仓库 | 模板消费者 | 能基于统一脚手架快速演进不同类型模块插件 |

## 5. 里程碑

1. 契约冻结：统一 `mount/unmount/query/list + slot/moduleId` 正式口径。
2. 脚手架实现：补齐 `chips-scaffold-module` 与 `module-standard`。
3. CLI 闭环：补齐 `chipsdev create module` 与工作区注册。
4. 质量门禁：完成模板检查、单测、生成工程集成测试。
5. 文档与工单收口：完成项目级文档与工单状态更新。

本文件定义的是项目级目标与边界。具体功能点见《02-模块插件脚手架功能需求规格》，质量门禁见《03-模块插件脚手架非功能与质量验收需求》。
