# 模块插件脚手架功能需求规格

> 文档状态：功能需求基线稿

---

## 1. 模板管理能力

### FR-MODULE-SCAFFOLD-001 模板枚举

- 脚手架必须提供模板枚举能力，用于返回当前可用模块模板列表。
- 模板元数据至少包含：
  - `id`
  - `name`
  - `description`
  - `version`
  - 是否集成 SDK / React / 主题系统 / 多语言系统

### FR-MODULE-SCAFFOLD-002 模板元数据校验

- 模板目录必须包含 `template.json`；
- 模板元数据缺失 `id/name/version` 任一字段时，脚手架必须以标准错误阻断；
- 模板目录为空或元数据不合法时，不得继续生成工程。

## 2. 工程创建能力

### FR-MODULE-SCAFFOLD-003 工程创建参数

- 创建模块工程时必须至少提供：
  - `projectName`
  - `targetDir`
  - `templateId`
  - `pluginId`
  - `moduleCapability`
  - `displayName`
  - `version`

### FR-MODULE-SCAFFOLD-004 目标目录保护

- 若目标目录已存在且非空，脚手架必须拒绝覆盖；
- 不允许用临时覆盖、清空目录等方式绕过此限制。

### FR-MODULE-SCAFFOLD-005 模板渲染

- 脚手架必须将模板文件完整渲染到目标目录；
- 文本文件使用占位符替换；
- 非文本文件按原样复制；
- 生成结果必须返回工程路径、模板 ID 与生成文件数量。

## 3. 标准模块模板能力

### FR-MODULE-TEMPLATE-001 工程骨架

- 标准模板至少包含：
  - `.eslintrc.cjs`
  - `manifest.yaml`
  - `package.json`
  - `tsconfig.json`
  - `chips.config.mjs`
  - `vitest.config.mts`
  - `src/index.ts`
  - `src/module/runtime.tsx`
  - `src/module/types.ts`
  - `src/module/i18n.ts`
  - `config/logging.ts`
  - `i18n/en-US.json`
  - `i18n/zh-CN.json`
  - `tests/unit/module-runtime.test.tsx`
  - `README.md`

### FR-MODULE-TEMPLATE-002 Manifest 正式约束

- 模板生成的 `manifest.yaml` 必须默认满足：
  - `type: module`
  - `entry: dist/index.mjs`
  - `capabilities: ["<moduleCapability>"]`
  - 必要的作者、版本、系统平台与引擎字段
- 若模板默认运行时会读取主题与语言状态，则必须默认声明 `theme.read` 与 `i18n.read`。

### FR-MODULE-TEMPLATE-003 React 运行时

- 模板必须默认提供可嵌入的 React 模块运行时；
- 运行时必须提供明确的 `mount/update/unmount` 生命周期；
- 模板必须体现 provider 化的运行时状态组织，而不是把 Bridge、主题、语言逻辑散落在多个组件中。

### FR-MODULE-TEMPLATE-004 主题与多语言

- 模板必须展示主题系统与多语言系统的正式接入方式：
  - 使用 `chips-sdk` 读取当前主题/主题 CSS；
  - 使用 `chips-sdk` 读取当前 locale；
  - 当调用方传入 `bridgeScopeToken` 时，必须优先以 scoped client 访问 Host；
  - 使用本地词典文件管理业务 UI 文案；
  - 响应 `theme.changed` 与 `language.changed` 事件。

### FR-MODULE-TEMPLATE-004A 模块独立身份

- 标准模板必须支持消费 `module.mount(...)` 返回的 `bridgeScopeToken`；
- 模块运行时建立 `chips-sdk` 客户端时，不得默认继续复用调用方宿主身份；
- 若调用方未提供 `bridgeScopeToken`，模板可回退到当前页面 Bridge，仅用于本地预览或降级场景。

### FR-MODULE-TEMPLATE-005 测试可运行

- 生成工程在默认状态下必须通过：
  - `npm run lint`
  - `npm run build`
  - `npm test`
  - `npm run validate`
  - `chipsdev package`

## 4. CLI 集成能力

### FR-MODULE-CLI-001 create module

- `chipsdev` 必须支持：

```bash
chipsdev create module <targetDir>
```

- CLI 必须自动：
  - 推导 `projectName`
  - 推导 `pluginId`
  - 推导 `moduleCapability`
  - 选择默认模板 `module-standard`

### FR-MODULE-CLI-002 工作区接入

- 当目标工程位于生态根工作区内时，CLI 必须：
  - 自动注册工作区；
  - 自动写入正确的 `volta.extends`；
  - 保留模板中的正式 semver 依赖。

### FR-MODULE-CLI-003 错误可诊断

- 当脚手架缺失、构建产物缺失、模板无效或参数不合法时，CLI 必须输出明确错误；
- 不得在报错时静默回退到其他插件类型或其他模板。

## 5. 文档与记录能力

### FR-MODULE-DOC-001 项目文档体系

- 模块脚手架仓库必须提供完整的需求文档、技术文档与开发计划；
- 每一章节应拆分为独立文档文件，方便后续迭代和审查。

### FR-MODULE-DOC-002 README 同步

- 标准模板生成的 `README.md` 必须说明：
  - 工程定位；
  - 快速开始；
  - 目录结构；
  - 主题/多语言/SDK 使用方式；
  - 正式约束与工作流。

### FR-MODULE-DOC-003 问题工单协同

- 如实现过程中发现新的跨生态缺口，必须先登记问题工单，再决定是否继续推进；
- 已解决的工单需在本轮交付完成时更新状态和修复说明。
