# 模块插件脚手架非功能与质量验收需求

> 文档状态：质量基线稿

---

## 1. 稳定性要求

- 模板渲染必须是确定性的：相同输入参数生成相同文件结构与文件内容。
- `chipsdev create module` 不能依赖未声明的人工步骤或隐式临时目录。
- 模板运行时在 `unmount` 时必须完整释放 React root 与事件监听。

## 2. 一致性要求

- 文档、Host 实现、SDK 封装、CLI 行为、模板 README 必须使用同一套模块正式口径。
- 生成工程中的 Manifest、构建配置、README、测试与模板目录说明必须同步一致。
- 不能同时保留 `module.load/unload` 与 `module.mount/unmount` 双口径。

## 3. 可维护性要求

- 模板变量命名必须语义明确，避免同义多名；
- 模板引擎逻辑保持最小化，复杂逻辑优先放在模板工程内部；
- 运行时状态管理与展示组件必须解耦，避免单组件膨胀。

## 4. 代码质量要求

- 不允许留下 TODO、FIXME、占位实现、空测试或永远不会运行的分支；
- 不允许引入未经批准的新第三方库；
- 生成工程默认代码必须能被 ESLint、Vitest 与 TypeScript 正常处理。

## 5. 测试要求

- 脚手架仓库自身至少应具备：
  - 模板枚举/渲染单测；
  - 模板目录完整性检查；
  - 生成工程最小自检；
- CLI 侧至少应具备：
  - `chipsdev create module` 工作区集成测试；
  - 生成工程的 `lint/build/test/validate/package` 闭环测试。

## 6. 文档同步要求

- 每次模块契约、模板目录、CLI 行为或测试链路发生变化，都必须同步更新文档；
- README 与需求/技术/开发计划文档不得长期背离；
- 问题工单与实现状态必须可相互追溯。

## 7. Git 与交付要求

- 开发必须在 `task-` 前缀分支上进行；
- 不允许删除现有文件，若未来确需移除，必须转入归档目录；
- 完成一轮正式开发后，必须先通过测试与文档收口，再等待审核，不得自行合并分支。

## 8. 验收标准

满足以下条件，视为本项目达到当前阶段验收要求：

1. `chips-scaffold-module` 可构建、可测试，模板完整；
2. `chipsdev create module` 可在临时生态工作区中创建真实模块工程；
3. 生成工程可通过 `lint/build/test/validate/package`；
4. 相关生态文档已更新到统一正式口径；
5. 项目级需求文档、技术文档、开发计划与问题工单已同步更新。
