# 阶段二：标准主题模板骨架实现

> 阶段目标：在不引入多模板与复杂集成的前提下，完成 `theme-standard` 标准主题插件模板及基础构建能力，使其在手工调用时即可生成可构建、可校验的主题工程。

---

## 1. 阶段范围

- 在本仓内创建标准主题插件模板目录与文件：`templates/theme-standard/`；
- 为脚手架核心层实现最小可用的模板渲染能力：
  - 支持 `CreateThemeProjectOptions` → 工程目录的生成流程；
  - 暂不与 `chips dev` 直接集成（与 CLI 的集成工作在阶段三进行）；
- 为 `theme-standard` 模板补齐：
  - Manifest、package、TS 配置与 chips 配置模板；
  - token 源文件模板（global/semantic/typography/motion/components）；
  - 五层 token 构建脚本与基础 CSS / icons / contracts / preview 模板；
  - 最小可用的单元测试与构建校验脚手架。

---

## 2. 关键任务

### 2.1 仓库初始化与基础配置

- 在 `chips-scaffold-theme` 根目录创建（如尚未存在）：
  - `package.json`：声明包名、基本脚本（lint/test/build 占位即可，但需可运行）；
  - `tsconfig.json`：脚手架实现代码的 TS 配置；
  - `.eslintrc` / `.prettierrc`（如生态已有统一模板，可复用）。
- 初始化 `src/core/`、`templates/`、`tests/` 等目录结构，但不写入临时代码。

### 2.2 theme-standard 模板资产落地

- 按《技术文档/01-主题包插件脚手架技术架构与模板设计》创建 `templates/theme-standard/` 目录结构：
  - `manifest.yaml.tpl`、`package.json.tpl`、`tsconfig.json.tpl`、`chips.config.mjs.tpl`、`README.md.tpl` 等配置类模板；
  - `tokens/` 子目录及 `global/semantic/typography/motion/components` token 源文件模板；
  - `src/build-tokens.ts.tpl`、`src/build-css.ts.tpl`、`src/validate-theme.ts.tpl` 等构建与校验脚本模板；
  - `styles/`、`icons/`、`fonts/`、`contracts/`、`preview/` 与 `tests/` 子目录及对应模板文件。
- 确保模板内容符合：
  - 主题包开发指南与主题接口规范；
  - Manifest 配置规范与插件开发规范；
  - Token 与主题对接标准；
  - 图标与字体系统设计规范。

### 2.3 渲染核心最小实现

- 在 `src/core/` 中实现：
  - 模板元数据加载与验证函数（与其他脚手架仓库尽量共享模型与实现形态）；
  - 占位符替换与文件复制函数；
  - `createThemeProject(options)` 的最小实现（接受参数、渲染 `theme-standard` 模板到给定目录）。
- 暂不暴露对 `chips dev` 的公共 API，仅提供内部可调用函数，便于阶段三集成。

### 2.4 基础测试与自测脚本

- 单元测试：
  - 针对占位符替换与模板元数据解析实现测试；
  - 针对 `createThemeProject` 生成的基本目录结构与关键文件存在性测试。
- 模板一致性测试：
  - 检查 `theme-standard` 模板目录完整性与关键文件存在；
  - 确认 token 源文件、构建脚本与契约文件模板之间的引用路径正确。
- 生成工程自测脚本：
  - 在临时目录中调用内部 `createThemeProject` 生成工程；
  - 在生成工程中执行：
    - `npm install`（可通过缓存优化或跳过真实安装，视 CI 能力而定）；
    - `npm run build`（至少执行 token 构建脚本，不要求发布级压缩）；
    - `npm run validate:theme`（在本地 Host 工作空间中调用 `chips host theme validate`，验证基础契约闭环）。

---

## 3. 依赖与前置条件

- 阶段一文档与规划已完成，且未发现阻断脚手架实现的架构级缺陷；
- 生态对主题系统、五层 token 架构、图标与字体体系、测试框架（Vitest 等）等技术选型已冻结或有明确口径；
- 如在实现过程中发现模板所需能力在 Host/SDK/组件库/CLI 中尚未落地，需立即：
  - 在问题工单中登记；
  - 评估是否可以通过“示例覆盖较小子集、暂不暴露完整组件列表”等方式暂缓，前提是生成工程仍然可构建与通过基础门禁；
  - 或将相关模板功能下沉至后续阶段。

---

## 4. 验收标准

- `templates/theme-standard/` 目录结构与内容完整，与技术文档描述一致；
- `src/core/` 中已实现最小可用的 `createThemeProject`，可在本地通过脚本调用生成工程；
- 从模板生成的主题工程在本机（至少一个支持平台）满足：
  - `npm install` 可成功或在约定策略下被安全跳过；
  - `npm run build` 能运行并产出 `dist/tokens.json` 与 `dist/theme.css`；
  - `npm run validate:theme` 能在本地 Host 上执行并通过基础校验（无致命错误）。
- 本阶段新增的代码与模板已通过本仓的基础质量门禁（lint + 单元测试）。

---

## 5. 输出物

- `templates/theme-standard/` 标准主题插件模板；
- `src/core/` 中的模板渲染最小实现（`createThemeProject` 及其依赖）；
- 覆盖模板渲染、模板一致性与基础构建的单元/集成测试；
- 阶段总结，记录：
  - 模板设计中做出的关键取舍（例如默认覆盖的组件范围、token 粒度等）；
  - 与后续阶段（扩展模板与 CLI 集成）的依赖关系与注意事项。
