# 阶段三：扩展模板与生态对接

> 阶段目标：在完成 `theme-standard` 模板的基础上，扩展多模板能力并与 `chips dev` 命令行完成集成，使开发者可以通过标准命令创建主题插件工程。

---

## 1. 阶段范围

- 在 `templates/` 下根据需要新增主题插件模板变体（例如 `theme-derived` 等）；
- 为脚手架核心层增加模板元数据与多模板管理能力；
- 在 `src/cli/` 中实现对外 API，并与 `chips dev create` 命令完成集成；
- 确保多模板场景下的测试与质量门禁覆盖。

---

## 2. 关键任务

### 2.1 模板变体设计与落地

- 与产品/架构方确认首批主题模板变体清单及目标场景：
  - 例如：
    - `theme-standard`：标准官方主题模板，覆盖核心组件与 Host UI；
    - `theme-derived`：基于父主题的差异化派生模板，仅覆盖部分 token；
  - 若暂时无法明确多模板清单，多模板能力仍需在代码层预留，但可仅维护单一模板。
- 为每个模板变体补齐：
  - 模板目录与元数据文件；
  - 与 `theme-standard` 一致的基础能力（token 构建、契约与校验脚本、测试骨架等），在此基础上调整继承策略与覆盖粒度。

### 2.2 模板元数据与选择逻辑

- 在 `src/core/` 中实现模板元数据加载模块：
  - 扫描 `templates/` 目录，构建主题模板清单；
  - 校验元数据文件与目录结构一致性；
- 为 CLI 集成层提供 `listThemeTemplates()` 接口：
  - 返回模板 ID、名称、描述等信息；
  - 用于在交互式命令行中展示可选主题模板列表。

### 2.3 CLI 集成层实现

- 在 `src/cli/` 中实现并导出：
  - `listThemeTemplates()`：供 `chips dev` 查询主题模板清单；
  - `createThemeProject(options)`：包装核心层的实现，并提供更友好的错误信息；
- 与 `chips dev` 仓库协作：
  - 在 `chips dev create` 中接入上述接口，当类型为 `theme` 时调用本脚手架；
  - 定义并实现命令行参数与交互式问答流程（例如选择模板、输入 `themeId`、 `publisher` 等）；
  - 如有需要，更新《Chips Dev 开发者命令行手册》中与主题脚手架相关的说明。

### 2.4 多模板测试与回归

- 扩展模板一致性测试：
  - 确保所有主题模板变体都有完整的关键文件与元数据；
  - 检查 token 源文件、构建脚本与契约文件模板在各模板中的一致性。
- 扩展生成工程 E2E 测试：
  - 对每个模板至少执行一轮“创建 → 安装依赖 → 构建 → `validate:theme`”的自动化检查；
- 针对 CLI 集成部分增加集成测试：
  - 可在 `chips dev` 仓库中实现端到端 CLI 测试，本仓至少提供可用于模拟调用的 API 与测试用例。

---

## 3. 依赖与前置条件

- 阶段二已完成 `theme-standard` 模板与基础渲染实现；
- `chips dev` 仓库已预留扩展点或存在明确的接入路径（例如通过依赖本包导出的 CLI API）；
- 如需在生态共用文档中新增“主题模板清单”或“主题脚手架使用说明”，需与架构/规范维护者提前对齐口径。

---

## 4. 验收标准

- 至少一个模板（如 `theme-standard`）已通过 `chips dev create` 端到端验证：
  - 在用户终端中可通过交互式命令完成主题工程创建；
  - 生成工程可构建、可执行 `validate:theme`，并在本地 Host 上通过基础校验；
- 多模板机制工作正常（若已有多个模板）：
  - `listThemeTemplates()` 返回的模板列表与实际目录一致；
  - 通过不同模板创建的工程目录结构与内容符合设计预期；
- 《Chips Dev 开发者命令行手册》中如有需要，已补充“主题包插件脚手架”相关使用说明；
- 本阶段新增/修改的代码与模板已通过本仓质量门禁与相关仓库（如 `chips dev`）的 CI 检查。

---

## 5. 输出物

- 支持多模板能力的脚手架核心与 CLI 集成实现；
- 至少一套已能通过 `chips dev create` 端到端验证的主题插件模板；
- 扩展后的模板一致性测试与 E2E 测试用例；
- 阶段总结，记录：
  - 模板变体之间的差异化点（例如继承粒度、覆盖范围）；
  - 与 `chips dev` 集成过程中发现并解决的问题；
  - 后续需要在生态共用文档中补充或调整的内容。
