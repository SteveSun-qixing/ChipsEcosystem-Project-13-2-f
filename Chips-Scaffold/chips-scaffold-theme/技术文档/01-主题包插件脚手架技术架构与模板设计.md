# 主题包插件脚手架技术架构与模板设计

> 文档状态：技术架构基线设计稿（仅文档，不含实现代码）  
> 适用范围：`Chips-Scaffold/chips-scaffold-theme` 仓库的总体技术方案、模板结构与集成方式

---

## 1. 总体定位与上下游关系

### 1.1 在生态中的位置

主题包插件脚手架是薯片生态“开发工具”侧的一层薄封装，位于：

- **上游**：开发者与 `chips dev create` 命令：
  - 开发者通过命令行选择「主题插件」类型与具体模板；
  - `chips dev` 根据配置从本仓库读取模板元数据与文件，生成新的主题插件工程。
- **下游**：Host Theme Runtime / SDK / 组件库：
  - 生成的主题插件工程打包为 `.cpk` 后由 Host `plugin.install` / `theme.apply` 等动作加载；
  - 主题 tokens 由 L11 Theme Runtime 解析，并通过 `theme.resolve/getAllCss/contract.get` 等接口向 L9/L10 暴露；
  - 组件库通过 `chips.comp.*` token 与 CSS 变量消费主题。

脚手架自身 **不参与运行时链路**，只承担：

- 标准化主题插件工程目录结构与配置；
- 严格对齐 `生态共用技术文档` 的主题系统设计与契约；
- 为主题开发者提供最小可用、可测试、可扩展的项目起步模板。

### 1.2 与其他脚手架仓库的关系

- 与 `Chips-Scaffold/chips-scaffold-app`、`chips-scaffold-basecard` 等仓库共同构成统一的脚手架体系；
- 顶层目录结构（`templates/需求文档/技术文档/开发计划`）保持一致，便于工具与文档复用；
- 仅在模板内容与生成工程结构上体现“主题插件”的专有差异。

## 2. 模板目录结构设计

主题脚手架内部以模板目录承载可选的工程基线。命名约定：

- 顶层使用 `templates/` 目录收纳所有模板；
- 主题模板使用前缀 `theme-`，例如：
  - `theme-standard`：标准官方主题模板（覆盖核心组件集与 Host UI）；
  - `theme-derived`：基于父主题的差异化派生模板；
  - 后续如需更多变体（高对比度主题、实验型主题等），可在后续阶段补充。

示例目录结构（规划）：


约束：

- 所有模板文件使用 `.tpl` 后缀，渲染生成工程时去掉该后缀；
- `manifest.yaml.tpl` 必须符合《Manifest配置规范》与《主题包开发指南》中 `type: theme` 的字段要求；
- token 源文件模板必须与 Style Dictionary 与 Theme Runtime 的期望结构一致（详见第 4 节）。

## 3. 生成工程目录结构约定

以 `theme-standard` 模板生成的主题插件工程为例，最终目录结构规划如下（略去部分细节文件）：

```text
my-theme/
├─ manifest.yaml
├─ package.json
├─ tsconfig.json
├─ chips.config.mjs
├─ README.md
├─ tokens/                    # 设计 token 源文件（Style Dictionary 输入）
│  ├─ global.json
│  ├─ semantic.json
│  ├─ typography.json
│  ├─ motion.json
│  └─ components/
│     ├─ button.json
│     ├─ input.json
│     └─ ...
├─ src/
│  ├─ build-tokens.ts        # 将设计 token 构建为 ref/sys/comp/motion/layout 结构
│  ├─ build-css.ts           # 基于 token 输出 CSS 变量与组件样式
│  └─ validate-theme.ts      # 调用 Host/SDK 执行主题校验（contract/contrast/motion）
├─ dist/
│  ├─ tokens.json            # 供 Theme Runtime 消费的五层 token 结构
│  ├─ theme.css              # 完整主题 CSS（供 theme.getAllCss 使用）
│  └─ assets/                # 打包后的 icons/fonts/preview 等
├─ styles/
│  ├─ base.css               # 全局与语义层 CSS 变量注入
│  ├─ components/
│  │  ├─ button.css
│  │  ├─ input.css
│  │  └─ ...
│  └─ motions.css
├─ icons/
│  ├─ icons.css
│  └─ svg/
│     └─ ...
├─ fonts/
│  └─ ...
├─ contracts/
│  └─ theme-interface.contract.json
├─ preview/
│  ├─ preview.png
│  └─ preview.html
└─ tests/
   ├─ tokens.spec.ts
   ├─ contract.spec.ts
   └─ build.spec.ts
```

关键约束：

- `manifest.yaml`：
  - 必须位于工程根目录；
  - `type: theme`、`themeId`、`version`、`parentTheme` 等字段符合《主题包开发指南》《Manifest配置规范》；
  - 权限声明默认使用最小集（通常只需要主题相关只读能力），由脚手架模板提供基线。
- `tokens/`：
  - 作为设计 token 源文件目录，遵循《主题包开发指南》中 `global/semantic/typography/motion/components` 分层；
  - 不直接暴露给 Theme Runtime，需经 `src/build-tokens.ts` 构建为 `dist/tokens.json`。
- `dist/tokens.json`：
  - 输出为 Theme Runtime 所需的五层结构：
    - `ref/sys/comp/motion/layout`；
  - 结构必须通过 `Chips-Host` 侧 `asThemeTokenLayers` 校验。
- `styles/` 与 `icons/`：
  - 通过 CSS 变量对接 token，命名遵循 `--chips-*` 与 `chips.comp.*` 的约定；
  - 图标与字体资源使用《图标与字体系统设计规范》中的资产来源与回退链。
- `tests/`：
  - 至少覆盖 token 完整性与契约映射、构建脚本关键路径、基础颜色对比度规则。

## 4. token 构建链路与五层结构映射

### 4.1 设计 token 源与目标结构

根据《主题包开发指南》和《主题系统设计》，脚手架采用“两层模型”：

- **源层（设计视角）**：面向设计工具与视觉设计师的 token 结构：
  - `tokens/global.json`：色板、间距、圆角等基础值；
  - `tokens/semantic.json`：primary/surface/error 等语义值；
  - `tokens/typography.json`：字体族、字号与字重；
  - `tokens/motion.json`：动效时长与缓动曲线；
  - `tokens/components/*.json`：组件级 token。
- **目标层（运行时视角）**：Theme Runtime 消费的五层结构：
  - `ref`：原始值层；
  - `sys`：语义值层；
  - `comp`：组件值层；
  - `motion`：动效值层；
  - `layout`：布局密度层。

脚手架需要提供默认的构建脚本（`src/build-tokens.ts`）完成从**源层 JSON**到**目标层 JSON**的映射：

- 允许使用 Style Dictionary 等工具作为中间构建管线（依赖选型需与《主题包开发指南》中推荐工具保持一致）；
- 映射规则与命名约定在脚本内通过常量与类型显式定义，避免散落在多个文件中；
- 生成的 `dist/tokens.json` 必须可以被 `Theme Runtime` 的 `mergeThemeLayers/resolveThemeFromLayers` 正常消费。

### 4.2 token 命名与组件对接

- 所有组件级 token 必须遵循《Token 与主题对接标准》中的前缀：
  - `chips.comp.button.*`、`chips.comp.input.*` 等；
  - 构建脚本负责将内部 JSON 键映射到符合前缀要求的扁平键名；
- `ref/sys` 层的键名设计应避免业务语义，保持通用性（例如 `color.blue.500` 而非 `todo-card-bg`）；
- `layout` 层建议集中管理列表密度、间距、容器高度等参数，后续可与 L8/L9 的布局策略联动。

## 5. 门禁与自测集成设计

主题脚手架需要为生成工程提供可直接运行的最小门禁链路，建议默认支持：

1. **本地构建与打包**
   - `npm run build`：
     - 调用 `build-tokens` 与 `build-css`；
     - 产出 `dist/tokens.json` 与 `dist/theme.css`；
     - 生成 `.cpk` 包（可通过 `chipsdev package` 或统一的 NPM 脚本封装），并保证包根保留 `manifest.yaml`、运行产物位于 `dist/`、压缩方式为 ZIP Store。
2. **主题契约与 token 完整性校验**
   - `npm run validate:theme`：
     - 在本地 Host 工作空间中安装生成的主题插件；
     - 调用 `chips theme validate`，确保 `theme.apply + theme.resolve` 不报错；
     - 使用 `theme.contract.get` 与 `resolve` 结果对比，确保声明覆盖的组件 token 集完整。
3. **颜色对比度与动效安全校验**
   - 可以在 `validate-theme.ts` 中集成基础规则：
     - 文本与背景对比度满足 WCAG 建议；
     - 动画时长与缓动曲线在白名单中；
   - 更严格的规则可以作为后续阶段的增强，不在当前基线强制实现。
4. **单元测试与集成测试**
   - 基础单元测试：
     - token 映射逻辑（`build-tokens`）；
     - 契约映射逻辑（从 tokens 到 `chips.comp.*`）；
   - 简单集成测试：
     - 在 headless 环境中调用 `chipsdev validate` + `chips plugin install` + `chips theme validate`，确认完整链路可用。

门禁脚本与测试文件本身只在生成工程中存在，脚手架仓库负责提供模板与约定，不直接维护具体主题。

## 6. 与其他脚手架的对齐要求

为保证薯片生态脚手架体系的一致性，主题脚手架在以下方面需与其他脚手架仓库对齐：

- 顶层目录结构：统一使用 `templates/需求文档/技术文档/开发计划`；
- 模板文件约定：统一使用 `.tpl` 后缀与一致的占位符语法；
- 生成工程约定：
  - 尽可能复用 `src/`、`tests/`、`config/` 等目录约定，仅在主题场景需要时新增 `tokens/`、`styles/`、`icons/` 等；
  - NPM 脚本命名优先与应用/卡片/模块脚手架保持一致（如 `dev/build/test/validate`）。

本技术文档作为主题包插件脚手架的架构与模板设计基线，后续实现与调整必须与本设计保持一致。如需重大变更，必须先更新本文件与相关需求/开发计划文档，再进行实现。
