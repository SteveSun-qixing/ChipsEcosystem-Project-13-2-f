## 1. 项目背景与目标

薯片生态中，界面视觉完全由 **主题插件（`type: theme`）** 决定。主题插件以主题包形式交付，承载颜色、字体、间距、动效、图标等全部视觉 Token 与 CSS 变量，实现“组件无头、主题注入”的前端架构：

- 组件库负责 L10 Component Headless，仅提供状态机、结构与 a11y，不含样式；
- L11 Theme Runtime 负责五层 token（`ref/sys/comp/motion/layout`）解析与作用域链覆盖；
- 主题包通过 `chips.comp.*` token 与 CSS 变量向上层注入视觉，实现主题切换与多样化前端风格。

为了让主题开发者以最低心智成本产出 **完全符合生态规范的主题插件**，需要提供一套官方的 **主题包插件脚手架**：

- 通过 `chipsdev create` 创建新主题插件项目时，默认使用本仓库提供的模板；
- 生成的主题插件工程天然符合主题系统设计、主题包目录结构、Manifest 规范、Token 与契约规范；
- 在不修改任何代码的前提下，即可通过 `chipsdev build/test/validate` 以及 `chips theme validate` 等基础门禁。

主题包插件脚手架自身是 **开发阶段工具，而非运行时代码**：

- 运行时加载的仍是由开发者基于脚手架生成的独立主题插件仓库打包出的 `.cpk` 主题包；
- 脚手架只负责：项目结构、配置文件、构建脚本、示例 token / CSS / 契约、测试骨架等，不参与 Host 运行时实现。

### 1.1 北极星目标

- 提供一套可持续 5–10 年演进的官方主题插件模板体系，覆盖主题系统的主线场景：
  - 标准主题（完整覆盖组件库与 Host UI 的默认主题）；
  - 派生主题（基于父主题仅覆盖差异 token 的主题系列）；
  - 生态官方主题（“薯片官方：默认/深色/高对比度”等）。
- 任何通过脚手架创建的主题插件工程，在遵循模板约定的前提下，都能：
  - 正确声明 `type: theme`、`themeId`（`publisher:name`）等 Manifest 字段；
  - 按五层 token 结构组织设计令牌，并通过 Theme Runtime 的解析与引用规则校验；
  - 通过 CSS 变量与 `chips.comp.*` token 正确对接组件库与 Host UI；
  - 内建基础的契约校验、颜色对比度与动效安全门禁脚本；
  - 拥有可运行的单元测试与最小集成测试，并通过基础测试门禁。
- 在不修改 Host / SDK / 组件库与全局契约的前提下，完成“脚手架 → 新主题插件工程 → Host 运行时”的端到端链路闭环。

### 1.2 本轮开发目标

本轮任务聚焦于 **从零建设 `Chips-Scaffold/chips-scaffold-theme` 仓库的文档与规划基线**，不编写任何实现代码，具体包括：

- 完成主题包插件脚手架的项目级需求文档：
  - 《01-主题包插件脚手架需求总览》（当前文档）；
- 完成主题包插件脚手架的技术文档骨架：
  - 总体架构、模板组织、token 构建链路、契约与门禁、命令行集成等章节（详见 `技术文档/`）；
- 完成分阶段的开发计划：
  - 阶段一：架构与需求冻结（当前阶段，仅文档与规划，不编写实现代码）；
  - 阶段二及后续：模板实现、CLI 集成、门禁脚本与测试体系搭建（详见 `开发计划/`）。

后续阶段在实现代码前，如发现生态层面的架构缺陷或能力缺口，必须先在《项目日志与笔记/问题工单.md》中登记工单，再继续推进。

### 1.3 成果边界与输出物

本项目最终目标输出（跨阶段）包括但不限于：

- 一组可被 `chips dev create` 消费的主题插件模板：
  - 标准主题模板（最小可用、覆盖官方组件集的完整主题）；
  - 可选的派生主题模板（基于父主题覆盖差异 token）；
- 与模板配套的工程与构建配置规划：
  - `package.json`、`tsconfig.json`、`chips.config.*` / `cpk.config.*` 等配置文件；
  - token 源文件（Style Dictionary 等）与五层 token 构建产物映射规范；
  - CSS / icons / fonts / preview 目录与文件命名约定；
- 完整的主题包插件脚手架需求文档、技术文档与开发计划；
- 与生态共用文档对齐的规范引用与补充说明（新增的跨生态正式口径会沉淀到 `生态共用技术文档`）。

不直接输出的内容：

- 不在本仓实现或修改 Host 内部实现（Kernel/Services/Runtime/渲染等）；
- 不在本仓修改 SDK 与组件库内部实现；
- 不输出市场宣传文案或非技术向材料（仅提供技术基线）。

## 2. 范围与边界

### 2.1 In Scope（本项目范围内）

- 主题包插件脚手架项目自身的需求、架构设计与实现规划：
  - 主题插件工程的目录结构与多模板组织方案；
  - token 源文件（设计令牌）与 Theme Runtime 所需五层 token 结构的映射规则；
  - 主题 CSS、icons、fonts、preview 等资源目录与命名规范；
  - 脚手架与 `chips dev create` 命令的对接方式与参数约定；
  - 模板内建的主题契约校验、颜色与动效门禁脚本、测试骨架等。
- 与其他仓库的集成接口说明：
  - 如何在模板中消费 `Chips-SDK` 的主题能力（`client.theme.*`）用于自测；
  - 如何保证生成项目的 Manifest、权限声明与 Host `theme.*` / `plugin.*` 动作保持一致；
  - 如何依照《主题包开发指南》《主题接口规范》《Token 与主题对接标准》组织产物。
- 本仓自身的开发计划、质量门禁与发布流程说明。

### 2.2 Out of Scope（本项目不做）

- 不在本项目中：
  - 修改或扩展插件类型系统（仍维持 `app/card/layout/module/theme` 五种类型）；
  - 更改 Theme Runtime、组件库或 Host 的既有架构设计与对外契约；
  - 引入未在生态技术选型中出现的第三方库（如新的打包器、样式引擎等）。
- 不承担以下职责：
  - 设计新的主题系统能力（例如额外的主题动作、主题存储后端）；
  - 修改 `生态共用技术文档` 中已冻结的协议、文件格式与 Bridge API 契约；
  - 管理具体主题包的设计视觉风格（颜色方案、品牌风格由主题设计者自行决定）。

### 2.3 信息归属边界

- 插件类型系统、Theme Runtime 五层架构、作用域链、主题公共接口（`theme.*`）等 **跨生态正式规范**，统一以 `生态共用技术文档` 为准，本项目文档只做引用与落地说明，不重复定义另一套口径。
- 本项目如需定义新的 **通用工程规范**（例如统一的主题插件工程目录结构、token 构建产物格式），需要：
  1. 先在 `生态共用技术文档/主题系统/` 中补充「主题包工程与脚手架规范」等章节；
  2. 再在本项目文档中引用对应章节，并补充仅对本脚手架有效的实现细节。
- 本仓的开发计划、阶段任务拆解、模板内部实现细节仅保存在 `Chips-Scaffold/chips-scaffold-theme` 目录，不进入生态共用文档。

## 3. 约束条件与质量要求

### 3.1 架构与规范约束

- **遵守十二层架构与职责边界**  
  - 脚手架不引入新的运行时层级，仅产出符合 L11 Theme Runtime 与 L10 Component Headless 期望的主题包；
  - 模板中的示例代码调用 Host 能力时，只能通过 SDK 或 Bridge API，禁止越层访问内部模块。

- **遵守主题系统与 token 规范**  
  - 模板必须以五层 token（`ref/sys/comp/motion/layout`）为落地目标，确保与 Theme Runtime 类型定义对齐；
  - token 命名与组件对接遵循 `chips.comp.<component-name>.*` 约定，并可追溯到 `sys` 层语义 token。

- **契约优先与门禁优先**  
  - Manifest 字段、权限声明、主题动作、错误码模型必须符合《Manifest配置规范》《主题接口规范》《插件契约规范》《Bridge-API规范》；
  - 模板应预置面向主题的基本门禁（token 完整性校验、契约校验、对比度/动效安全检查）的脚本入口。

### 3.2 工程与质量约束

- 禁止临时版、简化版、占位实现：
  - 模板内不得出现“TODO/占位组件/空实现函数”等未完成代码；
  - 生成的主题插件工程必须是可直接 `build + test + validate` 的完整工程。
- 禁止越界修改与未审批依赖：
  - 不得修改本任务范围外的仓库目录与文件；
  - 不得引入未在生态技术选型中列出的第三方库，如确有需要必须先提工单并经确认后再实施。
- 测试与门禁要求：
  - 脚手架自身需要具备模板一致性与最小 E2E 验证测试；
- 由脚手架生成的工程在默认配置下必须通过 `npm test` / `chipsdev test`、`chipsdev validate` 以及 `chips theme validate` 等基础门禁。

### 3.3 问题工单与阻断策略

如在脚手架设计或后续实现过程中发现：

- 生态层面的架构缺陷或能力缺口（例如 Theme Runtime 未暴露必要解析接口、组件库缺失关键 `chips.comp.*` 契约等）；
- 现有主题规范之间存在冲突且影响脚手架可实现性（例如 token 层级命名冲突、文件结构与运行时期望不一致）；
- `chips dev` 工具链尚未提供主题脚手架所需的模板扩展点；

则必须：

1. 在《项目日志与笔记/问题工单.md》中登记新工单，说明问题与期望行为；
2. 在 `项目日志与笔记/任务清单/` 下创建对应任务文件夹与说明文档（例如 `THEME-SCAFFOLD-XXXX/任务说明.md`）；
3. 暂停当前脚手架实现工作，等待用户处理与决策后再继续推进，绝对禁止以临时代码或降级实现绕过问题。

## 4. 干系人与协作边界

| 干系人 | 角色 | 关注点 |
|---|---|---|
| 主题插件开发者 | 使用脚手架创建主题包的第一方/第三方开发者 | 创建命令简单可靠、项目结构清晰、默认即具备 token 体系与契约校验、易于扩展 |
| Chips Dev 工具链 | 提供 `chips dev create/build/test/validate` 的命令行工具 | 脚手架导出的模板结构稳定、易集成、错误信息清晰可诊断 |
| Chips-Host 团队 | 维护 Host 内核、服务与 Theme Runtime | 脚手架生成的主题插件严格遵守 Host 对外接口基线与权限模型，不引入新的运行时代码路径 |
| Chips-SDK 团队 | 维护 SDK 封装与开发者工具 | 模板示例正确使用 `client.theme.*` 与其他能力，不绕过或重复实现 SDK 功能 |
| 组件库与主题团队 | 维护无头组件与主题系统 | 模板中正确对接组件库 token 契约与主题接口，不硬编码视觉样式、不破坏 token 体系 |
| 生态架构与规范维护者 | 维护生态共用文档与长线架构 | 脚手架文档与实现与全局规范口径一致，新增正式规范时能够同步回 `生态共用技术文档` |

## 5. 里程碑与阶段划分

主题包插件脚手架项目的开发阶段规划如下（详细见 `开发计划/00-开发计划索引.md`）：

1. **阶段一：架构与需求冻结**  
   - 阅读并对齐生态设计原稿与生态共用主题相关技术文档；  
   - 完成本仓需求文档、技术文档骨架与开发计划拆解；  
   - 不编写任何实现代码，确保方向正确且范围清晰。

2. **阶段二：基础主题模板骨架实现**  
   - 完成标准主题插件模板的目录结构与配置文件；  
   - 内建最小可用的 token 源文件、五层 token 构建脚本、CSS 与 icons 示例；  
   - 保证模板在本地可通过构建与基础测试。

3. **阶段三：模板扩展与生态对接**  
   - 设计并实现适用于不同主题类型的模板变体（标准主题、派生主题等）；  
   - 与 `chips dev create` 命令完成对接，支持按模板名或模式创建工程；  
   - 补充与 Host/SDK/组件库集成的契约测试与主题校验脚本。

4. **阶段四：测试体系与质量门禁**  
   - 建立脚手架自身的测试集（模板一致性测试、生成工程 E2E 验证等）；  
   - 定义并实现发布前质量门禁脚本；  
   - 确保后续模板演进不会破坏已有开发者项目的基本假设。

5. **阶段五：文档完善与发布验收**  
   - 完善主题包脚手架使用说明与示例文档；  
   - 对齐并在必要时补充 `生态共用技术文档` 中与主题脚手架相关的章节；  
   - 形成第一版可对外宣布的主题包插件脚手架发布版本。

上述阶段仅描述技术实施顺序，不涉及具体人力与时间排期。每个阶段结束前必须完成对应文档与测试的更新，方可进入下一阶段。
