# 测试策略与质量门禁设计（Chips-default）

> 文档状态：实现对齐稿  
> 适用范围：`ThemePack/Chips-default` 默认主题包的本地与 CI 验证

---

## 1. 目标与原则

- 为默认主题包提供**可重复、可自动化**的质量门禁。
- 覆盖 token 结构、关键组件契约、构建产物完整性。
- 不依赖临时实现或手工检查，所有检查均可通过脚本与测试用例执行。

---

## 2. 可用命令一览

在 `ThemePack/Chips-default` 目录下：

- `npm run build`
  - 执行 `build:tokens` 与 `build:css`；
  - 生成 `dist/tokens.json` 与 `dist/theme.css`。
- `npm run build:tokens`
  - `src/build-tokens.ts`；构建五层 token。
- `npm run build:css`
  - `src/build-css.ts`；拼接所有 CSS 文件。
- `npm run validate:theme`
  - `src/validate-theme.ts`；静态验证 token 结构与关键组件 token。
- `npm test`
  - 使用 Vitest 运行 `tests/*.spec.ts`，作为本主题包的基础测试集。

---

## 3. 构建与产物校验

### 3.1 Token 构建（build:tokens）

- 输入：
  - `tokens/ref.json`
  - `tokens/sys.json`
  - `tokens/motion.json`
  - `tokens/layout.json`
  - `tokens/comp/*.json`
- 输出：

```json
{
  "ref": { ... },
  "sys": { ... },
  "comp": { ... },
  "motion": { ... },
  "layout": { ... }
}
```

- 校验要点：
  - 所有 JSON 文件必须存在且可解析；
  - 合并后的 `comp` 层包含所有组件的 `chips.comp.*` token。

### 3.2 CSS 构建（build:css）

- 输入：
  - `styles/base.css`
  - `styles/components/*.css`
  - `styles/motions.css`
- 输出：`dist/theme.css`
- 校验要点：
  - 构建过程无错误或异常输出；
  - 最终 CSS 中不包含占位样式或 `TODO` 标记。

## 4. 主题验证脚本（validate:theme）

入口：`src/validate-theme.ts`，通过 `npm run validate:theme` 执行。

### 4.1 结构校验

步骤：

1. 读取 `dist/tokens.json` 并解析为 `ThemeTokenLayers`：

```ts
interface ThemeTokenLayers {
  ref: Record<string, unknown>;
  sys: Record<string, unknown>;
  comp: Record<string, unknown>;
  motion: Record<string, unknown>;
  layout: Record<string, unknown>;
}
```

2. 检查 `ref/sys/comp` 至少为对象，否则退出并输出错误信息。
3. 使用内部 `flattenLayer` 工具将五层展开为扁平键名：
   - 示例键名：
     - `chips.ref.color.gray-0`
     - `chips.sys.color.primary`
     - `chips.comp.button.root.surface.idle`

### 4.2 关键组件 token 校验

- 常量 `REQUIRED_COMPONENT_TOKENS` 定义了当前阶段必须存在的组件 token：

```ts
export const REQUIRED_COMPONENT_TOKENS: string[] = [
  // button
  "chips.comp.button.root.radius",
  "chips.comp.button.root.surface.idle",
  "chips.comp.button.root.surface.hover",
  "chips.comp.button.root.surface.active",
  "chips.comp.button.root.surface.disabled",
  "chips.comp.button.label.color.idle",
  "chips.comp.button.label.color.disabled",
  "chips.comp.button.focus.outline",
  // input
  "chips.comp.input.root.radius",
  "chips.comp.input.root.surface.idle",
  "chips.comp.input.root.surface.focus",
  "chips.comp.input.root.border.idle",
  "chips.comp.input.root.border.error",
  "chips.comp.input.value.color",
  "chips.comp.input.placeholder.color"
];
```

- 校验逻辑：
  - 对 `ref/sys/comp/motion/layout` 五层展开后的扁平变量表进行 `in` 检查；
  - 如有缺失，脚本输出缺失列表并设置 `process.exitCode = 1`；
  - 所有必需 token 存在时输出：
    - `Theme tokens structure is valid and required component tokens are present.`

该集合与组件库 `COMPONENT_TOKEN_MAP` 中 button/input 的需求保持一致，可视为当前阶段的最小契约覆盖集。

## 5. Vitest 用例设计

### 5.1 `tests/tokens.spec.ts`

- 目的：验证构建后的 token 文件结构。
- 行为：
  - 读取 `dist/tokens.json`；
  - 断言 `ref/sys/comp/motion/layout` 字段均存在且为对象。

### 5.2 `tests/contract.spec.ts`

- 目的：验证关键组件 token 契约。
- 行为：
  - 读取 `dist/tokens.json`；
  - 展开 `comp` 层为扁平键名；
  - 引用 `REQUIRED_COMPONENT_TOKENS`，逐个断言键存在。
- 说明：
  - 与 `validate:theme` 使用同一 token 列表，避免重复维护；
  - 后续如扩展验证范围，可通过扩充 `REQUIRED_COMPONENT_TOKENS` 实现。

### 5.3 `tests/build.spec.ts`

- 目的：验证构建流水线可在 CI 环境顺利执行。
- 行为：
  - 在测试中执行 `npm run build`；
  - 不对内容做细致断言，仅关注构建是否成功完成。

---

## 6. 建议的 CI 门禁

在集成到统一 CI 流水线时，建议按以下顺序执行：

1. `npm run build`
2. `npm run validate:theme`
3. `npm test`

所有步骤均通过才允许将默认主题包作为可发布版本：

- 若 `build` 失败：视为主题包结构或脚本错误；
- 若 `validate:theme` 失败：视为 token 契约不满足最小组件要求；
- 若 `npm test` 失败：视为基础保障用例不通过。

---

## 7. 后续扩展方向

- 当组件库侧对更多组件（如 `checkbox`、`radio`、`select` 等）增加更严格的契约要求时，可将对应 `chips.comp.*` token 加入 `REQUIRED_COMPONENT_TOKENS`；
- 如生态增加统一的主题包契约校验工具（基于 `contracts/theme-interface.contract.json`），本主题包可以在保持现有脚本的同时，将该工具接入 CI，作为更高层级的质量门禁。
