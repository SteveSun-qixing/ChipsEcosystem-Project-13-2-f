# 阶段三：CSS 与视觉表现实现

> 阶段目标：在五层 token 已对齐的基础上，为暗夜主题实现“简约优雅、干净无花哨”的 CSS 视觉表现，使常用组件在实际产品中具备统一、舒适的默认外观。

---

## 1. 阶段范围

- 基于阶段一确定的视觉风格与阶段二完成的 token 体系，在本主题包中实现：
  - 全局基础样式：背景、文本颜色、基础间距、滚动条等；
  - 常用交互组件（按钮、输入框、复选框、单选框、开关等）的视觉表现；
  - 布局与容器组件（对话框、Tabs、菜单、工具窗口等）的基础视觉；
  - 提示与反馈组件（通知、Toast、EmptyState、Skeleton等）的基础视觉。
- 所有 CSS 实现必须通过 token 与 CSS 变量（`--chips-*`）接入，不修改组件结构和语义。

---

## 2. 关键任务

### 2.1 全局基础样式

- 在 `styles/base.css` 中：
  - 定义全局字体、背景色、文案颜色等；
  - 使用 `chips.sys.*` token（经变量注入）作为颜色、间距、圆角等来源；
  - 保持样式简洁，不引入阴影、渐变等“花哨”效果（如确有必要只用极轻的阴影）。
- 确保 base 样式在不加载任何业务 CSS 时也能保证：
  - 文本对比度满足基础可读性要求；
  - 滚动条、选中状态等系统元素不会显得突兀。

### 2.2 交互组件视觉实现

- 在 `styles/components/` 下为核心交互组件补充样式（必要时可按组件命名划分文件）：
  - `button.css`：主按钮、次按钮、禁用态；
  - `input.css`：文本输入框、聚焦态；
  - 有需要时扩展：`checkbox.css`、`radio.css`、`switch.css` 等。
- 设计原则：
  - 所有颜色来自 token，不使用“magic number”颜色值；
  - 使用统一的圆角与间距；
  - 使用 `transition` 实现轻量 hover/focus 效果，不使用复杂 keyframes。

### 2.3 布局与容器组件视觉实现

- 对以下组件类型在 `styles/components/` 中实现基础视觉：
  - 对话框（dialog）：背景、边框、标题、按钮区域；
  - Tabs：标签列表、选中态下划线、内容区域；
  - 菜单（menu）：菜单项 hover/active 状态；
  - Tool-window/Dock-panel/Split-pane 等容器组件的边框与分隔线。
- 保证：
  - 这些容器在不同产品中看起来属于同一套设计系统；
  - 不使用渐变、花纹等“装饰性强”的视觉元素。

### 2.4 提示与反馈组件视觉实现

- 为 `notification`、`toast`、`empty-state`、`skeleton` 等组件提供基础视觉：
  - 提示类组件采用色块 + 文案 + 图标的简洁组合；
  - EmptyState 使用简洁轮廓与中性提示文案；
  - Skeleton 使用简单占位块，不做复杂动画。

### 2.5 与 token 的一致性检查

- 在 CSS 中大量使用 `var(--chips-...)` 时：
  - 对照 `dist/tokens.json` 与组件库的使用规范；
  - 避免临时引入与 token 不对应的新变量名。

---

## 3. 依赖与前置条件

- 阶段二已完成五层 token 对齐，并能够稳定生成 `dist/tokens.json`；
- 默认视觉风格与对比度要求已在需求与技术文档中明确；
- 如在 CSS 实现过程中发现 token 体系无法支撑某种必要视觉能力，应优先在文档中记录并评估是否需要扩充 token，而不是直接写死值。

---

## 4. 验收标准

- CSS 侧：
  - `styles/base.css` 与 `styles/components/*.css` 无语法错误，且不包含显式的“TODO”或被注释掉的占位逻辑；
  - 常用组件在 Demo 或 Host 集成环境中呈现出的视觉风格符合“简约、干净、优雅”的目标。
- 一致性侧：
  - CSS 中使用的所有设计值均来自 token 或派生变量，不出现未解释的硬编码颜色/尺寸；
  - 不擅自增加新的 `chips.*` token，而是使用已有 token 映射。
