# 暗夜主题包需求总览（Chips-theme-default-dark）

## 1. 项目背景与目标

### 1.1 北极星目标

交付一个可直接接入薯片生态正式主题系统的官方暗夜主题包，覆盖核心交互组件、卡片容器与查看器显示场景，在保持简约克制的前提下提供稳定、清晰、适合长时间使用的暗色体验。

## 2. 范围与边界

### 2.1 In Scope（本项目范围内）

- 基于正式主题包结构交付完整工程、契约、文档、测试与打包产物。
- 提供完整五层 Token：`ref / sys / comp / motion / layout`。
- 覆盖现有核心组件与卡片显示容器的暗夜视觉实现。
- 与 Host Theme Runtime、Card Viewer、统一卡片渲染链路保持一致。
- 保留 `data-scope / data-part / data-state` 契约，不修改组件结构。

### 2.2 Out of Scope（本项目不做）

- 不在本包内混合 light/dark 双模式切换。
- 不修改组件库交互逻辑、结构或运行时链路。
- 不引入生态技术选型之外的新第三方库。
- 不为兼容历史版本增加临时桥接或兜底实现。

## 3. 约束条件与质量要求

### 3.1 架构与规范约束

- 遵循《前端多样化和主题系统》《主题系统设计》《主题包开发指南》《主题接口规范》。
- 保持“一包一外观”的主题语义约束。
- 必须通过正式 Theme Runtime 下发 CSS 变量，不允许写死运行时主题来源。
- 若发现组件缺失、契约缺口或生态链路未打通，必须先提交问题工单再停止推进。

### 3.2 工程与质量约束

- 构建链路必须稳定产出 `dist/tokens.json` 与 `dist/theme.css`。
- 主题契约、最小功能集、构建脚本与测试必须全部可运行。
- 不留下 TODO、占位实现、简化版样式或临时主题兜底代码。
- 验证项至少覆盖：构建、契约、Token 结构、暗色视觉基线。

## 4. 干系人与协作边界

- ThemePack/主题工程维护方：负责主题 Token、CSS、文档与测试交付。
- Host Theme Runtime：负责主题安装、解析、应用与变量注入。
- Chips-CardViewer / card-service：负责消费统一主题快照，不再私有硬编码主题。
- 组件库：负责稳定暴露 headless 结构与样式契约挂点。

## 5. 里程碑与阶段划分

1. 冻结暗夜视觉基线与语义 Token。
2. 完成主题元信息与五层 Token 调整。
3. 完成 CSS 暗色化和组件层次优化。
4. 跑通构建、契约、测试与打包。
5. 同步更新 `.cpk-stage`、文档与交付产物。
