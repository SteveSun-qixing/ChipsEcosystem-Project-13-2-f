# 阶段四开发计划：卡片上传处理流水线

**阶段编号**：Phase 4  
**前置阶段**：Phase 3（房间与空间管理完成）  
**预计工时**：4-5 天（核心阶段，复杂度最高）  
**目标**：完整的卡片上传 → 资源 CDN 化 → HTML 渲染 → 入库流程，上传卡片后可通过 URL 访问其 HTML 版本。

---

## 交付标准

- 用户上传有效的 `.card` 文件后，服务端完成全部流水线步骤
- 卡片状态从 `pending` → `processing` → `ready` / `error` 可查询
- 卡片内嵌资源（图片等）上传到 MinIO，配置文件中的路径被替换为公开 CDN URL
- 通过 `GET /cards/{cardId}` 可获取 HTML 访问地址
- 前端可上传卡片并实时查看处理进度
- 流水线任何步骤失败，不留临时文件，错误信息清晰记录

## 任务清单

### 1. 卡片解包模块（`src/pipeline/card-unpack.ts`）

#### 1.1 安全解包

- [ ] 输入：`.card` 文件路径
- [ ] 校验文件魔数（ZIP: `50 4B 03 04`）
- [ ] 解压到 `/tmp/cps-pipeline-{uuid}/`
- [ ] **路径安全**：对每个 ZIP entry，检查 `path.resolve(tempDir, entry.path).startsWith(tempDir)`，拒绝路径穿越
- [ ] 解压完成后校验 `.card/` 目录存在

#### 1.2 元数据解析

- [ ] 读取 `.card/metadata.yaml`，用 `js-yaml` 解析
- [ ] 读取 `.card/structure.yaml`，解析结构
- [ ] 读取所有 `content/*.yaml`，建立 `Map<cardId, contentObject>`
- [ ] 签出 Zod 类型校验（`CardMetadata` schema、`CardStructure` schema）

#### 1.3 资源枚举

- [ ] 扫描解压目录根级文件（非 `.card/`、非 `content/` 目录下的所有文件）
- [ ] 返回资源文件列表：`{ relativePath: string, absolutePath: string, size: number, mimeType: string }[]`
- [ ] MIME type 通过文件扩展名映射（不依赖文件内容检测，保持简单）

#### 1.4 TypeScript 类型定义（`src/types/card.ts`）

```typescript
interface CardMetadata {
  chip_standards_version: string;
  id: string;           // 10位62进制
  name: string;
  created_at: string;   // ISO8601
  updated_at: string;
  theme?: string;
  tags?: string[][];
  cover_ratio?: string;
  visibility?: string;
}

interface CardStructureEntry {
  id: string;
  type: string;
}

interface CardStructure {
  structure: CardStructureEntry[];
  manifest?: {
    card_count: number;
    files?: Array<{ path: string; size: number; type: string }>;
  };
}

interface CardUnpackResult {
  tempDir: string;
  metadata: CardMetadata;
  structure: CardStructure;
  resourceFiles: ResourceFile[];
  contentMap: Map<string, Record<string, unknown>>;
}
```

### 2. CDN 上传模块（`src/pipeline/cdn-upload.ts`）

- [ ] 输入：`ResourceFile[]` 列表 + userId + cardId
- [ ] 对象 key：`chips-card-resources/{userId}/{cardId}/{relativePath}`
- [ ] 使用 `@aws-sdk/client-s3` PutObjectCommand 并发上传（`Promise.all` 限并发 5 个）
- [ ] 返回映射表：`Map<relativePath, cdnUrl>`
  - cdnUrl 格式：`{BASE_URL}/cdn/chips-card-resources/{userId}/{cardId}/{relativePath}`
- [ ] 上传失败抛出 `PipelineError`，记录哪个文件失败

---

### 3. URL 替换模块（`src/pipeline/url-replace.ts`）

- [ ] 输入：`contentMap`（各 YAML 内容对象） + `cdnUrlMap`（文件名 → CDN URL）
- [ ] 策略：对 content YAML 中所有字符串值，检查是否匹配 cdnUrlMap 中的任何文件名
  - 匹配规则：字符串值 === `relativePath`（文件名全匹配）
  - 匹配到则替换为对应 CDN URL
- [ ] 对象递归遍历（支持嵌套结构）
- [ ] 返回替换后的 `contentMap`
- [ ] 同时替换 `cover.html` 内的资源路径引用（若存在 `.card/cardcover/` 中的资源）

### 4. HTML 渲染模块（`src/pipeline/html-render.ts`）

**当前阶段采用服务端模板渲染方案（JSDOM + 模板）**，原因：
- 无 Electron Host 运行时
- Puppeteer 需要额外 Chromium，镜像体积爆炸
- 视觉一致性可在后续版本迭代

#### 4.1 渲染器接口（`src/pipeline/render/renderer.interface.ts`）

```typescript
interface CardRenderer {
  readonly cardType: string;
  render(contentConfig: Record<string, unknown>, cdnUrlMap: Map<string, string>): string;
}
```

#### 4.2 渲染器注册表（`src/pipeline/render/renderer-registry.ts`）

- [ ] 维护 `Map<cardType, CardRenderer>`
- [ ] 注册全部内置渲染器
- [ ] `getRenderer(type)` → 返回对应渲染器，未知类型返回 `FallbackRenderer`

#### 4.3 各类型内置渲染器

每种渲染器输出独立的 HTML 片段，供组合进 index.html：

- [ ] **MarkdownCardRenderer**：解析 `content` 字段（Markdown 文本），使用 `marked` 库（MIT）转为 HTML，含基础 CSS 样式
- [ ] **ImageCardRenderer**：读取图片路径（已替换为 CDN URL），生成 `<figure><img>` 结构，支持 `title` 字段
- [ ] **VideoCardRenderer**：生成 `<video controls>` 标签，支持字幕 `<track>` 标签
- [ ] **TextCardRenderer**：纯文本 / 富文本（从 content YAML 读取 HTML 或 plain text）
- [ ] **CodeCardRenderer**：代码块，带语言类名，内联 highlight.js 样式（MIT）
- [ ] **FallbackRenderer**：未知类型展示灰色占位块，注明类型名称

> **注意**：`marked` 和 `highlight.js` 需确认是否可加入技术选型，两者均为 MIT 协议。需用户确认后再正式引入。

#### 4.4 index.html 组合输出（`src/pipeline/html-render.ts`）

- [ ] 读取 structure.yaml，按顺序处理每个基础卡片
- [ ] 对每个基础卡片，取 contentMap 中对应的配置，调用对应渲染器生成 HTML 片段
- [ ] 组合为完整的自包含 HTML（内联基础 CSS，引用 CDN 资源）
- [ ] 模板结构：

```html
<!DOCTYPE html>
<html lang="zh-Hans">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{cardName}</title>
  <style>/* 基础样式，内联 */</style>
</head>
<body>
  <div class="chips-card">
    <div class="chips-card-header">
      <h1>{cardName}</h1>
    </div>
    <div class="chips-card-content">
      <!-- 各基础卡片 HTML 片段 -->
    </div>
    <div class="chips-card-footer">
      <span>由薯片社区托管</span>
    </div>
  </div>
</body>
</html>
```

### 5. 流水线编排（`src/pipeline/card-pipeline.ts`）

#### 5.1 主流程函数

```typescript
async function runCardPipeline(params: {
  cardFilePath: string;
  cardDbId: string;
  userId: string;
  db: DrizzleDB;
  storage: StorageService;
}): Promise<void>
```

#### 5.2 执行步骤

- [ ] 更新状态为 `processing`
- [ ] 步骤1：调用 `unpackCard(cardFilePath)` → `CardUnpackResult`
- [ ] 步骤2：调用 `uploadResourcesTocdn(resources, userId, cardDbId)` → `cdnUrlMap`
- [ ] 步骤3：调用 `replaceUrls(contentMap, cdnUrlMap)` → 替换后的 contentMap
- [ ] 步骤4：调用 `renderCardToHtml(metadata, structure, contentMap, cdnUrlMap)` → HTML string
- [ ] 步骤5：将 HTML 上传到 MinIO（`chips-card-html/{userId}/{cardDbId}/index.html`）→ `htmlUrl`
- [ ] 步骤6：更新数据库：
  - `html_url = htmlUrl`
  - `card_metadata = metadata (JSON)`
  - `card_structure = structure (JSON)`
  - `cover_url` = 若卡片有封面图（CDN URL），填入
  - `status = 'ready'`
- [ ] 步骤7（finally）：`rm -rf tempDir`

#### 5.3 错误处理

- [ ] 任何步骤抛出异常，捕获后：
  - 更新 `status = 'error'`，`error_message = error.message`
  - 确保 `rm -rf tempDir`（在 finally 中）
  - 不再向上抛出（流水线在后台异步执行，不影响上传接口响应）

#### 5.4 后台执行机制

- [ ] 使用 Node.js 原生异步（`setImmediate` 或无阻塞 Promise）
- [ ] 流水线在上传接口返回 202 后异步执行
- [ ] 不使用消息队列（本阶段单机部署，复杂度优先）
- [ ] 若后续需要扩展，此处为未来引入队列的接口边界

### 6. 服务端：上传与查询路由

#### 6.1 文件上传路由（`src/routes/upload.ts`）

- [ ] `POST /api/v1/upload/card`（需认证）
  - 使用 `@fastify/multipart` 处理文件流
  - 字段：`file`（.card 文件）+ `roomId?`（可选，所属房间 UUID）+ `visibility?`（`public`/`private`，默认 `public`）
  - 校验：文件类型（扩展名 `.card`），文件大小（≤ 500MB）
  - 写入临时文件
  - 创建 `cards` 数据库记录（`status = 'pending'`）
  - 触发异步流水线（`runCardPipeline`，不 await）
  - 返回 202：`{ data: { cardId, status: 'pending' } }`

#### 6.2 卡片查询路由（`src/routes/cards.ts`）

- [ ] `GET /api/v1/cards/:cardId`（公开，私有需认证）
  - 返回：id, title, htmlUrl, coverUrl, status, visibility, cardMetadata(摘要), createdAt, user 信息

- [ ] `GET /api/v1/cards/:cardId/status`（需认证，仅上传者）
  - 返回：`{ status, errorMessage, updatedAt }`
  - 用于前端轮询

- [ ] `PATCH /api/v1/cards/:cardId`（需认证）
  - 可更新：`roomId`、`visibility`
  - 不可更新：内容相关字段（内容已入库）

- [ ] `DELETE /api/v1/cards/:cardId`（需认证，仅上传者）
  - 级联删除 MinIO 中的相关资源（`chips-card-resources/{userId}/{cardId}/` 目录下所有对象 + HTML）
  - 删除数据库记录
  - 返回 204

- [ ] `GET /api/v1/users/me/cards`（需认证）
  - 分页返回当前用户所有卡片（含各状态）
  - 支持过滤：`?roomId=&status=&visibility=`

## 技术决策备忘

| 问题 | 决策 | 原因 |
|------|------|------|
| HTML 渲染方案 | JSDOM + 服务端模板 | 避免 Puppeteer 镜像体积，保持简单可控 |
| Markdown 渲染库 | `marked`（待用户确认） | MIT 协议，轻量 |
| 代码高亮库 | `highlight.js`（待用户确认） | MIT 协议，服务端可用 |
| 流水线执行 | 同进程异步，不用队列 | 单机部署复杂度优先 |
| 资源上传并发 | `Promise.all` 限并发 5 | 平衡速度与 MinIO 压力 |
| 临时文件清理 | always in `finally` | 保证不积累垃圾文件 |

> **待确认事项（需用户决策）**：
> 1. 是否可以引入 `marked`（MIT，Markdown 渲染）
> 2. 是否可以引入 `highlight.js`（MIT，代码高亮）

## 测试用例要求

| 用例 | 要求 |
|------|------|
| 上传有效 .card 文件 | 返回 202，cardId 存在，状态为 pending |
| 上传非 .card 文件 | 返回 400 |
| 上传超大文件 | 返回 413 |
| 流水线完整执行 | status 最终变为 ready，html_url 有值 |
| 流水线步骤失败（注入错误） | status 变为 error，error_message 有内容，无临时文件残留 |
| 路径穿越攻击（恶意 ZIP entry） | 拒绝解压，流水线报错 |
| 删除卡片 | DB 记录删除，MinIO 资源清理 |
