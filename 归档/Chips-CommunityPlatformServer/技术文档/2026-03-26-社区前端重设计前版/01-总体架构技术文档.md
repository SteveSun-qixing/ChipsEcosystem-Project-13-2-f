# 总体架构技术文档

## 1. 文档范围

本文描述社区服务器仓库当前已经落地的整体结构，只覆盖当前仓库真实存在的实现，不扩展未来规划。

对应目录：

- `packages/server`
- `packages/web`
- `packages/admin`
- `deploy`

## 2. 仓库结构

### 2.1 `packages/server`

职责：

- 提供 REST API
- 承担上传编排与权限控制
- 维护数据库、Redis、对象存储接入
- 调用 Host 运行时完成 `.card` 重打包与文件转换插件执行

### 2.2 `packages/web`

职责：

- 提供社区前台 Web 界面
- 承担登录、注册、空间浏览、房间浏览、上传、个人工作台
- 卡片查看时只负责跳转到转换后的 HTML 页面

### 2.3 `packages/admin`

职责：

- 提供运营后台前端
- 管理平台统计、用户状态与内容删除

### 2.4 `deploy`

职责：

- Docker 构建与编排
- Nginx 入口反向代理
- 运行环境变量样例

## 3. 运行时组件

当前项目是“模块化单体 + 多前端静态站点”的部署形态，不是拆分微服务集群。

### 3.1 后端依赖

- PostgreSQL：业务主数据
- Redis：登录失败计数、token 黑名单
- MinIO / S3：对象存储
- Host：作为唯一运行时承载卡片打包与正式转换插件执行

### 3.2 前端交付物

- Web 前台：静态站点
- Admin 后台：静态站点
- API：Fastify 进程
- CDN / 对象资源：经 Nginx 暴露 `/cdn/*`

## 4. 核心业务链路

### 4.1 卡片链路

1. 用户上传 `.card`
2. 服务端入库并返回 `pending`
3. 服务端解包卡片
4. 上传卡片资源到 CDN
5. 重写卡片内部资源 URL
6. 删除原本地资源文件并重打包为新 `.card`
7. 通过 Host 调用正式文件转换插件输出 HTML 目录
8. 上传 HTML 目录到 CDN
9. 回写 `htmlUrl`
10. 前台打开卡片时直接跳转到该 HTML 页面

### 4.2 箱子链路

1. 用户上传 `.box`
2. 服务端校验并解包
3. 读取 `metadata.yaml` 与 `structure.yaml`
4. 写入箱子记录
5. 箱子详情页展示结构与社区内卡片映射结果

### 4.3 空间 / 房间链路

1. 用户拥有公开资料与个人空间入口
2. 房间作为空间中的一级组织单元
3. 根目录表示 `roomId = null` 的内容
4. 空间页与房间页都按当前请求者的可见性返回内容与计数

## 5. 分层职责

### 5.1 `routes/`

- 负责 HTTP 接入
- 解析参数与调用 service
- 组装最终响应 DTO

### 5.2 `services/`

- 负责业务规则
- 承担权限判断、所有权校验、内容可见性过滤
- Host 集成层也归于 service

### 5.3 `pipeline/`

- 只负责卡片上传后的处理编排
- 不承载房间、用户、后台治理逻辑

### 5.4 `storage/`

- 封装对象存储桶、上传、删除、URL 构造、bucket 初始化

### 5.5 `db/`

- 维护 schema、迁移、连接池与管理员引导初始化

### 5.6 `plugins/`

- 维护 Fastify 鉴权与错误输出扩展

## 6. 架构边界

当前已经明确收口的边界：

1. Host 是唯一运行时承载
2. 社区服务器不自研第二条 HTML 转换正式链路
3. 卡片最终呈现效果由正式转换插件决定
4. 前台不再在卡片 HTML 外层套一层查看器 UI
5. 房间是第一版内容组织单元，但不是复杂布局容器

## 7. 当前不包含的能力

当前仓库尚未实现：

- 评论 / 点赞 / 收藏 / 关注
- 房间成员与协作权限
- 箱子在线布局运行时
- 社区卡片回下载为原生 `.card`
- 实时通知与消息系统

因此当前第一版的整体结构是：

- 社区平台后端负责“社区能力 + 上传编排 + 权限治理”
- Host 负责“正式运行时与插件能力”
- Web / Admin 负责前台与后台入口
