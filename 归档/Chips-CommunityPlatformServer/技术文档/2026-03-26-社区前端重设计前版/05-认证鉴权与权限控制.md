## 2. 认证模型

### 2.1 Token 组成

当前服务端采用“双 token + Redis 辅助状态”模型：

1. `access token`
   - 载体：`Authorization: Bearer <token>`
   - 用途：API 访问鉴权
   - 过期时间：`JWT_ACCESS_EXPIRES_IN`
2. `refresh token`
   - 默认载体：HttpOnly Cookie `chips_refresh_token`
   - Cookie path：`/api/v1/auth`
   - 过期时间：`JWT_REFRESH_EXPIRES_IN`
3. `jti`
   - access / refresh 各自独立生成 UUID
   - 用于 Redis 黑名单吊销

### 2.2 Token 负载

JWT 负载字段固定为：

- `userId`
- `role`
- `jti`

当前没有设备 ID、会话 ID、租户 ID 等额外字段。

### 2.3 登录与注册返回

`POST /api/v1/auth/register` 与 `POST /api/v1/auth/login` 当前行为一致：

- 在 JSON 响应中返回 `user` 与 `accessToken`
- 通过 Set-Cookie 写入 refresh token
- 不在 JSON 响应体中返回 refresh token 明文

因此，第一方 Web / Admin 前端采用的是“内存 access token + Cookie refresh token”模式。

## 3. Redis 参与点

### 3.1 登录失败计数

- key：`login:fail:{usernameLower}`
- 阈值：5 次
- 锁定时长：15 分钟

流程：

1. 登录前先检查 `AuthService.isAccountLocked(username)`
2. 用户不存在或密码错误时执行 `recordLoginFailure(...)`
3. 登录成功后执行 `clearLoginFailure(...)`

### 3.2 Token 黑名单

- key：`token:blacklist:{jti}`
- value：固定 `'1'`
- TTL：由 token 剩余有效期换算得到

使用场景：

1. refresh token 轮换时吊销旧 refresh token
2. logout 时吊销当前 refresh token
3. 访问受保护接口时校验 access token 的 `jti` 是否已被拉黑

## 4. Fastify 鉴权扩展

`authenticate.plugin.ts` 向 Fastify 注册了三个 `preHandler`：

### 4.1 `authenticate`

- 读取 `Authorization` 头
- 必须存在合法 Bearer token
- token 无效、缺失或已吊销时返回 401
- 成功后把 payload 挂到 `request.user`

适用接口：

- 当前用户资料接口
- 上传接口
- 内容编辑 / 删除接口
- 房间创建 / 更新 / 删除接口
- 卡片状态查询接口

### 4.2 `optionalAuthenticate`

- 若请求携带合法 token，则挂载 `request.user`
- 若没有 token 或 token 无效，则把 `request.user` 视为 `null`
- 不直接报错

适用接口：

- 用户空间
- 房间详情
- 房间内容列表
- 卡片详情
- 箱子详情
- 用户公开资料与公开内容列表

### 4.3 `requireAdmin`

- 先执行与 `authenticate` 相同的 token 解码
- 再校验 `payload.role === 'admin'`
- 非管理员返回 403 `AUTH_INSUFFICIENT_PERMISSION`

适用接口：

- `/admin/api/v1/*`

## 5. 角色模型

当前只有两种持久化角色：

1. `user`
2. `admin`

同时存在一个运行时访问态：

3. `guest`（未登录，仅是请求状态，不写入数据库）

### 5.1 `user`

- 可以管理自己的资料、房间、卡片、箱子
- 可以读取自己的私有内容
- 不具备后台治理能力

### 5.2 `admin`

- 登录链路与普通用户一致
- 通过 `/admin/api/v1/*` 获取平台统计、用户列表、内容列表
- 可以禁用用户、删除任意卡片 / 箱子
- 当前没有单独的“管理员查看私有前台页面”产品规则

### 5.3 `guest`

- 只能读取公开资源
- 不能调用任何写接口

## 6. 资源可见性规则

### 6.1 用户资料

- `GET /api/v1/users/:username`：公开资料，对所有请求者开放
- `GET /api/v1/users/me`、`PATCH /api/v1/users/me`、`PUT /api/v1/users/me/password`、`POST /api/v1/users/me/avatar`：仅本人

### 6.2 房间

房间可见性字段：

- `public`
- `private`

规则：

1. 房间所有者始终可见
2. `public` 房间对访客和其他已登录用户可见
3. `private` 房间对非所有者返回 404，而不是 403

当前房间访问入口：

- `GET /api/v1/users/:username/rooms`
- `GET /api/v1/rooms/:roomId`
- `GET /api/v1/users/:username/space`
- `GET /api/v1/rooms/:roomId/contents`

### 6.3 卡片

卡片可见性字段：

- `public`
- `private`

规则分两层：

1. **单卡读取**：`CardService.getAccessible(cardId, requesterId)`
   - 私有卡片对非所有者返回 404
2. **列表读取**：依据不同列表类型追加过滤
   - 房间内容列表：非房间所有者只能看到 `public + ready` 卡片
   - 根目录列表：非所有者只能看到 `public + ready` 卡片
   - 用户卡片列表：非所有者只能看到 `public` 卡片；当前实现不额外强制 `ready`
   - 发现 / 搜索：只返回 `public + ready` 卡片

### 6.4 箱子

箱子可见性字段：

- `public`
- `private`

规则：

1. 单箱子读取：私有箱子对非所有者返回 404
2. 房间内容列表：非房间所有者只能看到 `public` 箱子
3. 根目录列表：非所有者只能看到 `public` 箱子
4. 用户箱子列表：非所有者只能看到 `public` 箱子
5. 发现 / 搜索：只返回 `public` 箱子

## 7. 房间内内容权限细节

### 7.1 房间权限与内容权限分离

`RoomService.getAccessible(...)` 只判断“房间本身是否可访问”，不直接决定房间内每一项内容是否可见。

因此 `GET /api/v1/rooms/:roomId/contents` 当前会：

1. 先校验房间可访问
2. 再把 `request.user?.userId` 与 `room.userId` 传给：
   - `CardService.listByRoom(room.id, requesterId, room.userId, qs)`
   - `BoxService.listByRoom(room.id, requesterId, room.userId, qs)`

正式口径已经修复为：

- 只有房间所有者可以在房间内容列表中看到私有卡片 / 私有箱子
- “已登录但不是所有者”不会放宽内容可见性

### 7.2 房间计数的可见性口径

`RoomService.getContentCounts(roomId, requesterId, ownerUserId)` 已按请求者身份分支：

- 所有者：返回完整计数
- 非所有者：
  - 卡片只统计 `public + ready`
  - 箱子只统计 `public`

这保证了：

1. 公开空间页不会泄露私有内容数量
2. 公开房间详情页的计数与当前请求者实际可见内容一致

## 8. 所有权校验

### 8.1 房间所有权

以下操作必须先通过 `RoomService.assertOwnedByUser(roomId, userId)`：

- 上传内容到房间
- 修改房间
- 删除房间
- 上传房间封面
- 把卡片 / 箱子移动到房间

失败时返回 403 `ROOM_FORBIDDEN`。

### 8.2 卡片 / 箱子所有权

卡片与箱子的更新 / 删除都采用“只认所有者”的方式：

- 找不到资源 → 404
- 找到但不属于当前用户 → 仍返回 404

当前未给普通用户暴露 403 的“越权编辑”区别化口径。

## 9. 管理后台权限

### 9.1 统一入口

`routes/admin.ts` 在路由级统一挂载：

- `fastify.addHook('preHandler', fastify.requireAdmin)`

因此后台所有接口都依赖管理员角色。

### 9.2 后台能力边界

当前后台可以：

- 查看平台统计
- 分页查看用户列表
- 禁用 / 启用用户
- 分页查看全部卡片 / 箱子
- 强制删除任意卡片 / 箱子

当前后台还不能：

- 代登录用户
- 查看细粒度审计日志
- 修改房间内容可见性策略
- 绕过前台接口读取私有用户空间页面

## 10. 错误口径

### 10.1 401

典型场景：

- 缺失 access token
- token 无效
- token 已吊销
- refresh token 缺失或无效
- 账号被禁用后重新登录

### 10.2 403

当前主要用于：

- 非管理员访问后台
- 非房间所有者把内容挂入他人房间

### 10.3 404 伪装不可见

以下资源对无权访问者优先返回 404，用于减少资源存在性泄露：

- 私有房间
- 私有卡片
- 私有箱子
- 不属于当前用户的卡片删除 / 更新
- 不属于当前用户的箱子删除 / 更新

## 11. 当前未覆盖能力

当前权限系统还没有：

- 成员制房间
- ACL 明细表
- 协作者角色
- 分享链接签名
- MFA
- 设备级会话管理
- 管理员细分权限域

因此，当前正式口径仍然是“单所有者 + 公开/私有二元可见性 + 管理员后台治理”。
