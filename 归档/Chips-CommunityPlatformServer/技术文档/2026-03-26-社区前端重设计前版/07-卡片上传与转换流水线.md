# 卡片上传与转换流水线

## 1. 文档范围

本文对应当前已经落地的 `.card` 上传、资源 CDN 化、重打包、插件转换与 HTML 发布链路，覆盖：

- `packages/server/src/routes/upload.ts`
- `packages/server/src/pipeline/card-pipeline.ts`
- `packages/server/src/pipeline/card-unpack.ts`
- `packages/server/src/pipeline/cdn-upload.ts`
- `packages/server/src/pipeline/url-replace.ts`
- `packages/server/src/services/host-integration.ts`
- `packages/server/src/storage/s3.ts`

## 2. 入口与异步模型

### 2.1 上传入口

卡片上传入口为：`POST /api/v1/upload/card`

路由层职责：

1. 仅接受 `.card` 文件
2. 校验文件大小不超过 `MAX_CARD_SIZE_MB`
3. 解析表单字段：
   - `roomId?`
   - `visibility?`
4. 若指定 `roomId`，先校验该房间归当前用户所有
5. 把上传内容落到临时文件
6. 先创建数据库卡片记录，初始状态为 `pending`
7. 通过 `setImmediate(...)` 异步触发正式流水线
8. 立即返回 `202 Accepted`

### 2.2 数据库初始状态

卡片记录在上传受理后立即插入：

- `title = 处理中…`
- `status = pending`
- `visibility = public | private`
- `roomId = 目标房间或 null`
- `fileSizeBytes = 原始上传大小`

这使前端可以立刻拿到 `cardId`，再通过状态接口轮询处理结果。

## 3. 正式流水线顺序

`runCardPipeline(...)` 当前顺序已经固定为：

1. 解包 `.card`
2. 上传资源文件到 CDN
3. 重写 `content/*.yaml` 与 `cover.html` 内的资源 URL
4. 删除本地资源文件
5. 将重写后的目录重新打包为新的 `.card`
6. 删除原始上传包
7. 通过 Host 已安装的正式文件转换插件把新 `.card` 转为 HTML 目录
8. 上传 HTML 目录到 CDN
9. 回写数据库为 `ready`
10. finally 清理全部临时文件

## 4. 解包阶段

`unpackCard(...)` 的关键职责：

1. 校验 ZIP 魔数，拒绝伪造文件
2. 解压到系统临时目录
3. 过滤 `__MACOSX`、`.DS_Store`
4. 防止 ZIP 路径穿越
5. 自动处理“顶层包装目录”
6. 验证 `.card/` 目录存在
7. 读取：
   - `.card/metadata.yaml`
   - `.card/structure.yaml`
   - `content/*.yaml`
   - `.card/cover.html`（若存在）
8. 递归枚举资源文件，排除：
   - `.card/`
   - `content/`

### 4.1 元数据校验口径

当前要求 `metadata.yaml` 至少可解析出：

- `id`（若缺失则尝试 `card_id`）
- `name`（若缺失则尝试 `title`）

缺失时直接视为损坏卡片，返回 `FILE_CORRUPT`。

## 5. 资源上传与 URL 替换

### 5.1 资源上传

`uploadResourcesToCdn(...)` 会把卡片根目录资源上传到对象存储桶 `chips-card-resources`，key 规则为：

- `{userId}/{cardDbId}/{relativePath}`

返回映射表：

- `相对路径 -> CDN URL`
- `文件名 -> CDN URL`

当前并发上限为 5。

### 5.2 URL 替换

资源上传完成后，流水线会调用：

- `replaceContentUrls(...)`
- `replaceCoverHtmlUrls(...)`

统一把卡片内部原本指向本地资源的引用替换为 CDN URL。

### 5.3 重写后卡片的持久化策略

重写完成后会：

1. 把替换后的 `content/*.yaml` 写回临时目录
2. 把替换后的 `cover.html` 写回 `.card/cover.html`
3. 删除原先本地资源文件

因此后续重新打包出来的是一个“资源已 CDN 化”的新 `.card`。

## 6. 重打包与原包清理

### 6.1 重打包

服务端不会自己拼 `.card` 压缩包，而是调用 Host 正式能力：

- `hostIntegration.packCard(tempDir, rewrittenCardPath)`
- 下游实际调用 Host 内核能力 `card.pack`

### 6.2 原始上传包删除

新 `.card` 生成后，流水线会立即删除最初上传的原始包：

- 原始本地资源文件已删
- 原始上传 `.card` 已删
- 后续只保留“资源 URL 已替换”的重打包 `.card` 作为转换输入

这与当前业务要求一致：先完成 CDN 化，再基于新卡片做正式 HTML 转换。

## 7. HTML 转换

### 7.1 唯一正式链路

当前仓库的唯一正式 HTML 转换链路是：

1. `hostIntegration.init()` 初始化独立 Host 工作区
2. 动态安装卡片插件、主题插件、模块插件
3. 调用 `hostIntegration.convertCardToHtml(...)`
4. 由 Host 内核通过 `module.invoke` 执行模块能力：
   - capability：`converter.file.convert`
   - method：`convert`

### 7.2 输出参数

当前 HTML 转换的固定选项为：

- `source.type = card`
- `target.type = html`
- `output.overwrite = true`
- `options.html.packageMode = directory`
- `options.html.includeAssets = true`
- `options.html.includeManifest = false`

也就是说，服务端当前要求转换插件输出目录态 HTML，而不是单文件 HTML。

### 7.3 已归档的旧本地渲染实现

仓库中原先残留的一套本地 HTML 拼装实现已于 2026-03-25 移入：

- `Chips-CommunityPlatformServer/归档/2026-03-25-遗留本地HTML渲染实现/`

它不再属于活动源码，不得重新接回正式链路。

## 8. HTML 发布

转换插件输出目录后，流水线会：

1. 遍历 HTML 输出目录中的全部文件
2. 上传到对象存储桶 `chips-card-html`
3. 上传 key：`{userId}/{cardDbId}/{relativePath}`
4. 记录 `index.html` 对应 URL 为最终 `htmlUrl`
5. 若卡片存在封面 HTML，也会额外上传 `cover.html`

## 9. 数据库回写

流水线成功结束后会回写：

- `cardFileId`
- `title`
- `htmlUrl`
- `coverUrl`
- `cardMetadata`
- `cardStructure`
- `status = ready`
- `errorMessage = null`
- `updatedAt`

### 9.1 封面 URL 选择

当前 `coverUrl` 的选择策略比较简单：

- 从已上传的 CDN 资源 URL 中，挑选第一个图片后缀资源作为封面

这属于当前实现细节，并不等价于未来长期冻结的生态公共契约。

## 10. 失败与清理

### 10.1 失败处理

任一步骤失败后：

1. 记录错误日志
2. 提取 `errorMessage`
3. 把数据库状态写为 `error`

### 10.2 finally 清理

无论成功失败，都会尝试清理：

- 解包临时目录
- 重打包后的新 `.card`
- HTML 输出目录
- 未删除的原始上传包

## 11. 前台查看行为

当前前台卡片详情页 `packages/web/src/pages/CardDetailPage.tsx` 的行为是：

1. 先调用 `GET /api/v1/cards/:cardId`
2. 如果卡片 `status === ready` 且存在 `htmlUrl`
3. 直接 `window.location.replace(card.htmlUrl)`

因此平台不会再对 HTML 套一层卡片查看 UI；用户最终看到的就是转换插件产出的 HTML 页面。
