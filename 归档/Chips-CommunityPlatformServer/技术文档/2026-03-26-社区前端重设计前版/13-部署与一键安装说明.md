# 部署与一键安装说明

## 1. 目标与边界

- 本文档描述 `Chips-CommunityPlatformServer` 当前仓库内已经落地的单机 Docker Compose 部署方案。
- 当前默认方案是：
  - HTTP 单入口；
  - Nginx 统一反向代理；
  - Web / Admin / Server 分容器构建；
  - PostgreSQL / Redis / MinIO 同机编排。
- 当前默认编排不内置 HTTPS 证书管理；若生产环境需要 TLS，建议在外层网关或反向代理层补齐。

## 2. 当前部署资产

### 2.1 文件清单

- 环境变量：`Chips-CommunityPlatformServer/deploy/.env`
- 示例环境变量：`Chips-CommunityPlatformServer/deploy/.env.example`
- Compose：`Chips-CommunityPlatformServer/deploy/docker-compose.yml`
- Server 镜像：`Chips-CommunityPlatformServer/deploy/Dockerfile.server`
- Web 镜像：`Chips-CommunityPlatformServer/deploy/Dockerfile.web`
- Admin 镜像：`Chips-CommunityPlatformServer/deploy/Dockerfile.admin`
- 主反向代理：`Chips-CommunityPlatformServer/deploy/nginx/nginx.conf`
- Web 静态站点配置：`Chips-CommunityPlatformServer/deploy/nginx/web.conf`
- Admin 静态站点配置：`Chips-CommunityPlatformServer/deploy/nginx/admin.conf`
- Docker 上下文裁剪：`.dockerignore`

### 2.2 容器组成

| 服务 | 容器名 | 作用 |
|---|---|---|
| `postgres` | `ccps_postgres` | 主数据库 |
| `redis` | `ccps_redis` | token 黑名单与登录失败计数 |
| `minio` | `ccps_minio` | S3 兼容对象存储 |
| `server` | `ccps_server` | Fastify API 服务 |
| `web` | `ccps_web` | 用户前台静态站点 |
| `admin` | `ccps_admin` | 管理后台静态站点 |
| `nginx` | `ccps_nginx` | 对外统一入口 |

## 3. 当前构建口径

### 3.1 Docker 构建上下文

- 三个镜像统一使用生态工作区根目录作为构建上下文；
- 原因是 `packages/server` 依赖多个生态内本地 `file:` 包，同时 `HostIntegration` 默认还会从生态目录读取：
  - `Chips-Host`
  - `Chips-BaseCardPlugin/*`
  - `ThemePack/Chips-default`
  - `Chips-ModulePlugin/Chips-HtmltoImage-Plugin`
  - `Chips-ModulePlugin/Chips-CardtoHTML-Plugin`
  - `Chips-ModulePlugin/Chips-FileConversion-Plugin`

### 3.2 为什么不能只用项目子目录做上下文

- 若只以 `Chips-CommunityPlatformServer/` 为 Docker build context：
  - `packages/server/package.json` 中的本地 `file:` 依赖无法解析；
  - 服务端默认插件来源路径在容器内也不存在；
  - 一键安装链路会在镜像构建或 Host 初始化阶段失败。

## 4. 静态发布链路

### 4.1 Web 前台

- 先在 Node 构建阶段执行 `pnpm --filter @ccps/web build`
- 再把 `packages/web/dist` 复制到 Nginx 容器
- `web.conf` 负责：
  - `/assets/*` 长缓存
  - 其他路由回退到 `/index.html`

### 4.2 Admin 后台

- 先在 Node 构建阶段执行 `pnpm --filter @ccps/admin build`
- 再把 `packages/admin/dist` 复制到 Nginx 容器
- `admin.conf` 同样负责 SPA 回退
- 主站 Nginx 通过 `location /admin/ { proxy_pass http://ccps_admin/; }` 去掉 `/admin/` 前缀后转发

## 5. 反向代理口径

主站 Nginx 当前承担四类入口：

- `/api/` → `ccps_server`
- `/cdn/` → `ccps_minio`
- `/admin/` → `ccps_admin`
- `/` → `ccps_web`

这意味着：

- 前台与后台都是静态站点；
- API 与静态资源统一走同一站点根域；
- 若 `S3_PUBLIC_URL` 未配置，服务端返回的对象 URL 会统一收口到 `/cdn/...`。

## 6. 一键启动步骤

### 6.1 准备配置

1. 打开 `Chips-CommunityPlatformServer/deploy/.env`
2. 如需替换默认 Host 插件来源，可同时调整：
   - `HOST_CARD_PLUGIN_PATHS`
   - `HOST_THEME_PLUGIN_PATHS`
   - `HOST_MODULE_PLUGIN_PATHS`
   - `HOST_ACTIVE_THEME_ID`
3. 至少确认以下字段：
   - `BASE_URL`
   - `JWT_SECRET`
   - `S3_ACCESS_KEY`
   - `S3_SECRET_KEY`
   - `ADMIN_USERNAME`
   - `ADMIN_PASSWORD`

### 6.2 启动

在 `Chips-CommunityPlatformServer/deploy/` 下执行：

```bash
docker compose up -d --build
```

### 6.3 首次验证

完成启动后依次验证：

1. `http://localhost/api/v1/health`
2. `http://localhost/`
3. `http://localhost/admin/`
4. `http://localhost:9001`

## 7. 当前正式验证记录

本轮对部署链路做过以下核对：

- `docker compose -f docker-compose.yml config` 已通过；
- 项目正式构建已通过：`npx pnpm@9 build`；
- Docker daemon 在当前 CLI 会话中不可连接，因此镜像真机构建仍需用户本机 Docker 终端继续复核。

## 8. 当前注意事项

### 8.1 pnpm 版本

- 项目要求 `pnpm >= 9`
- 若本机全局 `pnpm` 版本过低，可临时使用：

```bash
npm_config_cache=/tmp/ccps-npm-cache npx pnpm@9 build
```

### 8.2 Docker 环境

- 若 CLI 报错 `Cannot connect to the Docker daemon`，说明当前会话未接入 Docker daemon；
- 这属于本机环境问题，不是仓库内 Compose 文件语法问题。

### 8.3 Host 插件路径

- 默认 Host 集成仍依赖生态工作区中的插件与主题目录；
- 如果要把社区服务器单独抽离为独立发布仓，需要后续再把这些依赖改造成真正可独立分发的构件来源。
