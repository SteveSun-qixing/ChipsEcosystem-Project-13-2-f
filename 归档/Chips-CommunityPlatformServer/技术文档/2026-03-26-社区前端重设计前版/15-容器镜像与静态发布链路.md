# 容器镜像与静态发布链路

## 1. 文档定位

- 本文档只描述当前项目自己的容器构建与静态发布实现细节。
- 对外公共部署规范仍以生态共用文档为上位参考；本文件不扩写成生态级标准。

## 2. Server 镜像

### 2.1 文件

- `Chips-CommunityPlatformServer/deploy/Dockerfile.server`

### 2.2 设计目的

- 构建 `packages/server`
- 让容器内既能运行 API 服务，又能访问默认 Host 插件来源
- 保证 `packages/server/package.json` 中的本地 `file:` 依赖可解析

### 2.3 当前关键实现

- 构建阶段与运行阶段都复制：
  - `Chips-Host`
  - `Chips-BaseCardPlugin/image-BCP`
  - `Chips-BaseCardPlugin/richtext-BCP`
  - `ThemePack/Chips-default`
  - `Chips-ModulePlugin/Chips-HtmltoImage-Plugin`
  - `Chips-ModulePlugin/Chips-CardtoHTML-Plugin`
  - `Chips-ModulePlugin/Chips-FileConversion-Plugin`
- 这样既能满足安装依赖，也能满足 `HostIntegration` 默认插件路径解析。

## 3. Web 镜像

### 3.1 文件

- `Chips-CommunityPlatformServer/deploy/Dockerfile.web`
- `Chips-CommunityPlatformServer/deploy/nginx/web.conf`

### 3.2 发布流程

1. Node 构建阶段执行 `pnpm --filter @ccps/web build`
2. 将 `packages/web/dist` 复制到 Nginx 镜像
3. 通过 `web.conf` 实现 SPA fallback

### 3.3 当前路由规则

- `/assets/*`：静态资源直出
- 其他路径：统一回退到 `index.html`

这保证：

- `/`
- `/upload`
- `/dashboard`
- `/@username`
- `/@username/rooms/:roomSlug`

都能被前台容器正确接住。

## 4. Admin 镜像

### 4.1 文件

- `Chips-CommunityPlatformServer/deploy/Dockerfile.admin`
- `Chips-CommunityPlatformServer/deploy/nginx/admin.conf`

### 4.2 发布流程

1. Node 构建阶段执行 `pnpm --filter @ccps/admin build`
2. 将 `packages/admin/dist` 复制到 Nginx 镜像
3. 通过 `admin.conf` 实现后台 SPA fallback

### 4.3 与主站的关系

- Admin 构建产物本身使用 `base: '/admin/'`
- 主站 Nginx 会去掉 `/admin/` 前缀再转发到 Admin 容器
- 因此前台主站与后台子路径可以共用一个外部域名

## 5. 主站 Nginx

### 5.1 文件

- `Chips-CommunityPlatformServer/deploy/nginx/nginx.conf`

### 5.2 当前职责

- 统一对外暴露 80 端口
- 将 `/api/` 代理到 Fastify
- 将 `/cdn/` 代理到 MinIO
- 将 `/admin/` 代理到 Admin
- 将其余路径代理到 Web

### 5.3 当前不做的事

- 不在当前默认 Compose 内直接管理 HTTPS 证书
- 不在当前默认 Compose 内加入 ACME / certbot
- 不在当前默认 Compose 内内嵌二级网关或服务发现组件

## 6. `.dockerignore`

### 6.1 文件

- `.dockerignore`

### 6.2 当前裁剪目标

- 排除各类 `node_modules`
- 排除已有 `dist`
- 排除工作区临时输出和调试目录

### 6.3 目的

- 降低工作区根目录作为 Docker build context 时的无关传输量
- 避免把本地构建产物和调试工作区误带入镜像

## 7. 当前已知边界

- 当前镜像链路已经能和现有生态工作区协同构建；
- 但它仍依赖生态工作区中多个兄弟目录；
- 若未来要把社区服务器单独分发为独立发布仓，需要继续收口：
  - Host 运行时依赖打包方式
  - 默认插件工件分发方式
  - 主题与基础卡片插件的独立安装来源
