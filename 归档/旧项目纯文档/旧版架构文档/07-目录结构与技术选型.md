## 一、仓库策略：每个插件独立仓库

### 1.1 核心原则

> **每个插件、每个模块都是独立的 git 仓库。完全解耦，完全独立。**

薯片生态不使用传统 monorepo 将所有代码放在一个仓库。每个插件和共享包都是独立仓库，可以独立开发、构建、测试、发布。

### 1.2 开发超级工作区

`Project-12` 是一个**超级项目（Super Project）**，通过 git submodule 或独立克隆聚合多个仓库，方便联调开发。这不是运行时结构，只是开发时的便利工具：

## 三、技术选型

### 3.1 薯片主机（Chips-Host）

| 技术 | 选择 | 原因 |
|------|------|------|
| 框架 | Electron | 唯一的桌面容器框架（已确定） |
| 语言 | TypeScript | 类型安全，与整个生态一致 |
| 构建工具 | electron-vite 或 electron-forge | Electron 专用构建工具 |
| 主进程框架 | 无（原生 Node.js） | 主进程足够简单，不需要额外框架 |
| 包管理 | pnpm | monorepo 支持，高效的磁盘利用 |
| 测试 | vitest | 快速，与 Vite 生态一致 |
| 日志库 | pino | 高性能 JSON 日志 |
| ZIP 库 | jszip（默认）/ yazl+yauzl（高性能） | 卡片/箱子文件处理 |
| YAML 库 | js-yaml | 配置文件解析 |
| 数据校验 | zod | 运行时参数验证 |
| ID 生成 | nanoid（定制 62 进制字符集） | 10 位 62 进制 ID |

### 3.2 官方插件

| 技术 | 选择 | 原因 |
|------|------|------|
| UI 框架 | React | 官方产品统一使用（第三方插件可自由选择） |
| 状态管理 | Pinia | React 官方推荐 |
| 构建工具 | Vite | 快速 HMR，优秀的构建性能 |
| 组件库 | @chips/component-library | 生态自有组件库 |
| SDK | @chips/sdk | Bridge API 封装 |
| CSS 方案 | CSS 变量 + CSS Modules | 主题系统兼容 |

### 3.3 第三方插件可选技术

第三方插件开发者可以使用任何 Web 技术：
- React、React、Svelte、Solid、vanilla JS……
- 任何 CSS 框架或方案
- 任何状态管理方案
- 唯一要求：使用 `window.chips.*` Bridge API 或 `@chips/sdk` 与系统交互

## 四、依赖层级

```
                      ┌──────────────────────┐
                      │    Chips-Host (主机)   │
                      │    只依赖 Node.js 包    │
                      │    不依赖任何前端框架    │
                      └───────────┬──────────┘
                                  │
                         被预加载脚本桥接
                                  │
              ┌───────────────────┼───────────────────┐
              │                   │                   │
    ┌─────────▼─────────┐ ┌──────▼──────┐ ┌─────────▼──────────┐
    │ @chips/sdk (SDK)   │ │@chips/      │ │ 第三方库            │
    │ 纯前端 TypeScript  │ │components   │ │ (插件自选)           │
    │ 封装 Bridge API    │ │ React 组件库 │ │                    │
    └─────────┬─────────┘ └──────┬──────┘ └────────────────────┘
              │                   │
              └────────┬──────────┘
                       │
              ┌────────▼────────┐
              │   插件应用代码    │
              │ (Vue/React/...)  │
              └─────────────────┘
```

**关键规则**：
1. **Chips-Host 不依赖任何前端框架**（Vue、React 等）。主机是纯 Node.js + Electron 应用。
2. **@chips/sdk 不依赖 Vue**。SDK 核心是纯 TypeScript，Vue composables 是可选的导出。
3. **@chips/component-library 依赖 React**。组件库只服务于 Vue 插件开发者。
4. **插件不依赖 Chips-Host**。插件只依赖 `window.chips`（Bridge API）和可选的 SDK。

## 五、构建与开发流程

### 5.1 开发模式

```bash
# 1. 启动薯片主机（开发模式）
cd Chips-Host
pnpm dev
# → 启动 Electron，加载开发版主进程
# → 支持主进程热重载

# 2. 启动编辑器插件（独立开发模式）
cd plugins/apps/editor
pnpm dev
# → 启动 Vite 开发服务器
# → 输出 http://localhost:5173
# → 在浏览器中开发 UI（使用 Mock Bridge API）

# 3. 在主机中加载开发中的插件
# 主机配置中添加开发插件路径
# 主机加载 http://localhost:5173 而非 file:// 路径
```

### 5.2 构建流程

```bash
# 构建 SDK（其他包依赖它）
cd Chips-SDK && pnpm build

# 构建组件库（插件可选依赖）
cd Chips-ComponentLibrary && pnpm build

# 构建所有插件（并行）
cd plugins && pnpm -r build
# 每个插件构建为 index.html + assets/ 的 Web 应用

# 构建主机
cd Chips-Host && pnpm build
# 生成 Electron 可执行文件

# 打包分发
cd Chips-Host && pnpm package
# 生成安装包，包含主机 + 内置插件
```

### 5.3 插件构建输出

每个插件构建后输出标准的 Web 应用结构：

```
dist/
├── manifest.yaml          # 复制自源码

├── renderer/              # 渲染组件（仅卡片/布局插件）
│   └── index.html
├── editor/                # 编辑组件（仅卡片/布局插件）
│   └── index.html
└── assets/
    ├── main-[hash].js

    └── style-[hash].css
```

### 6.2 插件分发

独立的插件以 ZIP 包形式分发：

```
my-plugin.zip
├── manifest.yaml
├── index.html
└── assets/
    ├── main.js
    └── style.css
```

用户通过以下方式安装：
1. 从应用市场下载并自动安装
2. 拖拽 ZIP 文件到薯片主机窗口
3. 在设置面板中选择本地 ZIP 文件安装
4. 命令行安装：`chips install ./my-plugin.zip`（`chips` 是用户 CLI，未来开发）

## 七、与原架构的迁移路径

### 7.1 可保留的部分

| 原有部分 | 新架构中的对应 |
|---------|-------------|
| Chips-core 的路由逻辑设计 | 迁移到 Chips-Host/kernel/ （TypeScript 重写） |
| Chips-Foundation 的服务接口设计 | 迁移到 Chips-Host/services/ |
| Chips-SDK 的 API 设计 | 保留并增加 Bridge 客户端封装 |
| Chips-ComponentLibrary | 保持不变 |
| Chips-Editor 的 UI 代码 | 迁移到 plugins/apps/editor/（移除 Electron 相关） |
| Chips-Viewer 的 UI 代码 | 迁移到 plugins/apps/viewer/（移除 Electron 相关） |
| BasicCardPlugin 的渲染/编辑组件 | 迁移到 plugins/cards/（调整为 iframe 模式） |
| 生态共用中的规范文档 | 保留，根据新架构更新 |

### 7.2 需要废弃的部分

| 废弃部分 | 原因 |
|---------|------|
| Chips-core (Rust) | 内核改用 TypeScript 实现在主进程中 |
| Chips-EcosystemSettings (Flutter) | 改为 Web 插件应用 |
| 各应用的 electron/ 目录 | 统一由 Chips-Host 管理 |
| 各应用的独立 Electron 配置 | 只有 Chips-Host 是 Electron 应用 |

### 7.3 建议的迁移顺序

```
阶段一：搭建薯片主机骨架
  1. 创建 Chips-Host 项目
  2. 实现内核（路由器、注册表、事件总线、生命周期）
  3. 实现基本的窗口管理器和插件管理器
  4. 实现预加载脚本（Bridge API）
  5. 实现核心基础服务（文件、配置、日志）

阶段二：迁移第一个应用插件
  6. 将 Chips-Viewer 的前端代码迁移为查看器插件
  7. 实现卡片服务和 ZIP 服务
  8. 验证卡片文件的打开和渲染

阶段三：迁移卡片插件
  9. 将 BasicCardPlugin 迁移为 iframe 模式的卡片插件
  10. 实现卡片渲染流程（应用插件 + 卡片插件 iframe）

阶段四：迁移编辑器
  11. 将 Chips-Editor 的前端代码迁移为编辑引擎插件
  12. 实现编辑相关的服务

阶段五：迁移其他应用
  13. 设置面板（从 Flutter 迁移为 Web 插件）
  14. 资源管理器、文件连接器、标签编辑器
```

## 八、性能预期

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 主机启动时间 | < 2 秒 | 从双击到内核就绪 |
| 插件加载时间 | < 1 秒 | 从请求到窗口显示 |
| 内核路由延迟 | < 1ms | 主进程内服务间调用 |
| Bridge 调用延迟 | < 5ms | 渲染进程 → 主进程 → 渲染进程 |
| 主进程内存 | < 100MB | 不含渲染进程 |
| 单个插件内存 | 50-200MB | 取决于插件复杂度 |
| 同时打开插件数 | 10+ | 受系统内存限制 |
