# 薯片生态依赖管理指南

---

## 一、依赖管理工具

生态使用 **pnpm-workspace** 管理多包依赖。每个插件和模块都是独立的 git 仓库，开发时通过超级项目（Project-12）聚合，使用 pnpm workspace 统一管理联调依赖。

根目录的 `pnpm-workspace.yaml` 定义开发时的工作区包：

```yaml
packages:
  # 薯片主机（核心）
  - 'Chips-Host'

  # 共享包（独立仓库）
  - 'Chips-SDK'
  - 'chips-component-library'

  # 官方应用插件（每个独立仓库）
  - 'plugins/apps/*'

  # 官方卡片插件（每个独立仓库）
  - 'plugins/cards/*'

  # 官方布局插件（每个独立仓库）
  - 'plugins/layouts/*'

  # 官方模块插件（每个独立仓库）
  - 'plugins/modules/*'

  # 官方主题插件（每个独立仓库）
  - 'plugins/themes/*'
```

> **注意**：每个插件是独立的 git 仓库，可以独立开发和发布。pnpm workspace 仅用于开发时联调。

## 二、依赖层级

```
┌─────────────────────────────────────────┐
│         第六层：插件应用                  │
│  Editor / Viewer / Settings 等          │
│  （每个独立仓库，每个独立 BrowserWindow） │
└─────────────────────┬───────────────────┘
                      │ 通过 Bridge API 调用
┌─────────────────────▼───────────────────┐
│      第五层：插件桥接层（Bridge API）     │
│  window.chips.* 标准接口               │
└─────────────────────┬───────────────────┘
                      │
┌─────────────────────▼───────────────────┐
│       SDK + 组件库（可选依赖）           │
│  @chips/sdk / @chips/components        │
└─────────────────────┬───────────────────┘
                      │
  ──── 以上在渲染进程 ─────── 以下在主进程 ────
                      │
┌─────────────────────▼───────────────────┐
│    第四层：基础服务层 + 共享功能模块      │
│  14 类核心服务 + 独立模块插件            │
├─────────────────────┬───────────────────┤
│        第三层：薯片内核                  │
│  TypeScript，路由器+注册表+事件总线     │
├─────────────────────┬───────────────────┤
│       第二层：平台抽象层（PAL）          │
│  适配 Windows/macOS/Linux 差异         │
├─────────────────────┬───────────────────┤
│        第一层：Electron 宿主            │
│  Chromium + Node.js                    │
└─────────────────────────────────────────┘
```

**原则**：上层可依赖下层，下层不可依赖上层。插件只能通过 Bridge API 访问底层能力，不能直接 import 主进程中的模块。

## 三、常用命令

### 安装依赖

```bash
# 在超级项目根目录安装所有工作区依赖
pnpm install

# 单个仓库安装
cd Chips-SDK && pnpm install
```

### 添加依赖

```bash
# 添加生产依赖
pnpm add lodash

# 添加开发依赖
pnpm add -D vitest

# 添加工作区内依赖（使用 workspace 协议）
pnpm add @chips/sdk@workspace:*
```

### 构建顺序

必须按依赖顺序构建：

```bash
# 1. 薯片主机（内核 + 基础服务 + Bridge）
cd Chips-Host && pnpm build

# 2. SDK
cd Chips-SDK && pnpm build

# 3. 组件库
cd chips-component-library && pnpm build

# 4. 各类插件（可并行）
cd plugins/apps/editor && pnpm build
cd plugins/cards/image && pnpm build
# ...
```

## 四、依赖添加规范

### 主机（Chips-Host）可用的依赖

主机的主进程是纯 Node.js 环境，可以使用：
- Node.js 内置模块（fs、path、crypto 等）
- Electron API
- 纯 Node.js 包（jszip、js-yaml、pino、zod、nanoid 等）
- 可选的原生模块（sharp、better-sqlite3 等）

**禁止**在主进程中引入前端框架（Vue、React）或 DOM 操作库。

### 插件可用的依赖

插件运行在渲染进程（BrowserWindow）中，可以使用：
- 任何 Web 前端框架（Vue 3、React、Svelte 等）
- @chips/sdk（Bridge API 封装）
- @chips/components（无头组件库，可选）
- 任何纯前端的 npm 包

**禁止**在插件中使用 Node.js 模块（fs、path 等）或直接调用 Electron API。

### 版本管理

- 所有包使用语义化版本（SemVer）
- 工作区内依赖使用 `workspace:*` 协议
- 外部依赖锁定到具体版本（通过 pnpm-lock.yaml）
- 定期检查安全漏洞：`pnpm audit`

## 五、独立仓库发布

每个插件仓库独立发布为 `.cpk` 包：

```bash
# 使用 Chips CLI 构建和打包
chips build      # Vite 构建，输出到 dist/
chips pack       # 打包为 .cpk 文件
chips publish    # 发布到薯片应用市场
```

SDK 和组件库发布为 npm 包：

```bash
cd Chips-SDK && pnpm publish
cd chips-component-library && pnpm publish
```
