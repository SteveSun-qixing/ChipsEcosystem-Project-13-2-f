# 基础卡片插件的结构

每个基础卡片都是一个完整的插件包，包含这个基础卡片相关的所有内容：渲染组件（前端代码）、编辑组件、配置文件结构定义、信息清单等。除了播放器、文本编辑器等底层通用组件之外，基础卡片相关的所有代码都包含在它自己的插件库中。

一个基础卡片插件由几个核心部分组成：渲染组件、编辑组件和信息清单。

## 渲染组件（前端代码）⭐ 核心

渲染组件是插件的前端代码，定义了这种卡片如何在查看器中显示。**这些代码使用网页技术编写**（HTML、CSS、JavaScript），是卡片最终呈现给用户的界面和交互逻辑的完整实现。

以视频卡片为例，渲染组件包含：
- **HTML 结构**：定义视频播放器的界面布局（播放区域、控制条、字幕显示区域等）
- **CSS 接口点**：暴露样式类名和 CSS 变量，供主题包注入样式（不包含具体颜色，只定义布局）
- **JavaScript 交互逻辑**：实现播放控制、字幕切换、进度记忆、快捷键响应等完整交互功能

**渲染流程**：
1. 卡片渲染器（CardRenderer，在公共基础层）读取卡片配置文件（content/ID.yaml）
2. 提取配置数据（视频路径、封面、字幕列表、播放参数等）
3. 调用视频卡片插件的渲染组件（前端代码）
4. 将配置数据填充到前端模板中
5. 生成完整的 HTML 网页
6. 在 iframe 窗口中加载该网页
7. 用户看到视频播放器并可以交互

不同类型的基础卡片有不同的渲染组件：
- **图片卡片**：前端代码实现图片查看器界面（缩放、旋转、幻灯片播放等）
- **Markdown 卡片**：前端代码解析 Markdown 语法并渲染为 HTML，支持目录、公式、代码高亮
- **代码卡片**：前端代码实现代码显示器（语法高亮、行号、代码折叠、复制按钮等）

渲染组件可以调用公共基础层的通用模块。例如，视频卡片插件可以调用 VideoPlayer 模块获得底层的视频播放能力，在此基础上实现视频卡片特定的功能。插件不需要从零实现视频解码和播放，只需要组织界面和交互逻辑。

## 编辑组件

编辑组件负责在编辑引擎中编辑卡片内容，提供用户界面让用户设置卡片参数。视频卡片的编辑组件提供文件上传功能、封面选择、字幕管理、播放参数设置等。图片卡片的编辑组件提供图片上传、标题输入、适配模式选择等。编辑组件修改配置后，保存到配置文件，渲染组件读取这些配置进行显示。

每种基础卡片的编辑组件界面和功能都不同，由插件开发者自行设计。

信息清单记录卡片类型的元数据，包括类型名称、唯一标识符、版本号、作者信息、支持的文件格式、图标、描述、依赖关系等。内核通过信息清单了解插件的能力和需求。

插件的工作流程完全由内核协调。用户双击卡片文件时，操作系统将文件传递给内核，内核识别为 .card 文件并路由到 CardRenderer（公共基础层的卡片渲染引擎）。CardRenderer 解析卡片文件，读取 structure.yaml 发现包含视频卡片。CardRenderer 调用视频卡片的 BaseCardRenderer，后者加载视频卡片插件的渲染组件（前端代码），读取配置数据，将数据填充到前端模板，生成 HTML 网页，创建 iframe 窗口并加载该网页，用户看到视频播放器界面。

编辑卡片时流程类似。用户拖拽视频卡片到画布，操作被发送到内核，内核路由到卡片编辑模块。用户点击卡片，内核识别需要加载视频卡片编辑组件，加载后显示在编辑面板窗口中。用户上传视频、设置参数，编辑组件通过内核接口将配置保存到卡片文件中。

## 配置文件和数据分离

基础卡片的配置保存在复合卡片文件的 content/ 目录下，每个基础卡片一个独立的配置文件。文件名格式是"ID.yaml"，使用十位 62 进制 ID。配置文件的具体结构由插件定义，不同类型的卡片有不同的配置项。

**关键设计**：
- **卡片文件保存内容数据**（配置文件）：视频路径、封面、字幕、播放参数等
- **基础卡片插件包含前端代码**：HTML 结构、CSS 接口点、JavaScript 交互逻辑
- **渲染器连接两者**：读取配置 → 调用插件前端代码 → 填充数据 → 生成 iframe

这种分离确保了：
1. 卡片文件轻量（只保存数据，不包含代码）
2. 插件可以独立更新（改进界面和功能，不影响已有卡片）
3. 主题可以灵活应用（前端代码只定义结构，样式由主题注入）

插件支持热插拔。用户在应用市场安装新插件时，内核接收到安装请求，下载插件包，解析元数据，注册到模块注册表，插件立刻可用。卸载插件时，内核从注册表移除，清理相关文件。更新插件时，内核会检查版本兼容性，确保已有的卡片仍能正常工作。

系统自带一些默认的基础卡片插件，包括富文本、Markdown、图片、视频、音乐、代码块、列表、打分等。第三方开发者可以开发更多类型的插件，比如思维导图卡片、表格卡片、3D 模型卡片等，发布到应用市场供其他用户使用。

插件之间是完全独立的。安装、卸载、更新某个插件不会影响其他插件。如果某个插件出错，内核会隔离错误，不会导致整个系统崩溃。插件可以声明依赖其他插件，内核在安装时会自动处理依赖关系。

用户可以安装多个实现相同功能的插件。比如系统自带富文本卡片编辑器，用户可以再安装一个高级富文本编辑器插件。在设置中可以指定默认使用哪个，也可以在创建卡片时手动选择。

## 与组件库的关系

基础卡片插件与薯片组件库（@chips/components）是解耦的关系。

**重要说明**：基础卡片的渲染组件和编辑组件在**各自的插件包**中实现，**不在组件库中**。这是薯片生态的核心架构设计。

**组件库的角色**：
- 提供通用 UI 组件（Button、Input、Modal 等）
- 提供卡片辅助组件（CardWrapper、CardLoading 等）
- 这些是**可选**的开发工具，插件可以选择使用或不使用

**插件可以**：
1. 直接使用公共基础层的能力（VideoPlayer、MarkdownParser 等）
2. 可选使用组件库的通用组件
3. 可选使用组件库的卡片辅助组件
4. 完全自定义实现所有 UI

这种设计确保了：
- 插件的独立性和可替换性
- 组件库的通用性和轻量化
- 基础层能力的复用性
