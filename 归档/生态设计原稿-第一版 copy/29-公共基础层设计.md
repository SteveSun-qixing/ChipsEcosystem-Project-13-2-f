# 公共基础层设计

## 架构定位

公共基础层位于薯片内核之上、插件桥接层之下，是整个生态的基础设施。它承载生态中所有插件和应用共用的通用能力，体现"底层厚、上层薄"的设计理念。

公共基础层分为两部分：

**A. 核心服务（运行在主进程中，数据/系统型）**：文件服务、卡片服务、箱子服务、资源管理、ZIP 处理、配置管理、主题管理、多语言、日志、标签数据库、凭证管理、数据序列化、平台服务、模块加载等。这些服务编译在薯片主机中，通过内核路由器对外提供能力。

**B. 共享功能模块（作为"模块插件"独立安装，UI 型）**：视频播放器、音频播放器、图片查看器、3D 渲染器、富文本编辑器、Markdown 渲染器、代码高亮器、文本编辑器等。这些模块各自独立仓库，可自由替换和升级，由上层插件按需加载使用，类似操作系统的共享库。

薯片内核只提供最核心的路由、注册、事件总线、生命周期管理等能力。公共基础层承载所有通用功能，让生态内的插件和软件保持轻量化，避免重复造轮子，提高代码模块复用率。

## 层级关系

从底层到顶层的完整架构是六层：Electron 宿主 → 平台抽象层（PAL）→ 薯片内核 → 基础服务层 + 共享功能模块 → 插件桥接层（Bridge API）→ 插件应用。

Electron 宿主提供 Chromium 引擎和 Node.js 运行时。平台抽象层适配不同操作系统的差异。薯片内核（TypeScript，运行在主进程中）提供中心路由、服务注册、事件总线、生命周期管理。基础服务层提供核心服务和可替换的共享功能模块。桥接层通过 contextBridge 向插件暴露 `window.chips.*` 标准接口。插件应用是运行在独立 BrowserWindow 中的 Web 应用。

## 包含的核心内容

### 核心服务（运行在主进程中）

卡片渲染相关服务是核心部分。卡片服务负责解析和操作 .card 文件，包括打开（解包）、保存（打包）、读取元数据和结构、管理基础卡片配置。箱子服务负责 .box 文件的对应操作。这些服务让上层插件通过 Bridge API 即可完成卡片和箱子的数据操作。

文件处理服务提供文件读写、压缩解压、格式转换等能力。ZIP 处理器用于创建和解析卡片和箱子文件（零压缩率 ZIP）。文件识别器根据文件扩展名和内容判断文件类型。

资源管理服务提供统一的资源访问层，屏蔽本地文件、网络资源、缓存的差异。上层插件通过 Bridge API 请求资源，资源管理服务自动处理获取、缓存和分发。

网络能力内置在主进程中，提供 HTTP 请求、下载管理等功能。网络协议适配器支持各种网络存储协议（WebDAV 等）的接入。缓存管理器处理网络资源的本地缓存。

配置管理服务提供层级化的配置系统。所有配置都通过配置管理器访问，支持四级配置覆盖（系统默认 → 用户全局 → 应用级 → 插件级），配置变化时自动通知相关模块。

主题管理服务负责主题包的注册、查询和分发。实现六级主题继承解析（组件 → 基础卡片 → 复合卡片 → 箱子 → 应用 → 全局），向插件提供主题 CSS。

数据序列化服务处理 YAML、JSON 等格式的解析和生成。所有配置文件的读写都通过这个服务。

日志系统提供统一的日志记录能力。所有服务和插件的日志都通过这个系统输出，支持分级过滤和文件输出。

多语言系统提供国际化支持。语言包的加载、文本翻译、语言切换等功能。

标签数据库服务管理文件的标签映射关系，支持标签查询和筛选。

凭证管理服务管理网络存储的认证信息，使用加密存储。

平台服务封装平台抽象层（PAL）的能力，通过内核路由向上层提供统一的平台接口（系统信息、屏幕信息、能力检测、全局快捷键等）。

模块加载服务管理共享功能模块的发现、加载和分发。

### 共享功能模块（作为模块插件独立安装）

通用媒体模块是基础层的重要部分。视频播放器模块提供视频解码和播放能力，支持多种视频格式。音频播放器模块提供音频播放和波形显示。图片查看模块提供图片显示、缩放、旋转等功能。三维模型渲染器模块提供 3D 模型的显示和交互。这些模块各自是独立的代码仓库和插件包。

文本处理模块包含多个独立模块。富文本编辑器模块提供富文本编辑能力。Markdown 渲染器模块将 Markdown 转换为 HTML。代码高亮器模块为各种编程语言提供语法高亮。文本编辑器模块提供基础的文本编辑能力。

这些共享功能模块不创建自己的窗口，由上层插件在自己的渲染进程中按需加载使用。加载方式有两种：作为 iframe 子组件加载，或作为 ES Module 动态导入。

### 无头组件库

界面组件库基于 Zag.js 和 Ark UI 构建，提供通用的界面元素。按钮、输入框、下拉菜单、滑块、弹窗等基础控件，这些控件只实现功能逻辑、结构和布局，**完全不包含任何视觉样式**。所有样式由主题包通过 CSS 变量和类名注入。组件暴露 CSS 类名接口点，主题包通过这些接口点定义具体的视觉表现。

## 模块组织原则

公共基础层的模块必须高度解耦。每个核心服务是独立的功能单元，通过内核路由器对外提供服务。服务之间不能直接依赖，必须通过薯片内核的路由系统通信。共享功能模块则各自是独立的代码仓库和插件包，可以被独立替换和升级。

接口设计需要保持高度一致的风格。所有服务使用统一的接口规范，参数传递方式统一，返回值格式统一，错误处理方式统一。上层插件通过 Bridge API 以一致的方式调用不同服务：`window.chips.invoke(namespace, action, params)`。

模块的职责单一明确。视频播放器只负责视频播放，不包含视频编辑功能。图片查看器只负责图片显示，不包含图片编辑功能。每个模块专注自己的核心功能。

模块支持按需加载。不是所有服务在启动时都加载，而是当某个功能需要时才加载对应的服务或模块插件。这减少了启动时间和内存占用。

## 协议设计

公共基础层定义了一套标准协议，规定上层插件如何调用基础层的能力。

模块调用协议规定了调用格式。上层插件通过 Bridge API 发送请求到主进程，主进程的内核路由器将请求路由到对应的服务处理器。所有调用都使用统一的请求响应格式。

资源访问协议规定了如何访问各种资源。本地文件、网络资源、嵌入资源等都通过统一的资源标识符访问，资源管理服务自动处理资源的获取和缓存。

事件通知协议规定了模块如何发出和接收事件。服务通过内核事件总线发布事件，插件通过 Bridge API 的事件订阅接口（`window.chips.on(eventName, callback)`）接收事件通知，实现松耦合的消息传递。

配置管理协议规定了配置的读写方式。所有配置都通过配置管理服务访问，支持多层级配置覆盖，配置变化时自动通知相关模块和插件。

## 与上层的关系

功能插件基于公共基础层开发。插件不需要自己实现视频播放、图片显示等基础功能，通过 Bridge API 调用对应的服务或加载对应的模块插件即可。插件只需要实现业务逻辑和界面编排部分，比如视频卡片插件负责管理视频的元数据、封面、字幕等，实际播放调用视频播放器模块插件。

软件开发工具包（SDK）基于 Bridge API 构建。第三方开发者通过 SDK 可以用类型安全的方式访问基础层的所有能力，开发自己的插件。SDK 是 Bridge API 的高层封装，提供更友好的开发接口。

## 版本管理

核心服务作为薯片主机的一部分，随主机一起更新，版本号与主机统一。

共享功能模块作为独立的模块插件，各自拥有独立的版本号，遵循语义化版本规范。上层插件在 manifest.yaml 中声明依赖的模块版本范围，安装时系统检查版本兼容性。

## 轻量化原则

公共基础层保持轻量化，只包含必需的功能。不包含任何业务逻辑，不包含具体的卡片实现，不包含具体的布局实现。这些都由上层插件提供。

不包含身份认证、用户管理、权限系统等功能。这些是社区服务器的职责，本地软件不需要。不包含数据同步、云端备份等功能，这些由具体的应用层实现。

只保留所有插件和软件都需要的通用能力。视频播放、图片显示、文本渲染等是通用需求，作为共享模块插件提供。具体的视频卡片如何组织数据、如何呈现界面，由视频卡片插件负责。

模块的实现也保持简洁。视频播放器使用成熟的开源库，不自己从零实现。图片处理使用现有的图像库。代码高亮使用现有的高亮库。基础层是这些开源工具的集成和封装，提供统一接口。

## 开源优先原则

公共基础层的开发遵循开源优先原则。在设计每个功能模块时，必须优先评估是否存在成熟的开源方案。凡是有开源方案的，采用提取和拆分开源项目模块的方式集成，仅对生态高度定制的部分进行全新设计。

视频播放、音频播放、PDF 解析、EPUB 阅读、Markdown 渲染、代码高亮、图片处理等功能都有成熟的开源实现，直接集成开源模块。ZIP 处理、文件识别、格式转换等也有开源库可用。只有卡片文件格式解析、薯片协议路由、主题引擎等生态特有的功能需要从零开发。

界面组件库采用 Zag.js（MIT 协议）作为组件状态机引擎，Ark UI（MIT 协议）作为框架绑定层。主题构建使用 Style Dictionary（Apache 2.0 协议）管理设计 Token。所有核心依赖均使用 MIT 或同等宽松的开源许可证。

这种方式大幅减少开发工作量，提高代码质量和稳定性，加快开发进度。开源模块经过社区验证，功能完善，性能优异。基础层主要工作是选择合适的开源库，提取所需功能，封装为统一接口，集成到生态中。
