# 全局主题管理系统

## 系统定位

全局主题管理系统是公共基础层的核心模块之一，负责整个薯片生态的主题登记、存储和分发。系统维护一个全局的主题注册表，记录所有已安装的主题包信息，为生态中的所有软件提供统一的主题查询和获取服务。

主题管理系统只负责主题包的管理，不负责主题的应用。主题如何应用到组件上由各个软件和插件自行实现，主题管理系统只提供主题包的存储和获取能力。这种职责分离让主题管理系统保持简洁，专注于管理这一单一职责。

## 主题存储

主题包是完整的前端视觉实现，包含所有组件的样式定义。主题管理系统将主题包存储在用户的本地目录中，每个主题包是一个独立的文件夹，包含样式文件、配置文件和可选的资源文件。

系统维护一个主题目录，所有已安装的主题包都存放在这个目录下。目录结构按照主题的发行商组织，每个发行商一个子目录，发行商目录下是该发行商发布的所有主题包。这种组织方式让用户可以方便地浏览和管理来自不同发行商的主题。

每个主题包目录包含一个配置文件，记录主题的基本信息：主题 ID、名称、版本、发行商、描述、支持的组件类型等。样式文件使用 CSS 格式，通过 CSS 变量系统定义视觉属性。主题包可以包含图片、图标等装饰性资源。

## 主题注册

主题管理系统启动时，扫描主题目录，读取每个主题包的配置文件，建立主题注册表。注册表记录每个主题的 ID、名称、版本、存储路径等信息，提供快速查询能力。

用户安装新主题时，系统将主题包解压到主题目录的对应位置，读取配置文件，将主题信息添加到注册表中。安装完成后，主题立即可用，所有软件都可以查询和使用这个新主题。

用户卸载主题时，系统从注册表中移除该主题的记录，删除主题包的文件夹。如果某个软件正在使用被卸载的主题，该软件会回退到默认主题或上一级主题。

## 默认主题

系统内置一个默认主题，这是唯一内置的主题。默认主题技术 ID 固定为 `chips-official.default-theme`，用于所有未显式配置主题的场景。

默认主题的样式文件随系统安装，存储在系统目录中，不在用户的主题目录中。用户不能卸载或修改默认主题，但可以安装其他主题覆盖默认主题的效果。

当某个组件请求主题但没有找到指定主题时，系统返回默认主题。当所有层级的主题配置都为空时，使用默认主题。默认主题是整个主题系统的兜底保障。

## 主题查询

生态中的软件通过主题管理系统查询可用的主题。查询接口返回所有已注册主题的列表，包含每个主题的基本信息。软件可以根据这个列表构建主题选择界面，让用户选择要使用的主题。

查询时可以指定过滤条件，比如只查询支持某种组件类型的主题，或者只查询某个发行商的主题。系统根据注册表中的信息进行过滤，返回符合条件的主题列表。

软件获取某个主题的详细信息时，系统返回该主题的完整配置，包括支持的组件类型、CSS 变量定义、资源文件列表等。软件根据这些信息决定如何应用主题。

## 主题获取

软件应用主题时，需要获取主题包的实际内容。软件向主题管理系统请求某个主题的样式文件，系统从主题包目录读取文件内容返回。

系统支持按需获取主题内容。软件可以只请求某个组件类型的样式，而不是获取整个主题包。系统解析请求，从主题包中提取对应的部分返回。这种按需获取减少了数据传输量，提高了加载效率。

主题包的资源文件通过资源路径访问。系统提供资源路径解析服务，将主题资源的相对路径转换为实际的文件路径，让软件可以正确加载主题中的图片、图标等资源。

## 主题安装

用户可以从多个来源安装主题包。最常见的方式是从应用市场下载安装，市场提供主题包的预览和下载功能。用户也可以直接导入本地的主题包文件，系统解析文件并安装到主题目录。

安装过程中，系统验证主题包的格式是否正确，配置文件是否完整，样式文件是否有效。验证通过后，将主题包复制到主题目录，更新注册表。验证失败则拒绝安装，返回错误信息。

系统支持主题包的更新。当用户安装一个已存在主题的新版本时，系统比较版本号，如果新版本更高，则替换旧版本的文件，更新注册表中的版本信息。

## 与软件的关系

主题管理系统是公共基础层的模块，为所有生态软件提供服务。编辑引擎、查看器、其他第三方软件都通过同一个主题管理系统获取主题，确保整个生态的主题体验一致。

软件启动时，可以查询主题管理系统获取可用主题列表，构建主题选择界面。用户选择主题后，软件从主题管理系统获取主题内容，应用到界面组件上。

软件不需要自己管理主题文件，所有主题相关的存储和管理都由主题管理系统负责。软件只需要通过标准接口查询和获取主题，专注于主题的应用逻辑。这种分离让软件更轻量，主题管理更统一。
