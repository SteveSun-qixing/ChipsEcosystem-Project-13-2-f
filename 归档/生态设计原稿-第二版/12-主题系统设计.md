# 主题系统设计

## 主题系统定位

薯片生态的组件库采用无头设计，只实现功能逻辑、结构和布局，完全不包含任何视觉样式代码。真正的样式和界面代码保存在各个主题包中。

主题系统负责前端的视觉层面，通过主题包实现界面的多样化外观。

---

## 主题包结构

主题包是完整的前端视觉实现，包含所有组件的样式定义：

- 按钮的颜色形状
- 输入框的边框样式
- 卡片的间距阴影
- 动画效果等

主题包使用纯 CSS 实现，通过 CSS 变量系统定义颜色、字体、圆角、阴影等视觉属性。

主题包还可以添加装饰性元素，比如背景图片、图标、自定义图形，但**不能修改组件的 HTML 结构和布局方式**。

---

## 主题包格式

每个主题包目录包含：

- `manifest.yaml`：主题配置文件
- `theme.css`：主样式文件
- 可选的图片、图标等资源文件

### manifest.yaml 示例

```yaml
id: chips-official.default-theme  # 技术 ID（必须唯一）
name: 默认主题                      # 显示名称
version: 1.0.0                     # 语义化版本
publisher: 薯片官方                 # 发行商
description: 薯片官方默认主题        # 描述
components:                         # 支持的组件类型列表
  - card.video
  - card.image
  - card.text
  - layout.grid
  - layout.list
```

**系统主键使用 `themeId`**（例如 `chips-official.default-theme`），界面可展示"薯片官方：默认主题"这类名称，但运行时解析必须使用 `themeId`。

## CSS 变量系统

主题包使用 CSS 变量 Token 系统，分为三层：

### 全局 Token（色板）
```css
:root {
  --chips-color-primary: #007AFF;
  --chips-color-secondary: #5856D6;
  --chips-color-success: #34C759;
  --chips-color-warning: #FF9500;
  --chips-color-error: #FF3B30;
  --chips-color-background: #FFFFFF;
  --chips-color-text: #000000;
}
```

### 语义化 Token（主要修改层）
```css
:root {
  --chips-button-bg: var(--chips-color-primary);
  --chips-button-text: #FFFFFF;
  --chips-card-bg: var(--chips-color-background);
  --chips-border-radius: 8px;
}
```

### 组件级 Token（细粒度）
```css
.theme-video-card {
  --video-control-height: 32px;
  --video-progress-color: var(--chips-color-primary);
}
```

## 原子级主题应用

主题支持原子级应用。每个基础卡片可以单独设置主题包，每个界面组件也可以单独设置主题包。一个复合卡片中的不同基础卡片可以使用不同的主题包。

---

## 主题配置层级

主题配置采用**六级覆盖机制**，查找优先级从高到低为：

1. **组件主题** → 2. **基础卡片主题** → 3. **复合卡片主题** → 4. **箱子主题** → 5. **应用主题** → 6. **全局主题**

配置文件中的主题字段默认为空。如果某个层级没有设置主题，就使用上一层级的主题。如果所有层级都没有设置，使用系统默认主题。

---

## 全局主题管理系统

全局主题管理系统是公共基础层的核心模块之一，负责整个薯片生态的主题登记、存储和分发。

### 主题存储

主题管理系统将主题包存储在用户的本地目录中，每个主题包是一个独立的文件夹。

目录结构按照主题的发行商组织，每个发行商一个子目录，发行商目录下是该发行商发布的所有主题包。

### 主题注册

系统启动时扫描主题目录，读取每个主题包的配置文件，建立主题注册表。

注册表记录每个主题的 ID、名称、版本、存储路径等信息，提供快速查询能力。

### 主题查询

生态中的软件通过主题管理系统查询可用的主题。查询接口返回所有已注册主题的列表。

查询时可以指定过滤条件，比如只查询支持某种组件类型的主题，或者只查询某个发行商的主题。

### 主题获取

软件应用主题时，需要获取主题包的实际内容。软件向主题管理系统请求某个主题的样式文件，系统从主题包目录读取文件内容返回。

系统支持按需获取主题内容。软件可以只请求某个组件类型的样式，而不是获取整个主题包。

### 默认主题

系统内置一个默认主题，技术 ID 固定为 `chips-official.default-theme`，用于所有未显式配置主题的场景。

默认主题的样式文件随系统安装，存储在系统目录中，不在用户的主题目录中。用户不能卸载或修改默认主题。

---

## 主题应用流程

1. 渲染器读取组件的主题配置
2. 如果为空则使用上级主题
3. 逐级向上查找，直到找到非空主题或到达全局主题
4. 如果所有层级都为空，使用默认主题
5. 加载对应主题包
6. 提取该组件类型的样式代码
7. 注入到组件的前端代码中
8. 生成包含样式的完整网页

---

## 前端组件和主题的关系

前端组件库提供功能性组件，这些组件实现交互逻辑、数据绑定、事件处理等功能，**但不定义视觉样式**。

组件的 HTML 结构和布局逻辑是固定的，但完全不包含样式代码。组件通过 CSS 类名暴露样式接口点，主题包通过这些类名定义具体的样式。

主题包针对每种组件类型提供样式实现。一个完整的主题包需要覆盖所有基础卡片类型和所有界面组件类型。

---

## 主题安装

用户可以从多个来源安装主题包：

- 从应用市场下载安装
- 直接导入本地的主题包文件

安装过程中，系统验证：
- 主题包的格式是否正确
- 配置文件是否完整
- 样式文件是否有效

验证通过后，将主题包复制到主题目录，更新注册表。

---

## 主题与前端多样化

同一个后端功能可以有多个前端产品，同一个前端产品可以有多个主题包。

用户可以选择前端产品（比如选择看板风格的待办界面），然后选择主题包（比如选择暗色主题），两者组合产生最终的视觉效果。
