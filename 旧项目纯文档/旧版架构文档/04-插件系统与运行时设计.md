## 一、插件类型体系

薯片生态的"一切皆插件"理念在新架构中得到更彻底的实现。所有插件分为五种类型：

### 1.1 应用插件（App Plugin）

完整的应用级插件，拥有独立的 BrowserWindow 窗口。

**示例**：编辑引擎、卡片查看器、生态设置面板、资源管理器、文件连接器、标签编辑器

**运行方式**：
- 每个应用插件在一个独立的 BrowserWindow（独立渲染进程）中运行
- 拥有完整的 Web 页面（index.html 入口）
- 通过 Bridge API 调用底层能力
- 可以加载卡片插件和布局插件

**文件结构**：
```
editor-plugin/
├── manifest.yaml          # 插件清单
├── index.html             # 入口页面
├── assets/
│   ├── js/
│   │   ├── main.js        # 主脚本
│   │   └── vendor.js      # 第三方库
│   ├── css/
│   │   └── style.css      # 样式
│   └── images/
│       └── icon.png        # 图标
└── locales/                # 插件自有的翻译文件（可选）
    ├── zh-CN.yaml
    └── en-US.yaml
```

### 1.2 卡片插件（Card Plugin）

基础卡片类型的实现，包含渲染组件和编辑组件。

**示例**：视频卡片、图片卡片、富文本卡片、Markdown 卡片、代码块卡片等 26 种

**运行方式**：
- 渲染组件以 iframe 方式加载在应用插件的页面中
- 编辑组件以 iframe 方式加载在编辑面板中
- 通过 postMessage 与宿主应用插件通信
- 可以间接调用 Bridge API（通过宿主应用中转）

**文件结构**：
```
video-card-plugin/
├── manifest.yaml          # 插件清单（声明类型为 card）
├── renderer/
│   ├── index.html         # 渲染组件入口
│   ├── renderer.js        # 渲染逻辑
│   └── renderer.css       # 渲染样式接口点
├── editor/
│   ├── index.html         # 编辑组件入口
│   ├── editor.js          # 编辑逻辑
│   └── editor.css         # 编辑样式
└── locales/
    ├── zh-CN.yaml
    └── en-US.yaml
```

### 1.3 布局插件（Layout Plugin）

箱子的布局展示方式。

**示例**：网格布局、瀑布流布局、列表布局、无限桌面布局、时间线布局等

**运行方式**：
- 与卡片插件类似，以 iframe 方式加载
- 接收卡片列表和布局配置，输出展示页面

**文件结构**：
```
waterfall-layout-plugin/
├── manifest.yaml
├── renderer/
│   ├── index.html         # 布局渲染入口
│   ├── layout.js          # 布局算法
│   └── layout.css         # 布局样式接口点
├── editor/                # 布局参数编辑界面（可选）
│   ├── index.html
│   └── editor.js
└── locales/
```

### 1.4 主题插件（Theme Plugin）

纯 CSS 样式包，不包含运行时代码。

**示例**：默认主题、深色主题、马卡龙配色、自然风光主题等

**运行方式**：
- 主题管理服务加载主题包的 CSS
- 注入到需要主题的组件/卡片/应用的渲染上下文中
- 不创建进程或窗口

**文件结构**：
```
macaron-theme/
├── manifest.yaml
├── tokens/
│   ├── global.css         # 全局 Token（色板）
│   ├── semantic.css       # 语义化 Token
│   └── components/        # 组件级 Token
│       ├── button.css
│       ├── input.css
│       ├── card-wrapper.css
│       └── ...
├── decorations/           # 装饰性资源（可选）
│   ├── background.png
│   └── icons/
└── preview.png            # 主题预览图
```

### 1.5 模块插件（Module Plugin）

公共基础层中可被多个插件共享的 UI 功能模块。类似操作系统的共享库（.dll / .so）——安装一次，多个插件共用。

**示例**：视频播放器、音频播放器、图片查看器、3D 渲染器、富文本编辑器、Markdown 渲染器、代码高亮器

**运行方式**：
- 不创建自己的窗口，没有独立进程
- 由其他插件（通常是卡片插件或应用插件）在自己的渲染进程中加载
- 支持两种加载方式：作为 iframe 子组件、作为 ES Module 动态导入
- 各自独立仓库，可自由替换和升级

**文件结构**：
```
video-player-module/
├── manifest.yaml          # 插件清单（type: module）
├── dist/
│   ├── index.html         # iframe 入口（备选加载方式）
│   ├── index.js           # ES Module 入口
│   └── assets/
│       └── style.css
└── locales/
    ├── zh-CN.yaml
    └── en-US.yaml
```

**manifest.yaml 示例**：
```yaml
id: "chips-module-video-player"
name: "视频播放器"
type: "module"                      # 第五种类型
version: "1.0.0"
publisher: "薯片官方"

entry:
  module: "dist/index.js"           # ES Module 入口
  iframe: "dist/index.html"         # iframe 入口（备选）

exports:
  - name: "createVideoPlayer"
    description: "创建视频播放器实例"
  - name: "VideoPlayerComponent"
    description: "Vue 3 视频播放器组件"

permissions:
  - resource.fetch
```

**使用方式**：
```typescript
// 卡片插件或应用插件中加载共享模块
const player = await chips.invoke('module', 'load', {
  moduleId: 'chips-module-video-player'
});

// 方式 1：作为 iframe 加载
const iframe = document.createElement('iframe');
iframe.src = player.iframeUrl;
container.appendChild(iframe);

// 方式 2：作为 ES Module 动态导入
const mod = await import(player.moduleUrl);
mod.createVideoPlayer(container, { src: videoPath });
```

**模块插件清单**：

| 模块 | ID | 说明 |
|------|-----|------|
| 视频播放器 | chips-module-video-player | 视频解码、播放控制、字幕 |
| 音频播放器 | chips-module-audio-player | 音频播放、波形显示 |
| 图片查看器 | chips-module-image-viewer | 图片显示、缩放、旋转 |
| 3D 渲染器 | chips-module-3d-renderer | 3D 模型显示和交互 |
| 富文本编辑器 | chips-module-rich-text-editor | 富文本编辑能力 |
| Markdown 渲染器 | chips-module-markdown-renderer | Markdown → HTML |
| 代码高亮器 | chips-module-code-highlighter | 代码语法高亮 |
| 文本编辑器 | chips-module-text-editor | 基础文本编辑 |

## 二、插件清单（manifest.yaml）

每个插件必须包含一个 `manifest.yaml` 文件，描述插件的所有元信息：

```yaml
# 插件基本信息
id: "chips-official.video-card"           # 唯一标识符（发行商.名称）
name: "视频卡片"                           # 显示名称
version: "1.0.0"                          # 版本号（语义化）
type: "card"                              # 类型：app | card | layout | module | theme
publisher: "薯片官方"                      # 发行商
description: "支持多种视频格式的基础卡片"     # 描述
icon: "assets/icon.png"                   # 图标路径

# 入口文件
entry:
  renderer: "renderer/index.html"         # 渲染组件入口（card/layout 类型）
  editor: "editor/index.html"             # 编辑组件入口（card/layout 类型）
  main: "index.html"                      # 应用入口（app 类型）

# 能力声明
capabilities:
  fileTypes: [".mp4", ".webm", ".mkv"]    # 能处理的文件类型
  cardType: "VideoCard"                   # 卡片类型标识（card 类型）
  layoutType: null                        # 布局类型标识（layout 类型）

# 权限声明
permissions:
  - file.read
  - file.write
  - resource.fetch
  - card.read
  - card.write

# 依赖声明
dependencies:
  host: ">=1.0.0"                         # 主机版本要求
  plugins: []                             # 依赖的其他插件

# 窗口配置（仅 app 类型）
window:
  width: 1200
  height: 800
  minWidth: 600
  minHeight: 400
  resizable: true
  frame: true                             # 是否使用系统窗口框架

# 多语言
locales:
  default: "zh-CN"
  supported: ["zh-CN", "en-US", "ja-JP"]
```

### 3.2 插件安装流程

```
1. 用户触发安装（从应用市场、本地文件、或拖入主机窗口）
2. 插件管理器接收安装请求
3. 验证阶段：
   3.1 解析 manifest.yaml
   3.2 检查 ID 唯一性（是否已安装同 ID 插件）
   3.3 检查主机版本兼容性
   3.4 检查依赖插件是否已安装
   3.5 验证必需文件是否存在
4. 安装阶段：
   4.1 将插件文件复制到对应类型的插件目录
   4.2 在插件注册表中记录插件信息
   4.3 通知内核路由器更新可用服务
5. 安装完成：
   5.1 发布 plugin.installed 事件
   5.2 通知用户安装成功
```

### 3.3 插件卸载流程

```
1. 用户触发卸载
2. 检查是否有其他插件依赖此插件
3. 如果插件当前正在运行，先停止它
4. 从插件注册表移除记录
5. 删除插件目录
6. 发布 plugin.uninstalled 事件
7. 通知用户卸载成功
```

### 3.4 插件更新

```
1. 接收新版本的插件包
2. 验证新版本的 manifest.yaml
3. 如果插件当前正在运行，通知它保存状态
4. 停止旧版本
5. 备份旧版本到临时目录
6. 安装新版本
7. 尝试启动新版本
8. 如果启动失败，回滚到旧版本
9. 启动成功则删除备份
```

## 四、应用插件运行时

### 4.1 创建窗口

当用户打开一个应用插件时：

```typescript
// 主进程中的 PluginWindowManager
async openAppPlugin(pluginId: string, params?: any) {
  // 1. 从注册表获取插件信息
  const plugin = this.registry.getPlugin(pluginId);

  // 2. 确定入口文件路径
  const entryPath = path.join(plugin.installPath, plugin.manifest.entry.main);

  // 3. 创建 BrowserWindow
  const win = new BrowserWindow({
    width: plugin.manifest.window?.width || 1200,
    height: plugin.manifest.window?.height || 800,
    webPreferences: {
      preload: path.join(__dirname, 'preload/bridge.js'),
      contextIsolation: true,      // 上下文隔离
      nodeIntegration: false,       // 禁止 Node.js 集成
      sandbox: true,                // 沙箱模式
      webSecurity: true,            // Web 安全策略
    }
  });

  // 4. 加载插件入口页面
  win.loadFile(entryPath);

  // 5. 传递启动参数
  win.webContents.on('did-finish-load', () => {
    win.webContents.send('plugin-init', {
      pluginId,
      params,
      permissions: plugin.manifest.permissions
    });
  });

  // 6. 记录窗口映射
  this.windowMap.set(win.id, pluginId);

  return win;
}
```

### 4.2 预加载脚本（Bridge 注入）

每个应用插件的 BrowserWindow 都使用相同的预加载脚本，通过 `contextBridge` 注入 Bridge API：

```typescript
// preload/bridge.ts
import { contextBridge, ipcRenderer } from 'electron';

contextBridge.exposeInMainWorld('chips', {
  // 核心调用
  invoke: (namespace: string, action: string, params?: any) =>
    ipcRenderer.invoke('chips:invoke', { namespace, action, params }),

  // 事件订阅
  on: (eventName: string, callback: Function) => {
    const listener = (_event: any, data: any) => callback(data);
    ipcRenderer.on(`chips:event:${eventName}`, listener);
    return () => ipcRenderer.removeListener(`chips:event:${eventName}`, listener);
  },

  // 事件发送
  emit: (eventName: string, data?: any) =>
    ipcRenderer.send('chips:emit', { eventName, data }),

  // 窗口控制
  window: {
    close: () => ipcRenderer.send('chips:window:close'),
    minimize: () => ipcRenderer.send('chips:window:minimize'),
    maximize: () => ipcRenderer.send('chips:window:maximize'),
    setTitle: (title: string) => ipcRenderer.send('chips:window:setTitle', title),
    openPlugin: (pluginId: string, params?: any) =>
      ipcRenderer.invoke('chips:window:openPlugin', { pluginId, params }),
  }
});
```

### 4.3 插件中的使用示例

```typescript
// 插件前端代码（任何 Web 框架）

const metadata = await window.chips.invoke('card', 'getMetadata', {
```

## 五、卡片插件运行时

### 5.1 渲染组件加载

卡片插件的渲染组件在应用插件的页面内以 iframe 加载：

```
应用插件 BrowserWindow
├── 应用的 UI（Vue/React/...）
│   ├── 菜单栏
│   ├── 工具栏
│   └── 卡片显示区域
│       ├── iframe: 基础卡片 A 的渲染组件
│       ├── iframe: 基础卡片 B 的渲染组件
│       └── iframe: 基础卡片 C 的渲染组件
```

### 5.2 渲染流程

```
1. 应用插件调用 chips.invoke('card', 'getStructure', { cardPath })
   → 主进程读取 structure.yaml，返回基础卡片列表

2. 对每个基础卡片：
   a. 应用插件调用 chips.invoke('card', 'getBaseCardConfig', { cardPath, baseCardId })
      → 主进程读取 content/ID.yaml，返回配置数据

   b. 应用插件查找该类型的卡片插件路径
      → chips.invoke('plugin', 'getCardPlugin', { cardType: 'VideoCard' })
      → 返回插件的渲染组件路径

   c. 应用插件创建 iframe，加载渲染组件
      → iframe src = "file:///path/to/video-card-plugin/renderer/index.html"

   d. 渲染组件加载完成后，应用插件通过 postMessage 传递配置数据
      → iframe.contentWindow.postMessage({ type: 'init', config: {...} })

   e. 渲染组件接收数据，渲染卡片内容
      → 如果需要访问资源（视频文件等），通过 postMessage 请求宿主应用代为调用 Bridge API

3. 如果设置了主题，应用插件获取主题 CSS 并注入到每个 iframe
```

### 5.3 卡片 ↔ 应用通信协议

卡片插件（iframe 内）与宿主应用插件之间通过 `postMessage` 通信：

```typescript
// 宿主应用 → 卡片 iframe
iframe.contentWindow.postMessage({
  type: 'init',              // 初始化
  config: { ... },           // 卡片配置数据
  theme: { css: '...' },     // 主题 CSS
  resources: { ... }         // 资源映射表
}, '*');

// 卡片 iframe → 宿主应用
window.parent.postMessage({
  type: 'bridge-request',     // 请求宿主代为调用 Bridge API
  requestId: 'xxx',
  namespace: 'resource',
  action: 'fetch',
  params: { uri: 'video.mp4' }
}, '*');

// 宿主应用 → 卡片 iframe（响应）
iframe.contentWindow.postMessage({
  type: 'bridge-response',
  requestId: 'xxx',
  result: { url: 'file:///cached/video.mp4' }
}, '*');

// 卡片 iframe → 宿主应用（尺寸变化通知）
window.parent.postMessage({
  type: 'resize',
  height: 600                 // 内容高度变化
}, '*');
```

## 六、插件生命周期

### 6.1 应用插件生命周期

```
             ┌──────────┐
             │  已安装   │
             └────┬─────┘
                  │ openPlugin()
                  ▼
             ┌──────────┐
             │  加载中   │ ← 创建 BrowserWindow，加载 index.html
             └────┬─────┘
                  │ did-finish-load
                  ▼
             ┌──────────┐
             │  初始化   │ ← 接收启动参数，初始化插件状态
             └────┬─────┘
                  │ plugin-ready
                  ▼
        ┌────────────────────┐
        │       运行中        │ ← 正常响应用户操作
        └────┬──────────┬────┘
             │          │
    窗口最小化│          │ 用户关闭/调用 close
             ▼          ▼
       ┌──────────┐ ┌──────────┐
       │  后台中   │ │  停止中   │ ← 保存状态，释放资源
       └────┬─────┘ └────┬─────┘
            │            │
    窗口恢复│            │ 完成清理
            ▼            ▼
       ┌──────────┐ ┌──────────┐
       │  运行中   │ │  已停止   │ ← BrowserWindow 销毁
       └──────────┘ └──────────┘
```

### 6.2 崩溃恢复

当插件渲染进程崩溃时：

```
1. Electron 检测到渲染进程崩溃
2. 主进程收到 'render-process-gone' 事件
3. 记录崩溃日志
4. 通知用户"XX 插件异常退出"
5. 提供"重新打开"选项
6. 用户选择重新打开时，创建新的 BrowserWindow 加载插件
7. 尝试恢复上次保存的状态
```

### 6.3 资源回收

- 关闭插件窗口时，BrowserWindow 被销毁，渲染进程自动回收
- 长时间后台的插件可以被挂起（暂停渲染进程的 JavaScript 执行）
- 内存压力大时，主进程可以选择性地关闭最久未使用的后台插件

```json
        "cardType": "VideoCard",

      }
    }
  },
  "fileTypeHandlers": {
    ".card": ["chips-official.viewer", "chips-official.editor"],
    ".box": ["chips-official.viewer"],
    ".mp4": ["chips-official.video-card"],
    ".png": ["chips-official.image-card"]
  },
  "defaultHandlers": {
    ".card": "chips-official.viewer",
    ".box": "chips-official.viewer"
  }
}
```

注册表在主机启动时加载到内存，插件安装/卸载时更新并持久化。

## 八、编辑引擎内部插件化

编辑引擎作为一个应用插件，自身也拥有高度插件化的内部架构。编辑引擎的内部扩展也是独立仓库，通过编辑引擎的扩展注册 API 加载。

### 8.1 内部扩展架构

```
编辑引擎应用插件
├── 引擎框架（固定核心）：窗口壳、菜单栏、IPC 桥接、扩展注册器
│
├── 扩展槽 A：编辑引擎布局（可替换，同时只激活一个）
│   ├── 无限画布布局（独立仓库/插件）
│   ├── 工作台布局（独立仓库/插件）
│   ├── 页面式布局（独立仓库/插件，未来）
│   └── 看板布局（独立仓库/插件，未来）
│
├── 扩展槽 B：工具面板（可添加/移除，可同时激活多个）
│   ├── 文件管理器面板（独立仓库/插件）
│   ├── 卡箱库面板（独立仓库/插件）
│   ├── 编辑面板（独立仓库/插件）
│   ├── 历史记录面板（独立仓库/插件）
│   └── 第三方自定义面板（社区开发）
│
├── 扩展槽 C：功能扩展（可添加，后台运行）
│   ├── 撤销重做模块（独立仓库/插件）
│   ├── 搜索替换模块（独立仓库/插件）
│   ├── 自动保存模块（独立仓库/插件）
│   └── 协作编辑模块（未来，独立仓库/插件）
```

### 8.2 编辑引擎扩展注册

内部扩展在编辑引擎的 BrowserWindow 内运行（不创建新窗口），可以作为 iframe 子组件或 ES Module 动态导入：

```typescript
// 编辑引擎启动时，从已安装的扩展中加载
const layouts = await chips.invoke('plugin', 'query', {
  type: 'editor-layout'
});
const panels = await chips.invoke('plugin', 'query', {
  type: 'editor-panel'
});

// 注册到编辑引擎内部
for (const layout of layouts) {
  editor.registerLayout(layout.id, layout.entryUrl);
}
for (const panel of panels) {
  editor.registerPanel(panel.id, panel.entryUrl);
}
```

### 8.3 编辑引擎扩展的 manifest

编辑引擎内部扩展使用与生态插件相同的 manifest.yaml 格式，但 type 字段使用子类型标识：

```yaml
id: "chips-official.infinite-canvas-layout"
name: "无限画布布局"
type: "editor-layout"              # 编辑引擎布局扩展
version: "1.0.0"
publisher: "薯片官方"

entry:
  main: "dist/index.html"

# 声明这是编辑引擎的内部扩展
host_plugin: "chips-official.editor"  # 宿主插件 ID
```

这些编辑引擎扩展与其他插件使用相同的 .cpk 打包格式和安装流程，用户可以在设置面板中管理。
