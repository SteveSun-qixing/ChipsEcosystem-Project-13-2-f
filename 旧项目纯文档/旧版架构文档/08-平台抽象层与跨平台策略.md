## 一、薯片生态的定位：软件级操作系统

薯片生态不是一个普通的应用程序，而是一个**运行在操作系统之上的软件级生态系统**。它的目标是：

> **给用户设备的操作系统套一层壳**。用户日常大部分操作都在薯片生态内完成，全平台体验统一。

类比关系：

```
传统软件生态：
  操作系统 → 各种独立软件

薯片生态：
  操作系统 → 薯片底层环境（常驻运行）→ 各种薯片应用（用户安装使用）

类似于：
  Android/iOS → 微信 → 微信小程序
  操作系统 → Docker → 容器化应用
  Chrome OS → Chrome 浏览器 → Web 应用/PWA
  操作系统 → JVM/CLR → Java/.NET 应用
```

**关键特征**：
- 薯片底层**长期无感运行**在用户电脑上（开机自启、系统托盘）
- 用户直接安装上层应用，使用体验与普通软件一致
- **底层厚、上层薄**：大部分能力在底层，上层应用只是薄壳
- 一次开发，全平台运行（Windows、macOS、Linux）
- 底层尽可能接管操作系统能力，在生态内提供统一抽象

## 三、平台抽象层（PAL）设计

### 3.1 核心理念

平台抽象层（Platform Abstraction Layer）位于操作系统和薯片内部世界之间。它的职责是：

1. **向下适配**：针对不同操作系统的差异，用不同的适配器实现相同的功能
2. **向上统一**：向生态内部提供完全统一的接口，内部模块不需要知道运行在什么系统上
3. **最大公约数**：覆盖 Windows、macOS、Linux 共有的能力
4. **预留扩展**：为平台独有能力保留架构接口

### 3.2 适配器体系

每一类系统能力都有一个抽象接口和多个平台适配器：

```typescript
// 抽象接口（薯片内部使用的统一接口）
interface FileSystemAdapter {
  readFile(path: string): Promise<Buffer>;
  writeFile(path: string, data: Buffer): Promise<void>;
  listDirectory(path: string): Promise<FileEntry[]>;
  watchPath(path: string, callback: WatchCallback): Disposable;
  getHomePath(): string;
  getTempPath(): string;
  getAppDataPath(): string;
  showInFileManager(path: string): Promise<void>;
  // ...
}

// Windows 适配器
class WindowsFileSystemAdapter implements FileSystemAdapter {
  getHomePath() { return process.env.USERPROFILE; }
  getAppDataPath() { return process.env.APPDATA + '/Chips'; }
  showInFileManager(path) { shell.openPath(path); /* explorer.exe */ }
  // ...
}

// macOS 适配器
class MacOSFileSystemAdapter implements FileSystemAdapter {
  getHomePath() { return process.env.HOME; }
  getAppDataPath() { return process.env.HOME + '/Library/Application Support/Chips'; }
  showInFileManager(path) { shell.showItemInFolder(path); /* Finder */ }
  // ...
}

// Linux 适配器
class LinuxFileSystemAdapter implements FileSystemAdapter {
  getHomePath() { return process.env.HOME; }
  getAppDataPath() { return process.env.HOME + '/.config/chips'; }
  showInFileManager(path) { /* xdg-open */ }
  // ...
}
```

### 3.3 平台能力清单

**第一类：核心能力（所有平台必须支持，最大公约数）**

| 能力域 | 统一接口 | 说明 |
|--------|---------|------|
| 文件系统 | `platform.fs.*` | 文件读写、目录操作、路径解析、文件监听 |
| 窗口系统 | `platform.window.*` | 窗口创建、管理、最小化/最大化/全屏 |
| 系统托盘 | `platform.tray.*` | 托盘图标、菜单、通知 |
| 通知系统 | `platform.notification.*` | 系统通知推送 |
| 剪贴板 | `platform.clipboard.*` | 文本/图片/文件剪贴板操作 |
| 网络 | `platform.network.*` | HTTP/WebSocket/网络状态检测 |
| 进程管理 | `platform.process.*` | 子进程、Worker 管理 |
| 系统信息 | `platform.system.*` | OS 类型、版本、CPU、内存、屏幕 |
| 电源管理 | `platform.power.*` | 电池状态、休眠锁定 |
| 快捷键 | `platform.shortcut.*` | 全局快捷键注册 |
| 自动启动 | `platform.autoLaunch.*` | 开机自启动配置 |
| 文件关联 | `platform.fileAssoc.*` | 文件类型注册 |
| URL 协议 | `platform.protocol.*` | 自定义协议处理（chips://） |
| 对话框 | `platform.dialog.*` | 打开/保存文件、消息框 |
| Shell 集成 | `platform.shell.*` | 打开外部链接、在文件管理器中显示 |
| 屏幕信息 | `platform.screen.*` | 多显示器、分辨率、DPI |
| 拖放 | `platform.dragDrop.*` | 系统级拖放支持 |
| 主题检测 | `platform.appearance.*` | 系统暗色/亮色模式检测 |
| 辅助功能 | `platform.accessibility.*` | 系统无障碍设置检测 |
| 本地存储 | `platform.storage.*` | 键值对持久化存储 |

**第二类：扩展能力（部分平台支持，需要能力检测）**

| 能力域 | 平台 | 说明 |
|--------|------|------|
| 桌面壁纸 | Windows | 接管桌面壁纸显示 |
| 触控栏 | macOS | Touch Bar 集成 |
| 菜单栏 | macOS | 系统菜单栏集成 |
| 任务栏缩略图 | Windows | 任务栏预览和按钮 |
| 系统主题色 | Windows/macOS | 读取系统强调色 |
| Spotlight/搜索 | macOS | 系统搜索集成 |
| D-Bus | Linux | 系统总线通信 |
| 桌面小组件 | Windows/macOS | 系统小组件 |

**第三类：预留能力（当前不实现，保留接口）**

| 能力域 | 说明 |
|--------|------|
| 蓝牙 | 设备发现和数据传输 |
| USB 设备 | USB 设备检测和通信 |
| 摄像头/麦克风 | 媒体设备访问 |
| GPS/位置 | 位置信息获取 |
| 系统健康 | CPU/内存/磁盘实时监控 |
| 打印 | 打印机管理和打印 |

### 3.4 能力检测机制

```typescript
// 插件可以查询平台能力
const hasWallpaper = await chips.invoke('platform', 'hasCapability', {
  capability: 'desktop.wallpaper'
});

if (hasWallpaper) {
  await chips.invoke('platform', 'setWallpaper', { path: '...' });
}

// 获取所有可用能力
const capabilities = await chips.invoke('platform', 'listCapabilities');
// → ['fs', 'window', 'tray', 'notification', 'clipboard', 'desktop.wallpaper', ...]
```

### 3.5 平台独有功能架构

平台独有功能通过**平台扩展插件**实现：

```
plugins/
├── platform-extensions/
│   ├── windows-desktop-wallpaper/      # Windows 桌面壁纸接管
│   │   ├── manifest.yaml
│   │   │   platform: windows           # 声明仅限 Windows
│   │   └── ...
│   ├── macos-touchbar/                 # macOS Touch Bar 集成
│   │   ├── manifest.yaml
│   │   │   platform: macos
│   │   └── ...
│   └── linux-dbus-integration/         # Linux D-Bus 集成
│       ├── manifest.yaml
│       │   platform: linux
│       └── ...
```

主机在启动时检测当前平台，自动加载对应的平台扩展插件。

## 四、常驻运行模型

### 4.1 系统服务模式

薯片底层长期无感运行在用户电脑上：

```
用户开机
  → 操作系统启动
  → 薯片主机自动启动（注册为开机启动项）
  → 主机进入后台模式
     ├── 主进程运行（内核 + 基础服务 + PAL）
     ├── 系统托盘图标就绪
     └── 等待用户操作或文件事件

用户双击 .card 文件
  → 操作系统传递给薯片主机（已在运行中）
  → 主机即时创建 BrowserWindow（无冷启动延迟）
  → 加载查看器插件，显示卡片内容
  → 用户体验：文件秒开

用户点击薯片应用图标
  → 操作系统传递给薯片主机
  → 主机创建 BrowserWindow
  → 加载对应的应用插件
  → 用户感觉像打开了一个普通软件

用户关闭所有窗口
  → 主机不退出，回到后台模式
  → 系统托盘图标保持
  → 下次打开应用时秒启动
```

### 4.2 应用快捷方式

每个安装的应用插件在操作系统中创建快捷方式：

```
用户桌面/开始菜单/Dock：
  ├── 薯片编辑引擎.lnk     → 启动主机 + 加载编辑引擎插件
  ├── 薯片查看器.lnk        → 启动主机 + 加载查看器插件
  ├── 薯片视频播放器.lnk     → 启动主机 + 加载视频播放器插件
  └── 待办清单.lnk          → 启动主机 + 加载待办插件
```

快捷方式实际上都指向同一个薯片主机可执行文件，通过不同的启动参数指定加载哪个插件：

```bash
# Windows
chips-host.exe --plugin=chips-official.editor

# macOS
open -a "Chips Host" --args --plugin=chips-official.editor

# Linux
chips-host --plugin=chips-official.editor
```

对用户来说，每个"应用"就像独立的软件一样，有自己的图标和名称。但实际上它们都运行在同一个薯片主机进程中。

### 4.3 资源管理策略

```
主机后台空闲时：
  ├── 主进程：~50MB（内核 + 必要服务）
  └── 无渲染进程（所有窗口已关闭）

打开一个应用：
  ├── 主进程：~60MB（加载了更多服务）
  └── 渲染进程 A：~100-200MB（应用插件）

打开多个应用：
  ├── 主进程：~70MB（共享不变）
  ├── 渲染进程 A：~100-200MB
  ├── 渲染进程 B：~100-200MB
  └── 渲染进程 C：~100-200MB
  （注：多个渲染进程共享 Chromium 基础内存，实际增量远小于独立应用）

内存压力时：
  → 自动挂起最久未使用的后台插件窗口
  → 释放其渲染进程内存
  → 用户重新切换时自动恢复
```

## 五、跨平台策略

### 5.1 桌面端：Electron

三大桌面平台（Windows、macOS、Linux）使用 Electron：
- 一套代码，三平台运行
- PAL 适配器处理平台差异
- 系统集成通过 Electron API + 原生模块

### 5.2 移动端：预留架构

当前阶段专注桌面端，但架构设计预留移动端扩展能力：

```
薯片内核（TypeScript）
├── 桌面端运行时：Electron（主进程 = Node.js）
├── 移动端运行时：Capacitor（未来，主进程 = Native Bridge）
└── Web 端运行时：ServiceWorker + 浏览器 API（未来）
```

**关键设计**：
- 内核和基础服务的代码不依赖 Electron 特有 API
- 平台相关的代码全部在 PAL 适配器中
- 插件应用是纯 Web 代码，天然跨平台
- 桥接 API 的接口定义与平台无关，不同运行时提供不同的实现

### 5.3 全平台一致性保证

```
插件开发者写一次代码
  → 使用 chips.* Bridge API（平台无关）
  → 桌面端：Electron 主进程的 PAL 适配器处理
  → 移动端：Capacitor 原生桥的 PAL 适配器处理（未来）
  → Web 端：浏览器 API 的 PAL 适配器处理（未来）

所有平台：
  → 相同的 Bridge API 接口
  → 相同的主题和设计系统
  → 相同的组件库
  → 相同的插件格式
```

## 六、操作系统能力接管策略

### 6.1 逐步接管

薯片底层不是一次性接管所有系统能力，而是分阶段逐步扩展：

**第一阶段（核心能力）**：
- 文件管理（通过标签文件系统替代系统文件管理器）
- 文件关联（.card/.box 文件的默认处理）
- 剪贴板集成（跨薯片应用的剪贴板共享）
- 通知推送
- 系统托盘

**第二阶段（生产力能力）**：
- 全局搜索（薯片内容的系统级搜索集成）
- 全局快捷键
- 文件预览（系统文件预览集成）
- 自动化（类似 macOS Shortcuts / Windows Power Automate）

**第三阶段（深度集成）**：
- 桌面壁纸接管（Windows）
- 桌面小组件
- 系统主题同步
- 跨设备同步（通过薯片云）

### 6.2 能力注册架构

所有系统能力通过统一的注册机制管理：

```typescript
interface PlatformCapability {
  id: string;                    // 'fs.read', 'desktop.wallpaper', ...
  category: string;              // 'core' | 'extended' | 'reserved'
  platforms: string[];           // ['windows', 'macos', 'linux']
  status: 'active' | 'available' | 'reserved';
  adapter: PlatformAdapter;      // 具体实现
}

// 注册新能力（后续可以随时添加）
platformManager.registerCapability({
  id: 'desktop.wallpaper',
  category: 'extended',
  platforms: ['windows'],
  status: 'available',
  adapter: new WindowsWallpaperAdapter()
});
```

这种注册机制确保了：
- 现有能力可以被枚举和发现
- 新能力可以随时通过插件添加
- 插件可以查询能力是否可用再决定行为
- 不同平台的能力差异被优雅处理
