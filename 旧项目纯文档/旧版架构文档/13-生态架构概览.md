## 一、架构层次

薯片生态采用**12层架构**，从底层到顶层：

```
┌─────────────────────────────────────────────────────────┐
│                   第六层：插件应用                         │
│   编辑引擎 / 查看器 / 设置面板 / 资源管理器等              │
│   每个应用是 Web 页面，运行在独立的 BrowserWindow 中       │
├─────────────────────────────────────────────────────────┤
│                第五层：插件桥接层（Bridge API）             │
│   通过 contextBridge 暴露 window.chips.* 标准接口         │
│   是插件访问系统能力的唯一通道                              │
├─────────────────────────────────────────────────────────┤
│            第四层：基础服务层 + 共享功能模块                 │
│   核心服务（14类，运行在主进程中）                          │
│   共享模块（视频播放器、图片查看器等，独立模块插件）          │
├─────────────────────────────────────────────────────────┤
│                   第三层：薯片内核                         │
│   TypeScript，运行在 Electron 主进程中                    │
│   消息路由器 + 服务注册表 + 事件总线 + 生命周期管理器       │
├─────────────────────────────────────────────────────────┤
│               第二层：平台抽象层（PAL）                     │
│   适配 Windows / macOS / Linux 差异                      │
│   向上提供统一的平台能力接口                               │
├─────────────────────────────────────────────────────────┤
│                第一层：Electron 宿主                       │
│   Chromium 引擎 + Node.js 运行时                         │
│   唯一的原生二进制程序                                    │
└─────────────────────────────────────────────────────────┘
```

## 二、核心组件

### 2.1 薯片主机（Chips Host）

- **定位**：整个生态唯一的 Electron 应用程序，长期在用户设备后台运行
- **包含**：Electron 宿主 + PAL + 内核 + 基础服务 + 窗口管理 + 插件管理 + Bridge
- **特点**：操作系统只识别这一个软件，所有上层应用以插件形式运行在主机内部

### 2.2 薯片内核

- **语言**：TypeScript
- **运行环境**：Electron 主进程（Node.js）
- **职责**：中心路由、服务注册、事件总线、生命周期管理
- **特点**：
  - 所有服务间通信必须经过内核路由
  - 模块完全解耦，禁止直接通信
  - 支持按需加载、健康检查、异常恢复

### 2.3 平台抽象层（PAL）

- **职责**：适配不同操作系统的差异，向上提供统一接口
- **设计**：每类系统能力（文件系统、窗口、通知、剪贴板等）都有抽象接口和多平台适配器
- **原则**：内核和服务层代码不直接调用平台特定 API，一律通过 PAL

### 2.4 基础服务层

- **语言**：TypeScript
- **运行环境**：Electron 主进程
- **核心服务**（14 类）：
  - **文件处理**：文件服务、ZIP 服务、序列化服务（YAML/JSON）
  - **业务服务**：卡片服务、箱子服务、资源管理服务
  - **系统服务**：配置管理、日志系统、多语言系统
  - **管理服务**：主题管理、标签数据库、凭证管理
  - **平台服务**：封装 PAL 能力为路由可调用服务
  - **模块加载**：管理共享功能模块的发现和分发
- **共享功能模块**（作为模块插件独立安装）：
  - 视频播放器、音频播放器、图片查看器、3D 渲染器
  - 富文本编辑器、Markdown 渲染器、代码高亮器、文本编辑器
  - 各自独立仓库，可自由替换和升级

### 2.5 插件桥接层（Bridge API）

- **机制**：通过 Electron 的 contextBridge 向渲染进程注入 `window.chips.*` 接口
- **职责**：是插件访问系统能力的唯一通道
- **包含**：`chips.invoke()`（通用调用）、`chips.on()`（事件订阅）、`chips.window.*`（窗口控制）、`chips.dialog.*`（对话框）、`chips.plugin.*`（插件信息）等

### 2.6 SDK 层（Chips-SDK）

- **语言**：TypeScript
- **职责**：Bridge API 的类型安全封装，为插件开发者提供友好的开发接口
- **核心模块**：
  - CardAPI / BoxAPI / FileAPI（业务 API 封装）
  - PluginManager / ThemeManager（管理功能封装）
  - RendererEngine（卡片渲染辅助）
  - React Composables（useTheme、useI18n、useCard 等）
  - Card Runtime Protocol（`chips-card-runtime`）：卡片 iframe 与宿主之间的 postMessage 通信协议，支持消息类型 `ready | init | bridge-request | bridge-response | resource-request | resource-response | resize | theme-change | language-change`

> **全链路事件说明**：SDK 的 `onThemeChanged` / `onLanguageChanged` 钩子底层订阅内核事件总线的 `theme.changed` / `language.changed` 事件。当 Host 执行 `theme.apply` 后，事件经内核总线 → event-forwarder → 渲染进程 IPC → SDK 钩子 → 应用回调。对于卡片 iframe 场景，宿主应用（viewer/editor）收到回调后通过 card-runtime protocol 的 `theme-change` / `language-change` 消息广播给所有已初始化的卡片实例。

### 2.7 生态设置面板（Chips Ecosystem Settings）

- **技术栈**：React（Web 应用插件）
- **职责**：设备侧生态环境运维治理、系统级设置管理
- **运行方式**：作为标准应用插件运行在薯片主机的 BrowserWindow 中
- **核心能力**：
  - **运行治理**：进程管理、健康检查、日志报告
  - **系统设置**：语言、主题、插件、文件系统策略
  - **插件管理**：安装、更新、停用插件
  - **协作桥接**：生态内外文件交换与策略统一落地

## 三、中心路由原则

这是整个架构最核心的设计原则：

```
主进程内服务间通信：
  服务A ──invoke──→ [内核路由器] ──转发──→ 服务B
                    ↑             ↓
                  响应 ←───────── 处理结果

插件（渲染进程）调用服务：
  插件 ──chips.invoke()──→ [Bridge/IPC] ──→ [内核路由器] ──→ 服务
                            ↑                              ↓
                          响应 ←──────── [IPC] ←────────── 处理结果
```

**关键点**：
1. 主进程内服务之间不能直接 import 调用，必须通过 `kernel.router.invoke()`
2. 插件只能通过 Bridge API（`window.chips.*`）访问系统能力
3. 内核负责权限验证和安全控制
4. 所有通信都会被记录到日志

├── chips-component-library/       ← 无头组件库（Zag.js + Ark UI）

## 五、运行模式

采用**单一主机模式**。薯片主机是唯一安装在系统中的 Electron 应用，长期后台运行。所有上层软件以 `.cpk` 格式的插件安装到主机中。

不提供“独立可执行应用打包”模式。任何产品形态都必须通过插件安装到主机，由主机统一加载与治理。

每个安装的应用插件在操作系统中创建独立的快捷方式。用户双击快捷方式时，快捷方式指向主机并带上启动参数，主机创建 BrowserWindow 加载对应插件。对用户来说，体验与使用独立软件完全一致。

多个插件共享同一个 Chromium 引擎和主进程服务，系统资源占用远少于独立运行多个 Electron 应用。每个插件的渲染进程互相隔离，一个插件崩溃不影响其他插件。

插件分为五种类型：
- **应用插件**（App Plugin）：完整应用，拥有独立窗口
- **卡片插件**（Card Plugin）：基础卡片类型的渲染和编辑组件
- **布局插件**（Layout Plugin）：箱子的布局展示方式
- **模块插件**（Module Plugin）：共享功能模块，可被多个插件加载使用
- **主题插件**（Theme Plugin）：CSS 样式主题包
