## 组件库开发现状（React基线）

### 当前事实

- 组件库主包已是 React 基线：`@chips/component-library`，依赖 `@ark-ui/react`。
- 当前导出组件与能力（13 组件 + theme/i18n/config/adapters/telemetry）已可支撑基础交互，但距离阶段四定义的首版组件广度仍有缺口。
- 组件库内部主题运行时为 `ThemeProvider + Token stack`（React Context），可以把 token 转换为 CSS Variables 下发到子树。

### 与阶段目标差异

- 阶段四计划定义：`@chips/components` + Vue 3 + 24 组件首批清单 + 每组件至少 5 测试。
- 实际现状：React 主包 + 13 组件 + 多个测试文件低于 5 用例（如 `tabs.test.tsx:1`、`input.test.tsx:2`、`checkbox.test.tsx:2`）。

## 主题系统开发现状（Token -> Theme包 -> 运行时）

### 1) Token 与主题包层

- `chips-theme-default` 已形成 `tokens/global.css -> animations/icons/components -> theme.css` 聚合入口。
- default 主题已提供颜色、半透明层、阴影、圆角、动效时序等 token，支持现代简约风的基础表达。
- 主题包接口点围绕 `data-scope + data-part + CSS变量`，具备形状/透明度/阴影/描边/动效的可配置入口。

### 2) Host 主题运行时层

- ThemeService 已注册：`list/get/getCSS/getAllCss/getCurrent/setCurrent/install/uninstall/resolve`。
- `getAllCss` 会分层返回 `tokens/components/animations/icons` 聚合结果，供应用一次性注入。
- Control Plane 侧暴露 `theme.apply`，内部通过路由转发到 `theme.setCurrent` 并写入 `config theme.global`。

### 3) 主题契约质量现状

- default 主题 `manifest.yaml` 存在重复 `theme` 字段（结构质量问题）。
- default 校验脚本对“多余组件”严格 fail；obsidian 仅做存在性检查；macaron 强制 24 组件。三套质量门禁口径不一致。

## 前端架构逻辑（UI -> SDK -> Bridge -> Host）

### 逻辑链路

1. UI 调用 SDK 或直接 `window.chips.invoke`。
2. SDK/Bridge 将请求序列化到 `chips:invoke` IPC。
3. Host kernel router 在主进程分发到对应服务（theme/control-plane 等）。
4. 服务返回标准化结果，前端按契约更新状态并注入 CSS。

### 当前技术现实

- `chips-app-viewer`：React + `@chips/component-library` 直连。
- `chips-app-editor`：Vue + `@chips/components`（通过 shim alias 到 `src/shims/chips-components`）。
- `chips-app-settings`：Vue 与 React 双入口并存（`main.ts` 与 `main.tsx`）。
- `chips-template-app`：仍是 Vue 模板，并继续引用 `@chips/components`。

## 关键链路（主题切换/渲染/i18n/错误）

### 1) 主题切换链路

- settings/template 侧：`theme.apply`（控制平面） -> Host `theme.setCurrent` -> `config theme.global`。
- CSS 刷新：`theme.getAllCss` -> 应用侧按层注入。

### 2) 主题渲染链路

- 组件侧：`data-scope + data-part` 作为主题样式挂点。
- 主题侧：`theme.css` 统一导入 tokens/animations/icons/components。

### 3) i18n链路

- 模板/settings 主流程已大量使用 i18n key。
- viewer 局部仍存在硬编码中英文文案，破坏“零硬编码文本”约束。

### 4) 错误链路

- ThemeService 基本按标准错误结构抛错（`code/message/details`）。
- 应用层存在历史 fallback 分支与 legacy action 回退，表明契约尚未完全收敛。

## 契约一致性与分裂点

1. **路由动作分裂**
   - 标准目标强调 `theme.apply / theme.getCurrent / theme.getAllCss / theme.resolve`。
   - Host ThemeService主路由使用 `setCurrent`；Control Plane 再包装 `apply`。

2. **主题层级字段分裂**
   - 标准文档与 Foundation/SDK存在 `compositeCard`。
   - Host `ThemeResolver` 链路没有 `compositeCard` 字段。

3. **返回结构分裂**
   - 标准 `getCSS` 口径为 `Promise<{ css: string }>`。
   - SDK `use-theme`/BridgeThemeAPI 当前仍以 `Promise<string>` 使用。

4. **包名分裂**
   - 规划与大量 Vue 代码使用 `@chips/components`。
   - React组件库主包是 `@chips/component-library`。

## 重复实现与复杂度热点

1. **双框架重复实现热点**
   - settings/template 同时维护 Vue 与 React（或历史残留），组件调用与状态逻辑重复。
2. **shim 复杂度热点**
   - editor 通过 250 行 shim 人工模拟组件库，长期会形成语义漂移。
3. **主题校验重复且不一致**
   - 各主题仓库分别维护 validate.js，策略不统一。
4. **主题事件语义分叉**
   - SDK内部常用 `theme:changed`，Bridge/UI常监听 `theme.changed`，语义边界模糊。

## 第三方接入复杂度分析

### 当前接入门槛

- 新开发者需要同时理解：Theme包契约、Host路由动作、SDK差异、框架选择与包名差异。
- 当选择 Vue 时，还要额外理解 shim 口径；当选择 React 时，要面对部分文档仍是 Vue 叙述。
- 结果是“能跑起来”与“长期可维护”之间存在明显落差。

### 对第三方最敏感的复杂点

- 接口命名不统一（`apply` vs `setCurrent`，`theme.changed` vs `theme:changed`）。
- 模板与官方推荐栈不统一（模板仍偏 Vue + `@chips/components`）。
- 组件库覆盖不足时，第三方会自行补组件，进一步加剧生态分叉。
