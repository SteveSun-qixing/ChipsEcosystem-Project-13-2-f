## 北极星架构（React + Ark UI + Style Dictionary + 声明式框架层）

目标是形成“单一前端技术主线 + 可扩展主题内核 + 统一桥接协议”的长期架构：

1. **UI主线**：React 作为官方前端基线。
2. **无头组件内核**：Ark UI（基于 Zag 状态机）承担行为与可访问性。
3. **设计令牌流水线**：Style Dictionary 统一 token 源、构建与多端输出。
4. **声明式框架层（新建）**：在组件库之上提供 `View/Stack/Grid/Form/List` 等语义原语，统一布局与系统能力组合方式。
5. **桥接分层**：`UI Hook层 -> Runtime Client层 -> Bridge传输层`，隔离 UI 与协议细节。

该北极星满足当前架构原则：插件仍通过 Bridge 接入，前端保持“上层薄壳、能力下沉到底层服务”。

## 开发阶段路线（阶段A / 阶段B / 阶段C）

## 阶段A：统一基础面（协议、包名、分层）

目标：先消除“同名不同义”和“双轨并存”。

- 收敛主题能力统一入口（见“目标接口与类型面”）。
- 收敛组件包命名与导出面（停止 `@chips/components` 与 `@chips/component-library` 双口径并存）。
- 收敛桥接客户端分层，抽离通用 Runtime Client，去掉多仓重复桥接胶水。
- 收敛主题层级字段语义（`compositeCard` 是否保留必须一次性定版并贯穿 Host/SDK/文档）。

## 阶段B：声明式UI层与新主题系统（研究设计）

目标：把“组件集合”升级为“可编排的声明式 UI 系统”。

- 构建声明式原语：`View`、`Stack`、`Grid`、`Form`、`List`。
- 建立语义布局/间距/交互状态协议，减少业务侧直接拼装低层组件。
- 引入新主题系统：
  - token 分层与别名策略统一（global/semantic/component）；
  - 主题契约自动生成与自动验收；
  - `theme.contract.get` 作为运行时可查询契约接口。
- 把系统级能力封装为“能力组件”与 hooks（如文件对话框、主题注入、语言切换、错误边界）。

## 阶段C：生态化与第三方开发者体验强化

目标：让第三方“低认知成本接入，默认正确”。

- 提供 React 官方模板（app/card/module/theme）与单一示例链路。
- 提供约束型脚手架与合规检查（路由契约、主题契约、i18n、测试门禁）。
- 提供声明式 UI 层文档、桥接协议文档、主题设计与调试工具。
- 完成历史实现收敛（shim、双入口、重复 bridge-client）。

## 目标接口与类型面（主题/声明式UI/状态模型/桥接分层）

## 1) 主题能力目标面（统一命名）

- `theme.apply`
- `theme.getCurrent`
- `theme.getAllCss`
- `theme.resolve`
- `theme.contract.get`

说明：以上五个动作作为唯一对外主题运行时面；任何旧动作（如 `setCurrent`）只保留内部兼容，不再作为前端业务入口。

## 2) 声明式 UI 原语

- `View`
- `Stack`
- `Grid`
- `Form`
- `List`

说明：原语层只承载结构语义与状态编排，不承载视觉；视觉仍由主题包注入。

## 3) 状态三层边界

- 本地状态（Local State）：组件瞬时交互状态。
- 领域状态（Domain State）：编辑器/查看器/设置面板业务状态。
- 服务状态（Service State）：来自 Host 服务的远程状态与缓存快照。

## 4) 桥接分层

- UI Hook层：面向页面和组件的调用入口。
- Runtime Client层：协议编解码、错误归一化、重试与回退策略。
- Bridge传输层：`window.chips.*` 传输与事件订阅。

## 质量门禁蓝图

1. **契约门禁**：主题动作、返回结构、层级字段、事件命名必须自动验收。
2. **组件门禁**：每个组件>=5单测；每个服务模块>=8单测；契约测试与无障碍测试强制通过。
3. **文本门禁**：零硬编码文案检查（i18n key 必须可追溯）。
4. **样式门禁**：组件库零视觉样式检查（禁止内联视觉样式与私有颜色硬编码）。
5. **主题门禁**：主题包契约一致性（manifest、components、imports、dist）统一校验器。

## 迁移与收敛原则

1. **先统一协议，再统一实现**：先锁定动作/类型/事件命名，再迁移 UI 层。
2. **去 shim 化**：shim 仅作为过渡资产，最终全部归档。
3. **模板先行**：官方模板必须先升级到北极星架构，再推动业务仓库收敛。
4. **单向依赖**：声明式层依赖组件层；组件层不反向依赖业务层。
5. **禁止新增双轨**：新能力只允许进入统一主线，不再接受 Vue/React 双实现并行扩张。

## 阶段验收标准

### 阶段A 验收

- 主题接口统一为 `theme.list/theme.apply/getCurrent/getAllCss/resolve/contract.get`。
- 桥接层完成三层化；重复 bridge-client 不再新增。
- 包名与导入路径收敛为单一标准。

### 阶段B 验收

- `View/Stack/Grid/Form/List` 在至少一个核心应用完成替换落地。
- 新主题系统契约可自动生成、自动校验、运行时可查询。
- 状态三层边界在代码结构和类型层可被静态检查。

### 阶段C 验收

- 第三方模板与脚手架默认产出“契约合规 + 测试可过”的工程。
- 历史双轨资产完成归档，主干不再依赖 shim 或旧命名。
- 生态仓库在同一接口命名体系下可互操作。
