## 2. 中心路由架构规范 ⭐

### 2.1 核心原则

**所有服务和插件的通信都必须经过薯片内核路由器**,这是整个架构最核心的设计原则。薯片内核使用 TypeScript 实现，运行在 Electron 主进程中。

```
✅ 正确: 主进程内服务A → kernel.router.invoke() → 服务B
✅ 正确: 插件(渲染进程) → window.chips.invoke() → IPC → 内核路由器 → 服务
❌ 错误: 服务A → 直接 import → 服务B
❌ 错误: 插件 → require('fs') → 直接操作文件系统
```

### 2.2 实现要求

1. **禁止直接调用**: 主进程内的服务不得直接 import 调用其他服务的方法
2. **统一接口**: 主进程内通过 `kernel.router.invoke(namespace, action, params)` 调用
3. **Bridge 隔离**: 插件（渲染进程）只能通过 Bridge API（`window.chips.invoke()`）访问系统能力
4. **异步通信**: 所有跨模块/跨进程调用必须是异步的
5. **权限验证**: 内核路由器负责验证所有调用的权限

### 2.3 示例

```typescript
// ❌ 错误示范（主进程内直接 import）
import { FileService } from '../services/file/file-service';
const result = await fileService.read('/path');

// ✅ 正确方式（主进程内通过路由器）
const result = await kernel.router.invoke('file', 'read', { path: '/path' });

// ❌ 错误示范（插件中直接使用 Node.js）
const fs = require('fs');
const data = fs.readFileSync('/path');

// ✅ 正确方式（插件中通过 Bridge API）
const data = await window.chips.invoke('file', 'read', { path: '/path' });
```

## 3. 多语言支持规范

### 3.1 强制要求

**所有界面文本和提示信息都必须支持多语言切换,不允许在代码中硬编码任何文本内容。**

**薯片生态采用系统统一管理的多语言方案**,核心原则：
- **零硬编码**：代码中不允许出现任何明文字符串
- **系统接管**：所有文本显示由系统统一管理
- **编码替换**：开发时使用词汇key，打包时替换为系统编码

> **完整规范**: 详见 [多语言系统规范](./11-多语言系统规范.md)

### 3.2 实现要求

1. **使用系统多语言API**: 所有文本通过 `t()` 函数获取
2. **开发阶段使用key**: 开发时使用人类可读的key（如 `common.save`）
3. **打包时自动替换**: 构建工具自动将key替换为系统词汇编码
4. **使用配置表**: 所有文本必须在配置表中定义，不得硬编码
5. **支持多语言**: 系统支持 zh-CN、zh-TW、en-US、ja-JP、ko-KR 等

### 3.3 示例

```typescript
// ❌ 错误: 硬编码文本
button.textContent = "保存";
alert("保存成功!");

// ✅ 正确: 开发时使用key
import { t } from '@chips/i18n';
button.textContent = t('common.save');
alert(t('message.save_success'));

// 打包后自动替换为编码
button.textContent = t('i18n.core.000003');  // 系统自动处理
alert(t('i18n.ui.100050'));
```

### 3.4 开发阶段词汇表

开发时在项目中维护 `dev_i18n.yaml`：

```yaml
# dev_i18n.yaml (开发阶段)
common:
  save: "保存"
  cancel: "取消"
  ok: "确定"
  delete: "删除"

message:
  save_success: "保存成功"
  delete_confirm: "确定要删除吗？"

error:
  network: "网络错误"
  file_not_found: "文件未找到"
```

### 3.5 系统词汇编码

打包时，构建工具会将开发用的key替换为系统词汇编码：

| 开发Key | 系统编码 | 说明 |
|--------|---------|------|
| `common.save` | `i18n.core.000003` | 核心词汇"保存" |
| `common.cancel` | `i18n.core.000002` | 核心词汇"取消" |
| `common.ok` | `i18n.core.000001` | 核心词汇"确定" |
| `message.save_success` | `i18n.ui.100050` | UI提示信息 |

**注意**: 
- 优先使用系统已有词汇编码，避免重复定义
- 系统未包含的词汇会自动分配插件编码段（i18n.plugin.xxxxxx）
- 建议通用词汇提交到系统词汇表，获取正式编码

## 4. 配置系统规范

### 4.1 强制要求

**所有配置项、路径、数值都通过配置文件管理,不允许硬编码。**

### 4.2 配置层级

配置系统采用层级结构,按优先级依次覆盖:

```
1. 默认配置 (代码中定义)
   ↓
2. 系统配置 (system.yaml)
   ↓
3. 用户配置 (user.yaml)
   ↓
4. 运行时配置 (临时覆盖)
```

### 4.3 示例

```typescript
// ❌ 错误: 硬编码配置
const maxSize = 1000;
const timeout = 5000;
const apiUrl = "https://api.example.com";

// ✅ 正确: 使用配置系统
const maxSize = Config.get('cache.max_size', 1000);
const timeout = Config.get('timeout.default', 5000);
const apiUrl = Config.get('api.base_url');
```

### 4.4 配置文件示例

```yaml
# config.yaml
cache:
  max_size: 1000
  ttl: 3600

timeout:
  default: 5000
  upload: 30000
  download: 60000

api:
  base_url: "https://api.example.com"
  version: "v1"

paths:
  data_dir: "~/.chips/data"
  plugin_dir: "~/.chips/plugins"
```

## 7. 撤销重做规范

### 7.1 强制要求

**所有用户操作都需要支持撤销和重做功能。**

### 7.2 实现要求

1. **命令模式**: 使用命令模式实现操作
2. **操作栈**: 维护操作历史栈
3. **限制大小**: 历史栈有最大大小限制
4. **快捷键**: 支持Ctrl/Cmd+Z撤销,Ctrl/Cmd+Shift+Z重做

### 7.3 示例

```typescript
interface Command {
  execute(): Promise<void>;
  undo(): Promise<void>;
  redo(): Promise<void>;
}

class AddCardCommand implements Command {
  constructor(
    private cardId: string,
    private cardData: any
  ) {}
  
  async execute() {
    await CardManager.add(this.cardId, this.cardData);
  }
  
  async undo() {
    await CardManager.delete(this.cardId);
  }
  
  async redo() {
    await this.execute();
  }
}

// 使用
const command = new AddCardCommand('card123', cardData);
await CommandManager.execute(command);

// 撤销
await CommandManager.undo();

// 重做
await CommandManager.redo();
```

## 8. 错误处理规范

### 8.1 强制要求

**错误处理使用标准化的错误代码,并提供友好的提示信息。**

### 8.2 错误代码格式

```
[分类]-[编号]

示例:
- VAL-1001: 验证错误
- PERM-2001: 权限错误
- RES-3001: 资源错误
- SYS-9001: 系统错误
```

### 8.3 错误处理示例

```typescript
// 定义错误类
class ChipsError extends Error {
  constructor(
    public code: string,
    message: string,
    public details?: any
  ) {
    super(message);
    this.name = 'ChipsError';
  }
}

// 抛出错误
if (!card) {
  throw new ChipsError(
    'RES-3001',
    t('error.card_not_found'),
    { card_id: cardId }
  );
}

// 捕获错误
try {
  await operation();
} catch (error) {
  if (error instanceof ChipsError) {
    Logger.error(`Error ${error.code}: ${error.message}`, error);
    UI.showError(error.message);
  } else {
    Logger.error('Unknown error', error);
    UI.showError(t('error.unknown'));
  }
}
```

## 12. 前后端分离规范

### 12.1 强制要求

**前端和后端必须完全分离,前端只负责界面展示,后端只负责业务逻辑。**

### 12.2 职责划分

**前端职责**:
- 界面渲染
- 用户交互
- 数据展示
- 表单验证(辅助)

**后端职责**:
- 业务逻辑
- 数据处理
- 数据验证(主要)
- 权限控制

**禁止前端**:
- 包含业务逻辑
- 直接操作数据库
- 直接访问文件系统
- 直接调用其他模块

**禁止后端**:
- 包含UI代码
- 操作DOM
- 包含样式代码

## 16. 安全规范

### 16.1 输入验证

**所有外部输入都必须验证**:
```typescript
// ✅ 正确
function processCardId(cardId: string) {
  // 验证格式
  if (!/^[0-9a-zA-Z]{10}$/.test(cardId)) {
    throw new ChipsError('VAL-1002', 'Invalid card ID');
  }
  
  // 继续处理
}
```

### 16.2 路径安全

**防止路径遍历攻击**:
```typescript
// ❌ 危险
const filePath = userInput;
fs.readFile(filePath);

// ✅ 安全
const safePath = path.join(baseDir, path.basename(userInput));
if (!safePath.startsWith(baseDir)) {
  throw new Error('Invalid path');
}
fs.readFile(safePath);
```

### 16.3 HTML注入防护

```typescript
// ❌ 危险
element.innerHTML = userInput;

// ✅ 安全
element.textContent = userInput;
// 或使用DOMPurify清理
element.innerHTML = DOMPurify.sanitize(userInput);
```

## 18. 版本管理规范

### 18.1 语义化版本

版本号格式: `主版本.次版本.修订号`

- **主版本**: 不兼容的API变更
- **次版本**: 向后兼容的功能新增
- **修订号**: 向后兼容的问题修正

### 18.2 变更记录

```markdown
# Changelog

## [1.2.0] - 2026-01-30

### Added

### Fixed
- 修复内存泄漏问题

## [1.1.0] - 2026-01-15

### Added
- 初始版本
```
