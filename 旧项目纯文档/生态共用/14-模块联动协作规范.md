# 模块联动协作规范

---

## 一、通信机制

### 1.1 请求-响应模式

模块间的同步调用使用请求-响应模式，所有请求必须经过内核路由。薯片内核使用 TypeScript 实现，运行在 Electron 主进程中。

**主进程内服务间通信**（直接函数调用，延迟极低）：
```
服务A                    内核路由器                  服务B
  │                        │                        │
  │──invoke(ns,action)──→│                        │
  │                        │──验证&路由────────→│
  │                        │                        │──处理
  │                        │←──返回结果──────────│
  │←──转发结果──────────│                        │
```

**插件（渲染进程）调用服务**（通过 Bridge API + IPC）：
```
插件(渲染进程)          Bridge/IPC          内核路由器          服务
  │                        │                  │                │
  │──chips.invoke()──→│                  │                │
  │                        │──IPC──────→│                │
  │                        │                  │──路由────→│
  │                        │                  │                │──处理
  │                        │                  │←──结果────│
  │                        │←──IPC──────│                │
  │←──Promise resolve──│                  │                │
```

### 1.2 事件发布订阅模式

模块间的异步通知使用事件总线：

```typescript
// 发布事件
eventBus.emit('card:created', { cardId: 'xxx' });

// 订阅事件
eventBus.on('card:created', (data) => {
  // 处理事件
});
```

**特点**：
- 发布者和订阅者互不依赖
- 并发向所有订阅者分发
- 订阅者异常不影响其他订阅者

## 二、各层协作方式

### 2.1 应用层 → SDK 层

应用通过 SDK 提供的 API 访问生态能力：

```typescript
import { ChipsSDK } from '@chips/sdk';

const sdk = new ChipsSDK();
await sdk.initialize();

// 使用 SDK API
const card = await sdk.card.read(cardId);
await sdk.card.update(cardId, newData);
```

### 2.2 SDK 层 → 公共基础层

SDK 内部调用 Foundation 的模块：

```typescript
// SDK 通过 CoreConnector 与内核通信
const response = await connector.request({
  module: 'foundation.file',
  action: 'read',
  params: { path: '/path/to/file' }
});
```

### 2.3 插件 → SDK 层

插件通过 PluginContext 访问 SDK 能力：

```typescript
const plugin: PluginRegistration = {
  id: 'my-plugin',
  activate: (ctx: PluginContext) => {
    // 注册命令
    ctx.registerCommand('my-command', handler);
    
    // 注册渲染器
    ctx.registerRenderer('my-card-type', renderer);
    
    // 订阅事件
    ctx.on('card:updated', handleUpdate);
  }
};
```

## 三、渲染器协作

### 3.1 渲染流程

```
卡片数据 → RendererEngine → 获取渲染器 → 创建上下文 → 应用主题 → 渲染输出
```

### 3.2 基础卡片渲染

基础卡片（26 种）的渲染器在 Foundation 中统一实现，SDK 的 RendererEngine 负责调度：

```typescript
// Foundation 提供基础卡片渲染器
const textCardRenderer: CardRenderer = {
  name: 'text-card-renderer',
  supportedTypes: ['rich-text', 'plain-text', 'markdown'],
  render: async (context) => {
    // 渲染逻辑
    return { html, css };
  }
};
```

### 3.3 插件扩展渲染

插件可以注册自定义渲染器：

```typescript
ctx.registerRenderer('custom-card-type', {
  render: async (context) => {
    return { html, css };
  }
});
```

## 四、资源访问协作

### 4.1 统一资源标识

所有资源通过统一的资源标识符访问：

```
chips://local/card/abc123/content    # 本地卡片内容
chips://network/image/xyz.png        # 网络图片
chips://embed/video/data             # 嵌入资源
```

### 4.2 资源加载链

```
应用请求资源 → SDK.ResourceManager → CoreConnector → Foundation 文件模块
                                                          ↓
返回资源数据 ← 缓存（如有）←────────────────────────── 读取/下载
```

## 五、主题协作

### 5.1 主题应用层级

```
全局主题（Foundation） → SDK 主题引擎 → 组件主题变量 → 渲染输出
```

### 5.2 主题变量传递

主题引擎将主题变量注入到渲染上下文：

```typescript
const themeVariables = themeManager.getCurrentVariables();
const renderContext = {
  theme: themeVariables,
  card: cardData,
  // ...
};
```

## 六、模块注册与发现

### 6.1 模块注册

模块启动时向内核注册：

```yaml
# module.yaml
id: "my-module"
version: "1.0.0"
routes:
  - path: "my-module.action1"
    handler: "handleAction1"
  - path: "my-module.action2"
    handler: "handleAction2"
dependencies:
  - "@chips/foundation": "^1.0.0"
```

### 6.2 路由发现

内核维护路由表，查找目标模块：

```
请求路径: "card.read" → 查找路由表 → 目标模块: "Chips-SDK.CardAPI"
```

## 七、协作原则

1. **禁止直接依赖**：模块间只能通过内核路由或事件总线通信
2. **接口统一**：所有模块使用统一的请求响应格式
3. **状态隔离**：每个模块维护自己的状态，不共享内存
4. **错误传播**：错误通过标准格式返回，不抛出跨模块异常
5. **版本兼容**：通过语义化版本确保模块间兼容性
