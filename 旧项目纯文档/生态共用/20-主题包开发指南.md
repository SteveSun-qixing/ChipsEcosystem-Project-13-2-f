## 1. 目标

主题包负责提供视觉层，不承载业务逻辑。组件逻辑由无头组件库提供，样式由主题包注入。

## 2. 主题包目录建议

```text
my-theme/
├── manifest.yaml
├── contracts/
│   └── theme-interface.contract.json
├── tokens/
│   ├── ref.json
│   ├── sys.json
│   ├── comp.json
│   ├── motion.json
│   ├── layout.json
│   └── global.css
├── components/
│   ├── button.css
│   ├── input.css
│   └── ...
├── icons/
│   ├── icons.css
│   └── svg/
├── animations/  # 或 motions/
│   └── *.css
└── dist/
    ├── tokens/
    ├── components/
    ├── icons/
    └── animations/
```

---

## 3. 关键规范

1. 组件保持零样式，主题包只负责样式注入  
2. Token 通过 Style Dictionary 构建，不手写重复产物  
3. 样式命名与组件 CSS 接口点保持一致  
4. 不在主题层引入业务耦合样式  
5. 主题插件必须声明兼容性字段
6. 一个主题包只表达一种主题语义（不在包内区分白天/夜间标签）
7. `manifest.theme` 仅允许 `tokensVersion`、`inherits` 字段
8. 组件变量命名仅允许 `--chips-sys-*`、`--chips-comp-*`、`--chips-layout-*`、`--chips-motion-*`
9. `theme` 类型插件豁免 `ui.layout` 字段（仅页面型插件强制）

---

## 4. manifest 核心字段（theme）

```yaml
schemaVersion: "1.0.0"
id: "my-publisher.my-theme"
name: "i18n.plugin.600001"
version: "1.0.0"
type: "theme"
publisher: "my-publisher"

entry:
  tokens: "tokens/global.css"
  themeCss: "theme.css"

theme:
  tokensVersion: "1.0.0"
  inherits: "chips-official.default-theme"

compatibility:
  host: ">=1.0.0"
  platform:
    - "win32"
    - "darwin"
    - "linux"
```

> 入口路径规范：`entry.tokens` 与 `entry.themeCss` 必须使用包根相对路径，禁止使用 `dist/` 前缀。
> 主题模式规范：主题清单不再使用白天/夜间标签字段；运行时通过 `themeId` 切换不同主题包。
> 阶段06口径：`theme` 插件不要求 `ui.layout` 字段。

## 5. 构建流程

```bash
pnpm install
pnpm build          # 触发 Style Dictionary 与主题产物构建
chipsd validate
chipsd pack
```

## 6. 验收清单

1. Token 五层输出完整（`ref/sys/comp/motion/layout`）  
2. 组件样式覆盖所需接口点  
3. 图标/动画资源可被正确加载  
4. Host 与独立调试模式样式表现一致  
5. 主题切换后组件视觉立即生效
6. `pnpm build`、`pnpm validate`、`pnpm test`、`chipsd validate` 全绿

## 7. 常见问题

### 7.1 样式未生效

优先检查：

- 是否只构建了 token，未产出组件 CSS  
- 主题入口路径与 manifest 是否一致  
- Host 场景下是否完成 CSS 注入链路

### 7.2 组件视觉错位

优先检查：

- 组件接口类名是否与主题 CSS 对齐  
- 是否误写业务特定样式覆盖了主题基础样式

## 8. 主题全链路事件流

主题包开发者需要了解主题在运行时的完整生效链路，以便排查"切换主题后样式未更新"等问题。

```
用户切换主题
  → Host theme-service.handleApply(themeId)
    → cssCache.clear()
    → emitThemeChanged(themeId)          // 发布 theme.changed 事件（propagate: true）
      → 内核事件总线
        → event-forwarder（通配符 '*' 订阅）
          → 所有渲染进程窗口（IPC）
            → SDK onThemeChanged 回调
              → viewer/editor 获取新 CSS（theme.getAllCss）
                → card-runtime protocol 广播 theme-change 消息
                  → 每个卡片 iframe 接收并应用新样式
```

关键点：
1. `theme.apply` 会清除 Host 侧 CSS 缓存并发布 `theme.changed` 事件
2. 事件通过 `propagate: true` 传播到所有渲染进程
3. 卡片 iframe 通过 `theme-change` postMessage 接收更新，无需自行订阅 Bridge 事件
4. 主题包的 CSS 产物质量直接影响全链路末端的渲染效果
