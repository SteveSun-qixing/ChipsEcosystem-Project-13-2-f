## 一、仓库架构

### 1.1 整体结构

```
Project-12/                          ← 超级项目（开发聚合用）
│
├── [Submodule] 核心层
│   ├── Chips-Host                   → 薯片主机 (Electron + TypeScript)
│   ├── Chips-SDK                    → SDK (TypeScript)
│   └── chips-component-library      → 无头组件库 (Ark UI + Vue 3)
│
├── [Submodule] 应用插件
│   ├── chips-app-editor             → 编辑引擎
│   ├── Chips-Viewer                 → PotatoEcosystem-Viewer
│   ├── Chips-TagEditor              → PotatoEcosystem-TagEditor
│   └── Chips-FileConnector          → PotatoEcosystem-FileConnector
│
├── [Submodule] 插件
│   └── BasicCardPlugin/Rich-Text-Basic-Card-Plugin
│                                    → PotatoEcosystem-RichText-BCP
│
├── [Submodule] 待开发项目
│   ├── Chips-AppMaker             → PotatoEcosystem-AppMaker
│   ├── Chips-CommunityServer      → PotatoEcosystem-CommunityServer
│   └── Chips-ResourceManager      → PotatoEcosystem-ResourceManager
│
├── 生态共用/                    ← 共享规范
├── 生态设计原稿（一切标准）/     ← 设计文档
└── ProductFinishedProductTestingSpace/  ← 测试空间
```

### 1.2 Submodule 说明

Submodule 是独立的 Git 仓库，拥有：
- 独立的版本控制
- 独立的 GitHub 远程仓库
- 独立的 `main` 和 `develop` 分支

主仓库记录每个 Submodule 的特定 commit，确保版本一致性。

## 二、分支规范

### 2.1 分支模型

```
main ─────────────────────────────────────────→ (生产)
  │
  └── develop ───────────────────────────────→ (开发主线)
        │
        ├── feature/功能名 ──→ (合并后归档)
        │
        └── feature/另一功能 ──→ (合并后归档)
```

### 2.2 分支说明

| 分支 | 生命周期 | 用途 | 合并后处理 |
|------|---------|------|-----------|
| `main` | 永久 | 生产环境代码 | 只接受 PR 合并 |
| `develop` | 永久 | 开发集成分支 | 日常开发基础 |
| `feature/*` | 开发期 | 新功能开发 | 归档到 `archive/feature/*` |
| `release/*` | 发布期 | 版本发布准备 | 归档到 `archive/release/*` |
| `hotfix/*` | 修复期 | 紧急修复 | 归档到 `archive/hotfix/*` |
| `archive/*` | 永久 | 历史归档 | 只读，不修改 |

### 2.3 命名规范

```bash
# 功能分支
feature/add-user-auth
feature/optimize-search

# 发布分支
release/1.0.0
release/2.1.0

# 修复分支
hotfix/fix-login-crash
hotfix/security-patch
```

## 三、日常开发流程

### 3.1 开始新功能

```bash
# 1. 进入目标项目（主仓库或子模块）
cd Chips-SDK

# 2. 确保在最新的 develop 分支
git checkout develop
git pull origin develop

# 3. 创建功能分支
git checkout -b feature/新功能名

# 4. 开发...编码...

# 5. 提交代码
git add .
git commit -m "feat: 功能描述"

# 6. 推送到远程
git push -u origin feature/新功能名
```

### 3.2 提交信息规范

```
<类型>: <简短描述>

[可选的详细描述]

[可选的关联 issue]
```

**类型列表：**

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | feat: 添加用户登录功能 |
| `fix` | 修复 | fix: 修复搜索结果为空的问题 |
| `docs` | 文档 | docs: 更新 API 文档 |
| `style` | 格式 | style: 调整代码缩进 |
| `refactor` | 重构 | refactor: 重构认证模块 |
| `test` | 测试 | test: 添加单元测试 |
| `chore` | 构建/工具 | chore: 更新依赖版本 |

### 3.3 完成功能并合并

```bash
# 方式一：在 GitHub 上创建 Pull Request（推荐）
# 1. 在 GitHub 上创建 PR: feature/xxx → develop
# 2. Code Review
# 3. Merge PR

# 方式二：本地合并
git checkout develop
git pull origin develop
git merge feature/新功能名
git push origin develop

# 归档功能分支（不删除，重命名为归档）
git branch -m feature/新功能名 archive/feature/新功能名
git push origin archive/feature/新功能名
git push origin --delete feature/新功能名   # 删除远程的旧名称
```

### 3.4 同步子模块更新到主仓库

```bash
# 在子模块中完成开发并推送后，回到主仓库
cd ..

# 主仓库需要记录子模块的新版本
git add Chips-SDK
git commit -m "update: Chips-SDK 功能更新"
git push
```

## 四、Submodule 操作

### 4.1 克隆仓库（含子模块）

```bash
# 方式一：克隆时初始化子模块
git clone --recurse-submodules <仓库URL>

# 方式二：先克隆，再初始化
git clone <仓库URL>
cd Project-12
git submodule init
git submodule update
```

### 4.2 更新子模块

```bash
# 更新所有子模块到远程最新
git submodule update --remote --merge

# 更新单个子模块
git submodule update --remote --merge Chips-SDK

# 仅同步到主仓库记录的版本（不拉取最新）
git submodule update
```

### 4.3 在子模块中开发

```bash
# 进入子模块
cd Chips-SDK

# 切换到 develop（子模块默认可能在 detached HEAD）
git checkout develop
git pull

# 正常开发流程...
git checkout -b feature/xxx
# 编码...
git add . && git commit -m "feat: xxx"
git push -u origin feature/xxx
```

### 4.4 批量操作

```bash
# 在所有子模块执行命令
git submodule foreach 'git checkout develop'
git submodule foreach 'git pull'
git submodule foreach 'git status'
```

## 五、依赖管理

### 5.1 依赖层级架构

薯片生态采用分层依赖架构，上层模块依赖下层模块：

```
┌─────────────────────────────────────────────────────────┐
│                      应用层                              │
│  Chips-Editor、Chips-Viewer、Chips-FileConnector 等      │
└─────────────────────────────┬───────────────────────────┘
                              │ 依赖
┌─────────────────────────────▼───────────────────────────┐
│                      组件层                              │
│  Chips-ComponentLibrary、Chips-TagEditor                 │
└─────────────────────────────┬───────────────────────────┘
                              │ 依赖
┌─────────────────────────────▼───────────────────────────┐
│                      SDK 层                              │
│  Chips-SDK（提供统一 API 接口）                          │
└─────────────────────────────┬───────────────────────────┘
                              │ 依赖
┌─────────────────────────────▼───────────────────────────┐
│                    公共基础层                            │
│  Chips-Foundation（通用工具、文件处理、渲染器等）         │
└─────────────────────────────┬───────────────────────────┘
                              │ 通信
┌─────────────────────────────▼───────────────────────────┐
│                      内核层                              │
│  薯片内核（TypeScript，运行在 Electron 主进程，中心路由）   │
└─────────────────────────────────────────────────────────┘
```

### 5.2 pnpm 工作区

项目使用 pnpm 工作区管理 npm 依赖，配置文件 `pnpm-workspace.yaml`：

```yaml
packages:
  # 核心模块
  - 'Chips-SDK'
  - 'Chips-Foundation'
  
  # 应用模块
  - 'Chips-Editor'
  - 'Chips-Viewer'
  - 'Chips-FileConnector'
  
  # UI 组件
  - 'Chips-ComponentLibrary'
  - 'Chips-TagEditor'
  
  # 插件
  - 'BasicCardPlugin/Rich-Text-Basic-Card-Plugin'
  - 'Chips-CardConversionPlugin/*'
```

### 5.3 安装依赖

#### 全局安装（推荐）

```bash
# 在主仓库根目录执行，安装所有子模块依赖
pnpm install
```

#### 单个模块安装

```bash
# 进入子模块目录
cd Chips-SDK
npm install

# 或使用 pnpm
pnpm install
```

### 5.4 构建顺序

由于模块间存在依赖关系，构建时需要按顺序进行：

```bash
# 推荐：使用脚本按依赖顺序构建
./脚本库/start-foundation.sh

# 或手动按顺序构建：
# 1. 构建薯片主机（内核 + 基础服务 + Bridge）
cd Chips-Host && pnpm build

# 2. 构建 SDK
cd Chips-SDK && pnpm build

# 3. 构建组件库
cd chips-component-library && pnpm build

# 5. 构建应用（可并行）
cd Chips-Editor && npm run build
cd Chips-Viewer && npm run build
```

### 5.5 模块间依赖配置

各模块通过 `package.json` 的 `peerDependencies` 声明依赖：

```json
// Chips-Editor/package.json
{
  "peerDependencies": {
    "@chips/sdk": "^1.0.0",
    "@chips/components": "^0.1.0",
    "@chips/foundation": "^1.0.0"
  }
}
```

```json
// Chips-SDK/package.json
{
  "peerDependencies": {
    "@chips/foundation": "^1.0.0"
  }
}
```

### 5.6 添加新依赖

```bash
# 添加生产依赖
cd Chips-SDK
pnpm add lodash

# 添加开发依赖
pnpm add -D vitest

# 添加工作区内的依赖（使用 workspace 协议）
pnpm add @chips/foundation@workspace:*
```

### 5.7 更新依赖

```bash
# 更新单个依赖
pnpm update lodash

# 更新所有依赖
pnpm update

# 交互式更新
pnpm update -i
```

### 5.8 依赖检查

```bash
# 检查过期依赖
pnpm outdated

# 检查重复依赖
pnpm dedupe --check

# 查看依赖树
pnpm list --depth=2
```

### 5.9 常见问题

#### 依赖版本冲突

```bash
# 清理并重新安装
rm -rf node_modules pnpm-lock.yaml
pnpm install
```

#### peerDependencies 警告

在 `.npmrc` 中配置：

```ini
auto-install-peers=true
strict-peer-dependencies=false
```

#### 子模块依赖未安装

```bash
# 确保子模块代码已更新
git submodule update --init --recursive

# 重新安装依赖
pnpm install
```

## 六、版本发布流程

### 5.1 准备发布

```bash
# 1. 从 develop 创建 release 分支
git checkout develop
git pull
git checkout -b release/1.0.0

# 2. 版本号更新、最后修复
# 修改 package.json / Cargo.toml 等版本号
git commit -m "chore: 准备 v1.0.0 发布"

# 3. 测试验证
```

### 5.2 完成发布

```bash
# 1. 合并到 main
git checkout main
git merge release/1.0.0

# 2. 打标签
git tag -a v1.0.0 -m "Release version 1.0.0"

# 3. 推送
git push origin main
git push origin v1.0.0

# 4. 合并回 develop
git checkout develop
git merge release/1.0.0
git push origin develop

# 5. 归档 release 分支
git branch -m release/1.0.0 archive/release/1.0.0
git push origin archive/release/1.0.0
git push origin --delete release/1.0.0
```

## 七、紧急修复流程

```bash
# 1. 从 main 创建 hotfix 分支
git checkout main
git checkout -b hotfix/紧急修复描述

# 2. 修复并提交
git commit -m "fix: 紧急修复描述"

# 3. 合并到 main
git checkout main
git merge hotfix/紧急修复描述
git tag -a v1.0.1 -m "Hotfix release"
git push origin main --tags

# 4. 合并到 develop（确保修复不丢失）
git checkout develop
git merge hotfix/紧急修复描述
git push origin develop

# 5. 归档 hotfix 分支
git branch -m hotfix/紧急修复描述 archive/hotfix/紧急修复描述
git push origin archive/hotfix/紧急修复描述
git push origin --delete hotfix/紧急修复描述
```

## 八、常用命令速查

### 7.1 状态查看

```bash
git status                    # 查看工作区状态
git branch -v                 # 查看所有分支
git log --oneline -10         # 查看最近10条提交
git submodule status          # 查看子模块状态
git diff --submodule          # 查看子模块变更
```

### 7.2 分支操作

```bash
git checkout <分支名>          # 切换分支
git checkout -b <新分支>       # 创建并切换
git branch -m <旧名> <新名>    # 重命名分支（用于归档）
git branch -d <分支名>         # 删除本地分支（谨慎使用）
git push origin --delete <分支> # 删除远程分支（谨慎使用）

# 归档分支（推荐方式）
git branch -m feature/xxx archive/feature/xxx
git push origin archive/feature/xxx
git push origin --delete feature/xxx
```

### 7.3 同步操作

```bash
git fetch origin              # 获取远程更新（不合并）
git pull                      # 拉取并合并
git push                      # 推送到远程
git pull --recurse-submodules # 拉取包含子模块
```

### 7.4 子模块操作

```bash
git submodule init            # 初始化子模块配置
git submodule update          # 更新到记录的版本
git submodule update --remote # 更新到远程最新
git submodule foreach '<命令>' # 批量执行命令
```

### 7.5 撤销操作

```bash
git checkout -- <文件>        # 撤销工作区修改
git reset HEAD <文件>         # 撤销暂存
git reset --soft HEAD^        # 撤销上次提交（保留修改）
git reset --hard HEAD^        # 撤销上次提交（丢弃修改）
git revert <commit>           # 创建反向提交
```

## 九、故障排除

### 8.1 子模块显示 "dirty"

```bash
# 原因：子模块有未提交的修改
# 解决：进入子模块处理

cd Chips-SDK
git status

# 方案A：提交修改
git add . && git commit -m "..."

# 方案B：丢弃修改
git checkout .
```

### 8.2 子模块处于 detached HEAD

```bash
# 原因：子模块指向特定 commit 而非分支
# 解决：切换到分支

cd Chips-SDK
git checkout develop   # 或 main
```

### 8.3 拉取后子模块版本不对

```bash
# 原因：主仓库记录了新的子模块版本，但本地未更新
# 解决：

git submodule update
```

### 8.4 子模块冲突

```bash
# 解决：更新到远程版本
git submodule update --remote --merge Chips-SDK

# 或强制重置
cd Chips-SDK
git fetch origin
git reset --hard origin/main
```

### 8.5 误操作恢复

```bash
# 查看操作历史
git reflog

# 恢复到之前的状态
git reset --hard <commit-hash>

# 查看归档的历史分支
git checkout archive/白皮书-V3
```
