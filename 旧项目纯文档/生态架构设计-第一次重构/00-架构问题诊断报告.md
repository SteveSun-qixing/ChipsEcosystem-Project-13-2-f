# 架构问题诊断报告

**编写时间**：2026-02-12
**诊断范围**：薯片生态底层架构（内核层、公共基础层、SDK层、应用层）

---

## 一、当前架构概述

当前薯片生态采用"微内核四层架构"：

```
第四层：软件产品（Editor、Viewer、AppTemplate、EcosystemSettings 等）
第三层：组件库 + SDK
第二层：公共基础层（Chips-Foundation）
第一层：薯片微内核（Chips-core，Rust）
```

每一层都是独立的包/项目，通过 pnpm workspace 管理。应用层的每个产品都是独立的 Electron 应用（Editor、Viewer、AppTemplate 等），EcosystemSettings 使用 Flutter 技术栈。

---

## 二、核心问题诊断

### 问题一：多 Electron 实例——违背"一个底层软件"目标 ⭐ 致命问题

**现状**：Editor、Viewer、AppTemplate、ImageViewer、VideoPlayer 分别是独立的 Electron 应用，各自拥有 `electron/main.ts` 入口文件和完整的 Electron 运行时。

**问题分析**：

1. **资源浪费严重**：每个 Electron 应用启动时独立加载 Chromium 内核（约 200MB RAM），Node.js 运行时（约 50MB RAM）。如果用户同时打开 3 个薯片软件，将消耗约 750MB+ 内存，仅用于重复的运行时环境。

2. **无法实现"共享内核"**：设计原稿第13篇明确描述"如果用户同时打开多个薯片软件，它们都运行在同一个内核进程中"，但独立 Electron 应用之间无法共享进程。每个应用都是操作系统级别的独立进程，无法共享已加载的模块和内存。

3. **违背核心愿景**：用户的核心愿景是"对于电脑来说，就只安装了一个薯片底层软件"，所有上层应用以插件形式运行。但当前架构中，每个产品都是需要独立安装的完整软件。

4. **操作系统限制**：部分操作系统（如 macOS 沙箱、部分 Linux 发行版安全策略）不允许一个程序启动另一个二进制程序。当前架构下，"打开另一个薯片软件"意味着启动另一个 Electron 二进制文件，在这些系统上会被拒绝。

### 问题二：Rust 微内核与 Electron 的割裂 ⭐ 严重问题

**现状**：`Chips-core` 是 Rust 编写的微内核，使用 Tokio 异步运行时。而所有应用层产品都是 Electron 应用（Node.js + Chromium）。

**问题分析**：

1. **跨语言桥接开销**：Rust 内核与 Electron 应用之间需要通过 IPC 或 N-API 桥接。每次内核调用都需要数据序列化/反序列化，增加延迟和复杂度。

2. **两套进程模型**：Rust 进程 + Electron 进程并行运行，增加了系统资源占用和进程管理的复杂度。

3. **职责错配**：内核的主要职责是消息路由和模块管理，这是 I/O 密集型任务（不是 CPU 密集型）。在 Electron 的上下文中，Node.js/TypeScript 完全能胜任这些工作，无需引入额外的 Rust 运行时。

4. **构建复杂度**：Rust 需要针对每个平台编译原生二进制文件，增加了 CI/CD 的复杂度和跨平台支持的难度。

5. **调试困难**：跨语言调试（Rust ↔ TypeScript ↔ 前端 JavaScript）极其困难，大幅增加了开发成本。

### 问题三：Flutter 技术栈割裂 ⭐ 严重问题

**现状**：`Chips-EcosystemSettings` 使用 Flutter（Dart）技术栈开发，而生态中其他所有产品都使用 Web 技术（TypeScript/Vue/React）。

**问题分析**：

1. **无法共享 UI 组件**：Flutter 的 Widget 与 Web 的组件完全不兼容，薯片组件库（@chips/components）无法在 Settings 中使用。

2. **主题系统割裂**：薯片主题系统基于 CSS 变量 Token，Flutter 使用完全不同的主题机制。Settings 面板无法与生态其他部分保持视觉一致性。

3. **额外运行时开销**：Flutter 需要 Dart VM，这是在 Electron（Chromium + Node.js）和 Rust 之外的第三个运行时，进一步增加资源消耗。

4. **违背插件化理念**：Settings 面板按照生态设计应该是一个可替换的插件/应用，但 Flutter 技术栈使它成为一个完全独立的技术孤岛。

### 问题四：Foundation 层过于单体化

**现状**：`@chips/foundation` 将视频播放器、图片查看器、Markdown 解析器、窗口管理器、拖拽系统等 30+ 个模块打包在一个 npm 包中。

**问题分析**：

1. **无法按需加载**：即使一个简单的插件只需要文件处理能力，也必须引入整个 Foundation 包。

2. **包体积膨胀**：随着模块增加，Foundation 包体积会持续增长，影响所有依赖它的项目。

3. **违背模块化原则**：设计原稿强调"极致模块化"和"按需加载"，但 Foundation 的单体结构与此矛盾。

### 问题五：构建时插件 vs 运行时插件

**现状**：基础卡片插件（BasicCardPlugin）是 npm workspace 中的包，需要在构建时通过 peerDependencies 链接。

**问题分析**：

1. **不是真正的动态插件**：插件需要在构建时解析依赖并编译，不能在运行时动态安装加载。

2. **无法实现应用市场**：用户不能从应用市场下载并即时安装一个新的基础卡片插件，必须重新构建应用。

3. **热插拔不可能**：设计原稿描述的"不重启软件即可安装、更新和卸载"在当前构建模型下无法实现。

### 问题六：缺乏统一入口

**现状**：没有一个"薯片底层软件"作为统一入口。用户面对的是多个独立的应用程序。

**问题分析**：

不符合"类似微信小程序生态"的目标。微信只有一个主应用，所有小程序在主应用内运行。当前薯片生态没有这个"主应用"。

---

## 三、根本原因

所有问题的根本原因可以归纳为一点：

> **当前架构将"软件产品"设计为独立的应用程序，而不是统一平台上的插件应用。**

这导致了：
- 多个 Electron 实例（因为每个产品是独立应用）
- Rust/Flutter 的技术割裂（因为各产品独立选择技术栈）
- Foundation 的单体化（因为它被当作所有独立应用的公共依赖）
- 构建时插件模型（因为插件被当作 npm 包而非动态加载的 Web 应用）

---

## 四、重构方向

基于以上诊断，核心重构方向是：

> **建立"薯片主机"（Chips Host）作为唯一的 Electron 应用，所有上层软件以 Web 插件的形式在主机内运行。**

这个方向将：

1. **解决多实例问题**：只有一个 Electron 进程，所有插件共享同一个 Chromium 和 Node.js 运行时。
2. **解决技术割裂问题**：内核运行在 Electron 主进程中（TypeScript），不需要 Rust 或 Flutter。
3. **实现真正的插件系统**：插件是 Web 应用，可以动态加载和卸载，真正的热插拔。
4. **实现统一入口**：薯片主机就是唯一安装在系统上的软件，用户通过它管理和使用所有插件应用。
5. **类似微信小程序**：与微信小程序生态的技术模型高度一致。
