## 一、设计哲学

薯片生态的底层架构遵循一个核心理念：

> **一个主机，万千插件。**

对于用户的计算机来说，薯片生态只表现为一个软件——**薯片主机**（Chips Host）。这是唯一安装在系统中的二进制程序。所有的编辑器、查看器、图片工具、视频工具、设置面板、开发工具等，都是运行在主机内部的**插件应用**。

这个模型与微信小程序生态高度类似：
- 微信是唯一的主程序 → **薯片主机**是唯一的主程序
- 小程序在微信内运行 → **插件应用**在薯片主机内运行
- 小程序通过微信 API 访问系统能力 → 插件通过**桥接 API** 访问系统能力
- 小程序是 Web 技术实现 → 插件应用是 **Web 技术**实现

## 二、架构层次

新架构从底层到顶层分为六层：

```
┌─────────────────────────────────────────────────────┐
│  第六层：插件应用（编辑引擎、查看器、设置面板……）       │
│  ─ 每个应用是一个 Web 页面，运行在独立的 BrowserWindow  │
├─────────────────────────────────────────────────────┤
│  第五层：插件桥接层（Bridge API）                       │
│  ─ 通过 contextBridge 暴露，是插件访问系统能力的唯一通道 │
├─────────────────────────────────────────────────────┤
│  第四层：基础服务层 + 共享功能模块                       │
│  ─ 核心服务：文件、资源、主题、多语言、配置管理等         │
│  ─ 共享模块：视频播放器、图片查看器、Markdown渲染器等     │
│  ─ 核心服务运行在主进程，共享模块由插件按需加载           │
├─────────────────────────────────────────────────────┤
│  第三层：薯片内核                                       │
│  ─ 消息路由、服务注册、事件总线、生命周期管理              │
│  ─ TypeScript，运行在 Electron 主进程中                 │
├─────────────────────────────────────────────────────┤
│  第二层：平台抽象层（PAL）                               │
│  ─ 适配不同操作系统的差异（Windows/macOS/Linux）         │
│  ─ 向上提供统一的平台能力接口                            │
├─────────────────────────────────────────────────────┤
│  第一层：Electron 宿主                                  │
│  ─ Chromium 引擎 + Node.js 运行时                      │
│  ─ 原生操作系统能力入口                                  │
└─────────────────────────────────────────────────────┘
```

### 第一层：Electron 宿主

这是整个生态唯一的原生二进制程序。基于 Electron 框架，提供：
- **Chromium 引擎**：为所有插件应用提供 Web 渲染能力
- **Node.js 运行时**：为主进程提供完整的系统访问能力
- **窗口系统**：创建和管理 BrowserWindow
- **IPC 通道**：主进程与渲染进程之间的安全通信

### 第二层：平台抽象层（PAL）

位于 Electron 宿主之上、薯片内核之下的平台适配层。它的职责是**向下适配不同操作系统，向上提供统一接口**：
- 每类系统能力（文件系统、窗口、通知、剪贴板等）都有一个抽象接口 + 多个平台适配器
- 内核和基础服务**不直接调用** `process.platform` 或任何平台特定 API，一律通过 PAL
- 覆盖 Windows、macOS、Linux 的**最大公约数**能力，同时为平台独有功能保留扩展接口
- 确保内核层和服务层代码的**平台无关性**，为未来移动端（Capacitor）扩展打下基础

详见文档 08。

### 第三层：薯片内核

运行在 Electron 主进程中的 TypeScript 模块，是整个生态的神经中枢：
- **消息路由器**：所有模块/插件间的通信都经过路由器中转
- **服务注册表**：记录所有已注册的服务及其提供者
- **事件总线**：发布-订阅事件系统
- **生命周期管理器**：管理插件和服务的启动、运行、停止

### 第四层：基础服务层 + 共享功能模块

分为两部分：

**A. 核心服务（运行在主进程中，数据型）**：
- **文件服务**：卡片/箱子打包解包、文件识别、格式转换
- **资源管理**：统一的资源访问层（本地、网络、缓存）
- **配置管理**：层级化配置系统
- **主题管理**：主题注册、查询、分发
- **多语言服务**：语言包管理和文本翻译
- **平台服务**：系统信息、能力检测、快捷键等平台能力封装
- 以及凭证管理、日志、标签数据库、序列化等

**B. 共享功能模块（作为"模块插件"独立安装，UI 型）**：
- 视频播放器、音频播放器、图片查看器、3D 渲染器
- 富文本编辑器、Markdown 渲染器、代码高亮器、文本编辑器
- 这些模块各自独立仓库，可自由替换和升级
- 由上层插件按需加载使用（类似操作系统的共享库）

服务按需懒加载，不在启动时全部初始化。

### 第五层：插件桥接层

通过 Electron 的 `contextBridge` 机制暴露给插件应用的标准 API。这是插件访问底层能力的**唯一通道**。插件不能直接访问 Node.js API、文件系统或网络，只能通过桥接 API 间接调用。

桥接 API 包括：核心调用、文件操作、卡片操作、资源访问、主题获取、窗口控制、模块加载等。

### 第六层：插件应用

每个插件应用是一个标准的 Web 应用（HTML + CSS + JavaScript），运行在独立的 BrowserWindow 渲染进程中。**上层应用是"薄壳"**，只负责 UI 布局和业务编排，所有重活（文件操作、媒体处理、资源获取等）由底层完成。插件分为五种类型：
- **应用插件**：编辑引擎、查看器、设置面板、资源管理器等
- **卡片插件**：基础卡片类型的渲染和编辑组件
- **布局插件**：箱子的布局算法和展示方式
- **模块插件**：共享功能模块（视频播放器、图片查看器等）
- **主题插件**：CSS 样式主题包

## 三、与原架构的关键变化

| 方面 | 原架构 | 新架构 |
|------|--------|--------|
| 系统中的程序数量 | 多个独立 Electron 应用 | **一个薯片主机** |
| 内核技术 | Rust 独立进程 | **TypeScript，运行在主进程** |
| 设置面板技术 | Flutter | **Web 技术（与生态统一）** |
| 上层产品形态 | 独立可执行文件 | **Web 插件应用** |
| 插件加载方式 | npm 包，构建时链接 | **Web 应用，运行时动态加载** |
| 进程模型 | 每个产品独立进程 | **共享主进程，插件独立渲染进程** |
| Foundation 形态 | 单体 npm 包 | **核心服务 + 独立仓库的模块插件** |
| 模块通信 | 跨进程 IPC | **主进程内直接调用 + IPC 到渲染进程** |


- **主进程（唯一）**：运行薯片内核、基础服务、窗口管理。是整个生态的大脑。
- **渲染进程（每个插件应用一个）**：每个打开的插件应用独占一个渲染进程。进程之间互相隔离，一个插件崩溃不会影响其他插件和主进程。
- **iframe（基础卡片渲染）**：基础卡片的渲染组件在应用插件的渲染进程内以 iframe 方式加载，不需要新的进程。

## 五、核心术语

| 术语 | 含义 |
|------|------|
| 薯片主机（Chips Host） | 唯一的 Electron 应用程序，是整个生态的容器 |
| 薯片内核（Chips Kernel） | 运行在主进程中的消息路由和服务管理核心 |
| 平台抽象层（PAL） | 适配不同操作系统差异的中间层 |
| 基础服务（Foundation Services） | 文件、资源、主题、多语言等通用服务模块 |
| 桥接 API（Bridge API） | 插件访问系统能力的标准接口 |
| 应用插件（App Plugin） | 拥有独立窗口的完整应用，如编辑引擎、查看器 |
| 卡片插件（Card Plugin） | 基础卡片类型的渲染/编辑组件 |
| 布局插件（Layout Plugin） | 箱子的布局展示插件 |
| 模块插件（Module Plugin） | 共享功能模块，如视频播放器、图片查看器 |
| 主题插件（Theme Plugin） | CSS 样式主题包 |
| 插件清单（Plugin Manifest） | 描述插件元数据和能力的 manifest.yaml 文件 |

## 六、关键设计原则

### 6.1 单一宿主原则
整个生态只有一个 Electron 二进制程序安装在系统中。没有任何插件需要独立的二进制文件。

### 6.2 Web 即插件原则
所有插件应用本质都是 Web 应用。任何能运行在浏览器中的代码都可以成为薯片插件。这意味着任何 Web 框架（Vue、React、Svelte、原生 JS）都可以用于开发插件。

### 6.3 桥接隔离原则
插件只能通过桥接 API 访问系统能力。不能直接使用 Node.js API、不能直接读写文件系统、不能直接发起网络请求。所有系统级操作都由主进程的基础服务代为执行。

### 6.4 中心路由原则（继承）
所有服务间的通信都经过薯片内核路由。服务之间不能直接调用。这与原架构的中心路由原则完全一致。

### 6.5 按需加载原则
基础服务、插件、主题都在需要时才加载。主机启动时只初始化最核心的内核和窗口管理。

### 6.6 崩溃隔离原则
每个插件应用运行在独立的渲染进程中。一个插件崩溃只影响该插件的窗口，不影响主进程和其他插件。主进程具备检测插件崩溃和自动恢复的能力。

## 七、与微信小程序模型的对比

| 微信生态 | 薯片生态 |
|----------|----------|
| 微信主程序 | 薯片主机（Chips Host） |
| 微信基础库 | 基础服务层 + SDK |
| 小程序框架 | 桥接 API + 预加载脚本 |
| 小程序（WXML/JS） | 插件应用（HTML/CSS/JS） |
| wx.* API | chips.* Bridge API |
| 小程序沙箱 | BrowserWindow sandbox |
| 小程序市场 | 薯片应用市场 |

两者的核心差异在于：
- 微信小程序使用自定义的 WXML/WXS，薯片直接使用标准 Web 技术
- 微信小程序有包体积限制，薯片本地运行无此限制
- 微信小程序的渲染层和逻辑层分离（双线程），薯片使用标准的 Electron 渲染进程

## 八、文件关联与系统集成

薯片主机注册以下系统级能力：

1. **文件关联**：
   - `.card` 文件 → 薯片主机 → 查看器插件
   - `.box` 文件 → 薯片主机 → 查看器插件

2. **URL 协议**：
   - `chips://open?plugin=editor&file=xxx` → 打开指定插件
   - `chips://install?plugin=xxx` → 安装插件

3. **系统托盘**：
   - 主机在后台运行时，系统托盘图标提供快捷操作

4. **自动启动**（可选）：
   - 用户可以设置开机自启动，加速后续使用

当用户双击 `.card` 文件时：
1. 操作系统调用薯片主机
2. 主机接收文件路径参数
3. 内核识别文件类型为卡片文件
4. 查找注册了 `.card` 处理能力的插件（默认为查看器插件）
5. 创建 BrowserWindow，加载查看器插件
6. 将文件路径传递给查看器插件
7. 用户看到卡片内容

整个过程中，操作系统只识别到"薯片主机"这一个程序。
