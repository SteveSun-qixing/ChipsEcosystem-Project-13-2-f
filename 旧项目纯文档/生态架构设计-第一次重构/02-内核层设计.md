**编写时间**：2026-02-12
**文档类型**：架构设计

## 一、内核定位

薯片内核（Chips Kernel）是整个生态的神经中枢，运行在 Electron 主进程（Node.js）中。它不包含任何业务逻辑，只提供最基本的基础设施能力：

- **消息路由**：在服务和插件之间转发消息
- **服务注册**：维护服务注册表，跟踪可用的服务
- **事件总线**：发布-订阅事件分发
- **生命周期管理**：管理服务和插件的启动/停止

内核是极度精简的。所有具体的功能（文件操作、资源管理、主题等）都由基础服务层提供，内核只负责调度和路由。

## 二、为什么选择 TypeScript 而非 Rust

### 2.1 Electron 上下文中的最优选择

在 Electron 架构中，主进程就是 Node.js。内核运行在主进程中意味着：

- **直接访问 Electron API**：窗口创建、IPC 管理、系统集成等，无需跨语言桥接
- **直接访问 Node.js API**：文件系统、网络、子进程等，无需序列化开销
- **与渲染进程零摩擦通信**：通过 Electron 内置的 IPC 机制
- **统一的调试体验**：整个应用栈（主进程 + 渲染进程）都可以用相同的工具调试

### 2.2 内核任务特征

内核的核心任务是**消息路由**——接收一个请求，查找目标处理器，转发请求，返回结果。这是典型的 I/O 密集型任务：

- 消息体积小（几 KB 的 JSON）
- 处理逻辑简单（查表、转发）
- 不需要复杂的计算

TypeScript/Node.js 的事件循环模型非常适合这种场景。Rust 的优势（零开销抽象、无 GC、内存安全）在这个场景下并不能带来显著收益。

### 2.3 Rust 的适当使用场景

Rust 不应被完全排除出生态，但应该用在正确的地方：

- **可选的原生模块**：ZIP 高速处理、图片编解码、视频元数据提取等 CPU 密集型任务
- **通过 N-API 集成**：作为 Node.js 的原生扩展，被基础服务层按需调用
- **非核心路径**：不在消息路由的关键路径上，不影响系统启动

这样既利用了 Rust 的性能优势，又避免了将其作为内核带来的复杂度。

## 三、内核组件

### 3.1 消息路由器（Message Router）

路由器是内核最核心的组件。所有模块间的通信都经过路由器。

**路由表结构**：
```
路由表 = {
  "namespace.action" → { 处理器, 权限要求, 超时设置 }
}

例如：
  "file.read"        → FileService.read()
  "card.getMetadata"  → CardService.getMetadata()
  "theme.getCurrent"  → ThemeService.getCurrent()
  "plugin.install"    → PluginManager.install()
```

**路由流程**：
```
请求发起方 → 路由器
  → 1. 解析请求的 namespace.action
  → 2. 在路由表中查找处理器
  → 3. 检查权限（请求方是否有权限调用此服务）
  → 4. 转发请求到处理器
  → 5. 处理器返回结果
  → 6. 路由器将结果返回给请求方
```

**两种请求来源**：

1. **主进程内服务间调用**：一个服务需要调用另一个服务（例如 CardService 需要 FileService 读取文件）。这种调用在同一个进程内完成，通过函数调用（非 IPC），延迟极低。

2. **渲染进程（插件）发起的调用**：插件通过 Bridge API 发送 IPC 消息到主进程，主进程的 IPC 处理器将消息转交给路由器，路由器找到处理器执行，结果通过 IPC 返回。

### 3.2 服务注册表（Service Registry）

注册表维护所有可用服务的信息。

**注册信息**：
```yaml
service:
  namespace: "file"
  provider: "FileService"
  version: "1.0.0"
  actions:
    - name: "read"
      params: { path: string }
      returns: Buffer
    - name: "write"
      params: { path: string, data: Buffer }
      returns: void
    - name: "exists"
      params: { path: string }
      returns: boolean
  status: "running" | "stopped" | "error"
```

**注册流程**：
1. 基础服务在初始化时，向注册表注册自己的命名空间和动作列表
2. 注册表验证命名空间的唯一性，记录服务信息
3. 路由器从注册表获取路由映射
4. 服务停止时，从注册表注销

**服务发现**：
- 路由器直接查询注册表获取处理器引用
- 插件可以通过 Bridge API 查询可用的服务列表
- 支持按命名空间、能力标签等条件筛选

### 3.3 事件总线（Event Bus）

事件总线实现发布-订阅模式，用于松耦合的异步通知。

**事件类型**：
- **系统事件**：`system.ready`、`system.shutdown`、`window.created`、`window.closed`
- **服务事件**：`file.changed`、`config.updated`、`theme.changed`、`plugin.installed`
- **插件事件**：`card.opened`、`card.saved`、`editor.modeChanged`

**事件流**：
```
发布者（服务或插件）
  → 事件总线
  → 所有订阅了该事件的处理器
```

**跨进程事件**：
- 主进程中的事件在主进程内分发（直接回调）
- 需要通知渲染进程的事件，通过 IPC 广播到相关的 BrowserWindow
- 渲染进程中的事件先通过 IPC 发送到主进程事件总线，再分发

### 3.4 生命周期管理器（Lifecycle Manager）

管理整个系统和各组件的启停顺序。

**系统启动流程**：
```
1. Electron 主进程启动
2. 初始化薯片内核
   2.1 创建消息路由器
   2.2 创建服务注册表
   2.3 创建事件总线
   2.4 创建生命周期管理器
3. 加载核心基础服务（按依赖顺序）
   3.1 配置管理器（最先加载，其他服务依赖它读取配置）
   3.2 日志服务
   3.3 文件服务
   3.4 插件管理器
4. 系统就绪，发布 system.ready 事件
5. 根据启动参数决定下一步
   - 无参数：显示启动器界面（或恢复上次会话）
   - 有文件参数：打开对应的处理插件
   - 有插件参数：直接启动指定插件
```

**系统关闭流程**：
```
1. 发布 system.shuttingDown 事件
2. 通知所有插件应用保存状态
3. 等待所有插件确认保存完毕（超时强制关闭）
4. 关闭所有 BrowserWindow
5. 停止所有基础服务（按依赖逆序）
6. 保存系统状态
7. 退出 Electron 进程
```

**插件生命周期**：
```
安装 → 加载 → 初始化 → 运行 → 暂停 → 恢复 → 停止 → 卸载
```

## 四、内核性能设计

### 4.1 路由性能

- 路由表使用 Map 数据结构，查找时间 O(1)
- 主进程内服务间调用是同步函数调用，无序列化开销
- 只有跨进程调用（插件 ↔ 主进程）才有 IPC 序列化开销

### 4.2 事件性能

- 事件订阅使用 Map<eventName, Set<handler>>，分发时间与订阅者数量线性相关
- 高频事件（如鼠标移动、滚动）应该在渲染进程本地处理，不通过事件总线
- 支持事件节流/防抖配置

### 4.3 启动性能

- 内核本身极轻量，启动时间 < 100ms
- 基础服务按需懒加载，只在第一次被调用时初始化
- 插件按需加载，只在用户打开时创建 BrowserWindow

### 4.4 内存管理

- 长时间未使用的插件窗口可以被挂起（Electron 的 backgroundThrottling）
- 基础服务中的缓存设置合理的上限和过期策略
- 主进程内存使用目标：< 100MB（不含插件渲染进程）

## 五、内核 API 概览

### 5.1 路由器 API

```typescript
// 注册路由
kernel.router.register(namespace: string, action: string, handler: Function)

// 注销路由
kernel.router.unregister(namespace: string, action: string)

// 调用服务（主进程内部使用）
kernel.router.invoke(namespace: string, action: string, params: any): Promise<any>

// 处理来自渲染进程的 IPC 请求
kernel.router.handleIPC(event: IpcMainInvokeEvent, request: BridgeRequest): Promise<any>
```

### 5.2 注册表 API

```typescript
// 注册服务
kernel.registry.register(serviceDescriptor: ServiceDescriptor)

// 注销服务
kernel.registry.unregister(namespace: string)

// 查询服务
kernel.registry.query(filter?: ServiceFilter): ServiceDescriptor[]

// 检查服务是否可用
kernel.registry.isAvailable(namespace: string, action: string): boolean
```

### 5.3 事件总线 API

```typescript
// 订阅事件
kernel.events.on(eventName: string, handler: EventHandler)

// 取消订阅
kernel.events.off(eventName: string, handler: EventHandler)

// 发布事件
kernel.events.emit(eventName: string, data?: any)

// 单次订阅
kernel.events.once(eventName: string, handler: EventHandler)
```

### 5.4 生命周期 API

```typescript
// 注册启动钩子
kernel.lifecycle.onReady(handler: Function)

// 注册关闭钩子
kernel.lifecycle.onShutdown(handler: Function)

// 启动服务
kernel.lifecycle.startService(namespace: string): Promise<void>

// 停止服务
kernel.lifecycle.stopService(namespace: string): Promise<void>
```

## 六、安全模型

### 6.1 权限分级

- **系统服务**：拥有完整权限，可以直接调用任何服务
- **官方插件**：拥有受信任权限，可以调用大部分系统服务
- **第三方插件**：受限权限，只能调用声明过的权限范围内的服务

### 6.2 权限验证

路由器在转发请求时检查权限：
1. 获取请求方的身份（哪个服务或插件）
2. 获取目标动作的权限要求
3. 检查请求方是否拥有所需权限
4. 权限不足则拒绝并返回错误

### 6.3 插件权限声明

插件在 manifest.yaml 中声明需要的权限：
```yaml
permissions:
  - file.read          # 读取文件
  - file.write         # 写入文件
  - resource.fetch     # 访问网络资源
  - card.read          # 读取卡片数据
  - card.write         # 写入卡片数据
  - window.create      # 创建新窗口
```

用户安装插件时，可以看到插件请求的权限列表，并决定是否授权。
