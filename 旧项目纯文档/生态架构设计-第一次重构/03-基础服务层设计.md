**编写时间**：2026-02-12
**文档类型**：架构设计

## 一、定位与职责

基础服务层是内核层之上、桥接层之下的通用能力层。薯片生态的核心理念是**底层厚、上层薄**——底层尽可能包含所有通用能力，上层应用只是调用底层的薄壳。

基础服务层分为两部分：

**A. 核心服务（编译在主机中，运行在主进程，数据/系统型）**：
- 文件、卡片、箱子、资源、ZIP、配置、主题、多语言、平台、凭证、日志、标签、序列化共 13 类
- 这些服务处理数据和系统操作，不涉及 UI 渲染
- 编译在薯片主机二进制中，不可独立替换

**B. 共享功能模块（作为模块插件独立安装，UI 型）**：
- 视频播放器、音频播放器、图片查看器、3D 渲染器、富文本编辑器、Markdown 渲染器、代码高亮器、文本编辑器等
- 这些模块提供 UI 渲染能力，由上层插件在渲染进程中按需加载
- 各自独立仓库、独立版本，可自由替换和升级（类似操作系统的共享库）
- 详见文档 04 中"模块插件"部分

本文档聚焦 A 类核心服务的设计。

**核心原则**：
- 核心服务**不包含任何业务逻辑**（不包含卡片的具体实现、布局的具体实现）
- 所有服务**通过内核路由器注册和调用**
- 服务**按需懒加载**，启动时不初始化所有服务
- 保持**轻量化**，只包含真正通用的数据/系统能力

## 三、懒加载策略

### 3.1 启动时加载的服务（核心依赖）

```
config.*        ← 其他服务需要读取配置
log.*           ← 系统需要日志能力
file.*          ← 基本文件操作
serializer.*    ← 配置文件解析需要
```

### 3.2 首次调用时加载的服务

```
card.*          ← 首次打开卡片文件时
box.*           ← 首次打开箱子文件时
resource.*      ← 首次访问外部资源时
zip.*           ← 首次处理 ZIP 文件时
theme.*         ← 首次需要主题时
i18n.*          ← 首次需要翻译时（通常很早）
platform.*      ← 首次查询平台能力时
module.*        ← 首次加载模块插件时
credential.*    ← 首次访问需要认证的资源时
tag.*           ← 首次使用标签功能时
```

### 3.3 加载机制

```typescript
// 懒加载示例：服务在第一次被路由器调用时才实例化
class LazyServiceLoader {
  private instances = new Map<string, Service>();
  private factories = new Map<string, () => Promise<Service>>();

  register(namespace: string, factory: () => Promise<Service>) {
    this.factories.set(namespace, factory);
  }

  async get(namespace: string): Promise<Service> {
    if (!this.instances.has(namespace)) {
      const factory = this.factories.get(namespace);
      const service = await factory();
      await service.initialize();
      this.instances.set(namespace, service);
    }
    return this.instances.get(namespace);
  }
}
```

## 四、Worker Threads 策略

对于 CPU 密集型任务，使用 Node.js Worker Threads 避免阻塞主进程事件循环：

### 适合放入 Worker 的任务

| 任务 | 原因 |
|------|------|
| ZIP 打包/解包 | 大文件处理耗时 |
| 文件哈希计算 | CPU 密集 |
| 图片缩略图生成 | 图像处理 |
| 批量文件扫描 | 大量 I/O + 文件处理 |
| 搜索索引构建 | CPU 密集 |

### 不适合放入 Worker 的任务

| 任务 | 原因 |
|------|------|
| 消息路由 | 需要快速响应，Worker 通信有开销 |
| 配置读取 | 简单 I/O，主线程足够 |
| 事件分发 | 需要直接访问订阅者引用 |
| IPC 处理 | 必须在主进程主线程 |

### Worker 池设计

```
主进程
├── 主线程（内核 + 路由 + IPC + 服务协调）
├── Worker 1（ZIP 处理队列）
├── Worker 2（文件扫描队列）
└── Worker 3（通用计算队列）
```

Worker 池大小根据 CPU 核心数动态调整，默认 2-4 个 Worker。

## 五、开源优先策略

继承原设计的开源优先原则。基础服务层的实现优先使用成熟的开源库：

| 功能 | 推荐开源库 | 说明 |
|------|-----------|------|
| ZIP 处理 | jszip / yazl+yauzl | ZIP 创建和解压 |
| YAML 解析 | js-yaml | YAML 解析生成 |
| 文件类型识别 | file-type | 根据内容判断文件类型 |
| 图片处理 | sharp (可选) | 缩略图生成，原生模块 |
| Markdown 解析 | markdown-it | Markdown → HTML |
| 代码高亮 | shiki / highlight.js | 语法高亮 |
| HTTP 客户端 | undici (Node.js 内置) | 网络请求 |
| WebDAV | webdav (npm) | WebDAV 协议支持 |
| 日志 | pino | 高性能 JSON 日志 |
| ID 生成 | nanoid | 高性能随机 ID |
| 数据校验 | zod | 运行时类型校验 |

**自研部分**（无现成开源方案或需要深度定制）：
- 卡片文件格式解析器（.card 专用结构）
- 箱子文件格式解析器（.box 专用结构）
- 中心路由器（生态专用消息路由）
- 主题引擎（原子级主题 Token 系统）
- 标签数据库（标签筛选和查询）
- 资源统一访问层（多源透明访问）

## 六、服务间依赖关系

```
配置服务 ← (所有服务都可能依赖配置)
    ↑
    │
日志服务 ← (所有服务都可能写日志)
    ↑
    │
序列化服务 ← 配置服务、卡片服务、箱子服务
    ↑
    │
文件服务 ← 卡片服务、箱子服务、资源服务
    ↑
    │
ZIP 服务 ← 卡片服务、箱子服务
    ↑
    │
资源服务 ← 卡片服务（获取外部资源）
    ↑
凭证服务 ← 资源服务（认证信息）
```

**关键规则**：服务之间的调用必须通过内核路由器，不能直接 import 和调用。即使在同一个进程中，也要通过 `kernel.router.invoke()` 调用。这确保了：
1. 调用可被日志记录和监控
2. 服务可以被独立替换
3. 权限可以被检查
4. 循环依赖可以被检测

## 七、可选的 Rust 原生加速模块

对于性能敏感的场景，可以开发 Rust 原生模块通过 N-API 集成：

| 模块 | 场景 | 预期收益 |
|------|------|----------|
| chips-zip-native | 大文件 ZIP 处理 | 10-50x 加速 |
| chips-hash-native | SHA256 计算 | 5-20x 加速 |
| chips-image-native | 图片缩略图 | 10-30x 加速 |

这些原生模块是**可选的**。系统始终有纯 TypeScript 的降级实现。安装了原生模块后，服务自动切换到更快的实现。

```typescript
// 优雅降级示例
async function computeHash(data: Buffer): Promise<string> {
  try {
    // 尝试使用 Rust 原生模块
    const native = require('chips-hash-native');
    return native.sha256(data);
  } catch {
    // 降级到纯 JS 实现
    const crypto = require('crypto');
    return crypto.createHash('sha256').update(data).digest('hex');
  }
}
```
