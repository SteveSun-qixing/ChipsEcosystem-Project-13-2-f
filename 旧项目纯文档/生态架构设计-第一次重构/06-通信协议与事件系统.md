## 一、通信模型概览

薯片生态中有三种通信路径：

### 路径一：主进程内服务间通信

```
服务 A → 内核路由器 → 服务 B → 结果 → 内核路由器 → 服务 A
```

- **传输方式**：进程内直接函数调用（同步/异步）
- **性能**：极低延迟（微秒级）
- **序列化**：无需序列化，直接传递对象引用
- **场景**：基础服务之间的协作

### 路径二：插件 ↔ 主进程通信

```
插件(渲染进程) → IPC → 主进程 IPC 处理器 → 内核路由器 → 服务 → 结果 → IPC → 插件
```

- **传输方式**：Electron IPC（ipcRenderer.invoke ↔ ipcMain.handle）
- **性能**：毫秒级延迟（JSON 序列化 + 进程间传输）
- **序列化**：Structured Clone 算法（支持大部分 JS 类型）
- **场景**：插件调用系统能力

### 路径三：插件 ↔ 卡片 iframe 通信

```
应用插件页面 → postMessage → 卡片 iframe → postMessage → 应用插件页面
```

- **传输方式**：window.postMessage（同一渲染进程内）
- **性能**：极低延迟
- **序列化**：Structured Clone 算法
- **场景**：应用插件与内嵌卡片渲染组件之间的数据传递

**Card Runtime Protocol（2026-02-27 实现完成）**：

壳层（viewer/editor）与 card iframe 之间使用 `chips-card-runtime` 协议通信，消息类型包括：

| 消息类型 | 方向 | 说明 |
|---------|------|------|
| `ready` | iframe → 壳层 | iframe 就绪，请求初始化数据 |
| `init` | 壳层 → iframe | 返回配置、主题 CSS/tokens、locale、词汇等初始化载荷 |
| `bridge-request` / `bridge-response` | 双向 | iframe 通过壳层代理调用 Bridge API |
| `resource-request` / `resource-response` | 双向 | iframe 请求资源（图片/视频等） |
| `resize` | iframe → 壳层 | iframe 内容尺寸变化通知 |
| `theme-change` | 壳层 → iframe | 主题变更广播（含新 CSS 和 tokens） |
| `language-change` | 壳层 → iframe | 语言变更广播（含新 locale 和词汇） |

`init` 载荷中注入真实的 Host 主题和语言数据，而非空值：
- 主题链路：壳层调用 `theme.getAllCss`（分层对象）+ `theme.resolve`（合并 tokens），并按 `tokens -> components -> animations -> icons` 拼接为 `theme.css: string`。
- 语言链路：壳层调用 `i18n.getCurrent`，注入 `locale`。

## 二、请求-响应协议

### 2.1 请求格式

所有通过路由器的调用都使用统一的请求格式：

```typescript
interface ChipsRequest {
  // 必填字段
  namespace: string;        // 服务命名空间，如 "file"、"card"、"theme"
  action: string;          // 动作名称，如 "read"、"getMetadata"
  params?: any;            // 参数（可选）

  // 由系统自动填充（请求方不需要设置）
  requestId?: string;      // 请求唯一 ID（路由器自动生成）
  source?: string;         // 请求来源标识
  timestamp?: number;      // 请求时间戳
}
```

### 2.2 响应格式

```typescript
interface ChipsResponse {
  requestId: string;        // 对应请求的 ID
  success: boolean;         // 是否成功
  data?: any;              // 成功时的返回数据
  error?: ChipsError;      // 失败时的错误信息
  timestamp: number;        // 响应时间戳
  duration: number;         // 处理耗时（毫秒）
}

interface ChipsError {
  code: string;            // 错误代码
  message: string;         // 人类可读的错误描述
  details?: any;           // 附加错误信息
}
```

### 2.3 命名空间规范

```
格式：{domain}.{action}

系统服务命名空间（保留）：
  file.*          文件操作
  card.*          卡片操作
  box.*           箱子操作
  resource.*      资源管理
  zip.*           ZIP 操作
  config.*        配置管理
  theme.*         主题管理
  i18n.*          多语言
  credential.*    凭证管理
  log.*           日志
  tag.*           标签
  serializer.*    序列化
  plugin.*        插件管理
  window.*        窗口管理
  system.*        系统操作

插件自定义命名空间：
  plugin.{pluginId}.*     插件自注册的服务
```

## 四、事件系统设计

### 4.1 事件模型

事件总线实现发布-订阅模式，支持两个作用域：

**主进程事件**：在主进程内分发，用于服务间的松耦合通知。
**跨进程事件**：从主进程广播到渲染进程，用于通知插件系统变化。

```
                     主进程事件总线
                    ┌──────────────┐
   服务 A ─发布──→ │   事件分发    │ ──→ 服务 B（主进程订阅者）
                    │              │ ──→ 服务 C（主进程订阅者）
                    │              │
                    │   跨进程广播  │ ──IPC──→ 渲染进程 A（插件订阅者）
                    │              │ ──IPC──→ 渲染进程 B（插件订阅者）
                    └──────────────┘
                          ↑
   插件 X ──IPC──→  IPC 接收器 ──→ 转发到事件总线
```

### 4.2 事件格式

```typescript
interface ChipsEvent {
  name: string;         // 事件名称
  data?: any;          // 事件数据
  source?: string;     // 事件来源
  timestamp: number;   // 事件时间戳
}
```

### 4.3 事件命名规范

```
格式：{domain}.{event}

示例：
  system.ready           系统就绪
  system.shuttingDown    系统关闭
  file.changed           文件变化
  file.created           文件创建
  file.deleted           文件删除
  card.opened            卡片打开
  card.saved             卡片保存
  card.closed            卡片关闭
  config.updated         配置更新
  theme.changed          主题变更
  language.changed       语言变更
  plugin.installed       插件安装
  plugin.uninstalled     插件卸载
  plugin.updated         插件更新
  window.created         窗口创建
  window.closed          窗口关闭
  window.focused         窗口获得焦点
```

### 4.4 事件过滤

插件可以选择性地订阅特定事件，减少不必要的 IPC 通信：

```typescript
// 只订阅卡片相关事件
chips.on('card.*', handler);

// 只订阅特定文件的变化
chips.on('file.changed', (data) => {
  if (data.path === myWatchedPath) {
    handleFileChange(data);
  }
});
```

主进程在广播跨进程事件时，检查每个渲染进程是否订阅了该事件，只向订阅者发送。

### 4.6 事件转发机制（2026-02-27 实现确认）

`event-forwarder.ts` 在内核事件总线上通配订阅 `'*'`，自动将所有点语义事件（如 `theme.changed`、`language.changed`）转发到所有 renderer 窗口。这是主题/i18n 全链路的关键基础设施。

事件传播链路：
```
Host 服务层 emit → 内核事件总线 → event-forwarder（'*' 订阅）→ 所有 renderer 窗口
                                                              → SDK useTheme/useI18n hooks 回调
                                                              → viewer/editor 壳层回调
                                                              → 壳层标准化主题快照后广播 card iframe
```

关键约束：
- 服务层发布事件时必须设置 `propagate: true`，否则事件不会被转发
- 事件名使用点语义（`theme.changed`），不使用冒号语义（`theme:changed` 已废弃）
- `theme-service.ts` 的 `handleApply` 在写配置后调用 `emitThemeChanged(themeId)` 发布事件
- `i18n-service.ts` 的 `handleSetCurrent` 在语言变更后调用 `emitLanguageChanged(locale)` 发布事件

### 4.5 事件优先级

事件处理器可以声明优先级（用于主进程内的服务）：

```typescript
// 高优先级处理器先执行
kernel.events.on('card.saved', handler, { priority: 'high' });

// 优先级顺序：high > normal > low
```

## 五、IPC 通道设计

### 5.1 通道命名

```
chips:invoke          请求-响应调用（ipcRenderer.invoke → ipcMain.handle）
chips:emit            插件发送事件（ipcRenderer.send → ipcMain.on）
chips:event:{name}    主进程向插件广播事件（webContents.send）
chips:window:{action} 窗口控制命令
chips:plugin-init     插件初始化数据传递
```

### 5.2 主进程 IPC 处理

```typescript
// 主进程 IPC 路由
ipcMain.handle('chips:invoke', async (event, request: ChipsRequest) => {
  const windowId = BrowserWindow.fromWebContents(event.sender)?.id;
  const pluginId = windowManager.getPluginId(windowId);

  // 权限检查
  if (!permissionManager.check(pluginId, request.namespace, request.action)) {
    throw { code: 'PERMISSION_DENIED', message: '权限不足' };
  }

  // 转发到路由器
  return kernel.router.invoke(request.namespace, request.action, request.params);
});

ipcMain.on('chips:emit', (event, { eventName, data }) => {
  const windowId = BrowserWindow.fromWebContents(event.sender)?.id;
  const pluginId = windowManager.getPluginId(windowId);

  // 将插件事件转发到事件总线
  kernel.events.emit(eventName, { ...data, source: pluginId });
});
```

## 六、超时与重试

### 6.1 超时策略

| 操作类型 | 默认超时 | 说明 |
|---------|---------|------|
| 普通服务调用 | 10 秒 | config.get、theme.getCurrent 等 |
| 文件操作 | 30 秒 | file.read、card.open 等 |
| 网络操作 | 60 秒 | resource.fetch 等 |
| ZIP 操作 | 120 秒 | 大文件打包/解包 |

插件可以在调用时指定自定义超时：

```typescript
// 使用默认超时
await chips.invoke('file', 'read', { path: '...' });

// 自定义超时
await chips.invoke('resource', 'fetch', {
  uri: 'https://...',
  _timeout: 120000  // 120 秒
});
```

### 6.2 重试策略

自动重试仅应用于幂等操作（如读取）：
- 网络超时：自动重试 3 次，间隔递增（1s, 2s, 4s）
- 文件锁冲突：自动重试 5 次，间隔 200ms
- 其他错误：不自动重试，直接返回错误

## 八、与原架构通信协议的对比

| 方面 | 原架构 | 新架构 |
|------|--------|--------|
| 主进程内通信 | 通过 Rust IPC | **直接函数调用**（无序列化） |
| 插件→主进程 | 多层桥接 | **一层 IPC（contextBridge）** |
| 消息格式 | 自定义协议 | **标准 JS 对象 + Structured Clone** |
| 事件系统 | Rust EventBus | **TypeScript EventEmitter + IPC 广播** |
| 跨插件通信 | 不支持/未定义 | **通过事件总线间接通信** |
| 调试工具 | 困难（跨语言） | **Chrome DevTools 原生支持** |
