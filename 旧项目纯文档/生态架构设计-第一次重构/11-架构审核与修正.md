## 修正一：架构层次缺少平台抽象层（影响文档 01）

**问题**：文档 01 的五层架构缺少文档 08 定义的平台抽象层（PAL），导致两篇核心文档的层级模型不一致。

**修正**：正式架构应为**六层**，从底层到顶层：

```
┌─────────────────────────────────────────────────────┐
│  第六层：插件应用（编辑引擎、查看器、设置面板……）       │
├─────────────────────────────────────────────────────┤
│  第五层：插件桥接层（Bridge API）                      │
├─────────────────────────────────────────────────────┤
│  第四层：基础服务层 + 共享功能模块                      │
├─────────────────────────────────────────────────────┤
│  第三层：薯片内核（路由 + 注册 + 事件 + 生命周期）      │
├─────────────────────────────────────────────────────┤
│  第二层：平台抽象层 PAL（适配不同操作系统）             │
├─────────────────────────────────────────────────────┤
│  第一层：Electron 宿主（Chromium + Node.js）           │
└─────────────────────────────────────────────────────┘
```

PAL 位于 Electron 宿主之上、薯片内核之下。内核和基础服务不直接调用 `process.platform` 或平台特定 API，而是通过 PAL 适配器访问系统能力。这确保了内核和服务层代码的平台无关性。

## 修正二：缺少第五种插件类型——模块插件（影响文档 03、04）

**问题**：需求明确指出"每种功能模块（图片查看器、视频播放器等）都是单独一个仓库"，且"公共基础层包含所有通用常用的功能模块"。但文档 03 只列出了主进程核心服务（文件、卡片、ZIP 等数据操作），未包含 UI 型共享模块（视频播放器、图片查看器、Markdown 渲染器等）。文档 04 只定义了四种插件类型，无法覆盖这些模块。

**修正**：新增第五种插件类型——**模块插件（Module Plugin）**。

### 模块插件的定位

模块插件是公共基础层中可被多个插件共享的 UI 功能模块。它不是独立应用（没有自己的窗口），也不是卡片/布局，而是**被其他插件加载使用的共享 Web 组件**。

类比操作系统的共享库（.dll / .so / .dylib）——安装一次，多个程序使用。

### 模块插件清单

| 模块插件 | ID | 说明 |
|---------|-----|------|
| 视频播放器 | chips-module-video-player | 提供视频播放能力 |
| 音频播放器 | chips-module-audio-player | 提供音频播放能力 |
| 图片查看器 | chips-module-image-viewer | 提供图片显示/缩放/旋转 |
| 3D 模型渲染器 | chips-module-3d-renderer | 提供 3D 模型显示 |
| 富文本编辑器 | chips-module-rich-text-editor | 提供富文本编辑能力 |
| Markdown 渲染器 | chips-module-markdown-renderer | Markdown → HTML |
| 代码高亮器 | chips-module-code-highlighter | 代码语法高亮 |
| 文本编辑器 | chips-module-text-editor | 基础文本编辑 |

### 模块插件的运行方式

模块插件不创建自己的窗口。它由其他插件（通常是卡片插件或应用插件）在自己的渲染进程中加载：

```typescript
// 视频卡片插件的渲染组件中
// 请求加载视频播放器模块
const playerModule = await chips.invoke('module', 'load', {
  moduleId: 'chips-module-video-player'
});
// playerModule.entryUrl → 模块的入口 URL

// 方式 1：作为 iframe 子组件加载
const iframe = document.createElement('iframe');
iframe.src = playerModule.entryUrl;

// 方式 2：作为 ES Module 动态导入（如果模块支持）
const player = await import(playerModule.moduleUrl);
player.createVideoPlayer(containerElement, options);
```

### manifest.yaml 示例

```yaml
id: "chips-module-video-player"
name: "视频播放器"
type: "module"                      # 第五种类型
version: "1.0.0"
publisher: "薯片官方"

entry:
  module: "dist/index.js"           # ES Module 入口
  iframe: "dist/index.html"         # iframe 入口（备选）

# 模块导出的能力
exports:
  - name: "createVideoPlayer"
    description: "创建视频播放器实例"
  - name: "VideoPlayerComponent"
    description: "Vue 3 视频播放器组件"

permissions:
  - resource.fetch                  # 需要获取视频资源
```

### 基础服务层的更新

文档 03 的基础服务层应区分两类：

**A. 核心服务（运行在主进程中，数据型）**：
file、card、box、resource、zip、config、theme、i18n、credential、log、tag、serializer、platform

**B. 共享功能模块（作为模块插件安装，UI 型）**：
video-player、audio-player、image-viewer、3d-renderer、rich-text-editor、markdown-renderer、code-highlighter、text-editor

核心服务编译在主机二进制中不可替换。共享功能模块是独立仓库的模块插件，可以被替换和升级。

## 修正四：目录结构与独立仓库的矛盾（影响文档 07）

**问题**：文档 07 展示的目录结构将所有插件放在 `Project-12/plugins/` 下，暗示 monorepo。但文档 10 明确要求每个插件是独立仓库。这两者矛盾。

**修正**：

文档 07 的目录结构应理解为**开发时的开发超级工作区**（类似 Android 的 repo 工具），而非运行时结构。实际中：

- 每个插件是独立的 git 仓库
- `Project-12` 是**超级项目（Super Project）**，通过 git submodule 或独立克隆聚合这些仓库
- 开发时可以在 pnpm workspace 中统一管理，但每个仓库可以独立开发、构建、发布
- CI/CD 中每个仓库独立流水线

**发布后的运行时结构**（在用户设备上）：

```
~/.chips/
├── host/                            # 薯片主机二进制
├── plugins/                         # 安装的插件（各自独立安装）
│   ├── apps/
│   ├── cards/
│   ├── layouts/
│   ├── modules/                     # 新增：模块插件
│   └── themes/
├── config/
├── cache/
├── logs/
├── workspace/
└── tags.db
```

## 修正五：补充 Style Dictionary 到主题架构（影响文档 09）

**问题**：你提到了 "Ark UI + Style Dictionary" 方案。文档 09 推荐了 Zag.js + Ark UI 但没有包含 Style Dictionary。

**修正**：在主题架构中加入 **Style Dictionary**（Apache 2.0 许可，Amazon 开源）。

### Style Dictionary 的作用

Style Dictionary 是设计 Token 的构建工具。它将一套 Token 定义（JSON/YAML）转换为多种平台的输出格式：

```
Token 定义文件（YAML/JSON）
  ↓ Style Dictionary 构建
  ├── CSS 自定义属性（Web 端使用）
  ├── iOS Swift 常量（未来移动端）
  ├── Android XML 资源（未来移动端）
  ├── TypeScript 类型定义
  └── 设计工具文件（Figma Token）
```

### 主题包开发流程

```
1. 设计师在 Figma 中定义设计 Token
2. 导出为 Style Dictionary 格式（tokens.json）
3. Style Dictionary 构建：
   - 生成 global.css（全局 Token）
   - 生成 semantic.css（语义化 Token）
   - 生成 components/*.css（组件级 Token）
   - 生成 types.d.ts（TypeScript 类型）
4. 加上图标、动画、装饰资源
5. 打包为主题插件 .cpk
```

### Token 定义示例

```json
{
  "chips": {
    "color": {
      "primary": {
        "value": "#6200ee",
        "type": "color",
        "description": "主色调"
      },
      "on-primary": {
        "value": "#ffffff",
        "type": "color"
      }
    },
    "radius": {
      "sm": { "value": "4px", "type": "dimension" },
      "md": { "value": "8px", "type": "dimension" },
      "lg": { "value": "16px", "type": "dimension" }
    },
    "button": {
      "radius": {
        "value": "{chips.radius.md}",
        "type": "dimension",
        "description": "引用语义化 Token"
      }
    }
  }
}
```

Style Dictionary 自动解析引用关系，生成最终的 CSS：

```css
:root {
  --chips-color-primary: #6200ee;
  --chips-color-on-primary: #ffffff;
  --chips-radius-sm: 4px;
  --chips-radius-md: 8px;
  --chips-radius-lg: 16px;
  --chips-button-radius: 8px; /* 已解析引用 */
}
```

## 修正六：补充编辑引擎的内部插件化设计（影响文档 04）

**问题**：需求明确指出"卡片编辑引擎本体要高度插件化，内部各种应用也要高度插件化"。但文档 04 只描述了生态层面的插件系统，没有描述应用内部的扩展机制。

**修正**：编辑引擎作为一个应用插件，自身也拥有内部扩展点系统。

### 编辑引擎的内部扩展架构

```
编辑引擎应用插件
├── 引擎框架（固定）：窗口壳、菜单栏、IPC 桥接
├── 扩展槽 A：编辑引擎布局（可替换）
│   ├── 无限画布布局（独立仓库/插件）
│   ├── 工作台布局（独立仓库/插件）
│   └── 页面式布局（未来，独立仓库/插件）
├── 扩展槽 B：工具面板（可添加/移除）
│   ├── 文件管理器面板（独立仓库/插件）
│   ├── 卡箱库面板（独立仓库/插件）
│   ├── 编辑面板（独立仓库/插件）
│   └── 历史记录面板（独立仓库/插件）
├── 扩展槽 C：功能扩展（可添加）
│   ├── 撤销重做模块
│   ├── 搜索替换模块
│   └── 自动保存模块
```

这些内部扩展也是独立仓库，通过编辑引擎的扩展注册 API 加载。它们在编辑引擎的 BrowserWindow 内运行（不创建新窗口），可以作为 iframe 子组件或直接作为 ES Module 动态导入。

### 编辑引擎扩展注册

```typescript
// 编辑引擎内部的扩展注册
editor.registerLayout('infinite-canvas', InfiniteCanvasLayout);
editor.registerPanel('file-manager', FileManagerPanel);
editor.registerFeature('undo-redo', UndoRedoFeature);
```

编辑引擎在自己的 manifest.yaml 中声明支持的扩展槽类型，用户可以通过设置面板管理这些内部扩展。

## 修正七："底层厚上层薄"原则需要贯穿始终

**问题**：需求的核心理念是"底层厚上层薄"——底层包含尽可能多的能力，上层应用只是调用底层的薄壳。但前几篇文档对这个原则的强调不够，特别是在说明上层应用应该有多"薄"方面。

**修正**：明确定义"厚底层"和"薄上层"的边界。

### 底层（薯片主机 + 模块插件）负责的事情

- 所有文件系统操作（读写、监听、搜索）
- 所有网络操作（HTTP、WebSocket、WebDAV）
- 所有卡片/箱子的解析和打包
- 所有资源获取和缓存
- 所有主题加载和注入
- 所有多语言翻译
- 所有配置管理
- 所有日志记录
- 所有平台适配
- 视频播放、音频播放、图片处理（通过模块插件）
- Markdown 渲染、代码高亮、富文本编辑（通过模块插件）
- 窗口管理、快捷方式、系统托盘
- 插件安装、卸载、更新、权限

### 上层应用插件只需要做的事情

- **界面布局**：定义自己的 UI 结构（几行 HTML/Vue/React）
- **业务编排**：调用底层 API，组织业务流程
- **用户交互**：处理点击、拖拽等用户操作
- **状态管理**：管理应用自身的 UI 状态

一个典型的应用插件**不需要**自己实现：文件读写、网络请求、视频播放、主题系统、多语言、配置存储。这些全部通过 `chips.invoke()` 调用底层完成。

### 示例：一个最简化的视频播放器应用

```html
<!-- 整个应用只有一个 HTML 文件 + 少量 JS -->
<div id="app">
  <div id="player-container"></div>
  <div id="controls">
    <button onclick="openFile()">打开文件</button>
  </div>
</div>
<script>
  async function openFile() {
    // 调用底层对话框能力
    const files = await chips.dialog.showOpenDialog({
      filters: [{ name: '视频', extensions: ['mp4','webm','mkv'] }]
    });
    if (!files) return;

    // 调用底层模块加载能力
    const player = await chips.invoke('module', 'load', {
      moduleId: 'chips-module-video-player'
    });

    // 加载视频播放器模块到容器中
    const iframe = document.createElement('iframe');
    iframe.src = player.entryUrl + '?file=' + encodeURIComponent(files[0]);
    document.getElementById('player-container').appendChild(iframe);
  }
</script>
```

这就是一个完整的视频播放器应用。所有重活（文件选择、视频解码、播放控制）都由底层完成。应用只负责 UI 布局和业务编排。
