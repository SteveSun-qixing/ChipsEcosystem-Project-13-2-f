# 内核层技术规格

**文档版本**: v1.0.0  
**编写日期**: 2026-02-12  
**对应任务**: 阶段一任务 1.2（内核层技术规格）  
**适用范围**: Chips-Host `src/main/kernel/`（TypeScript strict）

---

## 1. 文档目标与约束

本规格定义薯片内核四大组件的可落地实现标准：

1. 消息路由器（Router）
2. 服务注册表（Registry）
3. 事件总线（EventBus）
4. 生命周期管理器（LifecycleManager）

硬性约束：

- 所有服务间调用必须经过内核路由器，禁止直接 import 调用。
- 插件侧只能通过 Bridge API（`window.chips.*`）进入内核。
- 内核只提供基础设施能力，不承载业务逻辑。
- 内核代码必须平台无关；平台差异统一经 PAL 进入。
- 全部接口面向 TypeScript strict，禁止 `any`。

---

## 2. 内核总体架构

### 2.1 内核在主进程中的位置

内核位于六层架构的第三层（Electron 宿主 + PAL 之上、基础服务之下）：

```text
Electron Host
  -> PAL
    -> Kernel
      -> Foundation Services / Shared Modules
        -> Bridge
          -> Plugin Apps
```

### 2.2 四大组件关系

```text
Plugin/Service Request
  -> Router
    -> Permission Guard
    -> Registry (lookup route + service metadata)
    -> Target Service Handler
  -> Response

Registry
  -> stores service/action metadata + status
  -> queried by Router and LifecycleManager

EventBus
  -> in-process sync dispatch
  -> cross-process async broadcast via IPC adapter

LifecycleManager
  -> maintains service states and dependency DAG
  -> drives startup/shutdown/health-check
  -> updates Registry service status
```

### 2.3 内核启动顺序

1. 创建 Router（空路由表 + 缓存 + 验证器 + 权限钩子）
2. 创建 Registry（空服务表）
3. 创建 EventBus（订阅表 + 广播适配器）
4. 创建 LifecycleManager（依赖图 + 状态机）
5. 注册内核保留命名空间（`system.*`, `route.*`, `event.*`）
6. 加载核心服务（配置 -> 日志 -> 文件 -> 插件管理）
7. 发布 `system.ready`

## 3. 消息路由器（Router）

## 3.1 目标

统一承载主进程内服务调用与插件 IPC 调用，提供：

- O(1) 主路由查找
- 可预测匹配顺序（精确 -> 前缀 -> 通配符）
- 权限检查与参数校验
- 路由缓存（LRU + TTL）
- 统一错误响应

### 3.2 数据结构设计

```typescript
export type ChipsNamespace = string;
export type ChipsAction = string;
export type RouteKey = `${string}.${string}`;

export interface RoutePermissionRequirement {
  permissions: string[];
  requireAll: boolean;
}

export interface RouteDescriptor<Params, Result> {
  namespace: ChipsNamespace;
  action: ChipsAction;
  key: RouteKey;
  handler: RouteHandler<Params, Result>;
  priority: number;
  timeoutMs: number;
  permission?: RoutePermissionRequirement;
  schemaId?: string;
  enabled: boolean;
}

export type RouteHandler<Params, Result> = (
  params: Params,
  context: RouteInvocationContext
) => Promise<Result>;

export interface RouteInvocationContext {
  requestId: string;
  caller: CallerIdentity;
  timestamp: number;
  deadline: number;
}

export interface CallerIdentity {
  id: string;
  type: "kernel" | "service" | "plugin";
  pluginId?: string;
  windowId?: number;
}

export interface RouterCacheEntry {
  descriptor: RouteDescriptor<unknown, unknown>;
  matchedBy: "exact" | "prefix" | "wildcard";
  cachedAt: number;
  expiresAt: number;
}
```

内部索引：

- `exactRoutes: Map<RouteKey, RouteDescriptor<unknown, unknown>[]>`
- `prefixRoutes: RouteDescriptor<unknown, unknown>[]`
- `wildcardRoutes: RouteDescriptor<unknown, unknown>[]`
- `lruCache: Map<RouteKey, RouterCacheEntry>`（配合双向链表实现 LRU）

> 每个桶内按 `priority DESC` 排序；同优先级按注册时间先后。

### 3.3 路由匹配策略

匹配顺序固定为：

1. 精确匹配：`namespace.action`
2. 前缀匹配：`namespace.actionPrefix*`
3. 通配符匹配：`namespace.*` 或 `*.*`

命中第一个可用（`enabled=true` 且权限通过）的处理器立即返回。

### 3.4 LRU 缓存策略

- 默认容量：`512`
- 最大容量：`4096`
- TTL：`60_000ms`
- 仅缓存成功匹配结果；不缓存 miss
- 以下事件触发缓存失效：
  - 注册/注销路由
  - 路由启停状态变化
  - 权限模型版本变化

### 3.5 请求处理流水线

```text
receive -> normalize -> validate -> lookup
-> permissionCheck -> executeWithTimeout -> mapResult
-> emitMetrics -> return
```

伪代码：

```text
function invoke(namespace, action, params, context):
  key = namespace + "." + action
  validate key/params/context
  descriptor = findInCacheOrRouteTable(key)
  if descriptor missing: throw ROUTE_NOT_FOUND
  if no permission: throw PERMISSION_DENIED
  result = run handler with timeout
  return result
```

### 3.6 路由优先级规则

当多个处理器都匹配：

1. `priority` 数值大者优先
2. 若同优先级，精确 > 前缀 > 通配符
3. 若仍冲突，较早注册优先（稳定顺序）

### 3.7 Router 接口定义（完整）

来源：`Chips-core/src/router/route_table.rs`

## 4. 服务注册表（Registry）

### 4.1 目标

提供统一服务元数据管理，支撑：

- 服务注册/注销
- 命名空间冲突检测
- 按命名空间/能力/状态查询
- 路由器与生命周期联动

### 4.2 数据结构

```typescript
export type ServiceStatus =
  | "unloaded"
  | "loading"
  | "ready"
  | "running"
  | "stopping"
  | "stopped"
  | "error";

export interface ServiceActionDescriptor {
  action: string;
  routeKey: `${string}.${string}`;
  permissions: string[];
  timeoutMs: number;
  schemaId?: string;
}

export interface ServiceDescriptor {
  namespace: string;
  serviceId: string;
  version: string;
  capabilities: string[];
  dependencies: string[];
  actions: ServiceActionDescriptor[];
  status: ServiceStatus;
  registeredAt: number;
  updatedAt: number;
}

export interface RegistryQuery {
  namespace?: string;
  capability?: string;
  status?: ServiceStatus;
  serviceId?: string;
}
```

内部结构：

- `servicesByNamespace: Map<string, ServiceDescriptor>`
- `servicesById: Map<string, ServiceDescriptor>`
- `capabilityIndex: Map<string, Set<string /*serviceId*/>>`
- `statusIndex: Map<ServiceStatus, Set<string /*serviceId*/>>`

### 4.3 注册流程

```text
initialize service
-> validate descriptor
-> check namespace uniqueness
-> persist descriptor + indexes
-> notify router registerRoutes()
-> set status ready/running
```

### 4.4 注销流程

```text
check service status (must not be running unless force)
-> remove route bindings
-> remove indexes
-> remove descriptor
-> emit service.unregistered
```

### 4.5 查询 API

- 按命名空间：`namespace = "file"`
- 按能力标签：`capability = "fs.read"`
- 按状态：`status = "running"`
- 组合筛选（AND）

### 4.6 命名空间冲突检测

规则：

1. `namespace` 全局唯一，不允许重复注册
2. 保留命名空间（`system`, `route`, `event`）仅内核可注册
3. 发现冲突返回标准错误：

```json
{
  "code": "REGISTRY_NAMESPACE_CONFLICT",
  "message": "namespace 'file' already registered by service chips.file-service",
  "details": {
    "namespace": "file",
    "existingServiceId": "chips.file-service"
  }
}
```

## 5. 事件总线（EventBus）

### 5.1 目标

实现主进程内松耦合事件通信，并支持跨进程广播：

- 精确订阅 + 通配符订阅（`card.*`）
- 订阅过滤器
- 同步分发（主进程）
- 异步广播（到渲染进程）
- 订阅器超时与隔离

### 5.2 事件类型定义

```typescript
export interface ChipsEvent<TData = unknown> {
  id: string;
  name: string;
  source: string;
  data: TData;
  timestamp: number;
  propagate: boolean;
  metadata?: Record<string, string | number | boolean>;
}

export interface EventFilter {
  source?: string;
  dataPathEquals?: Array<{
    path: string;
    value: string | number | boolean;
  }>;
}

export interface EventSubscription {
  subscriptionId: string;
  subscriberId: string;
  pattern: string;
  filter?: EventFilter;
  timeoutMs: number;
  enabled: boolean;
}

export type EventHandler<TData = unknown> = (
  event: ChipsEvent<TData>
) => Promise<void>;
```

### 5.3 订阅机制

- 精确订阅：`theme.changed`
- 通配符订阅：`theme.*`
- 全量订阅：`*`

匹配函数：

```text
if pattern == "*": match
else if pattern endsWith(".*"): prefix match
else exact match
```

### 5.4 发布机制

1. **主进程同步分发**（等待所有处理器结束，可统计结果）
2. **跨进程异步广播**（IPC fanout，只发给已订阅窗口）

### 5.5 超时处理

- 默认单订阅器超时：`5_000ms`
- 超时后标记该订阅本次失败，继续分发其他订阅器
- 连续 N 次超时（默认 10）自动降级禁用并上报告警事件

### 5.6 事件隔离

- 订阅器异常不会中断全局分发
- 每个处理器独立 `try/catch + timeout wrapper`
- 结果统计分为 `success/failed/timeout`

## 6. 生命周期管理器（LifecycleManager）

### 6.1 服务状态机

规范状态机：

```text
unloaded -> loading -> ready -> running -> stopping -> stopped
                                   \-> error
loading  -> error
stopping -> error
stopped  -> loading (restart)
error    -> loading (recover)
```

### 6.2 依赖图数据结构（DAG）

```typescript
export interface DependencyNode {
  serviceId: string;
  dependencies: string[];
}

export interface DependencyGraph {
  nodes: Map<string, DependencyNode>;
  reverseEdges: Map<string, Set<string>>;
}
```

### 6.3 拓扑排序算法（Kahn）

伪代码：

```text
inDegree[service] = number of dependencies
queue = all nodes where inDegree == 0
order = []
while queue not empty:
  node = queue.pop()
  order.push(node)
  for dependent in reverseEdges[node]:
    inDegree[dependent] -= 1
    if inDegree[dependent] == 0:
      queue.push(dependent)
if order size != node size:
  throw circular dependency
```

### 6.4 环检测算法（DFS）

使用三色标记（white/gray/black）：

- 进入 `gray` 节点再次命中 `gray` 即存在环
- 记录路径用于错误详情输出

### 6.5 系统启动流程

1. 读取全部服务描述
2. 构建依赖图并执行环检测
3. 拓扑排序得到启动序列
4. 逐个执行 `initialize -> registerRoutes -> mark ready`
5. 启动必须常驻服务并标记 `running`
6. 发布 `system.ready`

### 6.6 系统关闭流程

1. 发布 `system.shuttingDown`
2. 停止接收新请求（Router drain）
3. 按拓扑逆序停止服务
4. 注销路由 + 更新注册表状态
5. 发布 `system.stopped`

### 6.7 健康检查机制

- 默认间隔：30 秒
- 检查内容：
  - 状态合法性
  - 最近错误计数
  - 核心 handler 心跳
  - 响应延迟阈值
- 结果等级：`healthy` / `degraded` / `unhealthy`

## 8. 标准错误模型

所有对外接口（含 Router/Bridge）统一返回：

```typescript
export interface StandardError {
  code: string;
  message: string;
  details?: unknown;
}
```

核心错误码：

- `ROUTE_NOT_FOUND`
- `ROUTE_TIMEOUT`
- `PERMISSION_DENIED`
- `REGISTRY_NAMESPACE_CONFLICT`
- `LIFECYCLE_CIRCULAR_DEPENDENCY`
- `SERVICE_NOT_READY`
- `EVENT_SUBSCRIPTION_NOT_FOUND`
- `INTERNAL_ERROR`

## 9. 性能要求（硬指标）

- 路由查找延迟（cache hit）：`< 0.10ms`
- 路由查找延迟（cache miss）：`< 0.35ms`
- 单事件主进程分发（10 订阅者）：`< 1.00ms`
- 内核常驻内存：`< 20MB`
- 内核就绪时间（不含服务加载）：`< 200ms`

采样与统计要求：

- 每 60s 输出一次 P50/P95/P99
- 错误率超过 1% 连续 5 分钟触发告警事件 `system.alert.kernel`
