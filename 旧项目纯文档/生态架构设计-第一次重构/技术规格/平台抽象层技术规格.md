## 1. 目标与边界

本规格定义 Chips Host 第二层平台抽象层（Platform Abstraction Layer, PAL）的完整实现标准，覆盖：

1. PAL 架构与分层边界（位于 Electron 宿主与内核之间）
2. 13 类适配器接口的完整 TypeScript 定义
3. 核心/扩展/预留三级能力清单（含平台差异）
4. PlatformManager 的注册、检测、查询与运行时决策机制
5. Windows/macOS/Linux 三平台实现规范
6. 平台扩展插件（platform extension plugin）规范

本规格用于阶段二任务 2.2（PAL 开发）和阶段二任务 2.5（platform 服务封装）的实现依据。

## 2. 架构定位与设计原则

### 2.1 六层架构定位

```text
第六层  插件应用（Web）
第五层  Bridge API（window.chips.*）
第四层  基础服务层 + 共享功能模块
第三层  Chips Kernel（router/registry/event/lifecycle）
第二层  PAL（本规格）
第一层  Electron 宿主（Chromium + Node.js）
```

PAL 作为系统能力唯一抽象面，向下吸收平台差异，向上输出统一契约。

### 2.2 强制设计原则

1. **平台无关性原则**  
   内核与基础服务禁止直接调用 `process.platform`、系统命令、注册表、plist、xdg 等平台细节。

2. **中心路由一致性原则**  
   非 `platform` 服务不得直接依赖 PAL 适配器；跨服务调用仍通过 `kernel.router.invoke()`。

3. **Bridge 隔离一致性原则**  
   插件侧不直接接触 PAL，仅可通过 `window.chips.invoke('platform', ...)` 间接访问。

4. **能力可检测原则**  
   平台能力必须可声明、可枚举、可检测，调用前可判定是否可用。

5. **降级可运行原则**  
   扩展能力不可用时必须提供明确错误码或无害降级路径，不能影响核心能力可用性。

### 3.2 运行流程

1. 主机启动时识别当前平台（windows/macos/linux）
2. PlatformManager 载入当前平台适配器集合
3. 注册核心能力（必须）
4. 探测扩展能力（可选）
5. 预留能力仅注册描述，不加载实现
6. `platform` 服务将 PAL 能力封装为路由动作对外提供

## 7. PlatformManager 设计

### 7.1 职责

`PlatformManager` 是 PAL 的唯一入口，负责：

1. 识别运行平台和上下文
2. 注册并管理适配器实例
3. 注册、检测、查询平台能力
4. 为 `platform` 服务提供查询 API
5. 统一错误码与降级策略

### 7.2 核心数据结构

```ts
export interface CapabilityFilter {
  level?: CapabilityLevel;
  status?: CapabilityStatus;
  platform?: PlatformKind;
  adapter?: AdapterToken;
}

export class PlatformManager {
  private readonly runtimeInfo: PlatformRuntimeInfo;
  private readonly adapters = new Map<AdapterToken, unknown>();
  private readonly capabilities = new Map<string, PlatformCapabilityDescriptor>();

  constructor(runtimeInfo: PlatformRuntimeInfo) {
    this.runtimeInfo = runtimeInfo;
  }

  getRuntimeInfo(): PlatformRuntimeInfo;

  registerAdapter<T>(token: AdapterToken, adapter: T): void;
  getAdapter<T>(token: AdapterToken): T;

  registerCapability(descriptor: PlatformCapabilityDescriptor): void;
  setCapabilityStatus(capabilityId: string, status: CapabilityStatus): void;

  hasCapability(capabilityId: string): boolean;
  requireCapability(capabilityId: string): PlatformCapabilityDescriptor;
  listCapabilities(filter?: CapabilityFilter): readonly PlatformCapabilityDescriptor[];

  initialize(): Promise<void>;
  dispose(): Promise<void>;
}
```

### 7.3 初始化算法

```ts
async function initializePal(): Promise<void> {
  const runtime = detectRuntimeInfo();
  const manager = new PlatformManager(runtime);

  const adapters = await loadPlatformAdapters(runtime.kind);
  for (const adapter of adapters) {
    manager.registerAdapter(adapter.token, adapter);
  }

  registerCoreCapabilities(manager);
  await probeExtendedCapabilities(manager);
  registerReservedCapabilities(manager);

  await loadPlatformExtensionPlugins(manager, runtime);
}
```

- `platform.getSystemInfo`

### 7.5 运行时平台判断逻辑

1. `process.platform` 映射：
   - `win32` -> `windows`
   - `darwin` -> `macos`
   - 其余桌面 Linux -> `linux`
2. 读取 `process.arch` 得到 `x64/arm64/...`
3. 读取 `os.release()` 与 `os.version()` 形成版本基线
4. Linux 额外读取 `XDG_CURRENT_DESKTOP` / `DESKTOP_SESSION`
5. 生成 `PlatformRuntimeInfo` 供全局使用

## 8. 各平台适配器实现规范

### 8.1 Windows 规范

1. **文件与路径**：使用 Win32 路径规则，统一在 PAL 输出标准化绝对路径。  
2. **文件关联/协议**：通过注册表（`HKCU\\Software\\Classes`）进行当前用户级注册。  
3. **自启动**：优先使用启动项（Run），企业场景可扩展任务计划策略。  
4. **托盘/通知**：托盘基于 Electron Tray，通知使用 Windows Toast。  
5. **快捷键**：处理保留组合键冲突（如系统级 Win+*）。  
6. **扩展能力**：任务栏缩略图、Jump List、Mica/Acrylic、壁纸能力。

### 8.2 macOS 规范

1. **目录规范**：`~/Library/Application Support/Chips` 等标准目录。  
2. **文件关联/协议**：使用 `CFBundleDocumentTypes` 与 `CFBundleURLTypes`。  
3. **自启动**：使用 LaunchAgent（用户级 plist），避免 root 级服务。  
4. **托盘/通知**：NSStatusBar/Notification Center 映射，适配权限模型。  
5. **窗口行为**：处理 traffic-light、全屏 Space 模式、represented file。  
6. **扩展能力**：Touch Bar、菜单栏扩展、Spotlight 协作。

### 8.3 Linux 规范

1. **目录规范**：遵循 XDG Base Directory（`$XDG_CONFIG_HOME` 等）。  
2. **文件关联/协议**：通过 `.desktop` + `mimeapps.list` + `xdg-mime`。  
3. **自启动**：`~/.config/autostart/*.desktop`。  
4. **通知/托盘**：优先 xdg-desktop-portal，兼容不同桌面环境。  
5. **快捷键**：Wayland 与 X11 差异必须显式处理失败返回。  
6. **扩展能力**：D-Bus、systemd user service。

### 8.4 通用实现约束

1. 所有适配器异步接口必须返回标准错误结构：  
   `{ code: string, message: string, details?: unknown }`
2. 所有外部命令调用必须有超时与退出码检查。
3. 所有平台私有逻辑必须封装在 `pal/platforms/<platform>/` 内。
4. 适配器之间不得互相直接依赖，统一由 `PlatformManager` 协调。

## 9. 平台扩展插件规范

### 9.1 定位

平台扩展插件用于承载非核心、平台独有能力（如壁纸接管、Touch Bar、D-Bus 深度集成）。

### 9.2 插件类型与清单

平台扩展插件在插件系统中使用 `type: module`，并在 `manifest.yaml` 声明 `platform` 字段。

```yaml
id: "chips-platform.windows-wallpaper"
name: "Windows Wallpaper Extension"
type: "module"
version: "1.0.0"

platform:
  targets: ["windows"]
  arch: ["x64", "arm64"]
  min_os_version: "10.0.19045"
  desktop_env: ["explorer"]

provides:
  capabilities:
    - "desktop.wallpaper.set"

permissions:
  - "platform.extended.desktop.wallpaper"
```

### 9.3 `platform` 字段定义

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `targets` | `string[]` | 是 | 支持的平台：`windows/macos/linux` |
| `arch` | `string[]` | 否 | 支持架构：`x64/arm64` 等 |
| `min_os_version` | `string` | 否 | 最低系统版本 |
| `desktop_env` | `string[]` | 否 | Linux/macOS 特定桌面环境约束 |

### 9.4 加载时机与条件

1. PAL 核心适配器初始化完成后执行扩展扫描
2. 过滤 `platform.targets` 与当前平台不匹配的插件
3. 校验架构、系统版本、权限声明
4. 校验通过后注册其 `provides.capabilities`
5. 任一插件加载失败只影响该插件，不影响核心 PAL 初始化

### 9.5 错误处理

扩展插件加载失败统一上报事件：

- 事件名：`platform.extension.load-failed`
- 数据：`{ pluginId, code, message, details }`
- 错误不阻断主机启动

## 10. 测试规格（供阶段二实现）

### 10.1 单元测试

1. 每个适配器接口至少 8 条用例（正常/错误/边界）
2. PlatformManager 注册、检测、过滤、降级逻辑完整覆盖
3. Linux/Windows/macOS 差异路径转换必须有参数化用例

### 10.2 集成测试

1. 三平台 CI 矩阵（Windows/macOS/Linux）
2. 核心能力清单逐项验证
3. 扩展能力在可用与不可用场景下的行为一致性验证

### 10.3 关键验收项

- 13 类适配器接口全部可编译
- 核心能力不少于 20 项且全平台可用
- 扩展能力不少于 10 项且可检测
- 预留能力不少于 5 项且有正式接口位
- `platform.*` 查询行为与能力注册表一致

## 11. 与其他层的对接边界

1. **向下（Electron/OS）**：只允许 PAL 触达平台私有 API
2. **向上（Kernel/Service）**：仅暴露 `PlatformManager` 与 `platform` 服务封装
3. **向外（Plugin）**：插件经 `window.chips.invoke('platform', ...)` 访问
4. **跨服务调用**：其他服务调用平台能力时仍需走内核路由

## 12. 实施结论

本规格将 PAL 从“概念层描述”落地为“可实现契约”，确保：

1. 平台差异被完整吸收，不向上泄漏
2. 核心能力稳定统一，扩展能力可插拔
3. 阶段二开发可以按接口直接编码
4. 后续移动端/Web 运行时可复用同一抽象模型（替换适配器实现）
