## 1. 插件系统总体架构

### 1.1 定位

插件系统负责统一定义并治理以下能力：

- 五类插件的运行模型、生命周期和互操作协议
- `manifest.yaml` 规范、校验和版本兼容规则
- `.cpk`（Chips Plugin Package）包格式与签名校验
- 插件安装、卸载、更新的事务化流程和回滚机制
- 插件注册表持久化与索引查询

### 1.2 分层边界

- `Chips-Host` 主进程（插件管理域）负责：解析/校验包、安装目录管理、注册表写入、窗口创建、权限裁决
- Bridge 层负责：将插件请求转换为 IPC 调用，执行权限检查
- 插件（渲染进程/iframe）负责：业务 UI 和交互，不直接访问 Node/Electron API

### 1.3 运行时目录（用户设备）

```text
~/.chips/
├── plugins/
│   ├── apps/
│   ├── cards/
│   ├── layouts/
│   ├── modules/
│   └── themes/
├── registry/
│   └── plugin-registry.json
├── workspace/
│   ├── install-staging/
│   └── update-backups/
├── cache/
└── logs/
```

## 2. 五种插件类型定义

### 2.1 类型矩阵

| 类型 | type 值 | 是否独立窗口 | 运行容器 | 典型入口 | 典型用途 |
|---|---|---:|---|---|---|
| 应用插件 | `app` | 是 | BrowserWindow | `entry.main` | 编辑器、查看器、设置面板 |
| 卡片插件 | `card` | 否 | iframe | `entry.renderer` + `entry.editor` | 基础卡片渲染与编辑 |
| 布局插件 | `layout` | 否 | iframe | `entry.renderer` + 可选 `entry.editor` | 箱子布局渲染 |
| 模块插件 | `module` | 否 | iframe / ES Module | `entry.iframe` / `entry.module` | 视频播放器、Markdown 渲染器 |
| 主题插件 | `theme` | 否 | 无运行容器（资源注入） | `entry.tokens` / `entry.themeCss` | Token 与样式注入 |

### 2.2 编辑器扩展（Editor Extension）归属规则

编辑器扩展不是第六种 `type`。扩展采用以下规则表达：

- 仍使用 `type: module`
- 通过 `extension` 字段声明扩展槽位
- 通过 `hostPlugin` 指定宿主（例如 `chips-official.editor`）

```yaml
extension:
  kind: editor-layout # editor-layout | editor-panel | editor-feature
  slot: workbench.main
hostPlugin: chips-official.editor
```

### 2.3 插件 ID 规范

- 规范格式：`publisher.plugin-name`（小写，点分发行者，短横线分名称）
- 正则：`^[a-z0-9]+(?:[.-][a-z0-9]+)*\.[a-z0-9][a-z0-9-]*$`
- Windows 兼容要求：ID 不得包含 `:` `*` `?` `"` `<` `>` `|` 等路径非法字符

## 3. `manifest.yaml` 完整规格

### 3.1 顶层字段总表

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---:|---|---|
| `schemaVersion` | string | 是 | - | Manifest 规范版本，当前固定 `1.0.0` |
| `id` | string | 是 | - | 插件唯一标识 |
| `name` | string | 是 | - | 插件展示名（可本地化） |
| `version` | string | 是 | - | SemVer 版本 |
| `type` | enum | 是 | - | `app | card | layout | module | theme` |
| `publisher` | string | 是 | - | 发行方标识 |
| `description` | string | 否 | `""` | 描述 |
| `icon` | string | 否 | `assets/icon.png` | 图标路径（相对包根） |
| `entry` | object | 是 | - | 类型相关入口定义 |
| `permissions` | string[] | 否 | `[]` | 权限声明 |
| `capabilities` | object | 否 | `{}` | 能力声明 |
| `compatibility` | object | 是 | - | 主机/SDK/平台兼容范围 |
| `dependencies` | object | 否 | `{ plugins: [] }` | 插件依赖 |
| `locales` | object | 否 | - | 多语言配置 |
| `window` | object | 条件 | - | 仅 `app` 类型 |
| `extension` | object | 条件 | - | 编辑器扩展声明（`module` 可用） |
| `hostPlugin` | string | 条件 | - | 扩展宿主插件 ID |
| `exports` | array | 条件 | `[]` | `module` 导出声明 |
| `theme` | object | 条件 | - | `theme` 类型专用 |
| `checksum` | object | 否 | - | 文件校验（可选增强） |

### 3.2 按类型必填字段

| 字段 | app | card | layout | module | theme |
|---|---:|---:|---:|---:|---:|
| `entry.main` | 必填 | - | - | - | - |
| `entry.renderer` | - | 必填 | 必填 | - | - |
| `entry.editor` | - | 必填 | 可选 | - | - |
| `entry.module` | - | - | - | 二选一 | - |
| `entry.iframe` | - | - | - | 二选一 | - |
| `entry.tokens` | - | - | - | - | 必填 |
| `entry.themeCss` | - | - | - | - | 必填 |
| `window` | 必填 | - | - | - | - |
| `exports` | - | - | - | 可选（推荐） | - |
| `theme` | - | - | - | - | 必填 |

### 3.3 权限字段规范

权限名称采用 `namespace.action` 形式，示例：

- `file.read`
- `file.write`
- `resource.fetch`
- `plugin.read`
- `window.control`

权限校验链路：

1. `windowId -> pluginId`
2. 从注册表读取 `manifest.permissions`
3. 匹配 API 所需权限
4. 放行或返回标准错误：`{ code, message, details }`

### 3.4 多语言字段规范

`manifest.yaml` 允许本地化文本字段使用 `i18n key` 引用：

```yaml
name: i18n.plugin.600001
summary: i18n.plugin.600002
```

当字段为 key 时，主机在展示层调用 `i18n.translate` 解析；当字段为原始文本时，安装器会在验证阶段给出 warning（不阻断本地安装，阻断市场发布）。

### 3.5 Manifest TypeScript 定义

```ts
export type PluginType = 'app' | 'card' | 'layout' | 'module' | 'theme';

export interface PluginManifest {
  schemaVersion: '1.0.0';
  id: string;
  name: string;
  version: string;
  type: PluginType;
  publisher: string;
  description?: string;
  icon?: string;
  entry: PluginEntry;
  permissions?: string[];
  capabilities?: Record<string, unknown>;
  compatibility: {
    host: string;
    sdk?: string;
    platform?: Array<'win32' | 'darwin' | 'linux'>;
  };
  dependencies?: {
    plugins?: Array<{ id: string; version: string; optional?: boolean }>;
  };
  locales?: {
    default: string;
    supported: string[];
    path?: string;
  };
  window?: AppWindowConfig;
  extension?: {
    kind: 'editor-layout' | 'editor-panel' | 'editor-feature';

    inherits?: string;

}
```

### 3.6 Manifest Zod 校验 Schema（实现参考）

```ts
import { z } from 'zod';

const semver = z.string().regex(/^\d+\.\d+\.\d+(?:-[0-9A-Za-z-.]+)?(?:\+[0-9A-Za-z-.]+)?$/);
const semverRange = z.string().min(1);
const pluginId = z
  .string()
  .regex(/^[a-z0-9]+(?:[.-][a-z0-9]+)*\.[a-z0-9][a-z0-9-]*$/);
const relPath = z.string().min(1).refine((v) => !v.startsWith('/'), 'must be relative path');

const baseSchema = z.object({
  schemaVersion: z.literal('1.0.0'),
  id: pluginId,
  name: z.string().min(1),
  version: semver,
  type: z.enum(['app', 'card', 'layout', 'module', 'theme']),
  publisher: z.string().min(1),
  description: z.string().optional(),
  icon: relPath.optional(),
  permissions: z.array(z.string().regex(/^[a-z][a-z0-9-]*\.[a-z][a-z0-9-]*$/)).default([]),
  capabilities: z.record(z.unknown()).default({}),
  compatibility: z.object({
    host: semverRange,
    sdk: semverRange.optional(),
    platform: z.array(z.enum(['win32', 'darwin', 'linux'])).optional()
  }),
  dependencies: z
    .object({
      plugins: z
        .array(
          z.object({
            id: pluginId,
            version: semverRange,
            optional: z.boolean().optional()
          })
        )
        .default([])
    })
    .default({ plugins: [] }),
  locales: z
    .object({
      default: z.string().min(2),
      supported: z.array(z.string().min(2)).min(1),
      path: relPath.optional()
    })
    .optional(),
  extension: z
    .object({
      kind: z.enum(['editor-layout', 'editor-panel', 'editor-feature']),
      slot: z.string().min(1)
    })
    .optional(),
  hostPlugin: pluginId.optional(),
  exports: z.array(z.object({ name: z.string().min(1), description: z.string().optional() })).optional(),
  theme: z
    .object({
      inherits: pluginId.optional(),
      tokensVersion: semver
    })
    .optional()
});

const appSchema = baseSchema.extend({
  type: z.literal('app'),
  entry: z.object({ main: relPath }),
  window: z.object({
    width: z.number().int().positive(),
    height: z.number().int().positive(),
    minWidth: z.number().int().positive().optional(),
    minHeight: z.number().int().positive().optional(),
    maxWidth: z.number().int().positive().optional(),
    maxHeight: z.number().int().positive().optional(),
    resizable: z.boolean().optional(),
    minimizable: z.boolean().optional(),
    maximizable: z.boolean().optional(),
    frame: z.boolean().optional(),
    transparent: z.boolean().optional(),
    backgroundColor: z.string().optional()
  })
});

const cardSchema = baseSchema.extend({
  type: z.literal('card'),
  entry: z.object({ renderer: relPath, editor: relPath }),
  capabilities: z.object({ cardType: z.string().min(1) }).passthrough()
});

const layoutSchema = baseSchema.extend({
  type: z.literal('layout'),
  entry: z.object({ renderer: relPath, editor: relPath.optional() }),
  capabilities: z.object({ layoutType: z.string().min(1) }).passthrough()
});

const moduleSchema = baseSchema
  .extend({
    type: z.literal('module'),
    entry: z
      .object({
        module: relPath.optional(),
        iframe: relPath.optional()
      })
      .refine((v) => Boolean(v.module || v.iframe), 'module or iframe entry is required')
  })
  .superRefine((value, ctx) => {
    if (value.extension && !value.hostPlugin) {
      ctx.addIssue({ code: z.ZodIssueCode.custom, message: 'hostPlugin is required for extension module' });
    }
  });

const themeSchema = baseSchema.extend({
  type: z.literal('theme'),
  entry: z.object({ tokens: relPath, themeCss: relPath }),
  theme: z.object({
    inherits: pluginId.optional(),
    tokensVersion: semver
  })
});

export const manifestSchema = z.discriminatedUnion('type', [
  appSchema,
  cardSchema,
  layoutSchema,
  moduleSchema,
  themeSchema
]);
```

## 4. `.cpk` 包格式规格

### 4.1 格式定义

- `.cpk` 是 ZIP 容器，扩展名固定为 `.cpk`
- 归档根目录必须直接包含 `manifest.yaml`，禁止多一层根目录包裹
- 文件路径统一使用 `/` 分隔

### 4.2 推荐目录结构

```text
my-plugin-1.0.0.cpk
├── manifest.yaml
├── README.md
├── LICENSE
├── preview.png
├── index.html                      # app 可有
├── renderer/                       # card/layout 可有
├── editor/                         # card/layout 可有
├── dist/                           # module 可有
├── tokens/                         # theme 必需
├── styles/                         # theme 必需
├── locales/                        # 可选
└── assets/                         # 可选
```

### 4.3 必需文件清单（按类型）

| 类型 | 必需文件 |
|---|---|
| `app` | `manifest.yaml` + `entry.main` 指向文件 |
| `card` | `manifest.yaml` + `entry.renderer` + `entry.editor` |
| `layout` | `manifest.yaml` + `entry.renderer` |
| `module` | `manifest.yaml` + `entry.module` 或 `entry.iframe` |
| `theme` | `manifest.yaml` + `entry.tokens` + `entry.themeCss` |

### 4.4 可选文件清单

- `README.md`：插件说明
- `LICENSE`：许可证
- `CHANGELOG.md`：变更记录
- `preview.png`：市场预览图
- `locales/*.yaml|json`：插件自有词汇
- `SIGNATURE`：签名摘要文件

### 4.5 包体积限制

- 默认硬限制：`500 MB`
- 单文件硬限制：`200 MB`
- 超限策略：安装器直接拒绝并返回 `PLUGIN_PACKAGE_TOO_LARGE`

### 4.6 签名机制

签名校验分三级：

1. 本地安装（开发模式）可不签名
2. 本地安装（生产模式）签名可选，缺失时警告
3. 应用市场下载包签名必需，不通过则拒绝安装

签名文件约定：

```text
SIGNATURE                # 签名元信息（JSON）
SIGNATURE.sig            # 对摘要签名（Base64）
CERTIFICATE.pem          # 发布者证书
```

摘要算法：`sha256`，校验目标为包内所有文件（排除签名文件本身）的路径+内容哈希集合。

## 5. 插件安装流程

### 5.1 安装来源

- 应用市场下载
- 本地 `.cpk` 文件选择
- 拖拽 `.cpk` 到主机窗口
- CLI 命令安装

### 5.2 流程总览（事务化）

```text
接收请求
  -> 创建安装事务 txId
  -> 解包到 staging
  -> manifest 校验
  -> 文件完整性与签名校验
  -> 依赖/兼容性校验
  -> 冲突检查
  -> 原子写入目标目录
  -> 注册表更新
  -> 运行时索引刷新
  -> 创建快捷方式（app）
  -> 发布 plugin.installed
```

### 5.3 详细校验步骤

1. 包文件存在且可读
2. ZIP 结构合法（无目录穿越、无绝对路径）
3. 存在 `manifest.yaml`
4. Manifest schema 校验通过
5. 所有入口文件存在
6. `compatibility.host` 与当前主机版本匹配
7. 依赖插件满足版本范围
8. 权限列表合法
9. 文件体积与数量符合限制
10. 签名策略校验通过（按来源）

### 5.4 文件复制策略

- 先写入：`~/.chips/workspace/install-staging/{txId}`
- 再原子替换：`rename(staging, ~/.chips/plugins/{typeDir}/{pluginId})`
- 旧版本并存保护：同 ID 不同版本升级走更新流程，不覆盖安装

### 5.5 注册表更新策略

- 安装成功后写入 `plugin-registry.json`
- 写入采用 `write tmp -> fsync -> rename`，防止电源中断导致损坏
- 更新索引：
  - `cardType -> pluginId`
  - `layoutType -> pluginId`
  - `module exports -> pluginId@version`
  - `file extension -> handler pluginIds`

### 5.6 快捷方式创建（仅 app）

创建命令统一为：

```text
chips-host --open-plugin=<pluginId>
```

平台实现交由 `platform.*` 服务与 PAL 适配器处理。

### 5.7 失败回滚机制

回滚原则：任何失败都应回到“安装前快照”状态。

| 失败阶段 | 回滚动作 |
|---|---|
| Manifest 校验失败 | 清理 staging，事务失败 |
| 依赖/兼容性失败 | 清理 staging，事务失败 |
| 目标目录写入失败 | 删除半写入目录，恢复旧版本指针 |
| 注册表写入失败 | 回滚目录变更，恢复旧注册表备份 |
| 快捷方式创建失败 | 回滚注册表和目录，清理快捷方式残留 |

安装失败对外错误格式：

```json
{ "code": "PLUGIN_INSTALL_FAILED", "message": "插件安装失败", "details": { "txId": "...", "stage": "registry" } }
```

### 5.8 安装事件通知

- `plugin.installing`
- `plugin.installed`
- `plugin.install-failed`

事件体至少包含：`pluginId`、`version`、`txId`、`timestamp`。

## 6. 插件卸载流程

### 6.1 卸载前检查

1. 插件是否存在
2. 是否被其他插件依赖
3. 是否存在运行中实例

### 6.2 卸载执行

```text
依赖检查
  -> 停止运行实例
  -> 删除快捷方式（app）
  -> 从注册表删除
  -> 删除插件目录
  -> 刷新内存索引
  -> 发布 plugin.uninstalled
```

### 6.3 卸载保护策略

- 默认拒绝卸载被依赖插件，返回 `PLUGIN_HAS_DEPENDENTS`
- `force` 卸载需要显式参数，并同时停用依赖插件

### 6.4 卸载错误码

- `PLUGIN_NOT_FOUND`
- `PLUGIN_HAS_DEPENDENTS`
- `PLUGIN_STOP_FAILED`
- `PLUGIN_UNINSTALL_FAILED`

## 7. 插件更新流程

### 7.1 版本比较规则

- 使用 SemVer 比较
- 新版本必须 `>` 当前版本
- 同版本重复安装视为幂等成功（可选跳过）

### 7.2 更新总流程

```text
下载/接收新包
  -> 版本检查
  -> 备份旧版本
  -> 安装新版本（复用安装流程）
  -> 健康检查
  -> 数据迁移（如需）
  -> 清理备份 or 回滚
```

### 7.3 热更新与重启更新

| 类型 | 默认策略 |
|---|---|
| app | 需要重启该插件窗口 |
| card | 支持热替换，已有 iframe 在下次渲染使用新版本 |
| layout | 支持热替换，布局实例可选择重载 |
| module | 支持按消费者维度热替换（不强制全局重启） |
| theme | 支持热更新并触发主题重注入 |

### 7.4 备份与回滚

备份目录：`~/.chips/workspace/update-backups/{pluginId}/{oldVersion}`

回滚触发条件：

- 新版本健康检查失败
- 启动失败
- 迁移脚本失败

回滚动作：

1. 停止新版本运行
2. 恢复旧版本目录
3. 恢复注册表指针
4. 发送 `plugin.update-rolled-back`

### 7.5 数据迁移规范

若插件涉及持久化数据结构变更，包内应声明：

```yaml
migrations:
  - from: "1.1.0"
    to: "1.2.0"
    script: "migrations/1.1.0_to_1.2.0.js"
```

迁移脚本执行原则：

- 超时默认 30s
- 失败即回滚
- 迁移日志写入 `logs/plugin-migration.log`

## 9. 应用插件生命周期设计

### 9.1 状态机

`installed -> loading -> initializing -> running -> background -> suspended -> stopping -> stopped`

异常分支：任意状态可进入 `crashed` 或 `error`。

### 9.2 状态转换规则

| 当前状态 | 触发事件 | 下一个状态 | 处理动作 |
|---|---|---|---|
| `installed` | 用户/系统 `openPlugin` | `loading` | 创建 BrowserWindow，加载入口 |
| `loading` | `did-finish-load` | `initializing` | 发送 `chips:plugin-init` |
| `initializing` | 收到 `plugin-ready` | `running` | 开启事件订阅与能力会话 |
| `running` | 窗口最小化/隐藏 | `background` | 降低刷新频率 |
| `background` | 系统内存压力 | `suspended` | 挂起非关键任务 |
| `background`/`running` | 用户激活窗口 | `running` | 恢复任务与渲染 |
| `running`/`background` | 关闭窗口 | `stopping` | 保存状态、释放资源 |
| `stopping` | 清理完成 | `stopped` | 销毁窗口映射 |
| 任意 | 渲染崩溃 | `crashed` | 记录日志并可重启恢复 |
| 任意 | 不可恢复错误 | `error` | 标记异常并阻止自动重启 |

### 9.3 BrowserWindow 默认配置

```ts
const defaultWindowOptions = {
  width: 1200,
  height: 800,
  minWidth: 640,
  minHeight: 480,
  show: false,
  webPreferences: {
    preload: 'preload/bridge.js',
    contextIsolation: true,
    sandbox: true,
    nodeIntegration: false,
    webSecurity: true,
    allowRunningInsecureContent: false
  }
};
```

`manifest.window` 可覆盖尺寸和窗口行为，不得覆盖安全配置字段。

### 9.4 插件初始化数据

`chips:plugin-init` 负载：

```ts
interface PluginInitPayload {
  pluginId: string;
  pluginType: 'app' | 'card' | 'layout' | 'module' | 'theme';
  version: string;
  permissions: string[];
  launchParams: unknown;
  locale: string;
  themeId: string;
  hostVersion: string;
}
```

### 9.5 后台挂起与恢复

- 后台超过 10 分钟且无活动订阅，进入 `suspended`
- 恢复条件：窗口重新聚焦、收到高优先级事件、用户显式唤醒

### 9.6 崩溃检测与恢复

- 监听 Electron `render-process-gone`
- 记录错误上下文（`pluginId`, `reason`, `exitCode`）
- 派发事件：`plugin.crashed`
- 提供恢复策略：`manual-reopen` 或 `auto-reopen`（最多 3 次，指数退避）

## 10. 卡片插件 iframe 运行时协议

### 10.1 iframe 创建参数

```html
<iframe
  sandbox="allow-scripts allow-same-origin"
  referrerpolicy="no-referrer"
  loading="lazy"
></iframe>
```

约束：

- 禁止 `allow-top-navigation`
- 禁止 `allow-popups-to-escape-sandbox`
- iframe 页面必须从插件安装目录受控加载

### 10.2 消息包统一格式

```ts
interface CardRuntimeMessage<T = unknown> {
  protocol: 'chips-card-runtime';
  version: '1.0.0';
  messageId: string;
  type:
    | 'init'
    | 'ready'
    | 'bridge-request'
    | 'bridge-response'
    | 'resource-request'
    | 'resource-response'
    | 'resize'
    | 'theme-change'
    | 'dispose'
    | 'error';
  timestamp: string;
  payload: T;
}
```

### 10.3 宿主 -> iframe 消息

- `init`：初始化数据（卡片配置、主题、上下文）
- `bridge-response`：Bridge 调用结果
- `resource-response`：资源请求结果
- `theme-change`：主题变更通知
- `dispose`：销毁通知

### 10.4 iframe -> 宿主消息

- `ready`：渲染组件就绪
- `bridge-request`：请求宿主代调 `chips.invoke`
- `resource-request`：请求资源解析
- `resize`：内容高度变化通知
- `error`：组件内部异常上报

### 10.5 初始化时序

```text
宿主创建 iframe
  -> iframe load
  -> iframe 发送 ready
  -> 宿主发送 init
  -> iframe 完成首屏渲染
  -> iframe 可持续发送 resize/bridge-request
```

### 10.6 资源请求中转

1. iframe 发送 `resource-request`
2. 宿主应用调用 `chips.invoke('resource', 'fetch', ...)`
3. 返回 `resource-response`
4. iframe 仅接收可访问 URL，不暴露宿主文件系统绝对路径

## 11. 模块插件加载协议

### 11.1 加载模式

- `iframe` 模式：隔离性高，适合复杂 UI 模块
- `ES Module` 模式：性能更高，适合纯渲染/工具模块
- `dual` 模式：插件同时提供两种入口，由消费者选择

### 11.2 路由调用接口

```ts
await kernel.router.invoke('module', 'load', {
  moduleId: 'chips-module-video-player',
  requestedVersion: '^1.2.0',
  consumerPluginId: 'chips-official.video-card',
  preferredMode: 'esm'
});
```

### 11.3 返回结构

```ts
interface ModuleLoadResult {
  moduleId: string;
  resolvedVersion: string;
  mode: 'iframe' | 'esm';
  iframeUrl?: string;
  moduleUrl?: string;
  exports: string[];
  integrity: string;
  cacheKey: string;
}
```

### 11.4 缓存策略

- 缓存键：`moduleId@resolvedVersion#mode`
- 同版本同模式复用已加载实例
- 支持 LRU 回收（默认 32 项）

### 11.5 版本冲突处理

当两个消费者请求不兼容版本：

- 尽量求交集版本（满足双方 range）
- 无交集时并行加载多版本，按 `cacheKey` 隔离
- 注册表记录每个消费者的版本绑定关系

### 11.6 卸载与失效

- 引用计数归零后可释放模块实例
- 插件更新后，旧版本模块标记 `stale`，新请求不再命中旧缓存
