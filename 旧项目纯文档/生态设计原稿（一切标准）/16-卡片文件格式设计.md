# 卡片文件格式设计

## 文件本质

卡片文件的本质是一个 ZIP 压缩包，后缀名改为 .card。压缩时使用零压缩率，即存储模式不压缩，这样可以最快速地访问文件内容。用户双击 .card 文件时，系统自动用卡片查看器打开。如果将后缀名改为 .zip，可以用任何解压工具打开查看内部结构。

## 目录结构

卡片文件的根目录包含三个部分：.card 隐藏文件夹、content 文件夹，以及所有的资源文件。

.card 文件夹是核心配置目录，包含三个必需文件。metadata.yaml 存储卡片的元数据信息，包括卡片 ID、名称、创建时间、主题配置、标签等。structure.yaml 定义卡片的结构，记录包含哪些基础卡片以及它们的顺序。cover.html 定义卡片的封面显示。如果用户上传压缩包或图片作为封面，相关资源会保存在 cardcover 文件夹中。

content 文件夹存放所有基础卡片的内容配置。每个基础卡片有一个独立的 YAML 配置文件，文件名就是该卡片的 10 位 62 进制 ID，扩展名是 .yaml。比如配置文件可能叫 a1B2c3D4e5.yaml 或 Xy9Zw8Vt7U.yaml。这些配置文件记录了该基础卡片的类型、参数和内容信息。

所有资源文件都直接放在卡片根目录中，不使用分类子文件夹。资源包括图片、视频、音频、字幕、文档等各类文件。如果卡片中嵌入了其他复合卡片，被嵌入的 card 文件也保存在根目录中。

## 元数据文件

metadata.yaml 记录卡片本身的所有信息。包括卡片的唯一 ID，这是一个 10 位 62 进制编号，使用 0-9、a-z、A-Z 字符组成，全局唯一。卡片名称，最长支持 500 个字符。创建时间和最后修改时间，使用 ISO 8601 格式，UTC 时区。

卡片使用的主题包字段保存技术 ID（`themeId`），例如 `chips-official.default-theme`。界面可展示“薯片官方：默认主题”这类显示名，但文件中保存的主键必须是技术 ID。主题包决定卡片的视觉外观。

标签系统采用数组结构，每个标签可以包含多个词。简单标签只有一个词，比如"动漫"。复杂标签由多个词组合，比如时间标签["时间戳", "15:20"]，地点标签["地点", "日本", "东京"]。这让标签可以表达键值对和复杂信息。

可选的扩展属性包括可见性设置、是否允许下载、是否允许二次创作、是否允许评论、许可证类型、年龄分级、内容警告等。

文件信息部分由系统自动生成，记录卡片的总大小、文件总数、完整性校验值（SHA256 算法）。

用户的使用信息（统计、观看进度、浏览历史等）不保存在卡片文件中，而是保存在查看器和编辑器的本地存储中。卡片文件只包含内容本身的信息。

## 结构文件

structure.yaml 定义卡片包含哪些基础卡片以及它们的展示顺序。文件中有一个 structure 数组，按照渲染顺序列出所有基础卡片。每个条目包含两个信息：卡片的 ID 和卡片的类型名称。

ID 是 10 位 62 进制编号，对应 content 文件夹中的配置文件。类型名称决定了使用哪个渲染器插件来显示这个卡片。比如类型是 VideoCard，系统就会加载视频卡片渲染器。

结构文件还包含一个 manifest 清单部分，提供快速索引和完整性验证信息。清单记录了卡片包的基本信息，包含多少个基础卡片，文件统计信息，以及所有资源文件的详细列表。资源列表中每个文件都记录了路径、大小、类型，视频文件还会记录时长，字幕文件会记录语言。

结构文件不包含任何内容、样式或属性配置。它只负责定义结构关系和提供文件索引。具体的内容配置都保存在 content 文件夹的独立文件中。

## 内容配置

content 文件夹存放每个基础卡片的内容配置文件。配置文件采用扁平化存储，所有文件平铺在 content 目录下，不再嵌套子文件夹。

每个配置文件都包含卡片类型标识，告诉系统这是什么类型的卡片。可以包含主题设置字段，默认为空，如果某个基础卡片需要使用特定主题，在这里记录主题包标识，会覆盖卡片级别的主题。如果为空，则使用上级的主题设置。可以包含布局参数，比如高度模式（自适应或固定）、宽高比等。

不同类型的基础卡片有不同的配置结构。视频卡片的配置包含视频文件路径、封面图片、字幕列表、播放参数等。图片卡片的配置包含图片文件路径、标题说明、适配模式等。Markdown 卡片的配置包含 Markdown 文件路径或内联内容、是否显示目录、语法高亮主题等。

配置文件中的资源引用支持相对路径、绝对路径和 URL。相对路径是相对于卡片根目录的。访问嵌套卡片中的资源时，可以将卡片文件名识别为文件夹，比如 card001.card/content.rtf 这样的路径。

## 封面文件

cover.html 定义卡片的封面显示。封面是用户浏览卡片列表或箱子时看到的预览内容。

封面支持多种创建方式。用户可以直接上传一个 HTML 文件作为封面。可以在输入框中填写 HTML 代码。可以上传一个压缩包，系统会在 cardcover 文件夹中解压，并使用其中的 index.html 作为封面入口。可以上传图片，图片会保存在 cardcover 文件夹中，系统会生成一个包含这个图片的 HTML 页面作为封面。

无论采用哪种创建方式，最终都会转化为 .card/cover.html 文件。如果用户没有设置封面，系统使用默认封面：白色背景，卡片名称居中显示。

封面 HTML 在 iframe 沙箱中运行，有安全限制。封面不能访问父窗口，不能提交表单，不能弹出窗口，不能访问网络。封面只能访问卡片内的资源，使用相对路径引用 cardcover 文件夹或根目录中的文件。

## 资源组织

卡片文件支持三种资源模式：全填充卡片、半空壳卡片、空壳卡片。

全填充卡片是所有资源都打包在卡片文件内部的模式。视频、图片、音频等所有资源文件都保存在 .card 压缩包的根目录中。用户只需要一个 .card 文件就能完整使用，不依赖外部资源。这种模式适合分享和备份。

空壳卡片是所有资源都引用外部文件的模式。.card 文件中只保存配置信息，基础卡片的配置文件中记录的是外部资源的路径或网址。比如视频卡片配置中的 video_file 字段可能是 "/Users/name/Videos/video.mp4" 这样的绝对路径，或者是网络地址。卡片文件体积很小，但需要外部资源才能正常工作。

半空壳卡片是部分资源在内部、部分资源在外部的模式。压缩包根目录中有一些资源文件，同时配置中还引用了外部资源。这种模式让用户可以根据实际情况灵活组织，重要的小文件打包在内部，大型文件引用外部节省空间。

用户添加资源时选择处理方式，系统自动记录。导入文件时可以选择复制到卡片内部或只记录路径。引用网络资源时只能记录地址。系统通过检查配置文件中的路径格式，判断资源是内部还是外部。

资源文件的命名规则是：如果用户导入的文件本身有文件名，就保留原文件名。如果是用户在编辑引擎中创建的新内容，使用所属卡片的名称命名，必要时添加序号区分。当两个文件的文件名相同但内容不同时，系统自动在后面添加编号。

在基础卡片的配置文件中，通过路径引用资源。相对路径表示资源在卡片内部，比如 "episode01.mp4"。绝对路径表示资源在文件系统的其他位置。网址表示资源在网络上。系统根据路径格式自动判断资源位置。

资源缺失时，卡片仍可以打开，但缺失的内容显示占位符和提示。用户可以在卡片设置中重新绑定资源，指定新的文件路径，或者下载资源到卡片内部。

## 文件命名

文件命名需要遵循一些规则。使用 UTF-8 编码，支持中文文件名。避免使用操作系统的特殊字符。文件扩展名使用小写。空格用连字符或下划线代替。content/ 目录中的配置文件只使用十位 62 进制 ID 命名，扩展名是 .yaml。

路径引用始终使用相对路径，相对于卡片根目录。使用正斜杠作为路径分隔符，确保跨平台兼容。

## 版本兼容

卡片格式有版本号，记录在 metadata.yaml 的 chip_standards_version 字段中。版本号遵循语义化版本规范，格式是"主版本号.次版本号.修订号"。主版本号变更表示不兼容的格式变更，次版本号变更表示向后兼容的功能新增，修订号变更表示向后兼容的问题修正。

当前的卡片格式版本是 1.0.0。未来格式演进时，查看器需要检查卡片的版本号，决定如何解析。生态不追求大版本向后兼容，遇到旧版本卡片时，用户需要使用对应版本的查看器，或者使用格式转换工具升级。

## 卡片打包

公共基础层提供卡片打包器模块，负责卡片文件夹与 .card 文件之间的转换。编辑引擎在工作区中以解压后的文件夹形式存储卡片，编辑完成后通过打包器将文件夹打包成标准的 .card 文件。

打包时，打包器遍历卡片文件夹的所有内容，按照卡片文件格式规范组织目录结构。.card 配置文件夹放在压缩包的根目录，content 文件夹包含所有基础卡片的配置文件，资源文件直接放在根目录。打包器使用零压缩模式创建 ZIP 文件，确保文件可以快速访问，然后将扩展名改为 .card。

解包时，打包器将 .card 文件作为 ZIP 文件解压到指定目录。解压后的文件夹结构与打包前完全一致，编辑引擎可以直接读写文件夹中的内容。打包器会验证解压后的结构是否符合卡片格式规范，确保 .card 配置文件夹和必要的配置文件都存在。

打包器提供验证功能，检查卡片文件的结构是否正确。验证包括检查必需文件是否存在、配置文件格式是否正确、资源引用是否有效等。应用可以在打包前验证文件夹结构，确保生成的 .card 文件符合规范。

打包器还提供元数据读取功能，可以在不完全解包的情况下读取卡片的元数据。这让应用可以快速获取卡片的基本信息，比如名称、创建时间、基础卡片数量等，用于文件列表显示或预览。
