# 插件契约规范

> 文档状态：vNext 架构对齐修订稿
> 适用阶段：阶段18及后续重构实施阶段
> 修订日期：2026-03-01

---

## 0. 契约元信息

### 0.1 适用范围
- 本规范定义插件与系统之间的协议，涵盖接口契约、生命周期契约、资源契约、安全契约。
- 目标对象：插件开发者、Host 运行时维护者、SDK 维护者。

### 0.2 核心约束（vNext 锁定）
- 插件类型：`app/card/layout/module/theme` 五类
- 交互接口：`window.chips.*` Bridge API
- 路由格式：`namespace.action`
- 逻辑尺寸单位：`cpx`（基准 1024cpx）
- 插件初始化必须走 `plugin-init` 握手

## 1. 插件类型定义

### 1.1 五类插件

| 类型 | 英文标识 | 运行时隔离 | 核心能力 |
|------|---------|-----------|---------|
| 应用插件 | `app` | 独立 BrowserWindow | 独立窗口应用 |
| 卡片插件 | `card` | iframe 优先，受控模式 | 卡片渲染/编辑 |
| 布局插件 | `layout` | 模块级隔离 | 卡片排列与容器 |
| 模块插件 | `module` | Host 托管隔离 | 无界面能力扩展 |
| 主题插件 | `theme` | 只读注入 | 视觉风格定义 |

### 1.2 插件会话字段
每个插件会话必须包含以下字段：

```typescript
interface PluginSession {
  pluginId: string;           // 插件唯一标识
  pluginType: PluginType;     // app | card | layout | module | theme
  permissions: string[];      // 已授权权限列表
  locale: string;              // 当前语言
  themeId: string;             // 当前主题
  launchParams: Record<string, unknown>;  // 启动参数
  sessionNonce: string;       // 会话唯一随机数
}
```

## 2. 接口契约

> vNext 对齐：采用 `window.chips.*` Bridge API，路由格式 `namespace.action`

### 2.1 Bridge API 入口

插件访问系统能力的主要接口为 `window.chips.*`，包含以下子域：

| 子域 | 核心能力 |
|------|---------|
| `window.chips.invoke` | 路由调用（`namespace.action` 格式） |
| `window.chips.on` | 事件订阅 |
| `window.chips.once` | 单次事件监听 |
| `window.chips.emit` | 事件发射 |
| `window.chips.window.*` | 窗口管理 |
| `window.chips.dialog.*` | 对话框操作 |
| `window.chips.plugin.*` | 插件管理 |
| `window.chips.clipboard.*` | 剪贴板操作 |
| `window.chips.shell.*` | 外部打开 |

### 2.2 路由命名规范

- 统一键格式：`` `${namespace}.${action}` ``
- 支持匹配模式：`exact` / `prefix` / `wildcard`
- 每路由可声明：`timeoutMs`、`invokeAccess`、`permission`、`schemaId`

### 2.3 调用上下文标准

`RouteInvocationContext` 固定字段：

```typescript
interface RouteInvocationContext {
  requestId: string;                    // 请求唯一标识
  caller: {
    id: string;
    type: 'app' | 'card' | 'layout' | 'module' | 'theme';
    pluginId?: string;
    windowId?: string;
    permissions?: string[];
  };
  timestamp: number;                      // Unix 时间戳
  deadline: number;                      // 超时截止时间
}
```

### 2.4 事件接口

事件命名采用点语义（如 `theme.changed`、`language.changed`），禁止使用冒号形式（`theme:changed`）。

## 3. 生命周期契约

> vNext 对齐：采用标准状态机 `unloaded -> loading -> ready -> running -> stopping -> stopped/error`

### 3.1 状态机定义

| 状态 | 含义 | 可转换至 |
|------|------|---------|
| `unloaded` | 未加载 | `loading` |
| `loading` | 加载中 | `ready` / `stopped` |
| `ready` | 就绪 | `running` / `stopped` |
| `running` | 运行中 | `stopping` |
| `stopping` | 停止中 | `stopped` |
| `stopped` | 已停止 | - |
| `error` | 错误状态 | `unloaded` |

### 3.2 plugin-init 握手

插件初始化必须走 `plugin-init` 握手流程：

1. **Manifest 校验** - 类型、权限、入口、契约版本
2. **签名与来源校验** - 官方/第三方渠道
3. **启动握手** - 分配 `sessionNonce`
4. **权限快照加载** - 最小权限原则
5. **沙箱注入** - 只暴露 `window.chips.*`

握手失败禁止进入 `running` 状态。

### 3.3 各阶段契约

| 阶段 | 契约要求 |
|------|---------|
| 安装 | 提供完整元数据（名称、版本、依赖、权限、`ui.layout` 声明；应用插件可选 `ui.window.chrome`；主题插件必须声明 `entry.tokens/themeCss`） |
| 激活 | 初始化资源，完成 `plugin-init` 握手 |
| 运行 | 响应事件和请求，保持响应状态 |
| 停用 | 保存状态，释放资源，清理定时器/监听器 |

## 4. 资源契约

> vNext 对齐：定义 CPU/内存/消息频率软配额，超配额触发降级

### 4.1 资源配额限制

| 资源类型 | 配额类型 | 超出处理 |
|---------|---------|---------|
| 内存 | 软配额 | 触发垃圾回收或降级 |
| CPU | 软配额 | 限频或暂停后台任务 |
| 存储 | 硬配额 | 拒绝写入，提示清理 |
| 网络请求 | 软配额 | 请求被拦截或延迟 |
| 消息频率 | 软配额 | 触发限频 |

### 4.2 降级策略

- **限频**：降低事件处理频率
- **暂停后台任务**：暂停非关键任务
- **告警**：通知开发者资源使用情况

## 5. 安全契约

> vNext 对齐：采用插件安全流水线7步流程

### 5.1 插件安全流水线

| 步骤 | 名称 | 职责 |
|------|------|------|
| 1 | Manifest 校验 | 类型、权限、入口、契约版本 |
| 2 | 签名与来源校验 | 官方/第三方渠道验证 |
| 3 | 启动握手 | `plugin-init`，分配 `sessionNonce` |
| 4 | 权限快照加载 | 最小权限集合 |
| 5 | 沙箱注入 | 只暴露 `window.chips.*` |
| 6 | 运行时审计 | 越权、超频、异常事件 |
| 7 | 会话结束清理 | 订阅、缓存、临时资源 |

### 5.2 权限声明

- 插件在 manifest 中显式声明所需权限
- 权限校验在 Host IPC 入站点执行
- 危险能力（文件写、外部打开、凭证访问）必须显式授权

### 5.3 敏感能力隔离

`credential`、`shell`、`window` 高风险动作需独立审计日志。插件越权调用返回标准 `PERMISSION_DENIED` 错误。

## 6. 版本契约

> vNext 对齐：遵循语义化版本规范，补充 manifest 固定字段

### 6.1 语义化版本

- **主版本**：不兼容变更时递增
- **次版本**：向后兼容新功能时递增
- **修订号**：向后兼容问题修复时递增

### 6.2 插件 manifest 固定字段

vNext 要求所有插件 manifest 必须包含以下布局声明：

```yaml
ui:
  layout:
    owner: page
    unit: cpx
    baseWidth: 1024
    contract: ./contracts/page-layout.contract.json
    minFunctionalSet: ./contracts/min-functional-set.json
```

应用插件如需声明原生窗口外观，可额外配置：

```yaml
ui:
  window:
    chrome:
      backgroundColor: "#ffffff"
      titleBarStyle: hidden
      titleBarOverlay:
        color: "#ffffff00"
        symbolColor: "#667085"
        height: 44
```

主题插件还必须声明运行时入口：

```yaml
entry:
  tokens: dist/tokens.json
  themeCss: dist/theme.css
```

### 6.3 版本检测

版本检测在插件加载时执行。系统检查当前系统版本是否在插件声明的兼容范围内。不兼容时会阻止插件运行或提示升级。

### 6.4 契约测试要求

每个插件必须存在同名契约测试：`<action>.contract.test.ts`。契约测试验证插件与系统的接口兼容性。

## 7. 冲突解决

> vNext 对齐：跨插件共享仅允许通过 Host 服务快照/事件广播/明确协作契约

### 7.1 冲突类型

| 冲突类型 | 处理方式 |
|---------|---------|
| 资源冲突 | 按优先级分配，低优先级等待或降级 |
| 功能冲突 | 用户选择，系统提示 |
| 依赖冲突 | 系统尝试解析兼容版本组合 |

### 7.2 跨插件共享边界

禁止插件直接共享可变内存引用。跨插件共享仅允许通过：
- Host 服务快照（可审计）
- 事件广播（只读载荷）
- 明确声明的协作契约（schema + permission）

## 8. 违约处理

### 8.1 处理等级

| 等级 | 适用场景 | 处理方式 |
|------|---------|---------|
| 警告 | 轻微违规（资源略高、响应略慢） | 通知开发者限时修复 |
| 停用 | 严重违规（安全违规、持续无响应） | 停用插件并通知用户 |
| 卸载 | 无法修复的违规或恶意插件 | 强制卸载并清理数据 |

---

## 9. 契约变更

### 9.1 变更原则

- 重大变更提前通知，提供充足适配时间
- 遵循向后兼容原则
- 废弃 API 保留过渡期

### 9.2 废弃流程

1. **废弃声明** - 标记将要删除的 API
2. **过渡期** - 返回警告信息，提示使用新 API
3. **移除** - 在公告后的版本移除
