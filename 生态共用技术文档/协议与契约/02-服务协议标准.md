# 服务协议标准

> 文档状态：vNext 架构对齐修订稿
> 适用阶段：阶段18及后续重构实施阶段
> 修订日期：2026-03-01

---

## 0. 协议元信息

### 0.1 适用范围
- 本标准定义 Host 内核、Host 服务域、插件运行时之间的通信规范
- 目标对象：Kernel 维护者、Service 开发者、SDK 维护者

### 0.2 核心约束（vNext 锁定）
- **中心路由原则**：服务间调用必须走内核路由，禁止跨服务直接 import
- **Bridge 隔离原则**：插件仅通过 `window.chips.*` 与系统能力交互
- **调用上下文**：`requestId` + `caller` + `timestamp` + `deadline`

### 0.3 架构归属声明（2026-03-06）
- Runtime Client、UI Hooks 及其上游渲染链路属于 Host 内置运行时能力。
- SDK 仅作为开发者工具包参与封装与类型约束，不承载协议运行时主实现。
- 用户运行环境仅依赖 Host；SDK 不作为运行时安装前提。

## 1. 全栈分层总览

> vNext 12层架构

| 层级 | 名称 | 核心职责 |
|------|------|---------|
| L1 | PAL | 平台能力抽象 |
| L2 | Host Kernel | router/registry/event/lifecycle |
| L3 | Host Services | file/resource/config/theme/i18n 等 |
| L4 | Plugin Runtime | 插件加载、隔离、会话、权限 |
| L5 | Bridge Transport | `window.chips.*` IPC 传输 |
| L6 | Runtime Client | 协议编解码、错误归一、重试 |
| L7 | UI Hooks | 面向页面的能力调用接口 |

### 1.1 依赖方向白名单
- 仅允许 `Ln -> Ln-1` 或 `Ln -> 公共契约层`
- 禁止跨层跳跃（例如 L7 直调 L5）

## 2. Bridge 三层架构

> vNext 核心架构：替代传统 HTTP/gRPC 传输

### 2.1 Bridge Transport（第五层）

对外入口：`window.chips.invoke/on/once/emit` + 子域 API

| 子域 | 能力 |
|------|------|
| `window.chips.invoke` | 路由调用（IPC 主通道 `chips:invoke`） |
| `window.chips.on/once/emit` | 事件订阅与发射 |
| `window.chips.window.*` | 窗口管理（子能力通道 `chips:window:*`） |
| `window.chips.dialog.*` | 对话框（子能力通道 `chips:dialog:*`） |
| `window.chips.plugin.*` | 插件管理（子能力通道 `chips:plugin:*`） |
| `window.chips.clipboard.*` | 剪贴板 |
| `window.chips.shell.*` | 外部打开 |

### 2.2 Runtime Client（第六层）

职责：
- 参数规范化、协议映射
- 错误归一（映射到 `BRIDGE_*` / `SERVICE_*` / `RUNTIME_*`）
- 超时控制
- 重试与回退（仅用于历史差异吸收，长期收敛为单动作）

### 2.3 UI Hooks（第七层）

提供的 Hook：
- `useTheme` / `useI18n` / `useFile` / `useWindow`
- `usePlugin` / `useConfig` / `useCard` / `useBox`

Hook 仅允许调用 Runtime Client，不可直连 Transport。

## 3. 路由契约描述符

> vNext 核心：所有服务注册必须提供 `RouteDescriptor`

### 3.1 RouteDescriptor 接口

```typescript
interface RouteDescriptor<I, O> {
  key: `${string}.${string}`;           // namespace.action 格式
  schemaIn: string;                     // 输入 JSON Schema ID
  schemaOut: string;                    // 输出 JSON Schema ID
  permission: string[];                 // 所需权限
  timeoutMs: number;                    // 超时时间
  idempotent: boolean;                  // 是否幂等
  retries: 0 | 1 | 2 | 3;               // 重试次数
  handler: (input: I, ctx: RouteInvocationContext) => Promise<O>;
}
```

### 3.2 RouteInvocationContext

```typescript
interface RouteInvocationContext {
  requestId: string;
  caller: {
    id: string;
    type: 'app' | 'card' | 'layout' | 'module' | 'theme';
    pluginId?: string;
    windowId?: string;
    permissions?: string[];
  };
  timestamp: number;
  deadline: number;
}
```

### 3.3 内核路由行为

- 参数 schema 校验
- 权限判定
- 路由级超时必配（超时报 `ROUTE_TIMEOUT`）
- 高频失败路由触发软熔断（窗口期内快速失败 + 可恢复探测）

## 5. 统一错误码标准

> vNext 标准：错误映射分层

### 5.1 错误分层

| 层级 | 错误前缀 | 说明 |
|------|---------|------|
| Bridge 层 | `BRIDGE_*` | IPC 传输层错误 |
| Service 层 | `SERVICE_*` | 服务业务逻辑错误 |
| Runtime 层 | `RUNTIME_*` | 超时/重试/熔断错误 |
| UI 层 | `UI_*` | 页面级错误 |

### 5.2 标准错误对象

```typescript
interface StandardError {
  code: string;           // 错误码
  message: string;        // 人类可读错误描述
  details?: unknown;     // 调试信息
  retryable?: boolean;   // 是否可重试
}
```

### 5.3 错误码示例

| 错误码 | 说明 | 可重试 |
|--------|------|--------|
| `BRIDGE_TIMEOUT` | Bridge 调用超时 | 是 |
| `BRIDGE_CONNECTION_LOST` | 连接断开 | 是 |
| `SERVICE_FILE_NOT_FOUND` | 文件不存在 | 否 |
| `SERVICE_PERMISSION_DENIED` | 权限不足 | 否 |
| `RUNTIME_ROUTE_TIMEOUT` | 路由超时 | 是 |
| `RUNTIME_CIRCUIT_OPEN` | 熔断器打开 | 是 |

---

## 6. 重试策略

- 可重试错误（`retryable=true`）才允许自动重试
- 默认指数退避：`200ms * 2^n`，最大 3 次
- 幂等动作可安全重试，非幂等动作需防重放策略（requestId 去重窗口）

---

## 7. 服务动作统一约束

每个动作必须定义：
- 输入 schema
- 输出 schema
- 权限
- 超时
- 错误码

幂等动作标注 `idempotent=true`。

---

## 8. 熔断机制

### 8.1 熔断位置
- 熔断在路由级内核层实现（不是服务协议层）

### 8.2 熔断状态
- **关闭**：正常转发请求
- **打开**：快速失败，不调用服务
- **半开**：允许试探性请求

### 8.3 恢复逻辑
- 半开状态探测成功后自动关闭
- 探测失败则保持打开状态

## 9. 监控追踪

### 9.1 统一日志字段

| 字段 | 说明 |
|------|------|
| traceId | 链路追踪 ID |
| requestId | 请求唯一标识 |
| pluginId | 插件标识 |
| namespace | 服务命名空间 |
| action | 动作名称 |
| durationMs | 耗时 |
| result | 结果状态 |
| errorCode | 错误码 |

### 9.2 链路追踪
- Transport 入站生成 `requestId`，贯穿 Runtime Client、Kernel、Service
- 事件链路记录 `source -> target`

## 10. 服务实现统一模板

每服务必须导出：
- `types.ts`（输入输出类型）
- `constants.ts`（动作名/错误码）
- `service.ts`（注册与实现）

每动作必须存在同名契约测试：`<action>.contract.test.ts`。

服务 `initialize()` 失败必须阻断系统进入 `running`。
