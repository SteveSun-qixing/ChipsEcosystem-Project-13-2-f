## 0. 协议元信息

### 0.1 适用范围
- 本协议定义薯片生态中各种文件类型的结构和规范

### 0.2 核心约束（vNext 锁定）
- **逻辑尺寸单位**：`cpx`（基准 1024cpx）
- **页面自治**：每页面定义自己的断点，不使用全局横竖模板

## 1. 卡片文件格式

卡片文件是内容存储的基本单元，扩展名为 `.card`。卡片文件采用 ZIP 格式压缩，内部包含元数据、结构和内容三部分。

元数据文件 `card/metadata.yaml` 存储卡片的标识、名称、版本、作者、创建时间等基本信息。元数据采用 YAML 格式，便于人类阅读和编辑。系统会根据元数据生成卡片的标识和验证信息。

结构文件 `card/structure.yaml` 定义卡片的内容组成，包括卡片类型、数据字段、引用资源等。结构信息帮助系统正确解析卡片内容，也支持编辑器的可视化编辑。

内容目录 `card/content/` 存储卡片的实际数据。数据可以是文本、图像、视频等多媒体内容。内容组织支持全引用、半引用、全内联三种模式，适应不同的使用场景。

### 1.1 卡片布局契约（vNext 新增）

卡片文件可选择包含布局声明：

```yaml
ui:
  layout:
    owner: page
    unit: cpx
    baseWidth: 1024
    contract: ./contracts/card-layout.contract.json
    minFunctionalSet: ./contracts/min-functional-set.json
```

- `owner`：布局所有者，`page` 表示页面级自治
- `unit`：逻辑尺寸单位，必须为 `cpx`
- `baseWidth`：基准宽度，固定为 `1024`
- `contract`：布局契约文件路径
- `minFunctionalSet`：最小功能集文件路径

## 箱子文件格式

箱子文件是卡片集合的容器，扩展名为。box。箱子文件采用ZIP格式压缩，内部可以包含多个卡片文件。

元数据文件box/metadata.yaml存储箱子的标识、名称、版本、类型等基本信息。箱子类型包括空箱子、半空箱子、全箱子三种，不同类型影响卡片的管理方式。

结构文件box/structure.yaml定义箱子包含的卡片列表和顺序。卡片列表记录每个卡片的引用路径，可以指向箱内卡片或外部卡片。

布局文件 `box/layout.yaml` 描述卡片在箱子中的排列方式。布局信息包括布局类型、卡片位置、尺寸等信息。布局由布局插件负责渲染。

### 2.1 箱子布局契约（vNext 新增）

箱子文件可选择包含布局声明：

```yaml
ui:
  layout:
    owner: page
    unit: cpx
    baseWidth: 1024
    contract: ./contracts/box-layout.contract.json
    minFunctionalSet: ./contracts/min-functional-set.json
```

## 插件文件格式

插件文件是功能扩展的封装，扩展名为。cpk。插件文件采用ZIP格式封装，包含清单文件和代码资源两部分。

清单文件 `manifest.yaml` 定义插件的标识、名称、版本、类型、入口点、依赖、权限等元数据。清单是插件系统识别和管理插件的基础。

插件包目录基线如下：

```text
<plugin>.cpk
├─ manifest.yaml
└─ dist/
   └─ ...
```

- `manifest.yaml` 位于包根目录；
- 运行时代码与构建产物位于 `dist/`；
- 不应在包内重复包含 `manifest.yaml` 的构建副本；
- 不应嵌套历史 `.cpk` 包或发布辅助文件。

### 3.1 插件 manifest 固定字段（vNext 新增）

所有插件必须包含以下布局声明：

```yaml
ui:
  layout:
    owner: page
    unit: cpx
    baseWidth: 1024
    contract: ./contracts/page-layout.contract.json
    minFunctionalSet: ./contracts/min-functional-set.json
```

### 3.2 插件类型声明

插件 manifest 必须声明类型：

```yaml
type: app | card | layout | module | theme
```

代码目录 `dist/` 存储插件的可执行代码。应用插件和卡片插件的代码运行在渲染进程。模块插件的代码可被其他插件引用。

资源目录 `assets/` 存储插件需要的静态资源，如图片、字体、国际化文件等。资源在运行时被加载器获取。

## 资源文件格式

资源文件是多媒体内容的存储形式，包括图像、音频、视频、文档等多种类型。资源文件独立存储，通过哈希值标识。

图像资源支持PNG、JPEG、WebP、SVG等格式。系统根据使用场景选择合适格式，如照片用JPEG，图标用PNG。图像可以包含多个分辨率版本，适应不同显示需求。

音频资源支持MP3、AAC、OGG、WAV等格式。音频文件可以包含元数据，如艺术家、专辑、时长等。系统支持流式播放和完整下载两种模式。

视频资源支持MP4、WebM、MKV等格式。视频可以包含多个分辨率和编码版本，适应不同网络条件。视频支持预览图和字幕。

## 文件校验机制

文件校验确保文件内容完整和来源可信。每个文件包含校验信息，读取时验证文件是否被篡改。

校验使用SHA-256哈希算法，生成文件内容的哈希值。哈希值存储在元数据中，读取时重新计算并比较。哈希不匹配表示文件已损坏或被篡改。

数字签名验证文件来源。发布者使用私钥对文件签名，系统使用公钥验证签名。签名验证确保文件来自官方发布，没有被恶意替换。

文件版本记录文件的修改历史。每次修改生成新的版本号，旧版本可以回滚。版本历史存储在元数据中，可以查看和比较。

## 文件加密机制

文件加密保护敏感内容的机密性。加密文件需要密钥才能解密读取。密钥管理确保只有授权用户可以访问文件内容。

加密使用AES-256算法，对称加密效率高。密钥使用用户密码或密钥文件保护。加密和解密过程对用户透明。

密钥派生从用户密码生成加密密钥。派生使用PBKDF2或Argon2算法，添加盐值防止彩虹表攻击。派生过程需要一定计算量，增加破解难度。

文件传输使用加密通道。HTTPS确保传输过程不被窃听。端到端加密可以进一步保护文件安全。

## 文件压缩策略

文件压缩减少存储空间和传输时间。不同类型文件使用不同压缩策略，平衡压缩率和性能。

打包文件如 `.card`、`.box`、`.cpk` 使用 ZIP 存储，不压缩。内容已经是压缩格式，再次压缩收益不大。零压缩加快打包和解包速度。

对 `.cpk` 而言，这里指 ZIP Store 模式。Host 当前安装器按 Store 模式解析本地文件头；若插件包使用 Deflate 等压缩方法，安装阶段会报 `PLUGIN_PACKAGE_INVALID`。

文本内容使用gzip压缩。gzip对重复模式多的文本效果好。压缩在传输时由HTTP协议自动处理。

媒体文件使用内容本身的压缩格式。图像使用WebP或AVIF等现代格式。视频使用H.264或H.265等高效编码。

## 文件命名规范

文件命名规范确保文件标识的唯一性和可读性。不同类型文件有不同的命名规则，便于识别和管理。

卡片和箱子文件使用ID作为文件名。ID是Base62编码的唯一标识符。文件名不包含中文或特殊字符，避免跨平台问题。

插件文件使用反向域名和版本号命名。如com.example.plugin-1.0.0.cpk。反向域名确保全球唯一性，版本号便于管理。

资源文件使用内容哈希作为文件名。哈希确保同名文件内容相同。系统可以自动去重，节省存储空间。

## 文件生命周期

文件生命周期管理文件的创建、使用、归档、删除等阶段。不同阶段有不同的存储策略和管理规则。

创建阶段文件刚生成，需要验证格式和内容。系统检查文件是否符合规范，计算校验信息。创建失败时给出明确错误提示。

使用阶段文件被频繁访问，系统优化读写性能。热数据保持在高速存储，冷数据可以迁移到低速存储。访问统计用于优化存储策略。

归档阶段文件不再活跃使用，转入长期保存。归档文件可以压缩存储，节省空间。归档文件仍可查询和恢复。

删除阶段文件被永久移除。删除前确认用户意图，防止误删。删除执行安全擦除，数据无法恢复。
