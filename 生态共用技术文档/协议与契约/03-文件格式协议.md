## 0. 协议元信息

### 0.1 适用范围
- 本协议定义薯片生态中各种文件类型的结构和规范
- 目标对象：文件解析器开发者、插件打包工具维护者

### 0.2 核心约束（vNext 锁定）
- **逻辑尺寸单位**：`cpx`（基准 1024cpx）
- **页面自治**：每页面定义自己的断点，不使用全局横竖模板
- **布局契约归属**：承载页面的插件或文档入口需声明 `ui.layout`；箱子文件本体通过 `active_layout_type + layout_configs` 关联布局，不直接声明 `ui.layout`

## 1. 卡片文件格式

卡片文件是内容存储的基本单元，扩展名为 `.card`。卡片文件采用 ZIP Store 模式封装，内部正式目录结构如下：

```text
<card>.card
├─ .card/
│  ├─ metadata.yaml
│  ├─ structure.yaml
│  ├─ cover.html
│  └─ cardcover/
└─ content/
   └─ *.yaml
```

元数据文件 `.card/metadata.yaml` 存储卡片的标识、名称、版本、作者、创建时间、封面比例等基本信息。元数据采用 YAML 格式，便于人类阅读和编辑。系统会根据元数据生成卡片的标识和验证信息。

结构文件 `.card/structure.yaml` 定义卡片的内容组成，包括基础卡片类型、数据字段、引用资源等。结构信息帮助系统正确解析卡片内容，也支持编辑器的可视化编辑。

内容目录 `content/` 存储卡片的实际数据。数据可以是文本、图像、视频等多媒体内容。内容组织支持全引用、半引用、全内联三种模式，适应不同的使用场景。

封面文件 `.card/cover.html` 是卡片封面的正式入口；若封面编辑器生成附属资源，则写入 `.card/cardcover/`。

补充约束：

- 解包态卡片工作目录与打包态 `.card` 内部目录结构必须保持一致；
- 正式资源路径以卡片根目录为基准记录，不额外嵌套 `card/` 前缀；
- Host `card.pack / card.unpack / card.readMetadata` 的解析口径必须与上述目录结构一致。

### 1.1 卡片布局契约（vNext 新增）

卡片文件可选择包含布局声明：

```yaml
ui:
  layout:
    owner: page
    unit: cpx
    baseWidth: 1024
    contract: ./contracts/card-layout.contract.json
    minFunctionalSet: ./contracts/min-functional-set.json
```

- `owner`：布局所有者，`page` 表示页面级自治
- `unit`：逻辑尺寸单位，必须为 `cpx`
- `baseWidth`：基准宽度，固定为 `1024`
- `contract`：布局契约文件路径
- `minFunctionalSet`：最小功能集文件路径

## 2. 箱子文件格式

箱子文件是卡片 URL 引用集合的正式消费容器，扩展名为 `.box`。箱子文件采用 ZIP Store 模式封装，内部正式目录结构如下：

```text
<box>.box
├─ .box/
│  ├─ metadata.yaml
│  ├─ structure.yaml
│  └─ content.yaml
└─ assets/
   └─ ...
```

元数据文件 `.box/metadata.yaml` 存储箱子标识、名称、创建时间、修改时间、标签和当前生效布局类型等基础信息。

结构文件 `.box/structure.yaml` 记录条目清单。每个条目都以 `entry_id + url` 为核心键，允许附带 `snapshot` 与 `layout_hints`，用于摘要展示、排序、分组和离线提示。

内容文件 `.box/content.yaml` 记录布局配置。顶层正式字段为 `active_layout_type` 与 `layout_configs`。布局插件的自定义参数保存在 `layout_configs[layoutType].props`，布局所需的箱子资源路径保存在 `asset_refs`。

`assets/` 是可选目录，仅保存箱子自有资源，例如布局背景图、占位图、摘要封面快照和布局私有附属文件。

正式约束：

- `.box` 内不保存被引用卡片实体，不嵌入 `*.card`；
- 箱子条目统一记录 URL，不再区分“内部 / 外部”；
- 不再区分三种箱子类型；
- 不再使用 `box/layout.yaml`；
- 箱子运行时资源解析由 Host 统一负责，布局插件只通过正式运行时契约取数。

### 2.1 箱子布局关联

箱子文件本身不声明 `ui.layout`。  
箱子的页面布局通过以下两层信息关联：

1. `metadata.yaml` / `content.yaml` 中的 `active_layout_type`；
2. `content.yaml` 中按 `layoutType` 建立的 `layout_configs`。

真正的页面布局契约由布局插件自身在 `manifest.yaml` 的 `ui.layout` 中声明。  
箱子消费阶段的 Host / 查看器 / 布局插件协作方式，以 `生态共用技术文档/协议与契约/08-箱子布局运行时契约.md` 为准。

## 3. 插件文件格式

插件文件是功能扩展的封装，扩展名为 `.cpk`。插件文件采用 ZIP 格式封装，包含清单文件和代码资源两部分。

清单文件 `manifest.yaml` 定义插件的标识、名称、版本、类型、入口点、依赖、权限等元数据。清单是插件系统识别和管理插件的基础。

插件包目录基线如下：

```text
<plugin>.cpk
├─ manifest.yaml
└─ dist/
   └─ ...
```

- `manifest.yaml` 位于包根目录；
- 运行时代码与构建产物位于 `dist/`；
- 不应在包内重复包含 `manifest.yaml` 的构建副本；
- 不应嵌套历史 `.cpk` 包或发布辅助文件。

### 3.1 插件 manifest 固定字段（vNext 新增）

所有插件必须包含以下布局声明：

```yaml
ui:
  layout:
    owner: page
    unit: cpx
    baseWidth: 1024
    contract: ./contracts/page-layout.contract.json
    minFunctionalSet: ./contracts/min-functional-set.json
```

应用插件可选声明窗口外观：

```yaml
ui:
  window:
    chrome:
      backgroundColor: "#ffffff"
      titleBarStyle: hidden
      titleBarOverlay:
        color: "#ffffff00"
        symbolColor: "#667085"
        height: 44
```

主题插件的 `entry` 字段必须为对象，并显式指向运行时产物：

```yaml
entry:
  tokens: dist/tokens.json
  themeCss: dist/theme.css
```

### 3.2 插件类型声明

插件 manifest 必须声明类型：

```yaml
type: app | card | layout | module | theme
```

代码目录 `dist/` 存储插件的可执行代码。应用插件和卡片插件的代码运行在渲染进程。模块插件的代码由 Host 加载并托管为无界面能力运行时，不作为其他插件直接 `import` 的正式边界。

资源目录 `assets/` 存储插件需要的静态资源，如图片、字体、国际化文件等。资源在运行时被加载器获取。

## 4. 资源文件格式

资源文件是多媒体内容的存储形式，包括图像、音频、视频、文档等多种类型。资源文件独立存储，通过哈希值标识。

图像资源支持 PNG、JPEG、WebP、SVG 等格式。系统根据使用场景选择合适格式，如照片用 JPEG，图标用 PNG。图像可以包含多个分辨率版本，适应不同显示需求。

音频资源支持 MP3、AAC、OGG、WAV 等格式。音频文件可以包含元数据，如艺术家、专辑、时长等。系统支持流式播放和完整下载两种模式。

视频资源支持 MP4、WebM、MKV 等格式。视频可以包含多个分辨率和编码版本，适应不同网络条件。视频支持预览图和字幕。

## 5. 文件校验机制

文件校验确保文件内容完整和来源可信。每个文件包含校验信息，读取时验证文件是否被篡改。

校验使用 SHA-256 哈希算法，生成文件内容的哈希值。哈希值存储在元数据中，读取时重新计算并比较。哈希不匹配表示文件已损坏或被篡改。

数字签名验证文件来源。发布者使用私钥对文件签名，系统使用公钥验证签名。签名验证确保文件来自官方发布，没有被恶意替换。

文件版本记录文件的修改历史。每次修改生成新的版本号，旧版本可以回滚。版本历史存储在元数据中，可以查看和比较。

---

## 6. 文件加密机制

文件加密保护敏感内容的机密性。加密文件需要密钥才能解密读取。密钥管理确保只有授权用户可以访问文件内容。

加密使用 AES-256 算法，对称加密效率高。密钥使用用户密码或密钥文件保护。加密和解密过程对用户透明。

密钥派生从用户密码生成加密密钥。派生使用 PBKDF2 或 Argon2 算法，添加盐值防止彩虹表攻击。派生过程需要一定计算量，增加破解难度。

文件传输使用加密通道。HTTPS 确保传输过程不被窃听。端到端加密可以进一步保护文件安全。

---

## 7. 文件压缩策略

文件压缩减少存储空间和传输时间。不同类型文件使用不同压缩策略，平衡压缩率和性能。

打包文件如 `.card`、`.box`、`.cpk` 使用 ZIP Store 模式，不做内容压缩。内容已经是压缩格式时再次压缩收益不大，零压缩可加快打包和解包速度。

对 `.cpk` 而言，这里指 ZIP Store 模式。Host 当前安装器按 Store 模式解析本地文件头；若插件包使用 Deflate 等压缩方法，安装阶段会报 `PLUGIN_PACKAGE_INVALID`。

文本内容在传输链路上可使用 gzip 压缩。媒体文件使用内容自身的压缩格式，如 WebP、AVIF、H.264、H.265 等。

---

## 8. 文件命名规范

文件命名规范确保文件标识的唯一性和可读性。不同类型文件有不同的命名规则，便于识别和管理。

卡片和箱子文件使用 ID 作为文件名。ID 是 Base62 编码的唯一标识符。文件名不包含中文或特殊字符，避免跨平台问题。

插件文件使用反向域名和版本号命名，如 `com.example.plugin-1.0.0.cpk`。反向域名确保全球唯一性，版本号便于管理。

资源文件使用内容哈希作为文件名。哈希确保同名文件内容相同。系统可以自动去重，节省存储空间。

---

## 9. 文件生命周期

文件生命周期管理文件的创建、使用、归档、删除等阶段。不同阶段有不同的存储策略和管理规则。

创建阶段文件刚生成，需要验证格式和内容。系统检查文件是否符合规范，计算校验信息。创建失败时给出明确错误提示。

使用阶段文件被频繁访问，系统优化读写性能。热数据保持在高速存储，冷数据可以迁移到低速存储。访问统计用于优化存储策略。

归档阶段文件不再活跃使用，转入长期保存。归档文件可以压缩存储，节省空间。归档文件仍可查询和恢复。

删除阶段文件被永久移除。删除前确认用户意图，防止误删。删除执行安全擦除，数据无法恢复。
