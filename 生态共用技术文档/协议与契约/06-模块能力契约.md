# 模块能力契约

> 文档状态：模块插件重构冻结稿
> 适用阶段：模块插件全链路重构实施阶段

## 1. 定位

模块插件是安装到 Host 中的无界面能力插件。它不创建窗口，不承载页面渲染，不以 DOM 挂载和前端插槽为正式能力边界。

模块插件对外暴露的是能力契约，而不是页面运行时。

## 2. 核心模型

### 2.1 Capability

模块对外暴露的稳定能力标识。调用方通过 capability 发现和调用模块能力。

### 2.2 Provider

某个模块插件对某项 capability 的正式提供者记录。

### 2.3 Method

某个 capability 下的具体方法。

### 2.4 Job

用于承载异步模块方法执行的任务实体。

## 3. 模块插件正式约束

- 模块插件安装到 Host 中，使用统一插件治理链路；
- 模块插件必须通过 Manifest 声明 `module.provides` 与 `module.consumes`；
- 模块插件之间允许相互调用，但必须继续通过 Host 模块服务；
- 应用插件、卡片插件、布局插件以及 Host 内部服务，都必须通过统一模块服务使用模块能力；
- 不存在“工作流模块”等额外模块类型，编排只是普通模块对其他模块的一种使用方式。

## 3.1 治理分工

- Host 管理统一调用治理框架：
  - provider 发现；
  - 默认 provider 解析；
  - 权限校验；
  - schema 校验；
  - 超时与取消；
  - job 生命周期；
  - 错误、事件和审计。
- 模块插件负责业务能力契约：
  - capability；
  - method；
  - 输入输出 schema；
  - 依赖能力；
  - 业务语义说明。
- 调用方负责按正式契约接入：
  - 开发期查看共享契约与模块仓库 contract 文件；
  - 运行期把 capability + method 请求交给 Host，由 Host 完成 provider 解析与请求转发。

## 3.2 长期场景覆盖

该模型需要长期稳定支撑以下场景，但这些只是场景，不是新的模块类型：

- 转换场景；
- 跨模块编排场景；
- 通用处理场景。

## 4. Manifest 结构

```yaml
id: chips.module.example
name: Example Module
version: 0.1.0
type: module
entry: dist/index.mjs
permissions:
  - file.read
module:
  apiVersion: 1
  runtime: worker
  activation: onDemand
  provides:
    - capability: example.process
      version: 1.0.0
      methods:
        - name: run
          mode: sync
          inputSchema: contracts/run.input.schema.json
          outputSchema: contracts/run.output.schema.json
  consumes:
    - capability: example.normalize
      versionRange: ^1.0.0
```

## 5. Host 模块服务动作

### 5.1 查询 Provider

```ts
module.listProviders({
  capability?: string,
  pluginId?: string,
  status?: "enabled" | "running"
}) -> { providers }
```

### 5.2 解析 Capability

```ts
module.resolve({
  capability: string,
  versionRange?: string
}) -> { provider }
```

### 5.3 调用方法

```ts
module.invoke({
  capability: string,
  method: string,
  input: Record<string, unknown>,
  pluginId?: string,
  timeoutMs?: number
}) ->
  | { mode: "sync", output: unknown }
  | { mode: "job", jobId: string }
```

### 5.4 Job 管理

```ts
module.job.get({ jobId }) -> { job }
module.job.cancel({ jobId }) -> { ack: true }
```

## 6. 事件

- `module.job.progress`
- `module.job.completed`
- `module.job.failed`
- `module.provider.changed`

## 7. Host 职责

- 维护模块注册表；
- 解析 capability 和 provider；
- 校验权限和 schema；
- 激活和回收模块 runtime；
- 调度同步调用和异步 job；
- 记录调用日志和审计信息。

## 8. 调用方职责

- 依赖正式 capability 契约，而不是依赖模块仓库 README；
- 通过 SDK / Bridge 正式模块服务发起调用；
- 处理同步结果或 job 结果；
- 不得直接 import 模块源码。

## 9. 模块开发者职责

- 正式声明 `module.provides` 和 `module.consumes`；
- 提供 input/output schema；
- 保持 capability 和 method 稳定；
- 按正式上下文 API 使用 Host 能力；
- 不绕过 Host 直连其他模块。

## 10. 文件转换模块补充

文件转换系统遵循本契约，不额外定义新的模块类型：

- 对外统一入口模块通过能力契约暴露文件转换服务；
- 原子转换模块按单一源/目标映射暴露能力；
- 编排模块内部继续通过 `ctx.module.invoke(...)` 调用原子模块；
- 调用方始终只依赖 capability / method / schema，不依赖具体插件源码或仓库结构。
