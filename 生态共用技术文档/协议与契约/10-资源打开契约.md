# 资源打开契约

**版本**：1.1.0
**更新时间**：2026-04-19
**状态**：正式版

---

## 1. 文档定位

本文档定义整个生态统一使用的资源打开动作 `resource.open`，用于让应用插件、查看器、复合卡片容器等，把“打开某个资源”的复杂路由统一委托给 Host。

核心原则：

1. 资源打开必须统一走 Host `resource.open`；
2. 调用方不得直接硬编码目标应用插件 ID；
3. 资源处理器发现、优先级匹配、启动参数组装与回退策略都由 Host 负责；
4. 应用开发者只声明能力并处理自己的启动上下文，不处理跨应用路由底层细节。

## 2. `resource.open`

### 2.1 请求

```ts
interface ResourceOpenRequest {
  intent?: string;
  resource: {
    resourceId: string;
    mimeType?: string;
    title?: string;
    fileName?: string;
    payload?: Record<string, unknown>;
  };
}
```

字段说明：

1. `intent` 表示打开意图，当前未提供时 Host 默认按 `view` 处理；
2. `resource.resourceId` 是正式资源标识，可为本地文件路径、`file://` URL、Host 托管资源 URL，或其他正式资源 ID；
3. `mimeType`、`title`、`fileName` 是帮助 Host 做更精确路由和更友好展示的可选提示字段。
4. `payload` 是可选的结构化上下文；Host 不解释其业务语义，只在命中正式应用处理器后继续透传。

### 2.2 响应

```ts
interface ResourceOpenResult {
  mode: 'plugin' | 'shell' | 'external';
  pluginId?: string;
  windowId?: string;
  matchedCapability?: string;
  resolved: {
    resourceId: string;
    filePath?: string;
    mimeType?: string;
    extension?: string;
    fileName?: string;
  };
}
```

字段说明：

1. `mode = 'plugin'` 表示 Host 已经找到正式应用插件处理器并完成启动；
2. `mode = 'shell'` 表示没有命中应用插件，Host 回退为系统默认本地路径打开；
3. `mode = 'external'` 表示资源是外部 URL，Host 回退为系统外链打开；
4. `matchedCapability` 仅在 `plugin` 模式下返回，表示最终命中的能力声明；
5. `resolved` 是 Host 已完成归一后的资源视图，供调用方记录日志、调试与诊断。

### 2.3 错误语义

1. 当请求结构不合法时，Host 返回 `SCHEMA_VALIDATION_FAILED` 或 `INVALID_ARGUMENT`；
2. 当资源既没有命中应用处理器，又不能按本地路径或外部链接回退时，Host 返回 `RESOURCE_HANDLER_MISSING`。

## 3. 处理器能力声明

应用插件如果要作为资源处理器，必须在 `manifest.yaml` 的 `capabilities` 中声明：

```yaml
capabilities:
  - resource-handler:view:image/*
```

正式格式：

```txt
resource-handler:<intent>:<mime>
```

匹配规则：

1. Host 先匹配精确 MIME 能力，例如 `resource-handler:view:image/png`；
2. 然后匹配主类型通配能力，例如 `resource-handler:view:image/*`；
3. 若仍未命中，再回退到 `file-handler:<ext>`；
4. 当前 `resource.open` 只会把 `type: app` 插件作为正式处理器候选。

补充约束：

1. `resource-handler:*` 主要用于跨应用资源转交；
2. `file-handler:<ext>` 主要用于本地文件扩展名匹配与系统文件关联；
3. 两者可以同时声明，Host 会优先采用 `resource-handler`。

## 4. Host 启动上下文注入

当 Host 通过 `resource.open` 命中应用插件时，启动上下文会统一补充：

```ts
interface ResourceOpenLaunchParams {
  trigger: 'resource-open-service';
  targetPath?: string;
  resourceOpen: {
    intent: string;
    resourceId: string;
    filePath?: string;
    mimeType?: string;
    extension?: string;
    fileName?: string;
    title?: string;
    matchedCapability?: string;
    payload?: Record<string, unknown>;
  };
}
```

正式要求：

1. `resourceOpen` 是当前生态资源打开路由的正式启动参数；
2. 当 Host 能解析出真实本地文件路径时，会继续补充 `targetPath`，用于兼容文件查看器类应用的既有启动恢复逻辑；
3. 资源处理器应用应优先读取 `launchParams.resourceOpen`，并允许把 `targetPath` 作为本地文件查看场景下的兼容回退。
4. 若上游通过 `resource.payload` 传入结构化上下文，Host 必须继续把它透传为 `launchParams.resourceOpen.payload`，不得在 Host 内部裁剪或改写其业务字段。

---

## 5. 复合卡片与查看器容器

涉及卡片、复合 iframe 或其他嵌入式资源容器时，正式要求如下：

1. 基础卡片节点只表达“我想打开一个资源”的意图，不直接启动其他应用；
2. 容器应用在收到资源打开事件后，应调用 `client.resource.open(...)` 或 `window.chips.invoke('resource.open', ...)`；
3. 卡片查看器、箱子查看器、未来的富媒体查看器都必须复用同一条资源打开路由，不得各自硬编码图片查看器、PDF 查看器等目标插件。

---

## 6. 典型链路示例

“卡片查看器点击图片 -> 图片查看器打开”的正式链路如下：

1. 基础卡片节点发出资源打开意图；
2. 卡片查看器接收到该意图后调用 `resource.open`；
3. Host 根据 `resource-handler:view:image/*` 解析出图片查看器；
4. Host 启动图片查看器，并通过 `launchParams.resourceOpen` / `targetPath` 传入资源；
5. 图片查看器恢复到对应图片查看状态。

补充：

1. 当上游不仅要“打开某个资源”，还要让目标应用恢复结构化业务上下文时，应把该上下文放入 `resource.payload`；
2. 当前“音乐基础卡片 -> 音乐播放器”正式使用该能力，把音乐卡片配置、已解析的音频/封面/歌词资源描述符一并透传到 `launchParams.resourceOpen.payload`；
3. Host 负责路由与透传，不负责解释这类业务级 payload 的内部语义。
