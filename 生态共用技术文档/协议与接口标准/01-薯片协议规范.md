# 薯片协议规范

> **文档状态**：vNext 冻结版
> **统一口径**：12层 vNext 架构、中心路由原则
> **版本**：2.0.0

---

## 协议概述

薯片协议是整个生态内部和外部通信的统一标准。该协议定义了模块间通信的消息格式、请求响应机制、事件分发系统、资源访问接口、错误处理规范。

协议遵循中心路由原则，所有通信必须经过薯片内核转发，模块之间不允许直接通信。协议支持多种底层传输方式，可以是进程间通信、网络请求或消息队列。协议消息包含类型标识和版本号，确保通信的兼容性。协议版本升级必须保持向后兼容。安全方面所有请求都需要权限验证。

## 中心路由架构

薯片内核维护路由表，记录所有可用的服务和服务地址。当模块A需要与模块B通信时，模块A向内核发送请求，内核验证请求格式和权限后，根据路由表将请求转发到目标模块。目标模块处理完成后将结果返回给内核，内核再转发给发起请求的模块。整个通信过程中，模块之间从未直接接触，确保了系统的解耦性。

路由表动态可调整，当用户安装新插件或修改配置时，内核自动更新路由规则。所有通信使用标准化的请求和响应格式，内核负责格式的验证和转换。

## 消息格式

### 请求消息格式

请求消息使用JSON格式，包含协议版本、消息ID、时间戳、服务名称、负载数据、可选参数。协议版本遵循语义化版本规范，消息ID使用UUID格式，时间戳使用ISO8601格式。可选参数包括超时时间、优先级、追踪标识等。

**请求消息JSON Schema**：

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",

  "type": "object",

  "properties": {
    "protocol_version": {
      "type": "string",
      "pattern": "^\\d+\\.\\d+\\.\\d+$",
      "description": "协议版本号，遵循语义化版本规范"
    },
    "message_id": {
      "type": "string",
      "description": "消息唯一ID，建议使用UUID"
    },
    "timestamp": {
      "type": "string",
      "format": "date-time",
      "description": "消息发送时间，ISO 8601格式，UTC时区"
    },
    "sender": {
      "type": "string",
      "description": "发送方模块ID"
    },
    "service": {
      "type": "string",
      "description": "请求的服务名称，格式: namespace.action"
```

### 调用上下文标准（RouteInvocationContext）

所有路由调用必须携带标准调用上下文，固定字段如下：

```ts
interface RouteInvocationContext {
  requestId: string;      // UUID 格式，请求唯一标识
  caller: {
    id: string;           // 调用者标识
    type: string;         // 调用者类型：app/card/layout/module/theme
    pluginId?: string;    // 插件 ID（如果是插件调用）
    windowId?: string;    // 窗口 ID
    permissions?: string[]; // 调用者权限列表
  };
  timestamp: string;      // ISO8601 格式
  deadline?: string;      // 超时截止时间
}
```

### 路由契约描述符

所有服务注册动作必须提供路由契约描述符，内核在注册时生成 `route-manifest`，供 SDK 与测试门禁校验：

```ts
interface RouteDescriptor<I, O> {
  key: `${string}.${string}`;    // namespace.action 格式
  schemaIn: string;             // 输入 JSON Schema ID
  schemaOut: string;            // 输出 JSON Schema ID
  permission: string[];         // 所需权限列表
  timeoutMs: number;            // 超时时间（毫秒）
  idempotent: boolean;          // 是否幂等
  retries: 0 | 1 | 2 | 3;      // 重试次数
}
```

### IPC 通道设计

薯片协议使用以下 IPC 通道进行通信：

| 通道名称 | 方向 | 说明 |
|----------|------|------|
| `chips:invoke` | 渲染进程 → 主进程 | 请求-响应调用（ipcRenderer.invoke → ipcMain.handle） |
| `chips:emit` | 渲染进程 → 主进程 | 插件发送事件（ipcRenderer.send → ipcMain.on） |
| `chips:event:{name}` | 主进程 → 渲染进程 | 主进程向插件广播事件（webContents.send） |
| `chips:window:{action}` | 渲染进程 → 主进程 | 窗口控制命令 |
| `chips:plugin-init` | 主进程 → 渲染进程 | 插件初始化数据传递 |

**主进程 IPC 处理示例**：

```typescript
// 主进程 IPC 路由
ipcMain.handle('chips:invoke', async (event, request: ChipsRequest) => {
  const windowId = BrowserWindow.fromWebContents(event.sender)?.id;
  const pluginId = windowManager.getPluginId(windowId);

  // 权限检查
  if (!permissionManager.check(pluginId, request.namespace, request.action)) {
    throw { code: 'PERMISSION_DENIED', message: '权限不足' };
  }

  // 转发到路由器
  return kernel.router.invoke(request.namespace, request.action, request.params);
});

ipcMain.on('chips:emit', (event, { eventName, data }) => {
  const windowId = BrowserWindow.fromWebContents(event.sender)?.id;
  const pluginId = windowManager.getPluginId(windowId);

  // 将插件事件转发到事件总线
  kernel.events.emit(eventName, { ...data, source: pluginId });
});
```

## 服务命名规范

服务名称采用命名空间加动作的格式。命名空间使用小写字母和连字符，动作使用小写字母和下划线。例如cardread表示读取卡片，cardwrite表示写入卡片。

标准命名空间包括卡片相关操作、箱子相关操作、资源管理、插件管理、文件系统操作、认证授权、配置管理、事件系统、日志系统。

## 权限验证

每个服务调用需要权限验证。权限模型定义基础权限和标准权限等级。权限检查流程包括查询模块权限列表、检查是否拥有所需权限、返回错误或继续处理。

## 错误处理

### 错误码分类体系（vNext 冻结标准）

错误代码采用分类前缀，遵循统一错误归一映射原则：

| 前缀 | 层级 | 说明 |
|---|---|---|
| `BRIDGE_*` | L5 Bridge Transport | IPC 传输层错误 |
| `SERVICE_*` | L3 Host Services | 服务域处理错误 |
| `RUNTIME_*` | L6 Runtime Client | 超时、重试、运行时错误 |
| `UI_*` | L12 App/Page | 页面级消费错误 |
| `PERMISSION_*` | 安全层 | 权限拒绝错误 |
| `RESOURCE_*` | 资源层 | 资源不存在等错误 |

### 标准错误对象格式

```ts
interface StandardError {
  code: string;           // 错误码，如 BRIDGE_TIMEOUT
  message: string;       // 面向用户的错误描述
  details?: unknown;      // 调试用详细信息
  retryable?: boolean;   // 是否可重试
}
```

### 常见错误代码（冻结口径）

| 错误码 | 说明 | 幂等 |
|---|---|---|
| `BRIDGE_TIMEOUT` | 路由调用超时 | 可重试 |
| `BRIDGE_CONNECTION_LOST` | IPC 连接中断 | 可重试 |
| `SERVICE_NOT_FOUND` | 服务或动作不存在 | 否 |
| `SERVICE_UNAVAILABLE` | 服务暂不可用 | 可重试 |
| `PERMISSION_DENIED` | 权限不足 | 否 |
| `RESOURCE_NOT_FOUND` | 资源不存在 | 是 |
| `RESOURCE_CONFLICT` | 资源冲突 | 否 |

## 版本兼容

协议版本号遵循语义化版本规范 `主版本.次版本.修订号`：

- **主版本**：不兼容的API变更
- **次版本**：向后兼容的功能新增
- **修订号**：向后兼容的问题修正

### 版本协商

内核在处理请求时检查协议版本：

1. 如果主版本号不同，拒绝请求
2. 如果次版本号不同，使用较低版本的功能子集
3. 修订号不影响兼容性

### 版本声明

模块注册时声明支持的协议版本范围：

```json
{
  "module_id": "card-manager",
  "protocol_versions": {
    "min": "1.0.0",
    "max": "2.0.0"
  }
}
```

## 异步处理

长时间任务采用异步模式，请求立即返回任务ID，模块订阅任务状态事件，任务完成后发送事件通知。

流式数据采用分块传输，支持大文件传输场景。

## 事件系统

事件类型使用后缀格式，创建事件使用created后缀，更新事件使用updated后缀，删除事件使用deleted后缀，错误事件使用error后缀。

模块向内核订阅事件，指定事件类型和过滤器。模块可以发布事件到内核，由内核分发到所有订阅者。

## 资源访问协议

资源访问使用统一的URI格式。URI格式为chips协议加权威机构加路径加查询参数。例如本地资源使用chipslocal路径，网络资源使用chipsnetwork路径。

资源请求包含URI、缓存选项、超时设置。资源响应包含URI、内容类型、数据大小、元数据。

## 性能优化

支持批量请求减少通信开销，内核接收批量请求后并行处理。内核提供缓存支持，相同请求可以返回缓存结果。对于大数据传输支持压缩，减少网络开销。

## 安全性

重要操作需要消息签名，使用RSA算法签名。跨网络通信必须使用TLS加密。内核对每个模块的请求进行限流，防止资源耗尽。

## 日志和追踪

所有请求和事件记录日志，包含时间戳、日志级别、消息类型、消息ID、服务名称、发送方、处理状态、执行耗时。

支持分布式追踪，包含追踪ID、跨度ID、父子跨度ID，支持跨进程的请求追踪。
