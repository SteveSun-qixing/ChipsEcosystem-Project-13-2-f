# 运行时设置治理接口标准

> 适用范围：生态设置面板、未来所有运行时治理类应用、Host 内部治理工具。  
> 文档目标：冻结主题治理、多语言治理、应用插件治理和组件主题验收的正式接口口径，避免各应用自行拼装一套设置链路。

## 1. 职责边界

- 本标准只覆盖 Host 运行态内可即时生效的治理能力。
- 安装器负责系统级安装、升级、卸载、文件关联和操作系统注册动作，不在本标准范围内。
- 治理类应用不得绕过 Bridge 直接修改 Host 内部文件或运行时对象。

## 2. 主题治理基线

### 2.1 数据来源

主题治理必须组合以下正式接口：

| 目标 | 接口 | 说明 |
| --- | --- | --- |
| 读取已安装主题记录 | `plugin.query({ type: 'theme' })` | 返回所有已安装主题插件运行时记录，并补齐主题治理元数据 |
| 读取当前可切换主题集合 | `theme.list()` | 仅返回当前工作区已启用主题插件 |
| 读取当前主题 | `theme.getCurrent()` | 返回当前活动主题 |
| 安装主题 | `plugin.install(manifestPath)` | 安装 `.cpk`、插件目录或 manifest 输入 |
| 切换主题 | `theme.apply(id)` | 切换当前主题 |
| 卸载主题 | `plugin.uninstall(pluginId)` | 卸载主题插件 |

### 2.2 正式视图模型要求

治理类应用需要在自身适配层生成统一主题视图模型，至少包含：

```ts
interface ThemeGovernanceRecord {
  pluginId: string;
  themeId: string;
  displayName: string;
  version: string;
  installed: boolean;
  enabled: boolean;
  current: boolean;
  installPath: string;
  installedAt: number;
}
```

其中：

- `installed` 来源于 `plugin.query`；
- `enabled` 由 `theme.list()` 交叉比对得到；
- `current` 由 `theme.getCurrent()` 判定。

### 2.3 事件要求

- 主题切换成功后以 `theme.changed` 作为统一刷新事件源。
- 主题安装或卸载后，治理类应用应重新拉取主题记录，不允许只在前端本地猜测结果。

## 3. 多语言治理基线

| 目标 | 接口 |
| --- | --- |
| 当前语言 | `i18n.getCurrent()` |
| 已安装语言列表 | `i18n.listLocales()` |
| 切换语言 | `i18n.setCurrent({ locale })` |
| 运行时刷新事件 | `language.changed` |

治理类应用必须在语言切换后执行运行时重渲染，禁止通过重启整个应用实现语言生效。

## 4. 应用插件治理基线

| 目标 | 接口 |
| --- | --- |
| 列出已安装应用插件 | `plugin.query({ type: 'app' })` |
| 安装应用插件 | `plugin.install(manifestPath)` |
| 启用应用插件 | `plugin.enable(pluginId)` |
| 停用应用插件 | `plugin.disable(pluginId)` |
| 卸载应用插件 | `plugin.uninstall(pluginId)` |

治理类应用在“列表场景”统一以 `plugin.query` 为正式基线，不要求自行读取插件目录结构。

`plugin.query` 的正式返回记录至少应包含：

```ts
interface PluginGovernanceRecord {
  id: string;
  manifestPath: string;
  installPath: string;
  enabled: boolean;
  type: 'app' | 'card' | 'layout' | 'module' | 'theme';
  name: string;
  version: string;
  description?: string;
  capabilities: string[];
  installedAt: number;
  theme?: {
    themeId: string;
    displayName: string;
    publisher?: string;
    parentTheme?: string;
    isDefault: boolean;
  };
  layout?: {
    layoutType?: string;
    displayName: string;
  };
}
```

## 5. 文件输入治理基线

- 文件选择统一使用 `platform.dialogOpenFile`。
- 拖拽安装本质上仍需转化为 Host 可识别的正式路径输入，再调用 `plugin.install`。
- 治理类应用不得自行解压 `.cpk` 或直接复制插件文件到 Host 插件目录。

## 6. 组件展示与主题验收基线

- 组件展示页必须直接使用 `@chips/component-library` 正式导出的组件。
- 展示页可额外读取 `theme.getCurrent()` 或 `theme.contract.get()` 做辅助说明，但不能替代真实组件渲染。
- 当组件库新增正式组件时，治理类应用的展示注册表也必须同步更新，作为主题验收的一部分。

## 7. 权限基线

最小权限集合：

- `theme.read`
- `theme.write`
- `i18n.read`
- `i18n.write`
- `plugin.read`
- `plugin.manage`
- `platform.read`

如治理类应用需要打开外部链接、展示日志或读写配置，应在自身 manifest 中显式追加对应权限并同步文档。
