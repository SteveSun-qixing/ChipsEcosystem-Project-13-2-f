# 运行时设置治理接口标准

> 适用范围：生态设置面板、未来所有运行时治理类应用、Host 内部治理工具。  
> 文档目标：冻结主题治理、多语言治理、应用插件治理和组件主题验收的正式接口口径，避免各应用自行拼装一套设置链路。

## 1. 职责边界

- 本标准只覆盖 Host 运行态内可即时生效的治理能力。
- 安装器负责 Host 本体的系统级安装、升级、卸载、文件关联和操作系统注册动作，不在本标准范围内。
- 应用插件快捷方式属于 Host 运行态治理能力：治理类应用只能调用 Host 正式接口创建/删除应用快捷方式，不能直接操作系统 API。
- 治理类应用不得绕过 Bridge 直接修改 Host 内部文件或运行时对象。

## 2. 主题治理基线

### 2.1 数据来源

主题治理必须组合以下正式接口：

| 目标 | 接口 | 说明 |
| --- | --- | --- |
| 读取已安装主题记录 | `plugin.query({ type: 'theme' })` | 返回所有已安装主题插件运行时记录，并补齐主题治理元数据 |
| 读取当前可切换主题集合 | `theme.list()` | 仅返回当前工作区已启用主题插件 |
| 读取当前主题 | `theme.getCurrent()` | 返回当前活动主题 |
| 安装主题 | `plugin.install(manifestPath)` | 安装 `.cpk`、插件目录或 manifest 输入 |
| 启用刚安装主题 | `plugin.enable(pluginId)` | 主题安装后必须启用，后续 `theme.list()` / `theme.apply()` 才能消费 |
| 切换主题 | `theme.apply(id)` | 切换当前主题 |
| 卸载主题 | `plugin.uninstall(pluginId)` | 卸载主题插件 |

### 2.2 正式视图模型要求

治理类应用需要在自身适配层生成统一主题视图模型，至少包含：

```ts
interface ThemeGovernanceRecord {
  pluginId: string;
  themeId: string;
  displayName: string;
  version: string;
  installed: boolean;
  enabled: boolean;
  current: boolean;
  installPath: string;
  installedAt: number;
}
```

其中：

- `installed` 来源于 `plugin.query`；
- `enabled` 由 `theme.list()` 交叉比对得到；
- `current` 由 `theme.getCurrent()` 判定。

### 2.3 事件要求

- 主题切换成功后以 `theme.changed` 作为统一刷新事件源。
- 主题安装或卸载后，治理类应用应重新拉取主题记录，不允许只在前端本地猜测结果。
- 若治理类应用提供“安装后立即可切换”体验，则必须执行 `plugin.install -> plugin.enable -> 重新拉取主题列表` 的正式链路，禁止跳过启用步骤。

## 3. 多语言治理基线

| 目标 | 接口 |
| --- | --- |
| 当前语言 | `i18n.getCurrent()` |
| 已安装语言列表 | `i18n.listLocales()` |
| 切换语言 | `i18n.setCurrent({ locale })` |
| 运行时刷新事件 | `language.changed` |

治理类应用必须在语言切换后执行运行时重渲染，禁止通过重启整个应用实现语言生效。

## 4. 运行时插件治理基线

### 4.1 应用插件

| 目标 | 接口 |
| --- | --- |
| 列出已安装应用插件 | `plugin.query({ type: 'app' })` |
| 安装应用插件 | `plugin.install(manifestPath)` |
| 启用应用插件 | `plugin.enable(pluginId)` |
| 停用应用插件 | `plugin.disable(pluginId)` |
| 卸载应用插件 | `plugin.uninstall(pluginId)` |
| 读取应用快捷方式状态 | `plugin.getShortcut(pluginId)` |
| 创建/更新应用快捷方式 | `plugin.createShortcut(pluginId, { replace? })` |
| 删除应用快捷方式 | `plugin.removeShortcut(pluginId)` |

治理类应用在“列表场景”统一以 `plugin.query` 为正式基线，不要求自行读取插件目录结构。

`plugin.query` 的正式返回记录至少应包含：

```ts
interface PluginGovernanceRecord {
  id: string;
  manifestPath: string;
  installPath: string;
  enabled: boolean;
  type: 'app' | 'card' | 'layout' | 'module' | 'theme';
  name: string;
  version: string;
  description?: string;
  capabilities: string[];
  installedAt: number;
  theme?: {
    themeId: string;
    displayName: string;
    publisher?: string;
    parentTheme?: string;
    isDefault: boolean;
  };
  layout?: {
    layoutType?: string;
    displayName: string;
  };
}
```

应用快捷方式治理视图还应补充：

```ts
interface PluginShortcutGovernanceRecord {
  pluginId: string;
  name: string;
  location: 'desktop' | 'launchpad';
  launcherPath: string;
  executablePath: string;
  args: string[];
  iconPath?: string;
  exists: boolean;
}
```

说明：

- Windows 的 `location: 'desktop'` 表示桌面快捷方式；
- macOS 的 `location: 'launchpad'` 表示由 Host 写入 `~/Applications/Chips Apps/*.app` 后出现在启动台的应用入口；
- `name` 是 Host 实际用于系统入口展示的名称，优先取 `manifest.ui.launcher.displayName`；
- `args` 为 Host 真实写入快捷方式的启动参数，当前至少包含 `--workspace=<workspacePath>` 与 `--chips-launch-plugin=<pluginId>`；
- `exists` 必须由 Host 重新检查目标路径得出，不允许前端仅凭本地缓存推断；
- 治理类应用只负责读取状态与触发 Host 接口，不得自行创建 `.lnk`、`.app` 或其他平台文件。

图标展示要求：

- 应用插件治理页优先消费 `PluginShortcutGovernanceRecord.iconPath`，并将其转换为正式文件 URL 作为系统入口图标展示；
- 当 `iconPath` 缺失、为空或文件不可用时，治理页必须回退到运行时统一图标组件，例如 `ChipsIcon` + `{ name: "apps" }`；
- 治理页不得把 `iconPath` 误解释为 `IconDescriptor`，也不得在前端手工拼装一套平台图标推断规则；
- 列表、详情页、按钮和空态中的其他运行时 UI 图标仍必须统一走 `ChipsIcon`。

创建 / 删除语义补充：

- 当 `plugin.createShortcut(pluginId, { replace: false })` 且现有快捷方式仍存在时，Host 直接返回当前状态，不重复写入；
- 当 `replace: true` 时，Host 允许覆盖已有入口；
- `plugin.removeShortcut(pluginId)` 必须同时清理 Host 工作区内的快捷方式登记记录；
- 应用插件卸载时，Host 会优先尝试删除其快捷方式，治理类应用无需自行补删。

刷新策略补充：

- 应用插件治理页在执行创建 / 删除快捷方式后，必须重新拉取 `plugin.getShortcut(pluginId)`，不得前端手改状态；
- 若 Host 已发出 `plugin.shortcut.changed` 事件，治理类应用可将其作为后续增强的刷新触发源，但首版实现仍以动作完成后的主动刷新为准。

### 4.2 卡片 / 布局 / 模块插件

| 目标 | 卡片插件 | 布局插件 | 模块插件 |
| --- | --- | --- | --- |
| 列表 | `plugin.query({ type: 'card' })` | `plugin.query({ type: 'layout' })` | `plugin.query({ type: 'module' })` |
| 安装 | `plugin.install(manifestPath)` | `plugin.install(manifestPath)` | `plugin.install(manifestPath)` |
| 启用 | `plugin.enable(pluginId)` | `plugin.enable(pluginId)` | `plugin.enable(pluginId)` |
| 停用 | `plugin.disable(pluginId)` | `plugin.disable(pluginId)` | `plugin.disable(pluginId)` |
| 卸载 | `plugin.uninstall(pluginId)` | `plugin.uninstall(pluginId)` | `plugin.uninstall(pluginId)` |

正式要求：

- 列表场景统一以 `plugin.query` 为基线，不得自行读取插件目录结构；
- 安装动作完成后，治理类应用应通过 `plugin.get(pluginId)` 或下一次 `plugin.query(...)` 校验插件真实 `type`，禁止把错误类型的安装结果直接混入当前治理页；
- 卡片插件治理页消费 `PluginGovernanceRecord.capabilities` 中扁平化后的 `cardTypes` 作为展示身份；
- 布局插件治理页消费 `PluginGovernanceRecord.layout.layoutType` 与 `layout.displayName`；
- 模块插件治理页消费 `PluginGovernanceRecord.capabilities` 作为能力标记集合；
- 卡片 / 布局 / 模块插件的安装、启用、停用、卸载完成后，都必须重新拉取对应 `plugin.query({ type })` 结果，或消费 `plugin.installed` / `plugin.enabled` / `plugin.disabled` / `plugin.uninstalled` 事件触发刷新。

## 5. 文件输入治理基线

- 文件选择统一使用 `platform.dialogOpenFile`。
- 拖拽安装本质上仍需转化为 Host 可识别的正式路径输入，再调用 `plugin.install`。
- 治理类应用不得自行解压 `.cpk` 或直接复制插件文件到 Host 插件目录。

## 6. 组件展示与主题验收基线

- 组件展示页必须直接使用 `@chips/component-library` 正式导出的组件。
- 展示页可额外读取 `theme.getCurrent()` 或 `theme.contract.get()` 做辅助说明，但不能替代真实组件渲染。
- 当组件库新增正式组件时，治理类应用的展示注册表也必须同步更新，作为主题验收的一部分。

## 7. 权限基线

最小权限集合：

- `theme.read`
- `theme.write`
- `i18n.read`
- `i18n.write`
- `plugin.read`
- `plugin.manage`
- `platform.read`

如治理类应用需要打开外部链接、展示日志或读写配置，应在自身 manifest 中显式追加对应权限并同步文档。
