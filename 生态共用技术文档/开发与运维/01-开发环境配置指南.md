# 开发环境配置指南

## 环境要求

- Node.js 18 及以上
- npm 10 及以上
- Git
- macOS / Windows / Linux 中任一受支持开发环境

说明：

- 生态根 `package.json` 统一维护 Volta 工具链版本；
- 各工作区 `package.json` 必须通过 `volta.extends` 指向生态根 `package.json`，保证在子工程目录直接执行 `chipsdev`、`chips` 等命令时，Volta 可以解析根工作区 hoist 的二进制。

## 生态仓库初始化

薯片生态一方仓库统一在生态根目录初始化依赖：

```bash
cd /Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f
npm install
```

说明：

- 根目录 `package.json` 是生态工作区唯一入口；
- `Chips-CardViewer`、`Chips-SDK`、`Chips-ComponentLibrary`、`Chips-BaseCardPlugin/*`、`ThemePack/*` 等都由该工作区统一解析；
- 不再使用 `npm link`、`file:` sibling 依赖或手工复制 SDK / 组件库。

## 日常开发

完成根安装后，可进入任意工作区工程执行命令，例如：

```bash
cd Chips-CardViewer
npm run build
npm run test
```

`chipsdev` 由工程内 `chips-sdk` 工作区依赖提供，无需全局安装。若开发机使用 Volta，生态工作区通过根 `package.json` 的 Volta 配置与各子工程的 `volta.extends` 保证直接命令调用稳定可用。

## 新工程创建

在生态根目录使用 `chipsdev create` 创建一方工程时：

- 模板先输出正式发布依赖；
- 若目标目录位于生态根工作区内，`chipsdev` 会自动：
  - 将新工程注册到根工作区（当现有 glob 未覆盖时）
  - 为新工程写入指向生态根的 `volta.extends`

这是一方工程接入生态仓库的唯一正式方式。

## 代码规范

- 提交前运行测试与校验；
- 提交信息使用 Conventional Commits；
- 涉及生态契约、包边界、工作区结构的改动，必须同步更新文档。

## 常见问题

### 为什么不再允许 `npm link`

`npm link` 无法提供稳定、可复现、可审计的生态级安装快照，也会让 CI、本地开发与脚手架生成行为不一致。

### 为什么模板不能写工作区协议

模板是对外生成面，不能耦合当前生态仓库结构；是否接入生态工作区，由 `chipsdev create` 在一方工程场景中统一决定。

## 持续集成

生态级流水线最少包含：

```bash
npm install
npm run verify:workspace
```

然后再执行对应工程的构建、测试、验证与打包流程。
