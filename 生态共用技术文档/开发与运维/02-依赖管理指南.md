# 依赖管理指南

## 概述

薯片生态在 2026-03-09 起统一采用“生态根工作区 + semver 包依赖 + 工作区自动链接”方案：

1. 生态根目录 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f` 提供唯一工作区根 `package.json`。
2. 生态内一方仓库之间的正式依赖，统一声明为 semver 版本，由 npm workspaces 自动链接到本地工作区包。
3. 脚手架模板输出面向发布的 semver 依赖；当模板工程落在生态根工作区内时，由 `chipsdev create` 自动注册到根工作区。
4. 为保证 Volta 与 npm workspaces hoist 的命令解析一致性，生态根 `package.json` 统一维护 Volta 工具链版本，各工作区 `package.json` 通过 `volta.extends` 指向生态根。

这套方案是生态仓库内的唯一正式依赖策略，不再保留旧的 sibling `file:` 引用方案。

## 生态根工作区

根工作区 `package.json` 是生态内依赖解析的唯一入口，当前覆盖：

- `Chips-*`
- `Chips-BaseCardPlugin/*`
- `Chips-ComponentLibrary/packages/*`
- `Chips-ComponentLibrary/packages/adapters/*`
- `Chips-Scaffold/*`
- `ThemePack/*`

规则：

- 一方工程新增后，必须纳入根工作区；
- 一方工程执行安装时，以根工作区 lockfile 为准；
- 一方工程的 `package.json` 必须通过 `volta.extends` 指向生态根 `package.json`；
- 生态级校验使用根脚本 `npm run verify:workspace`。

## 依赖声明规则

### 生态内一方工程

以下场景统一声明正式 semver 版本，并由工作区自动链接：

- `Chips-CardViewer` -> `chips-sdk`
- `Chips-CardViewer` -> `@chips/component-library`
- 基础卡片插件工程 -> `chips-sdk`
- 组件库聚合包 -> `@chips/hooks` / `@chips/components` / `@chips/card-runtime`

禁止：

- `file:../Chips-SDK`
- `file:../Chips-ComponentLibrary/...`
- `npm link`
- 将 SDK / 组件库复制进业务仓库

### 脚手架模板

模板文件中的依赖声明必须保持“正式发布口径”：

- `chips-sdk`: `^0.1.0`
- `@chips/component-library`: `^0.1.0`

模板中禁止写死：

- `file:...`
- 指向生态仓库绝对/相对路径的依赖来源

原因：

- 模板是对外生成面，不应耦合当前生态仓库绝对结构；
- 模板默认只输出运行时必要目录与 README，不预创建 `需求文档/`、`技术文档/`（或“技术手册”）与 `开发计划/` 目录；
- 生态内创建工程时，由根工作区自动完成本地包链接与 Volta 继承配置。

## 安装命令

推荐命令：

```bash
# 在生态根目录执行
npm install
```

也允许在某个工作区工程目录中执行：

```bash
cd Chips-CardViewer
npm install
```

此时 npm 会回到上层工作区根进行解析，仍然以生态根 lockfile 为准。

若在工作区工程目录中直接执行 `chipsdev`、`chips`、`vite`、`vitest` 等命令，Volta 也会通过 `volta.extends` 回溯到生态根并解析 hoist 后的根级二进制。

## 版本策略

- 一方工作区依赖使用正式 semver 版本，不再在仓内维护本地路径依赖。
- 模板与对外文档使用 semver 版本范围。
- 发生破坏性变更时，仍按语义化版本规范升级实际包版本。

## 锁文件策略

- 生态根工作区的 lockfile 是生态联调、CI 与验收的唯一安装快照。
- 子仓保留的历史 lockfile 不作为生态联调依据。
- 新增或修复工作区依赖后，必须更新根 lockfile。
- 根 `package.json` 的 Volta 配置与各工作区 `volta.extends` 共同构成命令解析快照的一部分，修改后必须纳入校验。

## CI / 验收

生态级流水线至少应执行：

```bash
npm install
npm run verify:workspace
```

然后再执行代表性仓库的构建、测试和验证命令。

## 最佳实践

- 先设计正式包边界，再声明依赖；
- 一方包必须能被工作区直接解析，禁止临时桥接；
- 组件库聚合包只能依赖正式子包，不能跨目录直接偷用源码；
- 一切依赖方案必须先写入文档，再进入模板和仓库实现；
- 发现依赖缺口时，先提工单，再改生态层，不允许在业务仓库补临时代码。
