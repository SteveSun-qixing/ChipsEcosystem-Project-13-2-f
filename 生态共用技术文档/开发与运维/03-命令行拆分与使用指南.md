# 薯片生态命令行拆分与使用指南

> 本文档定义 `chips` 与 `chipsdev` 的职责边界、工作区模型和协作方式，是生态内所有命令行实现与文档的统一依据。

## 1. 总体结构

薯片生态提供两套正式命令行：

- `chips`：面向终端用户与运维，管理用户工作区中的 Host 运行时；
- `chipsdev`：面向开发者，管理开发工作区中的工程流程与 Host 联调能力。

旧写法 `chips host`、`chips dev` 已全部废弃。

## 2. 工作区边界

### 2.1 用户工作区

`chips` 默认使用 `~/.chips-host`，可通过 `CHIPS_HOME` 覆盖。

用户工作区保存：

- Host 运行状态
- 用户侧插件记录
- 用户侧主题配置
- 用户侧日志与诊断信息

### 2.2 开发工作区

`chipsdev` 默认使用项目链路中的 `.chips-host-dev`：

- 先从当前目录向上查找 `.chips-host-dev`
- 找到即复用
- 找不到则在当前工程根创建

开发工作区保存开发态联调需要的全部 Host 状态，与用户工作区完全隔离。

### 2.3 隔离原则

以下数据不得跨工作区共享：

- 当前主题
- 主题插件启用状态
- 插件安装与启停记录
- Host 配置
- 运行日志

因此，出现“`chips` 已切换暗夜主题，但 `chipsdev run` 仍是默认主题”时，首先应检查是否处于不同工作区。

## 3. `chips` 的职责

`chips` 只负责用户工作区中的 Host 管理：

- `start/stop/status`
- `config`
- `logs`
- `plugin install/uninstall/list/enable/disable/query`
- `theme list/current/apply/resolve/contract/validate`
- `update check/install`
- `doctor`
- `open`

`chips` 不负责工程创建、构建、测试、打包。

## 4. `chipsdev` 的职责

`chipsdev` 负责两类能力：

### 4.1 工程工作流

- `init`
- `create`
- `server`
- `debug`
- `build`
- `test`
- `lint`
- `e2e`
- `package`
- `validate`
- `login`
- `publish`
- `run`

### 4.2 开发工作区 Host 管理

`chipsdev` 额外提供与 `chips` 同级的 Host 管理命令，但所有操作落在开发工作区：

- `start/stop/status`
- `config`
- `logs`
- `plugin`
- `theme`
- `update`
- `doctor`
- `open`

实现方式上，`chipsdev` 会把这些命令委托给 Host CLI 核心，但通过开发工作区路径和 `CHIPS_WORKSPACE_KIND=dev` 固定其上下文。

## 5. 典型工作流

### 5.1 用户侧运行

```bash
chips plugin install /绝对路径/主题包.cpk
chips plugin enable theme.theme.chips-official-default-dark-theme
chips theme apply chips-official.default-dark-theme
chips open /绝对路径/示例.card
```

### 5.2 开发侧联调

```bash
cd /Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer
chipsdev plugin install /绝对路径/主题包.cpk
chipsdev plugin enable theme.theme.chips-official-default-dark-theme
chipsdev theme apply chips-official.default-dark-theme
chipsdev run
```

### 5.3 主题包开发与验收

```bash
chipsdev build
chipsdev validate
chipsdev package
chipsdev plugin install /绝对路径/产物.cpk
chipsdev plugin enable <pluginId>
chipsdev theme validate
```

## 6. 主题链路约束

主题链路必须满足以下约束：

- 只有已启用主题插件才能进入 `theme.list/getCurrent/resolve/apply`；
- 主题插件安装、启用、禁用、卸载后，Host 必须刷新主题运行时状态；
- 应用启动初始主题必须从 Host 当前主题读取，不得写死默认值；
- 复合卡片窗口必须在主题运行时缓存键变化后重新渲染；
- 原生窗口背景在应用未显式提供时，必须取自当前主题 token；
- 卡片渲染服务不得写死 `color-scheme: light` 之类的覆盖样式。

## 7. 迁移说明

若旧文档中仍出现以下描述，均按当前方案理解：

- `chips dev ...` -> `chipsdev ...`
- `chips host ...` -> `chips ...`
- “开发命令不负责插件和主题管理” -> 已失效，当前 `chipsdev` 已提供开发工作区的正式插件与主题管理能力

## 8. 质量门禁

命令行相关实现与文档必须同时满足：

- 边界清晰：用户工作区与开发工作区不混用；
- 行为一致：`chips` 与 `chipsdev` 在各自工作区内提供同口径 Host 管理能力；
- 文档一致：命令示例、技术文档、设计原稿和工单记录保持统一；
- 可验证：主题安装、启用、切换、查看器显示、卡片渲染与窗口背景均可通过正式命令链路复核。
