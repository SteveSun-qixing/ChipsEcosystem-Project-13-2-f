# 薯片生态命令行拆分与使用指南

> 本文档定义薯片生态命令行工具的整体结构与职责边界，是 Host/SDK/脚手架/模板等项目实现与文档的统一依据。

## 1. 总体结构

薯片生态提供两套独立的命令行工具，分别服务于终端用户/运维与插件/应用开发者：

- `chips`：薯片主机（Chips Host）命令行工具  
  - 随 Host 软件一起安装。  
  - 面向终端用户与运维，负责本地 Host 实例的生命周期管理与运行时操作。

- `chipsdev`：薯片开发者命令行工具（Chips Dev CLI）  
  - 随 SDK 与开发工具链一起安装。  
  - 面向插件/应用/主题/语言包等开发者，负责项目创建、开发调试、构建测试、打包发布等。

两套命令行工具相互独立，当前正式入口为 `chips` 与 `chipsdev`。旧文档中出现的 `chips dev` 与 `chips host` 一律视为历史写法，统一替换为 `chipsdev` 与直接 `chips`。

## 2. Host 命令行工具：chips

### 2.1 职责

`chips` 只负责“主机级”的操作，不承担任何工程构建或开发工作流逻辑：

- 管理本地 Host 实例的生命周期（启动、停止、状态查看）；
- 管理 Host 工作区与配置；
- 管理 Host 运行中的插件与主题；
- 提供日志导出与诊断能力；
- 提供文件入口（open）以模拟系统文件关联行为。

所有需要编译、打包、测试、Lint、脚手架创建等开发者工作流，均应通过 `chipsdev` 完成，而不是通过 `chips`。

### 2.2 基本用法

命令入口统一为：

```bash
chips <command> [options] [args]
```

核心命令集合（与《Chips Host 命令行使用手册》保持一致）：

- `chips start`：启动 Host；
- `chips stop`：停止 Host；
- `chips status`：查看运行状态；
- `chips config list|set|reset`：查看和修改配置；
- `chips logs`：导出运行日志；
- `chips plugin list|install|uninstall|enable|disable|query`：管理插件；
- `chips theme list|current|apply|resolve|contract|validate`：管理与校验主题；
- `chips update check|install`：更新检查与安装；
- `chips doctor`：诊断当前环境；
- `chips open <path>`：通过 Host 打开 card/box 等文件。

Host CLI 默认使用 `~/.chips-host` 作为工作区目录，可通过环境变量（例如 `CHIPS_HOME`）覆盖，具体行为以 Host 仓库实现与文档为准。

## 3. 开发者命令行工具：chipsdev

### 3.1 职责

`chipsdev` 是 SDK 配套的开发者命令行工具，主要职责包括：

- 初始化开发环境与配置；
- 从脚手架模板创建各类工程（应用、卡片插件、布局插件、模块插件、主题包、语言包等）；
- 启动本地开发服务器与调试环境；
- 执行测试、Lint、E2E 与验证；
- 编译构建、打包生成 `.cpk` 插件包；
- 与插件市场/发布系统集成（登录、发布、版本管理等）。

`chipsdev` 不负责直接操作 Host 运行时（例如启动/停止 Host、安装插件到运行中的 Host 等），这类操作统一通过 `chips` 完成。

### 3.2 基本用法

命令入口统一为：

```bash
chipsdev <command> [options] [args]
```

典型命令集合（与《Chips Dev 开发者命令行手册》对齐）：

- 初始化与创建：
  - `chipsdev init`：初始化开发环境/全局配置；
  - `chipsdev create`：从脚手架模板创建新工程。
- 开发与调试：
  - `chipsdev server`：启动开发服务器（热重载、调试入口等）；
  - `chipsdev debug`：以调试模式启动目标项目；
  - `chipsdev run`：在开发环境下启动 Host 运行时并运行应用插件（type: app）。
- 质量门禁：
  - `chipsdev test` / `chipsdev test --coverage`：运行单元测试与覆盖率；
  - `chipsdev lint`：运行代码规范检查；
  - `chipsdev e2e`：运行端到端测试；
  - `chipsdev validate`：执行项目级契约校验（硬编码检查、配置/manifest 校验等）。
- 构建与打包：
  - `chipsdev build` / `chipsdev build --release`：构建项目；
  - `chipsdev package`：打包生成 `.cpk` 文件。
- 发布与版本：
  - `chipsdev login`：登录开发者账号；
  - `chipsdev publish`：发布插件到市场；
  - `chipsdev version`：查看与管理版本信息。

### 3.3 配置文件与工作区

项目级配置统一使用：

- `chips.config.mjs`：Chips Dev 配置文件，描述源码目录、输出目录、入口文件、测试目录等。

示例（以 CardViewer 为例）：

```js
/**
 * Chips Dev 配置文件（卡片查看器应用）
 */

const config = {
  srcDir: 'src',
  outDir: 'dist',
  entry: 'index.html',
  testsDir: 'tests'
};

export default config;
```

`chipsdev` 负责解析并使用该配置驱动构建与测试流程。具体字段含义与扩展能力由 SDK 文档与 CLI 实现共同约定。

## 4. 常见协作模式

### 4.1 开发者本地工作流（典型路径）

1. 在生态根工作区安装依赖并获得 `chipsdev` 与 `chips`。生态一方工程统一通过根工作区 `npm install` 解析 `chips-sdk` 与 `chips-host`，项目内依赖统一使用正式 semver 版本，由 npm workspaces 自动链接本地包，不再使用 `npm link` / `file:`；根 `package.json` 统一维护 Volta 工具链版本并显式声明 `chips-host` / `chips-sdk` 根级工具依赖，各工作区通过 `volta.extends` 指向根 `package.json`，从而保证在子工程目录直接执行 `chipsdev`、`chips` 时都能解析工作区根的正式二进制。  
2. 使用 `chipsdev create` 从脚手架模板创建工程；  
3. 在工程目录中：
   - 使用 `chipsdev server` 本地开发；
   - 使用 `chipsdev test/lint/e2e/validate` 执行质量门禁；
   - 使用 `chipsdev build/package` 生成 `.cpk`。  
4. 使用 `chips plugin install` 安装 `.cpk` 到本地 Host；  
5. 使用 Host 提供的入口运行并调试插件/应用。

补充约束：

- `chipsdev package` 生成的 `.cpk` 必须符合 Host 安装契约：包根清单 + `dist/` 运行产物 + ZIP Store（零压缩）；
- 不再使用 `chips host plugin install` 历史写法，正式命令统一为 `chips plugin install`；
- `chipsdev create` 在生态根工作区内生成一方工程时，必须自动完成工作区注册、内部依赖适配与 `volta.extends` 写入；
- 若安装失败，应优先检查 `.cpk` 是否包含重复 manifest、历史 `.cpk` 嵌套或使用了压缩 ZIP 方法。

### 4.2 主题包开发与验证

- 通过 `chipsdev create` 创建主题包工程；
- 使用 `chipsdev build` / `chipsdev package` 生成主题包 `.cpk`；
- 使用 `chips plugin install` 安装主题；
- 使用 `chips theme validate` 校验主题契约；
- 通过 Host UI 或 CLI 切换主题进行验证。

## 5. 迁移与历史说明

- 早期文档与对话记录中出现的 `chips dev` 形式统一视为历史称呼，对应当前的 `chipsdev` 命令行工具；
- 当前版本不再提供 `chips dev` 二级命令形式，也不提供兼容别名；
- 所有正式文档与工程脚本在命令行拆分任务完成后应不再引用 `chips dev`，仅使用：
  - `chips`（Host CLI）；
  - `chipsdev`（开发者 CLI）。

## 6. 接口稳定性与测试

- Host CLI (`chips`) 的命令与输出结构以《Chips Host 命令行使用手册》与本指南为准，任何变更需更新对应文档并增加 CLI 冒烟测试；
- 开发者 CLI (`chipsdev`) 的命令与参数以《Chips Dev 开发者命令行手册》与本指南为准，脚手架与模板工程的 CI 应包含至少一条通过 `chipsdev` 跑通的端到端用例；
- 当命令行工具的接口发生破坏性变更时，应同步更新：
  - 生态共用技术文档；
  - 各项目 README 与技术文档；
  - 相关 CI 流水线中的验证脚本。
