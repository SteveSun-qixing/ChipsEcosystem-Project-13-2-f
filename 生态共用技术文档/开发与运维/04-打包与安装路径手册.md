# 打包与安装路径手册

> 文档定位：本文档是薯片生态“打包、安装、开发联调、正式发布”操作路径的主发布位置。  
> 适用范围：`chipsdev`、`chips`、Host 开发工作区、插件 `.cpk`、卡片 `.card`、箱子 `.box`、Host 安装器。  
> 说明：文件结构与格式契约仍分别以《文件格式协议》《CPK打包格式规范》《Manifest配置规范》为准；本文档重点说明“怎么打包、装到哪里、开发时走哪条链路、正式发布走哪条链路”。

## 1. 总体原则

薯片生态当前存在两条必须明确区分的正式路径：

### 1.1 开发环境安装路径

用于生态内部开发、联调、测试、调试。

特点：

- Host 不做正式系统安装；
- Host 运行态落在开发工作区 `.chips-host-dev`；
- 插件与主题通过 `chipsdev plugin ...` 安装到开发工作区；
- 应用插件窗口联调通过 `chipsdev run`；
- 模块插件 Electron 能力联调通过 `chipsdev module invoke`；
- 当前工作区中的源码不会直接被 Host 当成运行时入口，真正被消费的是 `.chips-host-dev/plugins/<pluginId>/` 下的已安装副本。

### 1.2 正式发布使用路径

用于对终端用户分发与正式环境使用。

特点：

- Host 通过正式安装器交付；
- 用户工作区由 `chips` 管理，默认是 `~/.chips-host`；
- 插件/主题以 `.cpk` 形式分发，并通过 `chips plugin install` 安装；
- 终端用户不使用 `chipsdev` 管理正式运行环境。

### 1.3 禁止混用

- 开发联调时，不要把 Host 装到系统里再作为主要开发宿主；
- 正式发布时，不要把 `.chips-host-dev` 当作终端用户运行目录；
- 不允许手工改写 `.chips-host-dev` 或 `~/.chips-host` 来替代正式命令；
- 不允许把“开发工作区中的源码目录”误当成 Host 正式加载入口。

## 2. 打包对象总览

当前生态存在四类打包对象：

| 对象 | 产物扩展名 / 形式 | 主要生产方式 | 主要消费方式 |
| --- | --- | --- | --- |
| 卡片文档 | `.card` | Host `card.pack` / SDK `client.card.pack()` | Host / SDK / 查看器 / 编辑器 |
| 箱子文档 | `.box` | Host `box.pack` / SDK `client.box.pack()` | Host / SDK / 箱子查看器 / 编辑器 |
| 插件与主题 | `.cpk` | `chipsdev package` | `chipsdev plugin install` 或 `chips plugin install` |
| Host 发布包 | `Chips.app`、`.pkg` | `Chips-Host` 安装器构建脚本 | 系统安装器与终端用户 |

统一基线如下：

- `.card`、`.box`、`.cpk` 当前都使用 ZIP Store 模式，也就是 ZIP 存储但不压缩；
- 插件工程根 `manifest.yaml` 是唯一正式清单源；
- `chipsdev build` 只生成构建产物；
- `chipsdev package` 才负责把正式清单与构建产物组装成 `.cpk`；
- Host 安装器不等于插件 `.cpk` 打包，它属于 Host 本体发布链路。

## 3. 开发环境安装路径

本节是开发、调试、联调的正式主路径。

### 3.1 目标

开发环境的目标不是“把 Host 正式安装到系统”，而是：

1. 在生态根工作区恢复工具链与依赖；
2. 让 Host 以开发版方式在 `.chips-host-dev` 中运行；
3. 把插件与主题反复重新打包、重新安装到开发工作区；
4. 用真实 Host 链路验证窗口、主题、模块能力和文件打开行为。

### 3.2 前置准备

在生态根目录执行：

```bash
cd /Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f
npm install
```

这一步会恢复：

- 根级 `chips-sdk` 与 `chips-host` 工具依赖；
- 各工作区共享的 `node_modules`；
- `chipsdev` 与 `chips` 的本地可执行文件解析能力。

补充约束：

- 生态根 `package.json` 是一方仓库唯一正式入口；
- 各子工程依靠 `volta.extends` 继承根工具链；
- 不再使用 `npm link`、手工复制 SDK、手工复制 Host 工具。

### 3.3 开发工作区模型

`chipsdev` 默认使用开发工作区 `.chips-host-dev`：

1. 先从当前目录向上查找 `.chips-host-dev`；
2. 找到则复用；
3. 找不到则在当前项目根创建。

开发工作区中至少会保存以下内容：

- `plugin-runtime.json`
- `plugins/`
- `config.json`
- 开发态 Host 运行状态与日志

当前正式隔离原则：

- 开发工作区与用户工作区不共享主题状态；
- 不共享插件启停状态；
- 不共享运行日志；
- 不共享 Host 配置。

### 3.4 Host 在开发环境中的正式使用方式

开发态 Host 的正式路径是“本地构建 + 开发工作区运行”，而不是正式系统安装。

常见链路如下：

```bash
cd /Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host
npm run build
```

之后：

- `chipsdev start/status/stop/config/logs/plugin/theme/doctor/open` 会把命令委托给 Host CLI，但上下文固定在 `.chips-host-dev`；
- `chipsdev run` 会拉起真实 Electron Host，并在开发工作区中重装启用当前应用插件；
- `chipsdev module invoke` 会拉起真实 Electron Host，并在开发工作区中重装启用目标模块插件。

Host 在开发态的正式职责是：

- 作为插件运行时承载；
- 作为主题运行时承载；
- 作为模块 Electron 能力宿主；
- 作为 `.card/.box` 打开、渲染与调度中心。

### 3.5 插件 / 主题在开发环境中的正式安装顺序

插件与主题的开发态联调建议统一走以下顺序：

```bash
chipsdev build
chipsdev validate
chipsdev package
chipsdev plugin install /绝对路径/产物.cpk
chipsdev plugin enable <pluginId>
```

主题插件额外增加：

```bash
chipsdev theme apply <themeId>
chipsdev theme validate
```

应用插件额外增加：

```bash
chipsdev run
```

模块插件额外增加：

```bash
chipsdev module invoke --capability <capability> --method <method> --input '<json>'
```

### 3.6 开发环境中 `chipsdev package` 的真实行为

`chipsdev package` 当前真实行为基于 `Chips-SDK/cli/index.js`：

1. 读取工程根 `chips.config.mjs`；
2. 读取工程根 `manifest.yaml`；
3. 要求 `config.outDir` 对应的构建目录已经存在；
4. 从 `manifest` 中收集需要一起进入包内的正式资源；
5. 以 ZIP Store 模式生成 `.cpk`；
6. 将工程根 `manifest.yaml` 写入包根；
7. 将构建产物与正式资源写入包内相对路径；
8. 输出到工程 `dist/` 目录。

当前配置默认值：

| 字段 | 默认值 |
| --- | --- |
| `type` | `app` |
| `srcDir` | `src` |
| `outDir` | `dist` |
| `entry` | `app` 类型默认 `index.html`；非 `app` 默认 `src/main.ts` |
| `testsDir` | `tests` |

当前 `.cpk` 文件名规则：

```text
<safePluginId>-<version>.cpk
```

其中：

- `safePluginId` 来自 `manifest.id`；
- 非字母、数字、`_`、`.`、`-` 会被替换成 `_`；
- 输出目录固定为工程根的 `dist/`。

### 3.7 开发环境中会被一起打进 `.cpk` 的资源

`chipsdev package` 当前会把以下正式资源纳入收集范围：

- `manifest.entry`
- `manifest.preview`
- `manifest.ui.layout.contract`
- `manifest.ui.layout.minFunctionalSet`
- `manifest.ui.launcher.icon`
- `manifest.screenshots[].path`
- `manifest.module.provides[].methods[].inputSchema`
- `manifest.module.provides[].methods[].outputSchema`

资源收集规则：

1. 路径必须是相对路径；
2. `chipsdev validate` 会先校验资源是否存在；
3. 打包时按“根目录首段”计算需要复制的目录根；
4. 包内不会重复写入额外的 manifest 副本；
5. 包内不会继续嵌套历史 `.cpk`。

### 3.8 开发环境中不会被打进 `.cpk` 的内容

当前 `chipsdev package` 会显式排除：

- 包内再次出现的 `manifest.yaml`
- `manifest.yml`
- `manifest.json`
- `publish-meta.json`
- 已生成的 `.cpk`

因此，开发者不应把发布辅助文件或旧包继续堆进正式包根。

### 3.9 开发环境中 `chipsdev plugin install` 的真实行为

开发工作区的插件安装由 Host `PluginRuntime.install(...)` 完成，支持三类输入：

1. 插件目录；
2. `.cpk` 文件；
3. `manifest.yaml` 文件路径。

统一行为如下：

1. 解析输入来源；
2. 读取 manifest；
3. 校验 manifest 与资源路径；
4. 将源内容复制到 `.chips-host-dev/plugins/<pluginId>/`；
5. 更新 `plugin-runtime.json`；
6. 若之前已经存在同 `pluginId`，则替换旧副本。

替换语义：

- 替换依据是 `pluginId`，不是文件名；
- 旧副本会先被删除，再复制新副本；
- 旧的启用状态会在运行时记录层被保留；
- 但正式操作顺序仍建议保持 `install -> enable`，不要依赖“旧状态碰巧保留”作为唯一约定。

### 3.10 开发环境中的主题引导

`chipsdev` 当前会在以下场景自动确保开发工作区至少存在一个已启用主题：

- `chipsdev theme ...`
- `chipsdev run`
- `chipsdev open`

当前自动引导行为：

1. 检查开发工作区是否已有已启用主题；
2. 若没有，则要求 `ThemePack/Chips-default/dist/theme.css` 等构建产物可用；
3. 自动安装默认主题；
4. 自动启用默认主题；
5. 若当前工作区还没有 `ui.theme` 配置，则应用默认主题。

这意味着：

- 开发环境里的 Host 首次跑起来时，不要求先做系统级主题安装；
- 但主题工程本身要先有可用构建产物。

### 3.11 开发环境推荐命令链

#### 主题包

```bash
cd ThemePack/Chips-default
npm run build
npm run validate:theme
chipsdev package
cd ../..
chipsdev plugin install /绝对路径/ThemePack/Chips-default/dist/theme.theme.chips-official-default-theme-1.0.0.cpk
chipsdev plugin enable theme.theme.chips-official-default-theme
chipsdev theme apply chips-official.default-theme
chipsdev theme validate
```

#### 卡片 / 布局 / 模块插件

```bash
chipsdev build
chipsdev validate
chipsdev package
chipsdev plugin install /绝对路径/产物.cpk
chipsdev plugin enable <pluginId>
```

#### 应用插件

```bash
chipsdev build
chipsdev validate
chipsdev package
chipsdev plugin install /绝对路径/产物.cpk
chipsdev plugin enable <pluginId>
chipsdev run
```

## 4. 正式发布使用路径

本节是对终端用户与正式分发的路径说明，不是日常开发主路径。

### 4.1 目标

正式发布路径解决的是：

- Host 本体如何对外分发；
- 插件 / 主题如何作为 `.cpk` 分发；
- 终端用户如何在用户工作区安装和启用插件；
- 终端用户如何通过 `chips` 使用已安装的正式运行环境。

### 4.2 Host 正式发布路径

当前 Host 正式发布链路以 `Chips-Host` 安装器为准。

macOS 当前正式命令：

```bash
cd /Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host
npm run build:installer:macos
```

当前构建行为：

1. 构建 Host；
2. 构建存在的内置插件工程；
3. 组装 `Chips.app`；
4. 生成 `.pkg` 安装包。

当前安装器默认关注的内置插件：

- `theme.theme.chips-official-default-theme`
- `com.chips.eco-settings-panel`
- `com.chips.photo-viewer`

其中：

- 默认主题与生态设置面板属于必选内置插件；
- 图片查看器是可选内置插件；
- 图片查看器缺失时，不会阻塞 Host 发布包生成。

当前正式产物形态：

- `Chips.app`
- `Chips-Host-<version>-macos.pkg`

### 4.3 正式用户工作区

终端用户的正式运行环境由 `chips` 管理，默认工作区是：

```text
~/.chips-host
```

此路径与 `.chips-host-dev` 完全不同。

正式用户工作区负责保存：

- 用户态插件安装记录；
- 用户态主题配置；
- 用户态日志与诊断；
- Host 正式运行状态。

### 4.4 正式插件 / 主题发布路径

插件与主题正式发布时，仍然先由工程侧生成 `.cpk`，但安装动作改为用户态：

```bash
chips plugin install /绝对路径/产物.cpk
chips plugin enable <pluginId>
```

主题额外增加：

```bash
chips theme apply <themeId>
chips theme validate
```

正式发布口径：

1. 工程侧通过 `chipsdev build/validate/package` 产出 `.cpk`；
2. 用户侧通过 `chips plugin install` 安装；
3. 只有已启用主题插件才会进入 `chips theme list/current/apply/resolve/validate`；
4. 终端用户不应操作 `.chips-host-dev`。

### 4.5 正式 `.cpk` 安装约束

Host 当前对 `.cpk` 的正式要求如下：

- 必须是 ZIP Store 模式；
- 包根必须存在 `manifest.yaml`、`manifest.yml`、`manifest.json` 或 `*.plugin.json`；
- 运行时产物应位于包内 `dist/`；
- 包内不应嵌套历史 `.cpk`、重复 manifest 或发布辅助文件。

推荐正式工作流：

```bash
chipsdev build
chipsdev package
chips plugin install /绝对路径/产物.cpk
```

### 4.6 正式使用时的文件打开与能力消费

正式安装后的 Host 负责：

- 通过 `chips open <path>` 或系统文件关联打开文档；
- 根据已安装 app 插件的 `file-handler:<ext>` 分发普通文件；
- 根据 `resource-handler:*` 分发资源打开请求；
- 根据已启用主题提供主题运行时；
- 根据已启用插件消费 `installPath + entry` 加载运行产物。

## 5. `chipsdev validate` 与打包门禁

`chipsdev validate` 是 `.cpk` 打包前的正式校验入口。

当前至少校验：

1. `manifest.id/name/version/type` 是否存在；
2. `manifest.permissions` 是否存在且为数组；
3. `runtime.targets` 是否完整；
4. `app` 插件是否声明合法的 `ui.surface`；
5. 非 `app` 插件是否错误声明 `ui.surface`；
6. `capabilityFallbacks` 是否合法；
7. `config.outDir` 是否存在；
8. `config.outDir` 是否非空；
9. manifest 引用的正式资源是否存在。

这意味着：

- `chipsdev build` 是 `chipsdev package` 与 `chipsdev validate` 的前置步骤；
- 构建目录为空也会被视为失败；
- manifest 中写错资源路径会直接阻断打包。

## 6. `.card` 与 `.box` 的打包路径

插件 `.cpk` 不是生态里唯一的打包对象，卡片与箱子同样有正式打包链路。

### 6.1 `.card`

当前正式入口：

- Host `card.pack`
- SDK `client.card.pack(cardDir, outputPath)`

正式语义：

- 输入是目录态卡片工作目录；
- 输出是 `.card` 文件；
- 使用 ZIP Store 模式；
- 打包前后目录结构口径必须一致；
- 编辑引擎保存流程应先编辑工作目录，再导出 `.card`。

### 6.2 `.box`

当前正式入口：

- Host `box.pack`
- SDK `client.box.pack(boxDir, outputPath)`

正式语义：

- 输入是解包态箱子工作目录；
- 输出是 `.box` 文件；
- 使用 ZIP Store 模式；
- 正式消费时通过 `box.openView` 等 Host 链路读取；
- 编辑器创建新箱子与保存现有箱子时，都应通过 `box.pack` 生成最终单文件产物。

## 7. 签名、来源与当前实现边界

当前公共文档中有些历史材料会提到“插件签名”，但基于当前 Host 运行时代码，正式实现边界应理解为：

- 本地开发与本地 `.cpk` 安装默认视为 `source = local`；
- 当 `manifest.source !== 'local'` 时，Host 会要求 `manifest.signature` 存在；
- 当前本地开发打包链路不要求开发者先生成签名再安装；
- 若未来接入市场发布与远端来源校验，应在发布链路文档中单独补充，不应把尚未落地的签名流程写成本地开发前置步骤。

## 8. 常见错误与排查

### 8.1 `未找到构建输出目录`

含义：

- 还没有先执行 `chipsdev build`；
- 或 `chips.config.mjs` 中 `outDir` 与真实构建输出不一致。

### 8.2 `manifest 资源不存在`

含义：

- `manifest.entry`
- `ui.launcher.icon`
- `ui.layout.contract`
- `ui.layout.minFunctionalSet`
- `preview`
- schema 文件

中至少有一个路径写错或文件缺失。

### 8.3 `PLUGIN_PACKAGE_INVALID`

常见原因：

- `.cpk` 不是 ZIP Store；
- 包根缺少 manifest；
- 包结构里混入历史 `.cpk`；
- 包内产物不在 `dist/`；
- 手工打包导致结构和 `chipsdev package` 不一致。

### 8.4 开发环境里还是加载旧代码

常见原因：

- 只执行了 `chipsdev build`，没有重新 `chipsdev package`；
- 只重新打了包，没有重新 `chipsdev plugin install`；
- 误以为 Host 直接读取源码目录，而不是开发工作区中的已安装副本。

### 8.5 主题看不到或切不过去

常见原因：

- 主题只安装了，但没有 `chipsdev plugin enable` 或 `chips plugin enable`；
- 主题构建产物 `dist/tokens.json`、`dist/theme.css` 缺失；
- 当前命令跑在错误工作区里。

### 8.6 开发时把 Host 装到系统里

这不是推荐开发路径。

当前开发基线应为：

- Host 在本地构建；
- Host 状态落在 `.chips-host-dev`；
- 插件通过 `chipsdev` 安装；
- 应用通过 `chipsdev run`；
- 模块通过 `chipsdev module invoke`。

正式安装器只用于发布与终端使用，不作为日常开发主路径。

## 9. 最终结论

当前生态“打包与安装”必须按以下口径理解：

1. 开发环境与正式发布是两条不同路径，不能混用。
2. 开发环境下，Host 不做正式系统安装，主要使用开发工作区 `.chips-host-dev`。
3. 正式发布下，Host 通过安装器交付，用户侧通过 `chips` 管理正式工作区。
4. 插件与主题统一通过 `chipsdev package` 生成 `.cpk`，再分别安装到开发工作区或用户工作区。
5. `.card`、`.box`、`.cpk` 当前统一采用 ZIP Store 模式。
6. `manifest.yaml` 是插件工程唯一正式清单源。
7. Host 实际加载的是“已安装副本”，不是工作区源码目录。
8. 开发调试主路径应优先保证“真实 Host + 开发工作区 + 正式安装副本”这三件事同时成立。
