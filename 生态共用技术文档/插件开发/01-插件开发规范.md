# 插件开发规范

## 开发环境

开发环境是进行插件开发的基础设施。开发者需要安装Node.js和包管理器用于依赖管理。安装薯片开发工具包chips-dev-tools获取开发辅助工具。配置本地开发服务器用于测试和调试。

开发目录结构包括源代码目录src存放TypeScript源码，输出目录dist存放编译后的JavaScript文件，测试目录tests存放测试用例，资源目录assets存放图片、字体等静态资源，配置文件包括package.json、tsconfig.json等。

## 项目初始化

项目初始化使用官方提供的脚手架工具。运行chips create命令创建新插件项目，脚手架会根据插件类型生成相应的项目结构和配置文件。

生成的配置文件包括cpk.config.js打包配置，tsconfig.json TypeScript配置，package.json项目描述文件。开发者需要修改这些配置以适应具体项目需求。

## 类型定义

类型定义文件为插件开发提供类型安全。项目模板自动包含chips.d.ts类型定义文件，其中定义了Bridge API的所有接口类型。

主要类型包括：ChipsBridge全局桥接API对象类型，各种服务的方法调用类型定义，事件回调的类型签名，配置选项的类型定义。开发者应充分利用类型检查减少运行时错误。

## 清单配置

manifest.yaml是插件的核心配置文件，定义了插件的元数据和能力。

必填配置项包括：插件唯一标识符使用反向域名格式，插件显示名称，遵循语义化版本规范的版本号，插件类型如app、card、layout、module、theme，插件入口文件路径。

可选配置项包括：作者信息和联系方式，插件功能描述，图标文件路径，项目主页URL，开源许可证类型，关键词数组用于搜索，截图数组用于展示，依赖声明列出需要的模块及其版本范围，权限声明列出需要的系统能力。

## 打包配置

打包配置文件定义如何将源代码转换为cpk插件包。入口点指定加载的入口文件路径。输出目录指定编译产物的输出位置。压缩选项指定是否使用压缩。

资源处理包括：需要包含的静态资源文件，需要排除的源文件，资源文件的复制规则。

代码转换包括：TypeScript编译选项的目标版本，模块系统的目标格式，源代码映射的生成规则。

## 尺寸单位

薯片生态使用 `cpx`（chips pixel）作为统一的逻辑尺寸单位。

- 逻辑基准宽度：1024cpx
- 换算公式：`px = cpx * (容器宽度 / 1024)`

所有布局常量应以 cpx 声明，渲染前统一换算到 px。换算结果保留 3 位小数。

- 字体：四舍五入到 0.5px
- 边框：最小 1px 可见阈值
- 位移：允许亚像素渲染

SDK 提供 `@chips/layout-math` 统一函数，禁止应用侧重复实现。

## 权限声明

权限声明定义插件需要的系统能力。权限在 manifest.yaml 的 permissions 字段中声明。

常用权限采用命名空间格式：

| 权限名称 | 说明 |
|----------|------|
| `file.read` | 读取文件 |
| `file.write` | 写入文件 |
| `file.stat` | 文件状态 |
| `file.list` | 列出目录 |
| `network.request` | 发起网络请求 |
| `clipboard.read` | 读取剪贴板 |
| `clipboard.write` | 写入剪贴板 |
| `window.control` | 窗口管理 |
| `global-shortcut` | 全局快捷键 |
| `theme.read` | 读取主题 |
| `theme.write` | 应用主题 |
| `i18n.read` | 读取国际化 |
| `i18n.write` | 修改语言 |
| `log.write` | 写入日志 |
| `log.read` | 读取日志 |
| `credential.manage` | 凭证管理 |
| `plugin.manage` | 插件管理 |
| `module.manage` | 模块管理 |
| `box.pack` | 打包箱子 |

权限声明应遵循最小权限原则，只请求实际需要的权限。

## 本地测试

本地测试是开发流程的重要环节。chips dev命令启动开发模式，自动编译源代码并重新加载插件。开发模式下可以实时查看代码修改的效果。

调试功能包括：浏览器开发者工具用于前端调试，日志输出用于追踪执行流程，断点调试用于精确定位问题。

测试策略包括：单元测试验证独立函数和类，集成测试验证组件间协作，端到端测试验证完整功能流程。

## 发布流程

发布流程将插件分发给用户。首先进行版本号规划，遵循语义化版本规范。然后执行打包命令生成cpk文件。接着创建发布说明，描述新版本的功能和改进。最后将cpk文件提交到应用市场。

发布前检查清单包括：所有测试通过，类型检查无错误，打包过程无警告，manifest配置完整正确，图标和截图准备齐全。

## 应用市场

应用市场是插件的分发平台。开发者注册开发者账号后可提交插件。提交时需要填写插件信息和上传cpk文件。应用市场对插件进行审核后发布。

开发者可以在开发者后台管理已发布的插件，包括版本更新、状态切换、信息修改等操作。
