# 应用插件开发

## 概述

应用插件是拥有独立窗口的完整Web应用，是用户直接交互的功能入口。开发应用插件需要遵循特定的技术规范和架构模式。

## 基本结构

应用插件的基本结构包括HTML入口文件、JavaScript或TypeScript代码、CSS样式文件、资源文件等。入口文件由工程根 `manifest.yaml` 中的 `entry` 字段指定，正式约定为指向构建产物，例如 `dist/index.html`。

对于 app 类型构建产物，HTML 中引用的静态资源必须能够在 Host 使用本地 `loadFile()` 加载入口时正常解析。正式约定为输出相对资源路径（例如 `./assets/...`），禁止依赖站点根绝对路径 `/assets/...`。

应用插件入口页必须显式声明正式 `Content-Security-Policy`。当前公共基线要求至少满足：

- 不允许缺省 CSP；
- 不允许在正式入口页为应用运行时放开 `script-src 'unsafe-eval'`；
- 本地资源、iframe 与 worker 所需白名单应按 Host 正式加载链路声明，而不是在控制台 warning 出现后再按单项目临时放宽。

## 前端框架

应用插件推荐使用 React 作为前端框架。React 是薯片生态的官方主技术栈，提供最完善的支持和最佳的开发体验。

生态提供基于 React 的组件库（`@chips/component-library`）和 SDK，确保与系统深度集成。

在生态一方仓库内：

- `@chips/component-library` 与 `chips-sdk` 统一通过根工作区解析；
- 应用插件仓库中的一方依赖统一声明为正式 semver 版本，由生态根工作区自动链接到本地包；
- 禁止继续使用 `file:` sibling 依赖或 `npm link`。
- TypeScript 应用必须直接消费上述包的正式声明文件，禁止在项目内自行补写模块声明兜底。

（旧版 Vue 技术栈已归档，仅维护不新增功能。）

### React 开发规范

- 遵循 Vercel 组合模式：组件采用组合模式、显式变体、上下文分层
- 避免 boolean props 爆炸，使用显式子组件与 slot
- 使用 UI Hooks 调用系统能力（`useTheme`、`useI18n`、`useFile` 等）

## 生命周期

应用插件的生命周期包括安装、激活、运行、停用、卸载等阶段。

安装阶段，插件包被解压到插件目录，manifest信息被注册到系统。

激活阶段，插件被加载到内存，初始化过程执行，创建必要的资源。

运行阶段，插件响应用户操作，处理业务逻辑，与系统服务交互。

停用阶段，插件被从内存卸载，释放占用的资源。

卸载阶段，插件文件被删除，清理注册信息。

## 窗口管理

应用插件通过窗口管理API创建和管理窗口。

创建窗口使用window.chips.invoke方法调用window服务的create子方法，传入窗口配置参数如宽高、位置、标题等。窗口配置支持设置窗口样式如是否可调整大小、是否置顶等。

窗口通信方面，不同窗口之间通过消息机制进行通信。主窗口可以向子窗口发送消息，子窗口也可以向主窗口发送消息。

### 系统启动入口与静默待机

首版应用插件允许通过 `manifest.yaml` 中的 `ui.launcher` 声明系统启动入口。该能力用于让 Host 为应用插件创建正式快捷方式，而不是让插件自行生成系统文件。

当前正式链路如下：

1. 治理类应用或 Host 内部工具调用 `plugin.createShortcut(pluginId, { replace? })`；
2. Host 读取目标插件的 `ui.launcher.displayName` 与 `ui.launcher.icon`；
3. Host 通过平台抽象层生成系统入口：
   - Windows：桌面 `.lnk`
   - macOS：`~/Applications/Chips Apps/*.app`，并由系统索引后显示到启动台
4. 用户点击快捷方式后，Host 统一经 `app-entry -> plugin.launch` 打开目标应用插件；
5. 当所有窗口关闭时，Host 仍保持后台常驻，用户可继续通过快捷方式再次拉起应用。

注意：

- 只有 `type: app` 插件允许声明 `ui.launcher`；
- 插件不得直接调用系统 API 写入 `.lnk`、`.app` 或其他平台快捷方式文件；
- 快捷方式启动与普通应用启动最终都统一收口到 `plugin.launch`。

### 启动图标与运行时 UI 图标的边界

- `ui.launcher.icon` 是操作系统入口图标文件路径，只用于 Host 创建桌面快捷方式、启动台入口或等价系统壳层资源；
- 应用运行时界面中的按钮、菜单、空态、工具栏、治理列表等图标，必须统一使用 `ChipsIcon` 与 `IconDescriptor`；
- 不允许把 `ui.launcher.icon` 当作运行时 UI 图标主链路，也不允许把 `IconDescriptor` 写进 `manifest.yaml` 代替启动图标文件；
- 若治理页拿到的是 `PluginShortcutRecord.iconPath`，应把它视为文件资源 URL 展示；无图标时再回退到运行时 `ChipsIcon`。

### 启动上下文恢复

应用插件如果需要感知“自己为什么被打开”，必须通过正式启动上下文读取，而不是解析命令行或 Electron 私有对象。

SDK 与 Bridge 当前提供同步入口：

```ts
const launchContext = client.platform.getLaunchContext();
```

或：

```ts
const launchContext = window.chips.platform.getLaunchContext();
```

返回结构：

```ts
interface PlatformLaunchContext {
  pluginId?: string;
  sessionId?: string;
  launchParams: Record<string, unknown>;
}
```

典型使用场景：

- 快捷方式启动：`launchParams.trigger === 'app-shortcut'`
- 文件关联打开：`launchParams.trigger === 'file-association'`，并可能包含 `targetPath`
- 开发工作流启动：`launchParams.trigger === 'chipsdev.run'`

例如卡片查看器这类文件处理型应用，应在启动时优先检查 `launchParams.targetPath`，并在存在时直接恢复文件打开状态。

## 数据存储

应用插件的数据存储使用storage API。

键值存储通过storage.get和storage.set方法读写数据。数据以键值对形式保存在本地，不同插件的数据互相隔离。

数据格式方面，支持存储字符串、数字、布尔值、对象、数组等JavaScript数据类型。对象和数组会被序列化为JSON字符串存储。

## 调用系统服务

应用插件通过Bridge API调用系统服务。

通用调用方式使用window.chips.invoke方法，传入服务名、方法名和参数对象。例如调用卡片服务读取卡片使用await window.chips.invoke('card', 'read', {cardId: 'xxx'})。

事件监听使用window.chips.on方法监听系统事件。例如监听卡片创建事件使用window.chips.on('card.created', callback)。

与启动入口直接相关的正式动作包括：

- `plugin.launch(pluginId, launchParams?)`
- `plugin.getShortcut(pluginId)`
- `plugin.createShortcut(pluginId, { replace? })`
- `plugin.removeShortcut(pluginId)`

普通业务应用通常不会为自己创建快捷方式；该能力主要由生态设置面板等治理类应用统一调用。

## UI组件

应用插件可以使用生态提供的UI组件库。

无头组件库提供基础UI组件如按钮、输入框、弹窗等。组件只有逻辑和结构，样式由主题注入。

使用组件时从组件库导入所需组件，传入props配置参数，组件自动渲染对应的HTML结构。

组件支持响应式布局，可以适配不同屏幕尺寸。

## 卡片显示与编辑链路选择

涉及卡片内容时，应用插件必须先确定自己属于哪一类宿主：

- 通用查看态或第三方集成宿主：
  - 复合卡片显示统一使用 `client.card.compositeWindow.render(...)`
  - 基础卡片编辑器统一使用 `client.card.editorPanel.render(...)`
  - 不得自行实现第二套插件源码装载器
- 官方 `Chips-EditingEngine`：
  - 编辑态预览与编辑遵循 `生态共用技术文档/组件库/08-编辑引擎基础卡片装配与编辑运行时标准.md`
  - 复合卡片预览由编辑引擎内部按基础卡片粒度拼装 iframe
  - 编辑面板由编辑引擎本地 `EditorHost` 托管，不再以 `editorPanel` 作为正式主链路

除官方编辑引擎外，其他应用插件不得复制本地基础卡片运行时。

## 主题应用

应用插件通过主题系统实现视觉一致性。

获取当前主题使用 `window.chips.invoke('theme.getCurrent')` 方法；获取完整主题 CSS 使用 `window.chips.invoke('theme.getAllCss')` 方法。

应用主题时优先通过 `ChipsThemeProvider` 或 Host 注入的 CSS 变量消费主题定义，避免在应用内自行维护整套主题样式。

主题切换时系统会发送 `theme.changed` 事件，插件可以监听并更新样式。

## 错误处理

应用插件需要妥善处理各种错误情况。

API错误捕获使用try-catch块包装API调用，错误对象包含错误码和错误信息。

用户友好的错误提示将系统错误转换为用户可理解的提示信息。

错误日志记录错误详情用于调试分析。

## 性能优化

应用插件需要关注性能表现。

代码分割将插件拆分为多个chunk，按需加载减少初始加载时间。

资源懒加载对非关键资源延迟加载，提高首屏渲染速度。

缓存策略合理使用浏览器缓存和本地存储，避免重复请求和数据获取。

## 安全考虑

应用插件需要注意安全问题。

输入验证对用户输入进行验证和清理，防止注入攻击。

敏感数据处理使用系统提供的安全存储，避免在本地明文保存敏感信息。

权限最小化只请求必要的权限，不过度获取用户数据。

## 测试策略

应用插件需要完善的测试覆盖。

单元测试验证业务逻辑和工具函数。

组件测试验证UI组件的渲染和行为。

集成测试验证插件与系统的交互。

端到端测试验证完整用户场景。

## 发布流程

应用插件开发完成后需要打包发布。

开发阶段建议直接在应用插件仓库中执行 `chipsdev run`，由 `chipsdev` 启动完整 Host 底层并加载当前应用插件。

正式构建使用 `chipsdev build` 与 `chipsdev package` 生成 `.cpk` 文件。提交前使用 `chipsdev validate` 检查清单、产物和契约是否完整。

应用插件打包后的 `.cpk` 必须满足 Host 安装契约：包根目录保留工程根 `manifest.yaml`，运行产物位于 `dist/`，压缩方式使用 ZIP Store。开发阶段执行 `chipsdev run` 时，同样以工程根 `manifest.yaml` 作为安装输入，而不是生成额外的 `dist/manifest.yaml`。建议在发布前执行一次 `chips plugin install <cpk路径>`，确认 Host 可直接安装。

当 Host 以工程根 `manifest.yaml` 作为开发态安装输入时，必须将 `entry` 所在的完整运行时根目录（通常是 `dist/`）一起装入插件工作区，不能只复制单个入口文件。

发布到应用市场需要通过审核流程。
