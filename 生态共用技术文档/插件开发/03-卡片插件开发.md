# 卡片插件开发

## 概述

卡片插件为基础卡片提供渲染和编辑能力。每个基础卡片类型对应一个卡片插件，定义该类型卡片在查看器、通用 Host 托管编辑器，以及官方编辑引擎中的正式行为。

当前正式链路有两类消费者：

- Host 通用链路：通过 `card.render/card.renderEditor` 与 SDK `compositeWindow/editorPanel` 消费；
- 官方编辑引擎链路：通过 Host 已安装并启用的卡片插件入口模块导出的 `basecardDefinition` 消费同一组插件导出。

因此，卡片插件必须把“内容渲染/编辑能力 + 配置契约”写成可复用的正式导出，而不是把运行时假设写死在某一个宿主里。

## 插件结构

卡片插件至少应包含以下模块：

- `manifest.yaml`
- `src/index.ts`
- `src/render/`
- `src/editor/`
- `src/schema/`
- `src/shared/`
- 测试目录与示例数据

推荐结构：

```text
src/
  index.ts
  render/
    view.tsx
    runtime.ts
  editor/
    panel.tsx
    runtime.ts
  schema/
    card-config.ts
  shared/
    utils.ts
    i18n.ts
```

其中：

- `render/` 负责查看态内容挂载；
- `editor/` 负责编辑态内容挂载；
- `schema/` 负责默认值、类型定义与配置约束；
- `shared/` 负责富文本清洗、格式化、翻译等可复用逻辑。

## manifest 配置

卡片插件的 `manifest.yaml` 必须满足以下要求：

- `type: card`
- `capabilities.cardTypes` 明确声明支持的基础卡片类型
- `entry` 指向构建产物入口，例如 `dist/index.mjs`

示例：

```yaml
id: chips.basecard.richtext
name: 富文本基础卡片插件
version: 0.1.0
type: card
entry: dist/index.mjs
capabilities:
  cardTypes:
    - base.richtext
permissions: []
```

约束：

- `capabilities.cardTypes` 是 Host 分发与编辑引擎描述符映射的统一依据；
- 若插件需要兼容 `RichTextCard -> base.richtext`、`ImageCard -> base.image` 这类别名，应在宿主描述符层处理，不应把别名判断散落在业务组件中。

## 正式导出契约

卡片插件入口必须导出以下正式能力：

```ts
export interface IconDescriptor {
  name: string;
  style?: "outlined" | "rounded" | "sharp";
  fill?: 0 | 1;
  wght?: number;
  grad?: number;
  opsz?: number;
  decorative?: boolean;
  label?: string;
}

export interface BasecardRenderContext {
  container: HTMLElement;
  config: Record<string, unknown>;
  themeCssText?: string;
  resolveResourceUrl?(resourcePath: string): Promise<string>;
  releaseResourceUrl?(resourcePath: string): Promise<void> | void;
}

export interface BasecardResourceImportRequest {
  file: File;
  preferredPath?: string;
}

export interface BasecardResourceImportResult {
  path: string;
}

export interface BasecardArchiveImportRequest {
  file: File;
  preferredRootDir?: string;
  entryFile?: string;
}

export interface BasecardArchiveImportResult {
  rootDir: string;
  entryFile: string;
  resourcePaths: string[];
}

export interface BasecardEditorContext {
  container: HTMLElement;
  initialConfig: Record<string, unknown>;
  onChange(next: Record<string, unknown>): void;
  resolveResourceUrl?(resourcePath: string): Promise<string>;
  releaseResourceUrl?(resourcePath: string): Promise<void> | void;
  importResource?(input: BasecardResourceImportRequest): Promise<BasecardResourceImportResult>;
  importArchiveBundle?(input: BasecardArchiveImportRequest): Promise<BasecardArchiveImportResult>;
  deleteResource?(resourcePath: string): Promise<void>;
}

export function renderBasecardView(ctx: BasecardRenderContext): (() => void) | void;
export function renderBasecardEditor(ctx: BasecardEditorContext): (() => void) | void;

export interface BasecardDefinition {
  pluginId: string;
  cardType: string;
  displayName: string;
  description?: string;
  icon?: IconDescriptor;
  aliases?: readonly string[];
  commitDebounceMs?: number;
  previewPointerEvents?: 'native' | 'shielded';
  createInitialConfig(baseCardId: string): Record<string, unknown>;
  normalizeConfig(input: Record<string, unknown>, baseCardId: string): Record<string, unknown>;
  validateConfig(config: Record<string, unknown>): { valid: boolean; errors: Record<string, string> };
  collectResourcePaths?(config: Record<string, unknown>): string[];
  renderView(ctx: BasecardRenderContext): (() => void) | void;
  renderEditor?(ctx: BasecardEditorContext): (() => void) | void;
}

export const basecardDefinition: BasecardDefinition;
```

正式要求：

- 两个导出都必须是“挂载到容器并返回 cleanup”的纯函数；
- `basecardDefinition` 是官方编辑引擎进入正式运行时链路的唯一插件入口；
- `basecardDefinition.cardType` 必须与 `manifest.yaml` 中的 `capabilities.cardTypes[*]` 对齐；
- `basecardDefinition.icon` 必须输出正式 `IconDescriptor`，供编辑引擎卡箱库与其他运行时 UI 统一渲染；
- 若插件需要在编辑引擎预览态屏蔽 iframe 内部指针交互，应通过 `previewPointerEvents = 'shielded'` 正式声明，而不是要求宿主按具体卡片类型写死特殊分支；
- 导出函数不得直接写文件、直接创建窗口或直接依赖宿主状态机；
- 编辑器导出必须允许被直接挂载到普通 DOM 容器，而不是假定自己必然运行在 Host 托管 iframe 中；
- 所有清理逻辑都必须在返回的 cleanup 中闭环。

## 查看态渲染开发

查看态渲染的正式职责是：根据配置把内容挂载到给定容器。

推荐实现方式：

1. 在 `render/view.tsx` 中创建纯 DOM 结构；
2. 在 `render/runtime.ts` 中处理主题 CSS 注入、根节点挂载与清理；
3. 在 `src/index.ts` 中把 `renderBasecardView` 暴露给宿主。

查看态约束：

- 不直接假定外层一定有边框、圆角或卡片壳层；
- 内容宽度以宿主提供的容器为准；
- 主题颜色、边框、文本色优先使用 `--chips-*` token；
- 宿主可能把该导出挂载在 Host 生成的 iframe 文档中，也可能挂载在编辑引擎创建的单卡 iframe 中。
- 若查看态需要访问卡片内部文件资源，应优先消费宿主提供的 `resolveResourceUrl/releaseResourceUrl`；当宿主未提供时，才回退到配置中的卡片根目录相对路径。

## 编辑器开发

编辑器的正式职责是：把配置编辑 UI 挂载到给定容器，并通过 `onChange` 输出新配置。

推荐实现方式：

1. 在 `editor/panel.tsx` 中实现编辑器界面；
2. 在 `editor/runtime.ts` 中处理容器尺寸、根节点样式与清理；
3. 把局部输入节流、富文本清洗等逻辑收口在插件内部；
4. 通过 `onChange` 输出新的配置快照。

编辑器约束：

- 不直接调用 `CardService`、Bridge 写文件或其他持久化接口；
- 不把卡片切换、会话 flush、防抖提交职责写进插件；
- 不要求宿主必须提供 iframe 事件桥；
- 资源型编辑器应优先使用宿主提供的 `resolveResourceUrl/releaseResourceUrl/importResource/importArchiveBundle/deleteResource`；
- 卡片内部资源写回配置时只记录“相对于卡片根目录”的路径；
- 不得把 `blob:`、`data:`、系统绝对路径等运行时 URL 持久化成卡片内部正式资源路径；
- 若需要销毁逻辑，必须通过根节点 cleanup 或显式销毁函数闭环。

## 与宿主链路的关系

### Host 通用链路

- Host 通过 `card.render` 或 `card.renderEditor` 解析插件；
- Host 负责生成正式 iframe 文档与事件协议；
- SDK `compositeWindow/editorPanel` 负责把这些文档交付给应用。

### 官方编辑引擎链路

- 编辑引擎通过 `plugin.query({ type: 'card' })` 读取当前工作区已安装且已启用的卡片插件；
- 编辑引擎只会加载插件 `manifest.entry` 指向的构建产物，例如 `dist/index.mjs`；
- 插件入口模块必须导出 `basecardDefinition`，编辑引擎据此建立运行时描述符注册表；
- `renderBasecardView` 被挂载到编辑引擎自己创建的单卡 iframe；
- `renderBasecardEditor` 被直接挂载到编辑面板 DOM 容器；
- 编辑器会话、防抖提交、自动保存由编辑引擎运行时负责。

重要边界：

- 普通应用和第三方集成不得自行扫描插件目录并直接 import 插件源码；
- 官方编辑引擎允许根据 Host 运行时返回的已安装插件记录加载其构建产物入口，但不得回退到工作区源码扫描；
- 插件实现不能因为某个宿主的例外而破坏通用导出契约。

## 配置数据结构

每种卡片类型都应定义正式的配置 schema。

推荐做法：

- 在 `schema/card-config.ts` 中声明类型；
- 在同文件中提供默认配置；
- 在宿主描述符层完成最终归一与校验；
- 在插件内部只消费“已归一”的配置。

示例：

```ts
export interface BasecardConfig {
  card_type: 'RichTextCard';
  content_format: 'markdown';
  content_source: 'inline' | 'file';
  content_text?: string;
  content_file?: string;
  locale?: string;
  theme?: string;
}

export const defaultBasecardConfig: BasecardConfig = {
  card_type: 'RichTextCard',
  content_format: 'markdown',
  content_source: 'inline',
  content_text: '',
  locale: 'zh-CN',
  theme: '',
};
```

富文本基础卡片补充约束：

- 正式持久化内容只能是 Markdown，不得把编辑器运行时 HTML 直接写回配置；
- 纯文本字符数不超过 `200` 时保存到 `content_text`；
- 超过 `200` 时保存为卡片根目录 `.md` 资源文件，并在配置中写入 `content_file`；
- 编辑态悬浮工具框、选区状态等都属于运行时状态，不进入配置。

网页基础卡片补充约束：

- 正式类型标识为 `base.webpage`，运行时别名可以是 `WebPageCard`；
- 正式来源字段只允许两类：远程 `source_url`，或卡片根目录相对网页目录 `bundle_root + entry_file`；
- 当来源为网页 ZIP 包时，编辑器应通过 `importArchiveBundle(...)` 让宿主先校验并解压，再把整个网页目录写入卡片根目录；
- 网页 ZIP 包的正式入口文件必须是 `index.html`；若压缩包只有一个顶层目录且其中存在 `index.html`，宿主可以在导入时自动扁平化到最终网页目录；
- 网页包正式配置应同时保存 `resource_paths`，用于宿主持久化阶段重建资源清单和删除旧网页资源；
- 网页基础卡片显示模式分为两类：
  - `fixed`：固定为宽:高 `7:16` 视口，并在窗口内滚动网页内容；
  - `free`：按网页内容高度展开，但最高不超过当前宽度的 `20` 倍；
- 网页基础卡片查看态必须把稳定视口信息传递给嵌入页：
  - 父层到网页的正式消息类型为 `chips:webpage-card:viewport`；
  - 网页包/同源网页应可消费 `window.__CHIPS_WEBPAGE_CARD_VIEWPORT__`、CSS 变量 `--chips-webpage-card-*`，并监听 `chips:webpage-card:viewport-change` 事件；
  - 若嵌入页希望跨源回传自身内容高度，应向父层发送 `chips:webpage-card:content-height`；
- 网页视口协议中的 `height` 表示传递给网页内部的稳定测量视口高度：
  - `fixed`：等于固定比例 `7:16` 对应的正式视口高度；
  - `free`：优先对齐宿主当前真实窗口高度，并限制不超过固定比例高度；父层最终展开后的长卡高度只用于宿主布局，不能再次回写给网页内部，否则会形成测量反馈。
- 网页视口协议中的 `baseHeight` 表示内容暂不可测或回退场景下的最小外层高度。
- 对网页 ZIP 包，官方查看态可注入稳定虚拟视口层，把常见 `vh/dvh/svh/lvh` 高度单位和 `window.innerHeight` 等读取尽量映射到协议视口高度，以降低 `free` 模式下长网页误判窗口高度的风险。
- 若远程网页因为 `X-Frame-Options`、`frame-ancestors` 或等效策略拒绝 iframe 嵌入，卡片查看态只显示正式错误提示，不做降级绕过。

## 资源引用

渲染组件可能需要引用卡片文件中的资源。

正式约束：

- 相对路径资源必须相对于卡片根目录解析，运行时可通过 Host 或本地运行时注入的资源基准地址生效；
- 卡片内部资源路径统一从卡片文件根目录开始记录，不相对于 `content/`；
- 插件不直接假定自己拥有文件系统权限；
- 资源 URL 的生成由 Host 或宿主运行时完成，不由渲染组件私自拼系统绝对路径。

编辑态补充：

- Host 托管编辑器链路中，资源操作通过正式 `chips.card-editor:resource-*` 协议和 SDK `resources` 桥接完成；
- `importResource(...)` 返回的 `path` 必须是卡片根目录相对路径；
- `releaseResourceUrl(...)` 用于释放 `blob:` 等临时资源 URL；
- 若宿主未提供资源桥接，而当前编辑器又必须依赖内部资源操作，应显式报错，不得私接文件系统。

## 生命周期管理

卡片插件必须具备完整生命周期：

- 初始化阶段：创建 DOM、绑定事件、同步初始配置；
- 更新阶段：由宿主重新挂载或推送新配置；
- 销毁阶段：清理事件监听、定时器、编辑器实例和临时节点。

禁止事项：

- 把全局事件监听遗留在页面上；
- 依赖浏览器刷新才能释放状态；
- 把 cleanup 留成 TODO 或空实现。

## 主题适配

卡片插件必须适配统一主题系统。

正式要求：

- 使用 `--chips-*` 变量而不是写死主题颜色；
- 查看态与编辑态都必须使用同一套 token 体系；
- 宿主可能以“完整主题 CSS 注入”或“抽取后的变量注入”方式提供主题值，插件不得依赖其中某一种专有实现。

## 错误处理

卡片插件需要处理以下错误：

- 配置字段缺失或非法；
- 资源加载失败；
- 编辑器实例初始化失败；
- 富文本或嵌入内容需要清洗时的安全问题。

要求：

- 错误应局部隔离在当前基础卡片范围内；
- 不抛出会让宿主整页崩溃的未处理异常；
- 对用户可见的错误信息应尽量可理解。

## 测试策略

卡片插件需要至少覆盖以下测试：

- 查看态挂载测试；
- 编辑态挂载测试；
- 配置归一与校验测试；
- cleanup 与重复挂载稳定性测试；
- 在 Host 通用链路中的集成测试；
- 在编辑引擎本地运行时中的集成测试。

链路补充测试要求：

- 验证通过 SDK `compositeWindow/editorPanel` 调用时，插件在 Host 托管链路中行为正确；
- 验证通过编辑引擎本地运行时挂载时，单卡 iframe 预览与编辑器会话行为正确；
- 同一份插件导出在两条正式链路中的主题适配结果应保持一致。
