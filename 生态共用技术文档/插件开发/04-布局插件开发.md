# 布局插件开发

## 概述

布局插件为箱子提供页面级展示与交互能力。  
它的正式定位不是“解析卡片文件”，而是“消费 Host 提供的箱子会话能力，并把条目摘要、资源和交互组织成一个完整前端页面”。

主发布边界：

- 箱子文件结构以 `生态共用技术文档/文件格式规范/02-箱子文件格式规范.md` 为准；
- 箱子运行时取数契约以 `生态共用技术文档/协议与契约/08-箱子布局运行时契约.md` 为准；
- 本文档定义布局插件自身的开发方式、导出约定和工程边界。

## 工程创建

官方布局插件工程的正式初始化入口为：

```bash
chipsdev create layout <targetDir>
```

当前该命令会生成：

- `manifest.yaml`
- `chips.config.mjs`
- `src/view/`
- `src/editor/`
- `src/schema/`
- `src/shared/`
- `contracts/`
- `i18n/`
- `tests/`

生成工程默认不包含项目级 `需求文档/`、`技术文档/` 与 `开发计划/` 目录；这些文档只保留在脚手架仓与具体业务仓内部维护。

## 插件结构

布局插件至少应包含以下模块：

- `manifest.yaml`
- `src/index.ts`
- `src/view/`
- `src/editor/`
- `src/schema/`
- `src/shared/`
- 测试目录与示例箱子数据

推荐结构：

```text
src/
  index.ts
  view/
    page.tsx
    runtime.ts
  editor/
    panel.tsx
    runtime.ts
  schema/
    layout-config.ts
  shared/
    algorithm.ts
    i18n.ts
    transforms.ts
```

其中：

- `view/` 负责查看态页面挂载；
- `editor/` 负责布局参数编辑与预览；
- `schema/` 负责默认配置、归一和校验；
- `shared/` 负责排序、分组、布局计算和可复用转换逻辑。

## manifest 配置

布局插件的 `manifest.yaml` 必须满足以下要求：

- `type: layout`
- `entry` 指向构建产物入口，例如 `dist/index.mjs`
- `layout.layoutType` 声明公开布局类型标识
- `layout.displayName` 声明布局显示名称

示例：

```yaml
id: chips.layout.grid
name: 网格布局插件
version: 1.0.0
type: layout
entry: dist/index.mjs
permissions:
  - box.read
  - theme.read
  - i18n.read
layout:
  layoutType: chips.layout.grid
  displayName: 网格布局
```

约束：

- `layout.layoutType` 必须与箱子 `metadata.active_layout_type` / `content.layout_configs` 使用的键一致；
- Host 通过 `plugin.getLayoutPlugin({ layoutType })` 解析布局插件；
- 布局插件消费箱子数据时正式依赖 `box.read`，而不是 `file.*` 或网络直连权限。

## 正式导出契约

布局插件入口模块必须导出 `layoutDefinition`。

其中 `BoxSessionInfo`、`BoxEntrySnapshot`、`BoxEntryPage`、`BoxEntryQuery`、`ResolvedRuntimeResource` 以 `08-箱子布局运行时契约.md` 为准。

```ts
export interface IconDescriptor {
  name: string;
  style?: "outlined" | "rounded" | "sharp";
  fill?: 0 | 1;
  wght?: number;
  grad?: number;
  opsz?: number;
  decorative?: boolean;
  label?: string;
}

export interface BoxLayoutRuntime {
  listEntries(query?: BoxEntryQuery): Promise<BoxEntryPage>;
  readEntryDetail(request: {
    entryIds: string[];
    fields: Array<
      'documentInfo'
      | 'coverDescriptor'
      | 'previewDescriptor'
      | 'runtimeProps'
      | 'status'
    >;
  }): Promise<Array<{ entryId: string; detail: Record<string, unknown> }>>;
  renderEntryCover(entryId: string): Promise<{
    title: string;
    coverUrl: string;
    mimeType: string;
    ratio?: string;
  }>;
  resolveEntryResource(request: {
    entryId: string;
    resource: {
      kind: 'cover' | 'preview' | 'documentFile' | 'custom';
      key?: string;
      sizeHint?: { width?: number; height?: number };
    };
  }): Promise<ResolvedRuntimeResource>;
  readBoxAsset(assetPath: string): Promise<ResolvedRuntimeResource>;
  prefetchEntries(request: {
    entryIds: string[];
    targets: Array<'cover' | 'preview' | 'documentInfo'>;
  }): Promise<void>;
  openEntry(entryId: string): Promise<{
    mode: 'document-window' | 'external';
    documentType?: 'card' | 'box';
    windowId?: string;
    pluginId?: string;
    url?: string;
  }>;
}

export interface BoxLayoutRenderContext {
  container: HTMLElement;
  sessionId: string;
  box: BoxSessionInfo;
  initialView: BoxEntryPage;
  config: Record<string, unknown>;
  runtime: BoxLayoutRuntime;
}

export interface BoxLayoutEditorContext {
  container: HTMLElement;
  entries: BoxEntrySnapshot[];
  initialConfig: Record<string, unknown>;
  onChange(next: Record<string, unknown>): void;
  readBoxAsset?(assetPath: string): Promise<ResolvedRuntimeResource>;
  importBoxAsset?(input: { file: File; preferredPath?: string }): Promise<{ assetPath: string }>;
  deleteBoxAsset?(assetPath: string): Promise<void>;
}

export interface BoxLayoutDefinition {
  pluginId: string;
  layoutType: string;
  displayName: string;
  icon?: IconDescriptor;
  createDefaultConfig(): Record<string, unknown>;
  normalizeConfig(input: Record<string, unknown>): Record<string, unknown>;
  validateConfig(config: Record<string, unknown>): {
    valid: boolean;
    errors: Record<string, string>;
  };
  getInitialQuery?(config: Record<string, unknown>): BoxEntryQuery | undefined;
  renderView(ctx: BoxLayoutRenderContext): (() => void) | void;
  renderEditor?(ctx: BoxLayoutEditorContext): (() => void) | void;
}

export const layoutDefinition: BoxLayoutDefinition;
```

正式要求：

- `layoutDefinition.layoutType` 必须与 `manifest.yaml` 中的 `layout.layoutType` 对齐；
- `layoutDefinition.icon` 若声明，必须是正式 `IconDescriptor`，供查看器、编辑引擎卡箱库与治理页统一渲染；
- `renderView` 和 `renderEditor` 都必须是“挂载到容器并返回 cleanup”的纯函数；
- 配置默认值、归一和校验必须收口在插件内部，宿主只消费统一结果；
- 导出函数不得直接写文件、直接读文件系统、直接发起网络定位箱子条目；
- 查看态页面根节点应至少铺满宿主容器可视区，并允许内容自然增长；宿主侧也必须为布局容器提供正式的全宽与可收缩约束，例如 `width: 100%`、`max-width: 100%`、`min-width: 0`；不要用固定 `height: 100%` 搭配 `overflow: hidden` 截断页面，否则会破坏宿主中的正式局部滚动，或因横向尺寸链路不完整导致右侧留白；
- 所有计时器、观察器、事件订阅和对象 URL 都必须在 cleanup 中闭环。

## 查看态渲染开发

查看态渲染的正式职责是：使用摘要态数据尽快生成可交互页面，并在需要时向 Host 申请更多数据。

推荐实现方式：

1. 在 `view/page.tsx` 中实现页面结构；
2. 在 `view/runtime.ts` 中处理容器挂载、主题样式注入和清理；
3. 先用 `initialView` 渲染首屏；
4. 滚动、悬停、切换分组或打开详情时，再通过 `runtime.*` 继续取数。

查看态约束：

- 默认展示封面、标题、标签、摘要、状态等信息，不默认展开完整卡片；
- 默认展示卡片封面网格、标题、标签、摘要、状态等信息，不默认展开完整卡片；
- 只在当前交互确有需要时调用 `readEntryDetail` 或 `resolveEntryResource`；
- 显示封面时应优先使用 `renderEntryCover(entryId)`；
- 封面、预览和卡片入口都必须通过 Host 统一资源句柄消费；
- 点击封面或条目卡片时，必须调用 `runtime.openEntry(entryId)`；
- 当布局只是“直接显示文档封面”时，应优先把 `renderEntryCover(entryId)` 返回的 `title/coverUrl/ratio` 接入正式 `EmbeddedDocumentFrame` 组件；
- 若当前条目明确是卡片，也可以继续使用 `CardCoverFrame` 这一卡片专用包装组件；
- 不得把运行时资源 URL、临时绝对路径、凭证信息持久化回箱子文件。

## 编辑器开发

布局编辑器的正式职责是：编辑布局配置，并管理该布局在当前箱子中的自有资源。

推荐实现方式：

1. 在 `editor/panel.tsx` 中实现参数表单；
2. 在 `schema/layout-config.ts` 中定义默认配置、归一函数和字段校验；
3. 通过 `onChange` 输出新的配置快照；
4. 通过 `importBoxAsset/deleteBoxAsset` 管理该箱子专属资源。

编辑器约束：

- 不直接写 `content.yaml`、`structure.yaml` 或 `assets/` 文件；
- 不把宿主的会话 ID、临时 URL、绝对路径写进配置；
- 若布局需要背景图、占位图、主题映射文件等用户资源，必须作为箱子资源写入 `assets/`，并在 `asset_refs` 中记录；
- 若只是插件内置静态图标、默认样式或通用插画，应保留在布局插件包内，而不是复制进箱子。

## 与 Host 链路的关系

### 官方宿主层

查看器与编辑引擎消费布局插件时，正式共用同一条 `Host box-service -> SDK -> 应用壳层` 链路。

当前正式职责拆分如下：

1. Host `box-service` 通过 `plugin.query({ type: 'layout' })` 读取已安装布局插件；
2. Host 根据 `installPath + entry` 加载布局插件构建产物，并校验 `layoutDefinition`；
3. Host 对外提供 `box.listLayoutDescriptors/readLayoutDescriptor/normalizeLayoutConfig/validateLayoutConfig/getLayoutInitialQuery` 等动作，统一收口描述符、默认配置、配置归一与校验；
4. Host 通过 `box.renderLayoutFrame` 生成查看态布局文档，通过 `box.renderLayoutEditor` 生成编辑器文档；
5. SDK 只负责这些 Host 动作的类型化封装、iframe 装载和运行时消息桥接；
6. 查看器、编辑引擎等上层应用只调用 SDK 或 Bridge 正式接口，不直接加载布局插件。

约束：

- 查看器与编辑引擎不得各自复制一份布局插件加载逻辑；
- SDK 不承载布局插件主实现，SDK 仍只负责 API、文档窗口封装与桥接；
- 官方应用若需新增布局宿主能力，应先收口到 Host `box-service`，再接入具体应用仓。

### 查看链路

1. 应用优先调用 `client.document.window.render({ filePath })` 打开文档；在明确当前文件一定是箱子时，也可以使用 `client.box.documentWindow.render({ boxFile })`；
2. SDK 对 `.box` 文件调用 Host `box.inspect`，读取 `active_layout_type`、`layout_configs` 与条目摘要快照；
3. SDK 再调用 Host `box.readLayoutDescriptor`、`box.normalizeLayoutConfig`、`box.getLayoutInitialQuery`，由 Host 统一解析布局描述符、归一配置并生成首屏查询；
4. SDK 调用 `box.openView` 创建箱子查看会话；
5. Host 返回 `sessionId`、箱子信息与首批条目摘要；
6. SDK 调用 `box.renderLayoutFrame`，由 Host 加载布局插件并把 `box + initialView + config + runtime` 注入 `layoutDefinition.renderView`；
7. SDK 把 Host 返回的 `documentUrl` 装载到 iframe，并桥接 `chips.box-layout:*` 运行时消息；
8. 布局插件运行中继续通过 `runtime.*` 请求数据与资源；
9. 关闭页面时，正式销毁链路同时回收 `box.closeView(sessionId)` 与 `box.releaseRenderSession(sessionId)`。

### 编辑链路

1. 编辑引擎把正式 `.box` 解包到当前工作区会话目录，并维持正式自动保存链路；
2. 箱子预览与卡片窗口一样，统一调用 `client.document.window.render({ filePath: session.boxFile, documentType: 'box' })`；
3. SDK 仍按查看链路调用 Host `box.inspect/readLayoutDescriptor/normalizeLayoutConfig/getLayoutInitialQuery/openView/renderLayoutFrame`，由 Host 生成正式箱子查看文档；
4. 右侧编辑面板调用 `client.box.editorPanel.render({ layoutType, entries, initialConfig, resources })`；
5. SDK 调用 Host `box.renderLayoutEditor`，由 Host 加载同一个布局插件并执行 `layoutDefinition.renderEditor`；
6. 编辑器文档通过 `chips.box-layout-editor:*` 协议把配置变更与资源请求回传给 SDK；
7. 编辑引擎收到 `onChange` 后，继续通过 `client.box.normalizeLayoutConfig(...)` 走 Host 正式归一逻辑，再写回会话目录并自动保存。

重要边界：

- 查看器、编辑器都不应复制一套 Host 资源解析逻辑；
- 应用层永远不能自己装载布局插件，也不能回退到工作区源码目录；
- 布局插件只能通过正式回调与 `box.*` 契约访问箱子数据；
- 箱子消费默认是摘要优先、按需加载，不允许把“全量预解析”写成布局插件前提。

## 配置与资源模型

布局插件配置正式落位于：

```yaml
active_layout_type: "chips.layout.grid"
layout_configs:
  chips.layout.grid:
    schema_version: "1.0.0"
    props:
      sort_mode: "manual"
      background:
        mode: "image"
        asset_path: "assets/layouts/grid/background/hero.webp"
      top_region:
        mode: "html"
        html: "<section>...</section>"
    asset_refs:
      - "assets/layouts/grid/background/hero.webp"
```

设计要求：

- 自定义参数写入 `props`；
- 箱子专属资源路径写入 `asset_refs`；
- 同一箱子可以同时保存多个布局类型的配置；
- 切换布局时，只切换 `active_layout_type` 与对应配置，不改变条目引用集合。

常见自定义参数包括：

- 排序方式、背景区域、顶部内容区；
- 信息密度、标题显隐、标签显隐；
- 排序键、分组键、默认筛选条件；
- 悬停行为、空态样式、占位图选择；
- 看板列定义、瀑布流列宽、列表紧凑度等布局特有字段。

## 交互与按需取数

布局插件常见的运行时取数场景：

- 首屏列表：直接使用 `initialView`
- 滚动分页：调用 `listEntries`
- 悬停或抽屉预览：调用 `readEntryDetail`
- 真实封面：调用 `renderEntryCover`
- 预览图：调用 `resolveEntryResource`
- 背景图、占位图、装饰素材：调用 `readBoxAsset`
- 即将进入视口的内容：调用 `prefetchEntries`

正式原则：

- 优先请求“最少够用”的字段；
- 只对即将消费的条目做温和预取；
- 同一条目详情应在插件内部做会话级缓存；
- 大数据量布局必须使用虚拟列表、窗口化渲染或增量布局计算。

## 性能与失败处理

性能建议：

- 首屏优先渲染摘要块，不等待真实资源全部返回；
- 把排序、分组和布局计算与渲染层解耦；
- 对容器尺寸变化、滚动事件和拖拽反馈做节流与批处理；
- 对资源句柄按 `cacheKey` 做复用，避免重复申请。

失败处理要求：

- 条目资源失败时显示占位态，而不是让整页崩溃；
- `BOX_RESOURCE_NOT_FOUND`、`BOX_RESOLVE_FAILED` 等错误应映射为布局可理解的状态；
- 不做无限重试，不自行实现另一套解析回退链路；
- 会话关闭后必须停止后续请求与异步状态写入。

## 测试与发布

布局插件至少应覆盖以下测试：

- `createDefaultConfig/normalizeConfig/validateConfig` 的契约测试；
- `renderView` 在摘要态、分页态、错误态下的渲染测试；
- `renderEditor` 的参数编辑与资源引用测试；
- 大数据量、滚动分页、分组切换和窗口缩放场景的表现测试。

发布前检查：

- `manifest.layout.layoutType` 与 `layoutDefinition.layoutType` 一致；
- 布局配置仅写入 `props` 与 `asset_refs`；
- 未把绝对路径、临时 URL、凭证写入箱子；
- `chipsdev build`、`chipsdev package`、`chipsdev validate` 全部通过；
- 至少完成一次 Host 侧安装与箱子会话联调验证。
