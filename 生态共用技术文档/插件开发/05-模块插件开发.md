# 模块插件开发

## 1. 概述

模块插件是安装到 Host 中的无界面能力插件。模块插件不创建窗口，不承载页面渲染，不以 DOM 挂载和前端插槽为正式能力边界。

模块插件对外提供的是正式能力契约，用于被 Host、应用插件、卡片插件、布局插件以及其他模块插件复用。

## 2. 正式定位

模块插件与其他插件类型的边界如下：

- 应用插件负责用户界面和业务流程；
- 卡片插件负责基础卡片渲染和编辑；
- 布局插件负责箱子布局呈现；
- 模块插件负责无界面能力提供；
- 主题插件负责视觉样式。

模块之间允许相互调用，但必须继续经过 Host 的正式模块服务。

## 3. 正式调用模型

模块插件正式使用 capability + method 模型，而不是 `slot + mount` 模型。

调用方通过正式模块服务完成：

- 查询 provider：`module.listProviders`
- 解析 capability：`module.resolve`
- 调用方法：`module.invoke`
- 查询任务：`module.job.get`
- 取消任务：`module.job.cancel`

Host 负责：

- provider 注册；
- capability 解析；
- 权限校验；
- schema 校验；
- 模块 runtime 激活；
- 同步调用和异步任务调度；
- 日志和审计。

调用责任划分：

- Host 管理统一调用治理框架；
- 模块开发者定义 capability / method / schema 等业务契约；
- 调用方开发时对照正式契约接入，运行时由 Host 解析 provider 并转发请求。

## 4. Manifest 规范

模块插件必须提供根目录 `manifest.yaml`，并满足以下正式要求：

```yaml
id: chips.module.example
name: Example Module
version: 0.1.0
type: module
entry: dist/index.mjs
permissions:
  - file.read
module:
  apiVersion: 1
  runtime: worker
  activation: onDemand
  provides:
    - capability: example.process
      version: 1.0.0
      methods:
        - name: run
          mode: sync
          inputSchema: contracts/run.input.schema.json
          outputSchema: contracts/run.output.schema.json
  consumes:
    - capability: example.normalize
      versionRange: ^1.0.0
```

补充约束：

- `type` 必须为 `module`；
- `entry` 指向 Host 加载的正式模块运行时代码；
- 模块正式能力声明位于 `module.provides`；
- 模块正式依赖声明位于 `module.consumes`；
- 不再把 `capabilities: ["module.xxx"]` 作为模块正式能力契约主入口。

## 5. 工程结构

推荐结构如下：

```text
src/
└─ index.ts
contracts/
├─ <method>.input.schema.json
└─ <method>.output.schema.json
tests/
├─ unit/
└─ integration/
```

官方模块脚手架默认不得生成：

- React 运行时模板；
- `runtime.tsx`；
- 主题 CSS 注入逻辑；
- i18n UI 词典模板；
- `mountModule(container)` 之类页面挂载入口。

## 6. 模块入口设计

模块入口推荐导出统一能力定义对象，例如：

```ts
export default defineModule({
  capability: "example.process",
  async activate(ctx) {},
  async deactivate(ctx) {},
  methods: {
    async run(ctx, input) {
      return { ok: true, input };
    },
  },
});
```

说明：

- 模块运行时上下文由 Host 注入；
- 模块不得自行构造第二套 Host 通信通道；
- 模块调用其他模块时，应继续使用 Host 提供的 `ctx.module.invoke(...)`。

## 6.1 转换链路补充

文件转换系统中的“编排模块”和“原子转换模块”都只是普通模块插件：

- `Chips-FileConversion-Plugin` 属于编排模块；
- `Chips-CardtoHTML-Plugin`、`Chips-HtmltoPDF-Plugin`、`Chips-HtmltoImage-Plugin` 属于原子转换模块；
- 它们统一通过 capability + method 契约接入 Host；
- 编排关系只体现在模块之间的能力调用，不新增新的插件类别或 Host 路由类别。

## 7. SDK 调用示例

```ts
import { createClient } from "chips-sdk";

const client = createClient();

const result = await client.module.invoke({
  capability: "example.process",
  method: "run",
  input: {
    sourcePath: "/workspace/demo.txt",
  },
});

if (result.mode === "sync") {
  console.log(result.output);
}
```

异步任务示例：

```ts
const started = await client.module.invoke({
  capability: "converter.card.export",
  method: "convert",
  input: {
    cardFile: "/workspace/demo.card",
    targetFormat: "html",
  },
});

if (started.mode === "job") {
  const snapshot = await client.module.job.get(started.jobId);
  console.log(snapshot.job.status);
}
```

## 8. 测试策略

模块插件至少应覆盖以下测试层次：

- 单元测试：method 行为、schema 约束、错误归一；
- 集成测试：Host provider 注册、能力调用、任务取消；
- 脚手架测试：生成工程 build/test/validate/package；
- 跨模块调用测试：验证模块依赖和组合路径。

重点门禁：

- 不得留下 TODO、占位实现或未使用的临时代码；
- `manifest.yaml`、`contracts/`、`README`、测试和脚手架模板必须同步更新；
- 模块禁用和卸载必须具备可验证的回收行为。

## 9. 推荐工作流

```bash
chipsdev create module Chips-ModulePlugin/my-module
cd Chips-ModulePlugin/my-module
npm install
npm run build
npm test
npm run validate
chipsdev package
```

## 10. 最佳实践

- capability 和 method 命名保持稳定；
- 把业务能力 schema 作为正式契约提交到仓库；
- 把外部依赖显式写入 `module.consumes`；
- 模块间调用统一走 Host；
- 每次功能迭代同步更新 README 与测试、契约等运维资料；不再把 `需求文档/`、`技术文档/`（或“技术手册”）与 `开发计划/` 目录当作脚手架的默认产物，而是由各业务仓按需维护，确保模板只输出必要的运行时代码与配置。
