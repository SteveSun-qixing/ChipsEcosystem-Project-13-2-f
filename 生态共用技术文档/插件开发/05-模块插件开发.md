# 模块插件开发

## 1. 概述

模块插件是生态中的共享功能模块，面向 Host 与其他插件提供可复用能力。典型场景包括视频播放器、图片查看器、Markdown 渲染器、富文本编辑器等。

模块插件与应用插件的最大区别在于：

- 不创建独立窗口；
- 由调用方按需挂载到业务 slot；
- 入口产物通常是 `dist/index.mjs` 形式的前端模块；
- 主题、多语言、日志与生命周期治理仍需遵循生态统一规范。

## 2. 当前正式链路

当前正式可交付链路分为两层：

1. 插件治理层：模块插件和其他插件一样，通过 Host 安装、启用、禁用、卸载；
2. 挂载治理层：调用方通过 Bridge / SDK 的 `module.mount/unmount/query/list` 管理 slot 与模块插件的挂载关系。

正式 Bridge 动作如下：

- `module.mount({ slot, moduleId }) -> { module }`
- `module.unmount({ slot }) -> { ack: true }`
- `module.query({ slot }) -> { module: ModuleState | null }`
- `module.list() -> { modules: ModuleState[] }`

其中：

- `slot` 是逻辑插槽标识，例如 `viewer.preview`、`editor.richtext`；
- `moduleId` 是模块插件 ID，必须指向已安装、已启用且 `type: module` 的插件；
- `ModuleState` 至少包含 `slot/moduleId/entry?/capabilities/active/mountedAt`。

## 3. Manifest 规范

模块插件必须提供根目录 `manifest.yaml`，并满足以下正式要求：

```yaml
id: chips.module.markdown-renderer
name: Markdown 渲染模块
version: 0.1.0
type: module
entry: dist/index.mjs
description: 供查看器或编辑器复用的 Markdown 渲染模块。
author:
  name: Chips Team
  email: dev@example.com
capabilities:
  - module.markdown-renderer
permissions:
  - theme.read
  - i18n.read
engines:
  chips: "^0.1.0"
os:
  - darwin
  - win32
  - linux
```

补充约束：

- `type` 必须为 `module`；
- `entry` 当前正式使用字符串路径，不使用旧草稿中的 `exports` 字段；
- 模块能力当前正式使用 `capabilities: string[]`；
- 若模块运行时要从 Host 读取主题或语言状态，应显式声明 `theme.read`、`i18n.read` 等权限；
- 旧文档中的 `moduleType` 字段当前不是 Host 已实现的正式清单字段，不应写入正式工程。

## 4. 入口与运行时设计

模块插件推荐保持单入口导出，即由 `src/index.ts` 汇总模块的公共运行时能力。推荐结构如下：

```text
src/
├─ index.ts
├─ module/
│  ├─ runtime.tsx
│  ├─ types.ts
│  └─ i18n.ts
└─ config/
   └─ logging.ts
```

推荐做法：

- 把对外导出的挂载函数、上下文类型、事件类型统一收口在 `src/index.ts`；
- React 组件层只负责 UI 与交互，不直接承担 Bridge 协议拼装；
- 通过 provider / context 把主题、语言、模块状态隔离到单独运行时层，避免 boolean 模式扩张；
- `unmount` 必须完整释放 React root、事件监听、定时器与 DOM 副作用。

说明：

- 模块入口中的具体导出名称属于模块工程自身的实现面；
- Host 当前正式对外冻结的是插件治理、Bridge 动作、Manifest 校验与安装产物约束；
- 因此模块工程内部 API 可以按业务抽象，但不得与正式 Bridge 契约冲突。

## 5. 主题与多语言

模块插件前端仍需严格接入生态主题系统与多语言系统：

- 主题状态应优先通过 `chips-sdk` 读取 Host 当前主题；
- 若模块渲染需要完整主题 CSS，可通过 `client.theme.getAllCss()` 获取并注入模块容器；
- 语言状态应通过 `client.i18n.getCurrent()` 获取，并监听 `language.changed` 事件刷新本地词典；
- 应用内 UI 文案应保存在模块工程本地 `i18n/*.json` 中，不把业务文案依赖到 Host `i18n.translate`。

推荐模式：

1. 使用 `chips-sdk` 建立 `Client`；
2. 启动时并行读取 `theme.getCurrent/getAllCss/i18n.getCurrent`；
3. 在本地 React 运行时层集中管理主题快照、语言快照与错误状态；
4. UI 组件只消费格式化后的文案和样式状态。

## 6. SDK 调用示例

```ts
import { createClient } from "chips-sdk";

const client = createClient();

const mounted = await client.module.mount("viewer.preview", "chips.module.markdown-renderer");
const snapshot = await client.module.query("viewer.preview");
const modules = await client.module.list();
await client.module.unmount("viewer.preview");
```

约束：

- 调用方必须自行保证 slot 命名稳定、唯一、可读；
- 模块插件工程不得继续示例旧动作 `module.load/unload`；
- 如需查询挂载结果，优先使用 SDK 已解包后的 `ModuleState`。

## 7. 测试策略

模块插件至少应覆盖以下测试层次：

- 单元测试：运行时状态变更、语言切换、主题样式注入、错误归一；
- 组件测试：React 组件的挂载、更新、销毁；
- 集成测试：通过 `chipsdev build/test/lint/validate/package` 验证正式工程链路；
- CLI 创建测试：通过 `chipsdev create module` 验证模板生成后可在生态工作区内完成安装、构建、测试、校验与打包。

重点门禁：

- 不能留下 TODO、占位实现或未使用的临时代码；
- `manifest.yaml`、`chips.config.mjs`、`README`、测试与模板目录必须同步更新；
- 模块卸载必须具备可验证的资源清理行为。

## 8. 推荐工作流

正式开发模块插件时，推荐统一使用以下流程：

```bash
chipsdev create module Chips-ModulePlugin/my-module
cd Chips-ModulePlugin/my-module
npm install
npm run lint
npm run build
npm test
npm run validate
chipsdev package
```

如工程位于生态根工作区内，`chipsdev create` 会自动完成：

- 工作区注册；
- `volta.extends` 写入；
- 正式 semver 依赖保留。

## 9. 最佳实践

- slot 命名按业务边界设计，不使用临时字符串；
- 模块能力标识保持稳定，避免频繁改名；
- React 结构优先使用显式 provider、子组件与 slot 组合，不通过 boolean props 扩张模式；
- 主题、语言、Bridge 调用统一收敛到运行时层，避免组件散落直接调用；
- 每次功能迭代后同步更新需求文档、技术文档、开发计划与 README。
