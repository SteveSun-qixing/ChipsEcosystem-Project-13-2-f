# ID生成规范

## 概述

ID是生态中各种实体的唯一标识符，包括卡片ID、箱子ID、用户ID、插件ID等。ID生成遵循一致性、唯一性、可读性的原则。

## ID类型

卡片ID标识每个卡片文件，采用十位六十二进制格式。字符集包含0-9、a-z、A-Z，共62个字符。ID长度为10位，理论支持约83万亿个不重复ID。

箱子ID标识每个箱子文件，采用与卡片ID相同的格式。

用户ID标识每个用户账户，采用更长的长度确保安全性。使用16位六十二进制格式。

插件ID标识每个插件，采用反向域名格式。例如com.example.my-plugin。

资源ID标识每个资源文件，采用哈希值格式。使用内容的SHA256哈希前16位十六进制字符。

### 资源ID与服务关联

根据架构手册第4.5节，资源ID与 `resource` 服务紧密关联：

| 资源ID格式 | 生成方式 | 对应服务动作 |
|---|---|---|
| SHA256前16位十六进制 | 内容哈希 | `resource.resolve()` |

### 资源ID使用场景

- **resource.resolve(resourceId)**：根据资源ID解析出资源的实际访问路径（URI）
- **resource.readMetadata(resourceId)**：读取资源的元数据信息
- **resource.readBinary(resourceId)**：读取资源二进制内容

资源ID的优势在于：
1. **内容寻址**：相同内容的资源始终生成相同的ID，便于去重和缓存
2. **完整性验证**：可通过哈希比对验证资源完整性
3. **分布式友好**：不依赖中心化ID生成服务

## 生成算法

卡片和箱子ID使用分布式ID生成算法，确保在多台机器上生成的ID不会冲突。

算法采用Twitter雪花算法变体。最高位固定为0保证ID为正数。41位时间戳记录毫秒级时间。10位机器标识区分不同生成节点。12位序列号在同毫秒内自增。

时间戳使用自定义纪元，从2024年1月1日开始。可以使用约70年。

机器标识从配置中心获取，确保每个运行实例有不同的标识。序列号在每次生成后自增，满4096后回滚。

### 机器标识与平台抽象层

根据架构手册第2层 PAL（平台抽象层）设计，机器标识的获取需要通过平台无关的方式：

1. **PAL层接口**：`IDGenerator.getMachineId()` 提供统一的机器标识获取接口
2. **实现分离**：不同平台（Windows/macOS/Linux）的机器标识获取逻辑在PAL层收敛
3. **安全存储**：机器标识存储在平台安全存储中（如Keychain/DPAPI），防止篡改

```
PAL层接口定义：
interface IDGeneratorService {
  getMachineId(): Promise<string>;
  generateId(type: 'card' | 'box' | 'user'): Promise<string>;
}
```

## 验证规则

ID必须符合对应类型的格式要求。卡片和箱子ID必须是10位六十二进制字符。用户ID必须是16位六十二进制字符。插件ID必须符合反向域名格式。资源ID必须是32位十六进制字符。

系统提供ID验证函数，可以检查ID格式是否正确。

## 冲突处理

由于采用分布式ID生成算法，冲突概率极低。理论情况下，每毫秒每机器可生成4096个ID，冲突概率约为1/83万亿。

当意外发生ID冲突时，系统检测到后重新生成。重新生成时会使用新的时间戳或序列号。

## 兼容性

不同版本的ID生成算法可能略有差异。旧版本生成的ID在新版本中应该仍然有效。跨版本ID应该被正确识别和处理。

## 存储格式

ID在内存中使用字符串类型存储。在文件系统中，ID作为文件名或字段值存储。在数据库中，ID作为主键或索引字段。

## 显示和输入

用户界面中显示ID时，通常使用简化或友好的形式。例如可以显示ID的前几位加上省略号，如abc123...。

系统输入框接受完整ID或简化形式。简化形式会自动匹配唯一的ID。

## 迁移和导入

导入外部数据时，外部ID需要转换为生态ID格式。转换时保留外部ID作为参考，生成新的生态ID作为主键。

数据迁移时，ID需要保持一致性。迁移工具会处理ID的映射和转换。

## 最佳实践

应用程序不应依赖ID的顺序或规律。ID只是唯一标识，不包含任何业务含义。存储ID时使用字符串类型，避免数值溢出。展示ID时考虑使用简化形式。
