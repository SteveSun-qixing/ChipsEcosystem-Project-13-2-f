# 薯片内核设计

> **版本**：vNext 架构设计冻结稿  
> **层级**：L2 Host Kernel  
> **适用阶段**：阶段18及后续重构实施阶段

## 内核概述

薯片内核是整个生态系统的核心组件，运行在Electron主进程中，为上层所有应用和插件提供基础运行时环境。内核采用TypeScript编写，采用微内核设计模式，将核心功能最小化并保持稳定，扩展功能通过模块化方式添加。

内核的设计目标是提供高性能、高可用、可扩展的基础设施，使得上层业务逻辑能够专注于功能实现而无需关注底层复杂性。内核不直接处理任何业务逻辑，它只负责协调和管理，让所有业务逻辑运行在独立的插件或服务进程中。

## L2层级定位

| 层级 | 名称 | 核心职责 | 只允许依赖 | 禁止依赖 |
|---|---|---|---|---|
| L1 | PAL | 平台能力抽象 | OS/Electron 原生能力 | 业务插件 |
| **L2** | **Host Kernel** | **router/registry/event/lifecycle** | **PAL** | **应用页面代码** |
| L3 | Host Services | file/resource/config/theme/i18n 等 | Kernel | 插件 UI |

**依赖约束**：
- L2只允许依赖L1（PAL）
- L2禁止依赖应用页面代码（L12）
- L3及以下层级可以依赖L2

## 消息路由器（核心组件）

### 路由架构

消息路由器是内核最核心的组件，负责所有跨进程、跨插件的消息传递。路由器的设计参考了分布式系统中的消息队列理念，但针对本地桌面环境进行了优化。

**路由统一键格式**：`namespace.action`
- 例如：`file.read`、`theme.apply`、`card.parse`
- 支持命名空间分组，便于权限管理和模块划分

**匹配模式**：
- `exact`：精确匹配，如 `file.read`
- `prefix`：前缀匹配，如 `file.*` 匹配所有file开头的动作
- `wildcard`：通配符匹配，如 `*.*` 匹配所有动作

### 路由契约描述符（实现约束）

所有服务注册动作必须提供 `RouteDescriptor`，禁止"只注册 handler 不注册契约"：

```typescript
interface RouteDescriptor<I, O> {
  key: `${string}.${string}`;                    // namespace.action 格式
  schemaIn: string;                              // 输入JSON Schema ID
  schemaOut: string;                             // 输出JSON Schema ID
  permission: string[];                          // 所需权限列表
  timeoutMs: number;                             // 超时时间（毫秒）
  idempotent: boolean;                           // 是否幂等
  retries: 0 | 1 | 2 | 3;                        // 重试次数
  handler: (input: I, ctx: RouteInvocationContext) => Promise<O>;
}
```

**示例**：
```typescript
const routeDescriptor: RouteDescriptor<FileReadRequest, FileReadResponse> = {
  key: 'file.read',
  schemaIn: 'schemas/file.read.request.json',
  schemaOut: 'schemas/file.read.response.json',
  permission: ['file.read'],
  timeoutMs: 5000,
  idempotent: true,
  retries: 0,
  handler: async (input, ctx) => {
    // 业务逻辑
  }
};
```

内核在注册时生成 `route-manifest`，供SDK与测试门禁校验。

### 调用上下文标准

`RouteInvocationContext` 固定字段，所有路由调用必须携带：

```typescript
interface RouteInvocationContext {
  requestId: string;                              // 唯一请求ID，用于链路追踪
  caller: {                                       // 调用方信息
    id: string;
    type: 'kernel' | 'service' | 'plugin' | 'app';
    pluginId?: string;
    windowId?: string;
    permissions?: string[];
  };
  timestamp: number;                              // 请求时间戳
  deadline?: number;                              // 超时截止时间
}
```

### 路由级超时与熔断

- **超时必配**：每路由必须配置 `timeoutMs`
- **超时响应**：超时时返回 `ROUTE_TIMEOUT` 错误码
- **熔断机制**：高频失败路由触发软熔断
  - 窗口期内快速失败
  - 可恢复探测
  - 指标：p50/p95延迟、超时率、错误率、缓存命中率

## 服务注册表

### 核心职责

服务注册表是内核的另一个核心组件，负责管理所有可用的服务。服务可以是内核内置的核心服务，也可以是插件提供的扩展服务。

注册表维护服务的元数据，包括：
- 服务名称
- 版本
- 状态
- 依赖关系
- 接口定义

### 服务生命周期状态

服务状态转换：
```
unloaded -> loading -> ready -> running -> stopping -> stopped/error
```

| 状态 | 说明 |
|------|------|
| unloaded | 未加载 |
| loading | 加载中 |
| ready | 已准备 |
| running | 运行中 |
| stopping | 停止中 |
| stopped/error | 已停止/错误 |

**内核负责**：状态可观测
**内核不负责**：服务业务细节

### 依赖管理

注册表支持服务的依赖管理：
- **强依赖**：必须在目标服务启动前启动
- **弱依赖**：可以并行启动
- **懒加载**：只有当服务首次被调用时才真正启动

## 事件总线

### 事件模型

事件总线是内核的消息发布订阅机制，用于组件之间的解耦通信。

**统一事件元信息**：
```typescript
interface EventPayload {
  id: string;                                     // 事件唯一ID
  name: string;                                   // 事件名称（点语义）
  source: string;                                 // 事件来源
  data: unknown;                                  // 事件数据
  timestamp: number;                              // 时间戳
  metadata?: Record<string, unknown>;             // 元数据
}
```

### 事件命名规范

- **统一采用点语义**：`theme.changed`、`language.changed`、`plugin.enabled`
- **禁止混用**：如 `theme:changed` 等冒号格式

### 事件匹配

- **层级事件命名空间**：例如 `card.*` 匹配所有卡片相关事件，`card.created` 只匹配卡片创建事件
- **通配符订阅**：可以订阅所有事件或某个前缀下的所有事件
- **过滤器**：支持订阅时指定数据过滤器

### 投递模式

- **广播模式**：将事件发送给所有订阅者
- **单播模式**：只发送给第一个订阅者
- **同步/异步处理**：同步处理确保事件按顺序处理，异步处理提高系统吞吐量

## 生命周期管理器

### 状态机

生命周期管理器负责管理所有组件的启动和停止顺序。

**状态定义**：
```typescript
type ComponentLifecycleState = 
  | 'unloaded'    // 未加载
  | 'loading'     // 加载中
  | 'ready'       // 已准备
  | 'running'     // 运行中
  | 'stopping'    // 停止中
  | 'stopped'     // 已停止
  | 'error';      // 错误状态
```

### 管理策略

- **启动顺序**：从叶子节点开始逐层向上启动
- **停止顺序**：从根节点开始逐层向下停止
- **热更新支持**：插件更新时，先停止旧版本，启动新版本，然后切换流量

### 生命周期事件

组件在生命周期中会经历多个状态转换，生命周期管理器会触发相应的生命周期事件：
- `lifecycle.ready`：服务就绪
- `lifecycle.running`：服务运行中
- `lifecycle.stopping`：服务停止中
- `lifecycle.stopped`：服务已停止

## 模块加载器

### 职责

模块加载器负责动态加载和执行JavaScript/TypeScript模块。

### 支持格式

- CommonJS
- ES Module
- AMD

### 隔离机制

- 每个插件的模块在独立的上下文中运行
- 避免插件之间的全局变量污染
- 模块缓存：避免重复加载相同的模块

## 统一错误对象

所有内核和服务返回的错误必须符合以下标准格式：

```typescript
interface StandardError {
  code: string;                                  // 错误码
  message: string;                                // 错误消息
  details?: unknown;                              // 错误详情
  retryable?: boolean;                            // 是否可重试
}
```

### 错误码规范

| 前缀 | 来源 | 示例 |
|------|------|------|
| `ROUTE_` | 路由层 | `ROUTE_TIMEOUT`、`ROUTE_NOT_FOUND` |
| `SERVICE_` | 服务层 | `SERVICE_UNAVAILABLE`、`SERVICE_INTERNAL` |
| `KERNEL_` | 内核层 | `KERNEL_INITIALIZE_FAILED` |

## 日志标准字段

内核日志必须包含以下统一字段：

```typescript
interface LogEntry {
  traceId: string;                               // 链路追踪ID
  requestId: string;                             // 请求ID
  pluginId?: string;                              // 插件ID
  namespace?: string;                             // 命名空间
  action?: string;                               // 动作名
  durationMs?: number;                           // 耗时
  result?: 'success' | 'error';                  // 结果
  errorCode?: string;                            // 错误码
}
```

## 架构原则约束

### 不可违反原则

1. **中心路由原则**：服务间调用必须走内核路由，禁止跨服务直接import调用
2. **Bridge隔离原则**：插件仅通过 `window.chips.*` 与系统能力交互
3. **平台无关性**：平台差异收敛到PAL，核心不直连平台API
4. **契约优先原则**：接口、事件、manifest、token、布局均契约化

### 依赖方向

- **允许**：`Ln → Ln-1` 或 `Ln → 公共契约层`
- **禁止**：跨层跳跃（例如L12直调L5）

## 验收标准

### 分层验收

- L2职责边界清晰（router/registry/event/lifecycle）
- L2只依赖L1（PAL）
- L2不依赖应用页面代码（L12）

### 契约验收

- 所有服务动作提供 `RouteDescriptor`
- 调用上下文符合 `RouteInvocationContext` 标准
- 错误对象符合 `StandardError` 标准
- 事件命名采用点语义

### 链路验收

- Transport入站生成 `requestId`，贯穿Runtime Client、Kernel、Service
- 事件链路记录 `source -> target`
