# 平台抽象层设计

> **版本**：vNext 已落地口径  
> **层级**：L1 PAL（Platform Abstraction Layer）  
> **当前实现状态**：Desktop / Headless 已正式落地；Web / Mobile 契约已冻结、实现入口已预留

## 架构归属声明（2026-04-08）

- Host 主责 L1-L9 运行时链路，并随 Host 一起发布。
- SDK 只消费 PAL 上层公开契约，不承载 PAL 主实现。
- Host Core、Kernel、Services 不允许直接感知 Electron、浏览器或移动原生 API；一切平台差异必须压回 PAL 与 Shell。

## 1. PAL 2.0 总览

PAL 2.0 已从“桌面窗口 + 文件系统”的单体适配器，升级为“能力分面 + 结构化能力快照”的正式模型。

当前 `chips-host` 已提供：

- `DesktopPalAdapter`
- `HeadlessPalAdapter`
- 面向未来 Shell 复用的 PAL Contracts

当前 PAL 公开的核心平台标识：

```ts
type HostKind = "desktop" | "web" | "mobile" | "headless";
type PalPlatformId = NodeJS.Platform | "web" | "android" | "ios" | "server";
```

## 2. L1 定位

| 层级 | 名称 | 核心职责 | 允许依赖 | 禁止依赖 |
|---|---|---|---|---|
| L1 | PAL | 原生能力抽象、能力矩阵、平台差异收敛 | OS / Electron / 浏览器 / 原生移动 API | Kernel、Services、插件业务逻辑 |
| L2 | Host Kernel | 路由、事件、生命周期、权限上下文 | PAL | 平台 API |
| L3 | Host Services | 资源、主题、插件、模块、surface 等正式服务 | Kernel + PAL | 平台 API |

强制约束：

- Host Core / Kernel / Services 中禁止直接使用 `process.platform`。
- Host Core / Kernel / Services 中禁止直接 `import` Electron API。
- 平台能力必须先进入 PAL，再由上层服务域消费。

## 3. 能力分面

当前 PAL 2.0 已冻结以下分面接口：

| 分面 | 责任 |
|---|---|
| `environment` | 平台信息、宿主类型、结构化能力快照 |
| `surface` | 跨平台界面容器打开、聚焦、尺寸、状态、关闭、枚举 |
| `storage` | 受控读写、目录枚举、监听、复制、移动、删除 |
| `selection` | 打开文件、保存文件、消息框、确认框 |
| `transfer` | 路径打开、外链打开、在系统中定位、分享 |
| `association` | 文件关联 / URL scheme / share target 能力快照 |
| `device` | 主屏、全部屏幕、电源状态 |
| `systemUi` | 剪贴板、托盘、通知、全局快捷键 |
| `background` | 保活与防休眠语义 |
| `ipc` | named pipe / unix socket / shared memory 通道 |
| `offscreenRender` | HTML 导出 PDF / 图片 |

当前仍保留一组 legacy aliases：

- `window`
- `fs`
- `dialog`
- `clipboard`
- `shell`
- `platform`
- `screen`
- `tray`
- `notification`
- `shortcut`
- `power`

这些别名仅用于兼容现有桌面链路与现有测试；新的跨平台语义一律以分面接口为准。

## 4. `surface` 取代单一 `window`

PAL 的正式 UI 容器语义已经从 `window` 升级为 `surface`：

```ts
type SurfaceKind = "window" | "tab" | "route" | "modal" | "sheet" | "fullscreen";
```

`surface.open()` 当前支持三类目标：

- `target.type = "plugin"`
- `target.type = "url"`
- `target.type = "document"`

`surface` 是跨平台主语义：

- Desktop Shell：当前实际落地为 `window`
- Web Shell：预留映射到 `route / tab / modal`
- Mobile Shell：预留映射到 `fullscreen / sheet`
- Headless Shell：默认不支持交互式 surface

这意味着：

- 上层正式不再以 `BrowserWindow` 作为统一容器模型；
- `window.*` 现在是桌面语义别名，不再是未来新增能力的主入口。

## 5. 能力快照

PAL 不再对上层暴露扁平字符串数组能力，而是暴露结构化能力快照：

```ts
interface PalCapabilitySnapshot {
  hostKind: HostKind;
  platform: PalPlatformId;
  facets: {
    surface: { supported: boolean; interactive: boolean; supportedKinds: SurfaceKind[] };
    storage: { localWorkspace: boolean; sandboxFilePicker: boolean; remoteBacked: boolean };
    selection: { openFile: boolean; saveFile: boolean; directory: boolean; multiple: boolean };
    transfer: { upload: boolean; download: boolean; share: boolean; externalOpen: boolean; revealInShell: boolean };
    association: { fileAssociation: boolean; urlScheme: boolean; shareTarget: boolean };
    device: { screen: boolean; power: boolean; network: boolean };
    systemUi: { clipboard: boolean; tray: boolean; globalShortcut: boolean; notification: boolean };
    background: { keepAlive: boolean; wakeEvents: boolean };
    ipc: { namedPipe: boolean; unixSocket: boolean; sharedMemory: boolean };
    offscreenRender: { htmlToPdf: boolean; htmlToImage: boolean };
  };
}
```

当前已落地快照语义：

- Desktop：`surface.supported = true`，`supportedKinds = ["window"]`
- Headless：`surface.supported = false`，其余后台能力按需要开放

## 6. 当前已落地的 Shell / Adapter 组合

| Host Shell | PAL Adapter | 当前状态 | 说明 |
|---|---|---|---|
| `DesktopHostShell` | `DesktopPalAdapter` | 已落地 | 正式桌面宿主，绑定 Electron IPC 与渲染协议 |
| `HeadlessHostShell` | `HeadlessPalAdapter` | 已落地 | 社区服务器、批处理、CI、后台任务宿主 |
| `WebHostShell` | Web PAL | 预留 | 契约已冻结，当前仓内未交付实现 |
| `MobileHostShell` | Mobile PAL | 预留 | 契约已冻结，当前仓内未交付实现 |

## 7. 文件与传输模型

PAL 仍允许 Desktop / Headless 在内部使用本地绝对路径，但上层正式语义已经分拆：

- `storage` 负责受控读写
- `selection` 负责用户选择文件 / 保存目标
- `transfer` 负责打开、下载、导出、分享
- `association` 负责系统入口能力

约束：

- `filePath` 不是新的跨平台正式模型；
- Web / Mobile Shell 未来可以用沙箱句柄、远端对象地址或系统文档 token 实现同一语义。

## 8. 错误与不支持语义

PAL 必须显式返回正式错误，不能通过空实现“假支持”。

当前未支持能力统一走：

- `PAL_UNSUPPORTED`

例如：

- Headless 上调用 `surface.open`
- 当前桌面适配器未实现的 `transfer.share`

## 9. 质量门禁

PAL 相关改动必须满足以下门禁：

1. 所有平台差异必须收敛到 PAL / Shell。
2. Host Core、Services 中不得新增平台特判。
3. `platform.getCapabilities()` 必须返回结构化快照，不得回退到字符串数组。
4. 新平台能力必须先定义“支持 / 降级 / 不支持”语义，再进入实现。
5. 如果新增公开能力影响 Bridge、SDK、Manifest 或插件运行时，必须同步回写共享文档。
