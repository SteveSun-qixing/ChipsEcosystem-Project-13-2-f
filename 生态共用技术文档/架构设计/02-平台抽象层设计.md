# 平台抽象层设计

> **版本**：vNext 架构设计冻结稿  
> **层级**：L1 PAL（Platform Abstraction Layer）  
> **适用阶段**：阶段18及后续重构实施阶段

## 平台抽象层概述

平台抽象层位于Electron宿主之上、薯片内核之下，是整个生态系统的技术底座。PAL采用适配器模式，为每个操作系统实现对应的适配器，运行时根据当前操作系统自动选择合适的适配器。

**核心职责**：屏蔽底层操作系统的差异，为上层提供统一的跨平台接口。

**设计原则**：
- 平台无关性原则：内核和服务层代码不直接调用平台特定API，一律通过PAL进行适配
- 最小接口原则：只暴露上层真正需要的平台能力，避免过度抽象导致的复杂性
- 适配器模式：每个操作系统对应一个适配器，通过运行时检测自动选择

## L1层级定位

| 层级 | 名称 | 核心职责 | 只允许依赖 | 禁止依赖 |
|---|---|---|---|---|
| **L1** | **PAL** | **平台能力抽象** | **OS/Electron 原生能力** | **业务插件** |
| L2 | Host Kernel | router/registry/event/lifecycle | PAL | 应用页面代码 |
| L3 | Host Services | file/resource/config/theme/i18n 等 | Kernel | 插件 UI |

**依赖约束**：
- L1只允许依赖OS/Electron原生能力
- L1禁止依赖业务插件
- L2及以上层级依赖L1

## 窗口管理

窗口管理是平台抽象层最重要的功能之一，负责创建和管理应用窗口。

**核心能力**：
- 窗口创建：`create(options)` - 支持配置标题、初始尺寸、最小尺寸、是否可调整大小、是否置顶、是否全屏等
- 窗口状态：`getState()` / `setState(state)` - 获取和设置窗口状态
- 窗口聚焦：`focus()` - 将窗口置前
- 窗口调整：`resize(width, height)` - 调整窗口尺寸
- 窗口关闭：`close()` - 关闭窗口

**平台差异处理**：
- 不同操作系统的窗口行为存在显著差异，包括窗口样式、动画效果、拖拽行为等
- 平台抽象层将这些差异标准化为统一的事件类型
- 应用只需要监听统一的事件名称，无需关心底层是哪个操作系统

## 文件系统访问

文件系统访问接口是平台抽象层提供的另一组核心接口。

**核心能力**：
- 路径处理：`normalize(path)` / `join(...paths)` / `resolve(path)` - 路径规范化、拼接、解析
- 文件读写：`readFile(path)` / `writeFile(path, data)` - 文件读写操作
- 文件状态：`stat(path)` - 获取文件元数据
- 文件列表：`list(dir)` - 列出目录内容
- 文件监控：`watch(path, callback)` - 监听文件变化

**平台差异处理**：
- Windows使用反斜杠而Unix使用正斜杠，PAL自动处理
- 路径分隔符由PAL统一转换
- 文件系统大小写敏感度差异由PAL处理

---

## 系统对话框

系统对话框用于与应用用户进行交互。

**核心能力**：
- 文件选择对话框：`openFile(options)` / `saveFile(options)` - 支持单文件、多文件、文件夹选择
- 消息对话框：`showMessage(options)` - 支持信息、警告、错误、确认等类型
- 颜色选择器：`pickColor(options)` - 颜色选择
- 自定义对话框：`showDialog(options)` - 自定义对话框

**平台差异处理**：
- 对话框样式和行为存在差异，PAL提供统一接口
- 按钮顺序、键盘快捷键等跨平台兼容性由PAL处理

---

## 剪贴板操作

剪贴板是应用间数据交换的重要渠道。

**核心能力**：
- 读取：`read(format?)` - 支持纯文本、富文本、HTML、图像、文件列表等格式
- 写入：`write(data, format)` - 写入指定格式数据
- 监控：`monitor(callback)` - 监听剪贴板变化

**平台差异处理**：
- 不同平台支持的剪贴板格式存在差异，PAL处理最大兼容性

---

## 系统托盘

系统托盘是桌面应用的重要功能。

**核心能力**：
- 创建托盘：`create(options)` - 支持自定义图标、提示文本
- 托盘菜单：`setMenu(items)` - 设置右键菜单
- 托盘事件：`onClick(callback)` / `onRightClick(callback)` - 点击事件处理

**平台差异处理**：
- 不同操作系统的托盘实现存在较大差异，PAL提供统一行为

---

## 电源管理

电源管理接口允许应用获取系统电源状态并作出响应。

**核心能力**：
- 电源状态：`getPowerState()` - 获取电池电量、是否接通电源、是否处于省电模式等
- 电源事件：`onPowerStateChange(callback)` / `onSleep(callback)` / `onWake(callback)` - 电源状态变化事件

---

## 全局快捷键

全局快捷键允许应用在非激活状态下响应键盘快捷键。

**核心能力**：
- 注册：`register(accelerator, callback)` - 注册快捷键
- 注销：`unregister(accelerator)` - 注销快捷键
- 冲突检测：快捷键冲突时通知应用

**平台差异处理**：
- 不同平台的全局快捷键注册机制存在差异，PAL封装了这些差异

---

## 进程间通信

进程间通信是桌面应用的基础能力。

**核心能力**：
- 命名管道：适用于单台机器上的进程间通信
- 共享内存：提供高性能的数据共享方式
- Unix域套接字：在类Unix系统上提供高效的本地通信
- 发布订阅：基于共享的消息总线实现一对多或多对多的通信

## PAL接口标准

所有PAL接口必须遵循以下标准：

### 接口定义规范
```typescript
interface PALWindow {
  create(options: WindowOptions): Promise<WindowHandle>;
  getState(handle: WindowHandle): Promise<WindowState>;
  setState(handle: WindowHandle, state: Partial<WindowState>): Promise<void>;
  focus(handle: WindowHandle): Promise<void>;
  resize(handle: WindowHandle, width: number, height: number): Promise<void>;
  close(handle: WindowHandle): Promise<void>;
}
```

### 错误处理规范
```typescript
interface PALError {
  code: string;              // 错误码，如 PAL_WINDOW_NOT_FOUND
  message: string;           // 错误消息
  platform?: string;         // 平台特定错误信息
}
```

### 平台适配器接口
```typescript
interface PlatformAdapter {
  readonly platform: 'win32' | 'darwin' | 'linux';
  readonly version: string;
  
  // 每个平台能力对应一个方法
  window: PALWindow;
  fs: PALFileSystem;
  dialog: PALDialog;
  clipboard: PALClipboard;
  tray: power: PALPower;
  shortcut: PALShortcut;
  PALTray;
  ipc: PALIPC;
}
```

## 验收标准

### 分层验收
- L1职责边界清晰（平台能力抽象）
- L1只依赖OS/Electron原生能力
- L1不依赖业务插件

### 接口验收
- 所有平台能力通过PAL接口访问
- 内核代码不直接调用`process.platform`
- 平台差异收敛到PAL层

### 测试验收
- 每个适配器通过平台特定测试
- 跨平台一致性测试通过
