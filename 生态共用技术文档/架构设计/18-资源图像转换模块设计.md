# 资源图像转换模块设计

> 文档状态：首版正式稿
> 适用范围：Host `resource` 服务、PAL `image` 分面、`chips-sdk` 资源 API、生态内需要把 TIFF 资源归一为 PNG 文件的调用方
> 关联公共契约：`生态共用技术文档/协议与契约/11-资源图像转换契约.md`
> 关联内部实现参考：`Chips-Host/技术文档/20-TIFF转PNG资源转换服务设计.md`

## 1. 模块定位

资源图像转换模块是 Host `resource` 服务中的一项正式能力，用于把本地 TIFF 资源归一为 PNG 文件，供查看器、音乐播放器、资源管理、入库预处理等链路复用。

它的正式定位是：

- 属于 L3 Host Services 的 `resource` 服务域；
- 通过正式动作 `resource.convertTiffToPng` 对外暴露；
- 通过 `chips-sdk` 的 `client.resource.convertTiffToPng(...)` 提供类型化调用封装；
- 平台差异继续收敛到 L1 PAL `image` 分面；
- 输出结果是已经落盘完成的 PNG 文件，而不是内存态图片对象。

它不是：

- 一个新的 Host 服务域；
- 一个新的插件类型；
- 文件转换系统中的模块插件能力入口；
- `platform.*` 命名空间下的通用平台原语。

## 2. 为什么收口在 `resource`

### 2.1 语义归属

这个能力的输入主语是“资源”，输出主语是“归一化后的资源文件”，因此应归属 `resource` 服务，而不是 `platform`：

- 调用方传入的是 `resourceId`，而不是“平台图片句柄”；
- Host 需要理解资源路径、资源读取、目标文件落盘和错误归一；
- 该能力面向生态业务语义，而不是向上直接暴露底层系统 API。

### 2.2 与 `platform` 的边界

`platform` 服务当前收口平台原语、能力快照、屏幕、电源与离屏导出，不应继续堆放“资源归一化”类动作。
TIFF 转 PNG 的平台差异虽然存在，但它只属于实现细节，应该隐藏在 PAL 内部，由 `resource` 服务继续对外提供统一语义。

### 2.3 与文件转换系统的边界

生态中的“文件转换系统”长期设计是模块插件体系下的统一编排入口，主要面向 `card/html/pdf/image` 等跨格式导出流水线。
本能力并不新增转换模块，不走 `module.invoke(converter.file.convert)` 路径，而是作为 Host 内建资源能力，解决“本地 TIFF 资源归一化”为 PNG 文件的基础问题。

## 3. 正式公共口径

本节内容属于对外正式口径；若与仓内实现说明发生冲突，以共享契约与共享接口文档优先。

### 3.1 正式动作与 SDK 封装

- Host 动作：`resource.convertTiffToPng`
- SDK 封装：`client.resource.convertTiffToPng(request)`

### 3.2 请求结构

```ts
interface ResourceConvertTiffToPngRequest {
  resourceId: string;
  outputFile: string;
  overwrite?: boolean;
}
```

### 3.3 响应结构

```ts
interface ResourceConvertTiffToPngResult {
  outputFile: string;
  mimeType: "image/png";
  sourceMimeType: "image/tiff";
  width?: number;
  height?: number;
}
```

### 3.4 权限与路由治理

- 所需权限：`resource.read`、`file.read`、`file.write`
- Host 路由超时：15 秒
- 公开路由清单：`Chips-SDK/src/contracts/route-manifest.json`

### 3.5 正式错误码

- `RESOURCE_TIFF_UNSUPPORTED_URI`
- `RESOURCE_TIFF_INVALID_SOURCE`
- `RESOURCE_TIFF_SOURCE_NOT_FOUND`
- `RESOURCE_TIFF_INVALID_OUTPUT`
- `RESOURCE_TIFF_OUTPUT_EXISTS`
- `RESOURCE_TIFF_CONVERSION_UNSUPPORTED`
- `RESOURCE_TIFF_CONVERSION_FAILED`

## 4. 分层拆分

正式调用链如下：

```text
调用方 / 应用插件 / 模块运行时
  -> chips-sdk client.resource.convertTiffToPng(...)
    -> Host Kernel route: resource.convertTiffToPng
      -> Host resource service route descriptor
        -> ResourceImageService
          -> PAL image.convertTiffToPng(...)
            -> 当前 Host 运行时的实际平台转换能力
              -> 落盘 PNG 文件
```

各层职责固定如下：

| 层级 | 组件 | 责任 |
|---|---|---|
| L3 外部调用面 | `chips-sdk` `createResourceApi()` | 参数裁剪、基础类型校验、正式路由调用 |
| L2 Kernel 路由 | `register-host-services.ts` 中的 `resource.convertTiffToPng` 描述符 | 权限、超时、指标、动作注册 |
| L3 资源语义层 | `packages/resource-image-service` | `resourceId` 归一、TIFF 签名校验、错误归一 |
| L1 PAL | `PALImage.convertTiffToPng(...)` | 平台分支、命令执行、目录创建、输出 PNG 尺寸读取 |
| OS / Host Runtime | 当前桌面或无界面宿主能力 | 实际完成 TIFF 到 PNG 的格式转换 |

## 5. 注册与契约收口

当前正式链路在代码中的收口位置如下：

- Host 服务注册：`Chips-Host/src/main/services/register-host-services.ts`
- Host schema 注册：`Chips-Host/src/main/services/register-schemas.ts`
- Host 资源语义实现：`Chips-Host/packages/resource-image-service/src/resource-image-service.ts`
- PAL 接口声明：`Chips-Host/packages/pal/src/types.ts`
- PAL Node 适配实现：`Chips-Host/packages/pal/src/node-adapter.ts`
- SDK API：`Chips-SDK/src/api/resource.ts`
- SDK 路由清单：`Chips-SDK/src/contracts/route-manifest.json`

这条链路的正式要求是：

1. Host 路由必须显式注册动作、权限和超时；
2. Host schema 必须注册请求/响应校验器；
3. SDK 必须提供类型化封装；
4. SDK 路由清单必须包含公开动作；
5. 公共共享文档必须同步说明对外口径。

## 6. 输入校验与归一化

本节区分“正式要求”和“当前 Host 参考实现”。

### 6.1 正式要求

- `resourceId` 只允许本地文件路径或 `file://` 本地资源；
- `outputFile` 必须是 PNG 文件路径；
- 如果目标文件已存在且未允许覆盖，必须失败；
- 源文件必须是 TIFF 资源；
- 输出文件必须与源文件不同。

### 6.2 当前 Host 参考实现

当前 `ResourceImageService` 与 PAL Node 适配器的具体处理顺序如下：

1. `chips-sdk` 先裁剪 `resourceId` / `outputFile` 两端空白，并校验 `overwrite` 是否为布尔值；
2. Host schema 再次校验请求是否为对象，以及两个字段是否为非空字符串；
3. `ResourceImageService` 把 `resourceId` 归一为本地路径：
   - `file://` 通过 `fileURLToPath(...)` 转为本地路径；
   - 其他显式协议 URI 直接拒绝；
   - 纯路径会通过 `path.resolve(...)` 归一为绝对路径；
4. `ResourceImageService` 读取源文件字节并做 TIFF 头签名校验；
5. 通过后再委托 PAL `image.convertTiffToPng(...)`；
6. PAL 校验输出路径、目录状态、覆盖策略并执行真正转换；
7. PAL 读取落盘 PNG 的尺寸信息并返回给服务层；
8. 服务层归一返回 `{ outputFile, mimeType, sourceMimeType, width, height }`。

### 6.3 TIFF 头签名识别

当前 Host 服务层显式识别以下签名：

- Little-endian TIFF：`49 49 2A 00`
- Big-endian TIFF：`4D 4D 00 2A`
- Little-endian BigTIFF：`49 49 2B 00`
- Big-endian BigTIFF：`4D 4D 00 2B`

这意味着：

- 服务层至少能快速拒绝明显不是 TIFF 的输入；
- 更深的 TIFF 完整性、解码支持度和具体像素转换，仍由 PAL 底层转换能力决定；
- 某些文件即使签名合法，也可能在实际转换阶段失败，并以正式错误码返回。

## 7. 错误模型与归一规则

### 7.1 公共错误语义

| 错误码 | 语义 |
|---|---|
| `RESOURCE_TIFF_UNSUPPORTED_URI` | 输入不是本地路径或 `file://` 本地资源 |
| `RESOURCE_TIFF_INVALID_SOURCE` | 输入不是可接受的 TIFF 源 |
| `RESOURCE_TIFF_SOURCE_NOT_FOUND` | 源文件不存在 |
| `RESOURCE_TIFF_INVALID_OUTPUT` | 输出路径非法、不是 `.png`、与源文件相同，或指向目录 |
| `RESOURCE_TIFF_OUTPUT_EXISTS` | 输出文件已存在且未允许覆盖 |
| `RESOURCE_TIFF_CONVERSION_UNSUPPORTED` | 当前 Host 运行时没有 TIFF 转 PNG 能力 |
| `RESOURCE_TIFF_CONVERSION_FAILED` | 底层转换执行失败 |

### 7.2 当前 Host 参考映射

当前 `ResourceImageService` 对 PAL 错误的归一映射如下：

| PAL / 底层错误 | 对外错误 |
|---|---|
| `PAL_IMAGE_SOURCE_NOT_FOUND` | `RESOURCE_TIFF_SOURCE_NOT_FOUND` |
| `PAL_IMAGE_INVALID_OUTPUT` | `RESOURCE_TIFF_INVALID_OUTPUT` |
| `PAL_IMAGE_OUTPUT_EXISTS` | `RESOURCE_TIFF_OUTPUT_EXISTS` |
| `PAL_IMAGE_UNSUPPORTED` | `RESOURCE_TIFF_CONVERSION_UNSUPPORTED` |
| `PAL_COMMAND_NOT_FOUND` | `RESOURCE_TIFF_CONVERSION_FAILED` |
| `PAL_COMMAND_FAILED` | `RESOURCE_TIFF_CONVERSION_FAILED` |

未命中的异常当前保持原样上抛，避免把未知故障误归一为错误语义不准确的公共错误。

## 8. PAL 图像分面设计

### 8.1 接口位置

PAL `image` 分面当前公开以下接口：

```ts
interface PALImage {
  convertTiffToPng(input: ConvertTiffToPngRequest): Promise<ConvertTiffToPngResult>;
}
```

其中：

```ts
interface ConvertTiffToPngRequest {
  sourceFile: string;
  outputFile: string;
  overwrite?: boolean;
}

interface ConvertTiffToPngResult {
  outputFile: string;
  width?: number;
  height?: number;
  format: "png";
}
```

### 8.2 为什么这部分在 PAL

TIFF 转 PNG 的底层差异天然是平台相关问题：

- 各系统可用的转换命令不同；
- 是否存在图像库、命令行工具、系统组件并不一致；
- 输出文件落盘、目录创建与系统路径处理属于宿主能力。

因此：

- 服务层不应出现 `process.platform` 分支；
- 服务层不应直接执行系统命令；
- 平台差异必须被压回 PAL。

## 9. 当前 Host 参考实现策略

本节属于“当前 Host 参考实现”，用于帮助生态其他仓库理解真实运行链路；它不是调用方直接依赖的公共 API。

### 9.1 当前平台策略

`NodeImage.convertTiffToPng(...)` 当前按平台采用以下顺序：

- macOS：`sips -s format png <source> --out <output>`
- Windows：`powershell -NoProfile -Command ...`，通过 `System.Drawing.Image` 读取并保存 PNG
- Linux / 其他 Node 宿主：
  - 优先 `magick`
  - 其次 `convert`
  - 都不可用时返回不支持

### 9.2 输出路径治理

当前 PAL 实现额外保证：

- `outputFile` 后缀必须为 `.png`
- `sourceFile` 与 `outputFile` 不能是同一路径
- 目标目录不存在时自动创建
- 若目标文件已存在且 `overwrite === true`，会先删除旧文件再写入新文件

### 9.3 输出尺寸读取

当前 PAL 在转换完成后会读取输出 PNG 的 IHDR 头信息，得到：

- `width`
- `height`

因此公共返回值中的像素尺寸来自“实际落盘结果”，而不是来自调用方输入或猜测值。

## 10. 与音乐播放器内置 TIFF 处理的关系

音乐播放器当前也包含 TIFF 相关代码，但两者职责不同，不能互相替代。

### 10.1 音乐播放器链路

音乐播放器中的：

- `Chips-MusicPlayer/src/utils/artwork-runtime.ts`
- `Chips-MusicPlayer/src/utils/tiff.ts`

用于处理“封装在音频元数据里的嵌入式封面字节”。

它的特点是：

- 输入不是本地 TIFF 资源文件，而是音频标签里的二进制 artwork bytes；
- 输出主要是浏览器运行时可展示的对象 URL 或 PNG 字节；
- 优先目标是播放器界面立即显示封面；
- 当前实现是纯 TypeScript / 浏览器侧解码路径，不依赖 Host 系统命令。

### 10.2 Host 资源转换链路

本文档描述的 Host 模块则用于：

- 处理本地 TIFF 资源文件；
- 产出可复用、可持久化的 PNG 文件；
- 服务查看器、资源管理、导入预处理等多个生态场景；
- 由 Host 正式能力承载，而不是单应用私有运行时代码。

结论：

- 音乐播放器里的 TIFF 处理是“应用内嵌字节解码方案”；
- Host `resource.convertTiffToPng` 是“全生态复用的正式资源转换方案”；
- 两者可以共存，但公共资源归一化能力必须继续以 Host 模块为准。

## 11. 质量门禁与测试收口

当前与本模块直接相关的质量门禁如下：

- Host 路由、schema、PAL 接口、SDK API、共享文档必须同步收口；
- 平台差异必须停留在 PAL，不得回流到服务层或调用方；
- 新增图像转换能力时，不得在应用插件或模块插件中复制第二套宿主级 TIFF 转 PNG 逻辑；
- 若公共契约变化，必须同步更新 Host、SDK 与共享文档。

当前已覆盖的代表性测试包括：

- `Chips-Host/tests/unit/host-services-pal-routing.test.ts`
  - 验证 `resource.convertTiffToPng` 会通过资源服务链路委托到 PAL `image` 分面
- `Chips-Host/tests/integration/host-services.test.ts`
  - 在 macOS 上使用真实 TIFF 样本验证可以落盘 PNG 文件
- `Chips-Host/tests/contract/route-manifest.contract.test.ts`
  - 验证公开路由清单包含 `resource.convertTiffToPng`
- `Chips-Host/tests/unit/service-activation.test.ts`
  - 验证 Host PAL 依赖组装中包含 `image.convertTiffToPng`
- `Chips-SDK/tests/resource.test.ts`
  - 验证 SDK 会通过正式动作调用该能力，并在调用前完成基础参数校验

## 12. 扩展策略

未来如果继续扩展资源图像归一化能力，应遵守以下原则：

1. 继续保留在 `resource` 服务域，不新增第二套路由命名空间。
2. 继续复用 `resource-image-service` 与 PAL `image` 分面，不让平台差异泄漏到上层。
3. 继续区分“公共契约”和“当前参考实现”，避免把某个系统命令写死成对外语义。
4. 若扩展到更多格式映射，应优先设计为：
   - 输入仍然是资源语义；
   - 输出仍然是受控文件语义；
   - 错误模型保持可归一。
5. 不允许在查看器、音乐播放器、资源管理器等业务仓重新复制一套宿主级资源转换基础设施。

## 13. 结论

资源图像转换模块是一个“公共资源归一化能力”，而不是单应用私有逻辑。
它在架构上通过 `resource` 服务收口资源语义，通过 PAL `image` 分面收口平台差异，通过 SDK 暴露正式调用方式，已经形成一条完整的、可复用的正式链路。

后续生态内凡是遇到“拿到本地 TIFF 资源文件，但后续链路需要 PNG 文件”的场景，应优先复用本模块，而不是自行实现新的转换通道。
