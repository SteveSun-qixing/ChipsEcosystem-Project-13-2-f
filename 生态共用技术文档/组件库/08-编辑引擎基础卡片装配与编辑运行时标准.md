# 编辑引擎基础卡片装配与编辑运行时标准

## 1. 适用范围

本标准仅适用于官方 `Chips-EditingEngine` 的编辑态复合卡片预览与编辑链路。

本标准不替代：

- `CardCoverFrame`
- `CompositeCardWindow`
- `client.card.editorPanel.render(...)`

上述链路仍然是生态对外公开的通用 Host 托管方案；官方编辑引擎在编辑态使用本标准定义的本地运行时，是生态内唯一正式例外。

## 2. 正式定位

编辑引擎编辑态采用“双运行时、单数据源”架构：

- 显示链路：`BasecardDescriptor -> BasecardFrameHost -> 单卡 iframe -> CompositeCardAssembler -> CardWindow`
- 编辑链路：`BasecardDescriptor.renderEditor -> EditorSessionStore -> EditorHost -> EditPanel`
- 数据源：`CardService` 中的解包态 `.card/` 数据

该链路的设计目标是：

- 保留每个基础卡片独立 iframe 的失败隔离与交互边界；
- 去掉整张复合卡片最外层的统一 iframe；
- 让编辑引擎壳层直接掌握节点选中、布局汇总和编辑会话编排；
- 不在主流程里为任意单一卡片类型保留特殊分支。

## 3. 核心契约

### 3.1 BasecardDescriptor

编辑引擎本地运行时以 `BasecardDescriptor` 作为统一入口：

```ts
interface BasecardDescriptor {
  pluginId: string;
  cardType: string;
  displayName: string;
  aliases?: string[];
  commitDebounceMs?: number;
  createInitialConfig(baseCardId: string): Record<string, unknown>;
  normalizeConfig(input: Record<string, unknown>, baseCardId: string): Record<string, unknown>;
  validateConfig(config: Record<string, unknown>): { valid: boolean; errors: Record<string, string> };
  renderView(ctx: {
    container: HTMLElement;
    config: Record<string, unknown>;
    themeCssText?: string;
  }): (() => void) | void;
  renderEditor?(ctx: {
    container: HTMLElement;
    initialConfig: Record<string, unknown>;
    onChange(next: Record<string, unknown>): void;
  }): (() => void) | void;
}
```

约束：

- 描述符注册由编辑引擎在构建期静态完成；
- 主流程统一通过 `cardType` 或别名解析描述符；
- `RichTextCard -> base.richtext` 这类别名归一属于描述符层能力，不得散落在业务组件中；
- 当前首个正式接入类型为 `base.richtext`，但它只是注册中心中的一个描述符，不是主流程特判。

### 3.2 BasecardFrameHost

`BasecardFrameHost` 负责：

- 为单个基础卡片创建空白 `srcdoc` iframe；
- 在 iframe 内挂载 `renderView`；
- 从外层包装节点采集 `--chips-*` 主题变量，生成注入 CSS；
- 使用 `ResizeObserver + requestAnimationFrame` 测量单卡高度；
- 对外汇报 `loading/ready/error` 三态；
- 在 `preview` 场景下把指针选中回调给上层；
- 在 `delegate` 场景下输出归一化后的交互意图。

稳定性要求：

- 父层回调身份变化不得导致基础卡片 iframe 被反复重挂载；
- 真实重建条件只能来自 `cardType/baseCardId/normalizedConfig/themeCacheKey` 等有效渲染输入变化；
- 错误必须在单卡范围内隔离，不得拖垮整张复合卡片。

### 3.3 CompositeCardAssembler

`CompositeCardAssembler` 负责：

- 按 `CardService` 当前 `basicCards` 顺序装配单卡 iframe；
- 读取 `layout.padding/layout.gap` 组织纵向布局；
- 汇总每个节点的高度、加载态和错误态；
- 把聚合后的高度回传给 `CardWindow`；
- 维护当前选中节点与可选状态。

当前正式视觉约束：

- 装配层不再额外包裹边框、圆角、阴影；
- 每个基础卡片在编辑引擎中的边缘应保持无感，外层只负责结构编排，不负责附加视觉装饰。

### 3.4 EditorSessionStore

`EditorSessionStore` 负责：

- 以 `cardId + baseCardId` 生成稳定会话键；
- 保存 `sourceConfig/draftConfig/validation/dirty/isCommitting/errorMessage`；
- 在源配置被外部刷新且会话未脏时同步草稿；
- 以 `descriptor.commitDebounceMs` 控制提交节流；
- 在卸载、切换节点时 flush 最后一版合法草稿。

默认约束：

- 若描述符未声明 `commitDebounceMs`，默认提交节流为 `260ms`；
- 提交统一回到 `CardService.updateBasicCard(...)`，不得在编辑器组件中直接写文件。

### 3.5 EditorHost

`EditorHost` 负责：

- 根据 `cardType` 解析描述符；
- 确保当前会话存在并订阅其快照；
- 直接把 `renderEditor` 挂载到编辑面板容器；
- 接收编辑器 `onChange`，更新草稿；
- 按会话状态防抖提交；
- 在错误时只影响当前编辑面板，而不影响预览链路。

正式约束：

- 官方编辑引擎的编辑器主链路不再使用 iframe 宿主；
- `PluginHost` 在编辑引擎中仅作为 `EditorHost` 的薄包装，不再承担 SDK iframe 编辑器职责。

## 4. 数据与状态流

### 4.1 单一真实数据源

- `CardService` 中的 `CompositeCard.structure.basicCards[*].data` 是唯一正式数据源；
- `App.tsx`、`CardService.createCard/openCard/addBasicCard/updateBasicCard` 必须统一走描述符归一；
- 编辑器只维护草稿，不直接成为正式数据源。

### 4.2 标准流转

1. `CardService` 打开或创建卡片时，对 `cardType` 与 `data` 做统一规范化；
2. `CardWindow` 把当前 `basicCards` 交给 `CompositeCardAssembler`；
3. 用户在预览中点击节点，`BasecardFrameHost` 通过本地回调把 `baseCardId` 传给壳层；
4. `EditPanel -> PluginHost -> EditorHost` 打开当前基础卡片会话；
5. 编辑器修改草稿；
6. `EditorSessionStore` 调用 `CardService.updateBasicCard(...)` 提交合法配置；
7. `CardService` 执行 `notify()`、自动持久化，并发出 `card:basic-card-updated`；
8. `CardWindow` 因数据变更刷新对应基础卡片 iframe 与整体高度。

## 5. 主题与交互正式要求

### 5.1 主题

- 主题来源仍然是编辑引擎外层 `ChipsThemeProvider`；
- 单卡 iframe 通过提取当前文档的 `--chips-*` 变量生成注入 CSS；
- 基础卡片自身应使用 `--chips-*` token，而不是硬编码应用主题值。

### 5.2 交互

- `interactionPolicy = 'delegate'` 时，单卡 iframe 输出与 `CompositeInteractionPayload` 同形的滚轮交互载荷；
- 当前正式实现接管的是滚轮滚动与 `ctrl/meta + wheel` 缩放意图；
- `CardWindow` 负责把这些意图映射为无限画布 `panByInput/zoomByFactorAtPoint`。

## 6. 基础卡片插件侧要求

要进入官方编辑引擎链路，基础卡片插件必须满足以下条件：

- 在插件入口导出 `renderBasecardView` 与 `renderBasecardEditor`；
- 视图导出必须是“挂载到容器并返回 cleanup”的纯函数；
- 编辑导出必须允许直接挂载到普通 DOM 容器，而不是假定自己必然运行在 Host 编辑器 iframe 中；
- 需要的配置 schema、默认值、清洗逻辑应作为可复用模块暴露，供描述符层消费；
- 插件代码不得把 `cardType` 解析、会话管理、自动保存职责反推给编辑引擎。

## 7. 禁止事项

- 不允许在编辑引擎主流程里为任意卡片类型保留硬编码分支；
- 不允许同时保留“本地装配链路”和“复合 iframe 链路”作为编辑态双正式路径；
- 不允许引入通用占位编辑器作为正式兜底；
- 不允许在运行时扫描插件安装目录后动态 import 源码；
- 不允许绕开 `CardService` 直接写 `.card/content/*.yaml`。

## 8. 与通用 Host 托管链路的边界

- `CompositeCardWindow + editorPanel` 继续服务查看器、第三方集成和其他 Host 托管场景；
- 官方编辑引擎只在编辑态使用本地运行时；
- 两条链路共享基础卡片插件导出、`cardType` 匹配规则和主题 token 体系；
- 若未来其他应用希望复用编辑引擎本地运行时，必须先把该应用纳入正式生态决议，不能自行复制实现。
