# 生态概述

> **文档状态**：与 vNext 架构设计手册对齐
> **适用阶段**：阶段18及后续开发实施阶段

## 核心理念

薯片生态是一个运行在操作系统之上的软件级生态系统，核心理念是让复杂内容像打开图片一样简单。用户双击扩展名为card的文件就能自动打开查看，可以把卡片嵌入到网页或其他软件中分享给他人直接使用。这种设计理念贯穿整个生态的每一个层面，从底层架构到上层应用都遵循这一原则。

整个生态只有一个底层软件作为核心载体，这就是薯片主机。基于Electron技术栈构建的薯片主机是整个生态的唯一入口，对于操作系统来说始终只识别为一个软件。所有上层应用无论功能多么复杂多样，都是运行在薯片主机内部的插件。这种模型与微信小程序生态高度类似：微信是唯一的主程序对应薯片主机，小程序在微信内运行对应插件应用在薯片主机内运行，小程序通过微信API访问系统能力对应插件通过桥接API访问系统能力，小程序是Web技术实现对应插件应用也是Web技术实现。

## 全栈分层架构（12层）

薯片生态采用12层架构设计，明确实现级分层边界：

| 层级 | 名称 | 核心职责 | 只允许依赖 |
|---|---|---|---|
| L1 | PAL | 平台能力抽象 | OS/Electron 原生能力 |
| L2 | Host Kernel | router/registry/event/lifecycle | PAL |
| L3 | Host Services | file/resource/config/theme/i18n 等 | Kernel |
| L4 | Plugin Runtime | 插件加载、隔离、会话、权限 | Kernel+Services |
| L5 | Bridge Transport | `window.chips.*` IPC 传输 | Runtime |
| L6 | Runtime Client | 协议编解码、错误归一、重试 | Bridge Transport |
| L7 | UI Hooks | 面向页面的能力调用接口 | Runtime Client |
| L8 | Declarative UI | `View/Stack/Grid/Form/List` 语义原语 | UI Hooks + Component Layer |
| L9 | Unified Rendering | Normalize/Validate/Resolve/Commit | Declarative UI + Theme Runtime |
| L10 | Component Headless | 状态机、结构、a11y | Theme Runtime(接口) |
| L11 | Theme Runtime | token 解析、作用域覆盖、变量注入 | Theme Package |
| L12 | App/Page Domain | 页面状态、业务流程、编排 | L7-L11 |

**依赖方向白名单**：仅允许 `Ln -> Ln-1` 或 `Ln -> 公共契约层`，禁止跨层跳跃。

## 架构归属声明（2026-03-06）

- Host 是唯一运行时承载，L1-L9（含 L6 Runtime Client、L8 Declarative UI、L9 Unified Rendering）均为 Host 内置实现并随 Host 发布。
- 组件库主责 L10 Component Headless，只提供状态机、结构与可访问性，不承载渲染运行时。
- SDK 定位为开发者工具包，仅提供类型、安全封装、脚手架与测试辅助，不承载运行时主实现。
- 用户设备仅安装 Host 即可运行完整链路；SDK 仅在开发与集成阶段按需使用。

## 核心组件

薯片生态包含几个核心组件，它们共同构成了整个内容创作和管理体系的基础。

卡片是薯片生态的基本内容单元。一个卡片可以包含文字、图片、视频、音乐、代码、列表、打分组件等各种内容形态，也可以嵌套其他卡片形成更复杂的内容结构。卡片分为基础卡片和复合卡片两种类型。基础卡片是单一功能的原子组件，比如视频卡片、图片卡片、富文本卡片等，它们不能单独存在，必须作为复合卡片的组成部分。复合卡片是可以保存为独立文件格式的完整卡片容器，包含一个或多个基础卡片。当用户在编辑引擎中拖拽第一个基础卡片时，系统自动创建复合卡片容器来容纳它。复合卡片可以层层嵌套，形成无限复杂的内容结构。

箱子是卡片的容器，在文件系统中表现为单个`.box`文件。箱子内部保存元数据、卡片URL清单、卡片摘要、布局配置以及布局自身资源，但不直接保存被引用卡片实体。用户消费箱子时，由Host负责解析这些URL，并按布局插件的申请动态寻找和加载对应卡片信息，再把结果交给布局插件渲染页面。布局通常以封面、名称等摘要态展示卡片，同一个箱子可以切换不同的布局方式，引用数据保持不变，只是展示形式发生改变。这种设计让用户可以根据不同场景和需求灵活切换查看方式，而无需修改底层引用模型。

社区是网络上的公共平台，由社区服务器提供软件支持。社区服务器由存储服务器和社区平台两部分组成。存储服务器负责卡片和箱子的云端存储，社区平台提供用户、房间、空间等功能。用户可以在社区中创建房间，保存箱子和卡片，浏览其他用户的内容并与之互动。

## 页面级响应式与 cpx 尺寸体系

### 页面自治原则
- 每页面定义自己的断点、信息重排、交互降级策略
- 平台层不提供全局横竖模板

### cpx 逻辑尺寸单位
- 逻辑宽基准：`1024cpx`
- 换算公式：`px = cpx * (containerWidthPx / 1024)`
- SDK 提供 `@chips/layout-math` 统一函数，禁止应用侧重复实现

## 软件产品

薯片生态提供一系列软件产品来满足用户的不同需求。这些产品本质上都是运行在薯片主机中的插件应用。

卡片编辑引擎用于创建和编辑卡片内容，是生态中最重要的生产力工具。编辑引擎本身是插件化的，支持多种布局模式以适应不同的工作场景。无限画布布局提供无限可扩展的工作空间，适合创意设计和自由形式的内容组织。工作台布局采用分区窗口组合方式，适合专注于单个卡片的精细编辑。页面式布局类似传统笔记软件，适合线性内容创作。看板布局类似项目管理工具，适合任务和进度管理。网格布局以网格方式显示卡片封面，适合内容概览和浏览。用户可以根据工作场景切换不同布局，也可以在不同任务间选择最合适的布局模式。

卡片查看器用于查看卡片和箱子的内容，是消费卡片内容的主要入口。查看器通过调用渲染链路来展示各种类型的基础卡片内容。查看器还可以内置各种文件查看工具，这些工具的界面部分由应用或卡片插件承担，所需解析、索引、转换、预处理等无界面能力可以由模块插件提供，并按需要替换和扩展。

资源管理器是基于标签系统的文件浏览工具，帮助用户组织和查找卡片资源。不同于传统的树状文件夹结构，资源管理器采用标签来组织文件，提供更灵活的内容组织方式。

生态设置面板是整个生态的配置管理入口，提供四类核心功能：初始化安装处理新用户首次使用引导，运行治理负责插件管理和主题管理，全局设置管理各种偏好配置，协作桥接处理与其他软件的集成。

## 插件系统

薯片生态采用一切皆插件的设计哲学，这是整个生态架构的基础原则。基础卡片是插件，箱子布局是插件，编辑引擎的布局模式是插件，查看器中的各种工具是插件，甚至文件管理器和程序坞也是插件。这种设计确保了系统的极大灵活性和可扩展性。

插件支持热插拔机制，这是生态的重要特性。大多数插件可以不重启软件即可完成安装、更新和卸载操作。用户只需安装自己需要的插件，不需要的可以随时卸载。第三方开发者可以开发各种插件发布到应用市场，为生态贡献更多功能。

每个插件都可以形成独立的软件产品入口体验。待办能力可以做成独立入口的应用插件，视频能力可以做成独立的播放器应用插件。它们都由薯片主机加载运行，数据使用卡片文件格式保存，不依赖任何数据库系统。这种设计让功能扩展变得极为简单，无需为每个新功能开发独立的应用程序。

插件分为五种类型，涵盖生态的各个方面。应用插件是拥有独立窗口的完整应用，如编辑引擎、查看器、设置面板、资源管理器等。卡片插件是基础卡片类型的渲染和编辑组件，定义每种卡片类型的外观和行为。布局插件是箱子的布局算法和展示方式，控制卡片如何排列和呈现。模块插件是安装到 Host 中的无界面能力模块，用于向 Host 与其他插件提供可复用能力。主题插件是CSS样式主题包，定义整个生态的视觉外观。

## 底层架构

薯片主机是整个生态唯一的底层程序，基于Electron技术栈构建，长期在用户设备后台运行。所有上层软件都是运行在主机内部的Web插件应用。这种架构确保了资源的有效利用，多个插件共享同一个Chromium引擎和主进程服务，系统资源占用远少于独立运行多个应用程序。

薯片内核是整个生态的神经中枢，使用TypeScript实现，运行在Electron主进程中。内核提供中心路由系统、服务注册、事件总线、生命周期管理等核心能力。所有服务间的通信都必须经过内核路由，服务之间不能直接通信，这是最核心的架构原则。插件通过Bridge API（window.chips.*）访问系统能力，Bridge API将请求转发到主进程的内核路由器。内核还负责管理所有插件的生命周期，统一执行生态标准。

公共基础层位于薯片内核之上、插件桥接层之下，是整个生态的基础设施。它承载生态中所有插件和应用共用的通用能力，体现底层厚、上层薄的设计理念。公共基础层分为两部分：核心服务运行在主进程中，包括文件服务、卡片服务、箱子服务、资源管理、ZIP处理、配置管理、主题管理、多语言、日志、标签数据库、凭证管理、数据序列化、平台服务、模块加载等。共享功能模块作为模块插件独立安装，承担视频处理、图片处理、Markdown 渲染、内容转换、索引和规范化等无界面能力。这些模块各自独立仓库，可自由替换和升级。

前后端完全分离是生态的重要架构特征。前端只负责界面呈现，后端只负责业务逻辑，它们通过Bridge API和内核路由通信，从不直接接触。同一个后端功能可以有多个不同的前端实现，不同开发者可以开发不同风格的界面。前端和后端通过标准接口对接，后端完全不知道前端如何实现。

## 层次结构

薯片生态的内容层次从小到大依次分为六个级别，形成完整的内容组织体系。

基础卡片是最小的内容单位，代表不可再分的原子内容形态。每个基础卡片都有明确的类型，如文本、图片、视频、音乐、代码等。

复合卡片是可保存的最小单位，对应标准的card文件格式。一个复合卡片包含一个或多个基础卡片，是用户创建和保存内容的基本容器。

箱子是卡片的容器，在文件系统中表现为单个`.box`文件。箱子通过布局插件定义其展示方式，Host在消费时负责解析箱子记录的URL并按需向布局插件提供卡片信息和资源，支持多种布局算法的切换。

房间是用户在空间内的组织单位，可以进行多层嵌套。用户可以在房间中放置箱子和卡片，形成层级化的内容结构。

空间对应物理存储设备，如电脑、手机、硬盘、网盘等。系统自动识别各个存储设备，用户可以在不同空间间管理内容。

社区是网络上的公共平台，提供内容分享和社交功能。用户可以加入社区，浏览和下载其他用户分享的内容。

软件层次从底层到顶层分为六层。Electron宿主提供Chromium引擎和Node.js运行时，是整个生态的技术底座。平台抽象层适配不同操作系统的差异，向上提供统一的能力接口。薯片内核提供中心路由、服务注册、事件总线、生命周期管理等核心能力。基础服务层加共享功能模块提供各种通用服务。插件桥接层通过contextBridge向插件暴露标准接口。插件应用是运行在独立BrowserWindow中的Web应用，包括各种类型的功能插件。

这种双层层次结构确保了薯片生态既能有效组织内容，又能灵活构建应用，是整个生态设计的核心框架。

## 北极星目标与架构原则

### 北极星目标
- 构建可持续 5-10 年演进的插件生态前端基础设施
- 以单一标准支撑 `app/card/layout/module/theme` 五类插件
- 用户侧体验无感：功能、路径、交互语义稳定
- 开发者侧复杂度降低：契约统一、工具统一、模板统一

### 不可违反原则
1. **中心路由原则**：服务间调用必须走内核路由，禁止跨服务直接 import 调用
2. **Bridge 隔离原则**：插件仅通过 `window.chips.*` 与系统能力交互
3. **底层厚、上层薄**：能力尽量在底层固化，上层仅做业务编排
4. **平台无关性**：平台差异收敛到 PAL，核心不直连平台 API
5. **无头组件原则**：组件只负责结构和行为，视觉完全主题注入
6. **契约优先原则**：接口、事件、manifest、token、布局均契约化
7. **鲁棒优先原则**：默认内建隔离、回退、熔断、可观测

### React 架构技能基线
- `vercel-composition-patterns`：组件 API 采用组合模式、显式变体、上下文分层，避免 boolean props 爆炸
- `vercel-react-best-practices`：并发请求并行化、渲染/订阅优化、性能预算化、重渲染治理
