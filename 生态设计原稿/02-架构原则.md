# 架构原则

> **文档状态**：与 vNext 架构设计手册对齐
> **适用阶段**：阶段18及后续重构实施阶段


## 架构归属声明（2026-03-06）

- Host 主责 L1-L9 运行时链路（含 Runtime Client、Declarative UI、Unified Rendering）并随 Host 发布。
- 组件库主责 L10 Component Headless，只负责状态机、结构与 a11y。
- SDK 仅作为开发者工具包提供类型、封装、脚手架与测试辅助，不承载运行时主实现。
- 用户设备仅安装 Host 即可运行；SDK 仅在开发与集成阶段按需使用。

## 北极星目标

- 构建可持续 5-10 年演进的插件生态前端基础设施
- 以单一标准支撑 `app/card/layout/module/theme` 五类插件
- 用户侧体验无感：功能、路径、交互语义稳定
- 开发者侧复杂度降低：契约统一、工具统一、模板统一

## 不可违反原则

### 1. 中心路由原则

薯片生态采用中心路由架构，所有服务和插件的通信都必须经过薯片内核作为中转站。模块之间不能直接调用或通信，这是整个架构最核心的设计原则。

在主进程中，服务通过内核路由器的invoke方法调用其他服务。在渲染进程中，插件通过Bridge API发送请求，Bridge通过IPC将请求转发到主进程的内核路由器。内核接收请求后验证格式和权限，然后路由到目标服务。目标服务处理完成后将结果返回给内核，内核再转发给发起请求的插件。整个过程中，模块和插件从未直接接触。

内核不仅路由生态内部的通信，还负责生态内外的所有对接。从外部世界进入生态的所有内容，包括操作系统调用、文件系统访问、网络请求、用户输入等，都通过内核作为统一入口。内核提供文件识别、资源解析、用户输入捕获等能力，将外部数据转换成生态内的标准格式，然后路由到相应的模块。

内核维护一个完整的路由表，记录哪种请求应该路由到哪个模块。这个路由表可以动态调整，当用户安装了新的插件或者设置了新的默认处理模块时，内核会更新路由规则。所有的通信都使用标准化的请求和响应格式，内核负责格式的验证和转换。

这种架构让模块完全解耦。每个模块都是独立的孤岛，只知道如何与内核交互，不知道其他模块的存在。模块可以随时被替换成另一个实现相同接口的模块，而不影响系统的其他部分。内核还负责权限管理和安全控制，确保模块只能访问被允许的功能。所有的通信都会被记录到日志中，便于监控和调试。

### 2. Bridge 隔离原则

插件只能通过Bridge API访问系统能力，这是生态的另一个核心原则。Bridge API通过Electron的contextBridge机制暴露给插件，是插件访问底层能力的唯一通道。

插件不能直接访问Node.js API，不能直接读写文件系统，不能直接发起网络请求。所有系统级操作都由主进程的基础服务代为执行。插件只能通过调用window.chips.invoke方法发起请求，通过window.chips.on方法订阅事件。

Bridge API定义了一套完整的接口标准，涵盖文件操作、卡片管理、窗口控制、主题获取、模块加载等各个方面。每个接口都有明确的参数格式和返回值规范，确保插件与系统交互的一致性。

这种隔离设计带来了多方面的优势。首先提高了安全性，插件被限制在沙箱环境中运行，无法直接访问系统底层。其次简化了开发，插件开发者只需要学习一套标准接口，不需要了解底层实现细节。第三便于维护，底层实现可以自由修改而不影响上层插件，只要Bridge接口保持稳定。

### 3. 底层厚上层薄原则

底层尽可能包含所有能力，上层应用只是调用底层的薄壳，这是生态的重要架构原则。

公共基础层承载生态中所有插件和应用共用的通用能力。这些能力包括文件处理、媒体播放、文本渲染、格式转换等几乎所有应用都需要的基础功能。每个功能都由专门的模块负责实现，这些模块经过精心设计和优化，提供稳定可靠的公共服务。

上层应用专注于业务逻辑和界面呈现。具体来说，编辑引擎只负责用户交互和数据组织，具体的文件打包解包由文件服务处理。查看器只负责内容展示，具体的媒体解码播放由播放器模块负责。卡片插件只负责卡片数据的组织呈现，底层的渲染框架由系统提供。

这种设计避免了重复开发。每个基础功能只需要实现一次，所有应用都能复用。这降低了开发成本，提高了代码质量，也使得系统更加稳定。底层模块可以独立演进和优化，上层应用无需任何修改。

### 4. 平台无关性原则

内核和服务层代码不直接调用平台特定API，一律通过PAL进行适配，这是确保生态跨平台能力的关键原则。

平台抽象层位于Electron宿主之上、薯片内核之下。它的职责是向下适配不同操作系统，向上提供统一接口。每类系统能力都有一个抽象接口和多个平台适配器。内核和基础服务不直接调用process.platform或任何平台特定API，而是通过PAL接口请求能力。

PAL覆盖Windows、macOS、Linux的最大公约数能力，同时为平台独有功能保留扩展接口。当某个平台特有的能力被调用时，PAL会根据当前运行环境选择合适的适配器执行。这种设计确保了内核层和服务层代码的平台无关性，为未来移动端扩展打下基础。

开发者开发插件和应用时，也应该遵循这一原则。所有平台特定的能力都通过PAL获取，不直接使用平台API。这确保了应用在不同平台上都能正常运行。

### 5. 无头组件原则

组件只有逻辑和结构，零样式，视觉全部由主题包注入，这是生态视觉一致性的保障。

生态提供一套无头组件库，包含按钮、输入框、下拉菜单、滑块、弹窗等基础控件。组件只实现功能逻辑和HTML结构，完全不包含任何视觉样式。

组件通过CSS类名和CSS变量暴露样式接口点。主题包通过这些接口点注入具体的视觉定义，包括颜色、字体、圆角、阴影、动画等。所有样式都由主题包控制，同一套组件在不同主题下呈现完全不同的视觉效果。

这种设计实现了视觉与逻辑的彻底分离。组件开发者只需要关注功能实现，不需要关心视觉设计。主题开发者可以自由创作各种风格，而不需要了解组件的内部逻辑。两者独立演进，通过标准接口连接。

### 6. 契约优先原则

接口、事件、manifest、token、布局均契约化，所有能力必须通过明确的契约定义。

- **接口契约**：所有服务动作必须定义输入 schema、输出 schema、权限、超时、错误码
- **事件契约**：统一事件元信息 `id/name/source/data/timestamp/metadata`
- **manifest 契约**：插件必须通过 manifest schema 校验
- **token 契约**：主题必须通过 token 完整性校验
- **布局契约**：页面必须声明布局 contract

### 7. 鲁棒优先原则

默认内建隔离、回退、熔断、可观测。

- **失败隔离**：组件级 Error Boundary，单节点失败不拖垮整页渲染
- **回退策略**：主题失败回退默认主题，布局契约非法回退安全模板
- **熔断恢复**：同路由连续失败触发软熔断，半开状态探测成功后自动恢复
- **可观测**：统一日志字段（traceId/requestId/pluginId/namespace/action/durationMs/result/errorCode）

## 一切皆插件原则

每个功能都是独立的插件，统一使用cpk打包格式，这是生态可扩展性的基础。

插件是生态的基本构建单元。编辑引擎是插件，查看器是插件，资源管理器是插件，文件管理器是插件。每个基础卡片类型是插件，每种箱子布局是插件，甚至主题也是插件。系统中几乎所有功能都是插件，这种设计带来了极大的灵活性。

每个插件都是独立的代码仓库，有自己的版本管理和发布流程。插件之间通过标准接口通信，互不依赖。这使得插件可以独立开发、独立测试、独立部署，大幅提高了开发效率。

插件使用统一的cpk打包格式，本质是ZIP压缩包。cpk文件包含插件的所有代码、资源和配置文件。安装时，系统解压cpk文件，解析manifest清单，注册插件能力。卸载时，系统清理插件相关文件，不留残留。

热插拔是插件系统的重要特性。插件可以在不重启软件的情况下完成安装、更新和卸载。用户安装新插件后，系统动态加载插件代码，立即可用。更新插件时，系统先卸载旧版本再加载新版本。卸载插件时，系统移除插件所有文件和注册信息。

## 错误模型标准化

### 统一错误对象
```ts
interface StandardError {
  code: string;
  message: string;
  details?: unknown;
  retryable?: boolean;
}
```

### 错误归一映射
- `bridge` 层错误统一映射到 `BRIDGE_*`
- `service` 层错误统一映射到 `SERVICE_*`
- `runtime` 层超时/重试错误统一映射到 `RUNTIME_*`
- `ui` 层消费时仅接收标准对象，不直接暴露底层堆栈

## 插件独立成软件产品原则

每个应用插件都可以形成独立应用入口体验，数据使用卡片文件保存，不依赖数据库，这是生态应用形态的多样性保障。

应用插件是拥有独立窗口的完整Web应用。编辑引擎是一个应用插件，查看器是一个应用插件，设置面板是一个应用插件。每个应用插件在操作系统中创建独立的快捷方式，用户使用体验与普通软件一致。

应用插件的数据使用卡片文件格式保存，不依赖任何数据库。所有的内容都存储在card文件或box文件中，这些文件是标准的ZIP压缩包，可以被任何工具读取。这种设计避免了数据库依赖，使得应用更加轻量，也便于数据迁移和备份。

同一个功能可以有多个不同的应用插件实现。例如待办功能，可以有简洁风格的待办界面、看板风格的待办界面、日历风格的待办界面。这些不同的实现共享相同的后端服务，只是前端界面不同。用户可以根据自己的喜好选择使用哪个应用插件。

这种架构让生态具有极大的灵活性。开发者可以只开发前端不开发后端，复用已有的后端服务。也可以只开发后端不开发前端，使用已有的前端框架。还可以只开发主题包，改变现有应用的视觉风格。三者完全独立，可以自由组合。
