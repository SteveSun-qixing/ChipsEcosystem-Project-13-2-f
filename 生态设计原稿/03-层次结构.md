# 层次结构

> **文档状态**：与 vNext 架构设计手册对齐
> **适用阶段**：阶段18及后续重构实施阶段

## 软件层次（12层架构）

薯片生态的软件层次从底层到顶层分为 **12 层**，每层都有明确的职责边界和接口定义：

| 层级 | 名称 | 核心职责 | 只允许依赖 | 禁止依赖 |
|---|---|---|---|---|
| L1 | PAL | 平台能力抽象 | OS/Electron 原生能力 | 业务插件 |
| L2 | Host Kernel | router/registry/event/lifecycle | PAL | 应用页面代码 |
| L3 | Host Services | file/resource/config/theme/i18n 等 | Kernel | 插件 UI |
| L4 | Plugin Runtime | 插件加载、隔离、会话、权限 | Kernel+Services | 业务域实现细节 |
| L5 | Bridge Transport | `window.chips.*` IPC 传输 | Runtime | 页面业务状态 |
| L6 | Runtime Client | 协议编解码、错误归一、重试 | Bridge Transport | 组件渲染实现 |
| L7 | UI Hooks | 面向页面的能力调用接口 | Runtime Client | Transport 细节 |
| L8 | Declarative UI | `View/Stack/Grid/Form/List` 语义原语 | UI Hooks + Component Layer | Host 服务细节 |
| L9 | Unified Rendering | Normalize/Validate/Resolve/Commit | Declarative UI + Theme Runtime | Host 路由逻辑 |
| L10 | Component Headless | 状态机、结构、a11y | Theme Runtime(接口) | 视觉硬编码 |
| L11 | Theme Runtime | token 解析、作用域覆盖、变量注入 | Theme Package | 业务逻辑 |
| L12 | App/Page Domain | 页面状态、业务流程、编排 | L7-L11 | L1-L5 细节 |

### 依赖方向白名单
- 仅允许 `Ln -> Ln-1` 或 `Ln -> 公共契约层`
- 禁止跨层跳跃（例如 L12 直调 L5）

### 反模式黑名单
- 应用仓自建 bridge-client
- 组件层输出硬编码颜色/阴影/圆角
- 同一能力存在双动作名并长期并行

## 内容层次

薯片生态从最小的原子单位到全球社区，共有六个内容层次，形成完整的内容组织体系。

基础卡片是内容层次的最小单位，代表不可再分的原子内容形态。基础卡片有明确的类型区分，每种类型对应一种特定的内容形式。文本类型的基础卡片承载纯文本内容，支持富文本格式。图片类型的基础卡片承载图像文件，支持常见的图片格式。视频类型的基础卡片承载视频文件，支持各种编码格式。音频类型的基础卡片承载声音文件。代码类型的基础卡片承载程序代码，支持语法高亮。列表类型的基础卡片承载有序或无序的内容列表。评分类型的基础卡片承载评价打分信息。基础卡片的类型可以无限扩展，开发者可以开发更多类型的基础卡片插件来支持新的内容形式。

复合卡片是内容层次的第二层，是可保存的最小单位。复合卡片对应标准的card文件格式，是用户创建和保存内容的基本容器。一个复合卡片可以包含一个或多个基础卡片，这些基础卡片按照特定的顺序排列。复合卡片中的基础卡片可以是不同类型的，视频卡片、图片卡片、文本卡片可以共存于同一个复合卡片中。复合卡片支持嵌套结构，一个复合卡片可以作为基础卡片被包含在另一个复合卡片中。这种嵌套可以是无限层级的，用户可以创建极其复杂的内容结构。

箱子是内容层次的第三层，是卡片的容器。箱子在文件系统中表现为文件夹结构，包含多个卡片文件和一个配置文件。箱子的核心特性是布局系统，布局插件定义了箱子内部卡片的展示方式。同一个箱子可以切换不同的布局插件，布局插件决定了卡片的排列规则、排序方式、交互模式等。布局类型是无限可添加的，系统自带的布局插件包括网格布局、瀑布流布局、列表布局、看板布局等。第三方开发者可以开发更多类型的布局插件，用户从应用市场安装后即可使用。

房间是内容层次的第四层，是用户在空间内的组织单位。房间提供更高层次的内容组织能力，一个用户可以在自己的空间里创建多个房间，不同房间之间互不干扰。房间支持嵌套结构，用户可以建立层级关系来组织大量内容。例如用户可以创建“工作”房间作为顶层，在“工作”房间下创建“项目A”和“项目B”两个子房间。每个房间里可以放置箱子和卡片，形成清晰的内容层级结构。房间的嵌套深度没有限制，用户可以根据需要创建任意层级的房间结构。

空间是内容层次的第五层，对应物理存储设备。用户的每个设备都是一个独立的空间，包括电脑、手机、平板、硬盘、网络存储等。空间的容量由设备本身决定，系统自动识别各个存储设备，不需要用户手动设置。用户可以有多个空间，不同空间之间可以同步内容，实现跨设备的内容访问。空间的概念让用户不需要关心底层的存储管理，只需要在不同设备上使用同一个账号登录，即可访问自己在各个空间中的内容。

社区是内容层次的第六层，是网络上的公共平台。社区由社区服务器提供软件支持，包括存储服务器和社区平台两部分。存储服务器负责卡片和箱子的云端存储，社区平台提供用户管理、房间空间、内容分享等社交功能。用户可以在社区中创建公开房间，将本地的箱子同步上传到社区，获得分享链接。其他用户访问链接可以在线查看内容，喜欢的内容可以下载到自己的本地空间继续编辑和创作。社区让个人创作可以传播和被发现，形成活跃的内容生态。

## 层次之间的关联

内容层次和软件层次之间存在紧密的对应关系。内容层次定义了用户创建和组织的数字资产，软件层次定义了如何操作和管理这些资产。

当用户打开一个card文件时，软件层次自顶向下启动工作流程。用户双击触发操作系统的文件关联机制，薯片主机接收文件路径参数。内核识别文件类型后查找注册了对应处理能力的应用插件，通常是查看器插件。查看器插件创建BrowserWindow，加载自己的前端代码，通过Bridge API请求卡片服务读取card文件内容。卡片服务解析文件结构，提取各个基础卡片的信息。查看器根据这些信息调用各个卡片插件的渲染组件，在iframe中展示卡片内容。

编辑流程遵循类似的工作模式。用户启动编辑引擎，打开或创建卡片文件。编辑引擎通过Bridge API请求卡片服务获取当前卡片内容。用户在编辑界面中修改内容，编辑引擎通过Bridge API将更改保存到卡片服务。卡片服务负责更新card文件中的相关配置。整个过程中，各层各司其职，通过标准接口协作完成任务。

数据在层次之间流转有明确的路径。从用户输入到持久化存储，数据经过前端界面层、桥接层、内核路由层、服务层，最后到达文件系统。从文件系统到用户界面，数据经历相反的路径。每个层次只与相邻层次交互，数据格式在层间转换，确保信息的一致性和完整性。
