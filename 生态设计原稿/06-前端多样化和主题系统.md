# 前端多样化和主题系统

## 架构归属声明（2026-03-06）

- Host 主责 L1-L9 运行时链路（含 Runtime Client、Declarative UI、Unified Rendering）并随 Host 发布。
- 组件库主责 L10 Component Headless，只负责状态机、结构与 a11y。
- SDK 仅作为开发者工具包提供类型、封装、脚手架与测试辅助，不承载运行时主实现。
- 用户设备仅安装 Host 即可运行；SDK 仅在开发与集成阶段按需使用。

## 前端的多样化

薯片生态允许同一后端能力对应多个前端实现。应用层可以使用不同的技术栈、不同的交互方式和不同的视觉方案，但必须通过统一协议接入 Host，不能绕开生态底层自行建立一套平行运行时。

前端多样化的边界如下：

- 业务调用统一走 Host 对外接口与 Bridge API；
- 卡片显示统一走 Host 统一卡片显示链路；
- 组件视觉统一由主题系统提供，不允许在应用层和组件层硬编码一套独立皮肤体系；
- 主题、插件、窗口、配置、日志都必须落在当前工作区内闭环管理。

## 主题系统的作用

主题系统负责生态视觉表达。组件库只提供结构、状态与交互，不承载视觉样式。真正生效的颜色、字体、圆角、阴影、边框、动效与图标由主题包提供。

一个主题包只承载一种完整外观，不在包内区分 light/dark 或日夜模式。暗色外观必须作为独立 `themeId` 的正式主题包交付。

## 主题配置的层级

主题配置采用六级覆盖机制。主题权重从低到高为：全局 -> 应用 -> 箱子 -> 复合卡片 -> 基础卡片 -> 组件。解析时始终以高优先级覆盖低优先级，但最终运行结果必须由 Host 的 `theme.resolve` 统一产出，不允许应用自己实现另一套解析器。

基础卡片配置中的 `theme` 字段记录技术 ID（`themeId`），例如 `chips-official.default-theme`。系统内部始终按技术 ID 装载、解析和校验主题。

## 主题运行时链路

主题从安装到生效的正式链路如下：

1. 主题包以 `type: theme` 插件形式安装到当前工作区；
2. 只有已启用的主题插件才能进入 Host 主题服务的可用主题集合；
3. `theme.list/getCurrent/getAllCss/resolve/apply` 全部基于当前工作区和已启用主题插件执行；
4. Host preload 将当前主题 CSS、解析后的变量以及 `data-chips-theme-id`、`data-chips-theme-version` 注入文档根节点；
5. 应用壳层通过 `ChipsThemeProvider` 或等价运行时从 Host 当前主题初始化，而不是写死默认主题；
6. 复合卡片窗口在主题运行时缓存键变化后重新渲染 iframe 内容；
7. 原生窗口背景在应用未显式指定时，由 Host 从当前主题 token 推导；
8. 卡片渲染服务只负责注入主题 CSS 与变量，不得强行写死浅色模式或覆盖主题包自己的 `color-scheme`。

## 工作区边界

薯片生态同时存在两类正式工作区：

- `chips`：用户工作区，默认位于 `~/.chips-host`；
- `chipsdev`：开发工作区，默认位于项目链路中的 `.chips-host-dev`。

两类工作区的插件记录、主题配置、当前主题和运行状态彼此隔离。开发态看到的主题结果，只能以开发工作区为准；用户态看到的主题结果，只能以用户工作区为准。

## 主题接口（固定动作）

主题系统固定提供以下动作：

- `theme.list`：列出当前工作区中已启用的主题
- `theme.apply`：应用指定主题
- `theme.getCurrent`：获取当前主题
- `theme.getAllCss`：获取当前主题的完整 CSS 与当前 `themeId`
- `theme.resolve`：解析主题作用域链并返回解析结果与 token 视图
- `theme.contract.get`：获取当前主题契约接口点

## 前端组件和主题的关系

前端组件库只暴露结构和样式接口点。主题包通过 CSS 类名、数据属性和 CSS 变量完成样式落地。应用层可以组合主题，但不能破坏主题包与组件库之间的职责边界。

复合卡片、基础卡片、应用窗口和原生窗口外观必须共享同一主题链路，避免出现应用壳层、卡片内容区、系统窗口三套视觉状态不一致的问题。

## 标准接口的作用

统一接口的价值不只是让“能切换主题”，而是让主题切换在整个生态中保持可验证、可追踪、可维护：

- 主题插件启停边界明确；
- 工作区边界明确；
- 应用启动初始主题明确；
- 复合卡片内容刷新时机明确；
- 原生窗口背景与内容窗口的视觉来源明确。
