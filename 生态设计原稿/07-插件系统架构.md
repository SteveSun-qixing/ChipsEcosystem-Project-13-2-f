# 插件系统架构

## 一切皆插件的设计理念

薯片生态采用一切皆插件的设计哲学，这是整个架构的核心原则。系统中几乎所有功能都被实现为插件，从编辑器到查看器，从卡片类型到布局方式，从功能模块到视觉主题，全部以插件形式存在和运行。这种设计带来了极大的灵活性和可扩展性。

插件是生态的基本构建单元。编辑引擎本身是一个应用插件，提供创建和编辑卡片的功能。查看器是另一个应用插件，提供查看卡片和箱子的功能。资源管理器是文件浏览工具，设置面板是配置管理入口。每个基础卡片类型是独立的卡片插件，例如视频卡片插件负责视频内容的展示和编辑。箱子布局是布局插件，网格布局、瀑布流布局各是一个独立插件。各种媒体播放器是模块插件，视频播放器、音频播放器、图片查看器都可以作为独立插件。界面的视觉样式是主题插件，不同主题包提供不同外观。

每个插件都是独立的代码仓库，有自己的版本管理和发布流程。插件之间通过标准接口通信，互不依赖。这种架构使得插件可以独立开发、独立测试、独立部署，大幅提高了开发效率。开发者可以只关注自己负责的功能模块，不需要了解整个系统的所有细节。

## 五种插件类型

插件分为五种类型，每种类型有明确的职责和接口定义。

应用插件是拥有独立窗口的完整Web应用，运行在独立的BrowserWindow中。应用插件提供完整的用户界面和功能，是用户直接交互的对象。编辑引擎、查看器、资源管理器、设置面板、文件管理器等都是应用插件。每个应用插件在操作系统中可以创建独立的快捷方式，用户使用体验与普通软件一致。应用插件的数据使用卡片文件格式保存，不依赖数据库。应用插件之间可以相互调用，通过内核路由进行通信。

卡片插件是基础卡片的渲染和编辑组件。每个基础卡片类型对应一个卡片插件，定义该类型卡片如何显示和编辑。卡片插件分为渲染组件和编辑组件两部分。渲染组件负责在查看器中显示卡片内容，编辑组件负责在编辑引擎中修改卡片内容。渲染组件加载后在iframe中运行，与主页面隔离。编辑组件提供用户界面让用户设置卡片参数。卡片插件需要声明支持的卡片类型标识，系统根据卡片配置中的类型字段加载对应插件。

布局插件是箱子的展示方式定义。布局插件决定箱子内部卡片的排列规则、排序方式、交互模式等视觉呈现。布局插件包括两部分：编辑时的布局编辑器提供在编辑引擎中调整布局的用户界面，查看时的布局渲染器提供在查看器中展示布局的渲染逻辑。网格布局插件、瀑布流布局插件、列表布局插件、看板布局插件都是独立的布局插件。布局插件可以动态切换，切换时只改变展示方式，不影响底层数据。

模块插件是共享功能模块，提供可供其他插件复用的功能。模块插件不创建独立窗口，由其他插件按需加载使用。视频播放器模块提供视频解码和播放能力，音频播放器模块提供音频播放和波形显示，图片查看模块提供图片显示缩放旋转等功能，三维模型渲染器模块提供3D模型的显示和交互，富文本编辑器模块提供富文本编辑能力，Markdown渲染器模块将Markdown转换为HTML，代码高亮器模块为各种编程语言提供语法高亮。模块插件作为共享库被多个卡片插件或应用插件复用，避免重复实现。

主题插件是视觉样式包，定义界面的颜色、字体、圆角、阴影、动画等视觉属性。主题插件使用CSS和CSS变量实现，通过接口点与组件库连接。主题插件包含tokens定义颜色和字体，components定义各组件的样式，motions定义动画效果，icons定义图标资源。同一套组件在不同主题下呈现完全不同的视觉效果。主题插件是可选的，用户可以安装多个主题并在设置中切换。

**主题包标准结构**：
```
theme-pkg/
├── manifest.yaml
├── tokens/
│   ├── ref.json          # 原始值
│   ├── sys.json          # 语义值
│   ├── comp.json         # 组件值
│   ├── motion.json       # 动效值
│   └── layout.json       # 布局密度值
├── components/*.css      # 组件样式
├── motions/*.css         # 动画样式
├── icons/*               # 图标资源
├── contracts/theme-interface.contract.json  # 主题契约
└── preview/*             # 预览资源
```

**主题契约门禁**：每个主题包必须通过：
- token 完整性校验（缺失/冗余）
- 组件接口点校验（`data-scope/data-part`）
- 动效参数安全校验（最大时长、曲线白名单）
- 颜色对比度校验（文本可读性）

## cpk打包格式

插件使用统一的cpk格式打包，本质是一个ZIP压缩包。cpk文件扩展名标识了插件包类型，系统通过文件关联自动识别。

cpk压缩包内部有标准的目录结构。根目录包含manifest.yaml清单文件，这是必需的配置文件，描述插件的元数据、能力、依赖等。插件代码通常放在dist或build目录中，压缩打包时代码目录会被包含进去。静态资源如图片、字体等直接放在根目录或资源目录中。不同类型插件的目录结构略有差异，但manifest.yaml是统一的入口。

manifest.yaml记录插件的所有元数据信息。插件标识是唯一的名称，用于系统识别。版本号遵循语义化版本规范。插件类型标识这个插件属于五种插件类型中的哪一种。入口文件指定插件加载的入口点。权限声明列出插件需要的系统能力。依赖声明列出插件依赖的其他插件或模块。

cpk打包使用零压缩率即存储模式，这与卡片文件类似。零压缩可以快速解压安装，对性能要求较高的场景尤为重要。打包工具将插件所有文件打包成ZIP，然后重命名为cpk扩展名。

## 热插拔机制

热插拔是插件系统的重要特性，允许插件在不重启软件的情况下完成安装、更新和卸载操作。

安装插件时，系统接收用户的安装请求，下载或读取cpk文件。系统解压cpk文件到插件目录，解析manifest清单获取插件元数据。系统验证插件的完整性和权限声明，检查依赖是否满足。验证通过后，系统将插件注册到插件注册表，插件立即可用。安装过程不需要重启软件，用户可以继续使用其他功能。

更新插件时，系统先检查新版本与现有版本的兼容性。系统卸载旧版本插件，清理旧版本的文件和注册信息。系统安装新版本插件，重复安装插件的完整流程。更新过程可以无缝完成，用户无需退出正在使用的功能。

卸载插件时，系统从插件注册表中移除该插件。系统清理插件的所有文件和相关配置。如果有其他插件依赖被卸载的插件，系统会通知相关插件进行适配。

**热插拔机制提高了系统的可用性**。用户可以随时添加新功能，不需要中断当前工作。系统可以持续运行，不需要频繁重启。插件开发者可以快速迭代开发，即时测试新版本。

## 插件运行时安全流水线

插件运行时必须执行完整的安全流水线，确保系统安全：

1. **Manifest 校验**：校验类型、权限、入口、契约版本
2. **签名与来源校验**：官方/第三方渠道验证
3. **启动握手**：`plugin-init` 握手，分配 `sessionNonce`
4. **权限快照加载**：最小权限原则
5. **沙箱注入**：只暴露 `window.chips.*`
6. **运行时审计**：越权、超频、异常事件监控
7. **会话结束清理**：订阅、缓存、临时资源释放

## 插件独立仓库策略

每个插件都是独立的代码仓库，采用独立的版本管理。这种策略带来多方面优势。

独立仓库允许团队或开发者专注于特定功能。一个团队可以只负责视频卡片插件的开发，不需要了解整个编辑引擎的实现。另一个团队可以负责布局插件的开发。这种分工方式提高了开发效率，每个团队可以按照自己的节奏迭代。

独立仓库便于开源和社区贡献。第三方开发者可以独立开发插件，发布到应用市场。社区开发者可以 fork 官方插件进行定制，或者开发全新的插件。独立仓库降低了贡献门槛，促进了生态的繁荣。

独立仓库支持灵活的发布策略。官方插件可以使用稳定的发布周期，确保质量。第三方插件可以快速迭代实验性功能。用户可以根据需要选择不同发布策略的插件。

插件之间通过标准接口交互，版本兼容性由系统保证。插件声明自己依赖的其他插件版本范围，系统在安装时检查依赖是否满足。这种机制让独立仓库可以独立演进，同时保持系统的整体稳定性。
