# 卡片渲染机制


## 架构归属声明（2026-03-06）

- Host 主责 L1-L9 运行时链路（含 Runtime Client、Declarative UI、Unified Rendering）并随 Host 发布。
- 组件库主责 L10 Component Headless，只负责状态机、结构与 a11y。
- SDK 仅作为开发者工具包提供类型、封装、脚手架与测试辅助，不承载运行时主实现。
- 用户设备仅安装 Host 即可运行；SDK 仅在开发与集成阶段按需使用。

## 渲染流程概述

卡片渲染是生态中将卡片数据转换为可视化界面的核心机制。整个渲染流程涉及多个层次的协作，从文件系统到界面展示，经历了数据读取、结构解析、组件调用、iframe渲染等多个阶段。

渲染流程的起点是用户打开卡片文件。用户双击card文件或通过查看器打开卡片时，操作系统将文件路径传递给薯片主机。内核识别文件类型后，查找注册的文件处理服务。卡片服务接收请求，开始解析card文件的结构。卡片服务读取metadata.yaml获取卡片的基本信息，读取structure.yaml获取卡片包含的基础卡片列表，读取content目录下各个基础卡片的配置文件获取具体参数。

渲染流程根据卡片类型的不同分为基础卡片渲染和复合卡片渲染两种模式。基础卡片渲染针对单一类型的内容组件，复合卡片渲染针对包含多个基础卡片的复合结构。两种模式共享相同的底层机制，但在组件组装和布局方面有所区别。

## 基础卡片渲染

基础卡片渲染是整个渲染流程的基本单元。每个基础卡片都在独立的iframe中渲染，与其他卡片和主页面隔离。

渲染器首先根据卡片类型查找对应的卡片插件。系统根据structure.yaml中的类型字段，确定需要加载哪些卡片插件。每个基础卡片类型对应一个卡片插件，插件提供该类型的渲染组件。

加载渲染组件后，渲染器读取该基础卡片的配置文件。配置文件中包含该卡片的类型标识、参数设置、资源路径等信息。渲染器将配置数据传递给渲染组件的初始化接口。

渲染组件接收到配置数据后，生成对应的HTML内容。渲染组件使用自己定义的HTML结构和JavaScript逻辑，根据配置数据填充内容。如果卡片包含视频，渲染组件生成video标签和播放控制界面。如果卡片包含图片，渲染组件生成img标签和查看器功能。

生成的HTML被加载到iframe中运行。iframe提供了独立的浏览器上下文，与主页面隔离。这种隔离确保了安全性和稳定性：一个卡片渲染出错不会影响其他卡片和主程序。每个iframe运行在独立的域中，通过postMessage与主页面通信。

渲染完成后，用户可以在iframe中与卡片内容交互。点击视频卡片开始播放，滚动查看长文本内容，缩放查看图片细节。所有的交互都在iframe内部发生，不影响其他组件。

## 复合卡片渲染

复合卡片渲染处理包含多个基础卡片的复杂结构。复合卡片本质是一个容器，将多个基础卡片的iframe组合在一起展示。

复合卡片的渲染从读取structure.yaml开始。structure.yaml中的structure数组按顺序列出所有基础卡片。每个数组项包含卡片的ID和类型。渲染器按照这个顺序依次渲染各个基础卡片。

复合卡片的容器负责布局和协调。容器是一个主iframe或div容器，内部包含多个基础卡片的iframe。容器负责计算每个卡片的位置和尺寸，确定它们的排列方式。容器还负责处理整体的事件，如滚动、缩放等。

基础卡片的iframe按照顺序垂直排列。每个卡片的宽度与复合卡片宽度相同，高度由内容自适应决定。卡片之间有内边距和分隔线，形成清晰的视觉边界。如果某个基础卡片是折叠状态，容器只渲染其标题区域。

复合卡片还支持嵌套结构。一个复合卡片可以作为基础卡片被包含在另一个复合卡片中。这种嵌套可以是任意深度，形成树状结构。渲染时，系统递归处理嵌套结构，每个层级都在独立的iframe中运行。

## 主题应用

主题系统负责卡片的视觉样式。渲染过程中，主题信息被注入到各个组件中。

主题的配置信息存储在卡片的metadata.yaml中。themeId字段指定使用哪个主题包，例如chips-official.default-theme。系统根据这个ID加载对应的主题插件。

主题CSS通过多种方式应用。对于主页面和容器，主题CSS直接注入到页面头部。对于基础卡片的iframe，主题CSS通过iframe的src属性或postMessage传递。渲染组件读取主题变量，应用到自己的样式中。

主题的CSS使用CSS变量定义颜色、字体、间距等视觉属性。渲染组件使用这些CSS变量构建自己的样式。例如primary-color变量定义主色调，border-radius变量定义圆角大小。切换主题时，只需要更新CSS变量的值，所有使用这些变量的组件都会自动更新。

主题还包含组件级别的样式覆盖。每个基础卡片类型有对应的样式定义。例如视频播放器有特定的控制条样式，图片查看器有特定的缩放控件样式。这些样式通过data-scope和data-part选择器应用到具体组件。

## 渲染优化

渲染系统采用多种优化策略，确保流畅的用户体验。

懒加载是重要的优化手段。当复合卡片包含多个基础卡片时，系统优先渲染可见区域的卡片。远离视口的卡片延迟渲染，等到滚动到附近时再加载。这减少了初始渲染时间，降低了内存占用。

缓存机制避免重复渲染。相同配置的卡片第二次打开时，系统使用缓存的渲染结果。缓存键由卡片类型和配置数据的哈希值组成。当配置变化时，缓存失效，重新渲染。

渲染失败隔离确保系统的稳定性。如果某个基础卡片渲染出错，错误被捕获并隔离在该卡片的iframe内部。出错卡片显示错误提示，其他卡片正常显示。系统记录错误日志，便于后续调试和修复。

资源预加载提升后续操作的响应速度。当检测到用户可能即将打开某个卡片时，系统提前开始加载该卡片的配置和资源。例如用户在卡片列表中悬停在某个卡片上时，后台开始预加载该卡片的内容。

## 统一显示组件补充

渲染体系对外建议通过两个统一组件交付：

- **卡片封面组件（CardCoverFrame）**：展示封面 HTML 的 iframe 窗口，窗口下方显示卡片名称。
- **复合卡片窗口组件（CompositeCardWindow）**：展示复合卡片内容的 iframe 窗口。

业务应用（查看器、编辑器、未来应用）统一通过 SDK 显示接口调用这两个组件，传入卡片文件即可完成显示。渲染分发与拼接由 Host 内置渲染运行时处理，SDK 仅提供开发调用封装。

## 渲染器的职责

渲染器是协调整个渲染流程的核心组件。渲染器负责解析卡片结构、加载插件、创建iframe、处理事件、管理生命周期。

解析卡片结构时，渲染器读取card文件的所有配置信息，验证结构完整性，构建卡片的对象模型。加载插件时，渲染器根据类型查找对应的卡片插件，动态加载渲染组件代码。创建iframe时，渲染器设置iframe的属性和样式，注入必要的安全策略。处理事件时，渲染器接收iframe发来的消息，转发给相应的处理函数。管理生命周期时，渲染器跟踪每个卡片的加载状态，处理资源释放和内存清理。

渲染器作为公共服务运行在主进程中。它通过内核路由接收渲染请求，与卡片服务、插件服务、主题服务等协作完成渲染任务。渲染器暴露统一的API，查看器、编辑器等应用通过这些API请求渲染功能。
