# 公共基础层设计

## 架构定位

公共基础层位于薯片内核之上、插件桥接层之下，是整个生态的基础设施。它承载生态中所有插件和应用共用的通用能力，体现“底层厚、上层薄”的设计理念。

公共基础层分为两部分。A部分是核心服务，运行在主进程中，负责数据处理和系统功能。这些服务包括文件服务、卡片服务、箱子服务、资源管理、ZIP处理、配置管理、主题管理、多语言、日志、标签数据库、凭证管理、数据序列化、平台服务、模块加载等。这些服务编译在薯片主机中，通过内核路由器对外提供能力。B部分是共享功能模块，作为模块插件独立安装，提供无界面能力。它们包括视频处理、音频处理、图片处理、富文本能力、Markdown 渲染、代码高亮、内容转换、索引和规范化等。这些模块各自独立仓库，可自由替换和升级，由 Host 统一治理并向上层插件提供能力。

薯片内核只提供最核心的路由、注册、事件总线、生命周期管理等能力。公共基础层承载所有通用功能，让生态内的插件和软件保持轻量化，避免重复造轮子，提高代码模块复用率。

## 层级关系

从底层到顶层采用十二层实现口径：PAL → Host Kernel → Host Services → Plugin Runtime → Bridge Transport → Runtime Client → UI Hooks → Declarative UI → Unified Rendering → Component Headless → Theme Runtime → App/Page Domain。Electron宿主提供Chromium引擎和Node.js运行时，平台抽象层适配不同操作系统差异，插件应用运行在独立 BrowserWindow 中。

架构归属声明（2026-03-06）：Host 主责 L1-L9（含 Runtime Client、声明式 UI、统一渲染），组件库主责 L10，SDK 仅作为开发者工具包提供封装与类型，不承载运行时主实现。

## 核心服务（16个服务域）

核心服务运行在主进程中，提供系统级的功能支持。每个服务必须遵循服务动作统一约束。

### 服务域清单

| 服务域 | 核心动作面 | 输入/输出基线 | 幂等性 | 权限边界 | 典型错误码 |
|---|---|---|---|---|---|
| `file` | read/write/stat/list | path + options -> content/meta | `read/stat/list` 幂等 | `file.read/file.write` | `FILE_NOT_FOUND`, `FILE_PERMISSION_DENIED` |
| `resource` | resolve/readMetadata/readBinary | resourceId -> uri/meta/blob | `resolve/read*` 幂等 | `resource.read` | `RESOURCE_NOT_FOUND` |
| `config` | get/set/batchSet/reset | key/value -> ack/snapshot | `get` 幂等 | `config.read/config.write` | `CONFIG_KEY_INVALID` |
| `theme` | list/apply/getCurrent/getAllCss/resolve/contract.get | themeId/chain -> themeState/css | `list/get*/resolve` 幂等 | `theme.read/theme.write` | `THEME_NOT_FOUND`, `THEME_CONTRACT_INVALID` |
| `i18n` | getCurrent/setCurrent/translate/listLocales | locale/key -> locale/text | `get*/translate` 幂等 | `i18n.read/i18n.write` | `I18N_KEY_MISSING` |
| `window` | open/focus/resize/setState | windowConfig -> windowState | `focus` 幂等 | `window.control` | `WINDOW_NOT_FOUND` |
| `plugin` | install/enable/disable/uninstall/query | pluginManifest -> status | `query` 幂等 | `plugin.manage` | `PLUGIN_INVALID`, `PLUGIN_PERMISSION_DENIED` |
| `module` | listProviders/resolve/invoke/job.get/job.cancel | capability + method -> result/job | `listProviders/resolve/job.get` 幂等 | `module.read/module.invoke` | `MODULE_PROVIDER_NOT_FOUND` |
| `platform` | getInfo/getCapabilities/openExternal | none/url -> info/ack | `get*` 幂等 | `platform.read/platform.external` | `PLATFORM_UNSUPPORTED` |
| `log` | write/query/export | record/filter -> ack/bundle | `query` 幂等 | `log.write/log.read` | `LOG_EXPORT_FAILED` |
| `credential` | get/set/delete/rotate | secretRef -> secret/meta | `get` 幂等 | `credential.manage` | `CREDENTIAL_DENIED`, `CREDENTIAL_EXPIRED` |
| `card` | parse/render/validate | cardDoc -> ast/view/model | `parse/validate` 幂等 | `card.read/card.write` | `CARD_SCHEMA_INVALID` |
| `box` | pack/unpack/inspect | inputPath -> artifact/meta | `inspect` 幂等 | `box.pack` | `BOX_FORMAT_INVALID` |
| `zip` | compress/extract/list | files -> zip/meta | `list` 幂等 | `zip.manage` | `ZIP_CORRUPTED` |
| `serializer` | encode/decode/validate | payload + schema -> bytes/object | `decode/validate` 幂等 | `serializer.use` | `SERIALIZE_FAILED` |
| `control-plane` | health/check/metrics/diagnose | scope -> report | 幂等 | `control.read/control.write` | `CONTROL_TIMEOUT` |

### 服务动作统一约束
- 每个动作必须定义：输入 schema、输出 schema、权限、超时、错误码
- 幂等动作标注 `idempotent=true`
- 非幂等动作必须定义防重放策略（requestId 去重窗口）

### 统一错误对象
```ts
{ code: string; message: string; details?: unknown; retryable?: boolean }
```

### 服务实现统一模板
- 每服务必须导出：`types.ts`（输入输出类型）、`constants.ts`（动作名/错误码）、`service.ts`（注册与实现）
- 每动作必须存在同名契约测试：`<action>.contract.test.ts`
- 服务 `initialize()` 失败必须阻断系统进入 `running`

## 共享功能模块

共享功能模块作为模块插件独立安装，提供无界面能力。

通用媒体能力模块是基础层的重要部分。视频元数据提取与转码模块负责视频信息读取、快照和格式转换。音频分析模块负责波形、时长和元信息抽取。图片预处理模块负责缩放、旋转、压缩和格式转换。三维资源解析模块负责模型元数据提取和可消费中间结果生成。这些模块各自是独立的代码仓库和插件包。

文本处理模块包含多个独立模块。富文本规范化模块负责富文本结构整理。Markdown处理模块负责 Markdown 解析与目标格式输出。代码高亮模块负责语法分析和高亮结果生成。文本分析模块负责分词、抽取、校验和基础处理能力。

这些共享功能模块不创建自己的窗口，由上层插件在自己的渲染进程中按需加载使用。加载方式有两种：作为iframe子组件加载，或作为ES Module动态导入。

## 无头组件库

界面组件库基于Zag.js和Ark UI构建，提供通用的界面元素。按钮、输入框、下拉菜单、滑块、弹窗等基础控件，这些控件只实现功能逻辑、结构和布局，完全不包含任何视觉样式。所有样式由主题包通过CSS变量和类名注入。组件暴露CSS类名接口点，主题包通过这些接口点定义具体的视觉表现。

## 模块组织原则

公共基础层的模块必须高度解耦。每个核心服务是独立的功能单元，通过内核路由器对外提供服务。服务之间不能直接依赖，必须通过薯片内核的路由系统通信。共享功能模块则各自是独立的代码仓库和插件包，可以被独立替换和升级。

接口设计需要保持高度一致的风格。所有服务使用统一的接口规范，参数传递方式统一，返回值格式统一，错误处理方式统一。上层插件通过Bridge API以一致的方式调用不同服务。

模块的职责单一明确。视频转码模块只负责转码，不包含查看器界面。图片预处理模块只负责图片处理，不包含图片查看界面。每个模块专注自己的核心功能。

模块支持按需加载。不是所有服务在启动时都加载，而是当某个功能需要时才加载对应的服务或模块插件。这减少了启动时间和内存占用。

## 协议设计

公共基础层定义了一套标准协议，规定上层插件如何调用基础层的能力。

模块调用协议规定了调用格式。上层插件通过Bridge API发送请求到主进程，主进程的内核路由器将请求路由到对应的服务处理器。所有调用都使用统一的请求响应格式。

资源访问协议规定了如何访问各种资源。本地文件、网络资源、嵌入资源等都通过统一的资源标识符访问，资源管理服务自动处理资源的获取和缓存。

事件通知协议规定了模块如何发出和接收事件。服务通过内核事件总线发布事件，插件通过Bridge API的事件订阅接口接收事件通知，实现松耦合的消息传递。

配置管理协议规定了配置的读写方式。所有配置都通过配置管理服务访问，支持多层级配置覆盖，配置变化时自动通知相关模块和插件。

## 与上层的关系

功能插件基于公共基础层开发。插件不需要自己实现视频处理、图片处理、文本规范化、内容转换等基础功能，而是通过正式模块服务调用对应模块插件即可。插件只需要实现业务逻辑和界面编排部分。

软件开发工具包基于Bridge API构建。第三方开发者通过SDK可以用类型安全的方式访问基础层的所有能力，开发自己的插件。SDK是Bridge API的高层封装，提供更友好的开发接口和Vue 3 Composables。

## 版本管理

核心服务作为薯片主机的一部分，随主机一起更新，版本号与主机统一。

共享功能模块作为独立的模块插件，各自拥有独立的版本号，遵循语义化版本规范。上层插件在manifest.yaml中声明依赖的模块版本范围，安装时系统检查版本兼容性。

## 轻量化原则

公共基础层保持轻量化，只包含必需的功能。不包含任何业务逻辑，不包含具体的卡片实现，不包含具体的布局实现。这些都由上层插件提供。

不包含身份认证、用户管理、权限系统等功能。这些是社区服务器的职责，本地软件不需要。不包含数据同步、云端备份等功能，这些由具体的应用层实现。

只保留所有插件和软件都需要的通用能力。视频播放、图片显示、文本渲染等是通用需求，作为共享模块插件提供。具体的视频卡片如何组织数据、如何呈现界面，由视频卡片插件负责。

模块的实现也保持简洁。视频处理使用成熟的媒体库，不自己从零实现。图片处理使用现有的图像库。代码高亮使用现有的高亮库。基础层是这些开源工具的集成和封装，提供统一接口。

## 开源优先原则

公共基础层的开发遵循开源优先原则。在设计每个功能模块时，必须优先评估是否存在成熟的开源方案。凡是有开源方案的，采用提取和拆分开源项目模块的方式集成，仅对生态高度定制的部分进行全新设计。

视频播放、音频播放、PDF解析、EPUB阅读、Markdown渲染、代码高亮、图片处理等功能都有成熟的开源实现，直接集成开源模块。ZIP处理、文件识别、格式转换等也有开源库可用。只有卡片文件格式解析、薯片协议路由、主题引擎等生态特有的功能需要从零开发。

界面组件库采用Zag.js作为组件状态机引擎，Ark UI作为Vue 3框架绑定层。主题构建使用Style Dictionary管理设计Token。所有核心依赖均使用MIT或同等宽松的开源许可证。

这种方式大幅减少开发工作量，提高代码质量和稳定性，加快开发进度。开源模块经过社区验证，功能完善，性能优异。基础层主要工作是选择合适的开源库，提取所需功能，封装为统一接口，集成到生态中。
