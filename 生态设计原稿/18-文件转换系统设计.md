# 文件转换系统设计

## 1. 定位

文件转换系统是生态中的通用工具能力，不新增插件类型，也不新增独立 Host 服务域。它通过模块插件体系承载，并继续复用 Host 已有的：

- `module.listProviders / module.resolve / module.invoke / module.job.*`
- `card.render`
- `card.pack / card.unpack / card.readMetadata`

对调用方而言，文件转换系统表现为一个统一的文件转换入口；对实现层而言，它由一个编排模块和多个原子转换模块组成。

## 2. 正式结构

### 2.1 编排模块

`Chips-FileConversion-Plugin` 是对外统一入口，承担四个职责：

- 对应用提供统一公共入口；
- 维护可用转换 provider 的正式能力映射；
- 根据源类型和目标类型生成转换流水线；
- 汇总任务状态、错误和结果。

它本身只是普通模块插件，不是新的“路由器插件类型”。

### 2.2 原子转换模块

原子转换模块只负责单一格式映射：

- `card -> html`
- `html -> pdf`
- `html -> image`

其中：

- `Chips-CardtoHTML-Plugin` 负责从卡片生成目录态或压缩态 HTML；
- `Chips-HtmltoPDF-Plugin` 负责消费目录态 HTML 并生成 PDF；
- `Chips-HtmltoImage-Plugin` 负责消费目录态 HTML 并生成图片。

### 2.3 依赖关系

正式依赖图如下：

```text
调用方
  -> Chips-FileConversion-Plugin
       -> Chips-CardtoHTML-Plugin
       -> Chips-HtmltoPDF-Plugin
       -> Chips-HtmltoImage-Plugin
```

`pdf` 和 `image` 导出都必须先经过 `card -> html`，不允许各自复制一套卡片渲染实现。

## 3. 公共入口

对外统一入口仍然是 Host 模块服务。调用方通过能力契约调用编排模块，例如：

```ts
await client.module.invoke({
  capability: "converter.file.convert",
  method: "convert",
  input: {
    source: {
      type: "card",
      path: "/workspace/demo.card"
    },
    target: {
      type: "pdf"
    },
    output: {
      path: "/workspace/output/demo.pdf"
    }
  }
});
```

Host 继续负责统一的 provider 选择、schema 校验、job 生命周期和错误归一；文件转换系统本身不再额外定义第二套路由命名空间。

## 4. 流水线规则

### 4.1 支持矩阵

当前首版冻结以下组合：

- `card -> html`
- `card -> pdf`
- `card -> image`
- `html -> pdf`
- `html -> image`

### 4.2 组合规则

- `card -> html`：直接调用 `Chips-CardtoHTML-Plugin`
- `card -> pdf`：先 `card -> html(directory)`，再 `html -> pdf`
- `card -> image`：先 `card -> html(directory)`，再 `html -> image`
- `html -> pdf`：直接调用 `Chips-HtmltoPDF-Plugin`
- `html -> image`：直接调用 `Chips-HtmltoImage-Plugin`

### 4.3 中间产物规则

`card -> html` 必须同时支持两种输出模式：

- `directory`：目录态 HTML 中间产物，供后续 PDF / 图片模块消费；
- `zip`：最终交付给用户的 HTML 压缩包。

编排模块在 `pdf` / `image` 流水线中只能使用 `directory` 模式；直接导出 HTML 时才生成 `zip`。
中间 HTML 一律必须自包含资源并携带转换 manifest，不允许把“非正式中间产物”继续传给 PDF / 图片模块。

## 5. Card -> HTML 正式设计

### 5.1 渲染来源

卡片转 HTML 不允许复制基础卡片私有前端代码，也不允许在转换模块里实现另一套卡片显示器。正式来源是 Host 卡片渲染链路：

- 模块调用 `card.render(..., { target: "offscreen-render" })`
- Host 复用正式基础卡片插件导出和主题链路，生成复合卡片 HTML
- 转换模块在此基础上做离线化重写、资源收集、打包和清单生成

### 5.2 视觉一致性

HTML 导出结果必须与卡片渲染模块画面一致。这里的一致性至少包含：

- 基础卡片顺序一致；
- 主题样式一致；
- 基础卡片内 iframe 文档一致；
- 复合卡片导出时，基础卡片 iframe 文档必须外置为同目录 sibling HTML，并由 `index.html` 通过相对 `src` 引用；
- 资源路径可离线访问；
- 退化节点、错误态、封面与元数据口径一致。

### 5.3 交付结构

目录态 HTML 交付物结构如下：

```text
<output>/
├─ index.html
├─ <node-id>.html               # 复合卡片中每个基础卡片 iframe 的离线文档
├─ assets/
│  └─ content/                  # includeAssets=true 时存在
└─ conversion-manifest.json     # includeManifest=true 时存在
```

压缩态 HTML 交付物是上述目录结构的 ZIP 包。

`assets/theme/` 仅作为未来独立主题工件的保留位置。当前首版若 Host 渲染链路已把主题样式内联到 HTML，则不要求强制产出该目录。

## 6. HTML -> PDF / Image 正式设计

### 6.1 输入边界

PDF 和图片模块只接受目录态 HTML 中间产物，正式输入至少包括：

- HTML 根目录；
- 入口文件路径（通常为 `index.html`）；
- 与页面截图 / 打印相关的输出参数。

### 6.2 实现方式

这两个模块都不直接持有 Electron 页面实例，也不在模块内自行创建浏览器运行时。正式路径是由 Host 提供统一的 HTML 渲染导出能力，模块负责组装导出请求并消费导出结果。

Host 在受控 Electron 渲染环境内加载目录态 HTML：

- `html -> pdf` 使用 Electron 原生 PDF 输出能力；
- `html -> image` 使用 Electron 页面截图能力。

它们不再解析卡片，也不直接依赖基础卡片插件。

### 6.3 Host 渲染导出收口

`html -> pdf` 和 `html -> image` 必须共用一套 Host 侧 HTML 渲染容器与导出治理能力，至少统一以下内容：

- HTML 入口加载；
- 页面尺寸与缩放倍率设置；
- 字体、图片和资源稳定等待；
- 复合卡片 iframe 文档加载完成等待；
- 基础卡片高度上报与 iframe 高度同步完成等待；
- 复合卡片 `ready/resize` 稳定信号等待；
- 导出前页面状态确认；
- 取消、超时与资源释放；
- 警告和错误归一。

这样可以避免：

- 在模块内重复复制一套页面渲染实现；
- `pdf` 与 `image` 出现两套不一致的渲染口径；
- 通过 `puppeteer` 等独立运行时绕过 Host 正式能力边界。

## 7. 任务治理

转换流程默认采用 job 模式执行，正式要求如下：

- 编排模块负责汇总总进度，并把原子模块的阶段进度投影为统一进度事件；
- job 取消时，必须同时中止当前原子转换步骤并清理中间目录；
- 错误返回统一结构，且要区分：
  - 输入无效；
  - provider 不存在；
  - 渲染失败；
  - 资源打包失败；
  - PDF 输出失败；
  - 图片截图失败。

## 8. 批量转换

批量转换仍由编排模块统一承载：

- 多任务队列；
- 参数模板；
- 单文件结果与批次结果汇总；
- 中途取消与清理；
- 输出命名冲突处理。

批量只是执行模式扩展，不新增新的模块类型。
