**版本**：1.2.0  
**日期**：2026-03-02

## 核心理念

薯片生态是一个运行在操作系统之上的软件级生态系统。核心理念是让复杂内容像打开图片一样简单。用户双击 `.card` 文件就能自动打开查看，可以把卡片嵌入到网页或其他软件中分享给他人直接使用。

整个生态实行类似于微内核、微服务的架构设计，厚底层、薄上层。所有功能都拆分成一个个原子化的模块插件，实现最大程度的灵活性，同时减少重复代码开发。生态内所有逻辑都在内部进行，通过一套自己的薯片协议（`chips://`）进行通信和管理。

## 薯片主机

### 定位与架构

薯片主机（Chips Host）是整个生态唯一的底层程序，基于 Electron 技术栈构建，长期在用户设备后台运行。对于操作系统来说，始终只识别薯片主机这一个软件。所有上层应用无论功能多么复杂多样，都是运行在薯片主机内部的插件。

这种模型与微信小程序生态高度类似：
- 微信 → 薯片主机（唯一主程序）
- 小程序 → 插件应用（在主机内运行）
- 微信API → 桥接API（访问系统能力）
- 小程序用Web技术 → 插件应用也是Web技术

### 层次结构

薯片主机内部采用多层次架构，从底层到顶层依次为：

**第一层：PAL（平台抽象层）**  
负责适配不同操作系统（Windows、macOS、Linux）的差异，向上提供统一的平台能力接口。平台差异收敛到 PAL 层，核心代码不直连平台 API，确保跨平台能力。

**第二层：Host 内核**  
使用 TypeScript 实现，运行在 Electron 主进程中。提供中心路由、服务注册、事件总线、生命周期管理等核心能力。所有服务间的通信都必须经过内核路由，服务之间不能直接通信，这是最核心的架构原则。

**第三层：Host 服务域**  
包含 file、resource、config、theme、i18n、window、plugin、module、platform、log、credential、card、box、zip、serializer、control-plane 等 16 个核心服务。

**第四层：Plugin Runtime**  
负责插件加载、隔离、会话、权限管理。

**第五层：Bridge Transport**  
通过 Electron 的 contextBridge 机制向插件暴露 `window.chips.*` 标准接口，是插件访问系统能力的唯一通道。

**第六层：Runtime Client**  
负责协议编解码、错误归一、重试。

**第七层及以上**：UI Hooks、声明式UI层、统一渲染层、无头组件库、Theme Runtime、App/Page Domain。

### 单一主机模式

用户设备上只安装一个薯片主机，这是唯一的 Electron 应用程序。所有上层软件都是运行在主机内部的插件应用。每个插件以 `.cpk` 格式打包（本质是 ZIP 文件），安装时解压到插件目录，在操作系统中创建快捷方式。

用户双击快捷方式时，快捷方式指向薯片主机并带上启动参数指定加载哪个插件。主机从插件注册表中找到对应插件，创建 BrowserWindow 窗口，加载插件的 Web 入口页面。对用户来说，感觉就像启动了一个独立的软件。

如果用户同时打开多个薯片软件，它们都运行在同一个主机进程中。多个插件共享同一个 Chromium 引擎和主进程中的基础服务，系统资源占用远少于独立运行多个 Electron 应用。每个插件的渲染进程互相隔离，一个插件崩溃不会影响其他插件和主进程。

### 薯片命令行工具

生态提供 **chips** 用户端命令行工具，安装在系统 PATH 中，提供各种常用功能：

- 插件安装、卸载、列表查询
- 语言切换
- 主题切换
- 生态工作区路径设置
- 卡片和箱子文件的创建、查看、编辑
- 文件格式转换

命令行工具让用户可以通过终端快速操作生态，提升使用效率。

## 薯片SDK与开发工具

### 薯片SDK

薯片SDK为开发者提供各种开发工具，包括：

**接口与模块清单**：完整的薯片各种接口、模块以及调用方法清单。所有服务动作必须定义输入 schema、输出 schema、权限、超时、错误码。

**软件开发工具包**：基于 Bridge API 构建，提供类型安全的方式访问基础层的所有能力。SDK 是 Bridge API 的高层封装，提供更友好的开发接口和 Vue 3 Composables。

**通用能力封装**：封装了文件操作、渲染功能、媒体播放、界面组件、数据处理、网络请求等全方位能力。开发者只需要编写极少的代码（主要是界面调用），即可实现复杂功能。

### 薯片开发者命令行工具

**chipsdev** 是薯片生态的开发者命令行工具，提供各种开发命令：

- 项目初始化：从插件模板创建新项目
- 开发服务器启动：本地调试运行
- 插件打包：将代码打包为 .cpk 格式
- 硬编码检查：检测代码中的硬编码文本
- 国际化提取：从代码中提取需要翻译的文本
- 代码校验：检查代码是否符合生态规范

### 插件模板 ×6

生态提供六种插件模板，每个模板都是独立的项目仓库，提供插件开发的框架，实现项目初始化：

**1. 应用插件模板**  
用于创建完整的应用插件，如编辑引擎、查看器、资源管理器等。包含独立窗口框架、基础 UI 组件、业务编排模板。

**2. 卡片插件模板**  
用于创建基础卡片插件。包含渲染组件框架、编辑组件框架、信息清单模板、配置处理逻辑。

**3. 布局插件模板**  
用于创建箱子布局插件。包含布局编辑器框架、布局渲染器框架、配置结构定义。

**4. 模块插件模板**  
用于创建共享功能模块插件。包含模块加载接口、能力暴露定义、生命周期管理模板。

**5. 主题包模板**  
用于创建主题插件。包含完整的 Token 定义结构、组件样式接口点、动效定义、多语言支持模板。

**6. 语言包模板**  
用于创建多语言支持包。包含翻译文本结构、多语言切换接口、动态加载模板。

所有模板都遵循生态的开发规范，包含多语言支持、主题系统、日志记录等基础设施。开发者可以基于模板快速开发符合规范的插件。

## 薯片组件库与前端框架

### 无头组件库

薯片组件库采用无头设计（Headless Components），只负责实现功能逻辑、结构、布局，**不包含任何配色、样式、形状、动画、背景等外观**。视觉表现完全由主题包注入。

技术栈采用：
- **Zag.js**：作为组件状态机引擎
- **Ark UI**：作为 React 框架绑定层
- **Style Dictionary**：管理设计 Token
- **Tamagui (Core)**：作为核心优化层

在以上开源库的基础上进行小幅度的二次开发和自研，实现全生态深度的主题系统。

组件通过 CSS 类名和 CSS 变量暴露样式接口点。主题包通过这些接口点注入具体的视觉定义，包括颜色、字体、圆角、阴影、动画等。所有样式都由主题包控制，同一套组件在不同主题下呈现完全不同的视觉效果。

### 声明式前端框架

生态提供声明式前端框架（Declarative UI），提供各种常见布局语义原语：

- **View**：基础视图容器
- **Stack**：水平或垂直堆叠
- **Grid**：网格布局
- **Form**：表单布局
- **List**：列表布局

开发者使用这些语义原语构建界面，具体的渲染细节由底层统一处理。

### 自管理前端架构

薯片生态采用自管理前端的架构设计：

- 最下层有最高的优先级，如果最下层没有设置样式，就自动向上寻找
- 一直找到有样式设定的层级并应用
- 组件库只负责实现功能逻辑、结构、布局等作用，不包含样式、设计系统、动画
- 这些全部在主题包里实现

主题包可以将各种设计风格做成模板，包括：液态玻璃（Glassmorphism）、Material Design 3、Fluent UI、Bento、Neobrutalism、Atlassian Design、Linear、Notion 等。

### 多层级主题继承系统

主题系统是一个多层级继承系统，从最小单位向上寻找。主题配置采用六级覆盖机制：

**组件主题 → 基础卡片主题 → 复合卡片主题 → 箱子主题 → 应用主题 → 全局主题**

配置文件中的主题字段默认为空。如果某个层级没有设置主题，就使用上一层级的主题。如果所有层级都没有设置，使用系统默认主题。

## 插件系统

### 一切皆插件

薯片生态采用一切皆插件的设计哲学。基础卡片是插件、箱子布局是插件、编辑引擎布局是插件、查看工具是插件、主题包是插件，甚至文件管理器和程序坞也是插件。

插件分为六种类型：

**1. 应用插件**  
拥有独立窗口的完整 Web 应用，如编辑引擎、查看器、资源管理器、生态设置面板。每个应用插件在操作系统中可以创建独立的快捷方式，用户使用体验与普通软件一致。数据使用卡片文件格式保存，不依赖数据库。

**2. 卡片插件**  
基础卡片的渲染和编辑组件。每个基础卡片类型对应一个卡片插件，定义该类型卡片如何显示和编辑。分为渲染组件（查看器中显示）和编辑组件（编辑引擎中修改）。

**3. 布局插件**  
箱子的展示方式定义。决定箱子内部卡片的排列规则、排序方式、交互模式等。包括编辑时的布局编辑器和查看时的布局渲染器。

**4. 模块插件**  
共享功能模块，提供可供其他插件复用的功能。不创建独立窗口，由其他插件按需加载使用。如视频播放器、音频播放器、图片查看器等。

**5. 主题插件**  
视觉样式包，定义界面的颜色、字体、圆角、阴影、动画等视觉属性。使用 CSS 和 CSS 变量实现，通过接口点与组件库连接。

**6. 语言包插件**  
多语言支持包，提供界面文本翻译和语言切换能力。

### 插件包格式

所有插件使用统一的 `.cpk` 格式打包，本质是一个 ZIP 压缩包。压缩时采用零压缩率即存储模式，确保快速解压安装。

每个插件都是独立的代码仓库，有自己的版本管理和发布流程。插件之间通过标准接口通信，互不依赖。插件可以独立开发、独立测试、独立部署。

### 热插拔机制

热插拔是插件系统的重要特性。插件可以不重启软件即可完成安装、更新和卸载操作。用户安装新插件后，系统动态加载插件代码，立即可用。更新插件时，系统先卸载旧版本再加载新版本。

### 基础卡片插件结构

每个基础卡片插件是一个完整的插件包，包含这个基础卡片相关的所有内容：

**渲染组件（前端代码）**  
插件的前端代码，定义这种卡片如何在查看器中显示。包含 HTML 结构、CSS 接口点、JavaScript 交互逻辑。

**编辑组件**  
在编辑引擎中编辑卡片内容的界面。用户提供界面让用户设置卡片参数。

**信息清单**  
记录卡片类型的元数据，包括类型名称、唯一标识符、版本号、作者信息、支持的文件格式、依赖关系等。

**配置文件和数据分离**  
卡片文件保存内容数据（视频路径、封面、字幕、播放参数等），基础卡片插件保存前端代码。渲染器连接两者：读取配置 → 调用插件前端代码 → 填充数据 → 生成 iframe。

## 内容体系

### 基础卡片和复合卡片

基础卡片是最小的不可拆分内容单元，不能独立存在，必须作为复合卡片的组成部分。每种基础卡片都是一个完整的插件。基础卡片类型可以无限添加，系统预置常用的基础卡片类型。

复合卡片是可以保存为独立文件的最小单元，对应 `.card` 文件。包含一个或多个基础卡片，可以嵌套其他复合卡片，支持层层嵌套。用户拖拽第一个基础卡片到编辑引擎时，系统自动创建复合卡片容器。

基础卡片和复合卡片的根本区别在于能否独立存储。基础卡片没有独立的文件格式，它的数据保存在复合卡片的 `content/` 目录下。复合卡片有完整的文件结构。

### 箱子

箱子是卡片的容器，与卡片是并行同等地位的概念。在文件系统中表现为文件夹加同名 `.box` 文件。通过布局插件定义卡片的展示方式。

系统预置常用的箱子布局类型，数量可以无限添加。

布局不只是简单的页面排版，而是负责卡片展示的全方位功能，包括排序算法、页面交互方式、各种功能模块。每种布局由独立的插件提供。

箱子分为三种类型：全填充箱子（所有卡片在箱子内部）、空壳箱子（所有卡片在外部）、半空壳箱子（部分内部部分外部）。用户导入卡片时选择处理方式（复制、移动、登记地址），系统自动判断箱子类型。

### 层次结构

从小到大依次是：基础卡片 → 复合卡片 → 箱子 → 房间 → 空间 → 社区。

房间是用户在空间内的组织单位，可以层层嵌套。空间对应物理存储设备（电脑、手机、硬盘、网盘），系统自动识别，不需要用户手动创建。社区是网络上的公共平台，由社区服务器提供，用户可以创建房间、分享内容。

## 核心架构

### 中心路由架构原则

这是整个架构最核心的设计原则。所有模块的通信必须经过薯片内核作为中转站，模块之间不能直接通信。

当模块需要调用另一个模块的功能时，它向内核发送请求。内核接收请求后，验证格式和权限，然后路由到目标模块。目标模块处理完成后将结果返回给内核，内核再转发给发起请求的模块。整个过程中，两个模块从未直接接触。

内核不仅路由生态内部的通信，还负责生态内外的所有对接。从外部世界（操作系统、文件系统、网络、用户输入）进入生态的所有内容，都通过内核作为统一入口。内核提供文件识别、资源解析、用户输入捕获等能力。

这种架构让模块完全解耦，每个模块是独立的孤岛，只知道如何与内核交互。模块可以随时被替换成另一个实现相同接口的模块。

### 十二层架构

薯片生态采用十二层架构，从底层到顶层分为十二层：

| 层级 | 名称 | 核心职责 |
|------|------|----------|
| L1 | PAL | 平台能力抽象 |
| L2 | Host Kernel | router/registry/event/lifecycle |
| L3 | Host Services | file/resource/config/theme/i18n 等 |
| L4 | Plugin Runtime | 插件加载、隔离、会话、权限 |
| L5 | Bridge Transport | `window.chips.*` IPC 传输 |
| L6 | Runtime Client | 协议编解码、错误归一、重试 |
| L7 | UI Hooks | 面向页面的能力调用接口 |
| L8 | Declarative UI | View/Stack/Grid/Form/List 语义原语 |
| L9 | Unified Rendering | Normalize/Validate/Resolve/Commit |
| L10 | Component Headless | 状态机、结构、a11y |
| L11 | Theme Runtime | token 解析、作用域覆盖、变量注入 |
| L12 | App/Page Domain | 页面状态、业务流程、编排 |

公共基础层遵循**开源优先原则**：凡是有成熟开源方案的功能（视频播放、PDF 解析、Markdown 渲染、代码高亮等），采用提取和集成开源模块的方式，只对生态高度定制的功能（卡片文件格式解析、主题引擎等）进行全新开发。

### 极致模块化

每个功能都是独立模块，层层嵌套组合。所有模块通过内核路由通信，完全解耦。插件使用统一的 `.cpk` 格式打包，可以随时替换，只要实现相同接口。插件有独立版本号，遵循语义化版本规范，采用按需加载机制。

### 前后端完全分离

前端和后端完全解耦，通过内核通信，从不直接接触。

同一个后端功能可以有多个不同的前端实现。比如待办功能的后端是统一的，但可以有列表风格、看板风格、日历风格的界面。官方产品统一使用 React 框架开发。第三方开发者可以用 React、Svelte 等任何技术开发前端产品，只要通过薯片内核的标准接口与后端通信即可。

## 卡片渲染机制

### 渲染流程

薯片生态的所有卡片都使用网页技术渲染。卡片文件保存内容数据，基础卡片插件保存前端代码和布局逻辑，渲染器负责连接两者并生成完整的网页。

**基础卡片渲染流程**：
1. 卡片渲染器（CardRenderer）读取卡片配置文件
2. 提取配置数据（视频路径、封面、字幕列表等）
3. 调用对应基础卡片插件的渲染组件（前端代码）
4. 将配置数据填充到前端模板中
5. 生成完整的 HTML 网页
6. 在 iframe 窗口中加载该网页
7. 用户看到完整的交互界面

每个基础卡片渲染为一个独立的 iframe 窗口，iframe 背景设置为透明。

**复合卡片渲染**：
1. 渲染器读取 structure.yaml，获取基础卡片列表和顺序
2. 按顺序渲染每个基础卡片，每个生成一个 iframe
3. 将所有 iframe 垂直排列，组合成完整的卡片窗口
4. 顶部添加菜单栏（卡片名称和操作按钮）
5. 如果设置了主题包，主题包作为整个窗口的背景网页

最终输出一个完整的 iframe 窗口，可以嵌入到任何产品中。

### 渲染器在基础层的作用

公共基础层的 CardRenderer 模块包含完整的渲染逻辑。编辑引擎、查看器、第三方产品都调用同一个渲染引擎，确保卡片显示的一致性。

渲染器通过内核路由调用其他模块。需要解析 ZIP 时调用 ZIPProcessor，需要播放视频时调用 VideoPlayer。渲染器本身不直接实现这些功能，而是组织和调度。

## 文件格式

### 卡片文件

`.card` 文件本质是零压缩率的 ZIP 压缩包。包含 `.card/` 配置文件夹、`content/` 内容文件夹和资源文件。

**.card/ 配置文件夹**包含三个必需文件：
- **metadata.yaml**：元数据，记录卡片 ID（十位 62 进制编号）、名称、创建时间、修改时间、主题配置、标签等
- **structure.yaml**：结构定义，记录包含哪些基础卡片及顺序，每个条目包含 ID 和类型名称
- **cover.html**：封面定义，用于在列表或箱子中显示预览

**content/ 内容文件夹**存放每个基础卡片的配置文件，文件名是十位 62 进制 ID.yaml。配置文件的具体结构由插件定义，不同类型的卡片有不同的配置项。

**资源文件**直接放在根目录，包括图片、视频、音频、字幕、嵌套的卡片文件等。

卡片支持三种资源模式：
- **全填充卡片**：所有资源都打包在卡片文件内部，适合分享和备份
- **空壳卡片**：所有资源都引用外部文件，卡片体积小但依赖外部
- **半空壳卡片**：部分资源在内部、部分在外部，灵活平衡

资源路径可以是相对路径（卡片内部）、绝对路径（文件系统其他位置）或网址（网络资源）。

### 箱子文件

`.box` 文件本质是零压缩率的 ZIP 压缩包。包含 `.box/` 配置文件夹和可选的卡片文件。

**.box/ 配置文件夹**包含三个必需文件：
- **metadata.yaml**：箱子元数据
- **structure.yaml**：卡片列表，记录每个卡片的 ID、引用路径、是否在内部
- **content.yaml**：布局配置，由布局插件定义和解析

箱子默认扫描同级文件夹查找外部卡片，用户也可以在设置中添加额外的扫描路径，包括本地文件夹、网络地址、WebDAV 网盘等。

## 主题系统

### 主题包结构

薯片生态的组件库采用无头设计（基于 Zag.js + Ark UI），只实现功能逻辑、结构和布局，完全不包含任何视觉样式代码。真正的样式和界面代码保存在主题包中。

主题包是完整的前端视觉实现，包含所有组件的样式定义：按钮的颜色形状、输入框的边框样式、卡片的间距阴影、动画效果等。

主题包使用 CSS 变量 Token 系统，分为五层：
- **ref（原始值）**：最基础的数值
- **sys（语义值）**：语义化的设计决策
- **comp（组件值）**：组件级别的定义
- **motion（动效值）**：动画效果参数
- **layout（布局密度值）**：间距和尺寸

主题包可以添加装饰性元素（背景图片、图标），但不能修改组件的 HTML 结构和布局方式。

### 原子级主题应用

主题支持原子级应用。每个基础卡片可以单独设置主题包，每个界面组件也可以单独设置主题包。一个复合卡片中的不同基础卡片可以使用不同的主题包。

主题配置采用六级覆盖机制：**组件主题 → 基础卡片主题 → 复合卡片主题 → 箱子主题 → 应用主题 → 全局主题**。

配置文件中的主题字段默认为空。如果某个层级没有设置主题，就使用上一层级的主题。如果所有层级都没有设置，使用系统默认主题。

主题包系统主键使用 `themeId`（例如 `chips-official.default-theme`）。界面可展示"薯片官方：马卡龙配色"这类名称，但运行时解析必须使用 `themeId`。

## 产品族

### 生态设置面板

这是用户在设备上管理整个生态底层用的设置面板，是一个简单的套壳应用。

设置面板负责以下功能：
- **主题系统管理**：安装、切换、预览主题
- **多语言系统管理**：切换界面语言
- **插件管理**：安装、卸载、启用、禁用插件
- **运行情况查看**：系统状态、性能监控
- **生态工作区设置**：设置文件存储路径
- **其他系统设置**：未来会不断添加新页面和项目

所有接口都在主机里，生态设置面板只是一个简单的呈现所有接口，并提供对应的前端图形化操作 UI。

### 卡片查看器

用于查看卡片和箱子的软件。这是一个非常简单的套壳应用。

查看器直接在一个窗口中插入一个卡片渲染窗口组件，加上文件传递逻辑即可。卡片渲染由公共基础层的 CardRenderer 模块负责，查看器只需调用 SDK 中的显示卡片接口，传入卡片文件即可。

查看器提供窗口管理、导航控制、布局切换、收藏历史等扩展功能。

### 卡片编辑引擎

用于创建和编辑卡片的软件平台。这是一个工作台应用，是生态中最重要的生产力工具。

编辑引擎本身是框架，实际编辑功能由插件提供。所有操作实时自动保存。

**主要组件**：

**编辑面板窗口**：在编辑面板中提供 iframe 接口，用来接入嵌套基础卡片的编辑器。用户在编辑引擎中点击基础卡片时，编辑面板调用相应的基础卡片编辑器，用户可以在编辑面板中使用对应的编辑器编辑基础卡片内容。

**卡箱库**：显示已经安装在系统上的所有基础卡片类型和箱子布局类型。用户从卡箱库拖动卡片类型到工作区域即可创建新卡片。

**文件管理器**：管理用户在引擎中创建的所有卡片文件和箱子文件。

**程序坞（DOCK）**：显示所有的工具按钮，目前主要包括：编辑面板、卡箱库、文件管理器、编辑引擎设置。未来还会增加更多工具。

**支持多种布局模式**：

**无限画布布局**采用两层设计：
- **桌面层**：无限扩展的画布，用户可以自由摆放卡片和箱子，放大缩小，拖动平移
- **窗口层**：固定在屏幕位置，包含工具窗口（文件管理器、编辑面板、卡箱库）和程序坞

**工作台布局**采用分区窗口组合方式：左侧文件树，中间预览区，右侧编辑窗口。窗口可以拖动重排，适合专注单卡片编辑。

### 社区服务器

服务器端软件，提供多用户社区服务- **存储服务器。由两部分组成：
（Hub Storage）**：专门为薯片文件优化的网盘系统
- **社区平台（Hub Platform）**：完整的多用户操作环境，支持房间管理、内容管理、社区功能

可以部署在不同场景：个人小型社区、组织内网知识库、公网公共社区。

### 软件制作器

可视化的软件组装工具。用户通过图形化界面选择前端组件和后端模块，拖拽组装，制作器自动生成内核的路由配置。

### 资源管理器

轻量化的文件浏览工具，基于标签系统。以应用插件形式运行，可作为独立入口使用并管理任意类型文件。界面分为标签导航栏、主区域、侧边栏、状态栏。

## 标签文件系统

薯片生态采用全新的标签文件系统，替代传统的树状目录结构。整个文件系统是一个大的资源池，通过标签层层筛选来组织文件。

### 标签筛选路径

文件路径被重新定义为标签的筛选路径。用户依次选择标签，每个标签是一次筛选，最终得到符合所有标签条件的文件。同一个文件可以通过不同的标签路径到达。

标签支持键值对结构，如"月份：三月"、"创作者：小明"、"项目：网站改版"。标签可以嵌套，形成层级关系，如"地点：亚洲：中国"。

用户在某个筛选路径下创建文件，路径上的所有标签自动添加到新文件中。资源管理器中的"文件夹"是动态生成的虚拟视图，基于标签筛选结果。

### 文件系统映射

薯片生态提供两种文件操作模式：

**映射模式**：软件在内部建立文件系统的虚拟映射。所有薯片文件保存在软件工作文件夹，不污染用户原本的文件系统。标签信息保存在软件内部的标签数据库中。

**直接创建模式**：薯片文件直接创建在用户文件系统的当前位置。标签信息保存在隐藏配置文件或直接保存在 metadata.yaml 中。

用户可以选择使用哪种模式，或者针对不同文件夹使用不同模式。

---

## 资源管理

资源管理是内核提供的统一资源对接层。将本地文件系统、网络存储、网盘等各种环境统一抽象成总资源接口。模块通过内核的资源接口访问资源，不需要知道资源的实际位置和获取方式。

整个生态支持无感使用各种网络协议。认证信息保存在软件本地的秘钥管理工具中，不保存在卡片或箱子文件中。资源管理自动处理缓存，网络资源下载后缓存到本地。

卡片引用外部资源时，资源缺失可以重新绑定或下载到卡片内部。文件连接器提供批量绑定功能，支持手动连接、自动识别命名规律两种模式。

## 卡片使用和交互

### 创建和编辑

用户从卡箱库拖动基础卡片到桌面空白区域，自动创建复合卡片。继续拖动其他基础卡片到卡片窗口，根据鼠标位置决定插入位置（上半部分插入上方，下半部分插入下方）。

从操作系统拖动文件到卡片，系统识别文件类型，自动创建对应类型的基础卡片。拖动一个完整的卡片文件到另一个卡片，被拖动的卡片作为嵌套卡片插入。

点击基础卡片打开编辑面板窗口，显示对应的编辑组件插件。用户修改内容，实时保存到配置文件。编辑组件的界面完全由插件设计，不区分普通设置和高级设置。

### 模式切换

复合卡片通过锁定按钮切换编辑模式和查看模式：
- **编辑模式**：点击基础卡片打开编辑组件，可以调整顺序，可以添加删除卡片
- **查看模式**：可以与内容交互（播放视频、勾选列表），不打开编辑组件

### 窗口状态

卡片窗口有多种显示状态：
- **展开状态**：显示所有内容，高度根据内容自适应
- **收起状态**：高度缩小到约 16:9 比例，内容可滚动查看
- **封面状态**：只显示封面窗口，占用空间最小
- **最小化**：窗口完全关闭，不显示在桌面和程序坞

封面保存在 cover.html 中，通过网页框架显示。封面窗口大小固定，比例可调（支持 11 种比例），默认标准照片比例竖版。

## 拖动交互

薯片生态支持丰富的拖动操作：

- 从操作系统拖动文件到编辑引擎，自动创建对应卡片
- 从卡箱库拖动卡片类型到桌面或现有卡片
- 从桌面拖动卡片到另一个卡片实现嵌套
- 在无限桌面上拖动平移和调整窗口
- 在复合卡片内拖动调整基础卡片顺序

所有拖动操作都有实时视觉反馈：文件图标跟随、插入指示线、高亮区域、禁止符号等。系统根据拖动的起点、终点、修饰键等判断用户意图。

## 开发规范

所有模块必须遵循统一规范：

- **多语言支持**：所有界面文本支持多语言切换，不允许硬编码文本
- **变量配置化**：所有配置项通过配置文件管理，采用层级覆盖结构
- **日志记录**：记录所有重要操作、错误和警告，按级别分类
- **撤销重操作支持撤销和做**：所有用户重做
- **前后端分离**：前端只负责界面，后端只负责业务逻辑
- **主题包支持**：前端组件必须支持深度主题定制，不能硬编码样式
- **通过内核路由**：模块之间的所有通信必须经过内核路由
- **标准化错误处理**：使用统一错误对象 `{ code, message, details, retryable }`

## 薯片协议

生态定义了一套薯片协议，规范各方面的标准：

- **文件格式协议**：`.card` 和 `.box` 文件结构、配置文件格式、资源引用规范
- **接口标准协议**：前端接口标准、后端接口标准、模块调用协议
- **通信协议**：请求响应格式、路由调用标准、错误处理规范
- **主题协议**：Token 完整性校验、组件接口点校验、动效参数安全校验
- **事件协议**：统一事件元信息 `id/name/source/data/timestamp/metadata`

所有协议文档保存在生态文档中，所有项目都遵循这些协议，确保整个生态的一致性和互操作性。

基于 Electron 技术栈构建，可以跨平台运行：
- **桌面端**：Windows、macOS、Linux
- **移动端**：未来计划支持 iOS、Android

同一套代码，多平台运行。平台差异收敛到 PAL 层，核心代码不直连平台 API。

## 总结

薯片生态是一个庞大的软件级生态系统，目标是覆盖用户电脑的大部分使用场景，让全平台使用体验统一。

核心架构特点：

- **单一主机模式**：薯片主机是唯一安装的底层软件，所有应用以插件形式运行
- **厚底层、薄上层**：公共基础层包含所有通用功能，上层应用只是调用底层能力的薄壳
- **中心路由原则**：所有模块通过内核路由通信，完全解耦
- **一切皆插件**：所有功能都是插件，高度模块化，可热插拔
- **无头组件库**：只负责逻辑和结构，视觉完全由主题包注入
- **多层级主题继承**：从组件到全局的分级主题覆盖
- **前后端完全分离**：前端和后端通过标准接口对接，可自由组合
- **标签文件系统**：替代树状目录，灵活组织内容
- **文件格式规范**：`.card` 和 `.box` 文件作为数据载体，不依赖数据库

用户安装一个薯片底层软件后，可以自己添加安装很多插件小程序。生态内的所有小工具和软件都是插件的形式在整个底层软件上运转。

开发者可以用很方便、很简单的方式为薯片生态开发软件，不需要处理各种底层的逻辑和模块。只需按照薯片 SDK 和相关标准进行开发，就能开发出完美符合官方设计规范和交互规范的软件。

未来这个底层环境将长期无感运行在用户电脑上，用户直接安装各种上层应用，直接就能使用，使用体验和其他普通软件一样。薯片生态将逐步覆盖用户日常使用的大部分工具软件，成为一个真正的软件级生态系统。
