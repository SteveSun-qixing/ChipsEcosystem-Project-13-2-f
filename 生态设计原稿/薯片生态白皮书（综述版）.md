# 薯片生态白皮书（综述版）

**版本**：1.1.0  
**日期**：2026-01-31

---

## 核心理念

薯片生态是一个内容创作和管理系统，核心理念是让复杂内容像打开图片一样简单。用户双击 .card 文件就能自动打开查看，可以把卡片嵌入到网页或其他软件中，可以分享给他人直接使用。

---

## 内容体系

### 基础卡片和复合卡片

基础卡片是最小的不可拆分内容单元，不能独立存在，必须作为复合卡片的组成部分。每种基础卡片都是一个完整的插件。基础卡片类型可以无限添加，系统预置常用的基础卡片类型。

复合卡片是可以保存为独立文件的最小单元，对应 .card 文件。包含一个或多个基础卡片，可以嵌套其他复合卡片，支持层层嵌套。用户拖拽第一个基础卡片到编辑引擎时，系统自动创建复合卡片容器。

基础卡片和复合卡片的根本区别在于能否独立存储。基础卡片没有独立的文件格式，它的数据保存在复合卡片的 content/ 目录下。复合卡片有完整的文件结构。

### 箱子

箱子是卡片的容器，与卡片是并行同等地位的概念。在文件系统中表现为文件夹加同名 .box 文件。通过布局插件定义卡片的展示方式。

系统预置常用的箱子布局类型，数量可以无限添加。

布局不只是简单的页面排版，而是负责卡片展示的全方位功能，包括排序算法、页面交互方式、各种功能模块。每种布局由独立的插件提供。

箱子分为三种类型：全填充箱子（所有卡片在箱子内部）、空壳箱子（所有卡片在外部）、半空壳箱子（部分内部部分外部）。用户导入卡片时选择处理方式（复制、移动、登记地址），系统自动判断箱子类型。

### 层次结构

从小到大依次是：基础卡片 → 复合卡片 → 箱子 → 房间 → 空间 → 社区。

房间是用户在空间内的组织单位，可以层层嵌套。空间对应物理存储设备（电脑、手机、硬盘、网盘），系统自动识别，不需要用户手动创建。社区是网络上的公共平台，由社区服务器提供，用户可以创建房间、分享内容。

## 文件格式

### 卡片文件

.card 文件本质是零压缩率的 ZIP 压缩包。包含 .card/ 配置文件夹、content/ 内容文件夹和资源文件。

**.card/ 配置文件夹**包含三个必需文件：
- **metadata.yaml**：元数据，记录卡片 ID（十位 62 进制编号）、名称、创建时间、修改时间、主题配置、标签等
- **structure.yaml**：结构定义，记录包含哪些基础卡片及顺序，每个条目包含 ID 和类型名称
- **cover.html**：封面定义，用于在列表或箱子中显示预览

**content/ 内容文件夹**存放每个基础卡片的配置文件，文件名是十位 62 进制 ID.yaml。配置文件的具体结构由插件定义，不同类型的卡片有不同的配置项。

**资源文件**直接放在根目录，包括图片、视频、音频、字幕、嵌套的卡片文件等。

卡片支持三种资源模式：
- **全填充卡片**：所有资源都打包在卡片文件内部，适合分享和备份
- **空壳卡片**：所有资源都引用外部文件，卡片体积小但依赖外部
- **半空壳卡片**：部分资源在内部、部分在外部，灵活平衡

资源路径可以是相对路径（卡片内部）、绝对路径（文件系统其他位置）或网址（网络资源）。

### 箱子文件

.box 文件本质是零压缩率的 ZIP 压缩包。包含 .box/ 配置文件夹和可选的卡片文件。

**.box/ 配置文件夹**包含三个必需文件：
- **metadata.yaml**：箱子元数据
- **structure.yaml**：卡片列表，记录每个卡片的 ID、引用路径、是否在内部
- **content.yaml**：布局配置，由布局插件定义和解析

箱子默认扫描同级文件夹查找外部卡片，用户也可以在设置中添加额外的扫描路径，包括本地文件夹、网络地址、WebDAV 网盘等。

## 核心架构

### 中心路由架构原则

这是整个架构最核心的设计原则。所有模块的通信必须经过薯片内核作为中转站，模块之间不能直接通信。

当模块需要调用另一个模块的功能时，它向内核发送请求。内核接收请求后，验证格式和权限，然后路由到目标模块。目标模块处理完成后将结果返回给内核，内核再转发给发起请求的模块。整个过程中，两个模块从未直接接触。

内核不仅路由生态内部的通信，还负责生态内外的所有对接。从外部世界（操作系统、文件系统、网络、用户输入）进入生态的所有内容，都通过内核作为统一入口。内核提供文件识别、资源解析、用户输入捕获等能力。

这种架构让模块完全解耦，每个模块是独立的孤岛，只知道如何与内核交互。模块可以随时被替换成另一个实现相同接口的模块。

### 十二层架构

薯片生态采用十二层架构，从底层到顶层分为十二层：

**第一层：PAL（平台抽象层）** - 适配不同操作系统（Windows、macOS、Linux）的差异，向上提供统一的平台能力接口。

**第二层：Host内核** - 使用TypeScript实现，运行在Electron主进程中。提供中心路由、服务注册、事件总线、生命周期管理等核心能力。

**第三层：Host服务域** - 包含file、resource、config、theme、i18n、window、plugin、module、platform、log、credential、card、box、zip、serializer、control-plane等服务。

**第四层：Plugin Runtime** - 负责插件加载、隔离、会话、权限管理。

**第五层：Bridge Transport** - 通过Electron的contextBridge机制向插件暴露window.chips.*标准接口。

**第六层：Runtime Client** - 负责协议编解码、错误归一、重试。

**第七层：UI Hooks** - 面向页面的能力调用接口。

**第八层：声明式UI层** - 提供View、Stack、Grid、Form、List等语义原语。

**第九层：统一渲染层** - 负责Normalize、Validate、Resolve、Commit。

**第十层：无头组件库** - 状态机、结构、a11y，无视觉硬编码。

**第十一层：Theme Runtime** - token解析、作用域覆盖、变量注入。

**第十二层：App/Page Domain** - 页面状态、业务流程、编排。
- **共享功能模块**（作为模块插件独立安装）：视频播放器、音频播放器、图片查看器、3D 渲染器、富文本编辑器、Markdown 渲染器、代码高亮器、文本编辑器等。各自独立仓库，可自由替换

**第五层：插件桥接层（Bridge API）** - 通过 Electron 的 contextBridge 机制向插件暴露 `window.chips.*` 标准接口，是插件访问系统能力的唯一通道。SDK 封装 Bridge API 为类型安全的开发接口。薯片组件库基于 Zag.js + Ark UI 构建，采用无头设计（零样式），所有视觉表现由主题包注入。

**第六层：插件应用** - 编辑引擎、查看器、设置面板、资源管理器等完整应用，每个都是运行在独立 BrowserWindow 中的 Web 应用。插件分为五种类型：应用插件、卡片插件、布局插件、模块插件、主题插件。

公共基础层遵循**开源优先原则**：凡是有成熟开源方案的功能（视频播放、PDF 解析、Markdown 渲染、代码高亮等），采用提取和集成开源模块的方式，只对生态高度定制的功能（卡片文件格式解析、主题引擎等）进行全新开发。

### 极致模块化

每个功能都是独立模块，层层嵌套组合。插件分为五种类型：

- **应用插件**：完整的应用，如编辑引擎、查看器，拥有独立窗口
- **卡片插件**：基础卡片类型的渲染和编辑组件
- **布局插件**：箱子的布局展示方式
- **模块插件**：共享功能模块（视频播放器、图片查看器等），可被多个插件加载使用
- **主题插件**：CSS 样式主题包

每个插件是独立的代码仓库，使用统一的 `.cpk` 格式打包。插件通过 Bridge API 和内核路由通信，完全解耦。插件可以随时替换，只要实现相同接口。插件有独立版本号，遵循语义化版本规范，采用按需加载机制。

### 前后端完全分离

前端和后端完全解耦，通过内核通信，从不直接接触。

同一个后端功能可以有多个不同的前端实现。比如待办功能的后端是统一的，但可以有列表风格、看板风格、日历风格的界面。官方产品统一使用React框架开发。第三方开发者可以用React、Svelte等任何技术开发前端产品，只要通过薯片内核的标准接口与后端通信即可。

前端接口标准定义了前端如何调用后端功能、如何接收数据、如何处理事件。后端接口标准定义了后端模块提供哪些功能、接收什么参数、返回什么数据。内核的路由系统根据这两个标准进行转换和适配。

## 插件系统

### 一切皆插件

基础卡片是插件、箱子布局是插件、编辑引擎布局是插件、查看工具是插件、主题包是插件，甚至文件管理器和程序坞也是插件。

插件支持热插拔，大部分插件可以不重启软件即可安装、更新、卸载。插件可以组合成独立入口的应用产品，但运行时始终由 Chips Host 统一承载。待办、视频等能力都以插件方式复用，数据使用卡片文件保存，不依赖数据库。

### 基础卡片插件的结构

每个基础卡片插件是一个完整的插件包，包含这个基础卡片相关的所有内容。除了播放器、文本编辑器等底层通用组件之外，基础卡片相关的所有代码都包含在它自己的插件库中。

一个基础卡片插件由几个核心部分组成：

**渲染组件（前端代码）**：插件的前端代码，定义了这种卡片如何在查看器中显示。这些代码使用网页技术编写（HTML、CSS、JavaScript），是卡片最终呈现给用户的界面和交互逻辑的完整实现。

以视频卡片为例，渲染组件包含：
- HTML 结构：定义视频播放器的界面布局
- CSS 接口点：暴露样式类名和 CSS 变量，供主题包注入样式
- JavaScript 交互逻辑：实现播放控制、字幕切换、进度记忆等功能

**编辑组件**：在编辑引擎中编辑卡片内容的界面，由插件开发者自行设计。

**信息清单**：记录卡片类型的元数据，包括类型名称、唯一标识符、版本号、作者信息等。

### 配置文件和数据分离

这是关键设计：
- **卡片文件保存内容数据**（配置文件）：视频路径、封面、字幕、播放参数等
- **基础卡片插件包含前端代码**：HTML 结构、CSS 接口点、JavaScript 交互逻辑
- **渲染器连接两者**：读取配置 → 调用插件前端代码 → 填充数据 → 生成 iframe

这种分离确保了：卡片文件轻量（只保存数据），插件可以独立更新（改进界面不影响已有卡片），主题可以灵活应用（前端代码只定义结构，样式由主题注入）。

## 卡片渲染机制

### 渲染流程

薯片生态的所有卡片都使用网页技术渲染。卡片文件保存内容数据，基础卡片插件保存前端代码和布局逻辑，渲染器负责连接两者并生成完整的网页。

**基础卡片渲染流程**：
1. 卡片渲染器（CardRenderer）读取卡片配置文件
2. 提取配置数据（视频路径、封面、字幕列表等）
3. 调用对应基础卡片插件的渲染组件（前端代码）
4. 将配置数据填充到前端模板中
5. 生成完整的 HTML 网页
6. 在 iframe 窗口中加载该网页
7. 用户看到完整的交互界面

每个基础卡片渲染为一个独立的 iframe 窗口，iframe 背景设置为透明。

**复合卡片渲染**：
1. 渲染器读取 structure.yaml，获取基础卡片列表和顺序
2. 按顺序渲染每个基础卡片，每个生成一个 iframe
3. 将所有 iframe 垂直排列，组合成完整的卡片窗口
4. 顶部添加菜单栏（卡片名称和操作按钮）
5. 如果设置了主题包，主题包作为整个窗口的背景网页

最终输出一个完整的 iframe 窗口，可以嵌入到任何产品中。

### 渲染器在基础层的作用

公共基础层的 CardRenderer 模块包含完整的渲染逻辑。编辑引擎、查看器、第三方产品都调用同一个渲染引擎，确保卡片显示的一致性。

渲染器通过内核路由调用其他模块。需要解析 ZIP 时调用 ZIPProcessor，需要播放视频时调用 VideoPlayer。渲染器本身不直接实现这些功能，而是组织和调度。

## 主题系统

薯片生态的组件库采用无头设计（基于 Zag.js + Ark UI），只实现功能逻辑、结构和布局，完全不包含任何视觉样式代码。真正的样式和界面代码保存在主题包中。

### 主题包结构

主题包是完整的前端视觉实现，包含所有组件的样式定义：按钮的颜色形状、输入框的边框样式、卡片的间距阴影、动画效果等。

主题包使用 CSS 变量 Token 系统，分为三层：
- **全局 Token**：色板
- **语义化 Token**：主要修改层
- **组件级 Token**：细粒度定制

主题包可以添加装饰性元素（背景图片、图标），但不能修改组件的 HTML 结构和布局方式。

### 原子级主题应用

主题支持原子级应用。每个基础卡片可以单独设置主题包，每个界面组件也可以单独设置主题包。一个复合卡片中的不同基础卡片可以使用不同的主题包。

主题配置采用六级覆盖机制：**组件主题 → 基础卡片主题 → 复合卡片主题 → 箱子主题 → 应用主题 → 全局主题**。

配置文件中的主题字段默认为空。如果某个层级没有设置主题，就使用上一层级的主题。如果所有层级都没有设置，使用系统默认主题。

主题包系统主键使用 `themeId`（例如 `chips-official.default-theme`）。界面可展示“薯片官方：马卡龙配色”这类名称，但运行时解析必须使用 `themeId`。

## 产品族

### 卡片编辑引擎（Chips Editor）

用于创建和编辑卡片的软件平台。编辑引擎本身是框架，实际编辑功能由插件提供。所有操作实时自动保存。

支持多种布局模式，目前有两种基础布局：

**无限画布布局**采用两层设计：
- **桌面层**：无限扩展的画布，用户可以自由摆放卡片和箱子，放大缩小，拖动平移
- **窗口层**：固定在屏幕位置，包含工具窗口（文件管理器、编辑面板、卡箱库）和程序坞

**工作台布局**采用分区窗口组合方式：左侧文件树，中间预览区，右侧编辑窗口。窗口可以拖动重排，适合专注单卡片编辑。

### 卡片查看器（Chips Viewer）

用于查看卡片和箱子的软件。查看器调用公共基础层的 CardRenderer 显示卡片，不自己实现渲染逻辑。提供窗口管理、导航控制、布局切换、收藏历史等扩展功能。

查看器和编辑引擎可以是两个独立入口的应用插件，也可以是同一应用插件中的两种模式（通过锁定按钮切换）。

### 社区服务器（Chips Hub）

服务器端软件，提供多用户社区服务。由两部分组成：
- **存储服务器（Hub Storage）**：专门为薯片文件优化的网盘系统
- **社区平台（Hub Platform）**：完整的多用户操作环境，支持房间管理、内容管理、社区功能

可以部署在不同场景：个人小型社区、组织内网知识库、公网公共社区。

### 软件制作器（Chips Composer）

可视化的软件组装工具。用户通过图形化界面选择前端组件和后端模块，拖拽组装，制作器自动生成内核的路由配置。

采用单一主机模式。薯片主机（Chips Host）是唯一安装在系统中的 Electron 应用，长期后台运行。所有上层软件以 `.cpk` 格式的插件安装到主机中，每个应用插件在操作系统中创建独立快捷方式，用户体验与普通软件一致。

### 资源管理器

轻量化的文件浏览工具，基于标签系统。以应用插件形式运行，可作为独立入口使用并管理任意类型文件。界面分为标签导航栏、主区域、侧边栏、状态栏。

### 辅助工具

**文件连接器**：批量绑定外部资源到卡片的工具，支持手动连接（表格或连线）、自动识别（分析命名规律）两种模式。

**标签和重命名编辑器**：简单的小工具窗口，用于编辑文件名和标签，提供智能推荐和自动补全。

## 标签文件系统

薯片生态采用全新的标签文件系统，替代传统的树状目录结构。整个文件系统是一个大的资源池，通过标签层层筛选来组织文件。

### 标签筛选路径

文件路径被重新定义为标签的筛选路径。用户依次选择标签，每个标签是一次筛选，最终得到符合所有标签条件的文件。同一个文件可以通过不同的标签路径到达。

标签支持键值对结构，如"月份：三月"、"创作者：小明"、"项目：网站改版"。标签可以嵌套，形成层级关系，如"地点：亚洲：中国"。

用户在某个筛选路径下创建文件，路径上的所有标签自动添加到新文件中。资源管理器中的"文件夹"是动态生成的虚拟视图，基于标签筛选结果。

### 文件系统映射

薯片生态提供两种文件操作模式：

**映射模式**：软件在内部建立文件系统的虚拟映射。所有薯片文件保存在软件工作文件夹，不污染用户原本的文件系统。标签信息保存在软件内部的标签数据库中。映射采用简单的文件名匹配机制，通过文件名和修改时间判断。

**直接创建模式**：薯片文件直接创建在用户文件系统的当前位置。标签信息保存在隐藏配置文件或直接保存在 metadata.yaml 中。

用户可以选择使用哪种模式，或者针对不同文件夹使用不同模式。

---

## 资源管理

资源管理是内核提供的统一资源对接层。将本地文件系统、网络存储、网盘等各种环境统一抽象成总资源接口。模块通过内核的资源接口访问资源，不需要知道资源的实际位置和获取方式。

整个生态支持无感使用各种网络协议。认证信息保存在软件本地的秘钥管理工具中，不保存在卡片或箱子文件中。资源管理自动处理缓存，网络资源下载后缓存到本地。

卡片引用外部资源时，资源缺失可以重新绑定或下载到卡片内部。文件连接器提供批量绑定功能，支持手动连接、自动识别命名规律两种模式。

## 卡片使用和交互

### 创建和编辑

用户从卡箱库拖动基础卡片到桌面空白区域，自动创建复合卡片。继续拖动其他基础卡片到卡片窗口，根据鼠标位置决定插入位置（上半部分插入上方，下半部分插入下方）。

从操作系统拖动文件到卡片，系统识别文件类型，自动创建对应类型的基础卡片。拖动一个完整的卡片文件到另一个卡片，被拖动的卡片作为嵌套卡片插入。

点击基础卡片打开编辑面板窗口，显示对应的编辑组件插件。用户修改内容，实时保存到配置文件。编辑组件的界面完全由插件设计，不区分普通设置和高级设置。

### 模式切换

复合卡片通过锁定按钮切换编辑模式和查看模式：
- **编辑模式**：点击基础卡片打开编辑组件，可以调整顺序，可以添加删除卡片
- **查看模式**：可以与内容交互（播放视频、勾选列表），不打开编辑组件

### 窗口状态

卡片窗口有多种显示状态：
- **展开状态**：显示所有内容，高度根据内容自适应
- **收起状态**：高度缩小到约 16:9 比例，内容可滚动查看
- **封面状态**：只显示封面窗口，占用空间最小
- **最小化**：窗口完全关闭，不显示在桌面和程序坞

封面保存在 cover.html 中，通过网页框架显示。封面窗口大小固定，比例可调（支持 11 种比例），默认标准照片比例竖版。

## 拖动交互

薯片生态支持丰富的拖动操作：

- 从操作系统拖动文件到编辑引擎，自动创建对应卡片
- 从卡箱库拖动卡片类型到桌面或现有卡片
- 从桌面拖动卡片到另一个卡片实现嵌套
- 在无限桌面上拖动平移和调整窗口
- 在复合卡片内拖动调整基础卡片顺序

所有拖动操作都有实时视觉反馈：文件图标跟随、插入指示线、高亮区域、禁止符号等。系统根据拖动的起点、终点、修饰键等判断用户意图。

## 开发规范

所有模块必须遵循统一规范：

- **多语言支持**：所有界面文本支持多语言切换，不允许硬编码文本
- **变量配置化**：所有配置项通过配置文件管理，采用层级覆盖结构
- **日志记录**：记录所有重要操作、错误和警告，按级别分类
- **撤销重做**：所有用户操作支持撤销和重做
- **前后端分离**：前端只负责界面，后端只负责业务逻辑
- **主题包支持**：前端组件必须支持深度主题定制，不能硬编码样式
- **通过内核路由**：模块之间的所有通信必须经过内核路由

---

## SDK 能力封装

开发工具包封装绝大部分底层逻辑和通用能力。第三方开发者只需要编写极少的代码（主要是界面调用），即可实现复杂功能。

比如开发视频播放器软件，通过 SDK 调用公共基础层的 VideoPlayer 模块，几行代码就能实现完整功能。开发者只需要设计界面和用户体验。

SDK 提供的封装包括：文件操作、渲染功能、媒体播放、界面组件、数据处理、网络请求等全方位能力。

### 本地软件简化设计

本地软件（编辑引擎、查看器、资源管理器等）不包含身份认证、用户管理、权限系统等功能。这些是社区服务器的职责，本地软件保持简单。本地软件只负责内容的创建、编辑、查看和组织，所有数据保存为文件，不需要用户注册登录。

## 薯片协议

生态定义了一套薯片协议，规范各方面的标准：

- **文件格式协议**：.card 和 .box 文件结构、配置文件格式、资源引用规范
- **接口标准协议**：前端接口标准、后端接口标准、模块调用协议
- **通信协议**：请求响应格式、路由调用标准、错误处理规范

所有协议文档保存在"生态共用"文件夹中，所有项目都遵循这些协议，确保整个生态的一致性和互操作性。

## 跨平台

基于 Chromium 内核，可以跨平台运行：
- **网页端**：直接在浏览器使用
- **桌面端**：用 Electron 封装
- **移动端**：用 Capacitor 封装

同一套代码，多平台运行。

## 总结

薯片生态通过中心路由架构、六层底层设计、极致模块化、插件系统、标签文件系统、前后端分离、主题系统等设计，构建了一个灵活强大的内容创作和管理平台。

核心特点：
- **卡片文件只保存数据**，插件包含前端代码，渲染器连接两者
- **公共基础层提供通用能力**，插件只需实现业务逻辑
- **所有模块通过内核路由通信**，完全解耦
- **标签文件系统替代树状目录**，灵活组织内容
- **一切皆插件**，产品能力通过插件组合并由单一 Host 统一承载

用户可以像搭积木一样创建内容，通过标签灵活组织，通过插件自由扩展，通过主题自定义外观。所有内容保存为文件，不依赖数据库，可以自由分享和备份。
