# task015 审查总清单

## 任务目标

本任务用于审查“箱子”和“基础卡片 / 复合卡片”在当前生态中的模块分布、处理链路、调用接口、运行时职责，以及两者是否真正共用同一条正式链路。

用户当前关注的核心判断是：

1. 箱子布局卡片全链路是否应该和基础卡片保持同构；
2. 两者是否已经收口到同一套 Host / Bridge / SDK / 应用壳层体系；
3. 当前实现里，哪些部分已经共链路，哪些部分仍然分成了两套能力面；
4. 文档口径和代码实现之间是否一致；
5. 如果要把箱子链路重构为与卡片同构，完整重构方案应该如何设计。

## 当前阶段性判断

截至当前文档口径层结论：

1. 箱子和卡片在内容模型上是并行概念，不是同一实体的两种外观；
2. 两者都要求由 Host 作为唯一运行时承载，通过 Bridge / SDK / 应用壳层消费；
3. 两者都采用“正式文档 iframe + Host 受控 URL”的总体模式；
4. 但在公共服务动作面、插件类型、运行时会话模型上，文档已经明确分为 `card.*` 和 `box.*` 两套动作；
5. 官方编辑引擎对卡片和箱子也没有走完全相同的编辑态实现：卡片是本地基础卡片运行时，箱子仍然偏向 Host `box-service` + SDK 编辑面板。

以上只是当前阶段性判断，最终结论以 Host / SDK / Viewer / EditingEngine / 插件代码核查完毕后统一收口。
