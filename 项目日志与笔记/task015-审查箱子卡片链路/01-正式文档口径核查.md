## 文档口径下的核心结论

## 1. 概念关系

文档口径非常明确：

1. 基础卡片是原子内容单元；
2. 复合卡片是 `.card` 文件，对基础卡片做组合；
3. 箱子是 `.box` 文件，是卡片或箱子的引用容器；
4. 箱子不是卡片的一种类型，也不是复合卡片的另一种皮肤；
5. 箱子与卡片是并行概念，但都属于 Host 正式文档消费体系的一部分。

这意味着，从内容建模角度看，文档并不支持“箱子就是另一种卡片文件”的表述；文档支持的是“它们应共用同一宿主架构原则”，而不是“它们在服务动作面和插件接口上必须完全合并为一套名义接口”。

## 2. 文件格式层

### 卡片

文档定义：

1. `.card` 是 ZIP Store；
2. 关键目录是 `.card/` 和 `content/`；
3. `.card/metadata.yaml`、`.card/structure.yaml`、`.card/cover.html` 是核心；
4. 基础卡片内容配置位于 `content/*.yaml`；
5. 资源原则上位于卡片根目录。

### 箱子

文档定义：

1. `.box` 也是 ZIP Store；
2. 关键目录是 `.box/` 和可选 `assets/`；
3. `.box/metadata.yaml`、`.box/structure.yaml`、`.box/content.yaml`、`.box/cover.html` 是核心；
4. `structure.yaml` 记录 URL 引用和摘要快照；
5. 箱子不保存被引用卡片实体。

### 文件格式层对比结论

两者相同点：

1. 都是单文件、ZIP Store、正式文档产物；
2. 都有独立的隐藏配置目录；
3. 都有 `metadata.yaml`；
4. 都有正式 `cover.html` 入口；
5. 都允许附属封面资源目录；
6. 都要求通过 Host 返回受控 URL 再交给 iframe 挂载。

两者不同点：

1. `.card` 存内容实体，`.box` 存引用和布局配置；
2. `.card` 通过 `content/*.yaml` 组织基础卡片，`.box` 通过 `structure.yaml + content.yaml` 组织条目与布局；
3. `.card` 的内部主角是基础卡片插件，`.box` 的内部主角是布局插件。

## 3. 运行时总原则

设计原稿和共用技术文档都在强调同一件事：

1. Host 是唯一运行时承载；
2. 插件不能自行扫描磁盘、直连系统能力；
3. 应用层通过 Bridge / SDK / Host 正式动作消费能力；
4. 查看器只是壳层，不能重写第二套卡片或箱子解析链路；
5. 主题、多语言、受控文档 iframe 都属于正式公共运行时的一部分。

从这一层看，箱子和卡片的确属于同一个大架构，不允许应用私自各写一条旁路。

## 4. 服务动作面是否完全共用

文档答案是否定的。

### 卡片动作面

正式文档定义了：

1. `card.pack`
2. `card.unpack`
3. `card.validate`
4. `card.readMetadata`
5. `card.readInfo`
6. `card.parse`
7. `card.open`
8. `card.render`
9. `card.renderCover`
10. `card.renderEditor`

### 箱子动作面

正式文档定义了：

1. `box.pack`
2. `box.unpack`
3. `box.inspect`
4. `box.validate`
5. `box.readMetadata`
6. `box.renderCover`
7. `box.openView`
8. `box.listEntries`
9. `box.readEntryDetail`
10. `box.renderEntryCover`
11. `box.openEntry`
12. `box.resolveEntryResource`
13. `box.readBoxAsset`
14. `box.prefetchEntries`
15. `box.closeView`
16. `box.renderLayoutFrame`
17. `box.renderLayoutEditor`

### 判断

文档已经把两者分成了两类正式动作：

1. 卡片偏向“解析 + 复合渲染 + 卡片编辑器”；
2. 箱子偏向“查看会话 + 布局查看 + 布局编辑 + 条目动态取数”。

所以如果要求“对外完全只保留一套同名接口”，当前正式文档并不是这个方向。

## 5. SDK 层是否有统一壳层

有，但只是统一壳层，不是统一底层动作。

文档明确存在统一文档窗口：

1. `client.document.window.render({ filePath })`
2. `.card` 时委托 `client.card.compositeWindow.render(...)`
3. `.box` 时委托 `client.box.documentWindow.render(...)`

这说明：

1. 应用壳层可以不自己区分卡片和箱子；
2. 但 SDK 内部仍然是两条正式子链路；
3. 它们是在“上层统一入口”收口，不是在“底层动作面”合并。

## 6. 封面链路

文档对卡片封面和箱子封面给出的口径是高度对齐的。

共同点：

1. 都是独立于主文档内容的正式链路；
2. 都以 `cover.html` 为唯一正式入口；
3. 都由 Host 返回受控 `coverUrl`；
4. 都由 SDK / 统一文档组件把 `coverUrl` 挂进 iframe；
5. 都不自动注入主题、多语言和复合壳层。

这是当前文档里最接近“完全同构”的一段链路。

## 7. 查看态链路

文档定义下：

### 卡片查看态

1. Host `card.render`
2. 按基础卡片类型分发插件
3. 生成基础卡片 iframe
4. 拼接复合卡片文档
5. SDK `client.card.compositeWindow.render(...)`
6. 查看器承载最终复合 iframe

### 箱子查看态

1. Host `box.openView`
2. Host 解析 `.box`、创建查看会话
3. Host 按 `layoutType` 选择布局插件
4. Host `box.renderLayoutFrame`
5. 布局插件通过 `box.*` 会话动作继续按需请求
6. SDK `client.box.documentWindow.render(...)`
7. 查看器承载最终布局 iframe

### 查看态判断

两者同构的部分：

1. Host 统一托管；
2. SDK 统一交付 iframe；
3. 查看器只做壳层；
4. 都依赖插件输出；
5. 都交付受控文档 URL。

两者非同构的部分：

1. 卡片是 Host 先把基础卡片组合成复合卡片文档；
2. 箱子是 Host 先建立箱子会话，再让布局插件按需向 Host 取数；
3. 卡片的主分发键是 `cardType`，箱子的主分发键是 `layoutType`。

## 8. 编辑态链路

文档已经明确：卡片和箱子在官方编辑引擎里并没有保持完全一致。

### 卡片编辑态

1. 官方编辑引擎是正式例外；
2. 复合卡片预览走本地基础卡片运行时；
3. 编辑器直接挂 `renderBasecardEditor`；
4. 不再把 Host `CompositeCardWindow + editorPanel` 作为官方编辑引擎主链路。

### 箱子编辑态

1. 箱子窗口预览仍沿 Host 文档链路；
2. 右侧面板切换为箱子编辑器；
3. 布局编辑器通过 `box.renderLayoutEditor -> client.box.editorPanel.render(...)`；
4. 箱子工作目录通过 `box.unpack/pack` 和文件服务维护。

### 编辑态判断

文档口径下，官方编辑引擎中：

1. 卡片编辑态是“本地运行时特例”；
2. 箱子编辑态仍然比较依赖 Host `box-service` 和 SDK 箱子编辑面板；
3. 所以“编辑态全链路完全一模一样”这个判断，至少在文档层面不成立。

## 9. 插件契约层

文档对两类插件都定义了“宿主挂载纯函数 + 描述符”的模式。

### 基础卡片插件

核心导出：

1. `renderBasecardView`
2. `renderBasecardEditor`
3. `basecardDefinition`

### 布局插件

核心导出：

1. `layoutDefinition.renderView`
2. `layoutDefinition.renderEditor`
3. `layoutDefinition.createDefaultConfig / normalizeConfig / validateConfig`

### 契约层判断

这两者在“导出形态”和“宿主挂载方式”上高度相似，但它们的上下文对象明显不同：

1. 基础卡片上下文围绕单卡配置、资源 URL、编辑器资源桥；
2. 布局插件上下文围绕箱子会话、条目分页、条目封面、条目打开和箱子资源桥。

所以它们是“相似设计模式”，不是“完全同一接口”。

## 10. 文档口径下的阶段性总判断

如果把“同一条链路”理解为：

1. 都必须经由 Host；
2. 都必须通过正式插件和受控文档 URL；
3. 都可以由 SDK 统一文档窗口在上层壳层收口；
4. 都不允许应用层私自重写解析和分发；

那么文档口径支持“箱子和卡片处于同一宿主架构体系”。

如果把“同一条链路”理解为：

1. 使用同一组 Host 动作；
2. 使用完全同名的运行时接口；
3. 编辑态和查看态都走一模一样的流程；
4. 插件上下文对象和契约完全相同；

那么文档口径并不支持，当前正式文档已经把它们拆成了两套子链路。
