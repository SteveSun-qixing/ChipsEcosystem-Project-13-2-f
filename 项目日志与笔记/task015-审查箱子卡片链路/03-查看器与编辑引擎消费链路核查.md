# 查看器与编辑引擎消费链路核查

## 核查范围

本轮只看两个官方消费方：

1. `Chips-CardViewer`
2. `Chips-EditingEngine`

目标是确认：

1. 卡片和箱子在应用层是否真正被当作同一种“文档窗口”消费；
2. 官方编辑引擎里卡片和箱子的查看态、预览态、编辑态到底有多大差异；
3. 用户关心的“箱子布局卡片全链路应该和基础卡片一模一样”在消费方层面是否成立。

## 已核查模块

### 查看器

1. `Chips-CardViewer/manifest.yaml`
2. `Chips-CardViewer/src/App.tsx`
3. `Chips-CardViewer/src/components/CardWindow.tsx`

### 编辑引擎

1. `Chips-EditingEngine/src/components/CardWindow/CardWindow.tsx`
2. `Chips-EditingEngine/src/components/BoxWindow/BoxPreviewSurface.tsx`
3. `Chips-EditingEngine/src/components/EditPanel/EditPanel.tsx`
4. `Chips-EditingEngine/src/components/EditPanel/BoxEditorPanel.tsx`
5. `Chips-EditingEngine/src/components/EditPanel/PluginHost.tsx`
6. `Chips-EditingEngine/src/editor-runtime/EditorHost.tsx`
7. `Chips-EditingEngine/src/basecard-runtime/CompositeCardAssembler.tsx`
8. `Chips-EditingEngine/src/basecard-runtime/frame-host.tsx`
9. `Chips-EditingEngine/src/basecard-runtime/registry.ts`
10. `Chips-EditingEngine/src/services/box-document-service.ts`

## 1. 查看器层结论最清晰

### `Chips-CardViewer` 已同时声明两种文件处理能力

`manifest.yaml` 中同时声明：

1. `file-handler:.card`
2. `file-handler:.box`

### 启动入口使用统一探测

`App.tsx` 的启动逻辑是：

1. 从 `launchParams.targetPath` 取目标文件；
2. 调用 `client.document.detectType(filePath)` 识别 `.card/.box`；
3. 只要识别成功，就继续进入统一文档窗口。

### 查看器主组件使用统一文档窗口

`CardWindow.tsx` 的主流程是：

1. `client.document.detectType(filePath)`
2. `client.document.window.render({ filePath, locale, mode: 'view' })`
3. 为 iframe 统一挂接：
   1. `onReady`
   2. `onError`
   3. `onResourceOpen`

### 这一层的判断

在 `Chips-CardViewer` 中：

1. 卡片与箱子已经被统一当作“正式文档窗口”处理；
2. 查看器自己不重新解析 `.card/.box`；
3. 它只依赖 `client.document.*` 这一层；
4. 所以在查看器壳层，用户的“应共用一条链路”是基本成立的。

## 2. 编辑引擎中的卡片查看态不是 Host 文档窗口主链路

### 卡片封面仍用正式 Host 封面链路

`CardWindow.tsx` 中卡片封面使用：

1. `client.card.coverFrame.render({ cardFile })`

这部分依然走 Host 正式封面能力。

### 卡片主预览改为本地基础卡片运行时

卡片主预览区域并没有直接使用 `client.document.window.render({ filePath })`，而是：

1. `CompositeCardAssembler`
2. `BasecardFrameHost`
3. `basecard-runtime/registry`

即：

1. 编辑引擎本地读取卡片结构数据；
2. 本地装配多个基础卡片节点；
3. 本地加载基础卡片插件描述符；
4. 每个基础卡片节点以本地 iframe 形式挂载。

### 本地基础卡片运行时的插件来源

`basecard-runtime/registry.ts` 会：

1. 内建注册若干基础卡片描述符；
2. 通过 `client.plugin.query({ type: 'card' })` 同步已安装卡片插件；
3. 直接按插件 entry 加载 `basecardDefinition`。

### 判断

这说明编辑引擎里的卡片预览主链路已经不是：

1. Host `card.render`
2. SDK `client.card.compositeWindow.render`

而是官方编辑引擎自己的本地基础卡片运行时。

## 3. 编辑引擎中的卡片编辑态也是本地插件编辑器

### 右侧编辑面板对卡片走 `PluginHost`

`EditPanel.tsx` 中，卡片分支走：

1. `PluginHost`
2. `EditorHost`

### `EditorHost` 的行为

`EditorHost.tsx` 中：

1. 通过 `getBasecardDescriptor(cardType)` 取得基础卡片描述符；
2. 在本地维护编辑会话和草稿配置；
3. 提供本地资源导入、删除、压缩包导入；
4. 直接调用描述符的编辑器渲染能力；
5. 再把变更回写到编辑引擎自己的卡片状态。

### 判断

这说明卡片编辑态在官方编辑引擎里并不是以 `client.card.editorPanel.render(...)` 作为主入口，而是本地基础卡片插件运行时。

## 4. 编辑引擎中的箱子预览仍走统一文档窗口

### 箱子预览区域

`BoxPreviewSurface.tsx` 直接调用：

1. `client.document.window.render({ filePath: session.boxFile, documentType: 'box', locale })`

并统一挂接：

1. `onReady`
2. `onError`

### 判断

这说明在编辑引擎里，箱子预览态仍然依赖 Host + SDK 正式文档链路，而没有像卡片那样改成本地布局运行时。

## 5. 编辑引擎中的箱子编辑态也依赖 Host 布局编辑器

### 右侧编辑面板的箱子分支

`EditPanel.tsx` 中，箱子分支走：

1. `BoxEditorPanel`

### `BoxEditorPanel` 的行为

核心流程是：

1. `client.box.editorPanel.render(...)`
2. `client.box.editorPanel.onChange(...)`
3. `client.box.normalizeLayoutConfig(...)`
4. `boxDocumentService.updateLayoutConfig(...)`

同时通过 `resources` 给箱子编辑器 iframe 提供：

1. `readBoxAsset`
2. `importBoxAsset`
3. `deleteBoxAsset`

### 判断

箱子编辑态依然高度依赖：

1. Host `box.renderLayoutEditor`
2. SDK `client.box.editorPanel.render`

并没有切换到像卡片那样的本地布局插件运行时。

## 6. 编辑引擎中的箱子文档服务进一步证明其仍依赖 Host

`box-document-service.ts` 是很关键的证据。

### 创建箱子

创建时会调用：

1. `box.readLayoutDescriptor`
2. `box.normalizeLayoutConfig`
3. `box.pack`

### 打开箱子

打开时会调用：

1. `box.inspect`
2. `box.unpack`

### 保存箱子

保存时会调用：

1. `box.pack`
2. `box.inspect`

### 导入条目

导入箱子条目时会调用：

1. `box.readMetadata`

导入卡片条目时会调用：

1. `card.readInfo`

### 判断

编辑引擎内部虽然维护了箱子工作目录、自动保存和本地会话快照，但这些都是围绕 Host 正式 box/card 接口搭建的，不是另一套脱离 Host 的箱子解析链路。

## 7. 查看器与编辑引擎的对比结论

### 查看器层

1. 卡片和箱子高度统一；
2. 同一个 app；
3. 同一个 `client.document.window.render(...)`；
4. 同一套 iframe 事件处理模式。

### 编辑引擎层

卡片：

1. 封面走 Host 正式链路；
2. 主预览走本地基础卡片运行时；
3. 右侧编辑器走本地基础卡片插件编辑器。

箱子：

1. 预览走 `client.document.window.render(...box...)`；
2. 右侧编辑器走 `client.box.editorPanel.render(...)`；
3. 工作目录和保存流程仍持续调用 Host `box.* / card.*` 正式接口。

### 核心判断

在编辑引擎里：

1. 卡片和箱子没有保持完全相同的查看态与编辑态实现；
2. 卡片更偏向“本地基础卡片运行时”；
3. 箱子更偏向“Host box-service + SDK box API”；
4. 因此“编辑引擎里两者全链路一模一样”这一说法在当前实现中不成立。

## 8. 对用户核心问题的阶段性回答

如果只看官方查看器：

1. 可以认为卡片和箱子已经共享同一个文档窗口壳层。

如果把官方编辑引擎也纳入“全链路”：

1. 当前实现并不是完全同一条链路；
2. 尤其卡片编辑态与箱子编辑态差异很大；
3. 这不是只有文件格式不同那么简单，至少在消费方实现上已经出现显著分叉。
