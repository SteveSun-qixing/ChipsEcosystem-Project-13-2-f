# 插件契约与接口对比

## 核查范围

本轮聚焦插件层，确认：

1. 基础卡片插件和箱子布局插件是否本质上是一种插件；
2. 两者是否已经共享同一套运行时接口；
3. 它们只是“两个文件格式”不同，还是“插件上下文和宿主协作协议”也存在差异。

## 1. 基础卡片插件的正式导出形态

基础卡片插件当前普遍导出三层能力：

1. `renderBasecardView`
2. `renderBasecardEditor`
3. `basecardDefinition`

其中 `basecardDefinition` 一般包含：

1. `pluginId`
2. `cardType`
3. `displayName`
4. `description`
5. `icon`
6. `aliases`
7. `commitDebounceMs`
8. `createInitialConfig`
9. `normalizeConfig`
10. `validateConfig`
11. `collectResourcePaths`
12. `renderView`
13. `renderEditor`

### 基础卡片查看态上下文

基础卡片查看态收到的上下文普遍围绕单个基础卡片节点：

1. `container`
2. `config`
3. `themeCssText`
4. `resolveResourceUrl`
5. `releaseResourceUrl`
6. `openResource`

### 基础卡片编辑态上下文

基础卡片编辑态收到的上下文普遍围绕单个基础卡片配置编辑：

1. `container`
2. `initialConfig`
3. `onChange`
4. `resolveResourceUrl`
5. `releaseResourceUrl`
6. `importResource`
7. `importArchiveBundle`
8. `deleteResource`

## 2. 箱子布局插件的正式导出形态

箱子布局插件当前导出核心是：

1. `layoutDefinition`

1. `pluginId`

## 3. 两类插件的相同点

### 设计模式相似

两者都体现为：

1. 宿主提供一个 DOM 容器；
2. 插件导出一个描述符对象；
3. 描述符对象负责默认配置、规范化、校验、查看态和编辑态；
4. 实际渲染都发生在受控容器中；
5. 资源访问都不允许插件直接越过宿主。

### 生命周期形态相似

两者的 `renderView / renderEditor` 都是：

1. 由宿主挂载；
2. 返回清理函数或无返回；
3. 由宿主控制销毁。

### 结论

从软件工程设计模式上看，基础卡片插件和布局插件是同一种“宿主挂载式前端插件”。

## 4. 两类插件的关键差异

### 差异一：实体粒度不同

基础卡片插件面向：

1. 单个基础卡片节点

布局插件面向：

1. 整个箱子查看 / 编辑面

### 差异二：主标识键不同

基础卡片插件按：

1. `cardType`

布局插件按：

1. `layoutType`

### 差异三：查看态上下文不同

基础卡片查看态核心是：

1. 单卡配置
2. 主题样式
3. 单卡资源解析

布局查看态核心是：

1. `BoxSessionInfo`
2. `initialView`
3. `runtime`
4. 条目分页和动态取数

### 差异四：编辑态资源桥不同

基础卡片编辑器关心：

1. 卡片根目录资源
2. 导入普通资源
3. 导入压缩包网页资源

布局编辑器关心：

1. 箱子资产
2. 读取、导入、删除布局资源
3. 条目列表和布局配置

### 差异五：宿主分发逻辑不同

基础卡片插件由 Host `card-service` 基于 `cardType` 与插件 capability 分发。

布局插件由 Host `box-service` 基于 `layoutType` 与 layout 插件清单分发。

## 5. “只是两种文件格式”这句话是否成立

### 在文件容器层面

可以说两者确实都只是：

1. 不同的文档格式
2. 不同的内部 YAML 结构
3. 不同的内容组织方式

### 在插件运行时层面

不能简单说“只差文件格式”。

因为插件运行时至少还差：

1. 宿主分发键
2. 运行时上下文对象
3. 是否存在 session
4. 是否支持条目分页与按需取数
5. 编辑器桥接能力

## 6. 本轮插件层结论

### 成立的说法

1. 两类插件采用了非常相似的宿主挂载式架构模式；
2. 两类插件都由 Host 生成文档并交给 iframe；
3. 两类插件都依赖配置规范化、配置校验、查看态和编辑态描述符。

### 不成立的说法

1. 两类插件不是完全同一份接口；
2. `basecardDefinition` 与 `layoutDefinition` 不是同名、同参、同语义的统一契约；
3. 箱子布局插件并不是把基础卡片插件简单换个文件扩展名那么直接。

## 7. 对用户核心判断的支撑

如果用户想表达的是：

1. 两者都该属于同一个宿主插件体系；
2. 不该由应用私自写两套旁路；
3. 宿主要以一致的方式挂载插件；

那么当前实现基本符合。

如果用户想表达的是：

1. 箱子布局插件应当直接复用基础卡片插件同一份上下文和接口；
2. 两者只允许有文件解析差异，不允许运行时协议差异；

那么当前实现并不符合，现状依然是两套相似但不同的插件契约。
