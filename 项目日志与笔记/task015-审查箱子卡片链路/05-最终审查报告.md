# 箱子卡片链路最终审查报告

## 任务目标

本报告用于回答以下问题：

1. 与卡片和箱子相关的所有主要模块有哪些；
2. 当前生态中的处理管线链路是什么；
3. 相关调用接口有哪些；
4. 当前架构设计到底是怎样分层的；
5. 箱子布局卡片与基础卡片在当前实现里哪些地方一样，哪些地方不一样；
6. 用户提出的“箱子布局卡片全链路应该和基础卡片一模一样，并且是共用同一条链路”在当前代码与正式文档口径下是否成立。

## 三、处理管线链路

## 1. `.card` 打开与查看主链路

1. 桌面打开 `.card`
2. Host `file-association.ts`
3. Host `card.open`
4. `CardOpenService` 查找 `file-handler:.card`
5. `Chips-CardViewer` 被拉起
6. 查看器 `App.tsx` 读取 `targetPath`
7. 查看器 `client.document.detectType(...)` 识别为 `card`
8. 查看器 `client.document.window.render({ filePath })`
9. SDK 转调 `client.card.compositeWindow.render(...)`
10. SDK 转调 Host `card.render`
11. Host `card-service` 解包卡片、按 `cardType` 分发基础卡片插件、生成复合文档
12. Host 返回 `documentUrl`
13. SDK 创建 iframe
14. 查看器承载 iframe

## 2. `.box` 打开与查看主链路

1. 桌面打开 `.box`
2. Host `file-association.ts`
3. Host `box.inspect`
4. Host 查找 `file-handler:.box`
5. `Chips-CardViewer` 被拉起
6. 查看器 `App.tsx` 读取 `targetPath`
7. 查看器 `client.document.detectType(...)` 识别为 `box`
8. 查看器 `client.document.window.render({ filePath })`
9. SDK 转调 `client.box.documentWindow.render(...)`
10. SDK 顺序执行：
    1. `box.inspect`
    2. `box.readLayoutDescriptor`
    3. `box.normalizeLayoutConfig`
    4. `box.getLayoutInitialQuery`
    5. `box.openView`
    6. `box.renderLayoutFrame`
11. Host `box-service` 建立查看 session
12. Host 根据 `layoutType` 分发布局插件
13. Host 返回布局 `documentUrl`
14. SDK 创建 iframe 并附加 `chips.box-layout` runtime bridge
15. 布局插件在 iframe 内按需向 Host 拉取条目、封面、资源
16. 查看器承载 iframe

## 3. 卡片封面链路

1. 应用调用 `client.card.coverFrame.render({ cardFile })`
2. SDK 调 Host `card.renderCover`
3. Host 直接返回 `.card/cover.html`
4. SDK 创建 cover iframe

## 4. 箱子封面与箱子条目封面链路

### 箱子本体封面

1. 应用或宿主调用 `box.renderCover`
2. Host 返回 `.box/cover.html`

### 箱子条目封面

1. 布局插件或应用调用 `box.renderEntryCover`
2. Host 判断条目目标
3. 若目标是 box，复用 `box.renderCover`
4. 若目标是 card，复用 `card.readInfo(...cover...)`

## 5. 卡片编辑态链路

### Host / SDK 正式链路

1. `client.card.editorPanel.render(...)`
2. Host `card.renderEditor`
3. Host 基于 `cardType` 分发基础卡片插件编辑器

### 官方编辑引擎主链路

当前官方编辑引擎实际上并不以这一条为主，而是：

1. 本地 `EditorHost`
2. 本地基础卡片注册表
3. 本地资源导入 / 删除
4. 本地插件编辑器挂载

## 6. 箱子编辑态链路

### 预览

1. `BoxPreviewSurface`
2. `client.document.window.render(...box...)`
3. 仍走 Host 正式箱子查看链路

### 编辑

1. `BoxEditorPanel`
2. `client.box.editorPanel.render(...)`
3. `client.box.normalizeLayoutConfig(...)`
4. `boxDocumentService.updateLayoutConfig(...)`

### 文件读写与保存

1. `box-document-service` 调 `box.inspect`
2. `box.unpack`
3. `box.pack`
4. `box.readLayoutDescriptor`
5. `box.normalizeLayoutConfig`

## 五、相关架构设计结论

## 1. 大架构层面是统一的

当前文档与代码都支持以下统一原则：

1. Host 是唯一运行时承载；
2. 应用不能自己重新实现 Host 正式解析链路；
3. 对外能力通过 Bridge / SDK 暴露；
4. 查看器作为壳层承载正式文档 iframe；
5. 插件由 Host 受控分发；
6. 卡片和箱子都属于同一个正式文档体系。

## 2. 服务域层面是拆分的

虽然属于同一文档体系，但当前 Host 与 SDK 的正式能力域明确拆成：

1. `card.*`
2. `box.*`

这说明架构不是“只有一条内部动作链路”，而是“同一宿主体系下的两个正式服务域”。

## 3. 查看态模型并不相同

### 卡片

1. 先把复合卡片完整拼出来
2. 再把成品文档交给 iframe

### 箱子

1. 先建立查看 session
2. 再把布局插件挂进去
3. 布局插件通过 runtime bridge 按需请求条目数据

这两个查看态模型都合规，但并不是同一种内部处理方式。

## 4. 编辑态模型差异更大

### 卡片

官方编辑引擎主链路已经本地化：

1. 本地基础卡片运行时
2. 本地插件编辑器
3. 本地资源操作

### 箱子

官方编辑引擎仍然重度依赖：

1. Host `box-service`
2. SDK `box.documentWindow`
3. SDK `box.editorPanel`

这说明当前“全链路一致性”最大的分叉点就在编辑引擎。

## 5. 插件模式相似，但契约不同

### 相似处

1. 都是宿主挂载式插件；
2. 都有默认配置、规范化、校验、查看态和编辑态；
3. 都由 Host 生成文档或容器交给 iframe。

### 不同处

1. 基础卡片插件围绕单节点内容；
2. 布局插件围绕整个箱子查看 session；
3. 上下文对象、资源桥和 runtime 能力都不一样。

## 六、卡片与箱子对比结论

## 1. 相同点

1. 都是正式文档文件；
2. 都由 Host 承载；
3. 都通过 Bridge / SDK 对外；
4. 都可以在查看器层由同一个 `client.document.window.render(...)` 收口；
5. 都使用受控 iframe 文档 URL；
6. 都使用宿主分发插件；
7. 箱子条目对卡片的打开、信息、封面已复用卡片正式能力。

## 2. 不同点

1. 文件内容模型不同：
   1. `.card` 存内容实体
   2. `.box` 存引用和布局
2. Host 动作域不同：
   1. `card.*`
   2. `box.*`
3. 查看态模型不同：
   1. 卡片是预组合复合文档
   2. 箱子是 session + 按需取数
4. 插件类型不同：
   1. `basecardDefinition`
   2. `layoutDefinition`
5. 官方编辑引擎主链路不同：
   1. 卡片本地运行时
   2. 箱子 Host/SDK 正式链路

## 七、对用户核心判断的最终回答

## 1. 哪些层面可以说“应该一样”

如果用户说的“应该一样”是指：

1. 都必须在同一个 Host 架构下运行；
2. 都必须通过 Bridge / SDK 暴露；
3. 都必须通过正式 viewer / editor 壳层消费；
4. 应用层最好只用统一文档窗口入口；
5. 箱子内部卡片条目应复用卡片正式打开和封面能力；

那么当前生态大体上就是这个方向，而且很多部分已经做到。

## 2. 哪些层面当前并不是“一模一样”

如果用户说的“全链路一模一样”是指：

1. 使用完全同名的一组 Host 动作；
2. 查看态内部流程完全相同；
3. 编辑态内部流程完全相同；
4. 插件上下文完全相同；
5. 只有文件格式不同，其他都不该不同；

那么当前代码与正式文档都不支持这个判断。

当前真实状态更准确的表述是：

1. 卡片和箱子处于同一宿主文档架构；
2. 查看器壳层已经基本统一；
3. 箱子条目级能力已复用卡片正式能力；
4. 但 Host 服务域、查看态 session 模型、插件契约、编辑引擎主链路仍然是两套相似但不同的正式子链路。

## 3. 最终审查结论

### 结论一句话版

当前生态不是“箱子和基础卡片完全共用同一条内部链路”，而是“同一宿主架构下，查看器壳层统一、条目能力部分复用、底层服务域与编辑态实现仍分叉”的状态。

### 更精确的结论

1. 用户所说的“共用一条链路”在应用壳层、Host 承载原则、SDK 统一文档入口、箱子条目复用卡片能力这些层面已经部分成立；
2. 但在 Host 动作面、查看 session 模型、插件契约、编辑引擎卡片/箱子主实现这些关键层面并不成立；
3. 因此当前不能把“箱子布局卡片”和“基础卡片”描述为仅仅两种文件格式不同、其余链路完全相同。
