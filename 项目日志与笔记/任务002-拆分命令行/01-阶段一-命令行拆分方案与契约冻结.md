# 任务002-阶段一：命令行拆分方案与契约冻结

## 1. 背景

现状（在本阶段开始时）：

- 设计与共用文档中采用 `chips ...` / `chips dev ...` 的命令结构；
- 实际代码中仅实现了 Host 侧的 CLI（`Chips-Host/src/main/cli/index.ts`），开发者 CLI 尚未落地；
- 多个仓库（脚手架、主题包、CardViewer 等）的 package.json 与文档中引用了 `chips dev`。

最新决策要求：

- 将命令行彻底拆成两个独立命令：`chips` 与 `chipsdev`；
- `chips` 只负责 Host 相关操作；`chipsdev` 只负责开发工具链；
- 不保留 `chips dev` 形式，不做兼容层。

本阶段目标是在动代码之前，先把“新世界”的命令行结构与对外契约一次性设计清楚并冻结。

## 2. 目标

1. 明确命令行整体结构：
   - `chips`：Host CLI，具体命令命名、参数、返回结构；
   - `chipsdev`：开发者 CLI，命令清单、参数、返回结构。
2. 确定命令行与仓库/发布包的归属关系：
   - `chips` 随哪一个包/安装器发布（Chips-Host 自带，还是独立 npm 包壳层）；
   - `chipsdev` 在 `Chips-SDK` 中的具体实现与 npm 包形式（例如：`@chips/dev-cli`，导出 `chipsdev`）。
3. 冻结关键契约：
   - 配置文件命名（保留 `chips.config.mjs` 还是引入 `chipsdev.config.mjs` 等）；
   - 环境变量（如 `CHIPS_HOME` / 开发工作区路径相关变量）；
   - 基础输出格式（尤其是脚本/CI 将消费的 JSON 输出）。
4. 形成一份统一的“命令行拆分设计说明”，供后续阶段实现与文档改造使用。

## 3. 范围

### 3.1 在范围内

- 规范层设计：
  - `chips` 支持的子命令清单（start/stop/status/config/logs/plugin/theme/update/doctor/open/...）；
  - `chipsdev` 支持的子命令清单（init/create/server/debug/build/test/lint/e2e/package/validate/login/publish/version/...）；
  - 各命令的参数约定、退出码语义、错误分类（用户输入错误 / 系统错误 / 业务错误等）。
- 仓库归属与安装路径：
  - `chips` 与 `Chips-Host` 的关系；
  - `chipsdev` 与 `Chips-SDK` 的关系；
  - 是否需要为 `chipsdev` 单独建仓库（例如 `chips-dev-cli`）以及与 `Chips-SDK` 的耦合方式。
- 同名/相近名文件的处理策略：
  - 保留 `chips.config.mjs` 作为“开发构建配置”，由 `chipsdev` 消费；
  - Host 侧配置文件命名与存放位置（如 `~/.chips-host` 下的 JSON）。

### 3.2 不在范围内

- 具体 TypeScript/Node 实现细节；
- 安装器或安装 UI 的交互设计；
- 非 CLI 接口（如未来可能的本地 HTTP/IPC 服务）。

## 4. 产出物

1. 《命令行拆分设计说明》（建议落在 `生态共用技术文档/开发与运维/XX-命令行拆分设计说明.md`，具体编号在后续阶段创建）：
   - 描述 `chips` / `chipsdev` 的职责边界；
   - 命令清单与参数表；
   - 输出格式与错误语义约定（至少列出 JSON 输出结构示例与常见错误码）。
2. 对现有两份命令行手册的改造草案（结构级）：
   - 《01-Chips-Host命令行使用手册》：确认只覆盖 `chips`（Host CLI），不提及 `chipsdev`；
   - 《02-Chips-Dev开发者命令行手册》：确认改造为 `chipsdev` 手册，不再使用 `chips dev` 形式。
3. 阶段二〜六的接口依赖清单：
   - Host 侧 CLI 实现必须满足的接口约束；

   - 生态仓库适配时需要依赖的 CLI 功能列表（例如脚手架需要的 `create`/`build`/`validate`）。

## 5. 任务拆解

### 5.1 现状调研（只读）

- 列出现有文档中所有与命令行相关的正式规范：
  - `生态共用技术文档/命令行工具/01-Chips-Host命令行使用手册.md`
  - `生态共用技术文档/命令行工具/02-Chips-Dev开发者命令行手册.md`
  - `生态共用技术文档/协议与接口标准` 中与 CLI 相关的条目
  - `生态设计原稿` / 白皮书中关于 SDK 与命令行工具的描述
- 列出现有代码中使用 CLI 的所有入口：
  - 各仓库 package.json 脚本中 `chips dev` 的使用；
  - 测试代码或脚本中直接调用 `chips dev` / `chips` 的位置。

> 输出：一份“CLI 使用清单”，按仓库 + 文件路径罗列所有现有命令行调用，为后续改造提供输入。

### 5.2 拆分方案设计

- 明确定义：
  - `chips` 仅暴露 Host 相关命令，不包含任何开发构建能力；
  - `chipsdev` 仅暴露开发相关命令，不负责启动/停止 Host 或操作插件运行时。
- 为 `chipsdev` 定义命令集合：
  - 项目生命周期：`init` / `create` / `migrate`（如需要）；
  - 开发调试：`server` / `debug`；
  - 质量门禁：`test` / `lint` / `e2e` / `validate`；
  - 构建与打包：`build` / `package`；
  - 发布与版本：`login` / `publish` / `version`；
  - 其他：如 `doctor` / `config` 等（需要时新增）。
- 约定 CLI 输出格式：
  - 控制台人类可读输出；
  - 机读模式（例如 `--json` 参数），用于 CI 与脚本集成；
  - 基本错误码（0/非 0）及分类说明。

### 5.3 配置与工作区策略

- 确认 Host 侧工作区与配置：
  - 默认工作区路径（如 `~/.chips-host`），由 `CHIPS_HOME` 环境变量覆盖；
  - Host CLI 的状态/配置文件结构（例如 `host-state.json` / `plugins.json`）。
- 确认开发侧工作区与配置：
  - `chipsdev` 是否需要自己的全局配置（例如 `~/.chipsdev/config.json`）；
  - 项目级配置保持使用 `chips.config.mjs`，由 `chipsdev` 解析；
  - 是否需要约定额外的配置文件（如模板索引、脚手架配置等）。

### 5.4 契约文档编写

- 起草《命令行拆分设计说明》初稿，包含：
  - 总体架构图（Host / SDK / CLI / 脚手架 / 主题包之间关系）；
  - 命令清单与参数表；
  - 输出示例与错误模型；
  - 与现有设计文档的对应关系（哪些章节将被更新）。
- 与 Host / SDK / 脚手架相关需求文档对齐，确保后续阶段不会出现接口理解偏差。

## 6. 验收标准

- 已完成现状调研，获得一份覆盖所有仓库与正式文档的 CLI 使用清单；
- 已形成并评审通过的《命令行拆分设计说明》，包含：
  - `chips` / `chipsdev` 的职责边界；
  - 命令与参数清单；
  - 输出与错误契约；
  - 配置与工作区策略；
- 已在相关需求与设计文档中引用该说明，作为后续实现和文档改造的依据；
- 阶段二〜六的任务均可以在不修改本阶段结论的前提下推进。
