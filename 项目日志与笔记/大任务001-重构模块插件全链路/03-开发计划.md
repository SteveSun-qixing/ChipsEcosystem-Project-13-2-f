# 大任务001：重构模块插件全链路开发计划

## 1. 目标

按冻结后的新口径完成模块插件全链路重构，使模块插件最终可以在整个生态中正常创建、安装、注册、调用、组合和卸载。

## 2. 受影响范围

- `Chips-Host`
- `Chips-SDK`
- `Chips-Scaffold/chips-scaffold-module`
- 相关官方参考模块仓库（后续按实施阶段创建）
- 生态设计原稿
- 生态共用技术文档

## 3. 阶段划分

## 阶段一：文档冻结

### 目标

- 冻结模块插件正式定位；
- 冻结模块能力契约；
- 冻结 Host / SDK / 脚手架 / Manifest 统一口径。

### 交付物

- 项目级需求文档；
- 项目级技术文档；
- 项目级开发计划；
- 生态设计原稿更新；
- 生态共用技术文档更新。

### 完成标准

- 生态公共文档中不再把模块插件定义为 UI 挂载模块；
- 模块对外调用模型统一为 capability + method；
- 不再保留旧外部正式模块协议。

## 阶段二：Host 模块服务重构

### 目标

在 `Chips-Host` 中建立新的模块注册、解析和调用体系。

### 主要任务

- 设计并实现模块注册表；
- 实现 provider 注册、更新、移除链路；
- 实现 `module.listProviders / resolve / invoke / job.get / job.cancel`；
- 实现模块 runtime 激活与回收；
- 实现模块调用审计和错误归一；
- 清理旧模块公开外部协议。

### 验证

- Host build；
- Host unit / integration / contract tests；
- 模块调用成功、失败、取消路径测试。

## 阶段三：SDK 模块接口重构

### 目标

让 `chips-sdk` 成为唯一正式模块调用入口。

### 主要任务

- 重写 `src/api/module.ts`；
- 新增 provider 查询、能力解析、方法调用、job 管理类型；
- 删除或归档旧 UI 插槽式模块 API；
- 补齐 SDK 单测和 CLI 相关联动。

### 验证

- SDK tests；
- 模块接口类型校验；
- Host 联调测试。

## 阶段四：模块脚手架重做

### 目标

把 `chips-scaffold-module` 从 React UI 模板重做为 headless provider 模板。

### 主要任务

- 重写模板目录；
- 重写模板渲染逻辑；
- 重写 README 和模板说明；
- 更新 `chipsdev create module` 参数推导；
- 增加 contract 目录和测试骨架；
- 删除 UI 挂载式模板输出。

### 验证

- `build`
- `test`
- `test:templates`
- `test:e2e`
- 生成工程 `lint/build/test/validate/package`

## 阶段五：参考模块落地

### 目标

构建若干正式参考模块，验证新链路可用。

### 主要任务

- 选择至少两到三个代表性能力模块作为参考实现；
- 覆盖同步方法和异步 job 方法两类能力；
- 至少包含一次模块间调用验证；
- 输出对应的项目级文档和测试。

### 验证

- Host 可以发现并调用参考模块；
- 模块间调用链路通过；
- job 管理链路通过。

## 阶段六：业务接入

### 目标

让生态中的正式调用方接入新模块链路。

### 主要任务

- 将 Host 或相关官方应用迁移到新模块调用接口；
- 移除对旧模块 UI 口径的依赖；
- 验证应用插件对 capability 的声明和调用流程。

### 验证

- 官方应用能够通过 SDK 成功调用模块能力；
- 不再需要私有模块装载器；
- 不再依赖旧 `slot + mount` 公开口径。

## 阶段七：验收收口

### 目标

完成跨仓一致性收口和正式验收。

### 主要任务

- 文档、代码、测试、脚手架统一复核；
- 梳理剩余缺口并补录工单；
- 输出最终验收结论。

### 验证

- 文档与实现一致；
- 全链路测试通过；
- 模块插件从创建到使用闭环成立。

## 4. 风险与前置条件

### 4.1 关键风险

- Host 内部当前仍有第二套“module”命名（Kernel module loader），需要在实施期明确分名；
- 旧模块脚手架和旧文档口径散布范围较大，实施时容易遗漏；
- 模块调用与 job 模型一旦收口不稳，会反复影响 Host、SDK 和脚手架。

### 4.2 前置条件

- 文档口径先冻结；
- 用户确认新模块插件正式方向；
- 后续实施按阶段推进，不夹带兼容方案。

## 5. 验证基线

后续每一阶段至少应按改动范围运行对应正式脚本：

- `Chips-Host`
  - `npm run build`
  - `npm test`
  - 涉及公共契约时补 `npm run test:contract`
- `Chips-SDK`
  - `npm test`
- `Chips-Scaffold/chips-scaffold-module`
  - `npm run build`
  - `npm test`
  - `npm run test:templates`
  - `npm run test:e2e`

## 6. 阶段完成判定

只有当某一阶段的文档、代码、测试和验证全部收口后，才进入下一阶段。不得为赶进度跳过文档和测试，也不得先实现再事后补正式契约。
