## 3. 总体路线

### 3.1 核心原则

- 先冻结实施边界，再进入大规模编码；
- Host 先收口正式动作面、会话模型、资源句柄和 `.box` 入口；
- SDK 必须以 Host 正式动作面为唯一来源，不得继续保留旧箱子类型；
- 网格布局插件必须是首个正式布局插件，不得做演示版或只读占位版；
- 查看器与编辑引擎必须复用同一套正式布局插件导出契约，不得各写一套临时宿主；
- 不做兼容设计，不保留旧模型分支，不保留回退链路。

### 3.2 主关键路径

主关键路径如下：

- 阶段一 -> 阶段二 -> 阶段三 -> 阶段四 -> 阶段五 -> 阶段六 / 阶段七 -> 阶段八 -> 阶段九

说明：

- 阶段二是整个实现的主阻塞点，因为 Host 必须先成为箱子运行时中心；
- 阶段四负责先补齐布局插件脚手架工程基线，是阶段五的直接前置；
- 阶段五在阶段四完成后交付首个正式网格布局插件；
- 阶段六与阶段七都依赖阶段二、阶段三、阶段四、阶段五，但两者可并行推进；
- 阶段八只有在查看器和编辑引擎都已切到正式链路后才能进入；
- 阶段九负责最终收口，不允许留下旧模型活跃路径。

### 3.3 为什么这样排

- 当前真实代码中，Host 还没有箱子查看会话模型；
- SDK 仍暴露旧 `full/shell/mixed` 模型；
- 查看器仍是纯卡片应用；
- 编辑引擎仍创建目录态 `.box`；
- 布局插件仓还是空仓；
- 布局插件脚手架仓当前也不存在。

因此必须先建立 Host 真正的箱子运行时，再补齐布局插件脚手架工程基线，之后让 SDK、布局插件、查看器、编辑引擎全部围绕同一套正式契约收口。

### 3.4 动态推进原则

- 本任务进入正式连续实施阶段后，发现新的实现细节、依赖缺口或阶段顺序优化点时，优先直接补充和修订本任务计划文档；
- 不再把阶段内实现问题拆成新的阻断式工单，除非用户后续明确要求恢复工单流；
- 每次推进都要先核对相关代码、`manifest.yaml`、测试与正式链路，再回写计划；
- 若某一问题只影响当前阶段，则收口到当前阶段文档；若影响跨阶段依赖，则同步更新总计划与阶段索引。

## 6. 阶段完成判定

只有当某一阶段的文档、代码、测试与正式脚本验证全部收口，才能进入下一阶段。不得出现以下行为：

1. 跳过共享契约直接改应用仓；
2. 为了先跑通而保留旧字段、旧测试、旧路由；
3. 在查看器或编辑引擎中私建一套 Host 旁路逻辑；
4. 通过手工 patch 数据规避正式 `.box` 保存与打开链路；
5. 用 README 或临时说明代替正式文档、测试与门禁。
