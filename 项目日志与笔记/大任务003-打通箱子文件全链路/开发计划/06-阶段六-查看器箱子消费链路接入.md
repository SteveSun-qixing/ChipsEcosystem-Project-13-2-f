# 阶段六：查看器箱子消费链路接入

## 1. 阶段目标

让 `Chips-CardViewer` 从“纯卡片查看器”升级为“卡片 + 箱子查看器”，可以正式接收 `.box` 文件启动上下文、打开箱子会话、加载对应布局插件，并把箱子页面呈现给用户。

## 2. 入口条件

- 阶段二、阶段三、阶段四、阶段五已完成；
- Host 已经能正式打开箱子会话；
- 网格布局插件与共享宿主层已可用。

## 3. 涉及仓库

| 仓库 | 本阶段职责 |
|---|---|
| `Chips-CardViewer` | 接入 `.box` 查看模式与布局插件渲染 |
| `Chips-Host` | 如需微调 launchContext 或窗口承载契约，做最小闭环修正 |

## 4. 重点交付物

- 查看器双模式入口：`card` / `box`；
- `.box` 文件处理能力与清单更新；
- 箱子查看会话生命周期管理；
- 布局插件渲染容器与错误边界；
- 查看器测试和内部文档更新。

## 5. 任务拆解

### 5.1 改造应用入口与清单

需要更新查看器：

- `manifest.yaml` 中的文件处理能力；
- `.box` 查看所需权限；
- 启动上下文解析逻辑；
- 应用标题、空态、错误态多语言文案。

要求：

- `.card` 与 `.box` 两种入口共存，但都走正式 Host/SDK 链路；
- 不得为 `.box` 再写一套独立启动协议。

### 5.2 建立查看器箱子模式控制器

查看器内部需要新增箱子模式控制器，负责：

1. 识别启动目标是 `.card` 还是 `.box`；
2. 对 `.box` 调用 `client.box.openView`；
3. 保存 `sessionId`、`box`、`initialView`；
4. 在窗口卸载、目标切换、异常退出时调用 `client.box.closeView`；
5. 监听必要的会话级事件并刷新界面。

### 5.3 装载布局插件页面

查看器需要接入阶段五交付的共享布局宿主层：

1. 根据 `activeLayoutType` 解析布局插件；
2. 把容器、会话信息、初始摘要页、布局配置和 runtime 注入 `renderView`；
3. 提供正式错误边界和布局切换重挂载能力；
4. 确保主题与多语言事件仍能传达到布局页面。

### 5.4 补齐查看态体验

至少补齐以下正式体验：

1. 箱子加载中状态；
2. 箱子打开失败、布局插件缺失、资源缺失等错误提示；
3. 空箱子状态；
4. 布局切换后重渲染；
5. 关闭窗口或切换目标时正确释放旧会话。

### 5.5 回归卡片查看模式

查看器引入箱子模式后，必须确保现有卡片查看链路不被破坏：

- `.card` 打开路径保持工作；
- 主题、多语言与卡片错误态保持一致；
- 不得因为新增 `.box` 模式而引入卡片查看旁路逻辑。

## 6. 串并行安排

本阶段内部可受控并行：

- 一路改造 manifest 与启动上下文；
- 一路实现箱子模式状态机；
- 一路接布局插件宿主层与视图容器；
- 一路补测试与文案。

## 7. 验证门禁

- `cd Chips-CardViewer && npm run build`
- `cd Chips-CardViewer && npm run test`
- `cd Chips-CardViewer && npm run lint`
- `cd Chips-CardViewer && npm run validate`

同时必须验证：

- 查看器能从 `.box` 启动；
- `openView -> renderView -> closeView` 生命周期完整；
- `.card` 原链路无回归；
- 未出现查看器自行解析箱子文件、直接扫描磁盘或直接请求网络的旁路行为。

## 8. 阶段退出标准

- 查看器已经成为正式 `.box` 消费方；
- 用户可以通过查看器打开并浏览网格箱子页面；
- 若查看器仍只能靠手工塞假数据演示布局页面，则本阶段不算完成。
