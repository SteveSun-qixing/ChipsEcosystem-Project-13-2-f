# 阶段九：回归收口与发布级文档清理

## 1. 阶段目标

在全链路跑通后，完成所有旧口径清理、文档回写、门禁复验和发布级收口，确保箱子能力不是“能跑一次”的项目状态，而是正式可维护、可扩展、可持续演进的生态基础能力。

## 2. 入口条件

- 阶段八已完成；
- 端到端样例与自动化验证已经通过；
- 不存在阻塞上线的未闭环公共缺口工单。

## 3. 涉及仓库

| 仓库 / 目录 | 本阶段职责 |
|---|---|
| `Chips-Host` | 清理旧箱子活跃路径与文档漂移 |
| `Chips-SDK` | 删除旧类型与旧示例 |
| `Chips-BoxLayoutPlugin` | 完成 README、测试与发布级说明 |
| `Chips-CardViewer` | 清理纯卡片假设中的箱子冲突点 |
| `Chips-EditingEngine` | 清理目录态 `.box` 假设与旧保存逻辑 |
| `生态共用技术文档` | 在重新核对代码后修正文档漂移 |
| `项目日志与笔记` | 回写工单状态与联调结论 |

## 4. 重点交付物

- 旧箱子口径清退结果；
- 各仓最新 README / 技术文档 / 需求文档 / 开发计划同步；
- 工单状态回写；
- 最终回归验证记录。

## 5. 任务拆解

### 5.1 删除旧活跃路径

必须彻底删除或归档以下旧口径：

1. `full/shell/mixed`
2. `internal/external`
3. 目录态 `.box` 的正式路径
4. 旧 `BoxInspectResult` 结构
5. 只认 `box.inspect` 的查看入口
6. 旧布局字段名或旧 YAML 结构

要求：

- 不保留仍可进入生产链路的旧实现；
- 不允许以注释、开关或未使用代码形式“保底”保留旧分支。

### 5.2 回写所有受影响文档

需要重新核对代码后，更新：

- 生态共用技术文档中与箱子链路直接相关的漂移点；
- Host / SDK / 编辑引擎 / 查看器 / 布局插件仓内部技术文档；
- 如有项目级需求文档、README 与操作说明，也必须同步更新。

### 5.3 回写工单与验收结论

至少需要：

1. 在工单064与工单065中写回处理结果；
2. 记录最终哪些仓完成了什么；
3. 记录正式验证脚本与人工验收结果；
4. 标记仍需后续大任务处理的非阻塞事项。

### 5.4 统一质量门禁复验

在最终收口阶段，所有受影响仓库都要再跑一次正式脚本，确保不存在阶段间回归。

## 6. 串并行安排

本阶段可以并行做文档回写与清理，但最终必须统一复验：

- 旧代码清理分支；
- 文档回写分支；
- 工单回写分支；
- 统一门禁复验与结论收口。

## 7. 验证门禁

需要重新运行阶段八涉及的全部正式脚本，并补充：

- 根工作区 `npm run verify:workspace`
- 与文档、README、工单一致性的人工复核

必须确认：

- 代码、文档、测试三者不再漂移；
- 活跃链路中不再出现旧箱子模型；
- 没有 TODO、占位实现或“后续兼容”说明残留。

## 8. 阶段退出标准

- 箱子能力已经成为生态正式基础能力；
- 相关仓库文档、代码、测试和验证全部一致；
- 可以在不依赖临时说明、不依赖人工记忆的情况下继续维护和扩展。
