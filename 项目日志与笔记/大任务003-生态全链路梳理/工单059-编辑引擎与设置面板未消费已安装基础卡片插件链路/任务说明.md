# 工单059-编辑引擎与设置面板未消费已安装基础卡片插件链路

## 背景

本轮检查的核心问题之一是：用户安装 Host 后，再安装各种基础卡片插件，是否就能直接在编辑引擎和设置面板里使用和治理这些插件。继续核对编辑引擎、设置面板与 Host 插件运行时后，确认这条链路当前没有正式闭环。

## 证据

### 编辑引擎侧

1. `Chips-EditingEngine/src/basecard-runtime/registry.ts` 当前只静态注册：
   - `imageBasecardDescriptor`
   - `richtextBasecardDescriptor`
2. `Chips-EditingEngine/src/basecard-runtime/registrations/image.ts`
   与
   `Chips-EditingEngine/src/basecard-runtime/registrations/richtext.ts`
   直接从工作区 `Chips-BaseCardPlugin/*/src/index` 源码导入插件导出。
3. `Chips-EditingEngine/src/components/CardBoxLibrary/data.ts` 通过：
   - `import.meta.glob('../../../../Chips-BaseCardPlugin/**/manifest.yaml', ...)`
   在构建期扫描工作区清单，而不是从 Host 已安装插件治理状态读取。
4. `Chips-EditingEngine/src/basecard-runtime/frame-host.tsx` 中，当描述符不存在时会直接进入错误态：
   - `未注册基础卡片类型: ${cardType}`

### 设置面板侧

1. `Chips-EcoSettingsPanel/src/app/menu-registry.tsx` 当前只提供：
   - themes
   - languages
   - app-plugins
   - component-gallery
2. `Chips-EcoSettingsPanel/src/shared/runtime/settings-runtime-service.ts` 只有：
   - 主题治理
   - 应用插件治理
   - 没有卡片插件 / 模块插件 / 布局插件治理接口

### 主题治理补充缺口

1. `Chips-Host/src/runtime/plugin-runtime.ts` 安装插件时默认写入 `enabled: false`
2. `SettingsRuntimeService.installTheme(...)` 只做 `plugin.install(...)`
3. `Chips-Host/src/main/services/register-host-services.ts` 的 `loadInstalledThemes(...)` 只读取 `enabled` 的主题
4. `ThemePage.tsx` 仍会对刚安装但未启用的主题显示“应用”按钮

## 影响范围

- 用户安装新的基础卡片插件后，当前不能保证它会自动进入编辑引擎卡箱库；
- 也不能保证编辑引擎能正式渲染、编辑该卡片类型；
- 设置面板不能作为统一插件治理入口管理卡片插件、模块插件和布局插件；
- 主题“安装后立即可切换”的用户体验也没有闭环。

## 复现步骤

1. 查看编辑引擎基础卡片注册中心与卡箱库数据源；
2. 观察其均依赖工作区源码与构建期扫描，而不是 Host 已安装插件状态；
3. 查看设置面板菜单与运行时服务，确认只有主题和应用插件治理；
4. 查看主题安装与 Host 主题加载逻辑，确认“安装后直接应用”路径未闭环。

## 期望行为

1. 明确编辑引擎正式支持的基础卡片插件接入与治理策略；
2. 若支持用户安装基础卡片插件，应在编辑引擎和设置面板里有正式治理入口和消费链路；
3. 主题安装、启用、应用需在同一正式产品链路中闭环。
