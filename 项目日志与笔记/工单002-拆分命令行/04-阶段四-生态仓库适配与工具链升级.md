# 任务002-阶段四：生态仓库适配与工具链升级

## 1. 背景

在阶段二、三完成 `chips` 与 `chipsdev` 的实现之后，生态内已有的多个仓库仍然引用旧形式的命令：

- package.json 脚本与技术文档中大量使用 `chips dev ...`；
- README、需求文档、开发计划与技术说明中，将开发者命令行工具称为“Chips Dev（chips dev）”；
- 部分测试与 CI 脚本假定存在 `chips dev` 二级命令。

本阶段目标是统一完成所有仓库的命令行适配，使整个生态内部工具链完全切换到 `chips` / `chipsdev` 双命令体系。

## 2. 目标

1. 所有仓库不再引用 `chips dev`，全部改为 `chipsdev`；
2. 所有 Host 相关操作统一通过 `chips`（主要是 `chips ...` 系列命令）；
3. CI 与本地开发脚本在新命令下全部可用；
4. 脚手架与模板工程生成的项目默认就使用 `chipsdev` 相关命令。

## 3. 范围

### 3.1 在范围内（按仓库）

- `Chips-Host`：
  - 确认不再存在任何 `chips dev` 相关引用；
  - 如有文档/脚本涉及开发构建，统一调整为 `chipsdev`（通过 SDK 提供）。
- `Chips-SDK`：
  - 确认内部文档与示例只使用 `chipsdev`；
  - 若有历史对 `chips dev` 的引用，一并更新。
- `Chips-Scaffold`：
  - `chips-scaffold-theme`；
  - 其他脚手架项目（如 `chips-scaffold-app`、`chips-scaffold-basecard` 等）。
- `ThemePack/Chips-default`：
  - README、技术文档、开发计划中引用的命令；
  - package.json 中的脚本。
- `Chips-CardViewer`：
  - package.json 脚本中 `chips dev server/build/test/lint/validate` → `chipsdev ...`；
  - README 与技术文档。
- `Chips-BaseCardPlugin`、`Chips-ComponentLibrary`：
  - 如有 CLI 相关脚本与文档，一并调整。
- 生态共用/设计/规划文档中引用具体命令的部分（与阶段五有交叉，本阶段偏向“代码仓库内文档”）。

### 3.2 不在范围内

- 生态共用技术文档与设计原稿的“正式规范章节”改写（在阶段五集中处理），本阶段可先标注 TODO 或进行最小修订。

## 4. 产出物

1. 各仓库更新后的 package.json / scripts：
   - 所有脚本统一使用 `chipsdev` 调用开发 CLI；
   - 若需要 Host 配合（如 theme validate），使用 `chips ...`。
2. 各仓库更新后的 README 与技术文档：
   - 示例命令改为新形式；
   - 文本描述中不再提 `chips dev`。
3. CI 配置更新：
   - 所有引用 `chips dev` 的 CI pipeline 脚本改为 `chipsdev`；
   - 新增或更新 CLI 冒烟测试步骤。

## 5. 任务拆解

### 5.1 CLI 使用点扫描与清单

- 在整个工作区执行静态搜索：
  - `chips dev`；
  - `Chips Dev命令行工具` 等关键字；
  - 旧的 `chipsdev` 提法（需要统一成新定义）。
- 按仓库整理“CLI 使用清单”，包含：
  - 文件路径；
  - 当前命令；
  - 预期替换成的命令。

### 5.2 package.json 脚本适配

- `Chips-CardViewer`：
  - `"dev": "chips dev server"` → `"dev": "chipsdev server"`；
  - `"build": "chips dev build"` → `"build": "chipsdev build"`；
  - `"test": "chips dev test"` → `"test": "chipsdev test"`；
  - `"lint": "chips dev lint"` → `"lint": "chipsdev lint"`；
  - `"validate": "chips dev validate"` → `"validate": "chipsdev validate"`。
- `Chips-Scaffold` 各脚手架项目：
  - 若有调用 `chips dev create/build/test/validate` 的脚本或文档，一律改为 `chipsdev`；
  - 确认脚手架内部 README 与测试用例也同步更新。
- `ThemePack/Chips-default`：
  - 构建、打包与验证脚本改用 `chipsdev`；
  - Host 相关验证脚本使用 `chips theme ...`。

### 5.3 测试与 CI 脚本适配

- 更新各仓库 CI 配置（如 GitHub Actions/自建 CI）：
  - 安装 SDK / CLI 的步骤（例如 `npm install -g @chips/dev-cli`）；
  - 将原 `chips dev ...` 改为 `chipsdev ...`；
  - 验证 `chips ...` 调用环境是否准备就绪（如需要启动本地 Host）。
- 为核心仓库新增 CLI 冒烟测试：
  - 使用 `chipsdev` 在临时目录创建工程并跑通 build/test/validate；
  - 对主题包、脚手架等仓库执行最小验证流水线。

### 5.4 仓库内文档与示例更新

- 更新各仓库内的：
  - README；
  - 技术文档（技术架构、模板说明、集成说明等）；
  - 开发计划与阶段文档（如“集成 chips dev”的段落）；
  - 示例代码片段中的命令行调用。
- 统一术语：
  - “Chips Dev 命令行工具（chipsdev）”；
  - 明确 chipsdev 归属 SDK，不随 Host 安装。

## 6. 验收标准

- 在每个仓库中执行静态检查：
  - 不再存在 `chips dev` 字符串；
  - 所有 CLI 示例命令均符合新约定。
- 在 CI 中为代表性仓库（至少包括：Host / SDK / 一个脚手架 / ThemePack / CardViewer）跑通完整流水线：
  - 使用 `chipsdev` 完成创建/构建/测试/验证；
  - 使用 `chips` 完成必要的 Host 侧校验（如主题验证）。
- 通过一次“从零开始”的手工验证：
  - 新机器安装 SDK 与 `chipsdev`；
  - 使用脚手架创建主题包/应用工程，运行全部 scripts；
  - 打包 `.cpk` 后通过 Host 的 `chips` 命令完成安装与简单运行验证。
