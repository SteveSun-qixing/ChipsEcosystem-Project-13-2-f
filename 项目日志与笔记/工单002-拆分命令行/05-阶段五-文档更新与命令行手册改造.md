# 任务002-阶段五：文档更新与命令行手册改造

## 1. 背景

命令行拆分后，所有与 CLI 相关的正式文档需要统一更新，以避免“chips dev / chipsdev / Host 二级入口旧写法”多种叫法混用，造成后续读者混淆。

本阶段目标是完成生态层面所有正式文档的更新，使之与新的命令行设计完全一致，并对外形成清晰、一致的描述。

## 2. 目标

1. 所有正式文档中，命令行相关内容统一为：
   - Host 侧：`chips` / `chips ...`；
   - 开发者侧：`chipsdev ...`；
   - 不再出现 `chips dev` 形式。
2. 命令行手册与接口标准文档更新完毕：
   - 《Chips Host 命令行使用手册》；
   - 《Chips Dev 开发者命令行手册》；
   - 《Chips-Host 对外接口基线》等。
3. 设计原稿与白皮书中的 SDK 描述与新 CLI 方案一致。

## 3. 范围

### 3.1 在范围内

- `生态共用技术文档`：
  - `命令行工具` 目录下的两份手册；
  - `协议与接口标准` 中与 CLI 相关章节；
  - `开发与运维` 中与开发工具链相关的章节。
- `生态设计原稿`：
  - `薯片生态白皮书（综述版）.md`；
  - `19-SDK与协议.md` 及其它提及命令行工具的章节；
  - 任何明确说明“开发者命令行工具”的段落。
- `开发计划` 与 `需求文档`：
  - 所有提到 CLI 或工具链的需求与阶段规划；
  - 特别是与 SDK、Host、脚手架、主题包相关的任务文档。
- 各项目仓库内的说明文档：
  - README；
  - 技术架构与集成说明；
  - 测试策略与质量门禁设计等。
- 项目日志与问题工单中作为“规范引用”的部分。

### 3.2 不在范围内

- `.specstory/history` 等“对话历史/生成记录”文件可选更新，原则上不强制，但可视实际需要做全局替换或在索引中标注“历史文本，命令行方案已演进”。

## 4. 产出物

1. 更新后的《Chips Host 命令行使用手册》：
   - 全文使用 `chips` / `chips ...`；
   - 不提及 `chipsdev`，但可在“相关工具”中简要说明存在开发者命令行工具。
2. 更新后的《Chips Dev 开发者命令行手册》：
   - 标题与内容统一为 `chipsdev`；
   - 所有示例命令均使用 `chipsdev ...`；
   - 在前言中说明其归属于 SDK。
3. 更新后的 SDK/Host 相关设计文档：
   - SDK 章节中准确描述 `chipsdev` 的职责与范围；
   - Host 章节中准确描述 `chips` 的职责与范围。
4. 针对命令行拆分的说明章节：
   - 在合适位置（如《开发与运维》或《系统架构概述》）新增一小节，说明“为什么拆分 CLI、如何使用两套命令”。

## 5. 任务拆解

### 5.1 全局检索与分组

- 在整个工作区执行检索：
  - `chips dev`；
  - `chipsdev`；
  - `Chips Dev命令行工具`；
  - 其它可能的别写。
- 按文档类型分组清单：
  - 规范级文档（生态共用技术文档、架构与接口标准）；
  - 设计原稿与白皮书；
  - 需求与开发计划；
  - 项目 README 与技术文档；
  - 项目日志与问题工单。

### 5.2 命令行手册改造

- 《01-Chips-Host命令行使用手册.md》：

### 5.4 项目级文档与日志更新

- 每个项目仓库：
  - README 中的“如何开发/构建/测试”命令统一改为 `chipsdev`；
  - 技术文档中的 CLI 示例同步修改；
  - 测试策略/质量门禁文档中更新命令名称。
- 项目日志与问题工单：
  - 对“作为规范引用”的部分进行更新，使其指向新方案；
  - 保留历史记录但避免在“当前指导建议”部分使用旧命令名。
