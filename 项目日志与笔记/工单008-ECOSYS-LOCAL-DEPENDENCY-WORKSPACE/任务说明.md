# 工单008-ECOSYS-LOCAL-DEPENDENCY-WORKSPACE 任务说明

## 1. 背景

在处理 `项目日志与笔记/任务005-卡片查看器插件依赖无法按生态仓库方式安装.md` 时，对生态设计原稿、生态共用技术文档以及相关仓库配置进行了核查，确认当前文档口径与仓库落地状态存在明显偏差：

- 生态共用文档与仓库 README 普遍要求应用插件通过 `@chips/component-library` 与 `chips-sdk` 接入生态。
- `Chips-CardViewer/package.json` 已按该口径声明依赖：
  - `@chips/component-library`
  - `chips-sdk`
- 但当前工作区并没有一套正式可用的“生态仓库本地联动安装”方案，导致插件工程执行 `npm install` 时只能访问公共 registry。

这不是单一插件仓库的实现错误，而是生态工具链、包分发策略与工作区结构未打通的系统性缺口。

## 2. 已确认事实

### 2.1 文档与接入口径

- `生态共用技术文档/插件开发/02-应用插件开发.md` 明确写明生态提供基于 React 的组件库 `@chips/component-library` 与 SDK。
- `生态共用技术文档/组件库/01-组件库对外使用总览.md` 和 `Chips-ComponentLibrary/技术文档/08-开发者接入指南.md` 均以 `@chips/component-library` 作为统一接入包名。
- `生态共用技术文档/开发与运维/02-依赖管理指南.md` 已允许采用 monorepo/workspaces 作为大型项目依赖管理方式。

### 2.2 仓库现状

- `Chips-ComponentLibrary/package.json`
  - `name` 为 `chips-component-library`
  - `private: true`
  - 当前承担 monorepo 管理职责
  - 仓库内不存在名为 `@chips/component-library` 的实际 package
- `Chips-ComponentLibrary/packages/*` 当前拆分为多个包：
  - `@chips/components`
  - `@chips/hooks`
  - `@chips/card-runtime`
  - 其他内部包
- `Chips-SDK/package.json`
  - `name` 为 `chips-sdk`
  - `private: true`
  - 当前仓库根目录之外没有统一生态 workspace 根配置来让插件工程解析本地 sibling repo
- 生态根目录 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f` 当前没有统一 `package.json` / `workspaces` 配置。

### 2.3 实际复现

在 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer` 执行：

```bash
npm install
```

得到结果：

- npm 请求公共 registry 上的 `chips-sdk@^0.1.0`
- npm 请求公共 registry 上的 `@chips/component-library@^0.1.0`
- 安装流程报 404 并终止

说明当前生态仓库并未提供本地开发期正式依赖解析通道。

## 3. 为什么该问题不能在当前任务内直接继续开发

根据仓库规则，遇到以下情况必须先提工单并停止继续推进当前开发：

- 缺少生态标准组件/标准包导致无法接入；
- SDK/工具链接口不满足运行需求；
- 生态底层仓库没有打通，只能依赖临时开发代码或手工桥接。

本问题同时命中以上条件：

1. `@chips/component-library` 作为文档约定的标准接入包，目前没有实际可安装实现。
2. `chips-sdk` 与组件库没有形成生态级本地工作区安装策略。
3. 若在 `Chips-CardViewer` 内直接改成某个临时相对路径、手工复制导出、或手改 `node_modules`，都属于明确禁止的临时开发。

因此，本任务在当前阶段应先由生态架构与工具链层完成闭环，再恢复插件开发与修复工作。

## 4. 需要决策/补齐的内容

生态层必须先明确并落地以下方案之一，且需要形成正式文档：

1. **统一生态根工作区方案**
   - 在生态仓库根目录建立正式 `package.json` 与 `workspaces`
   - 统一定义各子仓如何被插件工程解析
   - 明确 lockfile、安装命令、CI 安装流程与脚手架生成规则

2. **组件库统一对外包方案**
   - 正式提供 `@chips/component-library`
   - 明确其与 `@chips/components`、`@chips/hooks`、`@chips/card-runtime` 的聚合关系
   - 明确导出入口、构建产物、版本策略与测试门禁

3. **SDK 本地开发消费方案**
   - 明确 `chips-sdk` 在生态仓库开发态如何被插件工程消费
   - 明确是否继续 `private`、是否通过 workspace 引用、是否需要发布产物目录
   - 明确 CLI 二进制 `chipsdev` 在插件工程 `npm scripts` 中的稳定可执行路径

4. **脚手架与文档同步**
   - `Chips-CardViewer`、`Chips-Scaffold` 模板、生态共用文档必须统一同一套安装口径
   - 不允许文档写一套、仓库实际实现是另一套
