# 工单010-DEV-RUN-DIST-MANIFEST-ENTRY-RESOLUTION 任务说明

## 1. 背景

在修复 `chipsdev` 直接命令被 Volta 拦截的问题后，继续对原始报错场景执行实机验证：

```bash
cd /Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer
chipsdev run
```

命令已经能够进入 `chipsdev` CLI、完成构建并启动 Host 安装流程，但在 `plugin.install` 阶段报错，说明原始“命令入口层”问题已闭环后，又暴露出新的安装链路契约问题。

根据仓库规范，遇到 SDK / Host / 契约层问题需要先登记工单，再停止继续修复。

## 2. 已确认事实

### 2.1 当前错误

`chipsdev run` 输出如下关键信息：

```json
{
  "code": "PLUGIN_ENTRY_NOT_FOUND",
  "message": "Plugin entry not found: dist/index.html",
  "details": {
    "route": "plugin.install",
    "lastError": {
      "sourcePath": "/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/dist/manifest.yaml",
      "entry": "dist/index.html"
    }
  }
}
```

### 2.2 当前工程配置

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/chips.config.mjs`
  - 应用入口为 `index.html`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/manifest.yaml`
  - `entry: dist/index.html`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/cli/index.js`
  - `chipsdev run` 先执行构建；
  - 再将根 `manifest.yaml` 同步到 `dist/manifest.yaml`；
  - 然后以 `dist/manifest.yaml` 作为 `plugin.install` 的 `manifestPath`

### 2.3 契约冲突点

当前链路存在以下不一致：

1. 工程根 `manifest.yaml` 的 `entry` 指向 `dist/index.html`
2. `chipsdev run` 安装时传给 Host 的却是 `dist/manifest.yaml`
3. 如果 Host 按“manifest 所在目录 + entry”解析入口，则会尝试解析到 `dist/dist/index.html`
4. 最终导致 `plugin.install` 找不到入口文件

因此，问题不是 Volta，也不是 `chipsdev` 命令找不到，而是：

- `chipsdev run` 对 manifest 的安装入口策略
- `manifest.entry` 的相对路径契约
- Host `plugin.install` 的入口解析规则

三者没有统一。

## 3. 为什么当前不能继续直接修复

这已经超出“命令入口解析”问题，进入生态级契约设计范围：

- 应用插件的 `manifest.entry` 到底应相对于工程根、打包根还是安装源 manifest 所在目录；
- `chipsdev run` 是否应该直接安装工程根 `manifest.yaml`，还是生成一份面向 `dist/` 的运行态 manifest；
- Host `plugin.install` 是否需要识别“开发态 dist manifest”与“正式 `.cpk` 包根 manifest”的差异。

这属于 SDK / Host / manifest 契约联动问题，必须由相关责任方统一决策后再修改，不能在当前对话里临时拍脑袋改一处。

## 4. 建议需要统一决策的点

1. **应用插件 manifest 入口基准**
   - 明确 app 类型 `manifest.entry` 的相对路径基准。

2. **`chipsdev run` 的安装输入契约**
   - 明确开发态是直接安装工程根 manifest，还是安装生成后的运行态 manifest。

3. **Host `plugin.install` 的解析规则**
   - 明确是否需要对开发态安装场景做正式支持，而不是隐含假设包结构只有 `.cpk` 一种。

4. **脚手架与文档同步**
   - `Chips-CardViewer`、应用脚手架模板、manifest 规范文档、`chipsdev run` 文档需要统一同一套规则。

## 5. 修复方案

本工单已于 2026-03-09 正式闭环，修复方案如下：

1. **收敛 manifest 单一事实源**
   - 明确应用插件工程根 `manifest.yaml` 为开发态与发布态的唯一正式清单源；
   - `manifest.entry` 继续保持指向构建产物，例如 `dist/index.html`。

2. **移除错误的 `dist/manifest.yaml` 生成逻辑**
   - `chipsdev build` 不再把工程根 `manifest.yaml` 复制到 `dist/manifest.yaml`；
   - 避免开发态安装链路出现“manifest 已在 dist 中，却仍声明 `entry: dist/...`”的二次路径拼接错误。

3. **修正 `chipsdev run` 安装输入**
   - `chipsdev run` 改为直接将工程根 `manifest.yaml` 传给 Host `plugin.install`；
   - Host 继续按既有契约，从该根 manifest 的 `entry` 指向构建产物 `dist/...` 完成安装与运行。

4. **补齐回归测试**
   - 在 `Chips-Host/tests/unit/plugin-runtime.test.ts` 中补充“根 manifest + dist entry”安装用例；
   - 在 `Chips-SDK/tests/run-cli-smoke-tests.cjs` 中补充 `chipsdev build` 不生成 `dist/manifest.yaml` 的回归校验。

5. **同步文档**
   - 已更新 manifest 配置规范、应用插件开发文档与 `chipsdev` 手册，统一说明“根 manifest 为唯一清单源”的正式契约。
