命令可以正常完成 Vite 构建，并输出：

## 2. 已确认事实

### 2.1 文档与当前现象不一致

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/生态共用技术文档/命令行工具/02-Chips-Dev开发者命令行手册.md:56`-`:62` 已明确写明：

1. `chipsdev run` 启动 Host 运行时；
2. 安装并启用当前应用插件；
3. 初始化插件会话；
4. 通过 Host 的 `window` 服务打开应用窗口。

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/README.md:22`-`:30` 也将 `chipsdev run` 定义为“启动完整 Host 底层并加载当前应用插件”。

当前实机行为与上述契约不一致。

### 2.2 `chipsdev run` 当前没有进入真正的 Electron 主机链路

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/cli/index.js:1164`-`:1168` 当前直接：

1. `new HostApplication({ workspacePath })`
2. `await app.start()`
3. 基于 `app.createBridge()` 创建 Runtime Client

但 `HostApplication` 只是：

- 创建 `Kernel`
- 创建 `NodePalAdapter`
- 注册 Host services

见 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/main/core/host-application.ts:33`-`:88`。

真正负责 Electron 主进程生命周期绑定的是：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/main/core/main-process.ts:64`-`:83`

这里才会：

1. 读取 Electron `app`
2. `await app.whenReady()`
3. 绑定 `before-quit` / `window-all-closed` / `activate`

而 `chipsdev run` 当前并没有走这条链路。

### 2.3 当前窗口服务在 Node CLI 场景下只会得到“内存态窗口”

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/pal/src/node-adapter.ts:131`-`:177` 表明：

- `NodeWindowManager` 启动时尝试读取 Electron `BrowserWindow`
- 仅当 `this.electronBrowserWindow` 存在时，才会创建真实窗口并调用 `loadWindowContent`
- 否则只创建一个内存中的 `WindowState` 记录并返回

而当前 `chipsdev run` 是从普通 Node CLI 进入，实测该链路中 `loadElectronModule()` 返回空值，因此 `window.open` 只会成功返回内存态对象，不会出现真实桌面窗口。

### 2.4 CLI 自身也不会被“空 Promise”保活

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/cli/index.js:1254`-`:1256` 末尾使用：

```js
await new Promise(() => {});
```

作为“保持进程存活”的实现。

但该写法本身不会创建事件循环活动句柄，因此当没有真实 Electron 主进程、窗口或 server 句柄存在时，Node 进程会直接退出。这与当前“输出成功后立刻回到 shell”完全一致。

### 2.5 `window.open` 返回值读取也与服务契约不一致

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/main/services/register-host-services.ts:833`-`:859` 中，`window.open` 返回的是：

```ts
{ window }
```

但 `chipsdev run` 在 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/cli/index.js:1230`-`:1233` 中却按 `opened.windowId` / `opened.id` 读取窗口 ID，而不是读取 `opened.window.id`。

这不是“无窗口”的主因，但说明 `chipsdev run` 与 Host 服务响应契约仍存在不一致，验收没有真正打通。

### 2.6 即使窗口链路补齐，当前应用静态资源仍然无法正常装载

#### 事实 A：manifest 文件安装路径只复制入口文件

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/runtime/plugin-runtime.ts:418`-`:488` 表明：

- 当 `plugin.install` 传入的是 manifest 文件路径时；
- 运行时会创建临时目录；
- 复制 manifest 文件；
- 然后只复制 `manifest.entry` 指向的入口路径；
- 最终再把这个临时目录安装到插件目录。

当前 `chipsdev run` 传入的是工程根 `manifest.yaml`，而不是工程目录，所以 `dist/index.html` 会被复制，但 `dist/assets/*` 不会被带上。

实机检查 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/.chips-host-dev/plugins/com.chips.card-viewer/dist/`，当前只有 `index.html`，没有 `assets/` 目录。

#### 事实 B：构建出的入口 HTML 使用绝对 `/assets/...`

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/dist/index.html:7` 当前输出：

```html
<script type="module" crossorigin src="/assets/index-WV_j79G5.js"></script>
```

同时 `chipsdev build` 的 Vite 配置在 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/cli/index.js:413`-`:434` 中没有设置相对 `base`。

这意味着即使把 `assets/` 一起复制过去，只要 Host 通过本地 `loadFile()` 加载 `dist/index.html`，浏览器仍可能把 `/assets/...` 解析为文件系统根路径，而不是插件安装目录下的 `dist/assets/...`。

因此当前问题不是单点 bug，而是：

1. Host 启动形态不对；
2. CLI 保活策略不成立；
3. manifest 文件安装链路不完整；
4. app 构建输出的资源路径也未对齐 Host 本地加载方式。

## 3. 根因判断

当前 `chipsdev run` 没有真正实现“开发态启动完整 Host 并打开应用窗口”的闭环，根因至少包含四层：

1. **启动形态错误**
   - 直接用 Node CLI 里的 `HostApplication`
   - 没有走 Electron 主进程启动链路

2. **窗口抽象退化为内存态**
   - `NodePalAdapter` 在无 Electron 模块时不会创建真实 `BrowserWindow`

3. **CLI 保活手段无效**
   - `await new Promise(() => {})` 不是有效的运行态保活方式

4. **静态资源安装/加载契约未打通**
   - manifest 文件安装只复制入口文件
   - 构建输出对本地 `file://` / `loadFile()` 场景不友好

因此，当前问题已经不是某一个仓库内“改一行代码”能安全闭环的局部问题，而是 SDK CLI、Host 启动模型、插件安装契约、app 构建契约四者之间没有统一。

## 6. 修复方案

本工单已于 2026-03-09 正式闭环，修复方案如下：

1. **收敛 `chipsdev run` 到真实 Electron Host 启动链路**
   - `chipsdev run` 不再在普通 Node CLI 进程中直接启动 `HostApplication`；
   - 改为由 SDK 启动 Electron 可执行文件，并进入 Host 新增的开发态主入口；
   - Host 侧统一通过 `HostMainProcess` 完成 `app.whenReady()`、主机启动、插件安装、握手与窗口打开。

2. **补齐插件窗口 preload 注入**
   - Host PAL 窗口层新增插件窗口 preload 配置；
   - 真实 `BrowserWindow` 现在会在插件窗口中注入 `window.chips.*`；
   - preload 通过 `additionalArguments` 接收 `pluginId/permissions` 上下文，避免页面处于“窗口打开了但 Bridge 不存在”的假启动状态。

3. **打通权限快照到窗口上下文**
   - 应用插件窗口打开时，`window.open` 现在会携带 `plugin.init` 会话中的权限快照；
   - 文件关联链路与 `chipsdev run` 链路都已统一传递该权限数据；
   - 这样 Electron preload 暴露的 Bridge 上下文与 Host 运行时权限模型保持一致。

4. **修正 manifest 文件安装的产物复制策略**
   - Host 在“以 manifest 文件作为安装输入”时，不再只复制单个 `entry` 文件；
   - 现在会复制 `entry` 所在的完整运行时根目录（如 `dist/`）；
   - 从而保证 `assets/`、chunk、样式等构建产物会一并进入插件安装目录。

5. **修正 app 构建的静态资源路径契约**
   - `chipsdev build` 在 app 构建场景下改为输出相对静态资源路径；
   - 生成的 `dist/index.html` 现在引用 `./assets/...`，可被 Host 本地 `loadFile()` 正常加载。

6. **补齐回归测试**
   - Host 单测已覆盖：
     - BrowserWindow preload 与桥接参数注入；
     - manifest 安装场景复制完整 `dist/assets/*`；
   - SDK smoke test 已覆盖：
     - app 构建产物生成 `dist/assets/`；
     - app 构建 HTML 使用相对 `./assets/...` 路径。
