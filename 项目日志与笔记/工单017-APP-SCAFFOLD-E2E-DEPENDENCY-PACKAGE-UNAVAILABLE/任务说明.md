# 工单017：应用插件脚手架生成工程的 E2E 验证依赖公开包不可安装

## 1. 背景

在补齐应用插件脚手架的主题链路、窗口外观基线与 Bridge API 文档后，需要执行 `chips-scaffold-app` 的生成工程 E2E 验证，确认模板产出的项目可以完整安装依赖、构建并通过基础门禁。

## 2. 现场证据

### 2.1 官方脚手架 E2E 在 `npm install` 阶段失败

在以下目录执行：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Scaffold/chips-scaffold-app`

命令：

```bash
npm run test:e2e
```

实际失败信息：

- `npm ERR! 404 Not Found - GET https://registry.npmjs.org/@chips%2fcomponent-library - Not found`

### 2.2 生成模板依赖公开安装源中不存在的包

应用脚手架模板 `package.json.tpl` 当前声明：

- `@chips/component-library: ^0.1.0`

但当前公开 npm 源上不存在该包，导致新生成工程无法在标准 `npm install` 流程中通过。

### 2.3 问题不在模板语法，而在生态分发契约未闭环

当前 `chips-scaffold-app` 的：

- 模板渲染测试通过；
- 模板结构检查通过；

说明脚手架本身可以生成工程骨架，但“生成工程 -> 安装依赖 -> 构建运行”的分发链路未闭环。

## 3. 根因判断

这是生态依赖分发与脚手架 E2E 验证环境之间的正式契约缺口：

1. 应用模板默认依赖了 `@chips/component-library`；
2. 当前验证环境按 README 与 E2E 脚本走标准 `npm install`；
3. 但组件库包未在该安装源中可解析，因此无法完成脚手架生成工程的全链路验证。

## 4. 影响范围

- `chips-scaffold-app` 生成的新应用插件工程无法按标准新项目流程直接安装依赖；
- 应用脚手架的 E2E 质量门禁当前不可信，无法完成真正闭环验收；
- 后续所有基于该模板创建的新应用工程都可能在第一步 `npm install` 就失败；
- 文档中的“快速开始 / npm install / npm run build”路径与实际生态交付状态不一致。

## 5. 复现步骤

1. 进入 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Scaffold/chips-scaffold-app`。
2. 执行 `npm run test:e2e`。
3. 等待脚手架生成临时应用工程并运行 `npm install`。
4. 观察安装阶段报错：`@chips/component-library` 404 Not Found。

## 6. 期望行为

- 官方应用脚手架生成的工程必须可以在标准开发环境中直接安装依赖并构建成功；
- 若依赖采用工作区、本地源或私有源，脚手架与文档必须给出正式且可执行的安装契约；
- E2E 验证必须覆盖真实依赖安装路径，而不是只验证模板文件是否生成。

## 7. 建议处理方向

1. 明确 `@chips/component-library` 与其他生态前端依赖的正式分发策略：
   - 发布到可访问的 npm 源；或
   - 为脚手架生成工程提供正式 workspace / file / registry 配置；
2. 同步更新应用脚手架模板 `package.json.tpl`；
3. 同步更新 `chips-scaffold-app` README 与相关生态文档中的安装说明；
4. 修复后重新执行 `npm run test:e2e`，将其恢复为正式质量门禁。

## 8. 正式处理结果

- 生态正式契约已明确为“模板保留 semver 版本声明，生态根工作区负责解析本地一方包”，不再依赖公开 npm 源提供 `@chips/component-library`；
- `chips-scaffold-app/scripts/run-generated-e2e.mjs` 已改为在临时生态工作区中通过 `chipsdev create app` 创建工程，并在工作区根执行 `npm install` 后再进入生成工程执行 `lint/test/build/validate`；
- 应用脚手架 README、需求文档、技术文档与测试策略已同步改为根工作区安装路径；
- 基础卡片脚手架与主题脚手架的相关 README / 文档说明也已同步为统一工作区依赖策略。

## 9. 需要协作方

- 组件库包分发维护方
- 应用脚手架维护方
- SDK / chipsdev 工具链维护方
- 生态依赖分发与安装契约维护方

## 10. 当前结论

应用插件脚手架的依赖安装链路已按本地工作区正式契约闭环。生成工程继续使用正式 semver 依赖声明，由生态根工作区统一解析本地 `chips-sdk` 与 `@chips/component-library`，E2E 验证也已切换到这一真实路径。
