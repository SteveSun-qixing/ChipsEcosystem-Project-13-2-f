# 工单026：生态设置面板所需的插件/主题治理契约未正式闭环

## 1. 背景

本工单来源于 2026-03-10 对 `Chips-EcoSettingsPanel` 的正式开发前核对。

用户已经明确要求：

- 生态设置面板必须严格按生态正式规范开发；
- 不允许临时开发、简化开发、旁路实现；
- 如遇到 Host / SDK / 组件库 / 公共基础层未打通，必须先登记问题工单，创建任务文件夹，并停止当前业务推进。

在对设计原稿、生态共用技术文档、Host 实现、SDK 封装、Bridge 暴露层进行逐项盘点后，已确认当前设置面板依赖的治理接口存在正式契约缺口，因此需要单独立项处理。

## 2. 结论

当前生态并不满足“直接开始高质量完成设置面板正式功能开发”的前提条件。

核心原因有三类：

1. 插件与主题治理的正式列表接口元数据不足；
2. SDK 对外能力与 Host 实际实现不一致；
3. 拖拽安装所需的文件路径解析能力尚未形成正式公开契约。

在这些问题处理前，继续实现设置面板会不可避免地落入以下违规路径之一：

- 前端自行解析插件 manifest；
- 直接依赖 preload 内部 helper；
- 在业务代码中兼容 SDK / Host 返回不一致；
- 为了赶进度补写临时转换层。

以上都不符合当前仓库规则，因此本工单视为正式阻断项。

## 3. 证据

### 3.1 Host 当前插件治理接口不足以支撑设置面板列表

`Chips-Host/src/main/services/register-host-services.ts` 当前仅正式实现：

- `plugin.install`
- `plugin.enable`
- `plugin.disable`
- `plugin.uninstall`
- `plugin.query`

其中 `plugin.query` 返回字段为：

- `id`
- `manifestPath`
- `installPath`
- `enabled`
- `type`
- `capabilities`
- `entry`
- `ui`
- `installedAt`

缺少设置面板列表要求的：

- 插件/主题显示名称
- 版本号
- 描述
- 主题 displayName / themeId
- 是否为当前主题的直接视图字段

这意味着设置面板若想展示“主题名称、版本、当前状态”“应用插件名称、版本、启用状态”等正式 UI，只能自己去解析 manifest 文件，属于越层实现。

### 3.2 主题管理场景缺少“全部已安装主题”的正式元数据链路

按当前正式文档与实现：

- `theme.list()` 只返回当前工作区“已启用”的主题；
- `theme.getCurrent()` 返回当前主题；
- `plugin.query({ type: 'theme' })` 只返回运行时记录，不含主题显示元数据。

因此：

- 已安装但未启用的主题无法通过正式主题接口获得完整列表信息；
- 又无法通过正式插件接口直接拿到主题名称与版本；
- 结果是“完整主题管理列表”需求无法闭环。

### 3.3 SDK 对外能力与 Host 实现存在不一致

`Chips-SDK/src/api/plugin.ts` 当前对外声明了：

- `client.plugin.getSelf()`
- `client.plugin.list()`
- `client.plugin.get()`
- `client.plugin.getCardPlugin()`
- `client.plugin.getLayoutPlugin()`

但 Host 当前并没有实现对应服务动作。

这会带来两个问题：

1. SDK 使用者会误判正式能力范围；
2. 设置面板若按 SDK 设计文档开发，将在运行时遇到接口不存在问题。

### 3.4 SDK 返回类型与 Host 实际返回结构不一致

当前已确认的典型不一致包括：

- `i18n.getCurrent()`：SDK 声明返回 `string`，Host 实际返回 `{ locale }`
- `i18n.translate()`：SDK 声明返回 `string`，Host 实际返回 `{ text }`
- `i18n.listLocales()`：SDK 声明返回 `string[]`，Host 实际返回 `{ locales }`
- `theme.getAllCss()`：SDK 声明返回 `string`，Host 实际返回 `{ css, themeId }`

这类问题会直接影响设置面板的类型安全、测试和实现正确性。

### 3.5 拖拽安装依赖的路径解析 helper 尚未正式冻结

现有业务示例 `Chips-CardViewer/src/components/DropZone.tsx` 已依赖：

- `window.chips.platform.getPathForFile(file)`

该能力目前来自：

- `Chips-Host/src/preload/create-bridge.ts`

但它尚未进入：

- Host 对外接口基线
- SDK platform API
- 设置治理正式契约文档

如果设置面板直接依赖它完成拖拽安装，会建立在未冻结接口上。

## 4. 对生态设置面板的直接阻断

当前阻断的首版能力包括：

1. 主题管理页的正式列表展示；
2. 应用插件页的正式列表展示；
3. 使用 SDK 的完整类型安全实现；
4. 拖拽安装能力的正式落地。

可实现但不能单独推进的内容包括：

- 纯壳层布局；
- 菜单注册表；
- 静态组件展示布局。

但由于用户要求交付“完整 100% 完美项目”，且禁止临时/占位/半成品推进，因此不能只做不闭环部分来规避阻断。

## 5. 建议解决方向

### 5.1 Host

补充面向治理场景的正式查询接口，至少二选一：

1. 扩展 `plugin.query` 返回完整元数据；
2. 或正式实现 `plugin.list/get` 并保证覆盖名称、版本、描述、类型、启用状态、安装时间等字段。

同时需要为主题治理明确统一口径，确保“已安装主题”和“已启用主题”都能被正式查询。

### 5.2 SDK

同步修正以下契约：

- `plugin.*` 只暴露 Host 已正式实现的动作，或推动 Host 一并补齐；
- `i18n.*` 返回类型与 Host 对齐；
- `theme.getAllCss()` 返回结构与 Host 对齐；
- 如保留 `getPathForFile`，补充类型与公开 API 说明。

### 5.3 文档

需要同步更新：

- `生态共用技术文档/协议与接口标准/06-Chips-Host对外接口基线.md`
- `生态共用技术文档/架构设计/14-Host服务域设计.md`
- `生态共用技术文档/插件开发/08-SDK使用指南.md`
- 如决定把拖拽路径解析能力正式公开，也需补入对应基线文档。

## 6. 当前处理决定

根据仓库规范，本工单已登记后：

- 立即停止继续推进 `Chips-EcoSettingsPanel` 的正式功能开发；
- 不在设置面板仓内编写旁路实现；
- 等待用户/协作方先处理生态契约问题，再继续推进业务开发。

## 8. 处理结果（2026-03-10）

本工单已完成闭环，正式处理结果如下：

1. Host 已补齐 SDK 先前声明但未落地的读取动作：
   - `plugin.list`
   - `plugin.get`
   - `plugin.getSelf`
   - `plugin.getCardPlugin`
   - `plugin.getLayoutPlugin`
2. Host 已将 `plugin.query` 升级为治理基线接口，在运行时记录基础上补齐正式元数据：
   - 通用字段：`name`、`version`、`description`、`capabilities`、`installedAt`
   - 主题字段：`themeId`、`displayName`、`publisher`、`parentTheme`、`isDefault`
   - 布局字段：`layoutType`、`displayName`
3. Host 运行时已在解析 manifest 时归一化主题/布局元数据，治理类应用不再需要自行读取 manifest 文件。
4. 插件权限边界已正式拆分为 `plugin.read` 与 `plugin.manage`：
   - 读取动作统一使用 `plugin.read`
   - 安装、启停、卸载等治理变更动作继续使用 `plugin.manage`
5. SDK 已与 Host 契约重新对齐：
   - `plugin.*` 读取动作与 Host 正式动作面一致；
   - `i18n.getCurrent/translate/listLocales` 已按 Host 返回结构做安全解包；
   - `theme.getAllCss()` 已按 `{ css, themeId }` 正式返回；
   - `platform.getPathForFile()` 已补入 SDK 平台辅助能力，供拖拽安装链路正式复用。
6. 生态共用技术文档与 SDK/项目技术文档已同步更新，治理接口标准、Host 对外接口基线、Bridge API 规范与 SDK 使用说明现已一致。
7. 验证结果：
   - `npm --workspace Chips-Host test` 通过（99/99）
   - `npm --workspace Chips-SDK exec vitest run tests/client.test.ts tests/tooling/contract-drift.test.ts tests/route-manifest.test.ts` 通过（9/9）

结论：

- `Chips-EcoSettingsPanel` 所依赖的插件/主题治理读取契约已正式闭环；
- 后续设置面板可以在不解析 manifest、不依赖未冻结旁路接口、不编写临时转换层的前提下继续推进正式开发。
