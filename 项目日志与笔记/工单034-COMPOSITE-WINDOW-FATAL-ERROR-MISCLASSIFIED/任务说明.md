# 工单034：复合卡片窗口 fatal 错误误判

> 当前状态：已闭环（2026-03-13）

## 1. 背景

用户反馈编辑引擎中的复合卡片预览“开始正常，操作一段时间后变成黄色三角感叹号”，同时控制台没有有效日志。

排查后确认，这不是浏览器原生崩溃，而是编辑引擎收到了 SDK `client.card.compositeWindow.onFatalError()` 事件，主动把预览区域切到了错误覆盖层。

## 2. 根因

根因位于 Host 统一复合卡片渲染脚本：

- 文件：`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/card-service/src/card-service.ts`
- 原逻辑把

```ts
errorPayload.length === Math.max(frameList.length, errorPayload.length)
```

作为是否发出 `chips.composite:fatal-error` 的条件。

这会把“部分节点失败、部分节点成功”的情况误判成整卡 fatal。例如：

- 成功 iframe 数：1
- 降级节点错误数：1

则旧逻辑会认为满足 fatal 条件，直接让编辑引擎把整个预览区改成黄三角覆盖层。

## 3. 正式处理结果

### 3.1 Host 侧

- 复合卡片脚本已改为基于总节点数 `totalNodeCount` 统计失败节点；
- 只有当 `failedNodeIds.size === totalNodeCount` 时才发出 `chips.composite:fatal-error`；
- 动态 iframe `error` 事件也会同步更新失败节点集合与节点状态，避免运行时错误分级再次漂移。

### 3.2 EditingEngine 侧

- `CardWindow` 在收到 `onFatalError` 或复合窗口 render promise reject 时，现已通过 `console.error` 输出完整上下文：
  - `cardId`
  - `cardFile`
  - `mode`
  - `error`

这样后续若再次出现真实 fatal，开发者可以直接从控制台拿到错误对象，而不是只看到黄三角。

## 4. 验证

- 已新增 Host 回归测试：
  - `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/tests/unit/card-service-rendering.test.ts`
- 新用例验证：
  - 当复合卡片存在“一个节点成功、一个节点失败”时，只会发出 `chips.composite:node-error`；
  - 不会再错误发出 `chips.composite:fatal-error`。

## 5. 结论

当前问题已经闭环。以后只有在“全部节点都失败”时，编辑引擎才会看到整卡 fatal 覆盖；部分节点失败将保持为局部降级，不再误伤整个预览窗口。
