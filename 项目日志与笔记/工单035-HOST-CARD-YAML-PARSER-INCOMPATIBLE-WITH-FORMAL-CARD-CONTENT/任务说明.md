# 工单035：Host 卡片 YAML 解析与正式卡片内容协议不兼容

> 当前状态：已闭环（2026-03-13）

## 1. 背景

用户在编辑引擎里编辑基础卡片一段时间后，复合卡片预览区出现黄色三角感叹号。  
排查后确认，这次并不是基础卡片插件渲染失败，而是 Host 在读取卡片内容 YAML 时失败，导致 `card.render` 路由直接报错。

## 2. 现场证据

通过 Host Runtime 直接复现：

- 卡片路径：`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/.chips-host-dev/2k7a0rnzTf.card`
- 路由：`card.render`
- 原始异常：

```text
ROUTE_EXECUTION_FAILED
Invalid indentation in YAML content
```

堆栈显示异常来自：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/shared/yaml-lite.ts`

而现场卡片内容文件由编辑引擎使用 `yaml.stringify(...)` 正式写出，正文字段中包含富文本 HTML，多行换行后仍是合法 YAML。

## 3. 根因

当前生态存在“写协议”和“读协议”不一致：

- EditingEngine 使用正式 `yaml` 包写 `.card` 内容；
- Host `CardService` 却仍用 `parseYamlLite` 读取 `.card/metadata.yaml`、`.card/structure.yaml` 与 `content/*.yaml`；
- `yaml-lite` 只能覆盖极窄的 YAML 子集，无法稳定解析 `yaml.stringify` 生成的富文本多行内容。

因此问题本质上不是单张卡片坏了，而是 Host 卡片读取链路没有遵循正式卡片文件协议。

## 4. 正式处理结果

### 4.1 Host 卡片读取链路

- 文件：`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/card-service/src/card-service.ts`
- 已新增正式解析函数 `parseCardYamlRecord(...)`；
- `.card/metadata.yaml`、`.card/structure.yaml`、`content/*.yaml` 现统一使用 `yaml.parse(...)`；
- 解析失败时会返回带 `filePath` 与 `cause` 的正式 `CARD_SCHEMA_INVALID` 错误上下文。

### 4.2 依赖声明

- 文件：`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/package.json`
- 已补齐 Host 对 `yaml` 的直接依赖声明，避免包级职责再依赖隐式 hoist。

### 4.3 回归测试

- 文件：`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/tests/unit/card-service-rendering.test.ts`
- 已新增用 `yaml.stringify(...)` 生成正式卡片内容文件的测试；
- 验证 `card.render` 能正常读取并渲染由 EditingEngine 写出的富文本 YAML 内容。

## 5. 验证结果

- `Chips-Host` `npm run build` 已通过；
- Host 关键回归测试已通过；
- 现场问题卡片通过 Host Runtime 直接调用 `card.render` 已返回成功，不再出现 `Invalid indentation in YAML content`。

## 6. 结论

当前问题已闭环。  
后续 EditingEngine 写出的正式 `.card` YAML 内容，Host 能按同一正式协议稳定回读，避免再次出现“编辑能保存、预览不能显示”的运行时断链。
