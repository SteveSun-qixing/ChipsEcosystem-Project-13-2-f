# 工单037：编辑引擎正式 TypeScript 编译门禁未闭环

## 1. 背景

在处理编辑引擎当前用户问题时，已经先按生态设计原稿、编辑引擎设计、资源管理器设计、拖动交互设计、主题系统和多语言规范完成相关阅读，并对 `Chips-EditingEngine` 中“文件管理窗口文件名前大空白”和“卡片文件无法从文件管理窗口拖到无限画布”进行了正式修复。

本轮修复完成后，已执行：

- 编辑引擎现有单元测试回归；
- 本次改动相关文件的 ESLint 校验；
- 编辑引擎工程的 `npx tsc --noEmit` 编译门禁检查。

前两项通过，但第三项失败，说明编辑引擎工程的正式 TypeScript 基线仍未闭环。

## 2. 阻断结论

当前 `Chips-EditingEngine` 不能宣称“正式 TypeScript 门禁可用”。虽然应用已有可运行链路，且单元测试通过，但严格 TS 编译仍存在多处失败点，覆盖：

1. Vite 运行时能力声明缺失；
2. 运行时代码中的遗留类型问题；
3. 已迁移模块中的失效引用；
4. 事件总线泛型签名未收口；
5. 现有测试文件的夹具和 mock 类型不满足严格 TS。

这属于工程基线问题，不是本次两个 UI/交互问题可以在同一回合中合法顺手收掉的“小修小补”。

```bash

npx tsc --noEmit
```

当前主要错误分组如下。

### 3.1 Vite 类型声明未接入

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/CardBoxLibrary/data.ts`

现象：

- 多处报错：`Property 'glob' does not exist on type 'ImportMeta'`

说明：

- 当前工程在源码中正式使用了 `import.meta.glob`，但 TypeScript 工程基线没有把对应的 Vite 类型声明接入进来。
- 这不是业务代码临时规避能解决的问题，应通过工程类型入口或配置正式闭环。

### 3.2 运行时代码仍有严格类型告警

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/EditPanel/EditPanel.tsx`

现象：

- `activeCard` 在使用点上仍可能为 `null | undefined`

说明：

- 编辑面板链路虽然可以运行，但严格 TS 认为当前判空边界不完整。

### 3.3 已迁移模块仍保留失效引用

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/EngineSettings/panels/FileModeSettingsPanel.tsx`

现象：

- 引用不存在的 `../../../core/state`
- 引用不存在的 `../../../types`

说明：

- 这属于迁移后的工程结构与源码引用未同步收口。
- 若继续在当前状态上叠加功能开发，会放大后续维护成本。

### 3.4 事件总线类型签名未统一

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/hooks/use-layout-switch.ts`

现象：

- 监听器参数 `({ from, to }) => void` 与事件总线当前的通用 `EventHandler<unknown>` 不匹配

说明：

- 当前实现依赖运行时约定，但静态类型层没有正式表达该契约。

### 3.5 现有测试文件不满足严格 TS

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/tests/unit/card-files.test.ts`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/tests/unit/card-window.test.tsx`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/tests/unit/plugin-host.test.tsx`

现象：

- 若干测试夹具缺少 `zIndex` 等正式字段；
- 若干回调参数仍是隐式 `any`；
- 局部表达式被推断为 `never`。

说明：

- 当前单元测试虽然能运行通过，但并不等价于“测试代码已满足严格 TS 工程标准”。

## 4. 影响范围

- `Chips-EditingEngine` 目前不能把 `tsc --noEmit` 作为正式验收门禁。
- 后续编辑引擎功能开发会在非完整类型基线下继续堆叠，增加回归风险。
- 若脚手架或其他应用工程以当前编辑引擎写法为参考，可能继续复制这些基线问题。

## 6. 期望行为

- 编辑引擎工程在保持严格 TypeScript 的前提下，直接通过：

```bash
npx tsc --noEmit
```

- Vite 类型声明、运行时代码、迁移遗留引用、事件总线类型、测试夹具类型一起闭环。
- 不允许通过以下方式临时绕过：
  - 关闭 strict；
  - 排除测试文件；
  - 大量 `as any`；
  - 本地补临时声明但不修正式工程入口。

## 7. 建议处理方向

### 7.1 工程类型入口

- 明确编辑引擎工程是否应正式接入 Vite 的 `ImportMeta` 类型声明；
- 若应接入，补齐对应类型入口并纳入脚手架/模板标准。

### 7.2 迁移遗留文件清理

- 清理 `FileModeSettingsPanel.tsx` 中对旧目录结构的失效引用；
- 对已迁移完成但未正式接线的面板代码做一次静态引用收口。

### 7.3 事件总线类型治理

- 为常用事件载荷建立明确的泛型或事件映射，而不是统一落到 `unknown` 后再在业务层假定结构。

### 7.4 测试代码类型基线

- 给现有测试夹具补齐正式配置字段；
- 修复 Vitest mock 的泛型/签名用法；
- 把测试代码也纳入正式 TS 门禁，而不是只依赖运行通过。
