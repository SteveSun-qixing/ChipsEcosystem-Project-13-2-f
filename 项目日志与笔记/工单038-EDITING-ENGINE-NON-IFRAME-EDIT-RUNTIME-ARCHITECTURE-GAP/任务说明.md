## 1. 背景

当前任务002希望对编辑引擎渲染模块做根本性重构，目标不是继续优化现有复合 iframe 链路，而是改成：

- 编辑引擎不再使用 SDK 当前交付的统一复合卡片 iframe 窗口作为正式编辑态渲染入口；
- 去掉复合卡片整体 iframe 包裹；
- 每个基础卡片单独渲染为窗口或节点；
- 由编辑引擎壳层负责拼装、选择、拖拽、布局、数据操作和交互代理；
- 最终形成更适合编辑态的复合卡片渲染架构。

用户同时要求：

- 不能做临时方案、兼容方案、占位方案；
- 必须严格按照生态正式接口、协议、主题系统和多语言系统开发；
- 如果发现生态架构缺口或 SDK/底层能力不足，必须先登记问题工单、创建任务文档并停止继续推进。

## 3. 当前正式链路现状

### 3.1 文档层正式口径

当前正式生态标准统一指向同一条链路：

1. Host 负责 L1-L9 运行时与统一渲染；
2. SDK 只做调用封装与入口交付；
3. 应用层统一通过 SDK 消费 Host 的正式显示/编辑链路；
4. 应用层不得私自实现另一套复合卡片渲染运行时。

最关键的约束来自：

- `生态共用技术文档/组件库/07-复合卡片窗口与SDK显示链路标准.md`
  - 明确要求应用层统一通过 SDK 调用；
  - 明确写明不允许应用层自行实现另一套 iframe 拼接、主题注入或兼容分支；
  - 明确写明不允许编辑引擎额外实现一套脱离 Host/SDK 的编辑器运行时。
- `生态共用技术文档/协议与接口标准/02-Bridge-API规范.md`
  - `card.render` 正式返回复合卡片渲染结果；
  - `card.renderEditor` 正式返回基础卡片编辑器文档；
  - 复合卡片和编辑器都通过 iframe 事件协议与应用壳层通信。

### 3.2 SDK 当前交付能力

`Chips-SDK/src/api/card.ts` 当前正式提供：

- `client.card.render(...)`
- `client.card.compositeWindow.render(...)`
- `client.card.editorPanel.render(...)`
- `onResize / onInteraction / onNodeSelect / onNodeError / onFatalError`
- `onReady / onChange / onError`

但这些接口的共同特征是：

- Host 返回 `view.body`；
- SDK 将其包装成 iframe；
- 应用拿到的是 iframe，而不是基础卡片节点运行时对象、场景树或可直接挂载的 React 组件。

### 3.3 Host 当前交付能力

`Chips-Host/packages/card-service/src/card-service.ts` 当前正式实现的是：

- `createChildFrameDocument()`：为基础卡片生成子 iframe 文档；
- `createCompositeDocument()`：把多个基础卡片 iframe 组装成复合卡片文档；
- `createEditorFrameDocument()`：生成基础卡片编辑器文档；
- `render()` / `renderEditor()`：返回完整 HTML 文档字符串。

这说明当前 Host 的正式交付边界仍然是“文档级运行时产物”，不是“编辑引擎可直接接管的基础卡片节点运行时”。

### 3.4 编辑引擎当前接入方式

`Chips-EditingEngine` 当前也已经接到正式链路：

- `CardWindow.tsx` 使用 `client.card.compositeWindow.render(...)`；
- `PluginHost.tsx` 使用 `client.card.editorPanel.render(...)`；
- 当前编辑引擎对正式链路的接入方式，与现行文档完全一致。

## 4. 根因判断

本次任务遇到的不是“编辑引擎还没接好现有接口”，而是“任务目标本身超出了现有正式架构边界”。

如果要正式实现“非 iframe 的编辑态复合卡片运行时”，当前至少缺少以下正式能力：

1. Host 没有向应用层交付“可组合的基础卡片节点树 / 场景树 / 运行时句柄”。
2. SDK 没有“编辑态节点运行时”入口，只有 iframe 文档入口。
3. 基础卡片插件契约当前只有：
   - `renderBasecardView`
   - `renderBasecardEditor`
   并没有“编辑态内联渲染节点”这一类正式导出。
4. 当前主题、i18n、错误隔离、节点高度、交互事件、拖拽命中全部围绕 iframe 正式链路设计，未定义应用壳层直接持有节点运行时的协议。

因此，这不是普通重构，而是新的正式架构设计议题。

## 5. 为什么当前不能继续开发

在未完成架构决议前，如果直接在编辑引擎仓内推进，会出现以下违规情况：

- 编辑引擎应用层自行实现第二套渲染运行时；
- 绕开现行 Host/SDK 正式边界；
- 需要私自重定义基础卡片插件装载方式；
- 需要私自重定义主题、多语言、拖拽、选中、错误隔离协议；
- 形成与生态共用文档并行、但未被正式承认的新标准。

这与当前仓库规则直接冲突：

- 不要自行决定改变架构设计；
- 不能开发简化版临时版；
- 要严格按照各种要求和协议开发；
- 遇到架构缺陷或生态不完善，先登记问题工单并停止当前推进。

## 6. 期望正式决议方向

在继续实现前，至少需要先明确以下问题：

### 6.1 架构归属

- 编辑态非 iframe 复合卡片运行时，最终归属 Host L9 统一渲染层，还是允许应用层持有特殊运行时？
- 如果仍归属 Host，Host 应新增什么正式动作或目标适配器？

### 6.2 插件契约

- 基础卡片插件是否需要新增第三种正式导出，例如编辑态渲染导出？
- 新导出的生命周期、事件模型、错误模型、清理机制如何定义？

### 6.3 主题与多语言

- 内联编辑态节点如何接入正式主题链？
- 是否仍由 Host 注入主题/i18n 上下文，还是转为应用层注入？

### 6.4 交互协议

- 节点选中、拖拽、插入指示、尺寸变化、输入法、滚动与缩放如何正式回传？
- 应用层和节点层之间的数据操作责任如何划分？

### 6.5 测试与门禁

- 哪些仓库需要同步补齐单测、契约测试、a11y 测试、性能基线和文档验收？

## 7. 当前结论

本工单记录的“原始标准冲突”仍然客观存在，但对当前任务而言，用户已经给出了明确决议，任务可以继续推进。

当前实施口径如下：

1. 保留 `Host + SDK` 的复合卡片显示链路；
2. `Chips-EditingEngine` 内部自建“基础卡片 iframe 组合层”、编辑器注册中心和编辑器会话管理；
3. 所有改动限制在编辑引擎仓与任务文档内；
4. 等任务方案与实现稳定后，再决定是否把该口径回写到更上层生态文档。
