# 联调修复记录：图片基础卡片编辑态无法显示已存在资源

## 1. 现象

- 在编辑引擎中，从文件管理器拖入一个复合卡片到无线画布；
- 复合卡片内包含图片基础卡片；
- 复合卡片展示态中图片可以正常显示；
- 点击进入编辑后，图片基础卡片编辑器内图片加载失败。

## 2. 影响范围

- 仓库：`Chips-EditingEngine`
- 模块：`src/editor-runtime/EditorHost.tsx`
- 受影响链路：编辑态基础卡片资源解析

## 3. 根因

- 编辑态资源读取走 `EditorHost.resolveResourceUrl(...)`；
- 当资源来自卡片根目录的已持久化文件时，早期实现会把磁盘文件重新读入内存，再转成临时 `blob:` URL 返回给基础卡片编辑器；
- 这条路径在旧卡片、特殊文件名和回切编辑场景下稳定性较差，会把“已经是正式落盘资源”的图片重新变成短生命周期运行时 URL；
- 因而出现“复合卡片预览可正常显示，但编辑器里无法稳定解析同一张图片”的分叉问题。

## 4. 修复

### 4.1 运行时修复

- 在 `Chips-EditingEngine/src/editor-runtime/EditorHost.tsx` 中调整 `resolveResourceUrl(...)` 策略；
- 对“当前编辑会话中尚未落盘的待导入资源”继续返回临时 `blob:` URL；
- 对“已经持久化在卡片根目录中的资源”改为直接返回稳定的 `file://` URL；
- 这样编辑态与复合卡片预览态对已落盘资源的解析路径保持一致，避免再次走不必要的二进制中转。

### 4.2 预览稳定性修复

- 在 `Chips-EditingEngine/src/basecard-runtime/frame-host.tsx` 中调整单卡刷新状态；
- 基础卡片首次挂载时仍显示 `loading`；
- 但当单卡已经进入 `ready` 后，后续因输入、批量导图或配置增量刷新触发的重新挂载，不再重新显示首屏 loading 遮罩；
- 旧内容在增量刷新过程中保持可见，避免复合卡片中的图片区域持续闪烁。

### 4.3 回归测试

- 在 `Chips-EditingEngine/tests/unit/plugin-host.test.tsx` 中新增回归用例；
- 覆盖“编辑器请求卡片根目录中已存在的非 ASCII 文件名图片资源”场景；
- 断言编辑态拿到的是稳定的 `file://` URL，而不是重新生成的 `blob:`。
- 在 `Chips-EditingEngine/tests/unit/basecard-frame-host.test.tsx` 中新增回归用例；
- 覆盖“单卡已经 ready 后，配置刷新不再重新显示 loading 遮罩”场景。

## 5. 验证

- 执行：

```bash
pnpm exec vitest run tests/unit/plugin-host.test.tsx tests/unit/basecard-frame-host.test.tsx
../node_modules/.bin/tsc --noEmit -p tsconfig.json
```

- 结果：相关单测与类型检查全部通过。

## 6. 当前结论

- 本次问题属于编辑引擎本地资源解析与单卡刷新稳定性细节缺陷；
- 不涉及新的生态能力缺口；
- 无需新增问题工单，可继续在当前正式链路上推进后续联调与开发。

## 7. 后续补充修复：批量新导图时部分图片编辑态 404

### 7.1 现象

- 用户在图片基础卡片中一次导入多张新图片；
- 复合卡片预览里有的图片能显示，有的图片不能显示；
- 编辑器控制台出现 `ERR_FILE_NOT_FOUND`，指向卡片根目录中的新图片文件名。

### 7.2 根因

- `EditorSessionStore.commit(...)` 在把资源导入操作提交给 `CardService.updateBasicCard(...)` 后，会立即清空当前编辑会话中的 `resourceImports`；
- 但 `CardService.persistCard(...)` 是异步执行的，真正把文件写入卡片根目录会晚于这次提交完成；
- 在这段时间窗内，编辑器若再次调用 `resolveResourceUrl(...)`，旧实现只检查编辑会话里的待导入资源，因此会错误地提前回退到 `file://卡片根目录/资源名`；
- 当批量导图较多、部分文件尚未落盘时，就会出现“同一批图片只有一部分能显示”的不稳定现象。

### 7.3 修复

- `EditPanel -> PluginHost -> EditorHost` 正式透传 `selectedBaseCard.pendingResourceImports`；
- `EditorHost.resolveResourceUrl(...)` 现在按以下顺序解析：
  1. 当前编辑会话中的待导入资源；
  2. `CardService` 当前基础卡片上仍未落盘的待导入资源；
  3. 卡片根目录中的稳定 `file://` 文件路径。
- 这样即使编辑会话已经完成提交，只要 `CardService` 还没有写盘完成，编辑器仍会继续拿到稳定可用的临时 URL。

### 7.4 回归测试

- `Chips-EditingEngine/tests/unit/plugin-host.test.tsx`
- 新增“提交后但持久化未结束时，编辑器继续通过待落盘资源返回 `blob:` 预览”用例。

### 7.5 当前结论

- 本次问题仍属于既有正式链路的实现缺陷，不是新的生态能力缺口；
- 修复后，图片基础卡片批量导图在编辑态与复合卡片预览态共享同一套稳定资源时间窗处理规则。

## 8. 后续补充修复：资源先写盘、图片列表配置未落盘

### 8.1 现象

- 用户在图片基础卡片编辑器中选择本地图片后，界面看起来“没有反应”；
- 实际卡片根目录已经写入图片文件，但 `content/<baseCardId>.yaml` 中的 `images` 仍为空；
- 结果是卡片资源目录与正式配置发生分叉，后续打开时仍显示空图片列表。

### 8.2 根因

- 图片编辑器在调用 `importResource(...)` 时，会先把资源加入 `EditorSessionStore.resourceImports`；
- 旧的 `EditorHost` 自动提交策略只要看到“存在待资源变更”，就会立刻或在普通 debounce 后提交；
- 这会让“草稿配置还没把这些图片写进 `images[*].file_path`”的中间态被错误提交；
- `CardService.updateBasicCard(...)` 因而拿到的是旧配置 `images: []`，却同时执行了资源写盘。

### 8.3 修复

- `EditorHost` 现在只有在“草稿配置相对源配置已经发生变化，并且这份草稿已经开始引用待资源变更”时，才允许对资源型变更执行提交；
- 对“只有资源队列变化、草稿配置还没跟上”的状态，自动提交与卸载 flush 都会直接跳过；
- 这样正式链路会保持单一原子语义：资源导入与配置引用必须一起进入 `CardService.updateBasicCard(...)`。

### 8.4 回归测试

- `Chips-EditingEngine/tests/unit/plugin-host.test.tsx`
- 新增“资源入队但草稿尚未引用时不得提交”的回归用例；
- 同时保留“草稿一旦引用新资源，立即提交”的用例，确保修复不影响正式资源链路。

## 9. 后续补充修复：图片布局从纵向切回网格后高度不回缩

### 9.1 现象

- 图片基础卡片先切到纵向布局；
- 再切回网格布局后，实际图片已经按网格渲染；
- 但复合卡片区域仍保持纵向布局时的长高度，底部留下大块空白。

### 9.2 根因

- `BasecardFrameHost.measureFrameHeight(...)` 旧实现同时读取 iframe 文档 `body/documentElement` 的 `offsetHeight/scrollHeight`；
- 当 iframe 先被纵向布局拉高后，文档视口本身也变高；
- 后续即使卡片真实内容已经缩短，`body/documentElement` 的尺寸仍会把旧高度带回测量结果，导致高度只能增长、无法正确回缩。

### 9.3 修复

- `BasecardFrameHost` 现在优先测量 `[data-chips-basecard-frame-root="true"]` 这个基础卡片真实内容根节点；
- 只有拿不到内容根节点时，才回退到文档级尺寸；
- 同时移除了会把 iframe 当前视口高度再次带回结果的 `body/documentElement offsetHeight` 依赖。

### 9.4 回归测试

- `Chips-EditingEngine/tests/unit/basecard-frame-host.test.tsx`
- 新增“从高布局刷新回短布局后，iframe 高度可从 420px 正常回缩到 128px”的回归断言。
