# 工单052-转换模块脚手架与共享文档口径漂移 任务说明

## 1. 背景

当前任务要求正式启动“卡片转换链路”建设，并明确提出以下约束：

1. 以 `Chips-FileConversion-Plugin` 作为对外统一文件转换编排模块；
2. 以 `Chips-CardtoHTML-Plugin`、`Chips-HtmltoPDF-Plugin`、`Chips-HtmltoImage-Plugin` 作为原子转换提供者模块；
3. 使用 `chips-scaffold-module` 初始化模块插件项目仓库；
4. 在开发前先核对生态设计原稿、生态共用技术文档、当前代码、`manifest.yaml`、测试链路；
5. 发现架构缺口、脚手架不可用、文档与代码不一致时，必须先登记工单并暂停推进。

本轮在真正开始生成 4 个转换模块仓库和相关文档前，先核对了以下内容：

- `生态设计原稿/18-文件转换系统设计.md`
- `生态共用技术文档/插件开发/05-模块插件开发.md`
- `生态共用技术文档/协议与契约/06-模块能力契约.md`
- `Chips-Scaffold/chips-scaffold-module/templates/module-standard/*`
- `Chips-Scaffold/chips-scaffold-module/tests/.tmp-e2e/module-standard-project/*`
- `Chips-Host/src/main/services/module-runtime.ts`
- `Chips-SDK/src/api/module.ts`

核对结果显示：当前模块插件“公共正式文档”与“脚手架实际生成基线”并未收口，已经直接影响本次转换模块链路的仓库初始化与文档冻结工作。

## 2. 证据

### 2.1 公共文档中的模块入口口径

`生态共用技术文档/插件开发/05-模块插件开发.md` 当前示例：

- Manifest 示例使用 `entry: dist/module.cjs`
- 工程结构示例使用 `src/module.ts`

`生态共用技术文档/协议与契约/06-模块能力契约.md` 当前示例：

- Manifest 示例同样使用 `entry: dist/module.cjs`

这些文档属于生态共用技术文档，按工作区规则会被其他模块仓库直接视为正式公共口径。

### 2.2 脚手架模板中的实际生成口径

`Chips-Scaffold/chips-scaffold-module/templates/module-standard/manifest.yaml.tpl` 当前模板：

- 生成 `entry: dist/index.mjs`

`Chips-Scaffold/chips-scaffold-module/templates/module-standard/src/index.ts.tpl` 当前模板：

- 实际源码入口文件为 `src/index.ts`

`Chips-Scaffold/chips-scaffold-module/tests/.tmp-e2e/module-standard-project/README.md` 当前生成工程 README：

- 也把 `manifest.yaml` 必须保持 `entry: dist/index.mjs` 作为正式约束

这说明脚手架交付出来的仓库基线，与共享文档对外发布的正式示例已经不一致。

### 2.3 转换系统设计原稿索引漂移

`生态设计原稿/00-文档索引.md` 当前把“文件转换系统设计”列为编号 25，但实际对应文件是：

- `生态设计原稿/18-文件转换系统设计.md`

这虽然不是代码层阻断，但会直接影响本次任务要求的“先读设计原稿，再按正式原稿更新共享文档和项目文档”的阅读路径。

## 3. 为什么这会阻断当前任务

本次任务不是单纯补一两页说明，而是要：

1. 用模块插件脚手架初始化 4 个正式转换模块仓库；
2. 为每个仓库建立 README、需求文档、技术文档、开发计划；
3. 同步更新生态设计原稿与生态共用技术文档；
4. 最终以这些文档作为后续开发、测试和验收基线。

如果在当前状态下继续推进，我必须自行决定以下事项：

1. 4 个转换模块的 `manifest.yaml` 入口到底写 `dist/index.mjs` 还是 `dist/module.cjs`；
2. 仓库源码入口到底写 `src/index.ts` 还是 `src/module.ts`；
3. 仓库 README 与项目内部技术文档到底跟脚手架模板走，还是跟共享文档走；
4. 后续对 `card -> html`、`html -> pdf`、`html -> image` 的 capability 契约文档应该引用哪一套模块工程基线。

这些都不是任务内可以自行拍板的“实现细节”，而是生态公共基线。如果我现在直接选一套写进去，会把新的不一致继续扩散到 4 个转换模块仓库和共享文档中，违反当前工作区规则。

## 4. 影响范围

### 4.1 直接影响

- `Chips-FileConversion-Plugin`
- `Chips-CardtoHTML-Plugin`
- `Chips-HtmltoPDF-Plugin`
- `Chips-HtmltoImage-Plugin`

这 4 个仓库当前都还没有正式内容。一旦用错误或未冻结的基线初始化，后面所有文档、测试、打包说明和实现入口都会跟着错。

### 4.2 连带影响

- 生态共用技术文档中的模块插件开发指南会继续与脚手架产物冲突；
- 后续转换链路开发计划文档无法判断应以哪套模块工程结构为主；
- 新增转换模块的 README 和仓库内部文档会在生成当日就和共享文档分叉；
- 设计原稿索引对转换系统的导航失准，会继续影响阅读和引用。

## 5. 建议收口顺序

建议先完成以下收口，再重启当前转换链路任务：

1. 统一模块插件正式工程基线：
   - `manifest.yaml` 正式入口产物路径；
   - 源码入口文件命名；
   - README 与共享文档示例口径。
2. 同步更新：
   - `生态共用技术文档/插件开发/05-模块插件开发.md`
   - `生态共用技术文档/协议与契约/06-模块能力契约.md`
   - `Chips-Scaffold/chips-scaffold-module/templates/module-standard/*`
   - 脚手架生成 README 示例
3. 修正：
   - `生态设计原稿/00-文档索引.md`
4. 基线收口后，再重新开始：
   - 4 个转换模块仓库初始化
   - 转换链路需求文档 / 技术文档 / 开发计划编写
   - 生态设计原稿与生态共用技术文档的转换链路正式补全
