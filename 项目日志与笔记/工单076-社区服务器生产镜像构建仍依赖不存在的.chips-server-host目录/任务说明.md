# 工单076-社区服务器生产镜像构建仍依赖不存在的.chips-server-host目录

## 背景

当前任务需要把最新的 `Chips-CommunityPlatformServer` 正式部署到远程 1Panel 服务器。

按工作区规则，部署前重新核对了：

- `生态设计原稿/20-社区服务器架构.md`
- `生态共用技术文档/协议与契约/09-社区平台API契约.md`
- `Chips-CommunityPlatformServer/技术文档/07-启动与运行说明.md`
- `Chips-CommunityPlatformServer/deploy/docker-compose.yml`
- `Chips-CommunityPlatformServer/deploy/Dockerfile.server`
- `Chips-CommunityPlatformServer/packages/server/src/services/host-integration.ts`

复核后确认：当前仓库的服务端生产镜像构建脚本仍然依赖一个仓库里并不存在的运行时目录，导致“按 Git 提交直接远程构建正式镜像”不能视为完全收口。

## 证据

### 1. `Dockerfile.server` 生产阶段复制了不存在的路径

`Chips-CommunityPlatformServer/deploy/Dockerfile.server` 当前生产阶段包含：

```dockerfile
COPY Chips-CommunityPlatformServer/packages/server/.chips-server-host ./packages/server/.chips-server-host
```

### 2. 仓库里不存在该目录

当前仓库中：

- `Chips-CommunityPlatformServer/packages/server/.chips-server-host/`

并不存在。

### 3. 该目录本质上是运行时工作区

`Chips-CommunityPlatformServer/packages/server/src/services/host-integration.ts` 当前实现会在服务启动时：

1. 以 `process.cwd()` 下的 `.chips-server-host` 作为工作区；
2. 启动前先 `rm -rf` 清空；
3. 再 `mkdir` 重建；
4. 之后由 Host 集成层动态安装插件与主题。

这说明 `.chips-server-host` 属于运行时临时工作区，而不是应当由仓库静态携带进生产镜像的发布资产。

## 影响范围

### 直接影响

1. 使用当前 `Dockerfile.server` 生产阶段直接构建社区服务器镜像时，会因为源路径不存在而失败；
2. 远程服务器若按当前 Git 提交直接执行正式 Docker 构建，部署链路会在镜像构建阶段中断；
3. 该问题会让“正式 Docker 全栈模式可直接从仓库提交构建”这一口径失真。

### 波及范围

1. `Chips-CommunityPlatformServer/deploy/Dockerfile.server`
2. `Chips-CommunityPlatformServer/deploy/docker-compose.yml` 中 `server` 服务的正式生产镜像链路
3. 所有基于仓库提交的远程自动化部署流程

## 复现步骤

1. 打开 `Chips-CommunityPlatformServer/deploy/Dockerfile.server`；
2. 查看生产阶段对 `.chips-server-host` 的 `COPY` 指令；
3. 检查 `Chips-CommunityPlatformServer/packages/server/` 目录；
4. 确认该目录并不存在；
5. 打开 `Chips-CommunityPlatformServer/packages/server/src/services/host-integration.ts`；
6. 确认 `.chips-server-host` 会在运行时被清空并重新创建。

## 期望行为

1. 服务端正式镜像构建不再依赖仓库中不存在的运行时目录；
2. `Dockerfile.server` 需要去掉对 `.chips-server-host` 的静态复制，或改为启动时动态准备；
3. 修复后重新验证：
   - 服务端正式镜像构建
   - Compose 全栈部署
   - Host 集成层启动与插件安装

## 暂行处理

1. 已登记本工单；
2. 本轮远程部署不直接依赖该有缺陷的 `production` 阶段服务镜像；
3. 改为使用同仓库 `Dockerfile.server` 的可构建阶段镜像运行服务端，以保持远程部署继续推进，同时不修改仓库源码。
