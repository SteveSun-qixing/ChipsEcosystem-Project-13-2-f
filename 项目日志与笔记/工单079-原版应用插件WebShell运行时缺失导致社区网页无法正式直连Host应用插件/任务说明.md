# 工单079-原版应用插件WebShell运行时缺失导致社区网页无法正式直连Host应用插件

## 1. 背景

在 task011 中，用户明确要求：

1. 不接受临时版网页查看器；
2. 希望社区网页点击卡片图片后，直接调用对应的原版应用插件；
3. 整个链路必须按 Host 统一运行时设计落地，而不是在社区前台重做一套页面逻辑；
4. 需要继续补齐 Web Shell 等底层能力，满足长期正式架构。

在继续把社区前台图片查看链路从“临时 Web 页面”收口为“原版 `Chips-PhotoViewer` 插件在浏览器里正式运行”时，确认当前生态底层并未具备所需运行时条件。

## 2. 证据

### 2.1 Host 当前没有 Web Shell

现有 shell 只有：

- `Chips-Host/src/main/shells/desktop-host-shell.ts`
- `Chips-Host/src/main/shells/headless-host-shell.ts`

不存在：

- `web-host-shell.ts`
- browser pal adapter
- browser surface runtime

### 2.2 Bridge 文档写了“预留”，代码没有实现

共享文档中明确写了 Browser Worker transport / WebView native bridge 为预留：

- `生态共用技术文档/架构设计/13-Bridge三层设计.md`

但代码层当前只看到：

- Electron preload transport
- Direct transport

相关证据：

- `Chips-SDK/src/core/bridge-adapter.ts`
- `Chips-Host/src/preload/create-bridge.ts`

### 2.3 原版应用插件 manifest 当前明确不支持 Web

图片查看器：

- `Chips-PhotoViewer/manifest.yaml`

卡片查看器：

- `Chips-CardViewer/manifest.yaml`

都写着：

```yaml
runtime:
  targets:
    web:
      supported: false
```

这意味着当前原版插件并没有被正式声明为可在 Web Shell 中运行。

### 2.4 原版 PhotoViewer 运行时强依赖 Host Bridge

关键代码：

- `Chips-PhotoViewer/src/hooks/useChipsBridge.ts`
- `Chips-PhotoViewer/src/hooks/useChipsClient.ts`
- `Chips-PhotoViewer/src/utils/launch-resource.ts`

表现为：

1. `window.chips` 缺失就直接报错；
2. 依赖 `client.resource.resolve(...)`；
3. 依赖 `client.platform.getLaunchContext()`；
4. 依赖 `platform.openFile / saveFile`；
5. 依赖主题和多语言事件。

社区前台当前并不能提供这些能力。

### 2.5 社区前台按文档本就不依赖 Host Runtime

现有边界文档写明：

- `Chips-CommunityPlatformServer/技术文档/01-重构基线与边界.md`

其中社区公开前端被定义为独立 Web 体系，不直接依赖 Host、Bridge、`window.chips` 或 Host 运行时。

这说明如果要转向“原版插件直跑 Web”，必须先改造生态底层边界本身，而不是继续在页面层做修补。

## 3. 影响范围

### 3.1 Host

缺失：

1. `WebHostShell`
2. Browser PAL adapter
3. browser-side `surface(route/modal/tab)` 正式承载
4. browser-side plugin session handshake
5. browser-side `window.chips` 注入链路

### 3.2 SDK

缺失：

1. 浏览器环境下对 Web Host transport 的正式适配；
2. 无 `window.chips` 时，当前只会抛出 `BRIDGE_UNAVAILABLE`，不能作为 Web Host runtime 工作。

### 3.3 官方应用插件

缺失：

1. `runtime.targets.web.supported = true`
2. Web 场景下的正式能力矩阵与降级行为治理

### 3.4 社区前台

当前不能通过正式生态链路直接运行 app 插件，只能重做页面来模拟结果，这与用户要求冲突。

## 4. 复现步骤

1. 查看 `Chips-Host/src/main/shells/`
   - 只存在 `desktop-host-shell.ts` 与 `headless-host-shell.ts`
2. 查看 `Chips-SDK/src/core/bridge-adapter.ts`
   - 浏览器里没有 `window.chips` 时，直接抛 `BRIDGE_UNAVAILABLE`
3. 查看 `Chips-PhotoViewer/manifest.yaml`
   - `runtime.targets.web.supported = false`
4. 查看 `Chips-CardViewer/manifest.yaml`
   - `runtime.targets.web.supported = false`
5. 查看 `Chips-PhotoViewer/src/hooks/useChipsBridge.ts`
   - `window.chips` 不存在即报错

## 5. 期望行为

需要新增正式生态底层能力：

1. `WebHostShell`
2. browser PAL adapter
3. browser transport / in-browser bridge host
4. browser-side `surface` 映射
   - route
   - modal
   - tab
5. browser plugin session/runtime handshake
6. 社区前台对 app 插件 surface 的正式承载
7. `Chips-PhotoViewer` / `Chips-CardViewer` 切换为 `web.supported: true`

完成之后，用户点击卡片图片时，才可以：

1. 由卡片内容发出正式 `resource.open` 意图；
2. Host Web runtime 匹配原版图片查看器插件；
3. 通过 `surface.open(target=plugin)` 打开原版插件的 Web surface；
4. 原版 `Chips-PhotoViewer` 在浏览器中运行，保持原 UI 与原交互。

## 6. 处理结果

本工单对应的社区服务器正式链路现已落地完成，不再继续使用临时网页查看器。

已完成事项：

1. `Chips-CommunityPlatformServer/packages/server/src/routes/host-runtime.ts`
   - 新增正式 Web 插件会话路由；
   - 新增插件入口 HTML 注入与 `bootstrap.js` 输出；
   - 新增插件静态资源分发；
   - 新增 Web 资源打开规划接口。
2. `Chips-CommunityPlatformServer/packages/server/src/services/host-integration.ts`
   - 新增 `openWebPluginSession`
   - 新增 `getWebPluginSession`
   - 新增 `getWebPluginEntry`
   - 新增 `resolveWebPluginAssetPath`
   - 新增 `closeWebPluginSession`
   - 新增 `resolveWebResourceOpenPlan`
3. `Chips-CommunityPlatformServer/packages/web`
   - 新增 `HostedPluginSurface`
   - 新增 `HostedPluginSessionPage`
   - 新增 `host-runtime.ts`
   - `CardDetailPage` 已切换为原版 `CardViewer` Web 会话承载；
   - 正式路由改为 `/host/plugins/:sessionId`。
4. 原版应用插件
   - `Chips-CardViewer/manifest.yaml` 已开启 `runtime.targets.web.supported = true`
   - `Chips-PhotoViewer/manifest.yaml` 已开启 `runtime.targets.web.supported = true`
   - `CardViewer` 已支持通过 `launchParams.webDocumentUrl` 恢复托管文档查看态
   - `PhotoViewer` 已支持通过 `launchParams.resourceOpen` 恢复 Web 图片查看态
5. 临时实现
   - 临时图片查看页已移入归档，不再参与正式编译。

## 7. 当前真实边界

1. 本工单已解决的是“社区服务器无法正式直连原版应用插件”的问题；
2. 当前正式落地点是社区服务器内部 Web 插件宿主链路；
3. `chips-host` 作为独立可复用包的通用 `WebHostShell` 公开导出，本轮没有单独交付为独立包级入口；
4. 这不影响社区前台通过正式链路运行原版 `CardViewer / PhotoViewer`。

## 8. 当前状态

- 状态：已解决（社区服务器正式 Web 插件宿主链路）
