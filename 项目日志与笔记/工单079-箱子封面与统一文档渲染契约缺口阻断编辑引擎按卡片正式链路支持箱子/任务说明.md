# 工单079-箱子封面与统一文档渲染契约缺口阻断编辑引擎按卡片正式链路支持箱子

## 背景

本轮任务原始目标是改造 `Chips-EditingEngine` 中箱子的交互方式，让箱子完全参考卡片的使用与交互方式，并且不要再沿“卡片一条渲染链、箱子一条渲染链”继续演化，而是直接优化现有卡片渲染接口和相关模块，使其正式支持箱子文件。

用户同时明确提出了几个关键要求：

1. 箱子需要完全参考卡片链路；
2. 箱子需要专门的渲染组件，但不能演化为第二套旁路管线；
3. 箱子也要有封面，且“封面和卡片的封面是完全一样的”；
4. 箱子嵌套到箱子时，内部箱子显示封面；
5. 编辑面板需要正式承载箱子的配置与内容列表编辑。

## 核对结论

继续核对设计原稿、共享技术文档、Host、SDK 与编辑引擎现有实现后，确认当前生态正式契约还没有闭环到可以直接按上述要求实施：

### 1. 箱子本体没有卡片同等级的正式封面入口

当前共享协议与格式文档中，卡片封面明确是：

- `.card/cover.html`
- `card.renderCover`
- `CardCoverFrame`

但箱子当前正式结构仍是：

- `.box/metadata.yaml`
- `.box/structure.yaml`
- `.box/content.yaml`
- `assets/`

并且只定义了 `cover_asset` / `coverAsset`，没有箱子版 `cover.html` 或等价正式封面文档入口。

### 2. Host / SDK 只有“箱子条目封面”，没有“箱子本体封面”

当前箱子消费正式链路里存在的是：

- `box.renderEntryCover`

它服务的是布局插件渲染条目封面，不是箱子本体封面。

当前缺失的是：

- `box.renderCover`
- 统一文档封面渲染接口
- 可直接复用 `CardCoverFrame` 同级能力的箱子本体封面渲染契约

### 3. 编辑引擎的正式编辑目标模型还是“卡片 / 基础卡片”

当前编辑引擎正式状态与编辑面板链路是：

- `CardContext.activeCardId`
- `CardContext.selectedBaseCardId`
- `EditPanel -> PluginHost -> EditorHost`

这条链路只覆盖“卡片窗口 + 基础卡片编辑器”，还没有“统一文档目标 = 卡片 | 箱子”的正式抽象。

如果在当前基础上直接把箱子编辑逻辑继续塞进 `BoxWindow`，虽然能做出界面，但本质上还是第二套旁路。

## 影响范围

如果不先补齐生态契约，继续直接写编辑引擎实现会带来以下问题：

1. 箱子封面口径会只存在于编辑引擎本地，无法成为整个生态共享正式行为；
2. 箱子嵌套、箱子列表、未来查看器或其他应用会再次出现第二套封面逻辑；
3. 编辑引擎内部会继续保留“卡片正式链路 + 箱子专用旁路”两条管线，和本轮用户要求相冲突；
4. 后续 Host / SDK 一旦补齐正式接口，当前本地实现大概率还要被推翻重做。

## 期望行为

需要先明确并收口以下正式内容：

1. 箱子本体正式封面的文件格式与运行时入口；
2. 统一文档渲染与编辑目标模型，明确卡片与箱子如何复用同一条正式链路；
3. 箱子嵌套时的正式封面消费口径；
4. 编辑引擎、Host、SDK、共享文档的一致收口顺序。

## 处理结果

当前已按“不要两条管线、直接完善现有卡片正式链路”的要求完成正式收口：

1. `Chips-Host` 已把箱子本体正式封面收口为 `.box/cover.html`，并新增 `box.renderCover`；
2. `Chips-SDK` 已同步 `box.renderCover`、`documentInfo / documentFile / document-window` 等正式契约；
3. `Chips-ComponentLibrary` 已抽象出通用 `EmbeddedDocumentFrame`，`CardCoverFrame` 改为其卡片封面专用包装；
4. `Chips-BoxLayoutPlugin` 已按新正式模型完成网格箱子编辑器与浏览器重构；
5. `Chips-EditingEngine` 已把箱子窗口收口为正式预览窗口，并把箱子编辑迁移到右侧编辑面板；
6. 相关共享文档与设计原稿已同步更新。

## 当前状态

本工单已闭环，不再阻断箱子按卡片正式链路接入编辑引擎。
