# 工单081 任务说明

## 工单名称

卡片校验接口文档与 SDK 实现漂移阻断箱子卡片统一链路审查

## 背景

本轮正在执行 `task015-审查箱子卡片链路`。用户要求调查：

1. 箱子相关模块和链路；
2. 箱子布局卡片与基础卡片是否共用同一条正式链路；
3. 所有相关技术文档、调用接口和架构设计；
4. 两者的差异点与一致点。

在进入 Host / SDK 代码层核查时，发现卡片正式接口 `card.validate` 已经出现共享文档、SDK 需求文档、SDK 实现、Host 路由四处不一致。这会直接影响对“卡片/箱子相关接口口径是否统一”的调查可信度，因此必须先登记工单。

## 证据

### 1. 共享系统接口标准口径

`生态共用技术文档/协议与接口标准/05-系统接口标准.md`

当前写法是：

1. `card.validate({ cardFile })`
2. 输入是卡片文件路径

### 2. SDK 使用文档口径

`生态共用技术文档/插件开发/08-SDK使用指南.md`

当前写法是：

1. `client.card.validate(cardDoc: CardDocument): Promise<ValidationResult>`
2. 输入是文档对象

### 3. SDK 需求文档口径

`Chips-SDK/需求文档/02-薯片 SDK 功能需求规格.md`

当前写法同样是：

1. `client.card.validate(cardDoc: CardDocument): Promise<ValidationResult>`

### 4. SDK 实际实现

`Chips-SDK/src/api/card.ts`

当前实现是：

1. `validate(card)`
2. 通过 `client.invoke("card.validate", { card })` 调用 Host

### 5. Host 实际路由

`Chips-Host/src/main/services/register-host-services.ts`

当前实现是：

1. `card.validate` 路由描述符接收 `{ cardFile: string }`
2. 内部调用 `ctx.getCardService().validate(input.cardFile)`

## 影响范围

1. `Chips-SDK/src/api/card.ts`
2. `Chips-Host/src/main/services/register-host-services.ts`
3. `生态共用技术文档/协议与接口标准/05-系统接口标准.md`
4. `生态共用技术文档/插件开发/08-SDK使用指南.md`
5. `Chips-SDK/需求文档/02-薯片 SDK 功能需求规格.md`
6. `task015-审查箱子卡片链路` 中关于卡片 / 箱子接口口径一致性的结论

## 复现步骤

1. 打开 `生态共用技术文档/协议与接口标准/05-系统接口标准.md`，核对 `card.validate` 的入参定义；
2. 打开 `生态共用技术文档/插件开发/08-SDK使用指南.md`，核对 SDK 方法签名；
3. 打开 `Chips-SDK/需求文档/02-薯片 SDK 功能需求规格.md`，核对需求文档签名；
4. 打开 `Chips-SDK/src/api/card.ts`，核对 `validate(card)` 的实际调用；
5. 打开 `Chips-Host/src/main/services/register-host-services.ts`，核对 `card.validate` 的 Host 正式路由入参与处理逻辑。

## 期望行为

1. `card.validate` 只能存在一套正式口径；
2. Host、SDK、共享文档、SDK 需求文档必须完全一致；
3. 在正式口径收口前，不能把这条接口继续当作稳定依据去推导“卡片与箱子接口层是否统一”。
