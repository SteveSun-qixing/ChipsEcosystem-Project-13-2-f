# 工单087 任务说明

## 工单标题

resource-readBinary 返回包装对象阻断音乐播放器嵌入封面链路

## 背景

在继续修复 `Chips-MusicPlayer` “导入 MP3 后显示嵌入专辑封面，并把提取出的 PNG 临时保存到 Host 工作区”时，先按工作区要求复核了设计原稿、共享文档和真实代码链路。

排查过程中确认：

1. 真实测试音频 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/ProductFinishedProductTestingSpace/测试音频.mp3` 的确包含嵌入式专辑封面；
2. 直接用 `Chips-MusicPlayer/src/utils/audio-metadata.ts` 解析磁盘字节时，可以得到：
   - `title = "Mine or Yours"`
   - `artist = "宇多田ヒカル"`
   - `album = "Mine or Yours"`
   - `artwork.mimeType = "image/tiff"`
3. 但播放器真实运行链路不是直接读磁盘字节，而是通过 `client.resource.readBinary(...)` 取二进制，再交给 `parseEmbeddedAudioMetadata(...)`。

继续核对后发现，真正阻断链路的是共享接口本身：

- SDK 类型和共享文档把 `client.resource.readBinary(...)` 描述成“返回裸二进制”；
- Host 实际却返回 `{ data: Buffer }`；
- 播放器当前按“裸二进制”口径使用，因此嵌入元数据与封面整体解析失败。

这已经超出单一业务仓问题，属于正式共享接口未收口。

## 证据

### 1. SDK 类型口径

- 文件：`Chips-SDK/src/api/resource.ts`
- 当前声明：
  - `readBinary(resourceId: string): Promise<ArrayBuffer>;`
  - 实现：`return client.invoke("resource.readBinary", { resourceId });`

这意味着 SDK 对调用方公开的语义是“拿到裸二进制”。

### 2. Host 实际返回口径

- 文件：`Chips-Host/src/main/services/register-host-services.ts`
- 当前 `resource.readBinary` 路由实际返回：

```ts
return { data };
```

也就是包装对象，而不是裸二进制。

### 3. 真实链路验证结果

使用真实 `HostApplication + createClient({ transport })` 调用：

```ts
await client.resource.readBinary(".../测试音频.mp3")
```

实际得到的结果对象形状为：

```ts
{
  data: <Buffer ...>
}
```

而不是 `ArrayBuffer` / `Uint8Array`。

### 4. 业务仓当前使用方式

- 文件：`Chips-MusicPlayer/src/App.tsx`
- 当前逻辑：
  - 调用 `client.resource.readBinary(metadataReadTarget)`
  - 再把返回值直接传给 `parseEmbeddedAudioMetadata(...)`

由于这里按“裸二进制”使用，而真实返回却是包装对象，最终会导致解析结果为空。

## 影响范围

### 直接影响

- `Chips-MusicPlayer` 无法稳定读取 MP3/FLAC/M4A 等文件中的嵌入标题、歌手、专辑、歌词和封面；
- 当前这条链路也会阻断“把嵌入封面提取为 PNG 并保存到 Host 工作区”的后续正式开发。

### 潜在影响

- 任何依赖 `client.resource.readBinary(...)` 获取裸字节的生态内应用插件；
- 共享文档、SDK 类型、Host 返回值三方继续漂移会扩大调用方的不确定性；
- 如果直接在业务仓兼容 `.data` 字段，会把错误契约写死进业务代码，后续正式修复共享接口时又会产生反向兼容负担。

4. 共享契约收口后，再继续完成音乐播放器的嵌入封面 TIFF -> PNG -> Host 工作区临时文件链路。

## 处理结果

1. `Chips-SDK/src/api/resource.ts` 已完成正式收口：
   - `resource.readBinary(...)` 对调用方继续返回裸 `ArrayBuffer`；
   - SDK 内部新增二进制载荷归一化，统一支持：
     - `ArrayBuffer`
     - `ArrayBufferView`
     - `{ data: Buffer }`
     - `{ data: { type: "Buffer", data: number[] } }`
2. `Chips-SDK/tests/resource.test.ts` 已补充桥接序列化测试，明确验证 SDK 会把嵌套 Buffer JSON 解包回裸字节。
3. `Chips-MusicPlayer/src/App.tsx` 已恢复按正式裸字节解析嵌入音频元数据，不再依赖错误的包装对象形状。
4. `Chips-MusicPlayer/src/utils/embedded-artwork-cache.ts` 已接入 Host 工作区 PNG 落盘链路：
   - 缓存目录：`<workspacePath>/.music-player-artwork-cache`
   - PNG：直接写入缓存目录
   - TIFF：先写临时 `.tiff`，再调用正式 `client.resource.convertTiffToPng(...)` 产出 PNG，最后删除临时 TIFF
   - 其他不稳定格式：运行时转码 PNG 后再落盘
5. `Chips-MusicPlayer` 已保留正式回退策略：仅当 Host 工作区 PNG 落盘失败或当前没有 `workspacePath` 时，才回退为运行时 `blob:` 封面 URL。
6. 共享文档与 SDK 内部技术文档已同步更新，避免后续业务仓再次读取 `.data` 做错误兼容。
