# 工单088 任务说明

## 工单标题

模块颜色选取器缺少正式图像像素解码与分析能力

## 背景

用户要求在 `Chips-ModulePlugin/Chips-ColorPicker` 中建设一个可供整个生态复用的颜色选取器模块，并明确提出：

1. 模块接收一张图片；
2. 输出两个颜色代码；
3. 其中一个颜色要浓郁、偏暗、适合作为纯色背景；
4. 另一个颜色要高亮度、高饱和度、适合作为个性色 / 强调色；
5. 两个颜色都要尽量代表整张图片的整体气质，而不是简单平均色。

## 证据

### 1. 当前正式图像公共能力只覆盖原始字节和 TIFF 归一化

- `生态共用技术文档/插件开发/08-SDK使用指南.md`
  - 只定义了 `client.resource.readBinary(...)`，语义是读取原始二进制。
- `生态共用技术文档/协议与契约/11-资源图像转换契约.md`
  - 只冻结了 `resource.convertTiffToPng`。
- `生态共用技术文档/架构设计/18-资源图像转换模块设计.md`
  - 当前图像能力仍定位在 Host `resource` 服务中的 TIFF -> PNG 转换，不包含通用像素解码或颜色分析。

结论：

- 当前正式口径没有：
  - `resource.decodeImage(...)`
  - `resource.readPixels(...)`
  - `resource.sampleImage(...)`
  - `module.image.palette.extract(...)`
  - 或任何等价的公共能力。

### 2. 模块插件正式上下文无法直接获得像素矩阵

根据：

- `生态共用技术文档/插件开发/05-模块插件开发.md`
- `生态共用技术文档/协议与契约/06-模块能力契约.md`

模块插件运行时正式只能通过：

- `ctx.host.invoke(...)`
- `ctx.module.invoke(...)`
- `ctx.job.*`

访问公共能力。

当前这些正式入口中，没有已经冻结的图像像素读取 / 颜色分析动作，因此模块内无法通过正式链路直接取得跨平台一致的像素矩阵。

### 3. 当前工作区没有已冻结的正式图像解码依赖基线

在生态根工作区核对现有依赖后确认：

1. 未发现已正式接入并可直接复用的 `sharp`、`jimp`、`pngjs`、`jpeg-js`、`node-vibrant`、`canvas` 等图像解码能力；
2. 执行：

```bash
node -e "try{console.log(require.resolve('canvas'))}catch(e){console.error('NO_CANVAS');process.exit(1)}"
```

得到：

```text
NO_CANVAS
```

3. 按工作区根 `AGENTS.md` 要求，当前不能在未获得确认的情况下擅自新增第三方图像处理依赖。

### 4. 颜色选取器不能接受简化实现

当前任务要求的结果是“高质量、智能、复杂也没关系，只追求效果”，并明确参考豆瓣、QQ 音乐等成熟设计场景。

这意味着不能用以下方式凑合：

- 平均色；
- 只取中心点颜色；
- 只按固定几何区域取样；
- 只支持单一图片格式；
- 写一套无法跨平台复用的私有命令链路。

因此该问题确实阻断正式开发，不是简单的工程排期问题。

## 影响范围

### 直接影响

- `Chips-ModulePlugin/Chips-ColorPicker` 当前只能停留在脚手架初始化状态；
- 本轮无法继续定义可信的正式输入 / 输出契约与实现细节。

### 生态影响

- `Chips-MusicPlayer` 的专辑封面取色；
- 音乐基础卡片的背景色提取；
- 作品卡片 / 海报卡片的背景色与强调色提取；
- 未来所有需要“从图片统一提取品牌色 / 背景色 / 个性色”的应用和插件。

如果不先收口公共能力，各业务仓很容易分别引入不同图像库或私有算法，形成：

- 输出风格不一致；
- 依赖治理混乱；
- 文档与代码再次漂移；
- 跨平台行为不可控。
