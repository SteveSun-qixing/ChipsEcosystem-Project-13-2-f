# 大任务001：重构模块插件全链路需求文档

## 1. 背景

当前生态中“模块插件”相关口径同时存在以下问题：

- Host、SDK、模块脚手架和文档长期围绕 `slot + mount + React runtime` 展开，默认把模块理解为可嵌入页面的前端运行时；
- 现有模块链路偏向“页面插槽挂载治理”，并没有形成“Host 托管无界面能力模块”的正式能力架构；
- 应用插件、卡片插件、布局插件如何稳定调用模块插件，缺少统一的 Host 级能力发现、契约校验、调用调度和任务治理模型；
- 模块之间可否互相调用、如何调用、如何编排、如何审计，缺少正式收口；
- 官方模块脚手架默认生成 React、主题、多语言和 DOM 挂载模板，与“无界面功能模块”的目标相冲突。

本次大任务目标不是修补旧方案，而是完成模块插件全链路重构的正式文档冻结，为后续开发提供唯一正确口径。

## 2. 目标

完成一套面向生态长期演进的模块插件正式方案，使模块插件最终具备以下特征：

- 安装到 Host 中；
- 不创建窗口，不承担界面展示职责；
- 作为 Host 托管的无界面能力模块存在；
- 通过正式能力契约为 Host、应用插件、卡片插件、布局插件以及其他模块插件提供可复用能力；
- 由 Host 统一负责发现、选择、校验、调度、隔离、审计和生命周期治理；
- 最终打通模块插件从创建、安装、注册、调用、组合到卸载的完整链路。

## 3. 范围

### 3.1 本次纳入范围

- 模块插件的正式定位、术语、能力边界；
- 模块插件的 Manifest 结构与能力声明方式；
- Host 模块注册表、能力解析、调用调度、任务治理、生命周期治理口径；
- SDK 对模块能力的正式调用方式；
- 模块插件之间相互调用的正式规则；
- 模块插件脚手架的正式基线；
- 模块插件在生态中的测试、验证和验收要求；
- 生态设计原稿、生态共用技术文档和项目级任务文档的统一收口。

### 3.2 本次不纳入范围

- 模块市场、签名分发、云端仓库等发布平台能力；
- 历史模块方案兼容桥、回退机制和双轨运行策略；
- 为旧 `slot + mount` UI 模块链路保留外部正式口径；
- 对未纳入本轮范围的业务模块做逐个实现。

## 4. 核心原则

### 4.1 模块只有一种

模块插件只有一种正式类型：Host 托管的无界面能力模块。

不单独设计“工作流模块”“转换模块”“渲染模块”等特殊模块类型。不同业务形态只体现在 capability 和 method 上，不体现在插件类型上。

### 4.2 Host 统一治理

模块插件的发现、解析、权限、校验、调度、任务、日志、审计和生命周期必须由 Host 统一治理。应用插件不得直接加载模块源码，也不得自行实现第二套模块装载器。

### 4.3 契约优先

应用插件和其他模块插件调用模块能力时，必须依赖正式能力契约，而不是依赖某个模块仓库 README 中的临时说明。Host 必须基于正式契约完成参数校验和调用调度。

在职责分工上，Host 负责统一调用治理框架，模块插件负责业务 capability / method / schema 契约，调用方负责按正式契约和 SDK 类型接入；运行时由 Host 负责解析 provider 并转发请求。

### 4.4 模块间调用必须走 Host

模块之间允许相互调用，但必须走 Host 提供的正式模块服务链路，禁止模块之间直接 `import` 调用。

### 4.5 无界面优先

官方模块脚手架默认不得生成 React、主题注入、多语言 UI 词典、DOM 挂载逻辑。模块能力可以服务于 UI 场景，但模块自身不是 UI 运行时。

### 4.6 不做兼容设计

本轮不保留旧 `module.mount/unmount/query/list` 页面挂载口径作为未来正式外部协议，不设计回退机制，不引入兼容性包袱。

## 5. 相关方

| 角色 | 关注点 |
|---|---|
| Host 维护者 | 模块注册、能力解析、权限治理、运行时隔离、任务调度 |
| SDK 维护者 | 模块能力调用 API、类型定义、错误归一、脚手架接入 |
| 官方应用插件维护者 | 如何稳定依赖模块能力、如何声明需求、如何接收结果 |
| 模块插件开发者 | 如何声明能力、实现方法、编写契约、打包发布 |
| 架构设计维护者 | 文档口径统一、长期扩展性、命名和分层边界 |

## 6. 功能需求

### 6.1 插件治理

- `FR-01` 模块插件必须继续使用统一插件治理链路安装到 Host 中，支持 install、enable、disable、uninstall。
- `FR-02` 模块插件启用后，Host 必须将其识别为 `type: module` 的正式能力提供者，而不是页面插槽挂载物。
- `FR-03` 模块插件卸载、禁用或失效时，Host 必须同步回收其 provider 注册、运行时实例、异步任务和审计上下文。

### 6.2 能力注册

- `FR-04` 模块插件必须通过正式 Manifest 声明自己提供的 capability、method、输入输出 schema、执行模式和依赖。
- `FR-05` Host 必须维护模块能力注册表，并支持按 capability 查询可用 provider。
- `FR-06` Host 必须支持同一 capability 存在多个 provider，并允许调用方显式指定 provider；未指定时由 Host 按正式策略解析默认 provider。

### 6.3 调用与调度

- `FR-07` 应用插件、卡片插件、布局插件、Host 内部服务以及其他模块插件，都必须通过统一模块调用接口使用模块能力。
- `FR-08` Host 必须在收到模块调用请求后，按 capability 和可选 provider 信息定位目标模块，并统一完成权限校验、schema 校验、运行时状态检查和请求转发。
- `FR-09` 模块插件必须支持同步方法和异步任务方法两种执行模式。
- `FR-10` 对于异步任务方法，Host 必须提供任务创建、状态查询、进度事件、取消和最终结果获取能力。

### 6.4 模块间调用

- `FR-11` 模块插件必须允许调用其他模块插件，但只能通过 Host 提供的正式模块调用 API 发起。
- `FR-12` 模块之间的组合和编排只是能力调用关系，不额外定义新的模块类型。

### 6.5 契约与验证

- `FR-13` Host 必须管理生态级统一调用框架和接口治理规则，包括公开动作面、provider 发现、选择、权限、schema、超时、任务、错误、事件和日志。
- `FR-14` 模块插件自身必须提供业务能力契约，包括 capability、method、输入输出 schema、依赖以及必要的对接说明；正式公共信息必须沉淀到生态共用技术文档和模块仓库契约文件中。
- `FR-15` SDK 必须为调用方提供唯一正式模块能力调用入口，不允许应用侧散落私有 `invoke("module.*")` 拼装；调用方提交 capability + method 请求后，必须由 Host 完成 provider 解析和请求转发。

### 6.6 脚手架

- `FR-16` 官方模块脚手架必须直接生成“无界面能力模块”工程，而不是 React 嵌入式运行时工程。
- `FR-17` 生成工程必须自带正式目录、Manifest、schema 合同目录、测试骨架、打包配置和 README。
- `FR-18` `chipsdev create module` 必须默认生成符合新契约的工程结构。

## 7. 非功能需求

- `NFR-01` 模块架构必须支撑生态未来长期扩展，不因某个单一历史业务场景而被绑死。
- `NFR-02` 模块调用链必须具备完整可观测性，至少覆盖 caller、provider、capability、method、duration、status、errorCode。
- `NFR-03` 模块运行时必须具备隔离能力，单个模块失败不能拖垮 Host 和调用方。
- `NFR-04` 模块权限必须独立于调用方权限生效，不允许模块复用页面宿主权限边界。
- `NFR-05` 模块契约必须可测试，包括 schema 校验、provider 解析、依赖校验、调用成功/失败路径、异步任务取消路径。
- `NFR-06` 模块脚手架默认生成的工程必须可构建、可测试、可校验、可打包，不留 TODO 和占位实现。

## 8. 交付物需求

本次大任务最终需要收口的交付物包括：

- 项目级需求文档；
- 项目级技术文档；
- 项目级开发计划；
- 生态设计原稿中与模块插件相关的正式设计更新；
- 生态共用技术文档中与模块插件相关的公共契约更新；
- 后续实施阶段需要完成的 Host / SDK / 脚手架 / 参考模块任务拆分基线。

## 9. 验收标准

- 模块插件正式定义完成收口，生态文档中不再把模块插件定义为 UI 挂载型模块。
- Host、SDK、Manifest、Bridge、脚手架文档对模块插件的对外口径统一为“Host 托管无界面能力模块”。
- 文档中存在统一的模块能力调用模型，明确 Host 管理统一调用框架、模块声明业务契约、调用方按正式契约使用。
- 官方模块脚手架目标基线明确为无界面能力工程。
- 后续实施所需阶段目标、受影响仓库和验证门禁已经清晰拆分。
