# 大任务001：重构模块插件全链路技术文档

## 1. 设计结论

模块插件的目标架构冻结为：

- 插件类型仍为 `type: module`；
- 运行形态为 Host 托管的无界面能力模块；
- 对外通过 capability + method 暴露能力；
- Host 统一负责 provider 注册、发现、选择、校验、调度、任务管理、权限治理和审计；
- 调用方统一通过 SDK / Bridge 的正式模块服务访问模块能力；
- 模块之间允许相互调用，但必须继续经过 Host 模块服务。

这意味着当前围绕 `slot + mount + React runtime` 的模块口径在后续实施中将整体退出正式外部协议。

## 2. 术语

### 2.1 模块插件

安装到 Host 中、由 Host 托管运行、对外提供无界面能力的正式插件。

### 2.2 Provider

模块插件中某一项已注册能力的提供者实例。一个模块插件可以提供一个或多个 capability。

### 2.3 Capability

对外暴露的稳定能力标识，例如：

- `converter.card.export`
- `renderer.markdown`
- `processor.richtext.normalize`

Capability 是调用方发现和调用模块能力的一级标识。

### 2.4 Method

某个 capability 下的具体方法，例如 `convert`、`render`、`normalize`、`index`。

### 2.5 Job

异步任务执行实体。用于处理长时间运行、可取消、需回报进度的模块方法。

## 3. 总体架构

## 3.1 分层

### A. 插件治理层

继续复用 Host 现有插件治理：

- `plugin.install`
- `plugin.enable`
- `plugin.disable`
- `plugin.uninstall`

模块插件先是正式插件，然后才是能力 provider。

### B. 模块注册层

当模块插件启用后，Host 解析其 Manifest 中的 `module` 声明，把 provider 信息注册到模块注册表。

### C. 模块运行层

Host 按需激活模块运行时，为其分配独立运行时身份、权限快照、日志上下文和任务上下文。

### D. 模块调用层

调用方统一通过模块服务发起：

- provider 查询
- capability 解析
- 方法调用
- job 管理

### E. 模块编排层

模块可以调用其他模块，但仍然通过模块服务完成，不形成第二套直连通道。

## 3.2 运行形态

第一版正式推荐冻结为 `worker` 运行时：

- 模块不运行在应用页面 DOM 环境中；
- 模块不直接依赖 React 渲染树；
- 模块通过 Host 注入的正式上下文访问系统能力；
- 模块不能直接获得 Electron 或 Node 原生对象。

## 3.3 长期场景覆盖

该架构需要长期稳定支撑以下三类高频业务场景，但它们只是使用场景，不是新的模块类型：

- 转换场景：例如导入、导出、转码、格式转换、内容投影；
- 编排场景：某个模块按业务流程调用其他模块完成组合任务；
- 通用处理场景：例如解析、规范化、索引、抽取、校验、富文本或 Markdown 处理。

架构设计应统一落在 capability + method + Host 治理框架上，而不是为这些场景再拆出新的插件类别。

## 4. Manifest 目标结构

模块插件 Manifest 目标结构如下：

```yaml
id: chips.module.card-export
name: Card Export Module
version: 0.1.0
type: module
entry: dist/module.cjs
description: 为生态提供卡片导出能力的正式模块插件。
permissions:
  - file.read
  - file.write
  - card.read
engines:
  chips: "^0.1.0"
module:
  apiVersion: 1
  runtime: worker
  activation: onDemand
  provides:
    - capability: converter.card.export
      version: 1.0.0
      description: 导出卡片为其他目标格式。
      methods:
        - name: convert
          mode: job
          inputSchema: contracts/convert.input.schema.json
          outputSchema: contracts/convert.output.schema.json
  consumes:
    - capability: renderer.card.snapshot
      versionRange: ^1.0.0
```

### 4.1 关键字段

- `entry`
  - 模块正式运行时代码入口；
  - 由 Host 在激活模块时加载；
  - 不再表示“页面可挂载模块前端入口”。
- `module.apiVersion`
  - 模块能力契约版本；
  - 用于后续能力协议治理。
- `module.runtime`
  - 当前冻结为 `worker`。
- `module.activation`
  - `onDemand` 或 `eager`；
  - 默认 `onDemand`。
- `module.provides`
  - 正式 provider 列表。
- `module.consumes`
  - 正式依赖的 capability 列表。

## 5. Host 模块注册表

Host 需要引入正式的模块注册表，至少维护以下信息：

```ts
interface ModuleProviderRecord {
  pluginId: string;
  capability: string;
  version: string;
  runtime: "worker";
  activation: "onDemand" | "eager";
  methods: Array<{
    name: string;
    mode: "sync" | "job";
    inputSchema: string;
    outputSchema: string;
  }>;
  permissions: string[];
  status: "enabled" | "disabled" | "running" | "error";
}
```

### 5.1 Host 职责

- 注册 provider；
- 查询 provider；
- 解析 capability 的默认 provider；
- 激活模块 runtime；
- 分发方法调用；
- 管理 job 生命周期；
- 统一记录审计日志。

## 6. 正式模块服务动作

建议冻结为以下动作面：

### 6.1 Provider 查询

```ts
module.listProviders({
  capability?: string,
  pluginId?: string,
  status?: "enabled" | "running"
}) -> { providers }
```

### 6.2 Capability 解析

```ts
module.resolve({
  capability: string,
  versionRange?: string
}) -> { provider }
```

### 6.3 方法调用

```ts
module.invoke({
  capability: string,
  method: string,
  input: Record<string, unknown>,
  pluginId?: string,
  timeoutMs?: number
}) -> 
  | { mode: "sync", output: unknown }
  | { mode: "job", jobId: string }
```

### 6.4 Job 管理

```ts
module.job.get({ jobId }) -> { job }
module.job.cancel({ jobId }) -> { ack: true }
```

### 6.5 Job 事件

通过事件总线发布：

- `module.job.progress`
- `module.job.completed`
- `module.job.failed`

### 6.6 治理分工与接入模型

模块插件正式链路中的职责分工冻结如下：

- Host 管理统一调用治理框架：
  - 正式动作名；
  - provider 注册与解析；
  - 权限校验；
  - schema 校验；
  - 超时与取消；
  - job 生命周期；
  - 错误码、事件和审计。
- 模块插件管理业务能力契约：
  - capability；
  - method；
  - 输入输出 schema；
  - 业务语义；
  - 外部依赖与 `module.consumes`。
- 调用方按正式契约接入：
  - 开发期查看生态共用技术文档中的公共契约与模块仓库中的 contract 文件；
  - 编码期通过 SDK 类型化接口发起调用；
  - 运行期由 Host 解析 provider 并把请求转发给目标模块。

## 7. 调用流程

## 7.1 应用插件调用模块插件

1. 应用插件通过 SDK 调用 `client.module.invoke(...)`；
2. Host 读取 capability 和可选 `pluginId`；
3. Host 从模块注册表定位 provider；
4. Host 校验调用方权限、provider 状态和输入 schema；
5. Host 激活目标模块 runtime（若尚未激活）；
6. Host 将请求分发给 provider method；
7. provider 返回同步结果或 job；
8. Host 统一归一化结果并返回调用方。

## 7.2 模块插件调用模块插件

1. 模块运行时使用 Host 注入的 `ctx.module.invoke(...)`；
2. 后续流程与应用插件调用一致；
3. 调用链中的 caller 变为当前模块 provider；
4. Host 继续统一记录审计、权限和错误。

## 8. Host 注入上下文

模块运行时上下文建议至少包含以下能力：

```ts
interface ModuleRuntimeContext {
  pluginId: string;
  capability: string;
  logger: Logger;
  permissions: string[];
  services: {
    file: FileServiceFacade;
    resource: ResourceServiceFacade;
    card: CardServiceFacade;
    box: BoxServiceFacade;
    config: ConfigServiceFacade;
  };
  module: {
    invoke(request: ModuleInvokeRequest): Promise<ModuleInvokeResult>;
  };
  jobs: {
    create<T>(executor: (job: ModuleJobController) => Promise<T>): Promise<{ jobId: string }>;
    reportProgress(jobId: string, payload: Record<string, unknown>): Promise<void>;
  };
  storage: {
    dataDir: string;
    cacheDir: string;
  };
}
```

## 9. SDK 目标接口

SDK 对外模块接口建议冻结为：

```ts
client.module.listProviders(options?)
client.module.resolve(capability, options?)
client.module.invoke(request)
client.module.job.get(jobId)
client.module.job.cancel(jobId)
```

### 9.1 边界

- SDK 负责类型、安全封装和结果解包；
- SDK 不承载模块运行时主实现；
- 调用方不得散落手写 `client.invoke("module.*")` 私有调用；
- SDK 模块类型应以正式共享契约为来源，不再从 README 或脚手架模板私有约定反推；
- 模块脚手架生成工程默认应消费上述正式 API。

## 10. 官方模块脚手架目标基线

官方 `chips-scaffold-module` 应生成如下目录：

```text
module-project/
├─ manifest.yaml
├─ package.json
├─ chips.config.mjs
├─ src/
│  └─ module.ts
├─ contracts/
│  ├─ <method>.input.schema.json
│  └─ <method>.output.schema.json
├─ tests/
│  ├─ unit/
│  └─ integration/
└─ README.md
```

### 10.1 明确禁止

- 不生成 React 运行时模板；
- 不生成 `runtime.tsx`；
- 不生成主题 CSS 注入模板；
- 不生成 i18n UI 模板；
- 不生成 `mountModule(container)` 之类 DOM 挂载入口。

## 11. 验证与测试设计

### 11.1 Host

- provider 注册测试；
- capability 解析测试；
- 多 provider 选择策略测试；
- schema 校验失败测试；
- 模块禁用/卸载回收测试；
- job 创建、进度、取消、完成测试；
- 模块间调用测试。

### 11.2 SDK

- 模块调用结果解包测试；
- 同步/异步两种模式测试；
- 错误归一测试；
- 类型定义测试。

### 11.3 脚手架

- 模板生成测试；
- 生成工程 build/test/validate/package 测试；
- Manifest 与 contract 文件完整性测试。

### 11.4 契约一致性

- Manifest 中 `module.provides/module.consumes` 与 contract 文件一致性测试；
- Host route schema 与 SDK 类型定义一致性测试；
- 生态共用技术文档中的正式动作面与代码路由清单一致性测试。

## 12. 重构边界说明

本技术方案明确以未来正式架构为目标，不为旧 `slot + mount + React runtime` 口径做兼容设计。后续实施阶段应以本文件为唯一技术收口依据，逐步完成 Host、SDK、脚手架和参考模块重构。
