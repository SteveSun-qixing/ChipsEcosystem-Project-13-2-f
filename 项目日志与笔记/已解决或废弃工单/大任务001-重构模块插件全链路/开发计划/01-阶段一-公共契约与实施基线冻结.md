# 阶段一：公共契约与实施基线冻结

## 1. 阶段目标

冻结模块插件在生态中的唯一正式口径，为后续 Host、SDK、脚手架和参考模块开发提供不可再漂移的实施基线。

## 2. 入口条件

- 用户确认模块插件正式方向；
- 当前任务级需求文档、技术文档已具备初稿；
- 已完成相关设计原稿、生态共用技术文档、Host/SDK/脚手架实际代码现状核对。

## 3. 重点交付物

- 项目级需求文档；
- 项目级技术文档；
- 项目级开发计划总览与分阶段计划；
- `生态共用技术文档/协议与契约/06-模块能力契约.md`；
- 相关公共架构文档、插件开发文档、Bridge/Host 对外接口文档；
- 相关生态设计原稿修订稿。

## 4. 任务拆解

### 4.1 公共架构冻结

- 明确模块插件只有一种正式类型：Host 托管无界面能力模块；
- 明确 capability + method 是唯一正式调用模型；
- 明确模块之间允许相互调用，但必须继续经过 Host；
- 明确不设计“工作流模块”“转换模块”等新类型，相关场景只体现在 capability 设计上。

### 4.2 契约分工冻结

- 明确 Host 管理统一调用治理框架；
- 明确模块开发者负责业务 capability / method / schema 契约；
- 明确调用方按共享契约与 SDK 类型接入，运行时由 Host 解析 provider 并转发；
- 明确公共信息与内部信息的文档归属边界。

### 4.3 Manifest 与运行时基线冻结

- 冻结 `type: module` 的 Manifest 结构；
- 冻结 `module.provides`、`module.consumes`、`module.runtime`、`module.activation` 等关键字段；
- 冻结第一版正式运行时为 `worker`；
- 冻结旧 `capabilities: []` 不再承担正式模块契约主入口。

### 4.4 Host 对外动作面冻结

- 冻结 `module.listProviders`；
- 冻结 `module.resolve`；
- 冻结 `module.invoke`；
- 冻结 `module.job.get`；
- 冻结 `module.job.cancel`；
- 冻结配套事件、错误、审计字段和验证要求。

### 4.5 脚手架目标冻结

- 冻结模块工程推荐目录结构；
- 冻结脚手架不再生成 React/slot 运行时；
- 冻结 contract 目录、测试骨架、README 基线和验证脚本要求。

## 5. 串并行安排

本阶段不建议再拆并行编码任务，必须先完成统一文档冻结。可接受的小规模并行仅限：

- 一路补项目级任务文档；
- 一路补生态共用技术文档和设计原稿；
- 最终统一复核口径后再收口。

## 6. 验证门禁

- 公共文档中不再正向描述模块插件为页面挂载模块；
- Host / SDK / 脚手架 / Manifest / Bridge / Host 接口文档口径统一；
- 需求、技术、开发计划三类项目文档已形成完整闭环；
- 长期场景覆盖已经明确写清：转换场景、跨模块编排场景、通用处理场景。

## 7. 阶段退出标准

- 可以作为后续所有实现任务的唯一正式文档基线；
- 后续阶段无须再讨论模块插件到底是什么、如何调用、谁管契约；
- 若仍存在关键概念冲突，则不得进入阶段二。
