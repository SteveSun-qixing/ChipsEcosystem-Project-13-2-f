# 阶段五：模块脚手架与创建链路重做

## 1. 阶段目标

把 `chips-scaffold-module` 从旧 React/slot 模块模板重做为正式的无界面能力模块工程生成器，并打通 `chipsdev create module` 创建链路。

## 2. 入口条件

- 阶段二完成；
- 阶段四至少完成 SDK 模块 API 名称和类型冻结；
- 阶段三完成模块 Manifest 与 Host 运行时上下文冻结检查点。

## 3. 重点仓库

- `Chips-Scaffold/chips-scaffold-module`
- `Chips-SDK` 中 `chipsdev` 入口的联动部分

## 4. 任务拆解

### 4.1 模板结构重做

- 删除旧 `runtime.tsx`、i18n UI、主题注入模板；
- 生成 `src/module.ts` 或等价 headless 入口；
- 生成 `contracts/`；
- 生成 `tests/unit`、`tests/integration`；
- 生成 README、Manifest、打包配置、校验脚本基线。

### 4.2 Manifest 模板重做

- 切换到 `module.provides/module.consumes`；
- 输出 `entry: dist/module.cjs` 或正式约定入口；
- 输出 `runtime: worker`、`activation: onDemand` 等标准字段；
- 不再输出旧 `capabilities` 主入口。

### 4.3 模板渲染引擎与 CLI

- 更新模板参数；
- 更新 `chipsdev create module` 的参数校验和默认值；
- 保证生成工程默认即符合新的目录与契约要求；
- 保证生成工程不带旧链路说明和示例。

### 4.4 生成工程测试

- `test:templates` 校验模板完整性；
- `test:e2e` 校验生成工程 build/test/validate/package；
- 增加 contract 文件和 Manifest 一致性测试；
- 增加 README 与公共文档口径一致性检查。

## 5. 串并行安排

- 可与阶段四并行推进；
- 模板目录与 README 重写可先行；
- 涉及 SDK 类型和 Host 动作名的模板代码必须等待阶段三、四冻结后再合流。

## 6. 验证门禁

- `chips-scaffold-module`：`npm run build`
- `chips-scaffold-module`：`npm test`
- `chips-scaffold-module`：`npm run test:templates`
- `chips-scaffold-module`：`npm run test:e2e`

重点验证项：

- 生成工程不再包含 React UI runtime；
- 生成工程自带 contract、测试和正式入口；
- 生成工程在不补临时代码的前提下可以走完 build/test/validate/package。

## 7. 阶段退出标准

- 新模块创建链路已经可用；
- 任何新模块仓库都不再从旧页面挂载模板起步；
- 参考模块开发可以基于该脚手架直接开展。
