# 阶段七：生态调用方接入与回归

## 1. 阶段目标

让生态中的正式调用方接入新模块链路，形成“调用方 -> SDK / Bridge -> Host -> 模块插件”的稳定生产路径。

## 2. 入口条件

- 阶段六完成；
- 至少已有一组稳定参考模块可以被调用。

## 3. 重点范围

- `Chips-Host` 内部调用方
- 官方应用插件
- 其他需要消费模块能力的官方仓库

## 4. 任务拆解

### 4.1 调用点梳理

- 识别当前生态中仍然依赖旧模块口径或私有调用的调用方；
- 识别未来需要正式调用模块能力的官方应用或服务；
- 输出迁移优先级。

### 4.2 调用方接入

- 按 capability + method 改造调用；
- 接入 SDK 正式模块 API；
- 处理同步结果与 job 结果；
- 补齐调用方侧权限声明、错误处理和观测字段。

### 4.3 回归验证

- 验证 caller -> Host -> provider 路径；
- 验证指定 provider 与默认 provider；
- 验证失败、超时、取消和 provider 不可用场景；
- 验证模块禁用或卸载后的调用方行为。

### 4.4 文档收口

- 更新调用方仓库中的需求文档、技术文档和使用说明；
- 更新生态共用技术文档中的示例；
- 清理对旧公开口径的正向描述。

## 5. 串并行安排

- 不同调用方可以并行改造；
- 但必须先确定公共错误处理和 job 处理模式；
- 高价值调用方优先，避免在多个边缘调用方上分散验证资源。

## 6. 验证门禁

- 接入仓库自身正式脚本通过；
- Host / SDK 全链路回归通过；
- 关键官方调用方能够稳定调用至少一个同步模块和一个异步模块。

## 7. 阶段退出标准

- 生态中已经存在真实正式调用方，而不是只有演示模块；
- 新链路已证明能够承接实际业务；
- 可以进入统一验收和旧链路清退阶段。
