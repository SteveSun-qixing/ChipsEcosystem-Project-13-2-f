# 大任务002：转换链路需求总览

## 1. 业务目标

生态需要一条统一、稳定、可扩展的文件转换链路，满足以下输出场景：

- 卡片导出为 HTML
- 卡片导出为 PDF
- 卡片导出为图片

对调用方而言，外部只感知到一个文件转换模块；对内部实现而言，正式结构拆分为一个编排模块和三个原子模块。

## 2. 范围

### 2.1 本次纳入范围

- 统一文件转换入口
- 原子转换模块的正式能力边界
- `card -> html -> pdf/image` 复用链路
- 转换任务的进度、取消、错误和结果归一
- 每个模块仓库的需求文档、技术文档、开发计划
- 生态设计原稿和生态共用技术文档同步收口

### 2.2 本次不纳入范围

- 外部文档导入到卡片
- Word / PDF / 音视频等其他源类型的转换
- 云端转换服务或远程队列
- 市场分发、签名和发布平台

## 3. 总体需求

### 3.1 统一入口

- 调用方只通过文件转换编排模块发起转换请求；
- 编排模块根据源类型和目标类型选择原子转换流水线；
- 调用方不直接依赖原子模块仓库结构。

### 3.2 原子职责

- `Chips-CardtoHTML-Plugin` 只负责 `card -> html`
- `Chips-HtmltoPDF-Plugin` 只负责 `html -> pdf`
- `Chips-HtmltoImage-Plugin` 只负责 `html -> image`

### 3.3 链路复用

- `card -> pdf` 必须复用 `card -> html`
- `card -> image` 必须复用 `card -> html`
- 不允许 PDF 和图片模块各自实现独立的卡片渲染逻辑

### 3.4 HTML 输出

- 必须复用基础卡片正式前端渲染代码
- 显示效果必须与卡片渲染模块一致
- 必须完成资源打包与路径重写
- 对用户交付时输出网页压缩包；对内部流水线可输出目录态中间产物

### 3.5 PDF / 图片输出

- 必须消费目录态 HTML 中间产物
- 使用 Electron 正式页面加载与导出能力
- 不直接依赖卡片结构解析

## 4. 质量需求

- 转换链路必须支持 job 进度与取消
- 中间产物必须可清理，不残留脏目录
- 输出文件命名、覆盖策略和错误码必须统一
- 文档、Manifest、脚手架基线和测试口径必须同步更新

## 5. 验收结果定义

本大任务验收完成时，应满足：

- 4 个模块仓库均完成正式文档基线
- 生态原稿和共用技术文档已冻结转换链路正式口径
- 调用方可以按统一入口导出 HTML / PDF / 图片
- HTML 视觉效果与 Host 渲染结果一致
- 编辑引擎导出面板具备接入这条正式转换链路的明确阶段计划
