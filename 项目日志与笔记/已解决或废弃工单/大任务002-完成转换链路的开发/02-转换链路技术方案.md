# 大任务002：转换链路技术方案

## 1. 总体结构

本次转换链路采用“一个编排模块 + 三个原子模块”的正式结构：

```text
调用方
  -> Chips-FileConversion-Plugin
       -> Chips-CardtoHTML-Plugin
       -> Chips-HtmltoPDF-Plugin
       -> Chips-HtmltoImage-Plugin
```

调用方只感知 `Chips-FileConversion-Plugin`；其余 3 个模块由编排模块在 Host 模块运行时内按能力契约调用。

## 2. 编排模块工作流

编排模块内部固定分为五层：

1. 请求归一层  
   负责校验源类型、目标类型、输出路径和参数模板。
2. 流水线规划层  
   把 `sourceType + targetType` 映射为正式 capability 链。
3. 步骤执行层  
   通过 `ctx.module.invoke(...)` 逐步调用原子模块。
4. 任务聚合层  
   汇总原子模块进度，映射为统一 job 进度。
5. 结果归一层  
   统一输出 artifact、warning、pipeline 和 error 结构。

## 3. Card -> HTML 技术路线

`Chips-CardtoHTML-Plugin` 不自行发明渲染器。正式路线如下：

1. 调用 Host `card.render(..., { target: "offscreen-render" })`
2. 获取 Host 生成的复合卡片 HTML 文档
3. 收集文档内嵌基础卡片 iframe 的资源基准路径
4. 重写资源引用到离线目录结构
5. 复制资源、生成 `conversion-manifest.json`
6. 输出目录态或压缩态 HTML 交付物

这样可以保证导出 HTML 与正式卡片显示链路共享同一套基础卡片渲染代码和主题结果。

## 4. HTML -> PDF 技术路线

`Chips-HtmltoPDF-Plugin` 只消费目录态 HTML 中间产物：

1. 读取 HTML 根目录与入口文件
2. 在受控 Electron 页面环境中加载 `index.html`
3. 等待页面稳定
4. 调用 PDF 输出能力生成最终文件
5. 返回输出路径、页数和警告

它不解析卡片文件，也不关心基础卡片结构。

## 5. HTML -> 图片技术路线

`Chips-HtmltoImage-Plugin` 与 PDF 模块一致，只在最后一步改为截图：

1. 读取 HTML 根目录与入口文件
2. 在受控 Electron 页面环境中加载 `index.html`
3. 设置目标尺寸、缩放倍率和背景策略
4. 执行页面截图
5. 输出 PNG / JPEG / WebP 文件

## 6. 中间产物策略

为兼顾“用户交付压缩包”和“内部二次转换复用”，`card -> html` 必须支持双模式：

- `directory`：目录态 HTML，中间产物，只供后续 `pdf/image` 流水线消费
- `zip`：压缩态 HTML，最终交付给用户

编排模块内部永远不能把 `zip` 直接传给 PDF / 图片模块。

## 7. 与编辑引擎的关系

编辑引擎当前正式只打通 `.card` 导出。后续接入转换链路时，应采用：

1. 先保存当前卡片状态
2. 如需导出 `.card`，继续调用 `card.pack`
3. 如需导出 HTML / PDF / 图片，调用统一文件转换编排模块
4. 在导出面板订阅 job 进度并反馈到 UI

编辑引擎不直接依赖原子模块。
