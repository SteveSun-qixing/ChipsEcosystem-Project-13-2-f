# 阶段一：全链路实施基线与仓库边界冻结

## 1. 阶段目标

冻结箱子正式实现的唯一实施基线，确保 Host、SDK、布局插件、查看器、编辑引擎在进入编码前已经对“做什么、不做什么、由谁负责什么”达成不可漂移的一致口径。

## 2. 入口条件

- 生态设计原稿中的箱子、布局插件、查看器章节已按新口径修订；
- 生态共用技术文档中的箱子业务、文件格式、运行时契约、服务协议和插件开发文档已完成首轮收口；
- 已核对 Host、SDK、编辑引擎、查看器和布局插件仓当前真实代码基线；
- 工单064已登记当前全链路缺口。

## 3. 涉及仓库

| 仓库 | 本阶段职责 |
|---|---|
| `项目日志与笔记` | 建立任务总览、开发计划与阶段文档 |
| `Chips-Host` | 核对当前 `box-service`、路由、schema、文件关联现状 |
| `Chips-SDK` | 核对当前 `BoxApi` 与契约清单现状 |
| `Chips-CardViewer` | 核对当前卡片查看壳与 `.box` 缺口 |
| `Chips-EditingEngine` | 核对当前箱子创建、工作区与布局库现状 |
| `Chips-BoxLayoutPlugin` | 冻结正式工程落位策略 |
| `Chips-ComponentLibrary` | 仅核对，不默认进入写入范围 |

## 4. 重点交付物

- 本大任务的总览文档与分阶段开发计划；
- 受影响仓库的写入范围与依赖矩阵；
- 箱子端到端目标流程图文字版；
- 布局插件宿主层共享位置决策；
- 自动化验证矩阵与样例夹具策略。

## 5. 任务拆解

### 5.1 冻结正式端到端流程

必须明确并固定以下唯一正式流程：

1. 编辑引擎创建或打开箱子；
2. 编辑链路在工作目录中维护 `metadata.yaml / structure.yaml / content.yaml / assets/`；
3. 保存或导出时由 Host `box.pack` 生成最终 `.box`；
4. Host 打开 `.box` 时通过 `box.openView` 建立查看会话；
5. 查看器加载对应布局插件；
6. 布局插件先消费摘要页，再按需向 Host 申请数据与资源；
7. 关闭查看页时由 Host 回收会话。

### 5.2 冻结仓库职责边界

必须在本阶段明确以下边界：

- `Chips-Host`：唯一运行时解析中心、会话中心、资源句柄中心；
- `Chips-SDK`：箱子正式 API、类型和调用封装；
- `Chips-BoxLayoutPlugin`：首个网格布局插件与布局编辑器；
- `Chips-CardViewer`：箱子查看消费方；
- `Chips-EditingEngine`：箱子创作与编辑消费方；
- `Chips-ComponentLibrary`：只有通用 UI 能力缺口时才进入写入范围。

### 5.3 冻结布局插件宿主方案

本阶段必须明确查看器与编辑引擎如何复用同一套布局插件导出契约。

必须在以下方案中收口为唯一正式实现方向：

1. 布局插件入口统一导出 `layoutDefinition`；
2. 官方应用只能基于 Host `plugin.query({ type: 'layout' })` 返回的正式安装记录与入口路径加载布局插件；
3. 不允许查看器和编辑引擎分别发明不同的布局插件宿主接口；
4. 若需要共享布局插件注册表或宿主适配层，必须先冻结共享代码落位位置，再进入编码。

### 5.4 冻结样例与验收夹具策略

必须明确后续端到端验证使用的正式夹具来源：

1. Host/SDK 单元与契约测试夹具；
2. 查看器和编辑引擎的应用级测试夹具；
3. `ProductFinishedProductTestingSpace` 或工作区样例目录中的真实 `.card` / `.box` 样例；
4. 包含本地 `file://`、网络 `https://`、缺失资源、禁用条目、空箱子、多布局配置等典型场景。

### 5.5 冻结旧口径清退清单

必须建立明确清退清单，后续阶段统一删除：

1. `full/shell/mixed`
2. `internal/external`
3. 目录态 `.box`
4. `box/layout.yaml`
5. `box.pack` 权限
6. 只支持 `box.inspect` 的文件打开逻辑
7. 查看器纯卡片假设
8. 编辑引擎“箱子仅是目录”的假设

## 6. 串并行安排

本阶段不建议拆并行编码任务，必须先完成统一冻结。允许的小规模并行仅限：

- 一路补本大任务计划文档；
- 一路继续核对相关仓的真实代码现状和正式脚本；
- 最终统一收口后进入阶段二。

## 7. 验证门禁

- 任务总览与分阶段计划全部成文；
- 端到端正式流程没有歧义；
- 受影响仓库范围、写入边界和验证脚本都已明确；
- 布局插件宿主层共享策略已冻结；
- 若仍存在“箱子到底由谁加载布局插件”这类关键歧义，则不得进入阶段二。

## 8. 阶段退出标准

- 后续所有编码阶段都可以把本阶段文档作为唯一实施基线；
- 不再需要重新讨论箱子模型、布局插件角色和 Host 主责范围；
- 若出现新的公共缺口，必须先登记工单再推进。
