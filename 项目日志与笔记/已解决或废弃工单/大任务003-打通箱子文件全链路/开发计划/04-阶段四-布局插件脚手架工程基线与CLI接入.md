# 阶段四：布局插件脚手架工程基线与CLI接入

## 1. 阶段目标

在 `Chips-Scaffold/chips-scaffold-boxlayout` 中正式建立布局插件脚手架仓，冻结布局插件工程目录、模板变量、测试基线、CLI 接入方式与生成工程验证门禁，为后续 `Chips-BoxLayoutPlugin` 实现提供唯一正式初始化来源。

## 2. 入口条件

- 阶段一至阶段三已完成；
- 箱子与布局插件共享契约已经冻结；
- 已确认工作区不存在现成的布局插件脚手架仓。

## 3. 涉及仓库

| 仓库 | 本阶段职责 |
|---|---|
| `Chips-Scaffold/chips-scaffold-boxlayout` | 建立布局插件脚手架正式工程 |
| `Chips-SDK` | 如模板示例需要依赖 SDK 正式类型，做最小对齐 |
| `生态共用技术文档` | 仅在代码与模板基线核对后修正文档漂移 |

## 4. 重点交付物

- 新仓 `chips-scaffold-boxlayout` 的需求文档、技术文档、开发计划；
- 正式脚手架工程骨架；
- 标准布局插件模板；
- 模板参数化与渲染逻辑；
- `chipsdev create` 的布局模板接入方案；
- `build/test/test:templates/test:e2e` 质量门禁。

## 5. 任务拆解

### 5.1 建立脚手架项目文档与范围冻结

需要先完成：

1. 项目级需求文档；
2. 项目级技术文档；
3. 项目级开发计划；
4. 与大任务003阶段依赖关系对齐。

### 5.2 冻结布局插件模板工程结构

模板生成工程至少应包含：

- `manifest.yaml`
- `package.json`
- `src/index.ts`
- `src/view/`
- `src/editor/`
- `src/schema/`
- `src/shared/`
- `tests/`
- `assets/`
- `i18n/`
- `contracts/`
- `README.md`

生成工程必须默认可构建、可测试、可校验，不得要求业务仓手工补主干基线。

### 5.3 冻结模板参数与 CLI 接入

至少冻结：

- 模板 ID；
- `pluginId / displayName / layoutType / description` 等核心参数；
- `chipsdev create` 与该脚手架的对接方式；
- 默认模板与未来多模板扩展策略。

### 5.4 冻结模板质量门禁

至少建立：

- 脚手架自身 `build`
- 脚手架自身 `test`
- 模板一致性检查 `test:templates`
- 生成工程端到端检查 `test:e2e`

### 5.5 与后续布局插件工程对接

必须保证阶段五 `Chips-BoxLayoutPlugin` 可以直接基于本脚手架生成或对齐，不允许再手写另一套工程标准。

## 6. 串并行安排

本阶段可受控并行：

- 一路完成项目文档；
- 一路建立脚手架工程骨架；
- 一路设计模板参数化与 CLI 接口；
- 一路补测试与质量门禁。

## 7. 验证门禁

阶段内必须形成并通过：

- `build`
- `test`
- `test:templates`
- `test:e2e`

同时验证：

- 生成工程目录、脚本、清单和 README 全部完整；
- 模板中无占位符泄露、无 TODO、无临时实现；
- 后续布局插件仓可直接以该模板为正式基线。

## 8. 阶段退出标准

- 布局插件脚手架已经成为正式工程初始化入口；
- 阶段五无需再讨论“布局插件工程应该长什么样”；
- 若后续仍需要手工拼工程骨架，说明本阶段未完成。
