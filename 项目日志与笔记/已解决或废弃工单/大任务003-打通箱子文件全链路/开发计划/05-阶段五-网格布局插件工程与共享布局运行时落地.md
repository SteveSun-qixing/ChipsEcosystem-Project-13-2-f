# 阶段五：网格布局插件工程与共享布局运行时落地

## 1. 阶段目标

在 `Chips-BoxLayoutPlugin` 中建立首个正式布局插件工程，交付可安装、可测试、可被查看器与编辑引擎复用的网格布局插件，并同步收口官方布局插件宿主层共享实现。

## 2. 入口条件

- 阶段二、阶段三、阶段四已完成；
- 阶段一已经冻结布局插件宿主层共享策略；
- `chips-scaffold-boxlayout` 已成为正式工程初始化基线。

## 3. 涉及仓库

| 仓库 | 本阶段职责 |
|---|---|
| `Chips-BoxLayoutPlugin` | 建立正式网格布局插件工程 |
| `Chips-ComponentLibrary` | 仅在发现缺失通用组件时进入 |
| `Chips-CardViewer` | 若共享布局宿主层放在查看侧共享目录，需要同步接线 |
| `Chips-EditingEngine` | 若共享布局宿主层放在编辑侧或公共目录，需要同步接线 |

## 4. 重点交付物

- 网格布局插件正式工程；
- `manifest.yaml`、`layout.layoutType`、`layoutDefinition`、测试和验证脚本；
- 查看态网格页面；
- 编辑态网格配置面板；
- 布局配置 schema、默认值、归一和校验逻辑；
- 官方布局插件宿主层共享实现；
- 如有必要，组件库补齐的公共封面或列表组件。

## 5. 任务拆解

### 5.1 建立正式工程骨架

必须为 `Chips-BoxLayoutPlugin` 建立完整正式工程，而不是临时演示目录。工程至少包括：

- `package.json`
- `manifest.yaml`
- `src/index.ts`
- `src/view/`
- `src/editor/`
- `src/schema/`
- `src/shared/`
- `tests/`
- `assets/`
- `i18n/`

并补齐：

- `build`
- `test`
- `lint`
- `validate`

### 5.2 实现布局插件正式导出

插件必须正式导出：

- `layoutDefinition.layoutType`
- `layoutDefinition.displayName`
- `createDefaultConfig`
- `normalizeConfig`
- `validateConfig`
- `getInitialQuery`
- `renderView`
- `renderEditor`

要求：

- `layout.layoutType` 与 `layoutDefinition.layoutType` 完全一致；
- 不直接依赖文件系统或网络；
- 所有清理逻辑必须闭环。

### 5.3 完成网格布局查看态

查看态至少实现以下能力：

1. 首屏消费 `initialView` 摘要态条目；
2. 渲染封面、标题、标签、状态与摘要；
3. 根据配置控制列数、间距、封面比例和信息密度；
4. 通过滚动分页或窗口化渲染支持大列表；
5. 通过 `resolveEntryResource` 动态请求封面或预览；
6. 通过 `readEntryDetail` 动态请求卡片元数据或状态；
7. 正式空态、错误态、加载态和资源缺失态。

### 5.4 完成网格布局编辑态

编辑态至少实现以下能力：

1. 网格列数、间距、封面比例、信息密度等参数编辑；
2. 主题适配与多语言接线；
3. 对当前箱子条目摘要的实时预览；
4. `props` 与 `asset_refs` 的正式输出；
5. 对无效配置的即时校验反馈。

### 5.5 落地共享布局宿主层

查看器与编辑引擎都要消费同一套布局插件导出契约，因此必须落地一个正式共享宿主层，负责：

1. 根据 `plugin.query({ type: 'layout' })` 读取已安装布局插件记录；
2. 从安装路径加载布局插件入口模块；
3. 校验 `layoutDefinition` 完整性；
4. 缓存已加载描述符；
5. 为查看器与编辑引擎提供一致的装载入口。

正式要求：

- 不得在两个应用仓各复制一份不同实现；
- 不得改为扫描源码目录；
- 若需要新增共享目录或共享包，必须使用阶段一已冻结的落位方案。

### 5.6 组件库受控补齐

如果当前组件库无法正式支撑网格布局所需的封面卡片、占位骨架、状态徽标或虚拟列表容器，则必须先在 `Chips-ComponentLibrary` 中补齐通用能力，再由网格布局插件消费。

禁止在布局插件内部内联一套仅供当前阶段使用的临时 UI 基础层。

## 6. 串并行安排

本阶段可受控并行：

- 一路建立插件工程与配置 schema；
- 一路实现查看态网格；
- 一路实现编辑态面板；
- 一路落地共享宿主层；
- 若进入组件库，则组件库补齐必须先于最终 UI 集成完成。

## 7. 验证门禁

至少运行：

- `cd Chips-BoxLayoutPlugin && npm run build`
- `cd Chips-BoxLayoutPlugin && npm run test`
- `cd Chips-BoxLayoutPlugin && npm run lint`
- `cd Chips-BoxLayoutPlugin && npm run validate`

若组件库进入写入范围，再运行：

- `cd Chips-ComponentLibrary && npm run verify`

同时必须验证：

- 网格布局插件可被安装并识别为 `layout` 插件；
- 共享宿主层能够正确加载已安装插件入口；
- 查看态和编辑态都不依赖临时 mock 结构才能运行。

## 8. 阶段退出标准

- 网格布局插件已成为正式工程，而不是空目录或示例项目；
- 查看器与编辑引擎都可以基于同一套共享宿主层消费 `layoutDefinition`；
- 若布局插件仍需要写死在某一个应用仓里运行，则本阶段不算完成。
