# 阶段八：样例工作区与端到端联调验收

## 1. 阶段目标

用真实工作区、真实 `.card` / `.box` 样例、正式脚本和自动化测试把箱子全链路打通，证明 Host、SDK、布局插件、查看器和编辑引擎已经形成可持续交付的完整系统。

## 2. 入口条件

- 阶段二至阶段七全部完成；
- Host、SDK、网格布局插件、查看器、编辑引擎都具备单仓自测能力；
- 样例夹具策略已在阶段一冻结。

## 3. 涉及仓库

| 仓库 / 目录 | 本阶段职责 |
|---|---|
| `Chips-Host` | 提供真实会话与文件关联能力 |
| `Chips-SDK` | 提供正式 API 给查看器与编辑引擎消费 |
| `Chips-BoxLayoutPlugin` | 提供网格布局查看与编辑实现 |
| `Chips-CardViewer` | 验证 `.box` 查看端到端流程 |
| `Chips-EditingEngine` | 验证创建、编辑、保存与预览端到端流程 |
| `ProductFinishedProductTestingSpace` / 测试夹具目录 | 提供真实样例数据 |

## 4. 重点交付物

- 正式样例工作区；
- 端到端联调用例；
- 多场景验收清单；
- 回归报告与问题收口记录。

## 5. 任务拆解

### 5.1 建立正式样例集

样例至少覆盖以下场景：

1. 工作区内 `file://` 卡片引用；
2. 网络 `https://` 条目；
3. 含封面快照和无封面快照条目；
4. 空箱子；
5. 启用 / 禁用混合条目；
6. 大数量条目分页与滚动；
7. 箱子自有资源；
8. 资源缺失、URL 解析失败和权限不足等异常场景。

### 5.2 打通创建到查看的完整流程

必须至少完成以下正式联调：

1. 在编辑引擎中新建箱子；
2. 添加多个条目与网格布局配置；
3. 保存并导出 `.box`；
4. 通过查看器或 `.box` 文件关联打开该文件；
5. 查看器通过 Host 打开会话并加载网格布局插件；
6. 用户可浏览摘要页并触发按需加载；
7. 关闭页面后会话正确回收。

### 5.3 覆盖关键回归场景

必须验证：

1. 主题切换后布局页面与编辑器预览都能正确刷新；
2. 多语言切换后布局页面与编辑器文案一致；
3. 条目较多时分页、懒加载或窗口化渲染稳定；
4. 查看器与编辑引擎同时打开同一箱子时不会破坏会话和保存结果；
5. 资源句柄不会泄露凭证或持久化到底层绝对路径。

### 5.4 自动化测试闭环

需要尽可能把端到端关键路径沉淀为自动化测试，而不是只保留人工演示：

1. Host 契约测试；
2. SDK 调用测试；
3. 查看器应用测试；
4. 编辑引擎应用测试；
5. 网格布局插件测试；
6. 工作区级 `verify:workspace` 回归。

## 6. 串并行安排

本阶段内部可受控并行：

- 一路准备样例工作区与样例 `.box`；
- 一路补自动化测试；
- 一路执行真实联调并记录问题；
- 所有问题统一回到对应仓库修复后再重新验证。

## 7. 验证门禁

至少运行：

- `cd Chips-Host && npm run build && npm test && npm run test:contract`
- `cd Chips-SDK && npm test`
- `cd Chips-BoxLayoutPlugin && npm run build && npm run test && npm run lint && npm run validate`
- `cd Chips-CardViewer && npm run build && npm run test && npm run lint && npm run validate`
- `cd Chips-EditingEngine && npm run build && npm run test && npm run lint && npm run validate`
- 根工作区 `npm run verify:workspace`

同时必须完成真实端到端人工验收：

- 编辑器创建 -> 保存 `.box` -> 查看器打开 -> 布局插件渲染 -> 动态取数 -> 关闭回收

## 8. 阶段退出标准

- 箱子全链路已经在真实工作区跑通；
- 自动化测试和人工验收都覆盖关键场景；
- 若仍只能证明单仓局部可用，而无法证明整条链路可用，则本阶段不算完成。
