# 大任务006：薯片社区服务器开发

## 任务概述

薯片社区服务器（Chips Community Platform Server，简称 CCPS）是薯片生态的网络公共平台组件，为多用户提供卡片与箱子的存储、发布、浏览和管理功能。本任务覆盖从零到一完整交付 CCPS，最终目标是一套可一键部署、稳定运行的社区服务器软件包。

**仓库路径**：`Chips-CommunityPlatformServer/`

---

## 产品定位与边界

### 本期目标（第一版）

- 用户注册与登录（用户名 + 密码，无需手机 / 邮箱验证码）
- 用户个人空间：网格列表展示所有房间以及直挂根目录的箱子和卡片
- 房间管理：创建 / 删除房间，以房间为单位管理访问权限（公开 / 私有）
- 卡片上传处理流水线：上传 → 提取内嵌资源 → 上传 CDN → 替换链接 → 卡片转换为 HTML → 入库
- 箱子上传：上传 `.box` 文件，解析并保存结构摘要
- 卡片 / 箱子公开浏览：通过固定 URL 访问已发布的 HTML 页面
- 管理后台：平台运营管理（用户、内容、存储情况总览及基础操作）
- 一键部署：提供 Docker Compose 一键启动方案

### 本期不包含

- 关注 / 粉丝社交功能
- 评论互动
- 高级推荐算法、热榜
- 实时通知推送（WebSocket/SSE）
- 邮件 / 手机号验证
- 多因子认证
- 版本历史管理
- 联邦协议（ActivityPub 等）

## 技术选型

| 类别 | 选型 | 说明 |
|------|------|------|
| 服务端运行时 | Node.js 20 LTS | 生态内已有技术基线 |
| API 框架 | Fastify | MIT 许可，高性能，插件生态完整 |
| 数据库 ORM | Drizzle ORM | MIT 许可，轻量、类型安全 |
| 数据库 | PostgreSQL 16 | MIT 许可，稳定可靠 |
| 对象存储 | MinIO（自托管）/ 标准 S3 接口 | Apache 2.0，用于卡片资源 CDN |
| 会话缓存 | Redis 7 | BSD 许可，用于 JWT 黑名单与速率限制 |
| 前端框架 | React 18 + Vite | 生态内已有技术基线 |
| 前端样式 | Vanilla CSS + 薯片主题 token | 与生态规范对齐 |
| 认证 | JWT（access + refresh token） | 无第三方依赖 |
| 容器化 | Docker + Docker Compose | 当前已收口为单机一键部署方案 |
| 卡片转 HTML | Host + 正式转换模块插件 | 保持与生态正式转换效果一致 |
| 语言 | TypeScript（全栈） | 与生态保持一致 |
| 包管理 | pnpm | 与生态工作区一致 |

> **CDN 说明**：默认使用 MinIO 的内置对象存储，通过 Nginx 反向代理对外提供静态资源访问，形成"内建 CDN"能力，无需依赖第三方 CDN 服务商，可一键自托管。

## 阶段规划

### 阶段一：项目脚手架与基础架构（Phase 1）

**目标**：搭建完整可运行的骨架，跑通 Hello World 端到端链路。

- 初始化 monorepo（`packages/server`, `packages/web`, `packages/admin`）
- 配置 TypeScript、ESLint、pnpm workspace
- 搭建 Fastify 服务，健康检查接口
- 配置 PostgreSQL + Drizzle ORM，初始化 schema migrations
- 配置 Redis 连接
- 配置 MinIO 连接与 bucket 初始化
- Docker Compose 编排（postgres、redis、minio、server、nginx）
- 环境变量管理（dotenv + Zod 校验）

**交付物**：`docker compose up -d --build` 后 API 健康检查可通，数据库 schema 存在。

---

### 阶段二：认证系统（Phase 2）

**目标**：用户注册、登录、JWT 认证完整闭环。

- 数据库 users 表设计
- `POST /auth/register`：用户名 + 密码注册（bcrypt 加密）
- `POST /auth/login`：登录，签发 access token + refresh token
- `POST /auth/refresh`：刷新 token
- `POST /auth/logout`：吊销 token（Redis 黑名单）
- Fastify 认证插件（JWT 验证中间件）
- 前端登录 / 注册页面

**交付物**：用户可注册登录，受保护接口正确鉴权。

---

### 阶段三：房间与空间管理（Phase 3）

**目标**：用户可以创建房间，管理权限，浏览个人空间。

- 数据库 rooms 表设计
- `POST /rooms`、`GET /rooms`、`DELETE /rooms/:id`
- 房间权限模型（公开 / 私有）
- `GET /users/:username`：个人空间首页数据接口
- 前端个人空间页面（网格房间列表）
- 前端房间管理页面

**交付物**：用户可在 UI 中创建 / 删除房间，公开房间可被任何人访问。

---

### 阶段四：卡片上传处理流水线（Phase 4）

**目标**：完整的卡片上传 → CDN 处理 → HTML 渲染 → 入库流程。

- 数据库 cards 表设计
- `POST /upload/card`：接收 `.card` 文件
- 卡片解包模块（读取 ZIP 结构，解析 `.card/metadata.yaml`、`structure.yaml`）
- 资源提取模块（枚举卡片内所有资源文件）
- CDN 上传模块（上传资源到 MinIO，获取公开 URL）
- URL 替换模块（将配置文件内的本地资源引用替换为 CDN URL）
- HTML 渲染模块（调用渲染服务将卡片转为独立 HTML）
- 入库（保存 card 记录，包括 CDN 化后的卡片数据和 HTML URL）
- `GET /cards/:cardId`：获取卡片元数据和 HTML 访问地址
- 错误处理与任务状态反馈

**交付物**：上传一个 `.card` 文件后，可通过 URL 访问其 HTML 版本。

---

### 阶段五：箱子上传与浏览（Phase 5）

**目标**：支持箱子文件上传，可以浏览箱子中的卡片列表。

- 数据库 boxes 表设计
- `POST /upload/box`：接收 `.box` 文件
- 箱子解包模块（解析 `.box/metadata.yaml` 与 `structure.yaml`）
- 卡片引用解析（将 structure.yaml 中的 URL 映射到社区内已有卡片）
- `GET /boxes/:boxId`：获取箱子元数据和卡片摘要列表
- 箱子浏览前端页面（展示卡片卡片列表/封面网格）

**交付物**：上传箱子后可以通过 URL 浏览其中卡片列表。

---

### 阶段六：公开浏览与内容页面（Phase 6）

**目标**：面向互联网用户的公开内容访问体验。

- 卡片详情公开页（嵌入 HTML iframe 或直接展示 HTML）
- 用户个人主页公开视图
- 房间公开浏览页（展示房间内的箱子和卡片）
- 基础 SEO（Open Graph、meta description）
- 全局头部导航与布局

**交付物**：任何人都可以通过 URL 访问用户分享的内容，体验完整。

---

### 阶段七：管理后台（Phase 7）

**目标**：平台运营管理界面。

- 管理员账户（超级管理员角色）
- `GET /admin/stats`：平台总览（用户数、卡片数、存储用量）
- `GET /admin/users`、`PUT /admin/users/:id`：用户管理
- `GET /admin/content`：内容列表，支持删除
- `GET /admin/storage`：存储用量分布
- Admin 前端页面（数据表格、图表）

**交付物**：管理员可登录后台查看平台运营状况并进行管理操作。

---

### 阶段八：一键部署包与文档（Phase 8）

**目标**：让任何人可以用最少步骤部署社区服务器。

- 完善 Docker Compose 配置（含 Nginx、SSL 模板）
- 编写部署文档（3 步部署指南）
- 编写配置参考文档（所有环境变量说明）
- 编写 API 文档（Swagger/OpenAPI 自动生成）
- 完善 README
- 集成测试（端到端验证主要流程）

**交付物**：完整可发布的软件包，文档齐全，测试覆盖主要流程。

## 关键设计决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| API 框架 | Fastify | MIT 许可，TypeScript 原生支持，性能优秀，插件体系完整 |
| 对象存储 | MinIO 自托管 | 完全自主可控，无需第三方服务，S3 兼容接口便于未来迁移 |
| CDN 资产处理 | 服务端流水线 | 保证资产版本一致性，避免客户端不稳定网络导致上传失败 |
| 卡片渲染 | 独立渲染服务 | 复用卡片渲染逻辑，不在主进程中执行重型渲染 |
| 认证 | JWT | 无状态，便于扩展，无需额外依赖 |
| 前端构建 | Vite | 生态内已有技术基线 |
| 部署 | Docker Compose | 最低部署门槛，一键启动，适合个人和小团队自托管 |
