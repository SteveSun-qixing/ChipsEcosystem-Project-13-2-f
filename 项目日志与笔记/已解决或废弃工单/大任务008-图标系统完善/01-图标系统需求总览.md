# 大任务008：图标系统需求总览

## 1. 背景

薯片生态的图标能力目前处于“资产已准备、规范已冻结、局部能力可用、运行时体系未完成”的状态：

1. Material Symbols 官方资产已经落库到 `design-assets/material-symbols/`；
2. 图标与字体系统设计规范已经冻结，明确要求统一 `ChipsIcon` 与主题 token；
3. Host 的应用启动图标链路已经可用；
4. 组件库、ThemePack、设置面板、编辑引擎、基础卡片插件与脚手架仍保留旧字符串图标、emoji、硬编码 SVG 和旧文档口径。

本任务的目标不是“补一点图标”，而是正式完成整个生态图标系统的一次彻底收口。

## 2. 总体目标

本任务完成后，生态图标系统必须满足以下总体目标：

1. 所有运行时 UI 图标都由统一图标组件输出；
2. 所有图标都由语义名驱动，而不是由 emoji、裸字符串字符或业务层内联 SVG 驱动；
3. 图标视觉参数统一纳入主题系统和 token；
4. 所有正式应用、插件、模板和共享文档对图标的口径保持一致；
5. 所有旧版图标、占位图标和 emoji 都从正式代码、正式模板、正式文档中移除；
6. 不保留兼容旧模型、回退逻辑或临时桥接层。

## 3. 架构目标

### 3.1 统一图标组件

生态内所有运行时 UI 图标统一使用 `ChipsIcon`（或同职责统一组件）输出。

该组件必须满足：

1. 以语义名消费 Material Symbols；
2. 支持 `outlined | rounded | sharp` 风格；
3. 支持 `fill / wght / grad / opsz` 变量轴；
4. 支持装饰图标与功能图标的可访问性分流；
5. 允许组件库默认图标、业务应用图标、插件元数据图标统一复用。

### 3.2 统一语义模型

全生态正式图标输入模型统一收敛为语义对象，不再继续使用 emoji 或任意 `string` 直接表达“图标本体”。

本大任务目标口径如下：

```ts
interface ChipsIconDescriptor {
  name: string;
  style?: 'outlined' | 'rounded' | 'sharp';
  fill?: 0 | 1;
  wght?: number;
  grad?: number;
  opsz?: number;
  decorative?: boolean;
  label?: string;
}
```

说明：

- `name` 是正式主键；
- `style/fill/wght/grad/opsz` 是渲染轴或风格覆盖；
- `decorative/label` 用于可访问性控制；
- 该对象用于运行时 UI 图标，不用于替代 `manifest.ui.launcher.icon` 这类操作系统入口图标文件路径。

### 3.3 主题系统统一接管图标视觉

图标视觉参数必须统一由主题系统提供，不允许在业务层或组件层硬编码。

至少要求正式落地：

1. `icon.color`
2. `icon.weight`
3. `icon.fill`
4. `icon.grade`
5. `icon.opticalSize`
6. `icon.size`

这些 token 需要形成：

- 共享 token 定义；
- ThemePack 默认值；
- 组件消费映射；
- 自动化测试覆盖。

### 3.4 Host 是操作系统入口图标的唯一承载

应用启动图标、快捷方式图标、启动台图标等系统级入口图标，仍由 Host 通过 `manifest.ui.launcher.icon` 的正式链路治理。

必须满足：

1. 应用插件只声明相对路径资源，不直接生成系统快捷方式文件；
2. Host 统一读取、复制、选择平台格式并创建快捷方式；
3. 设置面板等治理应用只读状态、触发接口，不绕过 Host；
4. 运行时 UI 图标与操作系统入口图标分层，不混为一套能力。

## 4. 正式需求

### 4.1 公共契约需求

1. 生态共用技术文档只能保留一套正式图标口径；
2. 主题包开发指南中的旧 `icons.css + svg + icon-sm/icon-md/icon-lg` 口径必须移除或改写；
3. 组件库、卡片插件、布局插件、设置治理文档中的 `icon` 字段说明要统一到新模型；
4. 编辑引擎基础卡片装配标准必须同步更新旧 `icon?: string` 口径。

### 4.2 共享资产需求

1. `design-assets/material-symbols/` 继续作为唯一正式图标上游来源；
2. 必须明确运行时所需字体、codepoints、生成物、追溯文件与命名规范；
3. 必须明确 App 图标源文件与 Material Symbols 运行时图标资源之间的边界；
4. 主题模板、默认主题与正式应用资产必须从正式源生成，不允许每仓手工维护一套漂移资源。

### 4.3 组件库需求

1. 正式提供 `ChipsIcon`；
2. 组件库默认图标改为统一图标组件，不再硬编码 SVG path；
3. 图标按钮、空态图标、时间图标、展开/折叠/关闭图标都必须走统一组件；
4. 所有相关契约、声明文件、token、测试与 a11y 都要一起收口。

### 4.4 ThemePack 需求

1. 默认主题包与暗色主题包都必须具备正式图标资源接入能力；
2. 主题包要提供图标字体声明、变量轴参数和图标 token 默认值；
3. 主题包模板要能生成正式图标目录和说明；
4. 不允许主题包继续停留在“只有 README，没有实际接线”的状态。

### 4.5 插件与应用需求

1. 基础卡片插件元数据中的 icon 必须从 emoji / string 改为语义图标描述；
2. 布局插件显示身份也必须统一到新模型；
3. 设置面板需要展示插件图标、治理状态与图标刷新结果；
4. 编辑引擎必须完成程序坞、卡箱库、文件管理器、设置面板、历史面板、导出面板等全部旧图标替换；
5. 查看器和其他应用如存在图标入口，也必须统一接入。

### 4.6 emoji 清零需求

本任务要求正式代码与正式文档中的 emoji 清零，覆盖范围包括：

1. 运行时代码；
2. 活跃模板代码；
3. 正式共享文档；
4. 正式项目文档；
5. 本大任务过程文档。

说明：

- 第三方上游追溯文件不做人为篡改；
- 已归档且不进入正式导航口径的历史材料，不作为首批强制修改对象；
- 但凡仍在正式运行、正式模板、正式文档入口中出现的 emoji，都必须清零。

### 4.7 无兼容要求

本任务明确不做以下内容：

1. 兼容旧字段；
2. 双字段双路径并存；
3. emoji 与新图标双写；
4. 内联 SVG fallback 与 Material Symbols 并存；
5. 旧主题口径与新主题口径长期共存。

所有旧口径都必须被正式替换，不保留回退机制。

## 5. 当前基线量化结果

截至 2026-03-25，已核对到的量化结果如下：

1. `Chips-EditingEngine`：76 处 emoji / 旧字符图标；
2. `Chips-BaseCardPlugin`：4 处 emoji / 旧字符图标；
3. `Chips-Scaffold`：2 处 emoji；
4. `生态共用技术文档`：8 处 emoji；
5. `Chips-ComponentLibrary`：不存在 `ChipsIcon` 正式实现；
6. `ThemePack/Chips-default/icons/` 与 `ThemePack/Chips-theme-default-dark/icons/` 当前无正式运行时图标资源；
7. `Chips-EcoSettingsPanel` 已能读取 `iconPath`，但治理页未渲染插件图标。

## 6. 验收要求

本大任务验收完成时，至少应满足：

1. 全仓正式运行时代码与正式模板不再出现 emoji 图标；
2. 全仓不存在旧的内联 SVG 默认图标实现主链路；
3. `ChipsIcon` 已成为所有正式 UI 图标统一出口；
4. 主题 token、ThemePack、组件库和应用层的图标视觉链路全部闭环；
5. 应用启动图标、插件图标、卡片/布局元数据图标、设置治理图标全部完成收口；
6. 脚手架生成的新工程默认走正式图标体系；
7. 相关测试、质量门禁、工作区级验证通过；
8. 共享文档、项目文档、任务文档全部与真实代码一致。
