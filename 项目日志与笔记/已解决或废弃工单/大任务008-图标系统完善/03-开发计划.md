# 大任务008：开发计划

## 1. 目标

按照已经冻结的图标系统方向，完成全生态图标体系的正式实施、旧图标替换、emoji 清退和文档收口，最终让图标成为可复用、可扩展、可验证的共享基础设施，而不是散落在各仓的局部实现。

## 3. 总体路线

### 3.1 核心原则

1. 先冻结公共契约，再动共享基础设施；
2. 先把 `ChipsIcon + 图标描述符 + 主题 token + ThemePack` 建成正式底座，再改应用；
3. 不做兼容设计，不保留旧字段、旧 fallback、旧 emoji 分支；
4. 启动图标链路与运行时 UI 图标链路分层治理，但都必须纳入同一大任务收口；
5. 任何阶段发现新的公共缺口，先回写工单与计划，再推进实现；
6. 每一阶段都必须以正式脚本和正式文档收口，不能只靠人工演示。

### 3.2 主关键路径

主关键路径如下：

- 阶段一 -> 阶段二 -> 阶段三 -> 阶段四 -> 阶段五 -> 阶段六 -> 阶段七 / 阶段八 -> 阶段九 -> 阶段十

说明：

1. 阶段三是运行时 UI 图标体系的主阻塞点，因为组件库必须先提供统一图标组件；
2. 阶段四依赖阶段三完成主题接线；
3. 阶段五负责把 Host / SDK / 治理链路接到同一套正式边界；
4. 阶段六把卡片和布局元数据图标模型改成正式语义对象；
5. 阶段七与阶段八都依赖前六阶段，但可并行推进；
6. 阶段九只有在正式应用侧口径稳定后才能统一升级脚手架；
7. 阶段十负责最终回归、文档一致性和发布级收口。

### 3.3 为什么这样排

当前真实代码中：

1. `ChipsIcon` 尚不存在；
2. ThemePack 还没有真正消费图标运行时资源；
3. 编辑引擎和基础卡片插件仍大量使用 emoji；
4. 设置面板虽能拿到 `iconPath`，但没有图标展示；
5. 脚手架仍在向新工程传播旧图标说明。

如果不先做公共底座，就会在应用层再次复制一批局部实现，导致第二轮漂移。

## 4. 阶段划分

### 4.1 阶段一：图标公共契约与文档基线冻结

目标：

- 清除共享文档内部互相冲突的图标口径；
- 冻结 `ChipsIcon`、图标描述符、ThemePack、Host 边界和 emoji 清零范围。

### 4.2 阶段二：共享图标资产与命名生成链路收口

目标：

- 把 Material Symbols 资产、App 图标源文件、生成物命名和追溯机制整理成单一正式来源。

### 4.3 阶段三：组件库统一图标组件与 Token 落地

目标：

- 在 `Chips-ComponentLibrary` 正式交付统一图标组件、类型、token、契约与测试。

### 4.4 阶段四：主题包与主题运行时图标接线

目标：

- 让默认主题包与暗色主题包具备真实图标运行时能力，而不是停留在 README。

### 4.5 阶段五：Host / SDK / Bridge / 治理图标链路收口

目标：

- 把启动图标、快捷方式图标、治理页图标状态与 SDK 类型一起收口。

### 4.6 阶段六：卡片布局描述符与插件元数据图标模型重构

目标：

- 把基础卡片插件、布局插件和编辑引擎内部描述符中的旧 `icon: string` 口径升级到正式语义模型。

### 4.7 阶段七：查看器、设置面板与布局宿主图标接入

目标：

- 完成治理应用和其他正式应用的图标接线与资产替换。

### 4.8 阶段八：编辑引擎全量图标替换与 Emoji 清退

目标：

- 一次性清理编辑引擎中散落的所有 emoji、字符图标与旧类型。

### 4.9 阶段九：脚手架模板与默认图标资产全面升级

目标：

- 让新生成项目默认就走正式图标体系，不再传播历史债务。

### 4.10 阶段十：全仓回归验证与发布级文档收口

目标：

- 完成所有仓的正式验证、工作区级验证和文档一致性收口。

## 5. 动态推进原则

1. 进入正式实施阶段后，发现新缺口时优先更新本大任务文档和工单；
2. 不允许绕过阶段顺序直接改下游应用；
3. 每次改文档前都要先复核相关代码、`manifest.yaml`、测试与正式链路；
4. 任何阶段未完成正式验证，不进入下一阶段；
5. 任何阶段都不能留下 TODO、占位实现、回退逻辑或“后续再统一”的临时代码。

## 7. 阶段完成判定

只有满足以下条件，某一阶段才算完成：

1. 代码、文档、测试和正式脚本全部收口；
2. 下游阶段所需的公共口径已经冻结；
3. 不再依赖旧图标模型；
4. 搜索结果中不再出现该阶段目标明确要求清除的旧实现；
5. 工单、任务日志和计划文档已同步更新。
