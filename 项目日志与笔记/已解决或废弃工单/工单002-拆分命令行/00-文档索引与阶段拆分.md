# 任务002-拆分命令行：文档索引与阶段拆分

## 1. 任务背景

当前薯片生态中，命令行工具在历史文档与设计中曾混用单一入口与二级命令空间：

- 用户侧 / 运维侧：历史上曾写作 `chips ...`
- 开发者侧：历史上曾写作 `chips dev ...`

最新决策要求将命令行彻底拆分为两个独立命令：

- `chips`：随 Host 软件发布，仅面向终端用户与运维，负责本地主机的启动、停止、配置、日志、主题、插件管理等操作。
- `chipsdev`：随 SDK 发布，仅面向开发者，负责插件/应用工程创建、构建、测试、打包、验证、发布等操作。

不再保留 `chips dev` 这一二级命令形式，不考虑历史兼容性。所有生态仓库与文档需要同步调整到新方案。

## 2. 最终目标

- 存在两套独立命令行工具：
  - `chips`：Host CLI，语义清晰，行为与《Chips Host 命令行使用手册》一致。
  - `chipsdev`：开发者 CLI，行为与《Chips Dev 开发者命令行手册》一致，但命令前缀从 `chips dev` 统一为 `chipsdev`。
- 生态内所有代码仓库（Host、SDK、脚手架、主题包、应用模板、基础卡片插件、组件库等）全部适配新命令：
  - package.json 脚本、CI 脚本、测试用例、技术文档、README 均不再引用 `chips dev`。
  - 所有“开发工具”相关操作统一通过 `chipsdev ...` 调用。
- 生态共用文档与设计原稿中：
  - Host 相关规范只引用 `chips ...`。
  - 开发流程与工具链规范只引用 `chipsdev ...`。
- 提供一套端到端验收清单，保证：
  - 开发者可以从零开始：安装 SDK → 使用 `chipsdev` 创建工程 → 构建/测试/验证 → 打包 `.cpk` → 使用 `chips` 安装并运行；
  - 终端用户与运维仅依赖 `chips` 完成本地主机管理。

## 3. 阶段拆分

本任务拆分为六个阶段，每个阶段单独一份任务文档：

- 阶段一（设计）：《01-阶段一-命令行拆分方案与契约冻结.md》
- 阶段二（Host）：《02-阶段二-Host命令行工具重构与 chips 发布.md》
- 阶段三（SDK）：《03-阶段三-SDK 命令行工具 chipsdev 实现与集成.md》
- 阶段四（生态适配）：《04-阶段四-生态仓库适配与工具链升级.md》
- 阶段五（文档）：《05-阶段五-文档更新与命令行手册改造.md》
- 阶段六（联调验收）：《06-阶段六-联调、CI 与验收闭环.md》

## 4. 里程碑与依赖关系（高层）

1. 先完成阶段一，冻结命令行拆分方案与对外契约。
2. 阶段二、三并行推进（Host CLI 与 SDK CLI 各自实现），但需遵守阶段一的统一规范。
3. 阶段四在阶段二、三的基础能力可用后开始，逐个仓库进行适配。
4. 阶段五贯穿阶段二〜四，可分批更新文档，但最终需要一次统一校对。
5. 阶段六在代码与文档改造完成后进行，负责端到端联调、CI 改造与最终验收。
