## 3. 根因判断

本问题根因不是 `chipsdev run` 子命令实现错误，而是：

1. 生态仓库采用 npm workspaces，当前安装策略为 `hoisted`；
2. `chips-sdk` 的二进制 `chipsdev` 仅落在根工作区 `/node_modules/.bin/chipsdev`；
3. 子工程 `/Chips-CardViewer/node_modules/.bin/` 中没有同名本地二进制；
4. 当前开发环境使用 Volta，执行 `chipsdev` 时首先命中 `/Users/sevenstars/.volta/bin/chipsdev` shim；
5. Volta 在子工程上下文中未把“根工作区 hoisted 的 bin”识别为“当前项目可执行文件”，因此直接报“你的项目里找不到 chipsdev”，命令在进入 `chips-sdk/cli/index.js` 之前就终止。

换句话说，这是一个：

- `Volta 可执行解析策略`
- 与 `npm workspaces hoisted 二进制布局`
- 以及“文档要求可在子工程直接执行 chipsdev”

三者之间未对齐的集成问题。

## 4. 为什么现有“已修复”结论与实际不一致

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/项目日志与笔记/任务004-chipsdev在插件工程内不可直接调用.md` 中的修复结论，更多证明了：

- 子工程声明了 `chips-sdk` 依赖；
- `npm run ...` 或 npm 参与 PATH 解析的调用方式可用；
- `chipsdev` CLI 在工作区内存在。

但它没有覆盖当前这条真实失败路径：

- 开发者在安装了 Volta 的终端中；
- 直接在子工程目录手敲 `chipsdev run`。

因此当前现象应认定为：

- 旧问题在“Volta + workspace hoist”场景下并未真正闭环，或
- 已闭环结论没有覆盖完整验收矩阵。

## 5. 影响判断

该问题会直接影响以下体验：

- 文档写法与实际终端行为不一致；
- 子工程目录内直接执行 `chipsdev` 系列命令失败；
- 开发者容易误判为 `chipsdev run` 实现损坏，实际却是命令入口层失败；
- 同一仓库中出现“`npm run build` 能用，但 `chipsdev build` 不能用”的分裂体验。

## 6. 当前结论

当前问题原因已经明确：

> `chipsdev run` 报错的直接原因，是 Volta 在 `Chips-CardViewer` 子工程目录下找不到“当前项目本地”的 `chipsdev` 可执行文件；而 `chips-sdk` 的二进制被 npm workspace hoist 到了生态根工作区，导致 Volta 不认这份根级二进制，命令在进入 `chipsdev` CLI 之前就失败。

这属于生态工具链集成问题，不属于业务代码问题。

本工单已于 2026-03-09 正式闭环，修复方案如下：

   - 保证新工程不会再次出现“文档允许直接执行 chipsdev，但 Volta 无法解析”的回归问题。

4. **补齐生态级校验**
