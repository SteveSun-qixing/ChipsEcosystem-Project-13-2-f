# 工单012：统一复合卡片显示链路未真正实现

## 1. 背景

在 `Chips-CardViewer` 联调中，卡片查看器应用已完成以下能力闭环：

- 拖拽卡片文件进入应用窗口；
- 通过 Host preload 正式解析拖拽文件本地路径；
- 调用 `chips-sdk` 的 `client.card.compositeWindow.render()`；
- 成功收到 Host `card.render` 返回结果；
- 将 iframe 成功挂载到页面并触发原生 `load`。

但最终内容区域为空白。

## 2. 现场证据

### 2.1 前端日志证据

实际日志已确认：

- 拖拽路径解析成功；
- `card.render` 调用成功；
- SDK 返回 iframe；
- iframe 已挂载且已触发 `load`；
- 不是窗口、Bridge、权限、拖拽、IPC 或 YAML 解析问题。

### 2.2 Host 渲染产物证据

对以下文件直接调用 Host `CardService.render(..., { target: 'card-iframe' })`：

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/ProductFinishedProductTestingSpace/薯片生态介绍.card`

返回 HTML 摘要显示：

- 有卡片标题 `薯片生态介绍`；
- 内容区仅出现两个文件名：
  - `6at8Obawde.yaml`
  - `Qzea3RKc0A.yaml`

即 Host 当前并未将 `content/*.yaml` 对应的基础卡片真正渲染为可视内容。

## 3. 根因判断

当前生态里的“统一复合卡片显示链路”并未真正实现到文档要求的程度。

当前实际行为：

- SDK `client.card.compositeWindow.render()`
  - 调 `card.render`
  - 得到一个声明式 UI HTML 片段
  - 塞入 `iframe.srcdoc`

当前缺失行为：

- 根据 `structure.yaml` 分发基础卡片节点；
- 查找对应基础卡片插件；
- 调用基础卡片插件渲染模块生成各自 HTML/iframe；
- 将多个基础卡片结果按结构和布局拼接；
- 交付真正的复合卡片窗口。

因此这不是 CardViewer 业务层 bug，而是 Host / SDK / 基础卡片插件运行时之间的生态底层能力缺口。

## 4. 受影响范围

- `Chips-CardViewer` 无法作为正式卡片查看器交付；
- 编辑器中的复合卡片预览链路同样不可信；
- 任何未来依赖 `client.card.compositeWindow.render()` 的应用都会得到错误结果；
- 生态共用文档当前描述的“统一显示链路”与真实实现不一致。

## 5. 按规范当前为什么必须停止

根据仓库开发要求：

- 遇到 SDK 工具和接口不能正常使用；
- 遇到生态底层、主机、内核、公共基础层未打通；
- 遇到需要临时开发才能继续推进；

必须先在《项目日志与笔记》的《问题工单.md》中登记，并创建独立任务说明文档，然后停止当前对话，等待用户处理后再继续。

本问题完全符合上述条件。

## 6. 建议后续由底层统一处理的方向

1. 明确 `card.render` 与 `client.card.compositeWindow.render()` 的职责边界；
2. 打通复合卡片 -> 基础卡片插件渲染分发链路；
3. 明确基础卡片渲染产物是 HTML、iframe 还是统一 runtime node tree；
4. 明确 ready / node-error / fatal-error 的正式事件来源；
5. 让 CardViewer、编辑器、其他应用统一复用同一条正式链路，而不是各自实现渲染逻辑。

## 7. 当前结论

当前 `Chips-CardViewer` 外层接入链路已基本打通，但生态正式要求的“统一复合卡片显示链路”尚未闭环。

因此本任务在此停住，等待工单012处理完成后再继续推进业务开发。
