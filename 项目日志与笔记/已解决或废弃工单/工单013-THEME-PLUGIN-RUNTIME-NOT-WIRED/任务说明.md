# 工单013：主题包安装与 Host 主题运行时装载链路未闭环

## 1. 背景

根据生态设计原稿、主题系统设计文档与默认主题包需求，薯片生态的视觉外观应由主题包统一提供，官方默认主题 `chips-official.default-theme` 应作为 Host 安装完成后的默认主题存在。

当前用户希望：

- 打包 `ThemePack/Chips-default`；
- 通过 `chips` 命令安装主题包；
- 让查看器、卡片渲染与整个 UI 链路都真正使用主题系统；
- 默认主题表现为白色、干净、优雅的官方默认主题。

在正式推进前，必须先确认主题包安装后能否被 Host 主题服务与 Theme Runtime 正式识别与消费。

## 2. 规范依据

### 2.1 生态与主题规范

- `生态设计原稿/06-前端多样化和主题系统.md`
  - 明确视觉样式由主题包负责，组件库不负责样式。
- `生态共用技术文档/架构设计/15-主题系统设计.md`
  - 明确 L11 Theme Runtime 负责 token 解析、作用域覆盖、变量注入；
  - 公开接口包括 `theme.list/apply/getCurrent/getAllCss/resolve/contract.get`。
- `生态共用技术文档/主题系统/01-主题包开发指南.md`
  - 明确主题包是正式插件类型，承载 CSS 与 token。
- `ThemePack/Chips-default/需求文档/01-默认主题包需求总览.md`
  - 明确官方默认主题应作为 Host 安装完成后的默认主题。
- `ThemePack/Chips-default/技术文档/01-默认主题包技术架构与集成说明.md`
  - 明确 Host 侧加载主题包后应形成 `ThemeRecord`，其 `css` 来自 `dist/theme.css`，`tokens` 来自 `dist/tokens.json`。

### 2.2 当前实现证据

- `Chips-Host/src/main/services/register-host-services.ts`
  - `buildState()` 直接内置了一个硬编码的 `state.themes` 列表；
  - `theme.list/apply/getCurrent/getAllCss/resolve` 仅围绕这份内存数据工作；
  - 当前没有看到从 `PluginRuntime` 中读取 `type: theme` 插件并组装为正式 `ThemeRecord` 的链路。
- `.chips-host-dev/plugin-runtime.json`
  - 当前开发工作区已安装插件中没有主题插件；
  - 即使后续通过 `chips plugin install` 安装主题包，Host 现有主题服务也缺少读取与挂载主题插件产物的实现证据。

## 3. 根因判断

当前生态中的“主题包 -> Host 主题服务 -> Theme Runtime -> UI 层”正式装载链路未闭环。

问题不只是“当前没安装主题包”，而是：

1. Host 主题服务仍以硬编码内存主题为主；
2. 主题插件安装产物没有正式进入 `theme.list/apply/getAllCss/resolve` 的数据源；
3. 默认主题包虽已具备正式工程与构建产物约定，但 Host 尚未把它作为唯一真实来源接入。

因此，直接继续做应用层开发、手动复制 CSS、或在查看器里绕过 Host 注入主题变量，都属于临时开发，不符合仓库要求。

## 4. 影响范围

- `chips` 命令安装主题包后，Host 不一定能真正消费其 CSS/tokens；
- `theme.getAllCss` 返回的可能仍是简化 stub，而不是正式默认主题包产物；
- `theme.resolve` 与 `theme.apply` 无法建立在真实主题插件之上；
- 所有依赖 Theme Runtime 的应用、组件库、卡片渲染链路都会受到影响；
- 用户要求的“默认主题白色、全生态统一、无临时实现”当前无法正式满足。

## 5. 复现与核查步骤

1. 查看 `ThemePack/Chips-default` 工程，确认其具备正式主题包结构与主题 ID `chips-official.default-theme`。
2. 查看 `Chips-Host/src/main/services/register-host-services.ts`，确认 `state.themes` 为硬编码列表。
3. 查看 `theme.list/apply/getAllCss/resolve` 实现，确认数据源仅来自 `state.themes`。
4. 查看 `.chips-host-dev/plugin-runtime.json`，确认当前已安装插件中没有主题插件。
5. 对照 `ThemePack/Chips-default/技术文档/01-默认主题包技术架构与集成说明.md`，确认 Host 设计预期与当前实现存在偏差。

## 6. 期望行为

- `ThemePack/Chips-default` 打包后可通过 `chips` 正式安装；
- Host 启动或主题服务初始化时，能够从已安装 `type: theme` 插件中加载 `dist/theme.css` 与 `dist/tokens.json`；
- `theme.list/apply/getCurrent/getAllCss/resolve/contract.get` 全部基于真实主题插件数据工作；
- 官方默认主题 `chips-official.default-theme` 应成为正式白色默认主题基线，而不是硬编码 stub；
- 主题包、Theme Runtime、组件库、应用层使用同一份正式主题来源。

## 7. 建议处理方向

1. 明确 Host 主题服务的数据源切换为已安装主题插件，而不是硬编码 `state.themes`；
2. 增加主题插件装载逻辑：读取 manifest、定位 `dist/theme.css` 与 `dist/tokens.json`、构造 `ThemeRecord`；
3. 将默认主题包的安装、启用、应用纳入 Host 启动与工作区初始化流程；
4. 对 `theme.getAllCss`、`theme.resolve`、`theme.apply`、`theme.contract.get` 建立真实集成测试；
5. 清理历史 stub 数据，避免出现“主题接口返回成功但实际未使用真实主题包”的假成功状态。

## 8. 正式处理结果

- Host 主题服务已切换为从已安装 `type: theme` 插件装载正式主题数据；
- `ThemePack/Chips-default` 的 `dist/theme.css` 与 `dist/tokens.json` 已进入 `theme.list/apply/getCurrent/getAllCss/resolve/contract.get` 的统一数据源；
- 默认主题包已完成打包、安装与主题命令校验，官方默认白色主题已成为正式运行时基线。

## 9. 需要协作方

- Host 仓库
- Theme Runtime 维护方
- ThemePack 默认主题包维护方
- SDK 主题接口维护方
- 组件库主题接入维护方

## 10. 当前结论

“主题包安装与 Host 主题运行时装载链路”已正式闭环。主题包现已成为 Host 主题运行时的唯一正式主题来源，不再依赖内存硬编码 stub。
