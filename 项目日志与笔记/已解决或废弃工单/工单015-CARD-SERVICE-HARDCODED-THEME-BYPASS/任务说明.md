# 工单015：复合卡片渲染链路绕过主题系统并硬编码暖色默认主题

## 1. 背景

用户要求：

- 复合卡片渲染必须正式使用主题系统；
- 取消所有硬编码主题；
- 卡片默认主题应为官方白色默认主题；
- 查看器、卡片渲染、主题系统链路应统一。

当前调查发现，复合卡片渲染结果中的淡棕色并非来自默认主题包，而是 `CardService` 内部硬编码主题。

## 2. 现场证据

### 2.1 CardService 内置暖色默认主题

`Chips-Host/packages/card-service/src/card-service.ts` 中 `createDefaultTheme()` 当前写死：

- `surface.canvas: #f5f4ef`
- `surface.card: #ffffff`
- `surface.border: #d6d0c5`
- `text.primary: #181512`
- `text.secondary: #5b5148`
- `accent.primary: #956a3a`

这是一套暖纸张色风格，不是官方默认主题包的白色视觉基线。

### 2.2 渲染入口默认直接使用硬编码主题

同文件 `render()` 中：

- `declaredThemeId = metadata.theme ?? 'chips-official.default-theme'`
- `theme = options?.theme ?? createDefaultTheme(declaredThemeId)`

即只要上游没有显式传入 `options.theme`，卡片渲染就直接绕回硬编码默认主题。

### 2.3 最终输出 HTML/CSS 继续固化这套暖色

`createBasecardThemeCss()` 与 `createCompositeDocument()` 会把：

- canvas / surface / border / accent 等值直接写入 iframe HTML；
- 其中 `accent.primary #956a3a` 与暖色半透明背景直接导致当前“淡棕色卡片”观感。

### 2.4 SDK 当前没有把正式主题快照传给 card.render

`Chips-SDK/src/api/card.ts` 当前调用 `card.render` 时，只传：

- `target: 'card-iframe'`
- `mode`

没有看到通过 Theme Runtime 获取当前主题快照并传入 `options.theme` 的链路。

## 3. 根因判断

当前正式问题不是“默认主题不好看”，而是“复合卡片渲染链路完全绕过了主题系统”。

具体表现：

1. CardService 自己构造默认主题；
2. SDK 显示接口没有把当前主题正式传入；
3. CardService 输出 HTML 时再次把这套硬编码主题固化进 iframe；
4. 最终卡片窗口与应用壳层、主题包、Theme Runtime 彼此脱节。

这属于生态底层链路缺口，不能在业务层用手动覆盖颜色的方式修补。

## 4. 影响范围

- 卡片查看器显示结果与默认主题包不一致；
- 未来编辑器预览、封面渲染、其他卡片消费应用都会出现同类问题；
- 主题包切换对卡片内容窗口不生效或只部分生效；
- 生态宣称的“统一主题系统”在卡片渲染链路上失真。

## 5. 复现步骤

1. 启动当前 Host + CardViewer 链路。
2. 打开任意复合卡片。
3. 观察复合卡片页面背景、边框与强调色呈淡棕暖色。
4. 查看 `Chips-Host/packages/card-service/src/card-service.ts`，可发现该配色来自硬编码默认主题。
5. 对照 `ThemePack/Chips-default/theme.css` 与默认主题 token，可确认默认主题应为白色浅色基线。

## 6. 期望行为

- `card.render` / `client.card.compositeWindow.render()` 必须消费正式 Theme Runtime 输出；
- 复合卡片、基础卡片 iframe 与应用壳层应共享同一主题来源；
- 默认卡片主题应来自官方默认主题包，而不是 `CardService` 私有默认皮肤；
- 彻底取消卡片渲染链路中的硬编码主题常量。

## 7. 建议处理方向

1. 取消 `CardService` 中私有 `createDefaultTheme()` 的正式渲染职责；
2. 由 Host 主题服务 + Theme Runtime 生成正式 `ThemeSnapshot` 后传入 `card.render`；
3. SDK `client.card.compositeWindow.render()` 需要与当前主题链路打通，而不是只传 `mode/target`；
4. `createBasecardThemeCss()` 与 `createCompositeDocument()` 只消费主题快照，不再定义私有视觉默认值；
5. 为复合卡片渲染补齐主题一致性测试，确保默认主题、切换主题、局部作用域主题都能生效。

## 8. 正式处理结果

- `card-service` 已移除私有暖色默认主题在正式渲染链路中的职责；
- SDK `client.card.compositeWindow.render()` 已改为把 Theme Runtime 的主题快照传入 `card.render`；
- 复合卡片与基础卡片 iframe 统一消费正式主题系统输出，不再通过私有暖色常量兜底。

## 9. 需要协作方

- Host `card-service`
- Host Theme Runtime / theme 服务
- SDK 统一显示接口维护方
- 基础卡片插件渲染链路维护方

## 10. 当前结论

卡片渲染主题链路已回归正式 Theme Runtime，默认淡棕色硬编码皮肤已移除，默认卡片视觉现与官方白色主题保持一致。
