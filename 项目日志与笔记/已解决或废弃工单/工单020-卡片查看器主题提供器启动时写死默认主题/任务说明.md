# 工单020：卡片查看器主题提供器启动时写死默认主题

## 1. 背景

本工单来源于 2026-03-10 对卡片查看器暗夜主题链路的专项排查。当前问题已经确认存在，需要作为正式生态问题单独跟踪。

## 2. 问题描述

- 卡片查看器应用入口当前在主题提供器上直接写死默认主题标识，而不是在启动时先读取当前活动主题。
- 这会导致应用的主题运行时初始状态固定落在默认主题，只能等待后续主题变更事件到来后才可能修正。
- 如果应用启动时当前主题已经是暗夜主题，但本次启动过程中没有新的主题变更事件，应用层主题上下文就会长期停留在错误的默认主题状态。

## 3. 影响范围

- 应用层主题上下文与 Host 当前主题状态可能不一致。
- 组件运行时、缓存键、依赖主题上下文的后续行为都可能基于错误初始值执行。
- 其他应用如果复制当前实现方式，会继续扩散同类问题。

## 4. 复现步骤

1. 将工作区当前主题切换为暗夜主题。
2. 直接重新启动卡片查看器应用。
3. 检查应用入口主题提供器的初始主题值，仍然是默认主题而不是当前主题。

## 5. 期望行为

- 应用启动时应以 Host 当前主题作为主题运行时的初始状态。
- 主题变更事件只用于增量同步，不应承担初始主题装载职责。

## 6. 需要协作方

- 卡片查看器维护方、组件库主题运行时维护方、Host 主题服务维护方

## 7. 当前状态

- 已处理，见下方正式处理结果与验证结果。

## 8. 正式处理结果

- 卡片查看器应用入口不再把 `chips-official.default-theme` 直接写死为唯一初始主题；
- 应用启动时先读取文档注入的 `data-chips-theme-id` 与 `data-chips-theme-version`，再调用 `client.theme.getCurrent()` 做校准；
- `theme.changed` 继续只承担增量同步职责，不再承担初始主题装载职责。

## 9. 验证结果

- `npm --workspace Chips-CardViewer test` 已通过；
- 重新启动查看器后，应用壳层初始主题与 Host 当前主题保持一致。

## 10. 当前状态

- 已闭环（2026-03-10）
