# 工单027：生态设置面板正式开发遇到 TypeScript 集成基线缺口

## 1. 背景

在继续推进 `Chips-EcoSettingsPanel` 正式开发时，已经按设计原稿、生态共用技术文档与当前 Host/SDK 契约开始把脚手架示例替换为正式设置面板实现。

本轮实现严格使用：

- React
- `chips-sdk`
- `@chips/component-library`
- 严格 TypeScript 模式（工程 `tsconfig.json` 已启用 `strict`）

在把页面、服务适配层、主题/多语言/插件治理逻辑接入正式工程后，对项目执行 TypeScript 编译校验时，发现当前生态基础能力仍存在两个正式阻断点，导致设置面板无法在“不降级类型检查、不写本地临时声明、不绕过正式组件接入”的前提下继续完成交付。

## 2. 阻断结论

当前阻断点包括：

1. `@chips/component-library` 缺少正式 TypeScript 类型声明；
2. `chips-sdk` 对外 `Client` 类型定义自身存在属性冲突，导致严格 TS 工程无法通过编译。

这两个问题都属于生态级基线缺口，不是设置面板业务层可以合法兜底的问题。

如果继续开发，将不可避免落入以下违规路径之一：

- 在设置面板项目本地补写 `declare module '@chips/component-library'` 之类的临时声明；
- 在业务代码中大量 `as any` 降级；
- 绕过 `chips-sdk` 的正式类型体系直接退回弱类型 Bridge 调用；
- 关闭严格 TS 或绕过类型门禁。

以上都不符合当前仓库规则，因此本工单视为正式阻断项。

## 3. 证据

### 3.1 组件库缺少正式类型声明

执行命令：

```bash
npm --workspace Chips-EcoSettingsPanel exec tsc --noEmit
```

出现错误：

- `Could not find a declaration file for module '@chips/component-library'`

当前解析到的包入口为：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-ComponentLibrary/packages/component-library/index.js`

但该包没有正式 `.d.ts` 导出，也没有在 `package.json` 中提供类型入口。

这意味着生态要求“前端使用 React + 薯片组件库 + strict TypeScript”目前无法正式同时满足。

### 3.2 SDK Client 类型定义存在属性冲突

同一次编译中还出现：

- `Interface 'Client' incorrectly extends interface 'CoreClient'`

具体文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/src/types/client.ts`

当前 `CoreClient` 中定义：

- `config: Readonly<ClientConfig>`

而 `Client` 又追加了：

- `config: import("../api/config").ConfigApi`

这两个 `config` 含义完全不同：

1. 一个是 SDK 客户端初始化配置；
2. 一个是 Host `config.*` 服务域 API。

结果是 `Client extends CoreClient` 在类型层面天然冲突，任何严格 TS 工程只要消费该定义，就会在编译期失败。

### 3.3 设置面板当前开发被直接阻断

在 `Chips-EcoSettingsPanel` 中，已经开始按正式架构接入：

- 应用壳层
- 主题治理页
- 多语言页
- 应用插件页
- 组件展示页
- 本地 i18n 资源与 Runtime Provider

但由于上面两个生态缺口，工程无法通过正式 TypeScript 校验，因此不能继续宣称“高质量完成正式开发”。

## 4. 影响范围

- `Chips-EcoSettingsPanel` 无法继续在严格 TS 与正式组件库/SDK 接入约束下完成开发。
- 所有新的 React + TypeScript 应用插件工程，只要正式依赖 `@chips/component-library` 与 `chips-sdk`，都可能遇到相同问题。
- 这不是设置面板单项目问题，而是生态前端工程基线问题。

## 5. 复现步骤

1. 在生态根工作区进入 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EcoSettingsPanel`。
2. 按正式方式在源码中引入 `@chips/component-library` 与 `chips-sdk`。
3. 执行：
   ```bash
   npm --workspace Chips-EcoSettingsPanel exec tsc --noEmit
   ```
4. 观察编译失败，包含：
   - 组件库缺少声明文件；
   - SDK `Client` / `CoreClient` 类型冲突。

## 6. 期望行为

- `@chips/component-library` 必须正式提供完整、可发布、可被 TS 工程消费的类型声明；
- `chips-sdk` 必须修复 `Client` 与 `CoreClient` 的类型冲突，明确区分 SDK 初始化配置与 Host `config.*` 服务域 API；
- 生态标准应用插件在 `strict` 模式下接入 React + SDK + 组件库后，应能直接通过 `tsc --noEmit`。

## 7. 建议处理方向

### 7.1 组件库

- 为 `@chips/component-library` 正式补齐类型产物；
- `package.json` 中提供明确的 `types`/`exports.types` 入口；
- 确保聚合包导出的所有组件、元数据与 helper 都有可消费类型。

### 7.2 SDK

需要重新命名或拆分当前冲突字段，避免：

- `CoreClient.config`（客户端初始化配置）
- `Client.config`（Host `config.*` 领域 API）

共享同一属性名。

最终方案应保持正式、清晰、无兼容包袱，而不是在业务层绕过去。

## 10. 处理结果

状态：已解决（2026-03-11）

### 10.1 组件库正式类型发布已闭环

已在 `Chips-ComponentLibrary` 内完成正式类型面收口：

- 为 `@chips/component-library`、`@chips/components`、`@chips/hooks`、`@chips/card-runtime` 明确补齐 `types` / `exports.types`
- 为聚合包新增 `index.d.ts`
- 为 `@chips/components`、`@chips/hooks`、`@chips/card-runtime` 提供正式声明入口
- 为聚合包补充运行时导出测试与类型冒烟测试，不再允许业务层本地声明兜底

同时对聚合层同名 helper 冲突进行了正式命名收口：

- `toComponentStandardError`
- `toCardRuntimeStandardError`

避免业务层依赖 `export *` 的歧义导出结果。

### 10.2 SDK Client 类型冲突已修复

已在 `Chips-SDK` 中将 Core Client 的初始化配置快照字段由：

- `config: Readonly<ClientConfig>`

正式调整为：

- `clientConfig: Readonly<ClientConfig>`

这样：

- `client.config.*` 继续保持 Host `config.*` 服务域 API 语义
- `client.clientConfig` 明确表示 SDK 客户端初始化配置快照

本次处理没有添加兼容别名，也没有在业务层绕过类型系统。

### 10.3 文档与质量门禁已同步

已同步更新：

- 生态共用组件库接入文档
- 应用插件开发文档
- 组件库仓库内部包设计、接入指南、卡片显示链路、测试策略、发布检查清单
- SDK Core Client 设计文档

组件库质量门禁已新增 `test:types`，并接入：

- `npm --workspace Chips-ComponentLibrary run verify`
- `npm --workspace Chips-ComponentLibrary run quality:gate`
