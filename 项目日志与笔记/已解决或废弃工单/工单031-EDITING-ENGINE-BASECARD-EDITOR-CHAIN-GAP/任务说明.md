# 工单031：编辑引擎基础卡片编辑器正式装载链路缺失

> 当前状态：已闭环（2026-03-13）

## 1. 背景

当前用户任务要求“编辑引擎可以正确加载并使用基础卡片”，并明确要求：

- 必须使用薯片 SDK 与正式生态链路；
- 不允许临时开发、简化版、占位实现；
- 如果发现生态底层、主机、内核、SDK、公共基础层没有打通，必须先登记问题工单并停止当前推进。

在本次排查中，已经确认编辑引擎当前问题不只是页面没接线，而是正式的“基础卡片编辑器装载链路”本身没有闭环。

## 2. 现场证据

### 2.1 编辑引擎应用层仍是 stub

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/EditPanel/PluginHost.tsx`

关键现状：

- `loadPlugin()` 内部把插件组件直接写成 `const component = null;`
- 没有通过 Host / SDK 请求基础卡片编辑器
- 失败后直接回退到 `DefaultEditor`

这说明编辑引擎目前并没有正式加载基础卡片插件编辑器。

### 2.2 编辑引擎卡片内容区也未接正式显示链路

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/CardWindow/CardWindow.tsx`

当前只是遍历基础卡片并输出占位块：

- 没有调用 `client.card.compositeWindow.render()`
- 没有把 Host 返回的复合 iframe 挂到编辑引擎窗口

也就是说，编辑引擎的“显示链路”与“编辑链路”都没有正式接通。

### 2.3 Host 只有基础卡片渲染链路，没有编辑器装载链路

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/card-service/src/card-service.ts`

当前 Host 已正式支持：

- 解析复合卡片结构
- 根据 `capabilities.cardTypes` 查找基础卡片插件
- 加载插件 entry
- 调用插件导出的 `renderBasecardView`
- 组装复合卡片 iframe 文档

但当前 Host 没有看到：

- `renderBasecardEditor` 对应的正式服务装载链路
- 基础卡片编辑器对应的 Host 路由动作
- “向编辑引擎交付编辑器 iframe / 编辑宿主”的正式实现

### 2.4 SDK 也缺少正式编辑器入口

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/src/api/card.ts`

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/src/api/plugin.ts`

当前 SDK 有：

- `card.render`
- `card.coverFrame.render`
- `card.compositeWindow.render`
- `plugin.getCardPlugin`

当前 SDK 没有：

- “请求基础卡片编辑器”的正式 API
- “交付编辑面板 iframe / editor host”的正式 API

因此编辑引擎无法仅通过 SDK 完成基础卡片编辑器装载。

### 2.5 基础卡片插件本身已经具备编辑器导出，但没有正式运行时接入

`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-BaseCardPlugin/richtext-BCP/src/index.ts`

当前富文本基础卡片插件已经导出：

- `renderBasecardView`
- `renderBasecardEditor`

说明插件侧的编辑器模块已经存在；缺的是 Host / SDK / 编辑引擎之间的正式装载契约与运行时链路。

## 3. 根因判断

当前真实问题是：

1. Host 正式闭环的是“基础卡片渲染链路”，不是“基础卡片编辑器链路”。
2. SDK 正式暴露的是“显示接口”，不是“编辑器装载接口”。
3. 编辑引擎 React 迁移后保留了 stub 宿主，没有接到正式链路。

因此“编辑引擎无法正确加载基础卡片插件”并不是单点 bug，而是生态正式能力缺口。

## 4. 正式处理结果

### 4.1 Host / 内核侧

- `CardService` 已补齐基础卡片编辑器装载能力，正式消费插件导出的 `renderBasecardEditor`；
- Host 已新增 `card.renderEditor` 路由和 schema 注册；
- 卡片服务已支持压缩态 `.card` 文件与解包态 `.card/` 目录两种输入形态。

### 4.2 SDK 侧

- 已新增 `client.card.editorPanel.render()`；
- 已新增 `onReady`、`onChange`、`onError` 事件订阅接口；
- 已新增 `client.card.compositeWindow.onNodeSelect()`，用于编辑模式下接收基础卡片节点选中事件；
- `route-manifest` 与客户端测试已同步更新。

### 4.3 编辑引擎侧

- 卡片窗口已统一改为 `client.card.compositeWindow.render({ cardFile, mode })`；
- 复合卡片 `preview` 模式点击基础卡片后，已通过正式事件链切换当前 `baseCardId`；
- 编辑面板已统一改为 `client.card.editorPanel.render({ cardType, initialConfig, baseCardId })`；
- 已移除 `PluginHost` 中把 `DefaultEditor` 作为正式路径兜底的做法；
- 已移除卡片窗口内部重复的基础卡片列表入口，交互统一回到“直接点击板块 -> 右侧编辑面板”的正式形态；
- 工作区卡片读写已对齐解包态 `.card/metadata.yaml`、`.card/structure.yaml`、`content/*.yaml` 与 `.card/cover.html`。

### 4.4 验证与文档

- Host 已补充渲染目录态 `.card`、`preview` 节点选中桥接与 `card.renderEditor` 回归测试；
- SDK 已补充 compositeWindow 节点选中、editorPanel 渲染与事件过滤测试；
- EditingEngine 已补充解包态卡片初始化与持久化测试，以及节点选中驱动编辑面板切换的单测；
- 生态共用文档、生态设计原稿和 SDK 技术文档已同步更新。

## 5. 当前结论

当前问题已经通过正式方案闭环：编辑引擎基础卡片显示链路与编辑链路均已切换到 Host + SDK 正式实现，不再依赖 stub、mock 或本地插件直连方案。
