# 工单032：编辑引擎主题与多语言运行时正式接线缺失

> 当前状态：已修复并闭环（2026-03-13）

## 1. 背景

当前用户任务明确要求：

- 前端使用 React 技术栈；
- 严格按照薯片生态主题系统与多语言系统规范开发；
- 不允许临时开发、简化版或绕过正式运行时的方案；
- 如果发现生态链路未打通或架构缺口，需要先登记问题工单并暂停推进。

在完成旧项目与新项目基础卡片显示链路、编辑链路和 Host/SDK 正式链路排查后，继续复核编辑引擎的应用壳层运行时接线，确认当前还存在一个未闭环问题：编辑引擎没有把主题系统和多语言系统按正式运行时方式接入。

## 2. 现场证据

### 2.1 编辑引擎当前只读取一次主题，没有正式订阅 `theme.changed`

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/App.tsx`

现状：

- 应用启动时调用一次 `client.theme.getCurrent()`；
- 只把 `themeId` 传给 `ChipsThemeProvider`；
- 没有传入正式 `eventSource/eventName`；
- 没有在应用层订阅 `theme.changed`。

这说明编辑引擎当前只有“初始主题读取”，没有“正式增量同步链路”。

### 2.2 组件库主题运行时只有在 props 变化或事件到达时才会刷新

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-ComponentLibrary/packages/hooks/src/index.js`

现状：

- `ChipsThemeProvider` 内部通过 `themeId/version` 或 `eventSource + eventName` 更新运行时状态；
- 若应用层不传 `eventSource`，则不会收到 `theme.changed` 事件；
- `cacheKey` 不会在主题切换后按正式链路刷新。

这意味着当前编辑引擎即使包裹了 `ChipsThemeProvider`，也没有真正形成可持续的主题切换运行时。

### 2.3 编辑引擎卡片窗口依赖 `themeRuntime.cacheKey` 重渲染复合卡片

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/CardWindow/CardWindow.tsx`

现状：

- 复合卡片窗口的 iframe 重渲染依赖数组中包含 `themeRuntime.cacheKey`；
- 但由于应用壳层没有把 `theme.changed` 接入 `ChipsThemeProvider`，`cacheKey` 不会按正式链路变化；
- 主题切换后，应用壳层和复合卡片窗口无法保证同步刷新。

这与 `生态共用技术文档/组件库/07-复合卡片窗口与SDK显示链路标准.md` 中“主题切换后复合卡片窗口重新触发整条渲染链”的要求不一致。

### 2.4 编辑引擎当前没有正式订阅 `language.changed`

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/services/i18n-service.ts`

现状：

- 当前只有 `initLocale()` 启动初始化；
- 当前只有 `setCurrentLocale()` 在本地主动设置时刷新词典；
- 应用层没有正式订阅 `language.changed`。

这意味着当用户在设置面板或其他应用里切换系统语言后，编辑引擎不会按规范自动刷新 UI 文案。

### 2.5 生态内已有正式参考实现，编辑引擎尚未对齐

参考文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/src/App.tsx`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EcoSettingsPanel/src/app/providers/RuntimeProvider.tsx`

现状：

- 卡片查看器已通过 `ChipsThemeProvider(themeId/version/eventSource/eventName)` 接入正式主题增量同步；
- 设置面板已正式订阅 `theme.changed` 和 `language.changed`；
- 编辑引擎当前没有对齐这些正式运行时接线模式。

## 3. 根因判断

当前问题不是 Host、SDK 或基础卡片插件能力缺失，而是编辑引擎应用壳层未把已有正式运行时能力接上：

1. 主题系统没有完成从 Host 到应用壳层的正式事件同步。
2. 组件库主题运行时没有收到编辑引擎提供的正式主题事件源。
3. 多语言系统没有完成 `language.changed` 的正式订阅与本地词典同步。

因此这属于编辑引擎应用层运行时集成缺口，而不是基础卡片渲染/编辑链路本体缺失。

## 4. 影响范围

- 编辑引擎壳层与卡片内容区可能出现主题状态不一致；
- 主题切换后复合卡片窗口不会稳定触发重新渲染；
- 语言切换后编辑引擎界面文案不会按规范刷新；
- 后续继续开发面板、工具窗口、设置项等 UI 功能会建立在不完整运行时之上，造成返工风险。

## 5. 期望处理方向

### 5.1 主题运行时

- 在编辑引擎应用壳层补齐正式主题启动读取与增量同步；
- `ChipsThemeProvider` 接入 `themeId/version/eventSource/eventName`；
- 视组件库实际消费情况补齐 resolver 接线；
- 确保 `themeRuntime.cacheKey` 能真实驱动复合卡片窗口重渲染。

### 5.2 多语言运行时

- 在编辑引擎应用壳层正式订阅 `language.changed`；
- 语言变更后同步刷新本地 i18n 注册表与界面；
- 验证工具窗口、卡片窗口、空状态、错误态等文案全部能动态刷新。

### 5.3 验收要求

- 主题切换后，应用壳层、工具窗口、复合卡片窗口结果一致；
- 语言切换后，编辑引擎 UI 文案无需重启即可更新；
- 增补对应单元测试/集成测试，不引入临时兼容分支。
