# 工单033：编辑引擎实时保存与复合预览时序竞态

> 当前状态：已闭环（2026-03-13）

## 1. 背景

在完成工单031的基础卡片正式编辑链路后，继续按用户要求回归“创建基础卡片 -> 编辑面板输入 -> 内容实时保存 -> 复合卡片实时显示”的完整链路，发现实时编辑体验仍然没有正式闭环。

用户的直接现象是：

- 创建基础卡片后内容不能实时保存；
- 在编辑面板修改内容后，复合卡片上仍显示空白或旧内容；
- 这与“必须完整打通创建、编辑、预览整条链路”的任务目标不一致。

## 2. 现场证据

### 2.1 基础卡片编辑器只在失焦/提交时发出变更

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-BaseCardPlugin/richtext-BCP/src/editor/panel.tsx`

现状：

- 富文本基础卡片编辑器原先只在 `focusout` / `submit` 时触发 `onChange`；
- 用户持续输入时不会把最新配置回传给编辑引擎；
- 因此“实时保存”链路在编辑器入口就没有被正式触发。

### 2.2 编辑引擎预览刷新早于磁盘持久化完成

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/core/card-service.ts`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/CardWindow/CardWindow.tsx`

现状：

- 编辑面板变更先更新内存中的 `basicCard.data`；
- `CardWindow` 原先按 `modifiedAt` 直接触发复合卡片 iframe 重渲染；
- 但 Host / SDK 复合卡片渲染读取的是磁盘上的 `.card/content/*.yaml`，不是编辑器内存快照；
- 当写盘仍在异步进行时，预览已提前刷新，最终读到的可能是旧内容甚至空内容。

## 3. 根因判断

当前问题不是 Host 统一渲染接口错误，而是编辑器变更发射时机和编辑引擎持久化/预览刷新时序不一致：

1. 编辑器输入期没有持续发出变更；
2. 持久化链路没有把连续输入串行化到“最新快照”；
3. 预览刷新依赖了内存态变更时间，而不是“磁盘已写完”的正式信号。

## 4. 正式处理结果

### 4.1 基础卡片编辑器恢复实时变更发射

- richtext 基础卡片编辑器已增加输入期防抖发射（`EMIT_DEBOUNCE_MS = 120`）；
- 保留失焦/提交时的 flush 行为；
- 空正文校验改为提示型，不再阻塞实时输入阶段的正式变更流。

### 4.2 编辑引擎持久化改为串行队列

- `CardService` 已新增按卡片维度串行执行的持久化队列；
- 每次写盘都基于当前卡片快照执行，避免旧写入覆盖新输入；
- 已新增 `isPersisting / persistedRevision / pendingPersistRevision / lastPersistedAt` 运行时状态。

### 4.3 复合卡片预览改为等待持久化完成后刷新

- `CardWindow` 已从依赖 `modifiedAt` 改为依赖 `persistedRevision`；
- 只有当最新配置真正落盘后，才推动 Host / SDK 重新渲染 iframe；
- 消除了“先刷新预览、后写盘”的空白卡片竞态。

## 6. 当前结论

当前问题已经通过正式方案闭环：基础卡片编辑器输入、编辑引擎持久化、Host/SDK 复合卡片预览三者已在同一正式时序下联通，用户现在可以获得稳定的实时保存与实时显示结果。
