# 工单036：编辑引擎复合卡片无限长模式缺少正式高度同步链路

> 当前状态：已闭环（2026-03-13）

## 1. 背景

当前用户任务包含以下明确要求：

- 复合卡片有“无限长模式”和“固定比例模式”两种显示方式；
- 无限长模式下，复合卡片窗口必须始终完整显示整张复合卡片的全部内容；
- 固定比例模式下，复合卡片窗口高宽比固定为 16:9；
- 模式切换通过复合卡片窗口顶部的收起 / 展开按钮完成；
- 必须严格走薯片 SDK、Host 正式显示链路，禁止临时开发或应用层绕过。

在进入实现前，对生态设计原稿、生态共用技术文档、EditingEngine 当前实现和旧项目行为进行了对照排查。结论是：当前第 1 条需求并不是单纯样式问题，而是正式接口链路缺口。

## 2. 现场证据

### 2.1 EditingEngine 已经接入正式复合卡片显示接口

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/CardWindow/CardWindow.tsx`

当前现状：

- 编辑引擎已通过 `client.card.compositeWindow.render({ cardFile, mode })` 获取正式复合 iframe；
- iframe 被挂到 `previewHostRef` 中，并被设置为 `width: 100%; height: 100%`；
- 外层壳层当前只消费 `onReady / onNodeSelect / onNodeError / onFatalError`。

这说明应用层已经切到正式显示链路，但没有拿到任何“内容总高度”信号。

### 2.2 CardWindowBase 的展开态是 `height: auto`，但拿不到真实内容高度

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/CardWindowBase/CardWindowBase.tsx`

当前现状：

- 收起态高度固定按 `width * 16 / 9` 计算；
- 展开态高度为 `auto`；
- 但是展开态内部承载的是 iframe，而 iframe 本身不会自动把内部文档高度同步给父窗口；
- 因为没有正式高度回传，展开态窗口并不能真正跟随复合内容增长。

### 2.3 Host 只把基础卡片节点高度同步到复合文档内部

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/card-service/src/card-service.ts`

当前现状：

- Host 复合文档内部会监听 `chips.basecard:height`；
- 收到单个基础卡片高度后，只更新复合文档内部对应节点 iframe 的高度；
- Host 没有继续把“整张复合卡片当前总高度”通过正式事件回传给应用壳层；
- 当前正式事件只有：
  - `chips.composite:ready`
  - `chips.composite:node-select`
  - `chips.composite:node-error`
  - `chips.composite:fatal-error`

因此 Host 正式链路目前止步于“内部节点尺寸编排”，没有对外暴露“复合窗口外层尺寸同步”。

### 2.4 SDK 没有复合卡片窗口尺寸订阅接口

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/src/api/card.ts`

当前现状：

- `client.card.compositeWindow` 当前只提供：
  - `render`
  - `onReady`
  - `onNodeSelect`
  - `onNodeError`
  - `onFatalError`
- 没有 `onResize`、`onLayoutChange` 或等价正式 API。

因此即使 Host 后续补发尺寸事件，当前编辑引擎也无法仅通过 SDK 正式接口消费该能力。

## 3. 根因判断

当前问题的根因不是 EditingEngine 样式写错，而是正式生态链路缺少“复合 iframe 总高度回传协议”：

1. Host 内部知道每个基础卡片节点高度，但没有把复合文档总高度继续对外发布。
2. SDK 没有封装复合窗口高度变化事件。
3. EditingEngine 作为应用壳层，严格按正式链路时无法知道展开态窗口应增长到多少。

因此“无限长模式完整显示全部内容”目前是生态正式接口缺口，不是单个应用仓库可独立闭环的小修复。

## 4. 为什么不能在当前任务里直接临时修

按当前仓库规则，以下做法都不允许进入主线：

- 在 EditingEngine 内部用私有 `window.message` 监听自行解析未经 SDK 正式暴露的事件；
- 在应用层直接探测 iframe 内部 DOM 高度；
- 给展开态窗口写死一个很大的高度；
- 按富文本卡片等个别类型估算高度；
- 在 EditingEngine 内部重建一套平行复合卡片渲染 / 测量链路。

这些都属于绕过正式 Host / SDK / 协议层的临时方案，与当前用户要求和生态规范相冲突。

## 5. 期望正式方案

建议按正式生态链路补齐：

### 5.1 Host

- 在复合卡片文档中新增正式尺寸事件，例如 `chips.composite:resize`；
- 在以下时机重新计算并发出总高度：
  - 单个基础卡片节点高度变化后；
  - 初始 ready 前后；
  - 主题切换导致复合文档重排后；
  - 复合文档自身 `ResizeObserver` 触发时。

建议 payload 至少包含：

- `height`
- `nodeCount`
- `reason`

### 5.2 SDK

- 新增 `client.card.compositeWindow.onResize(frame, handler)`；
- 保持与现有 `onReady/onNodeSelect/onNodeError/onFatalError` 同一安全过滤策略；
- 同步更新 SDK 测试、文档和契约说明。

### 5.3 EditingEngine

- 在复合卡片展开态订阅正式 resize 事件；
- 用正式高度值同步 `CardWindow` 展开态的窗口尺寸；
- 收起态继续严格保持 16:9 固定比例，不受内容高度影响。

## 6. 正式处理结果

### 6.1 Host

- `CardService` 生成的复合卡片文档已新增正式 `chips.composite:resize` 事件；
- 复合文档会在以下时机重新计算并回传整窗高度：
  - 初始装载；
  - 节点 iframe load / error 后；
  - 基础卡片通过 `chips.basecard:height` 回传新高度后；
  - 复合文档自身 `ResizeObserver` 检测到布局变化后。
- `resize` payload 当前固定包含：
  - `height`
  - `nodeCount`
  - `reason`

### 6.2 SDK

- `client.card.compositeWindow` 已新增 `onResize(frame, handler)`；
- 事件与 `onReady/onNodeSelect/onNodeError/onFatalError` 共享同一 `event.source + origin` 安全过滤逻辑；
- SDK 类型已新增 `CompositeResizePayload` 与正式 `reason` 联合类型。

### 6.3 EditingEngine

- `CardWindow` 已正式订阅 `client.card.compositeWindow.onResize(...)`；
- 复合卡片展开态使用正式高度驱动预览容器尺寸，从而让窗口跟随整张复合卡片内容增长；
- 收起态仍由 `CardWindowBase` 固定为 16:9，不受内容高度影响；
- 同时移除了因 `t` 依赖波动导致复合预览链路无谓重建的副作用，保持 resize 更新稳定。

### 6.4 组件库与契约

- `@chips/card-runtime` 已补齐 `resize` 消息解析；
- `CompositeCardWindow` 组件类型声明已新增 `onResize`；
- 组件库技术文档与 README 已同步纳入正式 `resize` 事件集。
