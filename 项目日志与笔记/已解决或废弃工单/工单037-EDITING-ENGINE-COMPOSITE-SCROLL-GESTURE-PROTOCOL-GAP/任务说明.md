# 工单037：编辑引擎复合卡片滚动与手势协议缺口

> 当前状态：已闭环（2026-03-14）

## 1. 背景

当前用户任务对编辑引擎无限画布的滚动缩放行为提出了明确要求：

- 需要同时适配鼠标、触摸板、触摸屏；
- 要能区分“桌面缩放”“窗口局部滚动”“无限长复合卡片驱动整桌面上下滚动”；
- 当用户正在无限长复合卡片上持续滚动并滑出该复合卡片时，必须暂时屏蔽后续滚动；
- 只有在用户中断当前滚动序列后，再回到桌面滚动，才允许恢复桌面缩放；
- 不允许使用临时方案、简化方案或绕开生态正式协议的做法。

按要求先对生态设计原稿、生态共用技术文档、EditingEngine 当前实现、SDK 显示接口与 Host 复合卡片文档生成链路进行了完整排查。结论是：当前需求并不是单纯的编辑引擎本地事件分发优化，而是正式生态链路缺少“复合卡片内部滚动与手势意图向外层应用壳层回传”的协议。

## 2. 现场证据

### 2.1 EditingEngine 当前只能监听外层 DOM，无法正式获知复合卡片内容区内部交互

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/layouts/InfiniteCanvas/InfiniteCanvas.tsx`

当前现状：

- 无限画布当前只在应用壳层根节点上注册原生 `wheel` 监听；
- 当前逻辑只能依赖外层 DOM `target` 的 class/层级做粗粒度判断；
- 它没有正式事件源去区分复合卡片 iframe 内部发生的滚轮、触摸板双指滚动、触摸屏滑动与捏合手势。

这意味着当前应用层能处理的只是“外层壳”交互，不是复合卡片内容区的真实手势来源。

### 2.2 SDK 生成的复合卡片 iframe 处于沙箱隔离态，应用壳层没有正式同源监听能力

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/src/api/card.ts`

当前现状：

- `client.card.compositeWindow.render()` 统一通过 `createFrameFromView()` 创建外层 iframe；
- 该 iframe 固定设置为 `sandbox="allow-scripts allow-forms"`；
- 当前没有 `allow-same-origin`；
- SDK 当前也没有暴露任何“把复合卡片内部原生 wheel/touch/pinch 转译为正式事件”的接口。

这说明应用壳层不能把复合卡片文档当作可直接挂接原生监听器的正式同源文档来使用。

### 2.3 Host 复合卡片正式协议未包含滚动与手势事件

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/card-service/src/card-service.ts`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/生态共用技术文档/协议与接口标准/02-Bridge-API规范.md`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/生态共用技术文档/组件库/07-复合卡片窗口与SDK显示链路标准.md`

当前现状：

- 复合卡片正式事件当前只有：
  - `chips.composite:ready`
  - `chips.composite:resize`
  - `chips.composite:node-select`
  - `chips.composite:node-error`
  - `chips.composite:fatal-error`
- 基础卡片当前只回传：
  - `chips.basecard:height`
  - `chips.basecard:select`

正式协议里没有任何“滚动意图”“触摸序列”“捏合缩放意图”回传能力。

### 2.4 复合卡片内部是多层 iframe 结构，应用层无法仅靠当前链路还原真实滚动来源

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/生态设计原稿/14-编辑引擎设计.md`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/生态共用技术文档/组件库/07-复合卡片窗口与SDK显示链路标准.md`

当前现状：

- 应用层拿到的只是一个复合 iframe；
- 复合 iframe 内部还包含多个基础卡片 iframe；
- 用户真正的滚动与触摸行为，往往发生在基础卡片 iframe 内部；
- 当前正式链路只把“高度变化”和“选中事件”回传到外层。

因此，编辑引擎壳层当前无法稳定知道：

- 当前滚动是不是来自无限长复合卡片内容区；
- 当前是普通滚动还是捏合缩放；
- 当前滚动序列是否还在同一个复合卡片上持续；
- 用户滑出复合卡片后，是否应继续屏蔽后续滚动，直到滚动中断再恢复其他机制。

## 3. 根因判断

当前问题的根因不是 `Chips-EditingEngine` 本地判断条件写得不够多，而是正式协议链路缺少关键交互事件：

1. Host 没有把复合卡片与基础卡片 iframe 内部的滚动 / 触摸 / 捏合行为转译成正式协议事件；
2. SDK 没有封装对应订阅接口，应用壳层无法正式消费；
3. EditingEngine 因此无法在不破坏 iframe 隔离的前提下，完成用户要求的交互状态机。

所以这属于生态正式链路缺口，不是单仓库内写几段额外 `wheel` 判断就能合规闭环的问题。

## 4. 为什么不能在当前任务里直接临时修

按当前仓库规则，以下做法都不能进入正式主线：

- 在应用层尝试直接探测沙箱 iframe 内部 DOM 或私接监听；
- 通过覆盖层截获复合卡片滚动，牺牲正式 iframe 交互能力；
- 只对鼠标滚轮做局部特判，忽略触摸板与触摸屏；
- 不补协议，直接在编辑引擎仓库内写一套私有滚动状态机并假设所有滚动事件都能被外层壳层拿到；
- 在局部场景下实现“桌面缩放”但无法稳定覆盖复合卡片内容区与滑出屏蔽逻辑。

这些都属于临时开发或绕过正式协议的方案，与当前任务要求冲突。

## 5. 期望正式方案

建议按正式生态链路补齐，至少包含以下方向：

### 5.1 Host

- 在复合卡片显示链路中新增正式交互意图回传能力；
- 由复合卡片文档统一接收并归一化基础卡片 iframe 内部的滚动 / 触摸 / 捏合行为；
- 对外向应用壳层回传正式事件，至少能表达：
  - 交互类型：普通滚动 / 捏合缩放 / 触摸滑动；
  - 来源作用域：复合卡片 / 基础卡片 / 降级节点或等价范围；
  - 归一化后的方向、位移、阶段信息；
  - 手势序列开始 / 持续 / 结束或等价状态。

### 5.2 SDK

- 新增正式订阅接口，供应用壳层统一消费复合卡片交互意图；
- 保持与当前 `onReady/onResize/onNodeSelect/onNodeError/onFatalError` 一致的来源校验与安全过滤；
- 更新类型、测试、文档与使用示例。

### 5.3 EditingEngine

- 基于正式协议实现无限画布交互状态机；
- 在桌面背景上把滚动解释为桌面缩放；
- 在工具窗口等本地页面中保持原生局部滚动；
- 在无限长复合卡片上把滚动解释为整桌面上下滚动；
- 在滚动序列从复合卡片滑出时，进入短暂屏蔽态，只有滚动中断后才恢复其他机制；
- 统一适配鼠标、触摸板、触摸屏。

## 6. 验收要求

- 不允许应用层直接绕过 Host/SDK 协议；
- 鼠标、触摸板、触摸屏三类交互至少要有正式单测 / 协议测试覆盖；
- 复合卡片内部滚动对桌面滚动 / 缩放的影响必须可预测、可复现、可验证；
- 文档、协议、SDK、Host、EditingEngine 需要一起闭环。

## 7. 处理结果

本工单已按正式生态链路闭环完成，处理结果如下：

1. Host `card.render` 正式新增 `interactionPolicy?: 'native' | 'delegate'`，并在 schema / 服务描述层完成参数校验。
2. Host 基础卡片 iframe 与复合卡片文档已补齐正式 `chips.basecard:interaction` / `chips.composite:interaction` 事件桥接，统一归一化滚轮、触摸滚动、捏合缩放意图。
3. SDK 已新增：
   - `CompositeInteractionPayload` / `CompositeInteractionPolicy` 类型；
   - `client.card.compositeWindow.onInteraction(...)`；
   - `interactionPolicy` 参数校验与透传。
4. EditingEngine 已正式实现：
   - 桌面背景滚轮缩放；
   - 工具窗口与非代理场景的原生局部滚动；
   - 展开态无限长复合卡片驱动整桌面平移与缩放；
   - 从复合卡片连续滚动滑出时的短暂屏蔽态，待滚动序列中断后再恢复桌面缩放；
   - 鼠标、触摸板、触摸屏三类交互的统一状态机。
5. 文档已同步更新：
   - Bridge API 规范；
   - SDK 使用指南；
   - 复合卡片窗口与 SDK 显示链路标准；
   - 编辑引擎设计原稿。
6. 验证已完成：
   - `Chips-EditingEngine` 定向单测通过；
   - `Chips-SDK` 定向单测通过；
   - `Chips-Host` 定向单测通过；
   - `Chips-Host` `tsc --noEmit -p tsconfig.json` 通过。

## 8. 当前结论

工单037已不再是生态阻塞项。  
EditingEngine 现已基于正式 Host / SDK / 协议链路实现复合卡片滚动与手势代理，不需要再以临时方案推进该需求。
