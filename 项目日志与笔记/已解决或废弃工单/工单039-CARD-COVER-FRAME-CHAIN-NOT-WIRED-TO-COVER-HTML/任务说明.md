# 工单039：正式封面渲染链路未接入 `.card/cover.html`

> 当前状态：已闭环

## 1. 背景

当前用户任务要求重新设计并开发复合卡片的封面编辑管理页面，且明确要求：

- 封面最终完全由 `cover.html` 决定；
- 不能做临时方案、简化方案、占位方案；
- 必须严格按照 SDK、Host、生态协议和共享文档实现；
- 一旦发现生态链路、SDK、Host、基础层未打通，必须先记录到《问题工单.md》，创建新任务文档，并停止当前推进。

在本次验收中，用户继续反馈：

- 封面编辑保存后，切换无限画布中的卡片到封面模式，没有显示真实封面；
- 实际仍然是灰色背景加卡片名称的占位视图；
- 用户进一步质疑“是否 SDK 中的封面渲染组件和接口的整个链路不对”。

经核对，当前问题并不只是编辑引擎一个 React 分支没接线，而是“应用层占位实现 + SDK/Host 正式封面链路未闭环”同时存在。

## 3. 现场证据

### 3.1 编辑引擎封面态仍是硬编码占位

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/src/components/CardWindow/CardWindow.tsx`

当前 `windowState === 'cover'` 时，编辑引擎直接渲染：

- `.card-cover__image`
- `.card-cover__placeholder`
- `.card-cover__title`

也就是说，当前封面态没有：

- 读取 `.card/cover.html`
- 调用 `client.card.coverFrame.render(...)`
- 挂载正式封面 iframe

这解释了为什么用户保存封面后，切换到封面模式仍然看到灰底占位。

### 3.2 SDK 已有 `coverFrame.render()`，但没有真正对接封面文件

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK/src/api/card.ts`

当前 `client.card.coverFrame.render(...)` 的实现是：

1. 调用 `client.invoke("card.render", { cardFile, options: { target: "card-iframe" } })`
2. 取回 `view.body`
3. 通过 `createFrameFromView(view.body, ...)` 包装成 iframe

也就是说，SDK 当前没有：

- 独立的封面渲染路由；
- `target: 'cover-frame'` 或等价目标；
- “明确消费 `.card/cover.html`” 的实现。

### 3.3 Host `CardService.render()` 根本没有读取 `cover.html`

文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/card-service/src/card-service.ts`

当前 `CardService.render()` 的正式行为是：

1. 解析 `.card/metadata.yaml`、`.card/structure.yaml` 和 `content/*.yaml`
2. 渲染结构中的基础卡片节点
3. 调用 `createCompositeDocument(...)` 生成复合卡片 HTML 文档

当前实现没有：

- 读取 `.card/cover.html`
- 装载 `cardcover/*` 资源形成封面显示文档
- 独立封面 target 或 cover render action

因此 SDK `coverFrame.render()` 即使被应用层调用，当前也拿不到基于 `cover.html` 的正式封面内容。

## 4. 根因判断

当前问题由两层原因叠加：

### 4.1 应用层直接原因

`Chips-EditingEngine` 的 `CardWindow` 封面态没有消费正式封面接口，而是保留了历史占位实现。

### 4.2 生态正式链路根因

即使应用层现在改成调用 `client.card.coverFrame.render(...)`，当前 SDK / Host 正式链路也没有证据表明其真正消费了 `.card/cover.html`，因为：

- SDK `coverFrame.render()` 只是复用 `card.render({ target: "card-iframe" })`
- Host `CardService.render()` 只会生成复合卡片文档
- Host 当前没有正式封面渲染入口

所以这不是一个可以在编辑引擎单仓库里放心闭环的普通 bug，而是正式封面显示链路未打通。

## 5. 影响范围

- 编辑引擎无限画布中的封面模式无法显示用户真实封面；
- 资源管理器卡片列表、卡片选择器、导入预览等任何未来依赖 `client.card.coverFrame.render(...)` 的应用都可能受影响；
- `CardCoverFrame` 组件标准当前缺少真实可用的 Host + SDK 落地链路支撑；
- 若继续仅在编辑引擎内直接读 `cover.html` 并私自挂 iframe，会绕过正式 SDK / Host 接口，违反当前任务要求。

## 6. 期望正式闭环方向

### 6.1 Host

- 为封面提供正式渲染入口；
- 明确读取 `.card/cover.html` 与其资源目录；
- 交付封面文档，而不是复合卡片文档。

### 6.2 SDK

- `client.card.coverFrame.render(...)` 改为调用正式封面渲染入口；
- 补齐相关参数校验、事件模型和契约测试；
- 保证返回值确实是封面 iframe。

### 6.3 应用层

- 编辑引擎封面态和其他封面消费场景统一切到 `coverFrame` 正式链路；
- 移除灰底占位分支作为正式实现；
- 补齐应用层回归测试，验证封面态显示真实 `cover.html`。

## 7. 最终处理结果

用户随后已明确要求“请帮我解决这个工单问题，然后继续开发”，因此本工单按正式 Host + SDK + EditingEngine 三段链路继续落地并完成闭环。

### 7.1 Host 正式封面渲染链路

- `Chips-Host/packages/card-service/src/card-service.ts`
  - 新增 `CardService.renderCover(cardFile)` 正式入口；
  - 直接读取 `.card/cover.html`，返回受控 `coverUrl`；
  - 对压缩 `.card` 文件使用可复用的持久化解包根目录，保证 iframe 通过相对路径稳定访问 `cardcover/*` 资源；
  - 不再把封面错误地复用为复合卡片 `createCompositeDocument(...)` 输出。
- `Chips-Host/src/main/services/register-host-services.ts`
  - 新增正式路由 `card.renderCover`。
- `Chips-Host/src/main/services/register-schemas.ts`
  - 新增 `card.renderCover` 的 request / response schema 注册。

### 7.2 SDK 正式封面接口

- `Chips-SDK/src/api/card.ts`
  - `client.card.coverFrame.render(...)` 已改为调用 `card.renderCover`；
  - 使用受控 URL 创建 iframe，而不是继续把 HTML 字符串通过 `srcdoc` 注入；
  - 按封面组件标准使用 `sandbox="allow-scripts allow-same-origin"`。
- `Chips-SDK/src/contracts/route-manifest.json`
  - 已登记 `card.renderCover` 路由。

### 7.3 EditingEngine 封面态接入

- `Chips-EditingEngine/src/components/CardWindow/CardWindow.tsx`
  - 已移除封面态灰底标题占位正式分支；
  - 改为挂载 `client.card.coverFrame.render(...)` 返回的真实封面 iframe；
  - 通过 `card:persisted` 事件在封面保存落盘后刷新显示；
  - 保留封面模式原有的拖拽与点击恢复交互。
- `Chips-EditingEngine/src/components/CardWindow/CardWindow.css`
  - 封面卡片容器已补齐阴影、厚度和加载/错误态样式。
