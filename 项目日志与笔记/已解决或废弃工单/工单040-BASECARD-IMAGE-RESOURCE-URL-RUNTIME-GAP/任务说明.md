## 3. 现场证据

### 3.1 共享文档要求资源 URL 由宿主正式生成

`生态共用技术文档/插件开发/03-卡片插件开发.md` 明确要求：

- 相对路径资源必须相对于卡片内容或宿主分发得到的资源根解析；
- 插件不直接假定自己拥有文件系统权限；
- 资源 URL 的生成由 Host 或宿主运行时完成，不由渲染组件私自拼系统绝对路径。

这说明图片、视频、音频等资源型插件不能靠插件自身私有逻辑绕开正式宿主链路。

### 3.2 Host 通用链路当前没有资源根注入

`Chips-Host/packages/card-service/src/card-service.ts` 当前链路中：

- `renderBasecardHtml(...)` 只向插件导出传入：
  - `container`
  - `config`
  - `themeCssText`
- `createChildFrameDocument(...)` 把基础卡片内容放进 `iframe srcdoc`；
- 当前基础卡片文档中没有对应卡片根目录的 `<base href>` 或等价资源根；
- `normalizeNodeConfig(...)` 目前仅对 `chips.basecard.richtext` 做专项归一，没有资源型插件的资源路径解析逻辑。

因此，如果图片配置中保存的是正式卡片文件中的相对路径，例如 `image.jpg` 或 `./image.jpg`，当前 Host 正式链路不会把它变成 iframe 内可直接加载的 URL。

### 3.3 EditingEngine 本地运行时也没有资源解析上下文

`Chips-EditingEngine/src/basecard-runtime/contracts.ts` 中：

```ts
normalizeConfig: (input: BasecardConfigRecord, baseCardId: string) => BasecardConfigRecord;
```

当前只传入：

- 原始配置；
- 基础卡片 ID。

没有：

- 卡片根路径；
- 资源服务实例；
- 资源 URL 生成器；
- 运行时临时资源字段治理能力。

`frame-host.tsx` 只是把 `normalizeBasecardConfig(...)` 的结果传给插件 `renderView(...)`，也没有资源 URL 注入步骤。

### 3.4 EditingEngine 当前持久化逻辑也没有运行时字段隔离

`Chips-EditingEngine/src/core/card-service.ts` 当前会把 `basicCard.data` 直接写回 `content/<id>.yaml`。

这意味着即便在编辑引擎侧临时给图片卡片配置加上运行时 URL 字段：

- 也会直接污染正式 YAML；
- 或必须先改写持久化与归一逻辑；
- 这已经超出单个图片插件内部实现范围，属于正式架构契约变更。

### 3.5 旧版图片基础卡片正式功能依赖本地文件资源

旧项目明确包含：

- 本地图片文件上传；
- 图片文件列表；
- 多种排版查看；
- 本地文件与 URL 两类来源；
- 编辑态与渲染态都需要访问图片资源。

因此无法通过“只支持 URL 图片”或“只支持远程资源”的方式继续推进。

## 4. 根因判断

当前正式生态链路对“文本型基础卡片”已经足够，但对“资源型基础卡片”缺少统一契约。

该缺口同时存在于两条正式链路中：

### 4.1 Host 通用渲染链路

- 没有把相对资源路径解析成基础卡片 iframe 可消费的正式 URL；
- 没有资源型插件的宿主归一注入步骤；
- 基础卡片 iframe 使用 `srcdoc`，天然不具备卡片文件根路径解析上下文。

### 4.2 EditingEngine 本地基础卡片运行时

- 描述符归一接口没有资源解析上下文；
- 预览 iframe 没有资源根；
- 本地持久化没有运行时字段剥离机制。

因此当前正式架构还不能支撑图片基础卡片这种资源型插件的完整迁移。

## 5. 为什么当前不能继续开发

如果继续推进，只会落入以下非正式方案之一：

- 在插件内部直接拼系统绝对路径；
- 在插件内部依赖文件系统权限或私有 Bridge；
- 只支持远程 URL，放弃本地图片；
- 把本地文件转为临时 Data URL 保存到配置；
- 在 EditingEngine 内引入只为图片卡片服务的私有运行时字段。

这些方案都违反当前任务要求：

- 禁止临时开发；
- 禁止简化版；
- 禁止自行决定改变架构设计；
- 必须严格按生态接口与协议开发。

## 7. 处理结果

### 7.1 EditingEngine 本地运行时闭环

- `BasecardDescriptor` 已正式扩展 `collectResourcePaths(...)`；
- `renderEditor` 上下文已正式支持：
  - `resolveResourceUrl(...)`
  - `releaseResourceUrl(...)`
  - `importResource(...)`
  - `deleteResource(...)`
- `EditorSessionStore` 已新增待导入资源与待删除资源状态；
- `CardService` 已在自动持久化中把待导入资源写入卡片根目录，并基于真实根目录文件重建 `structure.yaml manifest.resource_count/resources`；
- 待删除资源延迟到显式保存时执行，避免破坏编辑器撤销与会话稳定性；
- 编辑引擎单卡预览 iframe 已注入卡片根目录 `<base href>`，使配置中的相对路径资源可直接解析。

### 7.2 Host 通用链路闭环

- Host 基础卡片查看文档已注入卡片根目录 `<base href>`；
- Host 托管编辑器文档已新增正式资源桥协议：
  - `chips.card-editor:resource-request`
  - `chips.card-editor:resource-response`
  - `chips.card-editor:resource-release`
- Host 编辑器文档会把 `resolve/import/delete/release` 能力传递给插件 `renderBasecardEditor(...)`；
- Host 编辑器/查看器文档 CSP 已放行 `blob:` 图片资源，避免临时资源 URL 被 iframe 自身拦截。

### 7.3 SDK 通用链路闭环

- `client.card.editorPanel.render(...)` 已新增 `resources` 本地桥配置；
- `resources` 正式支持：
  - `rootPath`
  - `resolveResourceUrl(...)`
  - `releaseResourceUrl(...)`
  - `importResource(...)`
  - `deleteResource(...)`
- SDK 会在本地桥接编辑器 iframe 的资源请求，但不会把 `resources` 透传进 Host `card.renderEditor` 正式路由；
- `rootPath` 可作为只读资源解析回退，用于把卡片根目录相对路径生成 `file://` URL。

### 7.4 正式约束冻结

- 卡片内部资源路径统一从卡片文件根目录开始记录；
- 资源文件正式写入卡片根目录，而不是 `content/`；
- `blob:`、`data:`、系统绝对路径等运行时 URL 不得写回正式 YAML；
- 资源型基础卡片必须通过宿主资源能力访问内部资源，不得私接文件系统。
