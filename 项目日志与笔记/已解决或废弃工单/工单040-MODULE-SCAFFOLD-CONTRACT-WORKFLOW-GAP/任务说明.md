# 工单040：模块插件正式契约与脚手架创建链路未闭环

> 当前状态：已解决（2026-03-14）
>
> 本工单前 1–8 节保留发现当时的阻断背景与判断依据；第 9 节记录本轮正式修复结果、验证范围与当前结论。

## 1. 背景

用户当前明确要求：

- 先完整阅读生态设计原稿、生态共用技术文档和仓内现有实现；
- 模块脚手架必须严格按生态正式规范开发，禁止临时方案、简化方案和旁路实现；
- 如发现架构缺陷、SDK 接口不可用、文档与代码不一致或生态未打通，必须先登记工单、创建任务文件夹，并停止当前推进。

基于该要求，已对以下内容完成逐项核对：

- `生态设计原稿/`
- `生态共用技术文档/`
- `Chips-Host`
- `Chips-SDK`
- `Chips-Scaffold` 现有 `app/basecard/theme` 脚手架
- `chipsdev create` 当前实现

核对结论是：`Chips-Scaffold/chips-scaffold-module` 并不是“只差把空目录补全”的普通实现任务，而是存在上游正式契约和工具链同时未闭环的问题。若不先冻结这些基础口径，脚手架会把错误的模块协议直接固化进模板和示例工程。

## 2. 阻断结论

当前不满足继续正式开发模块插件脚手架的前置条件。

阻断不只是一处，而是同一条链路上的三类问题同时存在：

1. 模块脚手架仓库本体尚未建立；
2. `chipsdev create module` 正式入口未落地；
3. `module.*` 正式契约在 SDK、Host、Bridge 文档之间相互冲突。

在这些问题处理前继续推进，会必然落入以下违规路径之一：

- 在脚手架模板中自行决定 `module.*` 的正式调用方式；
- 让模板示例依赖当前错误或未冻结的 SDK 接口；
- 为了让模板“先跑起来”而增加临时兼容层；
- 先写脚手架，等 Host/SDK/文档未来再反向适配。

以上都不符合当前项目规则，因此本工单视为正式阻断项。

## 3. 证据

### 3.1 模块脚手架仓库当前为空

目录：

- `Chips-Scaffold/chips-scaffold-module/`

现状：

- 目录存在，但没有 `package.json`、`src/`、`templates/`、`tests/`、`需求文档/`、`技术文档/`、`开发计划/`。

说明：

- 当前模块脚手架不是“已有仓库待完善”，而是尚未进入正式建设状态。
- 这本身不是阻断的全部原因，但意味着一旦开工，模板结构、CLI API、测试体系和文档都需要一次性按正式标准建立。

### 3.2 `chipsdev create` 尚未支持 `module`

文件：

- `Chips-SDK/cli/index.js`

现状：

- `handleCreate()` 当前只处理：
  - `theme`
  - `app`
  - `card`
- 对其他类型直接报错：
  - “当前支持：app、card、theme。”

说明：

- 生态设计原稿和共用文档已经把模块插件列为五类正式插件之一，但开发者正式创建入口尚未跟进。
- 若先在 `chips-scaffold-module` 中定义一套 API，却不同时冻结 CLI 接入方式，后续仍会出现模板存在但不能按正式命令创建的断链。

### 3.3 SDK 与 Host 的 `module.*` 契约不一致

SDK 当前口径：

- 文件：
  - `Chips-SDK/src/api/module.ts`
  - `Chips-SDK/需求文档/02-薯片 SDK 功能需求规格.md`
  - `Chips-SDK/技术文档/06-主题与插件能力封装设计.md`
- 现状：
  - `client.module.mount(slot: string, moduleId: string): Promise<ModuleState>`
  - `ModuleState` 内含 `moduleId`
  - SDK 实际调用 `client.invoke("module.mount", { slot, moduleId })`

Host 当前口径：

- 文件：
  - `Chips-Host/src/main/services/register-host-services.ts`
  - `Chips-Host/src/main/services/register-schemas.ts`
  - `生态共用技术文档/架构设计/14-Host服务域设计.md`
  - `生态设计原稿/13-公共基础层设计.md`
- 现状：
  - `module.mount` 的 schema 要求 `slot + module`
  - Host 服务实际读取 `input.module`
  - 返回值是 `{ ack: true }`
  - `module.query` 返回 `{ module: ModuleRecord | null }`
  - `module.list` 返回 `{ modules: ModuleRecord[] }`

说明：

- SDK 现在的 `mount(slot, moduleId)` 调用无法和 Host 当前 schema/服务实现直接对齐。
- 即使忽略类型，SDK 期望的返回值和 Host 正式返回值也不一致。
- 这已经属于“SDK 工具和接口不能正常使用”的正式缺口。

### 3.4 Bridge 文档仍保留第三套模块语义

文件：

- `生态共用技术文档/协议与接口标准/02-Bridge-API规范.md`

现状：

- 文档仍写为：
  - `module.load(moduleId)`
  - `module.unload(moduleId)`
  - `module.list()`

说明：

- 这与 SDK 的 `mount/unmount`、Host 的 `mount(slot + module)` 都不一致。
- 目前生态中至少存在三套并行口径：
  1. `load/unload(moduleId)`
  2. `mount/unmount(slot, moduleId)`
  3. `mount/unmount(slot + module)`

### 3.5 模块脚手架无法合法决定模板示例应接哪套协议

模块脚手架正式交付必然包含以下内容：

- 模板工程 `README.md`
- 示例代码
- 测试夹具
- 与 `chipsdev create module` 的调用参数约定
- 对 SDK/Host 正式接口的演示

但在当前状态下，无法回答以下关键问题：

1. 模块模板是否应该示例 `client.module.mount(...)`？
2. 如果示例 `mount`，传的是 `moduleId` 还是完整 `module` 载荷？
3. 返回值是 `ModuleState` 还是 `{ ack: true }`？
4. 模块脚手架是否还应为 `module.load/unload` 兼容历史文档？

任何一个答案只要不是先由生态正式决议给出，都会变成脚手架侧自行拍板。

## 4. 对当前任务的直接影响

当前直接受阻的工作包括：

1. `chips-scaffold-module` 仓库实现；
2. `chipsdev create module` 集成；
3. 模块模板中的正式 API 示例与 README；
4. 模块脚手架的单元测试与生成后集成测试；
5. 面向该仓库的需求文档、技术文档、开发计划的最终冻结。

虽然理论上可以先写“纯模板渲染引擎”和“纯文档”，但这仍然会立刻遇到模块协议选择问题，而且用户明确要求的是完整正式项目，不允许占位式推进，因此不能通过只做局部壳层来绕过阻断。

## 5. 建议解决方向

### 5.1 先冻结模块管理正式契约

至少要统一以下事项：

1. 对外动作名到底采用：
   - `module.load/unload/list`
   - 还是 `module.mount/unmount/query/list`
2. 若采用 `mount`：
   - 入参是否为 `slot + moduleId`
   - 还是 `slot + module`
3. `mount/query/list` 的正式返回值结构是什么
4. SDK 是否只做轻量封装，还是需要补齐对象解包

### 5.2 同步收口文档、Host、SDK

需要统一更新至少以下位置：

- `生态共用技术文档/协议与接口标准/02-Bridge-API规范.md`
- `生态共用技术文档/架构设计/14-Host服务域设计.md`
- `生态共用技术文档/插件开发/05-模块插件开发.md`
- `Chips-SDK/src/api/module.ts`
- `Chips-SDK/需求文档/02-薯片 SDK 功能需求规格.md`
- `Chips-SDK/技术文档/06-主题与插件能力封装设计.md`
- `Chips-Host/src/main/services/register-host-services.ts`

### 5.3 契约冻结后再建设模块脚手架

冻结后，模块脚手架正式任务应至少覆盖：

1. `chips-scaffold-module` 包结构与模板渲染引擎；
2. 标准模块模板；
3. 需求文档、技术文档、开发计划三套文档；
4. 模板自检与生成后集成测试；
5. `chipsdev create module` 接入；
6. 相关 README 与生态共用文档更新。

## 9. 处理结果（2026-03-14）

### 9.1 契约已统一

本轮已将模块正式口径统一为：

- `module.mount({ slot, moduleId }) -> { module }`
- `module.unmount({ slot }) -> { ack: true }`
- `module.query({ slot }) -> { module: ModuleState | null }`
- `module.list() -> { modules: ModuleState[] }`

其中：

- Host 对外仍保持包装对象返回；
- SDK 对调用方解包为 `ModuleState` / `ModuleState[]`；
- `ModuleState` 当前至少包含 `slot/moduleId/entry?/capabilities/active/mountedAt`。
