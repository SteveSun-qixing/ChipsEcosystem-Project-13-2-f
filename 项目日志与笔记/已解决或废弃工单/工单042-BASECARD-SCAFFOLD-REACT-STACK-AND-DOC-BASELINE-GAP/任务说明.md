## 3. 现场证据

### 3.1 React 架构口径已明确冻结

设计原稿中已经明确写明：

- `生态设计原稿/02-极致模块化架构.md`：官方产品统一使用 React 框架开发；
- `生态设计原稿/08-开发规范和标准.md`：官方产品统一使用 React 框架开发，前端组件不得硬编码样式。

这说明当前官方基础卡片插件也应遵循 React 技术栈口径。

### 3.2 脚手架模板对 React 的表述与生成结果不一致

`Chips-Scaffold/chips-scaffold-basecard/技术文档/03-模板参数化与渲染引擎设计.md` 中明确要求模板源码与 React 组合模式、性能规范对齐。

但当前模板实际生成结果存在明显漂移：

- `templates/card-standard/package.json.tpl` 没有 `react` / `react-dom`；
- `templates/card-text-basic/package.json.tpl` 没有 `react` / `react-dom`；
- 生成出的 `image-BCP/package.json` 同样没有 React 依赖；
- 但 `tsconfig.json.tpl` 已固定 `jsx: "react-jsx"`。

这意味着模板在配置层已经指向 React JSX 语义，但在依赖层没有提供 React 运行时基线。

### 3.3 项目文档骨架也未随正式要求生成

当前生成的 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-BaseCardPlugin/image-BCP/` 没有：

- `docs/requirements/`
- `docs/technical/`
- `docs/development/`

而本任务明确要求项目必须具备正式需求文档、技术文档、开发计划，且开发过程中要持续同步更新。

这说明脚手架目前并没有交付符合当前项目治理要求的项目级文档骨架。

## 4. 根因判断

当前基础卡片脚手架仍停留在“基础 TypeScript 骨架”状态，没有完整跟上：

1. 官方 React 技术栈要求；
2. 项目文档驱动开发要求；
3. 生成项目即具备正式交付基线的要求。

这不是 `image-BCP` 单仓可以合理私下消化的问题，而是脚手架正式模板与生态架构口径不一致。

## 5. 为什么当前不能继续推进 image-BCP 迁移

如果继续推进，只会落入以下非正式做法之一：

- 在 `image-BCP` 项目里手动加 React 依赖，绕过脚手架缺口；
- 在 `image-BCP` 项目里手动补 docs 目录，让业务仓承担脚手架职责；
- 继续沿用当前无 React 依赖的骨架，违反本任务“官方前端统一使用 React 技术栈”的要求。

这些做法都会破坏“先用正式脚手架初始化项目仓库”的前提，也会让后续所有基础卡片插件继续复制同类问题。
