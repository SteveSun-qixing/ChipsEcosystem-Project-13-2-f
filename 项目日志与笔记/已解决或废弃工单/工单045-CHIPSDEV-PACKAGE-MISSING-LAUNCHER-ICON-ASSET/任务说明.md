# 工单045：chipsdev 打包链路未收集应用插件 launcher icon 资源

> 当前状态：已闭环（2026-03-16）

## 1. 背景

当前联调任务要求将 `Chips-EditingEngine` 重新打包并安装到当前开发 Host 工作区 `.chips-host-dev`，并且必须严格使用正式 `chipsdev` 工具链，不允许手工改包、绕过安装器或做临时开发。

在执行正式链路时，发现应用插件 `.cpk` 虽然可以生成，但安装阶段会因为包内缺少 launcher icon 资源而失败，说明当前 `chipsdev package` 对应用插件 manifest 资源的打包范围不完整。

## 2. 现场信息

### 2.1 目标项目

- 项目：`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine`
- manifest：`/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/manifest.yaml`
- 关键字段：
  - `type: app`
  - `entry: dist/index.html`
  - `ui.launcher.icon: assets/icons/app-icon.ico`

### 2.2 图标资源实际存在

- 项目内已有图标：
  - `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/assets/icons/app-icon.ico`
- 用户补充提供的设计资源目录：
  - `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/design-assets/Appicon`

这说明问题不在于“没有图标源文件”，而在于“正式打包链路没有把该资源打入 `.cpk`”。

## 3. 复现步骤

1. 进入：

```bash
/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine
```

2. 执行：

```bash
../node_modules/.bin/chipsdev build
../node_modules/.bin/chipsdev validate
../node_modules/.bin/chipsdev package
```

3. 观察 `package` 返回成功，并输出：

```text
/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-EditingEngine/dist/chips-official.editing-engine-0.1.0.cpk
```

4. 解包查看：

```bash
unzip -l dist/chips-official.editing-engine-0.1.0.cpk
```

5. 确认包内只有：

- `manifest.yaml`
- `dist/**`

没有：

- `assets/icons/app-icon.ico`

6. 执行安装：

```bash
../node_modules/.bin/chipsdev plugin install dist/chips-official.editing-engine-0.1.0.cpk
```

7. 观察安装失败，返回：

- `Plugin entry not found: assets/icons/app-icon.ico`
- `code: PLUGIN_ENTRY_NOT_FOUND`

## 4. 根因判断

### 4.1 当前打包资源收集范围不完整

核对 `Chips-SDK/cli/index.js` 后确认：

- `collectManifestAssetPaths(...)` 当前只会收集：
  - `manifest.entry`
  - `manifest.ui.layout.contract`
  - `manifest.ui.layout.minFunctionalSet`
  - `manifest.preview`
- 没有收集：
  - `manifest.ui.launcher.icon`

因此：

- `chipsdev validate` 会误判工程完整；
- `chipsdev package` 会生成“表面成功但运行时资源不完整”的 `.cpk`；
- `plugin install` 最终在 Host 侧因缺资源失败。

### 4.2 这是正式工具链缺口，不是单个项目资源问题

同类 manifest 结构也出现在：

- `Chips-CardViewer/manifest.yaml`
- `Chips-EcoSettingsPanel/manifest.yaml`

因此该问题具备生态级影响，不应通过项目内手工复制包内容等方式绕过。

## 5. 影响范围

- 应用插件正式安装链路不可靠；
- `chipsdev validate` 与 `chipsdev package` 的资源口径不一致；
- 应用脚手架模板、生态文档与实际工具行为发生漂移；
- 所有依赖 `ui.launcher.icon` 的 app 插件都可能受到影响。

## 6. 期望正式修复

### 6.1 chipsdev

- 扩展 `collectManifestAssetPaths(...)`，正式纳入所有 manifest 对外资源字段，至少包括：
  - `ui.launcher.icon`
- 同步更新 `validate` 阶段的资源存在性检查。

### 6.2 文档与脚手架

- 更新应用插件脚手架和共用技术文档，明确：
  - 应用图标的正式存放位置；
  - manifest 中图标路径的正式写法；
  - `chipsdev package` 的实际收集范围。

### 6.3 回归验证

- 至少补齐以下验证：
  - 应用插件 `launcher icon` 被正确打入 `.cpk`
  - `chipsdev plugin install <cpk>` 成功
  - `.cpk` 解包后可见图标资源
  - `validate` 与 `package` 对资源字段的口径一致
