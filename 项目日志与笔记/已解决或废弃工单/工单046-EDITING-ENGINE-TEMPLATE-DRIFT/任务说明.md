## 3. 现场证据

### 3.1 工程真实身份已经是正式编辑引擎

`Chips-EditingEngine/manifest.yaml` 当前明确声明：

- `id: chips-official.editing-engine`
- `name: 编辑引擎`
- `type: app`
- `entry: dist/index.html`

同时，项目代码目录已经包含大量真实业务结构，例如：

- `src/basecard-runtime/`
- `src/editor-runtime/`
- `src/components/CardWindow/`
- `src/layouts/InfiniteCanvas/`
- `src/layouts/Workbench/`
- `src/services/`

这说明它不是一个空白模板仓，而是实际的编辑引擎应用工程。

### 3.2 README 仍然是脚手架模板占位内容

`Chips-EditingEngine/README.md` 文件开头仍是：

```md
# {{ DISPLAY_NAME }} 应用插件工程（由 chips-scaffold-app 生成）
> 插件 ID：`{{ PLUGIN_ID }}`
> 模板：`app-standard`
> 生成自：`chips-scaffold-app`
```

其中还反复出现：

- `{{ PROJECT_NAME }}`
- `{{ PLUGIN_ID }}`
- 脚手架模板级说明

这与 `manifest.yaml` 中已经正式命名为“编辑引擎”的真实工程身份明显不一致。

### 3.3 运行时代码里仍残留模板身份

`Chips-EditingEngine/config/app-config.ts` 当前写有：

```ts

  featureFlags: {
    enableDebugPanel: false,
  },
};
```

这说明不只是 README 没有更新，连工程配置层都还留着模板占位。

### 3.4 i18n 与示例组件仍以模板工程身份存在

`Chips-EditingEngine/i18n/zh-CN.json` 当前顶层 key 为：


这类信息如果未经甄别直接进入项目文档、AGENTS 或生态共用文档，会把编辑引擎错误地描述成模板工程或示例应用。

`Chips-EditingEngine/template.json` 当前内容为：

## 4. 根因判断

当前问题不是单一文档遗漏，而是“脚手架来源痕迹没有随正式业务工程完成收口”：

1. 项目真实身份已变成编辑引擎正式应用；
2. 但 README、配置、i18n、示例组件、模板元数据仍有明显模板占位残留；
3. 导致“项目文档口径”“工程内部标识”“真实运行时代码”三者没有完全统一。

这会让后续所有依赖该项目文档的工作都存在误导风险，包括：

- 编写 `AGENTS.md`
- 更新项目内部技术文档
- 更新生态共用技术文档
- 校对插件身份、命名空间、i18n key 规划和样例说明

## 5. 为什么当前不能直接把 README 当成正式依据

本轮任务要求 `AGENTS.md` 必须认真参考设计原稿和技术手册，但不能把未来易变细节或错误口径写进去。

如果把当前 `Chips-EditingEngine/README.md` 直接当成正式依据，就会出现至少三种风险：

1. 把模板占位信息误写成正式项目规则；
2. 把 README 中的错误项目身份传播到共用文档或治理规则里；
3. 忽略当前工程内还没有完成收口的文档与元数据问题。

因此，本工单登记后，后续与 `Chips-EditingEngine` 相关的规则提炼只能优先参考真实代码、`manifest.yaml` 和正式技术文档，不能依赖当前 README。
