## 4. 发现的阻断性缺口

虽然“可创建工程”的闭环已经成立，但继续审查“未来 Host 是否能依靠当前模块架构扩展到更多功能域”时，发现当前模块链路还缺少关键治理层。

### 4.1 `module.mount` 只记录挂载状态，不创建模块运行时身份

证据文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/main/services/register-host-services.ts`

当前 `module.mount` 的行为是：

1. 校验 `slot` 非空；
2. 校验 `moduleId` 非空；
3. 校验目标 `slot` 没被占用；
4. 从插件运行时中取到对应插件记录；
5. 校验其 `type === module` 且已启用；
6. 把 `slot/moduleId/entry/capabilities/active/mountedAt` 写入 `state.modules`；
7. 返回 `{ module }`。

但缺失了：

- 模块专属 `PluginSession`；
- 模块专属 `sessionNonce`；
- 模块专属权限快照；
- 模块专属配额；
- 模块专属 Bridge 身份上下文；
- 模块级运行态审计切面。

这意味着模块当前只是“被登记到某个 slot 的插件记录”，还不是“真正进入 Host 运行时治理面的独立插件实例”。

### 4.2 Host 已有插件会话能力，但模块挂载链路没有接入

证据文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/runtime/plugin-runtime.ts`

当前 Host 已具备：

- `pluginInit(pluginId, launchParams)`；
- `completeHandshake(sessionId, nonce)`；
- `PluginSession.permissions/themeId/locale/sessionNonce`；
- `getQuota()/setQuota()`。

也就是说，Host 并不是没有会话治理能力，而是模块挂载链路没有使用它。

### 4.3 Kernel 权限校验只看调用方上下文

证据文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/packages/kernel/src/router.ts`

当前权限校验直接依赖：

- `context.caller.permissions`

因此谁发起 Bridge 调用，就以谁的权限集合过 `PermissionGuard`。

如果模块运行在调用方页面里，却没有独立的 Host Bridge 身份，那么模块自己的 manifest 权限声明就不会真正进入 Host 校验链。

### 4.4 模块模板默认直接复用当前页面 Bridge 客户端

证据文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Scaffold/chips-scaffold-module/templates/module-standard/src/module/runtime.tsx.tpl`

模板中的 `createRuntimeClient()` 当前逻辑是：

1. 若调用方传入 `client`，就直接复用它；
2. 否则只要当前页面存在 `window.chips`，就直接 `createClient()`。

这意味着模块运行时中的：

- `client.theme.getAllCss()`
- `client.theme.getCurrent({ pluginId: moduleId })`
- `client.i18n.getCurrent()`

虽然代码里传了 `moduleId` 作为业务参数，但真正的 Bridge 调用身份仍然来自当前宿主页面的 `window.chips` 上下文，而不是一个由 Host 分配的“模块插件调用上下文”。

### 4.5 独立插件窗口有 Bridge 身份注入，但模块没有等价链路

证据文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/preload/plugin-window.ts`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-Host/src/preload/create-bridge.ts`

当前 Host 能为独立插件窗口注入：

- `pluginId`
- `permissions`
- `launchContext`

并基于这些信息构造调用上下文。

但模块插件当前没有等价的 Host 托管预加载或注入链路，所以不会自动得到一份带自身身份的 `window.chips`。

### 4.6 文档把模块描述为独立插件治理对象，但实现还停留在“调用方自管”

证据文件：

- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/生态共用技术文档/协议与契约/01-插件契约规范.md`
- `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/生态共用技术文档/插件开发/05-模块插件开发.md`

当前文档同时声明：

- 五类插件都属于正式插件体系；
- 模块插件是“沙箱隔离”；
- 插件初始化必须走 `plugin-init`；
- 模块若读取主题/语言状态，应显式声明 `theme.read`、`i18n.read`。

但实际实现中：

- 模块没有独立运行时会话；
- 模块没有独立握手；
- 模块没有独立 Bridge 身份；
- `slot` 与 `capabilities` 也没有 Host 级契约匹配。

这说明“文档要求的治理强度”高于“当前实现实际提供的治理强度”。

## 5. 对未来 Host 扩展的具体影响

### 5.1 当前不能把模块权限当作真实安全边界

模板生成的 manifest 会写：

- `theme.read`
- `i18n.read`

但当前模块运行时大概率仍会以调用方应用/插件的 Bridge 身份去访问 Host。

这会带来两个问题：

1. 模块自身权限声明可能只是“说明性元数据”，而不是硬边界；
2. 未来若调用方权限更大，模块可能在不经过自身权限治理的情况下间接获得更高能力。

### 5.2 Host 还无法对模块做独立配额、独立审计和独立追责

既然没有模块专属会话，也就很难正式实现：

- 模块级消息频率限流；
- 模块级错误审计；
- 模块级资源消耗追踪；
- 模块级权限拒绝日志。

这对未来 Host 面向更多领域复用模块时，会直接影响运行时治理质量。

### 5.3 `slot + capability` 模型还没形成 Host 级治理规则

当前模块链路里：

- `capabilities` 会被写入 manifest；
- `module.mount` 会把它拷贝进返回的 `ModuleState`；
- 但 Host 不会校验“这个模块能力是否适配这个 slot”。

现在文档甚至明确写了：

- `slot` 命名稳定性由调用方自行保证。

这意味着当前 `slot` 更像约定字符串，而不是冻结后的正式契约面。对于未来要扩到更多功能域、更多入口位的 Host，这个治理强度还不够。

## 6. 为什么当前不能给出“架构已完全满足未来扩展”的通过结论

当前可以给出的结论只能分两层：

### 6.1 已通过的层

- 模块插件脚手架仓库已经存在；
- 模板、CLI、基本测试链路已经闭环；
- 作为“创建模块项目的开发工具”已经达到可用状态。

### 6.2 尚未通过的层

- 模块作为独立插件身份进入 Host 运行时治理的闭环没有完成；
- 模块权限、会话、配额、审计和 Bridge 身份还未打通；
- slot 与 capability 的 Host 级治理规则未冻结。

因此，当前还不能正式认定“模块插件架构已经足以支撑未来 Host 对各种功能和领域的长期扩展”。

## 7. 期望正式收口方向

### 7.1 路线 A：模块保持独立插件身份

若继续坚持“模块插件是正式插件、具备独立权限与隔离”的定位，则需要补齐：

1. `module.mount` 创建模块专属 `PluginSession`；
2. 模块级 `plugin-init` / `sessionNonce`；
3. 模块级 `pluginId/permissions/quota` 的 Bridge 上下文；
4. 模块级审计与限流；
5. slot 与 capability 的匹配契约。

### 7.2 路线 B：模块降级为调用方内嵌库

若模块本质只是“由调用方宿主加载的共享前端库”，则需要同步下调：

1. 插件契约文档中对模块“沙箱隔离”“独立握手”的表述；
2. 模块 manifest 中对独立权限语义的要求；
3. 模块脚手架模板对“模块自身向 Host 读主题/语言”的暗示；
4. Host 对模块链路的未来扩展承诺。

无论选哪条，都必须先统一文档、Host、SDK 与脚手架模板口径，再继续把模块架构向更多领域推广。

## 8. 处理结果

本工单已在当前任务分支完成正式收口，处理结果如下：

1. Host 模块挂载链路已接入独立运行时治理：
   - `module.mount` 现在会校验 namespaced slot 与 `requiredCapabilities`；
   - 挂载成功后会为模块插件创建正式 `PluginSession`、完成握手，并发放一次性的 `bridgeScopeToken`；
   - `module.unmount` 以及模块插件 `disable/uninstall` 会同步回收模块会话和作用域令牌。
2. Bridge 与 IPC 已新增 scoped 调用通路：
   - `window.chips.invokeScoped(action, payload, { token })`
   - `window.chips.emitScoped(event, data, { token })`
   Host 会在 IPC 入站时先把 token 还原为模块插件自身的 `pluginId/permissions/session`，再进入 kernel 权限校验链。
3. SDK 与模块脚手架已完成对齐：
   - SDK `client.module.mount(slot, moduleId, { requiredCapabilities? })` 已支持新的正式口径；
   - `ClientConfig` 支持 `bridgeScope: { token }`；
   - 模块标准模板在收到 `bridgeScopeToken` 后，会优先创建 scoped client，而不再默认复用宿主页面的 Bridge 身份。
