# 工单057-chipsdev开发工作区缺少模块插件的正式Electron联调入口

## 背景

本轮在 `Chips-HtmltoImage-Plugin` 的真实联调中，用户补充了实际开发习惯：

- 平时测试使用 `chipsdev` 启动

因此需要核对当前 `chipsdev` 是否已经为模块插件提供了正式的 Electron 联调路径。

## 原始问题结论

在处理前，`chipsdev` 的开发工作区命令体系没有为模块插件提供正式的 Electron Host 联调入口。

具体表现为：

1. `chipsdev start/stop/status/config/logs/plugin/theme/open` 都委托给 Host CLI；
2. Host CLI 通过 `HostApplication` 启动的是 Node Host；
3. Node Host 不具备 Electron `BrowserWindow`；
4. `platform.renderHtmlToImage` / `platform.renderHtmlToPdf` 正式实现都依赖 Electron；
5. `chipsdev run` 虽然能拉起 Electron Host，但当前只支持 app 插件，不能用于模块插件。

因此，当前模块插件即使在 `.chips-host-dev` 中完成安装和启用，也无法通过标准 `chipsdev` 工作流验证真实 Electron 宿主能力。

## 证据

### 1. `chipsdev` Host 管理命令全部走 Host CLI

文件：

- `Chips-SDK/cli/index.js`

关键实现：

- `HOST_MANAGED_COMMANDS` 包含：
  - `start`
  - `stop`
  - `status`
  - `config`
  - `logs`
  - `plugin`
  - `theme`
  - `update`
  - `doctor`
  - `open`
- `delegateHostManagedCommand(...)` 最终调用 `runHostCliCommand(...)`

### 2. Host CLI 启动的是 Node Host

文件：

- `Chips-Host/src/main/cli/index.ts`

关键实现：

- `withHost(...)` 内部直接：

```ts
const app = new HostApplication({ workspacePath: workspace });
await app.start();
```

### 3. `HostApplication` 使用 Node PAL

文件：

- `Chips-Host/src/main/core/host-application.ts`

关键实现：

```ts
this.pal = new NodePalAdapter(...)
```

这意味着 CLI 链路不会进入 Electron `app.whenReady()`，也不会拥有真实 `BrowserWindow`。

### 4. 图片/PDF 导出正式能力要求 Electron

文件：

- `Chips-Host/src/main/services/register-host-services.ts`

关键错误：

- `HTML to image export requires Electron BrowserWindow support`
- `HTML to PDF export requires Electron BrowserWindow support`

### 5. `chipsdev run` 当前只支持应用插件

文件：

- `Chips-SDK/cli/index.js`

关键判断：

```js
if (manifest.type !== 'app') {
  throw new Error('chipsdev run 当前仅支持应用插件 ...')
}
```

## 影响范围

- 所有依赖 Electron 宿主的模块插件当前都无法通过标准 `chipsdev` 工作流联调；
- 开发者会误以为“开发工作区插件已安装启用”就等价于“真实 Electron 能力已可用”；
- 当前 `chipsdev` 文档需要更清晰地区分：
  - 开发工作区 Node Host 管理命令
  - 真实 Electron Host 联调命令

## 复现步骤

1. 在模块插件仓库执行：

```bash
chipsdev build
chipsdev package
chipsdev plugin install /绝对路径/产物.cpk
chipsdev plugin enable <pluginId>
```

2. 通过开发工作区调用依赖 Electron 的模块 capability。

3. 观察返回：

```text
PLATFORM_UNSUPPORTED
HTML to image export requires Electron BrowserWindow support
```

4. 检查 `chipsdev run`，会发现它不能用于模块插件。

## 期望行为

需要正式收口以下之一：

### 方案A：补正式能力

为模块插件新增 `chipsdev` 的 Electron 联调入口，例如：

- 启动真实 Electron Host
- 复用 `.chips-host-dev`
- 安装并启用模块插件
- 允许发起一次 capability/method 调用
- 输出同步结果或 job 结果

### 方案B：冻结当前边界

如果现阶段不支持模块插件 Electron 联调，则必须在文档中明确声明：

- `chipsdev start` 只管理开发工作区的 Node Host CLI 上下文；
- 不能用于验证依赖 Electron `BrowserWindow` 的模块能力；
- 需要单独的 Electron 验证入口。

## 暂行处理

- 保留当前 `Chips-HtmltoImage-Plugin` 的 mock 回归测试；
- 不再把 `chipsdev start` 下的 `PLATFORM_UNSUPPORTED` 归因到插件业务实现；
- 等待工具链能力或正式边界先收口。

## 需要协作方

- `chipsdev` CLI 维护方
- Host Electron 入口维护方
- 生态文档维护方

## 处理结果

本工单已闭环，正式处理结果如下：

1. `Chips-SDK/cli/index.js` 已新增 `chipsdev module invoke` 命令：

```bash
chipsdev module invoke \
  --capability <capability> \
  --method <method> \
  [--input '<json>'] \
  [--input-file /绝对路径/input.json] \
  [--manifest /绝对路径/manifest.yaml] \
  [--timeout-ms 60000]
```

2. `chipsdev module invoke` 的正式行为已经收口为：
   - 解析并复用 `.chips-host-dev`
   - 自动引导默认主题进入开发工作区
   - 若当前目录或 `--manifest` 指向模块插件工程，则先执行正式构建
   - 拉起真实 Electron Host
   - 重新安装并启用目标模块
   - 通过 `module.invoke` 调用 capability/method
   - 若为异步 job，则等待到 `completed/failed/cancelled` 终态再退出
   - 输出 JSON 结果

3. Host 侧已新增并复用以下 Electron 开发入口文件：
   - `Chips-Host/src/main/electron/dev-plugin-utils.ts`
   - `Chips-Host/src/main/electron/dev-invoke-module.ts`
   - `Chips-Host/src/main/electron/dev-run-app.ts` 已改为复用共享插件重装逻辑

4. 为保证工具链正式可验，已补齐 SDK 测试：
   - 新增 `Chips-SDK/tests/run-cli-module-invoke-tests.cjs`
   - 修复 Electron 可执行路径解析逻辑，使测试可通过 fake Electron 注入验证 Electron runner 入口
   - 修复多组 CLI 工作区测试的 npm cache 隔离问题，避免用户目录缓存权限污染测试结果

5. 已同步更新公共文档，明确以下正式边界：
   - `chipsdev start/plugin/theme/...` 负责开发工作区管理
   - `chipsdev run` 负责应用插件窗口 Electron 联调
   - `chipsdev module invoke` 负责模块 capability/method 的 Electron 联调
   - 依赖 `BrowserWindow` 的模块能力不得只用 `chipsdev start` 验证
