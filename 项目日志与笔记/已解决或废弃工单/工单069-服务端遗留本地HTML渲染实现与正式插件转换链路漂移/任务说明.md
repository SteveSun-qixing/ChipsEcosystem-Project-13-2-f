# 工单069-服务端遗留本地HTML渲染实现与正式插件转换链路漂移

## 背景

当前正在按要求重写社区服务器的产品文档与技术文档。  
在继续核对卡片上传、CDN 替换与 HTML 转换链路时，发现仓库中仍保留一套未接线的本地 HTML 渲染实现。

用户已经明确要求：社区服务器不应自行维护一条与正式转换插件并行的 HTML 生成链路，而应通过 Host 已安装的正式转换插件进行处理，以确保最终效果与编辑引擎一致。

## 证据

### 1. 当前正式链路已经改为 Host + 转换插件

`Chips-CommunityPlatformServer/packages/server/src/pipeline/card-pipeline.ts` 当前实际调用：

1. `uploadResourcesToCdn(...)`
2. `replaceContentUrls(...)`
3. `hostIntegration.packCard(...)`
4. `hostIntegration.convertCardToHtml(...)`

### 2. Host 集成层实际走的是模块能力调用

`Chips-CommunityPlatformServer/packages/server/src/services/host-integration.ts` 中，
`convertCardToHtml(...)` 最终调用的是：

- `module.invoke`
- capability: `converter.file.convert`
- method: `convert`

并且会按环境变量或默认值动态安装卡片插件、主题插件和模块插件。

### 3. 仓库中仍残留一套本地 HTML 拼装实现

`Chips-CommunityPlatformServer/packages/server/src/pipeline/html-render/` 下存在：

- `index.ts`
- `registry.ts`
- `renderer.interface.ts`
- `renderers/*`

该目录实现的是服务端自行拼接 HTML 的旧方案。

### 4. 当前主流程并未引用这套旧实现

搜索 `renderCardToHtml`、`rendererRegistry` 和 `src/pipeline/html-render` 后，确认该目录没有被当前正式运行链路引用。

这说明它已成为遗留死代码，但仍会持续误导后续维护者。

## 影响范围

1. `Chips-CommunityPlatformServer/packages/server/src/pipeline/html-render/*`
2. `Chips-CommunityPlatformServer/技术文档/07-卡片上传与转换流水线.md`
3. `Chips-CommunityPlatformServer/技术文档/08-Host集成与插件管理.md`
4. `Chips-CommunityPlatformServer/需求文档/06-卡片与箱子产品规则.md`

## 复现步骤

1. 查看 `packages/server/src/pipeline/card-pipeline.ts`；
2. 查看 `packages/server/src/services/host-integration.ts`；
3. 搜索 `renderCardToHtml` 与 `rendererRegistry`；
4. 可确认当前正式实现与遗留本地渲染目录并存，但后者未被引用。

## 期望行为

1. 社区服务器仓库只保留一条正式 HTML 转换链路；
2. 该链路必须是 Host + 动态安装插件的正式文件转换链路；
3. 本地 HTML 拼装旧实现应移出活动源码目录，避免继续被误用；
4. 文档统一收口为正式插件转换方案。

## 暂行处理

1. 已登记总工单；
2. 已将遗留本地 HTML 渲染代码从活动源码目录移出；
3. 已继续按正式插件链路编写相关文档。

## 需要协作方

- 社区服务器后端维护方
- Host 集成维护方
- 文档维护方

## 处理结果

1. 已确认当前唯一正式卡片 HTML 转换路径为：
   - 资源上传 CDN
   - URL 替换
   - 重打包新的 `.card`
   - 通过 Host 调用正式转换插件输出 HTML
2. 已将遗留目录从 `packages/server/src/pipeline/html-render/` 移出，避免继续作为活动源码存在。
3. 后续技术文档与需求文档只再描述正式插件转换链路，不再保留“服务端本地拼 HTML”口径。
