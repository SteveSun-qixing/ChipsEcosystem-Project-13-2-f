# 工单073-通用文件关联与应用路由能力缺失导致图片查看器无法正式接入

## 1. 背景

本轮建设 `Chips-PhotoViewer` 时，用户提出了两类诉求：

1. 其他应用把图片交给内核后，由内核自动找到图片查看器并转发；
2. 用户把普通图片文件绑定到薯片生态后，由图片查看器接收打开。

初始核对发现，当时 Host 只对 `.card/.box` 提供正式文件关联接线，普通图片文件无法正式进入图片查看器。

## 2. 原始问题

初始阻塞主要有四点：

1. `app-entry.ts` 只接 `.card/.box`
2. `file-association.ts` 只对 `.card/.box` 做正式分发
3. 安装器只注册 `.card`
4. 图片查看器还没有正式声明图片类型处理能力

## 3. 本轮处理结果

本轮已经完成文件路径图片场景的正式收口：

### 3.1 Host 启动入口泛化

- `Chips-Host/src/main/electron/app-entry.ts`
- 结果：
  - 进程参数、macOS `open-file`、`second-instance` 现在都能接收任意带扩展名文件；
  - Host 不再把正式文件关联入口硬编码为 `.card/.box`。

### 3.2 Host 文件关联分发泛化

- `Chips-Host/src/main/core/file-association.ts`
- 结果：
  - 对非 `.card/.box` 文件，Host 会查询 `file-handler:<ext>`；
  - 命中后通过 `plugin.launch(...)` 把 `targetPath` 转发给目标 app 插件；
  - 未命中时才回退 `platform.shellOpenPath`。

### 3.3 安装器能力驱动注册

- `Chips-Host/src/main/installer/macos-installer.ts`
- 结果：
  - 安装器会扫描内置 app 插件 manifest 中的 `file-handler:<ext>`；
  - 把对应扩展名转换成 `Info.plist` 文档类型；
  - 图片类型注册不再靠手写常量，而是由内置插件能力决定。

### 3.4 图片查看器正式声明能力

- `Chips-PhotoViewer/manifest.yaml`
- 结果：
  - 已声明 `.png/.jpg/.jpeg/.webp/.gif/.bmp/.svg/.avif` 的 `file-handler` 能力；
  - 应用已完成查看、缩放、保存图片副本的首版交付。

### 3.5 “没装插件就不接图片”

- `Chips-Host/src/main/core/built-in-plugins.ts`
- `Chips-Host/src/main/installer/macos-installer.ts`
- 结果：
  - 图片查看器被收口为可选内置插件；
  - 若 Host 发行包不携带图片查看器，安装器不会注册图片类型；
  - Host 启动也不会因缺少图片查看器而失败；
  - 因此“没有安装图片查看器插件时，Host 不接住图片”已经成立。

## 4. 当前剩余边界

本工单解决的是“文件路径图片场景的正式接入”。

以下内容仍属于后续扩展，不阻塞当前图片查看器首版：

1. 非文件路径图片资源的票据化跨应用转交；
2. 多个图片查看器并存时的默认处理器治理；
3. 动态卸载图片查看器后的系统级文件关联回收。
