# 工单074 - Chips-SDK仓库级测试被脚手架类型定义缺失阻断

## 背景

在处理富文本基础卡片 Markdown 文本资源读取链路时，需要同步验证 `Chips-SDK` 新增的编辑器文本资源桥接能力是否在仓库级测试中正常收口。

执行 `Chips-SDK` 根目录 `npm test` 后，发现失败并不来自 SDK 本体，也不来自本轮新增的 `tests/card.test.ts` 覆盖，而是被测试链路中的脚手架工程构建异常提前阻断。

## 证据

- 执行目录：
  - `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-SDK`
- 执行命令：
  - `npm test`
- 实际报错：

```text
error TS2688: Cannot find type definition file for 'lodash-es'.
```

- 报错阶段：
  - `tests/run-cli-create-workspace-tests.cjs`
- 被触发的工程：
  - `Chips-Scaffold/chips-scaffold-app`

## 影响范围

- `Chips-SDK` 仓库级 `npm test` 当前不能稳定作为 SDK 自身改动的正式质量门禁；
- SDK 改动需要额外绕开总测试脚本，改用专项命令验证；
- 开发者容易把脚手架配置缺陷误判为 SDK API 或桥接层回归。

## 复现步骤

1. 进入 `Chips-SDK` 根目录。
2. 执行 `npm test`。
3. 等待测试流程进入 `tests/run-cli-create-workspace-tests.cjs`。
4. 观察 `Chips-Scaffold/chips-scaffold-app` 构建失败并报：
   - `Cannot find type definition file for 'lodash-es'`

## 期望行为

- `Chips-SDK` 仓库级测试应能稳定运行；
- 若脚手架工程属于必测项，则其类型依赖与 `tsconfig` 需要完整闭环；
- 若脚手架工程不应阻断 SDK API 回归，则应拆分测试入口或分层门禁。

## 暂行处理

- 本轮对 SDK 改动改用专项命令验证：
  - `npx vitest run tests/card.test.ts`
- 该专项验证已经通过，可证明当时的资源桥接改动与脚手架修复互不阻断；后续正文资源链路已统一收敛到“运行时资源 URL + 请求方自解析”正式模型。

## 需要协作方

- `Chips-SDK` 维护方
- `Chips-Scaffold` 维护方
- 开发工具链维护方

## 当前状态

- 已解决

## 处理结果

1. 已在 `Chips-Scaffold/chips-scaffold-app/tsconfig.json` 中显式声明：
   - `"types": ["node"]`
2. 该修改使脚手架工程只加载自身正式需要的 Node 类型，不再把生态根工作区中其他可见的 `@types/*` 包误纳入当前工程的隐式类型库扫描。
3. 修复后，`chips-scaffold-app` 已通过：
   - `npm run build`
   - `npm test`
   - `npm run test:templates`
   - `npm run test:e2e`
4. 与本轮 SDK 富文本资源桥修改直接相关的专项验证已通过：
   - `npx vitest run tests/card.test.ts`
5. `Chips-SDK` 仓库级 `npm test` 未继续全量执行，这是按当前用户要求“无需跑全量测试”主动停止，并非本工单仍存在未解决阻断。
