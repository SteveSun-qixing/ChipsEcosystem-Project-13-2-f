# 工单006-HOST-THEME-RUNTIME-CONTRACT 任务说明

## 1. 背景

在推进 `Chips-Host` Phase 7（L11 主题运行时）开发计划与技术文档补全过程中，发现 Host 现有 `theme` 服务实现与生态共用文档约定存在偏差，主要集中在：

- `theme.resolve` 响应结构与含义与《主题接口规范》不一致；
- `theme.contract.get` 仅为占位实现，未能体现组件级主题契约。

该偏差会直接影响后续 L11 Theme Runtime 的接口设计与实现质量，属于“契约与实现偏差 / 生态规范口径不一致”类问题，需要先统一口径再继续实施具体功能。

相关文档与代码位置：

- 规范文档：
  - `生态共用技术文档/主题系统/02-主题接口规范.md`
  - `生态共用技术文档/架构设计/14-Host服务域设计.md`
- Host 实现：
  - `Chips-Host/src/main/services/register-host-services.ts` 中 `themeService` 的 `resolve` 与 `contract.get` 实现
  - `Chips-Host/tests/integration/host-services.test.ts` 中对 `theme.resolve` 的当前用例

## 2. 问题描述

1. `theme.resolve` 接口：
   - 生态共用文档中约定：
     - 请求：`{ chain: string[] }`
     - 响应：`{ resolved: Array<{ id: string; displayName: string; order: number }> }`
   - 当前 Host 实现：
     - 请求：`{ chain: string[] }`
     - 响应：`{ tokens: Record<string, unknown> }`
     - 行为：对 `RuntimeState.themes` 中的 `ThemeRecord.tokens` 做简单深度合并，未区分五层 token，也未返回 resolved 链路元信息。

2. `theme.contract.get` 接口：
   - 生态共用文档中约定返回具有 `scope/parts/states/tokens` 的组件契约集合，供组件库与主题包校验使用。
   - 当前 Host 实现：
     - 忽略真实 theme 契约，仅返回固定占位结构（component 默认 `global`，slots 固定为 `background/foreground/border/radius/shadow`）。

3. 对 L11 Theme Runtime 的影响：
   - 无法在不调整 `theme.resolve` / `theme.contract.get` 语义的前提下，满足五层 token 解析、作用域链解析、变量表生成与契约门禁等完整能力要求；
   - 组件库与 SDK 无法可靠依赖当前 Host 实现作为最终接口；
   - 若直接在现有接口之上添加逻辑，后续一旦口径调整会带来重复重构成本。

## 3. 期望输出

1. 口径统一结论：
   - 明确 `theme.resolve` 的最终语义：
     - 是否保持“解析主题依赖链并返回链路元信息（resolved 列表）”，并将 token 视图另行通过其他接口提供；
     - 或者，更新生态共用文档，使 `theme.resolve` 直接返回解析后的 token 视图（例如五层合并后的扁平 token 表 + 作用域信息）。
   - 明确 `theme.contract.get` 的最终结构：
     - 输出是否需要完全映射主题包 `contracts/theme-interface.contract.json`；
     - 是否需要暴露每个组件的 `scope/parts/states/tokens` 以及校验结果。

2. 文档与代码调整方案：
   - 如需调整口径：
     - 更新 `生态共用技术文档/主题系统/02-主题接口规范.md` 与 `生态共用技术文档/架构设计/14-Host服务域设计.md`；
     - 在 `Chips-Host/技术文档/15-主题运行时实现说明.md` 中同步更新接口说明和数据模型；
   - 在 Host 侧：
     - 更新 `themeService` 的 `resolve` / `contract.get` 类型定义与实现；
     - 补充或修正对应的契约测试与集成测试；
     - 确保 `Chips-Host/tests/integration/host-services.test.ts` 等用例与新口径一致。

3. 对 L11 Theme Runtime 开发计划的影响评估：
   - 在 `Chips-Host/开发计划/07-L11主题运行时开发计划.md` 中补充“依赖本工单闭环”的说明；
   - 根据新的口径，调整 R2-R4 阶段中对 `theme.resolve` / `theme.contract.get` 的使用方式和输出结构。

## 4. 建议的处理步骤

1. 召开小范围评审（生态架构组 + 主题系统项目组 + 组件库项目组）：
   - 对比现有实现与文档；
   - 讨论 Theme Runtime 在 Host 与组件库之间的数据流最佳形态（链路元信息 vs. token 视图 vs. 契约视图）。
2. 形成决议后：
   - 更新生态共用文档中的接口定义；
   - 在 Host 仓单独开一个任务分支，完成 `theme.*` 接口实现与测试调整；
   - 在组件库/SDK 仓补充或调整对 `theme.*` 的调用与契约测试。
3. 完成后在本任务文档中记录：
   - 最终口径摘要；
   - 相关 PR / 提交链接；
   - 回归测试结果与影响面评估。
