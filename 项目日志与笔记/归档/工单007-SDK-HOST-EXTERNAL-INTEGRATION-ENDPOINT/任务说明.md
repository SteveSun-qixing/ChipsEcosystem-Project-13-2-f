# 工单007-SDK-HOST-EXTERNAL-INTEGRATION-ENDPOINT 任务说明

## 1. 背景

薯片生态在早期设计文本中曾假定“薯片 SDK 让第三方开发者可以将薯片能力集成到自己的软件中”，包括博客系统、知识库、课程平台等独立业务系统。但根据 2026-03-07 用户最新决策，当前阶段收紧 SDK 范围：SDK 仅面向薯片主机内部应用插件和生态内部工具，不再承诺直接支持这些第三方业务系统。

现有对外接口基线明确：

- 插件与应用内部通过 `window.chips.*` 访问 Host 能力（Bridge API）。
- 用户与运维通过 `chips` / `chipsdev` 两套命令行工具分别管理本地主机与开发流程（用户操作通过 `chips`，开发流程通过 `chipsdev`）。

在本工单提出时，生态共用层尚未有一份明确“SDK 如何在 Host 外部（Node/浏览器）与主机通信”的正式接口规范。随着 SDK 范围收敛为“仅面向主机内部应用与生态内部工具”，这部分缺口在当前阶段不再构成必须解决的需求。

## 2. 阻断问题

在为 `Chips-SDK` 制定需求与技术方案时，曾出现以下架构级阻断（现作为历史记录保留）：

1. **外部访问模式未定义**
   - 除插件内部 Bridge API 与命令行工具外，未发现用于 SDK → Host 的正式远程调用通道：
     - 无“本地 Host 服务端协议”文档（如 HTTP/IPC/WebSocket 等）；
     - 《前后端接口标准》明确用于社区服务器等外部服务，不适用于 Host 内核通信；
     - 《系统接口标准》与 Host 服务域设计仅描述 Host 内部服务与 Bridge 行为。
   - `chips host` 命令虽然适合自动化脚本，但当前文档并未将其定义为 SDK 可消费的稳定编程接口（命令参数、输出结构、错误语义等未在 SDK 语境下冻结）。

2. **SDK 需求与 Host 能力存在缺口**
   - 设计与白皮书要求 SDK 能在 Node.js / 浏览器环境中访问 Host 能力，典型需求包括：
     - 面向第三方服务的卡片解析与渲染；
     - 通过统一接口读取/写入 card/box 文件；
     - 访问 theme/config/i18n 等服务域。
   - 在缺少正式“Host-External”接口规范的情况下，SDK 无法在不自造通道的前提下实现上述需求。

3. **架构边界风险**
   - 若 SDK 仓自行通过：
     - 直接依赖 `Chips-Host` 内部模块，或
     - 自定义对 `chips host` CLI 的封装并假定输出结构，
   - 会与现有架构文档中“Host 内部接口统一由生态共用文档冻结”的原则冲突，也会给后续 Host 重构与 SDK 版本兼容带来高风险。

在用户明确收紧 SDK 范围之前，无法在保持架构一致性与长期演进能力的前提下，为“Host 外部 SDK 调用场景”设计出可实施的高质量方案。随着 SDK 范围收敛为“仅支持主机内部应用与生态内部工具”，该问题在当前阶段不再构成阻塞。

## 3. 任务目标

在不引入临时实现或隐式通道的前提下，本工单原始目标如下（现视作历史目标，当前阶段已不再要求立即完成）：

1. **明确 SDK-Host 外部通信模式的架构口径**
   - 由生态架构组牵头，Host/SDK/工具链/社区服务器相关项目组共同确定：
     - SDK 与 Host 在 Node.js 环境下的推荐接入模式（如本地服务、IPC 通道、CLI JSON 接口等）；
     - SDK 在浏览器环境下是否需要/允许直接访问本地 Host，以及安全与权限约束。

2. **形成正式的对外接口规范**
   - 在 `生态共用技术文档` 中新增或更新相关章节，例如：
     - 《SDK 使用指南》扩展“Host 外部接入”章节；
     - 如引入本地服务协议，则在“系统接口标准”或新的协议文档中描述具体协议、认证、错误码等；
     - 如采用 CLI JSON 模式，则在命令行手册中冻结输出结构与错误语义。

3. **为 `Chips-SDK` 实施提供稳定基础**
   - 为后续 SDK 实现提供：
     - 可依赖的 Host 外部调用入口；
     - 对应的错误模型与权限模型；
     - 版本兼容策略（例如：需要 Host ≥ 某个版本）。

## 4. 协作与输出物

### 4.1 生态架构组

- 牵头讨论并裁决以下问题：
  - SDK 是否在 vNext 阶段就需要支持“Host 外部”场景，还是先仅支持插件/应用内部场景；
  - 如果需要支持外部场景，推荐路线是：
    - 新增 Host 本地服务端接口（HTTP/IPC），还是
    - 将 CLI 标准化为 SDK 友好的 JSON/流式接口。
- 输出架构决策文档，明确：
  - 推荐模式；
  - 非推荐/禁止模式；
  - 安全与权限边界。

### 4.2 Host 开发组

- 依据架构决策：
  - 设计并实现必要的服务端能力（本地服务或 CLI 扩展）；
  - 提供结构化输出、错误码与认证机制；
  - 补齐契约与自动化测试。

### 4.3 SDK 开发组

- 在决策明确后：
  - 更新 `Chips-SDK/需求文档` / `技术文档` 中与外部接入相关的部分；
  - 基于统一接口实现 `endpoint/server` 相关功能；
  - 补充集成测试，确保跨环境行为一致。

### 4.4 文档协同

- 更新以下文档使其与最终实现一致：
  - `生态设计原稿/19-SDK与协议.md`
  - `生态设计原稿/薯片生态白皮书（综述版）.md` 中 SDK 相关章节
  - `生态共用技术文档/插件开发/08-SDK使用指南.md`
  - 相关架构与接口标准文档

## 5. 当前状态与闭环说明

- 当前状态：
  - 已在 `项目日志与笔记/问题工单.md` 中将本工单状态更新为“已闭环”，并记录了用户将 SDK 范围收紧为“仅面向主机内部应用与生态内部工具”的决策；
  - 已根据该决策更新相关生态设计与 SDK 文档，使其不再对第三方外部集成做承诺。
- 现阶段处理结论：
  - Host 无需在当前版本专门为外部 Node/浏览器 SDK 客户端设计新的对外服务端点，SDK 也不再需要实现对应能力；
  - 如未来重新提出“SDK 支持第三方外部系统集成”的需求，应通过新的工单单独立项设计，届时可复用本工单的历史记录作为背景资料。
