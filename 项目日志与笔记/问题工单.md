- 工单编号：工单009-CLI-VOLTA-WORKSPACE-BIN-RESOLUTION
- 日期：2026-03-09
- 发现阶段：`Chips-CardViewer` 本地运行 `chipsdev run` 问题排查
- 问题类型：生态工具链 / CLI 可执行解析 / Volta 与 npm workspaces 集成
- 问题描述：
  - 在 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer` 目录直接执行 `chipsdev run`，命令尚未进入 `chipsdev` CLI 逻辑，就先被 Volta 拦截并报错：`Could not locate executable chipsdev in your project`。
  - 当前生态根工作区已经将 `chips-sdk` 通过 npm workspaces 链接到根目录 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/node_modules/chips-sdk`，根级二进制 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/node_modules/.bin/chipsdev` 也存在。
  - 但子工程目录 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/node_modules/.bin/` 中没有 `chipsdev`，且 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/node_modules/chips-sdk` 也不存在；依赖被 npm 以 hoisted 策略提升到了根工作区。
  - Volta 在子工程目录执行全局 shim `/Users/sevenstars/.volta/bin/chipsdev` 时，没有沿 npm workspace 向上复用根工作区的 `.bin/chipsdev`，导致它认为当前工程缺少本地可执行文件并提前失败。
  - 该问题不是 `chipsdev run` 子命令实现报错，而是命令解析层在进入 CLI 之前失败。
- 影响范围：
  - 所有在子工程目录直接手敲 `chipsdev <command>` 的开发流程都会受影响，包括 `run`、`build`、`test`、`lint`、`validate`、`package`。
  - 文档中“在工程目录中直接使用 `chipsdev`”的口径，与当前 Volta + npm workspace hoisted 安装方式存在实际偏差。
  - `npm run build`、`npm exec chipsdev -- build` 这类经 npm 解析 PATH 的调用方式不受影响，因此容易造成“脚本可用、直接命令不可用”的认知错位。
- 复现步骤：
  1. 进入 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer`。
  2. 执行 `chipsdev version` 或 `chipsdev run`。
  3. 观察 Volta 直接报错，未进入 `chipsdev` 输出。
  4. 执行 `npm exec chipsdev -- version` 或 `../node_modules/.bin/chipsdev version`。
  5. 观察命令可正常输出版本信息，说明 CLI 本体存在，失败点在 Volta 可执行解析层。
- 期望行为：
  - 在符合生态规范的子工程目录内，开发者直接执行 `chipsdev <command>` 时应稳定可用，不依赖对 Volta/npm 差异的额外了解。
  - 生态文档、工作区安装策略与开发机命令解析行为应保持一致。
- 暂行处理：
  - 当前仅完成原因定位，不做修复。
  - 排查阶段确认可通过 `npm exec chipsdev -- <command>`、`npm run <script>` 或根工作区 `.bin/chipsdev` 间接执行 CLI。
- 需要协作方：SDK 仓库、生态工具链维护方、开发环境规范维护方
- 状态：已闭环

  处理结果（2026-03-09）：
  - 在生态根 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/package.json` 中补齐正式 Volta 工具链声明，统一锁定 Node 与 npm 版本；
  - 所有根工作区覆盖的一方工程 `package.json` 已补齐 `volta.extends`，统一指向生态根 `package.json`；
  - `chipsdev create` 已增强为：在生态根工作区内生成新工程时，除工作区注册外，还会自动写入正确的 `volta.extends`；

- 工单编号：工单010-DEV-RUN-DIST-MANIFEST-ENTRY-RESOLUTION
- 日期：2026-03-09
- 发现阶段：工单009 修复后的 `chipsdev run` 实机验证
- 问题类型：生态工具链 / Host 安装链路 / manifest 入口解析
- 问题描述：
  - 在 `Chips-CardViewer` 中修复 Volta 命令解析后，`chipsdev run` 已能正常进入 CLI 构建与 Host 安装流程，但在 `plugin.install` 阶段失败。
  - 实际错误为 `PLUGIN_ENTRY_NOT_FOUND`，详情显示：
    - `sourcePath`: `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer/dist/manifest.yaml`
    - `entry`: `dist/index.html`
  - 当前 `chipsdev run` 会先构建，再把工程根部 `manifest.yaml` 同步到 `dist/manifest.yaml`，然后以该 `dist/manifest.yaml` 作为安装入口。
  - 但 `Chips-CardViewer/manifest.yaml` 中的 `entry` 仍为 `dist/index.html`，当安装入口已经位于 `dist/manifest.yaml` 时，会导致 Host 继续按 `dist/index.html` 解析，进而查找不存在的二次拼接路径。
- 影响范围：
  - `chipsdev run` 当前仍无法在应用插件工程中完成完整启动闭环。
  - 所有采用“将根 manifest 复制到 `dist/manifest.yaml` 再直接安装”的 app 工程，都可能存在相同入口解析问题。
  - 涉及 `chipsdev build`、`chipsdev run`、Host `plugin.install`、manifest 契约与 app 脚手架模板之间的一致性。
- 复现步骤：
  1. 进入 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer`。
  2. 执行 `chipsdev run`。
  3. 观察构建成功后进入 Host 安装流程。
  4. 观察 `plugin.install` 返回 `PLUGIN_ENTRY_NOT_FOUND`，其中 `entry` 为 `dist/index.html`，而安装源是 `dist/manifest.yaml`。
- 期望行为：
  - `chipsdev run` 在 app 工程中应能基于统一 manifest 契约完成安装、启用、初始化与窗口打开。
  - `manifest.entry`、`dist/manifest.yaml` 的生成策略与 Host `plugin.install` 的入口解析规则必须保持一致。
- 暂行处理：无。根据仓库规范，先登记工单并停止继续推进当前修复。
- 需要协作方：SDK 仓库、Host 仓库、应用脚手架维护方、manifest 契约维护方
- 状态：已闭环

  处理结果（2026-03-09）：
  - 已确认工程根 `manifest.yaml` 是应用插件开发态与发布态的唯一正式清单源；
  - `chipsdev build` 已取消向 `dist/` 复制 `manifest.yaml` 的逻辑，避免生成错误的 `dist/manifest.yaml`；
  - `chipsdev run` 已改为直接将工程根 `manifest.yaml` 传给 Host `plugin.install`，由 Host 按 `entry: dist/...` 装载构建产物；

  - 已验证 `Chips-CardViewer` 中直接执行 `chipsdev run` 成功启动 Host 并打开应用插件，`chipsdev package` 与 `chips plugin install` 联调成功；
