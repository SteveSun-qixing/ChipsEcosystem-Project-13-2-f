- 问题类型：生态工具链 / 本地依赖分发 / 工作区集成
- 问题描述：
  - `Chips-CardViewer/package.json` 依赖 `@chips/component-library` 与 `chips-sdk`，但当前生态仓库不存在可被 `npm install` 直接解析的本地工作区安装方案。
  - `Chips-ComponentLibrary` 仓库当前没有名为 `@chips/component-library` 的实际可安装包；根 `package.json` 名称为 `chips-component-library`，且仅承担 monorepo 管理职责。
  - `Chips-SDK/package.json` 当前为 `private: true`，生态根目录也没有统一的 workspace 根配置，导致插件工程无法按“生态仓库内 sibling repo 联动开发”的方式解析本地 SDK/组件库依赖。
  - 实际复现时，在 `Chips-CardViewer` 目录执行 `npm install`，npm 直接请求公共 registry 并返回 `chips-sdk` / `@chips/component-library` 404。
- 影响范围：
  - `Chips-CardViewer` 当前无法稳定安装依赖、执行构建、测试与后续 Host 联调。
  - 所有依赖 `@chips/component-library`、`chips-sdk` 的生态插件工程与脚手架模板都会受阻。
  - 生态共用文档中“通过 npm 安装 SDK / 组件库”的接入路径与当前仓库现状不一致。
- 复现步骤：
  1. 进入 `/Users/sevenstars/Documents/ChipsCard/Develop/Project-13-2-f/Chips-CardViewer`。
  2. 执行 `npm install`。
  3. 观察 npm 解析 `chips-sdk@^0.1.0` 与 `@chips/component-library@^0.1.0`。
  4. 观察安装流程访问公共 npm registry 并返回 404，安装终止。
- 期望行为：
  - 生态仓库内所有插件工程应存在正式、稳定、可文档化的本地依赖安装方案。
  - `@chips/component-library` 应与生态文档中的包名、导出方式和安装方式保持一致。
  - `chips-sdk` 与组件库应能被插件工程在本地开发期稳定解析，并支持构建、测试、脚手架生成与 CI。
- 暂行处理：无。根据仓库规范，禁止引入临时桥接安装方案或手工改写 node_modules。
- 需要协作方：生态架构组、组件库仓库、SDK 仓库、脚手架/工具链维护方
